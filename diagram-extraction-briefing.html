Executive Summary

What We Built and Why

Architecture diagrams are the most information-dense artifacts in any consulting engagement — and PDF extraction tools destroy them. We designed and built a 5-pass extraction pipeline that converts PDF architecture diagrams into structured, verifiable knowledge graphs: Mermaid diagram-as-code, component tables with technology wiki-links, and graph-bound prose summaries. Every extracted element is individually verified against the source image. The system was validated through a 7-layer evidence stack — three independent blind research reports, three rounds of adversarial pressure testing, and six execution-level trap resolutions — all before writing a single line of production code.

The Problem

Architecture Diagrams Are Invisible to the Pipeline

PDF diagrams are the most information-dense artifacts in consulting decks. Four fundamental gaps made them invisible to folio.love's processing pipeline.

T

Text in Shapes Is Lost

pdfplumber.extract_text() captures flowing paragraph text. Component labels inside vector shapes — "API Gateway", "Auth Service" — are silently dropped.

D

Sparse Diagrams Are Destroyed

Pixel-histogram blank detection fires on line-on-white diagrams. Valid LLM results are overridden with pending() and excluded from downstream passes.

S

Schema Doesn't Fit

SlideAnalysis has consulting-slide types and framework options. No fields exist for nodes, edges, groupings, or Mermaid diagrams.

P

Prompts Don't Ask the Right Questions

The analysis prompt asks for consulting classification — not nodes, edges, flow direction, or component technologies.

Core Principle

Evidence-Driven Escalation

More multimodal context is not monotonically better. Sending 5 images to a simple 5-box flowchart introduces binding noise. The system starts simple and escalates only when evidence indicates insufficiency.

1 Before Pass A Deterministic signals

  • Uses Stage 1 data only — cannot use Pass A output to judge Pass A input
  • pypdfium2 >30 bounded words OR pdfplumber >200 vector lines → escalate to global + tiles
  • Otherwise → global image only (simple path)
  • A dense diagram with only global image may hallucinate 8 large boxes and miss 40 small ones

2 Before Pass B/C Pass A output drives

  • Pass A node count refines escalation for verification depth
  • Simple (≤10 nodes): claim-level verification with global image
  • Medium (11–25): claim-level verification with local highlights
  • Dense (>25): full verification + regional completeness sweep

The Architecture

Extract → Mutate → Verify → Sweep → Render

A five-stage LLM pipeline with deterministic escalation. Each pass builds on the verified output of the previous, with safety checks preventing error propagation.

A

Pass A — Extract LLM

Global 300 DPI image + PDF text inventory → LLM extracts a canonical graph JSON with nodes, edges, groups, and arbitrary stable IDs. Escalates to tiles for medium/dense diagrams based on deterministic Stage 1 signals (word count + vector count), not the LLM's own assessment.

B

Pass B — Mutate LLM + Python

A second LLM acts as visual critic: receives the graph + global image, outputs targeted mutations (add/remove/relabel/regroup/change direction). Python applies mutations deterministically — untouched elements are mathematically protected. If >40% elements change → sanity short-circuit: skip verification, mark for human review.

C

Pass C — Claim-Level Verification LLM Batched

Every node and edge becomes a verifiable claim. Batched 15–20 per LLM call with PIL-generated visual highlights (semi-transparent bounding boxes on the source image). Per-claim: confirmed / rejected / uncertain. Rejected elements removed; uncertain ones generate specific review questions.

S

Completeness Sweep Dense Only

For dense diagrams (>25 nodes): per-cluster regional prompt asks "what components or connections in this region are NOT in the current graph?" Discovers elements all prior passes missed. Added with low confidence + review flag.

R

Deterministic Rendering Pure Python

Verified graph → Mermaid (with recursive nesting + sanitization), graph-bound prose (no inferred semantics), component tables, connection tables, and technology wiki-links. Generated deterministically — never by the LLM.

Key Technical Decisions

Design Decisions That Emerged from Evidence

Each decision was pressure-tested across three blind review rounds. Six execution-level traps were caught and resolved before writing a single line of code.

I

Spatial IoU for Stable IDs

Node IDs are arbitrary (not hash-based). Cross-run stability via bounding box IoU > 0.80 — robust to label changes and tolerant of positional shifts. Human overrides survive re-extraction.

V

Claim-Level Highlights

Rectangular crops spanning source/target cut routing lines. Solution: PIL draws colored bounding boxes directly on the global image. "Look at the highlighted boxes. Is there a continuous line?"

M

Mutations, Not Rewrites

Pass B outputs a targeted mutation list — not a full JSON rewrite. Untouched nodes and edges are mathematically protected. Prevents hallucination regressions in verified elements.

F

Freeze / Override System

folio_freeze: true bypasses the entire pipeline. Frozen Markdown is parsed to hydrate a graph for deck-level aggregation — no LLM calls, no cost, no risk of overwriting human edits.

H

Honest Framing

"Extracts at high quality and flags what needs checking." Confidence reasoning, actionable review questions, and abstention when the system doesn't trust its own output. System learns from corrections.

E

Endpoint-Level Privacy

Not just provider-level controls. Retention policies vary by API endpoint within the same provider. folio.yaml governs allowed API surfaces with per-endpoint restrictions.

What This Enables

From Machine-Opaque to Queryable Data

Today, architecture diagrams are human-readable but machine-opaque — locked in PDFs that no tool can search, join, or compare. Once folio.love extracts them into structured data (nodes, edges, technologies, groups), they become a data source that can be combined with everything else.

Due Diligence at Machine Speed

Today, an analyst stares at a PDF diagram and manually cross-references against CMDB exports. folio.love makes every component, connection, and technology a structured field — enabling teams to join extracted architecture data against CMDB/ServiceNow and surface discrepancies like "diagram says Kafka, CMDB shows RabbitMQ" across hundreds of pages. The extraction is the unlock; the join becomes straightforward.

Triangulated Transformation Assessment

folio.love can extract both current-state and target-state architecture diagrams into structured component lists. Because the output is structured YAML — not pixels — teams can diff them, map components against CMDB records and interview findings, and express migration scope as structured deltas rather than slide annotations. Before extraction, this triangulation is manual; after, it's data work.

LLM-Powered Architecture Analysis

An LLM cannot meaningfully read a PDF architecture diagram — it sees pixels, not structure. folio.love's Markdown output changes this: extracted diagrams can be fed directly to an LLM for cross-cutting analysis. "What are the single points of failure across all 47 system diagrams?" becomes a prompt over structured data rather than a week of analyst review. The extraction makes the corpus LLM-readable.

Knowledge Reuse Across Engagements

folio.love outputs structured YAML frontmatter with tagged components and technologies for every diagram. This enables teams to query across past engagements — "show me every microservices architecture we've seen in financial services" — using Obsidian's Dataview plugin or any tool that reads Markdown + YAML. Architecture knowledge shifts from tribal memory to a searchable, structured library.

Cross-Deck Consistency & QA

When architecture diagrams are structured data, comparing them across workstreams or time periods becomes possible. "The Q1 deck shows 3 microservices, the Q3 deck shows 7 — what changed and does the cost model reflect it?" Today this requires manual slide-by-slide review. With structured extraction, the component lists exist as diffable data.
