# PRD Phase B Spec Review: Reviewer 2 Alignment Review

## 1. Architecture Compliance

- **Critical:** The spec introduces a new canonical top-level field, `provenance_links`, but the approved ontology schema does not allow that field anywhere today. The complete schema enumerates the allowed frontmatter fields by category and type, and `provenance_links` is absent; the roadmap also points back to that schema as the stable target. As written, §9.1 is a schema deviation, not an extension of an existing field family. Evidence: [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L552), [Folio_Ontology_Architecture.md](/Users/jonathanoh/Dev/folio.love/docs/architecture/Folio_Ontology_Architecture.md#L757), [Folio_Ontology_Architecture.md](/Users/jonathanoh/Dev/folio.love/docs/architecture/Folio_Ontology_Architecture.md#L787), [04_Implementation_Roadmap.md](/Users/jonathanoh/Dev/folio.love/docs/product/04_Implementation_Roadmap.md#L148).

- **Critical:** Provenance is specified as a standalone command and separate pipeline, but its machine state is stored under `_llm_metadata.enrich.axes.provenance`. The approved enrich contract defines `_llm_metadata.enrich` as the enrich block with a required top-level shape and enrich-owned axes; the provenance spec does not explain how a standalone `folio provenance` run populates or preserves that contract on notes with no prior enrich history. This is an architectural coupling with enrich lifecycle and schema semantics that is not justified in the approved documents. Evidence: [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L189), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L598), [folio_enrich_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_enrich_spec.md#L666), [folio_enrich_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_enrich_spec.md#L713).

## 4. Backward Compatibility

- **Major:** Reusing `_llm_metadata.enrich` creates an unaddressed stale/refresh dependency. The current shipped refresh logic preserves `_llm_metadata.enrich`, clears enrich fingerprint fields, and drops only `axes.relationships.proposals`; it does not know anything about a future `axes.provenance` subtree. The provenance spec assumes this namespace is already safe because enrich passthrough exists, but stale invalidation and cleanup would require new shared lifecycle rules. Evidence: [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L671), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1083), [cli.py](/Users/jonathanoh/Dev/folio.love/folio/cli.py#L79), [cli.py](/Users/jonathanoh/Dev/folio.love/folio/cli.py#L94), [folio_enrich_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_enrich_spec.md#L1155).

- **Minor:** The spec’s refresh implementation note is incomplete. It says to extend `_inject_enrich_passthrough` in `folio/output/frontmatter.py`, but the current refresh path also requires the extraction side in `folio/cli.py` to be updated, because passthrough is split between extraction and injection. Evidence: [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L671), [cli.py](/Users/jonathanoh/Dev/folio.love/folio/cli.py#L49), [frontmatter.py](/Users/jonathanoh/Dev/folio.love/folio/output/frontmatter.py#L376).

## 6. Deviation Report

- **Critical:** `provenance_links` is added outside the approved ontology schema and without document-type applicability rules.
- **Critical:** `_llm_metadata.enrich.axes.provenance` couples a separate command to the enrich namespace without an approved shared lifecycle/schema contract.
- **Critical:** The spec shifts the roadmap slice from deliverable-to-evidence provenance to evidence/interaction-source provenance.
- **Major:** PRD patch mapping is incomplete: `FR-606` and likely `FR-403` are missing, while `FR-705` is stretched beyond its current meaning.
- **Major:** “Inherits enrich protection rules” is factually inaccurate; enrich still permits metadata updates on protected notes.
- **Major:** Refresh stale handling for a provenance subtree is not aligned with the shipped enrich refresh implementation.
- **Minor:** The fingerprint reuse claim says “exactly,” but the proposed provenance fingerprint contract is only analogous.
- **Minor:** The implementation touchpoint for passthrough is underspecified because refresh uses both extraction and injection helpers.
