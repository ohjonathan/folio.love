**Completeness Check**
Rev 4 is materially better than Rev 3 on governance, stale-aware dedupe, and explicit test intent, but it still does not fully close the remaining concerns around stale-link repair, stale/orphan review at scale, dry-run consistency, and deterministic ordering. The happy path is specified; the operator-critical recovery path is still not.

**Quality Assessment**
The spec is directionally strong and much more reviewable than earlier revs. The appendices make the approval package coherent, the lifecycle states are clearer, and the acceptance criteria now frame PR D correctly as infrastructure-first. The remaining problem is that several “closed” concerns are only partially closed: the spec now names the behaviors, but it still leaves key repair and planning flows under-specified.

**UX Review**
Pending review UX is reasonably solid: stable IDs, pagination, and doc/range actions are useful. Stale/orphan UX is still the weak point. The spec asks reviewers to make high-stakes repair decisions without preserving enough historical context to verify a stale link, and it only offers one-by-one actions for the review mode most likely to explode after refreshes or note churn.

**Issues Found**

Critical
- Rev 4 still does not provide a trustworthy stale-repair UX because confirmed links do not retain the text a reviewer needs to verify them. `provenance_links` stores coordinates and hashes, confirmation removes the proposal record, and `refresh-hashes` is defined as a human visual-verification action, but the spec never persists prior claim/passage text or requires stale-review output to show old vs current content. In practice, a reviewer cannot reliably tell “same meaning, wording changed” from “different claim now occupying the same slot.” This is a direct miss on the stale lifecycle closure claim. §9.1, §13.3, §13.5 ([folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L379), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L395), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L665), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L688))

Major
- `--dry-run` is still not behaviorally consistent with a real run. The pipeline spec says real runs also perform confirmed-link staleness checks and process the `re-evaluate` queue, including bypassing protection when required, but dry-run stops before those steps. That means a protected note can appear metadata-only in dry-run and still trigger LLM work in the real run. The test plan also does not add an explicit parity test for this case. §10, §14.1, §17.13 ([folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L527), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L530), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L542), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L712), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L905))
- The protected-note repair story is broader than the text suggests. Rev 4 says `re-evaluate` bypasses protection “for that pair only,” but in v1 `supersedes` is singular, so the pair is effectively the entire source note against the target. Re-evaluating that pair re-extracts all claims and can generate unrelated new proposals on a `reviewed` or `overridden` note when the user only intended to repair one stale link. That is not a narrowly scoped repair path. §10, §13.5, §14.1 ([folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L491), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L497), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L689), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L712))
- Stale/orphan review still does not scale operationally. Rev 4 added pagination, but unlike pending review it does not add doc/range batch actions. After a refresh or target deletion event, operators are still forced into one-by-one `refresh-hashes`, `remove`, `acknowledge`, or `re-evaluate` actions. That is a real UX gap, not a cosmetic one, and it means the Round 3 “at scale” concern is only partially addressed. §13.2, §13.5 ([folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L652), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L656), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L682), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L686))
- The deterministic-ordering fix is incomplete. Rev 4 adds `proposal_id` as a tie-breaker, but both evaluation order and review order still depend first on “registry order,” and the spec never defines a canonical sort for registry entries. The document claims deterministic ordering and range safety across runs, but the highest-precedence ordering key is still unspecified. §8.9, §10.3, §18 ([folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L336), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L337), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L549), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L946))

Minor
- Review-state integration is still ambiguous. The spec adds `provenance_link_stale` as a review flag, but it does not say whether stale provenance should also auto-set `review_status: flagged`, when that flag clears after repair, or how acknowledged/orphaned states interact with the existing review model defined in the PRD and ontology. §14.3, Appendix B, FR-703/704 ([folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L741), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1100), [02_Product_Requirements_Document.md](/Users/jonathanoh/Dev/folio.love/docs/product/02_Product_Requirements_Document.md#L606), [02_Product_Requirements_Document.md](/Users/jonathanoh/Dev/folio.love/docs/product/02_Product_Requirements_Document.md#L634), [Folio_Ontology_Architecture.md](/Users/jonathanoh/Dev/folio.love/docs/architecture/Folio_Ontology_Architecture.md#L825))
- Orphan handling is stricter than necessary. The spec forbids `re-evaluate` for orphaned links even though the repair mechanism is pair-level semantic re-matching, not coordinate patching. That forces remove-and-rerun behavior for a case where the system could attempt targeted repair directly from stale review. §13.5 ([folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L689), [folio_provenance_spec.md](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L700))
