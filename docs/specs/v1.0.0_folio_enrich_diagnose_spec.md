# v1.0.0 — `folio enrich diagnose` (Tier-4 Graph Quality Layer Sub-Item A of 3)

## 1. Overview

**Delivers:** Tier-4 Roadmap row #4 sub-item A ("Graph quality layer") from `docs/product/04_Implementation_Roadmap.md` §"Week 26-27": the read-only `folio enrich diagnose` surface that surfaces enrichability blockers per parent enrich spec §7.7. Identifies registry-managed notes whose managed sections cannot be safely updated by `folio enrich` — exposing the blockers operators would otherwise discover only when graph-density work stalls silently.

**One-sentence summary:** `folio enrich diagnose [scope] [--json] [--limit N]` walks the enrichment plan via the existing `plan_enrichment` predicate, filters to non-eligible dispositions, maps each to a stable finding `code` per parent §7.7's vocabulary, and emits a deterministic report (text or JSON) that names the blocker, severity, subject, detail, recommended remediation, and trust-status annotation — never writes anything to disk.

**Target version:** v1.0.0 (new minor in `pyproject.toml`; first major-version-threshold crossing).

**Theme:** Tier-4 graph quality layer — sub-item A (per-note hygiene check). Establishes the diagnostic surface that downstream sub-items B (trust-aware graph behavior) and C (relation-schema validation) build on operationally — operators need diagnose first to understand which notes are at risk before the system starts silently filtering flagged inputs in B.

**Risk level:** `low` (feature) + `low` (migration). The core predicate is `plan_enrichment` (already shipped, fully tested) — diagnose is a thin filter + renderer over its output. Primary risks: (1) `folio/cli.py` group reshape regression (existing `folio enrich [scope]` invocation must keep working) — mitigated by `@cli.group(invoke_without_command=True)` pattern proven by `folio digest` at `folio/cli.py:1179-1234` plus regression tests ED-CLI-2 / ED-CLI-3; (2) scope creep into sub-item B via trust-posture overreach — mitigated by §6 firewall + manifest cardinality assertion #5 (grep firewall on `derive_trust_status` in diagnose code path); (3) finding-code drift if `_determine_disposition` reasons change — mitigated by exhaustive (disposition, reason)-tuple test coverage in §8 (ED-1..6).

### 1.1 Core behaviors

1. `folio enrich diagnose [SCOPE]` walks `plan_enrichment(config, scope=SCOPE, force=False, dry_run=True)` and filters returned `EnrichPlanEntry` rows to those that are **not safely enrichable**: dispositions `protect`, `conflict`, plus the special `(analyze, "stale")` tuple representing a stale enrich shell. Entries with disposition `skip` (fingerprint match) and `(analyze, "eligible")` (ready to enrich) are omitted — these are healthy.
2. Each surfaced entry maps deterministically to one of six **finding codes** (§5) with a fixed `severity` (error/warning/info), `subject_id`, `detail`, and `recommended_action`.
3. Text-mode output is sorted by `(severity_rank desc, code asc, subject_id asc)` — within a severity tier, findings group by finding code; within a code group, findings sort alphabetically by subject_id. Per-line shape `[severity [flagged?]] code subject_id: detail\n  Action: recommended_action`. JSON-mode output is the §7 envelope.
4. **Scope semantics** match `folio enrich`: optional positional `SCOPE` argument resolves to a deck_dir / markdown_path prefix; omitted = library-wide. Reuses `plan_enrichment`'s existing scope-prefix semantics (which calls the local `_matches_scope` helper at `folio/enrich.py:68`); `diagnose_notes` does NOT import from `folio.links`.
5. **Trust posture** (per operator-confirmed product shape; §6 firewall): every eligible note is surfaced regardless of `review_status`. Findings carry an additive `trust_status: "ok" | "flagged"` key (JSON) and a `[flagged]` annotation INSIDE the severity bracket (text — `[error [flagged]] code …`, mirroring `folio graph doctor` at `folio/cli.py:2176-2185` for cross-surface cohesion). NO default exclusion. NO `--include-flagged` flag. NO direct call to `folio.tracking.trust.derive_trust_status` — see §6 for the rationale and the manifest's grep-firewall enforcement.
6. **`--limit N`** caps the rendered findings post-sort; does NOT short-circuit `plan_enrichment` (which is O(n) over the registry already). When findings exceed N, the renderer prints a `(showing N of M; raise --limit to see more)` footer (text) and includes the truncation in the envelope `summary` (JSON).
7. **`--json`** emits the §7 envelope at `schema_version: "1.0"` with seven top-level keys: `schema_version`, `command`, `scope`, `limit`, `findings`, `summary`, `truncated`. Each finding carries the parent §7.7 fixed schema (5 keys: `code`, `severity`, `subject_id`, `detail`, `recommended_action`) plus an additive `trust_status` key (operator-confirmed extension of §7.7 — the parent spec's "fixed schema" is read as a MINIMUM contract, additive-only).
8. **Exit codes**: `0` always when the run completes (presence of findings is informational, not pass/fail — parity with `folio graph doctor`); `1` on fatal config / scope-resolution / registry / I/O errors; `2` on Click usage error (negative `--limit`).
9. **Read-only invariant**: diagnose calls `plan_enrichment(config, dry_run=True)`, which already honors the dry-run-writes-nothing rule per `folio/enrich.py:233-261`. Diagnose adds zero further mutations. ED-13 in §8 verifies `registry.json` mtime is unchanged across an invocation.
10. **No LLM calls**: diagnose surfaces what `_determine_disposition` already computes; no model traffic. ED-12 verifies the call graph contains zero LLM imports.

```python
DIAGNOSE_COMMAND_NAME = "enrich diagnose"


@dataclass(frozen=True)
class DiagnoseFinding:
    """A single enrichability blocker for a registry-managed note.

    Field order mirrors parent spec §7.7 "fixed schema" exactly for the
    first 5 keys (code, severity, subject_id, detail, recommended_action),
    then trails the additive `trust_status` key per operator-confirmed

    severity: str           # one of {"error", "warning", "info"}
    subject_id: str         # registry deck_id (e.g., "clienta_dd_q126_evidence_20260301_market-sizing")

    PEER-MIN-003 closure: by_code is typed Mapping (read-only contract)
    and constructed via types.MappingProxyType so frozen-dataclass
    immutability extends to the value level (consumers cannot mutate
    `result.summary.by_code["x"] = 1`).
    """
    total: int                                   # finding count (post-limit)
    by_code: Mapping[str, int]                   # MappingProxyType wrapper; ordered by code asc
    flagged_total: int                           # count of findings with trust_status == "flagged"

    command: str                              # always == DIAGNOSE_COMMAND_NAME
    scope: str | None                         # the requested scope (None == library-wide)
    limit: int | None                         # the requested limit (None == unbounded)
    findings: tuple[DiagnoseFinding, ...]     # length ≤ limit when limit is not None
    summary: DiagnoseSummary
    truncated: bool                           # True iff len(findings) < unfiltered_match_count


def diagnose_notes(
    config: FolioConfig,
    *,
    scope: str | None = None,
    limit: int | None = None,
) -> DiagnoseResult:
    """Run the read-only enrichability hygiene check.

    Performs scope-resolution preflight (CB-3 closure): when scope is
    non-None, verifies it matches at least one registered deck. Raises
    ScopeResolutionError on a typo'd scope (parent §7.7 contract: invalid
    scope is fatal exit non-zero — NOT a silent empty-finding result).

    Validates limit (ADV-SF-004 closure): raises ValueError if limit is
    not None and limit < 1.

    Walks plan_enrichment(config, scope=scope, force=False, dry_run=True),
    filters EnrichPlanEntry rows whose (disposition, reason) tuple maps to
    a §5 finding code, sorts by (severity_rank desc, code asc, subject_id
    asc) per CSF-1, and truncates to limit when set. Builds the §7 envelope.

    Pure-function: no side effects, no mutations, no file writes, no LLM
    calls. plan_enrichment honors dry_run=True per folio/enrich.py:233-261.
    """
    # CB-3 closure: scope-resolution preflight.
    # Parent §7.7 says invalid scope is fatal. We can't distinguish
    # "scope matches no eligible decks" from "scope is a typo" inside
    # plan_enrichment (which silently skips non-matches at folio/enrich.py:294-298).
    # Solve by verifying scope matches at least one registered deck.
    if scope is not None:
        _resolve_scope_or_raise(config, scope)  # raises ScopeResolutionError

    # ADV-SF-004 closure: function-level limit validation.
    if limit is not None and limit < 1:
        raise ValueError(f"limit must be >= 1 or None; got {limit!r}")

    plan: list[EnrichPlanEntry] = plan_enrichment(
        config, scope=scope, force=False, dry_run=True,
    )

    raw_findings: list[DiagnoseFinding] = []
    for entry in plan:
        finding = _entry_to_finding(entry)  # see §3.2
        if finding is not None:
            raw_findings.append(finding)

    # CSF-1 closure: deterministic order — severity rank desc (error → warning → info),
    # then code asc (groups within severity tier), then subject_id asc.
    raw_findings.sort(key=_finding_sort_key)

    unfiltered_count = len(raw_findings)
    if limit is not None:
        rendered = tuple(raw_findings[:limit])
    else:
        rendered = tuple(raw_findings)

    by_code_dict = dict(sorted(Counter(f.code for f in rendered).items()))
    summary = DiagnoseSummary(
        total=len(rendered),
        by_code=MappingProxyType(by_code_dict),  # PEER-MIN-003 closure: value-level immutability
        flagged_total=sum(1 for f in rendered if f.trust_status == "flagged"),
    )

    return DiagnoseResult(
        schema_version=DIAGNOSE_SCHEMA_VERSION,
        command=DIAGNOSE_COMMAND_NAME,
        scope=scope,
        limit=limit,
        findings=rendered,
        summary=summary,
        truncated=len(rendered) < unfiltered_count,
    )


class ScopeResolutionError(Exception):
    """Raised when --scope does not resolve to any registered deck.

    CB-3 closure: parent §7.7 says invalid scope is fatal exit non-zero.
    Diagnose treats a non-None scope that matches zero decks as fatal
    (typo guard). Catches case where operator types `folio enrich
    diagnose ClinetA` (typo) and gets a misleading "No findings."
    """
```

### 3.2 Internal helpers (`folio/enrich.py` private additions)

```python
# Severity rank for deterministic sort (descending = worst first).
_SEVERITY_RANK = {"error": 3, "warning": 2, "info": 1}


def _finding_sort_key(f: DiagnoseFinding) -> tuple[int, str, str]:
    """Three-level sort key (CSF-1 closure):
    severity desc (worst first), then code asc (group within tier),
    then subject_id asc (tie-break alphabetic).
    """
    return (-_SEVERITY_RANK[f.severity], f.code, f.subject_id)


def _resolve_scope_or_raise(config: FolioConfig, scope: str) -> None:
    """CB-3 closure: scope-resolution preflight.

    Verify scope matches at least one registered deck. Raises
    ScopeResolutionError if scope is non-empty and no deck's deck_dir or
    markdown_path satisfies the scope-prefix predicate.

    Caller is `diagnose_notes` only; called when scope is non-None.
    Library-wide scope (None) bypasses this check entirely.
    """
    library_root = config.library_root.resolve()
    registry_path = library_root / "registry.json"
    if not registry_path.exists():
        # Empty library is not "scope unresolvable" — let plan_enrichment
        # bootstrap the empty registry and return zero findings cleanly.
        return
    data = registry_mod.load_registry(registry_path)
    if data.get("_corrupt"):
        # CB-3 closure: explicitly-corrupt registry is fatal per parent §7.7
        # ("unreadable registry is fatal exit non-zero"). plan_enrichment
        # would silently rebuild — diagnose must not silently mask corruption.
        raise ScopeResolutionError(
            f"Registry at {registry_path} is corrupt and cannot be parsed safely. "
            f"Investigate manually before re-running diagnose."
        )
    for entry_data in data.get("decks", {}).values():
        if not isinstance(entry_data, dict):
            continue
        entry = registry_mod.entry_from_dict(entry_data)
        if entry.type not in ("evidence", "interaction"):
            continue
        if (_matches_scope(entry.markdown_path, scope) or
                _matches_scope(entry.deck_dir, scope)):
            return
    raise ScopeResolutionError(
        f"Scope '{scope}' did not match any registered evidence or interaction "
        f"deck. Run 'folio status' to list registered scopes."
    )


# PROD-SF-002 + PROD-SF-003 closure: action templates use f-string
# substitution at construction time so {subject_id} is materialized,
# not leaked as a literal token. Curation/review_status text now names
# the manual frontmatter-edit mechanism (no folio promote downward, no
# folio unflag command exists today).
def _entry_to_finding(entry: EnrichPlanEntry) -> DiagnoseFinding | None:
    """Map an EnrichPlanEntry to a DiagnoseFinding, or None if healthy.

    Matches the (disposition, reason) tuple returned by
    _determine_disposition. Reason strings are parsed by prefix where
    they encode a value (e.g. "curation_level=L2" → prefix
    "curation_level=").

    See §5 for the full mapping table.
    """
    disposition = entry.disposition
    reason = entry.reason
    subject_id = entry.entry.id

    if disposition == "protect":
        if reason == "frontmatter unreadable":
            code, severity, action = (
                "frontmatter_unreadable",
                "error",
                "Frontmatter cannot be parsed; restore the YAML block manually before re-running enrich.",
            )
        elif reason == "managed sections not identifiable":
            code, severity, action = (
                "managed_sections_unidentified",
                "warning",
                "Verify the note's `## Slide N` / `### Analysis` / `## Related` / `## Entities Mentioned` headings are intact and not nested inside fenced code blocks.",
            )
        elif reason.startswith("curation_level="):
            code, severity, action = (
                "protected_by_curation_level",
                "warning",
                "Body protection is intentional. To allow enrich to modify this note, edit the note's `curation_level` frontmatter field to a lower tier (L1 or L0) — no command demotion path exists today. Otherwise accept that enrich will not modify this note.",
            )
        elif reason.startswith("review_status="):
            code, severity, action = (
                "protected_by_review_status",
                "warning",
                "Body protection is intentional. To allow enrich to modify this note, edit the note's `review_status` frontmatter field back to `clean` (no `folio unflag` command exists today). Otherwise accept that enrich will not modify this note.",
            )
        else:
            # ADV-SF-003 / ED-15a closure: Defensive default for unknown
            # protect reasons. Surface as the unidentified-class.
            code, severity, action = (
                "managed_sections_unidentified",
                "warning",
                f"Unrecognized protect reason: {reason}. Investigate manually.",
            )
    elif disposition == "conflict":
        if reason == "managed body fingerprint mismatch":
            code, severity, action = (
                "managed_body_conflict",
                "error",
                f"Managed body content has been edited by hand. Reconcile manually, then re-run with `folio enrich --force {subject_id}` to overwrite.",
            )
        else:
            # ADV-SF-003 / ED-15b closure: Defensive default for unknown
            # conflict reasons (parallel to protect-class fallback).
            code, severity, action = (
                "managed_sections_unidentified",
                "warning",
                f"Unrecognized conflict reason: {reason}. Investigate manually.",
            )
    elif disposition == "analyze" and reason == "stale":
        # Stale enrich shell — input changed via folio refresh or
        # folio ingest; the note IS enrichable, but the previous enrich
        # state is invalidated. Surface as info.
        code, severity, action = (
            "enrich_status_stale",
            "info",
            f"Enrich state is stale (source was re-ingested or refreshed). Re-run `folio enrich {subject_id}` to refresh.",
        )
    elif disposition == "analyze" and reason != "eligible":
        # ADV-SF-003 / ED-15c closure: Defensive default for unknown
        # analyze reasons OTHER than the healthy "eligible" state.
        code, severity, action = (
            "managed_sections_unidentified",
            "warning",
            f"Unrecognized analyze reason: {reason}. Investigate manually.",
        )
    else:
        # disposition in {"analyze" (with "eligible" reason), "skip"}
        # → healthy, omit.
        return None

    review_status = entry.existing_fm.get("review_status", "clean")
    trust_status = "flagged" if review_status == "flagged" else "ok"

    return DiagnoseFinding(
        code=code,
        severity=severity,
        subject_id=subject_id,
        detail=reason,
        recommended_action=action,
        trust_status=trust_status,
    )
```

## 5. Disposition → Finding Code Mapping

Diagnose's contract is to surface every (disposition, reason) tuple from `_determine_disposition` (`folio/enrich.py:345-415`) that represents a non-eligible state, mapped deterministically to a stable finding code. Healthy states (eligible, fingerprint-match-skip) are omitted.

### 5.1 Finding-code table

## 6. Trust Posture (Firewall Against Sub-Item B)

### 6.1 Operator-confirmed shape (annotation-only)

Per the Phase-A AskUserQuestion (2026-04-17), `folio enrich diagnose` adopts an **annotation-only** trust posture (PEER-SF-004 closure: scope is annotation, NOT flagged-note enumeration):

1. **`trust_status` annotation on every surfaced finding.** Every `DiagnoseFinding` carries a `trust_status: "ok" | "flagged"` field. Text mode renders `[flagged]` INSIDE the severity bracket (e.g., `[error [flagged]]`) when `trust_status == "flagged"`. This means: when a note has a managed-section blocker AND its `review_status` is `flagged`, the operator sees the trust status alongside the blocker. A note whose ONLY state deviation is `review_status: flagged` (no blocker) is NOT a finding — diagnose surfaces blockers, not flagged-note censuses. For flagged-note enumeration, operators should use `folio status --flagged` (where supported) or `folio links review --include-flagged` per slice-4 discipline.
2. **No default exclusion.** Diagnose surfaces every eligible note regardless of `review_status` — when it has a managed-section blocker, its trust posture is annotated; when it doesn't, it's omitted as healthy.
3. **No `--include-flagged` flag.** Adding it would suggest a default exclusion exists; there is none.
4. **No `excluded_flagged_count` envelope key.** Same rationale.
5. **Direct frontmatter read, NOT `derive_trust_status`.** Trust is read as `entry.existing_fm.get("review_status") == "flagged"` directly. The shared helper `folio.tracking.trust.derive_trust_status` operates on a `RelationshipProposalView` (`folio/tracking/trust.py:6`) whose `.flagged_inputs` set marks WHICH proposal endpoint is flagged — not applicable to a single note.

### 6.2 Why this is NOT slice-4 (v0.6.4) inheritance

Slice 4 implements five rules for **discovery surfaces** (per `docs/specs/v0.6.4_trust_gated_surfacing_spec.md` §1, mirroring parent design spec §11; ALI-MIN-001 closure — rule numbering aligned with v0.6.4 enumeration):
1. Default-exclude flagged inputs.
2. Surface trust posture explicitly (`[flagged]` annotation tag).
3. `--include-flagged` operator override.
4. Always disclose exclusion counts ("N flagged inputs excluded").
5. No silent-empty (custom header when all proposals filtered).

Diagnose is not a discovery surface — it is a diagnostic health-check. Hiding flagged notes by default would defeat its purpose. **Rule 2 (annotation) is the only inherited piece.** Rules 1, 3, 4, 5 are explicitly NOT inherited.

### 6.3 Sub-item B firewall (manifest-enforced)

Sub-item B (deferred to a follow-on slice in Tier-4 row #4) extends slice-4 discipline to **all Tier-4 graph-oriented surfaces** (digest, graph, synthesize, search) per Roadmap §"Week 26-27" line 571-573. Specifically: blanket default-exclusion of `review_status: flagged` inputs across discovery surfaces.

Diagnose MUST NOT pre-implement any of B's surface area. The manifest enforces this with **G-scope-5** (CB-4 closure: extended to cover both `folio/enrich.py` and the marker-bounded diagnose CLI block in `folio/cli.py`):

```bash
# G-scope-5 (a): no derive_trust_status or --include-flagged in folio/enrich.py
! grep -nE 'derive_trust_status|include[-_]flagged' folio/enrich.py

# G-scope-5 (b): no --include-flagged inside the v1.0.0 diagnose CLI marker block in folio/cli.py
# (Existing --include-flagged on digest/search/graph doctor/synthesize lives OUTSIDE the marker block.)
! awk '/^# v1\.0\.0 diagnose CLI: BEGIN/,/^# v1\.0\.0 diagnose CLI: END/' folio/cli.py | grep -nE 'include[-_]flagged|derive_trust_status'
```

Plus runtime test ED-CLI-19 asserts `folio enrich diagnose --help` does NOT contain the literal `--include-flagged`. This is a hard gate at D.6. If sub-item B implementation later needs diagnose to consume `derive_trust_status`, that change lives in B's slice — not v1.0.0.

### 6.4 Why `trust_status` is an additive over §7.7's "fixed schema"

Spec interpretation: parent spec §7.7's `--json` schema enumerates 5 finding keys (`code`, `severity`, `subject_id`, `detail`, `recommended_action`) and labels them "fixed". This v1.0.0 spec reads "fixed" as the **MINIMUM** contract — additive optional keys are compatible. Justification:

1. Parent §7.7 also says "minimum finding classes" for the code enum (3 minimum, additive allowed). Symmetric reading is consistent.
2. Operator-confirmed product shape requires per-finding trust annotation.
3. The shared envelope discipline in v0.9.0 search (sub-slice 3) successfully argued for envelope-key additive growth via minor-version bump. Diagnose at v1.0.0 surfaces the additive `trust_status` key from initial release; future evolutions would minor-bump SCHEMA_VERSION per the same versioning policy (consumers MUST bump minor when adding a top-level or per-finding key).

A future doc-maintenance pass should amend parent §7.7's wording from "fixed schema" to "minimum schema" to reconcile the contract description with the additive-extension precedent established here. Tracked in §11 carry-forwards.

### 7.4 Envelope-vs-shared-§5 disclaimer

This envelope is deliberately independent of the v0.9.0 shared envelope (a **fresh schema_version 1.0** specific to diagnose). It is intentionally **NOT** the v0.9.0 shared `--json` envelope used by `folio search` / `folio synthesize` / `folio graph` (which ships at schema_version `"1.1"`). Justification:

1. The shared envelope is for relationship-proposal consumers — its `findings` are §5 proposal objects (11 keys: `proposal_id`, `proposal_type`, `source_id`, `target_id`, `subject_id`, `evidence_bundle`, `reason_summary`, `trust_status`, `schema_gate_result`, `producer`, `input_fingerprint`, `lifecycle_state`, plus `relation`, `flagged_inputs`).
2. Diagnose's `findings` are NOTE-level hygiene blockers, not proposals. Reusing the proposal-envelope shape would force null-padding 9+ keys per finding and break the §7.7 "fixed schema" minimum.
3. Diagnose's domain is independent: no shared collector, no `RelationshipProposalView`, no proposal lifecycle. Hoisting to a shared module would be premature abstraction.

## 9. Failure Policy

CB-3 closure: parent §7.7 specifies fatal-on-invalid-scope / fatal-on-unreadable-registry semantics. v1.0 v1 spec contradicted that with empty-findings-exit-0; v1.1 honors §7.7 via explicit scope-resolution preflight in `diagnose_notes` (raises `ScopeResolutionError`) and explicit corrupt-registry detection.

| Scenario | Behavior |
|---|---|
| Library lock held by concurrent `folio enrich` | Diagnose acquires no lock (parity with `folio graph doctor`). Concurrent reads are safe; concurrent registry mutation may produce a stale snapshot but no corruption. Documented as a non-concurrency-safe property. |
| `registry.json` missing | `_resolve_scope_or_raise` returns immediately (empty library is not "scope unresolvable"); `plan_enrichment` bootstraps in-memory with `dry_run=True` (no write). Diagnose returns empty findings, exit 0. (Library-wide scope = None bypasses preflight entirely.) |
| `registry.json` corrupted (parsable but `_corrupt: true` flag) | (CB-3 closure) `_resolve_scope_or_raise` raises `ScopeResolutionError`; CLI exits 1 with `✗ Registry at <path> is corrupt and cannot be parsed safely. Investigate manually before re-running diagnose.` Honors parent §7.7 fatal-on-unreadable-registry. |
| Scope arg supplied AND matches no registered evidence/interaction decks | (CB-3 closure) `_resolve_scope_or_raise` raises `ScopeResolutionError`; CLI exits 1 with `✗ Scope '<scope>' did not match any registered evidence or interaction deck. Run 'folio status' to list registered scopes.` NOT exit 0; NOT "No enrichment hygiene findings." (the v1.0 v1 false-clean is fixed). |
| Scope arg = None (library-wide) AND zero findings | `findings == []`, `summary.total == 0`, exit 0. Text mode prints "No enrichment hygiene findings." This is the genuine healthy-library case — no scope-resolution preflight applies. |
| `--limit N` ≤ 0 (CLI) | Click `IntRange(min=1)` rejects → exit 2. |
| `diagnose_notes(limit=N)` with N ≤ 0 (programmatic) | (ADV-SF-004 closure) `diagnose_notes` raises `ValueError("limit must be >= 1 or None; got -1")`. Programmatic callers receive the exception; CLI catches as ValueError → exit 1. |
| LLM profile invalid (config error) | `plan_enrichment` falls back to `profile_name = "default"` per `folio/enrich.py:264-268`; diagnose succeeds (LLM is not actually called). |
| `_determine_disposition` raises | The exception propagates from `plan_enrichment` — diagnose's outer `try/except` catches as `Exception` and exits 1 with `✗ Fatal error:`. |
| Per-note frontmatter parse failure | Already handled in `_determine_disposition` — returns `("protect", "frontmatter unreadable")`, surfaced as `frontmatter_unreadable` finding. Does not abort the run. |
| File system error on note read | `MarkdownDocument` parsing already guards in `plan_enrichment`; per-file errors do not abort the batch. |
| Future `_determine_disposition` reason not enumerated in §5.1 (`protect`, `conflict`, or non-`eligible` `analyze`) | (ADV-SF-003 closure) Surfaced as `managed_sections_unidentified` (warning) with `recommended_action = "Unrecognized {disposition} reason: {reason}. Investigate manually."` Does not silently disappear. |
| Future `_determine_disposition` disposition value not in {protect, conflict, analyze, skip} | (ED-15d) Returns None (omits the entry) AND emits a `RuntimeWarning` so a future disposition addition surfaces visibly to operators rather than silently. |
