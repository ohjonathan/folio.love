# v0.6.2 Provenance Lifecycle Rename Spec

## 1. Overview

This spec implements **PROV-1** from the slice-2 carry-forward table (`docs/specs/v0.6.1_proposal_lifecycle_rename_spec.md` §8). It renames the `ProvenanceProposal.status` field to `lifecycle_state` with the same value mapping as slice 2's `RelationshipProposal` rename: `pending_human_confirmation` → `queued`; `rejected` → `rejected`.

**Migration-only; no behavior change.** The `stale_pending` value (written by cli.py `_mark_enrich_stale`) is a provenance-specific lifecycle state that passes through unchanged — it is not in the 6-state `PROPOSAL_LIFECYCLE_STATES` vocabulary but remains valid for provenance proposals.

**Operator upgrade path:** no action required. `from_dict` reads legacy `status` transparently.

## 3. Storage Schema Migration

### 3.1 Field rename

Same pattern as slice 2 (`v0.6.1_proposal_lifecycle_rename_spec.md` §3):

| Layer | Before | After |
|-------|--------|-------|
| Persisted frontmatter key | `status` | `lifecycle_state` |
| Dataclass field | `ProvenanceProposal.status` | `ProvenanceProposal.lifecycle_state` |
| `to_dict()` output key | `"status"` | `"lifecycle_state"` |

### 3.2 Value mapping

Reuses `_STATUS_TO_LIFECYCLE` from `folio/pipeline/enrich_data.py`:

| Old value | New value |
|-----------|-----------|
| `"pending_human_confirmation"` | `"queued"` |
| `"rejected"` | `"rejected"` |
| `"stale_pending"` | `"stale_pending"` (pass-through; provenance-specific) |

### 3.3 Backward-compat read strategy

`from_dict`: same `is None` guard pattern as slice 2. Import `_STATUS_TO_LIFECYCLE` from `enrich_data`.

Raw-dict readers: same `is None` fallback. For lines that compare to `"pending_human_confirmation"`, use dual-check `state in ("queued", "pending_human_confirmation")` to handle both new and legacy frontmatter. For lines that compare to `"rejected"`, the value is unchanged — no dual-check needed.

### 3.4 cli.py `_mark_enrich_stale` (lines 133-134)

Current:
```python
if proposal.get("status") == "pending_human_confirmation":
    proposal["status"] = "stale_pending"
```

After:
```python
state = proposal.get("lifecycle_state")
if state is None:
    state = proposal.get("status")
if state in ("queued", "pending_human_confirmation"):
    proposal["lifecycle_state"] = "stale_pending"
```

The dual-value check (`"queued"` or legacy `"pending_human_confirmation"`) ensures both old and new frontmatter are handled. After the write, the proposal has `lifecycle_state: stale_pending`.
