# Spec v1.0 — proposal-review-hardening-v0-6-0

## 1. Overview

**Delivers:** the smallest coherent implementation slice of the Tier 4 Proposal Review Hardening proposal (`docs/specs/tier4_discovery_proposal_layer_spec.md` rev 4). Specifically, **rejection-memory enforcement at surfacing time** for relationship proposals, plus a **cumulative-acceptance-rate diagnostic** surfaced via `folio graph doctor`. After this slice ships, a rejected proposal stays hidden from operator review until its material input basis changes — the core §10.1 invariant from the proposal doc — and operators can inspect per-producer acceptance-rate health without any new persistence, timestamps, or producer-state files.

**One-sentence summary:** when `folio links review` collects pending relationship proposals, it now filters out any proposal whose `basis_fingerprint` matches a prior rejection recorded in the same frontmatter, and `folio graph doctor` surfaces each producer's cumulative acceptance rate with a 10-reviewed-proposal warm-up.

**Target version:** v0.6.0 (next minor after v0.5.2 graph-ops layer).

**Theme:** Tier 4 proposal-layer governance — operator-observable rejection memory.

**Risk level:** `medium` (feature) + `high` (migration). The feature itself is a narrow filter plus a diagnostic aggregator. The migration risk is higher than initially rated: `collect_pending_relationship_proposals` has **seven call sites across three files** (not two as v1.0 incorrectly claimed), and its return-type change from `list[...]` to `tuple[list, dict[str, int]]` touches all of them. v1.1 explicitly enumerates every call site in §2 in-scope and §7 Migration; the implementer must update every site in one commit to keep the test suite green.

## 2. Scope

### In-scope (this spec commits)

- [ ] **Rejection-memory filter** in `collect_pending_relationship_proposals` (`folio/links.py`) that excludes proposal views whose rejection-key tuple (defined in §4.2) matches an existing `status == "rejected"` entry in the same document's `_llm_metadata.<producer>.axes.relationships.proposals` list.
- [ ] **Fingerprint-based revival.** When a producer emits a new proposal with the same `(source_id, target_id, relation)` but a *different* `basis_fingerprint` than a prior rejection, the new proposal surfaces normally with a `revived` marker (see §4.2 and §4.4 for render contract). Stale invalidation for fingerprint drift is implicit; no new mechanism required.
- [ ] **Suppression counter** per producer: when the filter excludes a proposal view during collection, a per-producer count is recorded in the return value so operator-facing callers can surface the suppression-count disclosure line. Counter scope: aggregated across all documents collected in the call (not per-document).
- [ ] **`collect_pending_relationship_proposals` return-type migration** — function return type changes from `list[RelationshipProposalView]` to `tuple[list[RelationshipProposalView], dict[str, int]]`. All **seven** call sites update in this slice:
  1. `folio/links.py:174` — `relationship_status_summary`
  2. `folio/links.py:269` — `_find_pending_view`
  3. `folio/links.py:369` — `confirm_doc`
  4. `folio/links.py:381` — `reject_doc`
  5. `folio/graph.py:84` — `graph_status` (`len(...)` → `len(views)`)
  6. `folio/graph.py:150` — second `graph_status` / diagnostic path
  7. `folio/cli.py:~1597` — `folio links review` subcommand dispatch (rendering the new suppression-count line — see §4.4)
- [ ] **Cumulative acceptance-rate diagnostic** on `folio graph doctor` (`folio/graph.py`): for each producer observed in the library, compute `accepted / (accepted + rejected)` across all managed documents. Sources: rejections in `_llm_metadata.<producer>.axes.relationships.proposals` (`status == "rejected"`); acceptances in `_llm_metadata.links.confirmed_relationships` (by `producer` field, populated by `confirm_proposal`). Suppression line is always rendered to make the feature discoverable.
  - `folio graph doctor`: add a new section `### Producer acceptance rates` that renders the per-producer table (defined in §4.3/§4.4). Section ordering: appears after existing severity-sorted findings, before the final summary line. When no producers are observed, renders `"No producer acceptance-rate data yet."` rather than empty column headers.

Existing `findings` schema unchanged (additive). Schema version tag bumped if needed per existing conventions.

## 4. Technical Design

### 4.1 Component overview

Three changes across two existing files; no new modules.

### 4.2 `folio/links.py` — rejection-memory filter

**File:** `folio/links.py` (MODIFY)

**Functions affected:**
- `collect_pending_relationship_proposals` (currently lines ~107–155): add filter step; change return type.
- Helper `_build_rejection_key(proposal_dict) -> tuple[str, str, str, str]` (CREATE, private). Returns `(source_id, target_id, relation, basis_fingerprint)`. Producer is implicit via the enclosing per-producer iteration context — NOT included in the key because the key is only ever compared within one producer's list.
- Helper `_is_rejection_suppressed(candidate_key, rejected_keys) -> bool` (CREATE, private). Pure function; returns `candidate_key in rejected_keys`.
- Return-type extension: `collect_pending_relationship_proposals` returns `tuple[list[RelationshipProposalView], dict[str, int]]`. The dict maps `producer → count` of proposals suppressed-by-rejection-memory in the call's scope. Both elements are always present; counter is empty dict `{}` when no suppressions occurred.

**Approach (pseudo-code aligned with current code's safety patterns):**

```text
all_views: list[RelationshipProposalView] = []
suppression_counts: dict[str, int] = {}

for managed_doc in scope:
    _, fm = _read_markdown(managed_doc.path)                # current helper
    if fm is None: continue
    metadata = fm.get("_llm_metadata", {})
    if not isinstance(metadata, dict): continue
    for producer, producer_axes in metadata.items():
        if producer == "links":                              # SF-1: reserved namespace
            continue
        if not isinstance(producer_axes, dict): continue
        axes = producer_axes.get("axes", {}) or {}
        proposals = (axes.get("relationships", {}) or {}).get("proposals", []) or []

        # Build rejected-key set; use _proposal_from_raw to normalize field access (SF-2)
        rejected_keys = {
            _build_rejection_key(_proposal_from_raw(managed_doc.id, producer, p))
            for p in proposals
            if isinstance(p, dict) and p.get("status") == "rejected"
        }

        for p in proposals:
            if not isinstance(p, dict): continue
            if p.get("status") != "pending_human_confirmation": continue
            candidate = _proposal_from_raw(managed_doc.id, producer, p)
            key = _build_rejection_key(candidate)
            if _is_rejection_suppressed(key, rejected_keys):
                suppression_counts[producer] = suppression_counts.get(producer, 0) + 1
                continue

            # Revival detection: a rejection exists for same (source, target, relation) but different fingerprint
            same_relation_rejections = {
                rk for rk in rejected_keys if rk[:3] == key[:3] and rk[3] != key[3]
            }
            view = RelationshipProposalView.from_proposal(candidate, doc=managed_doc, revived=bool(same_relation_rejections))
            all_views.append(view)

return all_views, suppression_counts
```

**Constraints:**
- Filter is **per-document-per-producer**. A rejection in doc A does NOT suppress a proposal with the same key in doc B (which may arise from doc B's own evidence). This matches the `_llm_metadata.<producer>` scope of rejection state — state lives next to the proposal.
- `basis_fingerprint` equality is string-equal. Normalization is producer-side responsibility per proposal rev 4 §7.
- `pending_human_confirmation` is the canonical pending state in current code; the proposal doc's `lifecycle_state: queued` maps to this. The spec does NOT rename the field in this slice (follow-up migration).
- **SF-1 skip:** `"links"` key under `_llm_metadata` is reserved for `confirmed_relationships` by `_ensure_confirmation_meta` (`folio/links.py:202`); the aggregator must skip it.
- **SF-2 safety:** field access routes through `_proposal_from_raw` so missing `source_id`/`target_id` keys are populated via its `setdefault` behavior instead of raising `KeyError`.
- **Revival marker:** the `RelationshipProposalView` gains an optional `revived: bool` attribute (default `False`). Rendering layer (§4.4) consumes this to add the `(revived — basis changed)` annotation.

### 4.3 `folio/graph.py` — cumulative acceptance-rate diagnostic

**File:** `folio/graph.py` (MODIFY)

**Function affected:**
- `doctor` (existing, generates the `folio graph doctor` report): add a new section `### Producer acceptance rates` with a table.

**New helper:**
- `_aggregate_producer_acceptance_rates(config: FolioConfig) -> list[ProducerAcceptanceRate]` (CREATE, public within module).

**Data sources (both required — PR-A-15 / gemini data-path correction):**
- **Rejections:** iterate `_llm_metadata.<producer>.axes.relationships.proposals` across all managed docs; count entries with `status == "rejected"`.
- **Acceptances:** iterate `_llm_metadata.links.confirmed_relationships` across all managed docs; count entries by `producer` field.

**Output shape:**

```python
@dataclass(frozen=True)
class ProducerAcceptanceRate:
    producer: str
    accepted: int
    rejected: int
    total_reviewed: int  # accepted + rejected
    rate: float | None   # None if warm-up
    warmup: bool         # True when total_reviewed < 10
```

**Module-level constants (CREATE in `folio/graph.py`):**

```python
_ACCEPTANCE_WARMUP_COUNT: int = 10       # proposal rev 4 §9.2
_ACCEPTANCE_GATE_RATE: float = 0.50      # proposal rev 4 §9.2
```

All three statuses and all test assertions reference these names (SF-4 / P-S2).

**Doctor-report rendering (post CPY-2 relabel):**

```text
### Producer acceptance rates

| Producer | Accepted | Rejected | Reviewed | Rate | Status |
|----------|----------|----------|----------|------|--------|
| relationship-suggester-a | 23 | 4 | 27 | 0.85 | ok |
| relationship-suggester-b | 2  | 8 | 10 | 0.20 | low-acceptance (< 50%) |
| relationship-new         | 3  | 1 | 4  | —    | warmup (< 10 reviewed) |

Status column is informational only in v0.6.0; `low-acceptance` producers continue to surface proposals at normal priority in this slice.
```

Row-ordering: `low-acceptance` rows first, then `ok`, then `warmup` — matches Johnny's "problems first" triage pattern (UX-3 closure). **Within each bucket, rows sort alphabetically by producer name** (deterministic; no flakiness under re-runs).

**Rate-boundary behavior:** `rate < 0.50` → `low-acceptance` status; `rate >= 0.50` → `ok` status. `rate == 0.50` is "ok" (strict-less-than threshold, matches the `(< 50%)` display copy). The CPY-2 relabel (`below-gate` → `low-acceptance (< 50%)`) discloses the threshold inline, parallel to the existing `warmup (< 10 reviewed)` idiom (CPY-1).

**Missing-producer handling (SF-8 / P-S3 / FV-1 closure):**

If an entry in `_llm_metadata.links.confirmed_relationships` lacks a `producer` field (legacy / corrupted data), the aggregator excludes it from per-producer rates and increments a module-level counter surfaced as an additional diagnostic row:

```text
Data-integrity: N accepted entries excluded (missing `producer` field).
```

This row renders below the main table when the count > 0; elided entirely otherwise. The counter is also exposed in `--json` output under `producer_acceptance_rates_data_integrity.missing_producer_count`.

**Constraints:**
- Warm-up threshold = `_ACCEPTANCE_WARMUP_COUNT` (10); rate threshold = `_ACCEPTANCE_GATE_RATE` (0.50).
- `low-acceptance` is a status label only; this slice does NOT change surfacing behavior based on it (see §2 out-of-scope).
- `suppressed` and `stale` lifecycle states are excluded from denominator (proposal rev 4 §9.2). Current code has only `pending_human_confirmation` and `rejected` (plus accepted-via-relocation) — the filter is no-op today but coded defensively for future-proofing.
- Aggregator's iteration over `_llm_metadata.items()` MUST skip the reserved `"links"` key (SF-1).

### 4.4 `folio/cli.py` — rendering updates

**File:** `folio/cli.py` (MODIFY; scope-lock already permits)

**Why v1.0 said "two callers":** it was wrong. v1.0's claim was based on unverified author assumption, not grep. v1.1 lists the verified seven.

### `folio graph doctor --json` — breaking change (declared in v1.2)

The `--json` output shape change in §4.4 is **not additive**, contrary to v1.1's framing. Current code at `folio/cli.py:1758-1759` emits `json.dumps(findings)` — a top-level JSON **array**. v1.2 changes this to a top-level **object** wrapping `findings`, `producer_acceptance_rates`, and `producer_acceptance_rates_data_integrity` keys.

**Consumers affected:** any tool that parses `folio graph doctor --json` output as a top-level JSON array. (Internal folio callers: none identified; the `--json` flag is a v0.5.x addition with no known scripted consumers.)

**Migration path chosen:** top-level-object replacement. Rationale: folio is pre-1.0; breaking CLI JSON shape at a minor version is acceptable when (a) no known scripted consumers exist, (b) the new shape cleanly extends rather than renames, and (c) the change is documented in a visible CHANGELOG entry at ship time.

**Alternative considered and rejected:** gating the new shape behind a `--json-v2` flag while keeping the top-level array as default. Rejected because the new shape is materially more useful (named keys for three distinct data classes) and the deprecation cost of carrying two JSON shapes through the pre-1.0 version arc outweighs the migration cost for a near-zero external-consumer surface.

**Phase C requirement:** CHANGELOG entry for v0.6.0 MUST name this as a breaking change to `folio graph doctor --json` output shape.

### Backwards-compatibility for stored frontmatter

- Rejection entries from prior versions (pre-v0.6.0) have the expected shape (`status: rejected`, `basis_fingerprint: <string>`). No frontmatter migration required.
- New fields: none. This slice reads existing frontmatter; it does not write new fields.

### Deprecation paths

- None. The spec is additive in behavior.

### Rollback plan

- Single-commit revert of the two file changes in `folio/links.py` and `folio/graph.py` restores v0.5.2 behavior. No data migration is needed. The only observable change is that operators would once again see previously-rejected proposals re-appear.

## 8. Risk Assessment

**Overall risk:** `medium`.

### Risks and mitigations

| Risk | Severity | Likelihood | Mitigation |
|------|----------|-----------|-----------|
| Filter false-negative (legitimate proposal suppressed) | High (breaks intended functionality) | Low | Comprehensive per-doc, per-producer, per-fingerprint tests. Revival test explicit. |
| Filter false-positive (rejected proposal leaks back to operator) | High (defeats feature) | Low-Medium | Test uses exact fingerprint match; covers equal-fingerprint-suppressed and different-fingerprint-revived. |
| Aggregator reads only one data source → undercounts | Medium (doctor output misleading) | Medium (PR-A-15 identified this class of bug on the spec itself) | Explicit test `test_doctor_aggregator_reads_both_sources` asserts both paths contribute. |
| `basis_fingerprint` collision between semantically-different proposals | Medium | Very Low (basis_fingerprint is producer-hashed with multiple input fields) | Out-of-scope to redesign fingerprint; current collision rate is empirically zero per existing v0.5.2 tests. Flag in Open Questions if Phase B reviewers want reinforcement. |
| Return-type change breaks an unexpected caller | **High** | Low (now; mitigated) | §7 enumerates all seven call sites verified via `rg`; §4.5 tests 7–10 + §4.6 tests 10–11 regression-cover each migration. Severity raised from Low to High per B.1 finding X-1 / P-B2 — the *consequence* is high (silent breakage of 5 CLI subcommands); the *likelihood* is now Low because the inventory is verified and under test. |
| Warm-up padding attack (PR-A-16) | Low | Very Low in closed-set producer model | Documented in test-file docstring; not architecturally addressed in this slice. |

### Adversarial attack notes

An adversarial reviewer attempting to break this spec should focus on:

1. Does the filter handle lists that are modified during iteration? (Python iteration guards: no; the code builds the `rejected_fingerprints` set before the emission loop.)
2. Does `_remove_or_update_proposal` in the confirm/reject path interact with the filter? (It runs synchronously, writing frontmatter; subsequent `collect_*` reads fresh frontmatter.)
3. Does a producer ever emit two proposals with the same `basis_fingerprint` but different `status` transitions? (Yes — the same suggestion re-emitted in a later enrich run; this is the *target* of the filter.)
4. What if `_llm_metadata.links.confirmed_relationships` is missing the `producer` field on older accepted entries? (Regression guard: the aggregator must tolerate missing `producer` and exclude those entries from per-producer rates with a warning.)
5. Circular canonical-state interactions: does adding a canonical relation via `_add_canonical_relationship` ever conflict with rejection memory? (No: canonical relations live at `fm["relationships"]`; rejection memory lives at `_llm_metadata.<producer>.axes.relationships.proposals`. Different keys; no interaction.)

### Monitoring

Post-ship observability: `folio graph doctor` output provides the acceptance-rate diagnostic directly. Operators can run it any time to check producer health without running tests.
