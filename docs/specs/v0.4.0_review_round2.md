## New Issues in Rev 2

### N1. Prompt migration strategy missing (Should-Fix)

The current `ANALYSIS_PROMPT` (`analysis.py:23-59`) instructs the LLM to return prose-formatted output:
```
Format your response exactly as:
Slide Type: [type]
Framework: [framework]
...
Evidence:
- Claim: [what this evidence supports]
  Quote: "[exact text from slide]"
```

Section 2.4 commits to JSON structured output as the sole response contract. This means the prompt itself must be rewritten to request JSON. The spec defines the target JSON schemas (5.5) and normalization rules (5.6) but doesn't address:

1. **How the prompt changes.** The current prose-format prompt needs to be replaced with a JSON-requesting prompt. This is a non-trivial prompt engineering task that affects analysis quality.
2. **Cache invalidation is automatic** — the `_prompt_version` check (analysis.py:950) hashes the prompt text, so changing the prompt already triggers full re-analysis. Good, but worth calling out explicitly so implementers know existing caches will be wiped.
3. **Quality regression risk.** Switching from prose to JSON output may change analysis quality. The spec should acknowledge this and recommend a quality validation step (e.g., run the same 10-20 slides through old prose prompt vs. new JSON prompt, compare results).

**Recommendation:** Add a subsection under 5.5 or 5.8 noting that:
- The existing `ANALYSIS_PROMPT` must be rewritten to request JSON
- `_prompt_version` hashing ensures automatic cache invalidation on prompt change
- A quality comparison (old prompt vs. new JSON prompt) should be run during v0.4.0-a before shipping the prompt change

### N2. `ProviderOutput.truncated` usage unspecified (Minor)

Section 5.2 defines `ProviderOutput` with a `truncated: bool` field but the spec never says how to detect or use it. Questions:

- How does each adapter detect truncation? (Anthropic: `stop_reason == "max_tokens"`, OpenAI: `finish_reason == "length"`, Gemini: `finish_reason == "MAX_TOKENS"`)
- What happens when `truncated=True`? Section 5.6 says unparseable JSON becomes pending, but truncated JSON would fail parse anyway, making the field redundant unless the orchestrator wants to log a more specific reason.

**Recommendation:** Either document the truncation detection per provider and the specific orchestrator behavior, or drop the field and let JSON parse failure handle it (simpler).

### N3. `create_client` lifetime unclear (Minor)

The protocol (Section 5.3) has `create_client(api_key: str) -> object`. Questions:

- Is a new client created per slide, per conversion, or once at startup?
- All three SDKs support client reuse. Creating per-slide would be wasteful.

**Recommendation:** Add a note: client is created once per conversion command invocation and reused across all slides. The orchestrator calls `create_client()` during route resolution, not per-slide.

### N4. `ErrorDisposition` doesn't carry the original exception (Minor)

`classify_error()` returns `ErrorDisposition(kind, message)` but discards the original exception. The orchestrator may need it for logging (stack trace) or for re-raising on permanent failure.

**Recommendation:** Either add an `original: Exception` field to `ErrorDisposition`, or document that the orchestrator should catch and log the exception before calling `classify_error()`.

### N5. `--llm-profile` disables fallbacks — is this intentional? (Clarification)

Section 3.5 says `--llm-profile` "disables configured fallbacks for that invocation." This means a CLI override is strictly single-provider with no safety net. If the user picks a flaky provider, they get pending analysis with no fallback.

This may be the right choice (deterministic > resilient for debugging), but it's worth calling out explicitly as intentional rather than leaving it ambiguous.

### N6. Routing uses `convert` but codebase uses `convert_pass1` / `convert_pass2` (Minor)

Section 3.1 shows a single `convert` route. The round 1 spec and the codebase structure distinguish `convert_pass1` and `convert_pass2` as separate stages. Section 2.3 locks them to the same profile, which makes a single `convert` route correct — but the spec should explicitly note that the prior `convert_pass1` / `convert_pass2` route names from rev 1 are collapsed into `convert` because same-profile policy makes separate routes pointless for v0.4.0.

### N7. Evidence count cap of 10 — should rejected extras be logged? (Minor)

Section 5.6 caps evidence at 10 items per slide. What happens to items beyond 10? Silent discard could mask providers that produce high-quality evidence at positions 11+.

**Recommendation:** Log a warning when capping occurs, including total count before cap. Keep first 10 (highest-positioned, which is what JSON array ordering gives you).

### N8. `_profile_version` in cache may be over-scoping (Minor)

Section 7.1 adds both `_provider_version` and `_profile_version` to cache metadata. But a profile is just a name — two profiles with the same provider and model should produce identical results. Invalidating cache on profile rename forces unnecessary re-analysis.

**Recommendation:** Consider whether `_provider_version` + `_model_version` is sufficient for correctness. If a user renames `high_quality_anthropic` to `primary_claude` but keeps the same provider/model, should the cache invalidate? Probably not. If you keep `_profile_version`, document the rationale (e.g., different profiles might have different system prompts in the future).

## Assessment

The spec is now implementable. The Technical Design section (5) gives a developer enough to code against: protocol, types, registry pattern, JSON schemas, normalization rules, and a clear refactor boundary. The test strategy and cache design are concrete. The incremental delivery slices are sensible.

The new issues above are all minor or should-fix level — none are blocking. The biggest risk is N1 (prompt rewrite quality regression), which is a practical implementation concern rather than a spec gap.

**Verdict: Approve.** Address N1 (prompt migration) and N8 (profile cache scoping) before implementation starts. The rest can be resolved during implementation.
