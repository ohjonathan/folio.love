## R1: Adversarial Reviewer

**Mandate:** Find the failure modes, race conditions, security gaps, and unstated assumptions that will break production.

### A1. JSON structured output is not universally reliable (Should-Fix)

The spec (§2.4) commits to JSON structured output as the "sole supported response contract." This is the correct portability choice, but the spec understates the reliability gap:

- **Anthropic** has no native JSON mode for `messages.create()`. You must prompt for JSON and hope. Claude is *usually* compliant, but there is no schema enforcement mechanism comparable to OpenAI's `response_format: { type: "json_object" }`.
- **OpenAI** has `response_format: { type: "json_object" }` which is reliable, but GPT models occasionally prepend a markdown code fence (` ```json `) before the JSON object, which breaks `json.loads()`.
- **Gemini** has structured output via `response_mime_type: "application/json"` and optional `response_schema`, which is the most reliable option.

The spec's §5.5 Rule 2 says "if provider-native JSON mode is unavailable or unreliable, adapters request a single raw JSON object and the orchestrator extracts that object before parsing." This is vague — *who* strips the code fence? *What* signal tells the orchestrator to try extraction vs. discard?

**Recommendation:** Add a concrete `_extract_json(raw_text: str) -> str` utility to the orchestrator layer, documented in §5.8. It should:
1. Attempt `json.loads(raw_text)` directly
2. If that fails, strip markdown code fences and retry
3. If that fails, mark as pending

This is a one-function addition but it prevents the "works with Anthropic, crashes with GPT" failure mode.

### A2. `create_client()` returns `object` — type safety gap (Minor)

§5.3 defines `create_client(self, api_key: str) -> object`. Returning `object` provides zero type safety. The `analyze()` method then accepts `client: object`. If someone passes the wrong provider's client, it will fail with an opaque SDK error deep inside the adapter, not at the interface boundary.

**Recommendation:** Either use `Self`-typed generics (`ClientT`) bounded per adapter, or accept that this is a pragmatic compromise and add a runtime type check in `analyze()`. Document the choice.

### A3. Fallback switches mid-batch with no per-slide provenance (Should-Fix)

§6.2 says fallback activates after one retry on the active profile. §5.9 says the client is "reused for the remaining slides." This means a 50-slide deck could have slides 1-22 analyzed by Anthropic and slides 23-50 by Gemini, with no per-slide indicator of which provider produced each result.

The `_llm_metadata` block (§7.4) records `fallback_used: true/false` at the document level but not which slides were affected. For evidence validation, this matters — different providers have different hallucination profiles and the 80% word overlap threshold ([analysis.py:477](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L477)) was tuned for Claude.

**Recommendation:** Add a per-slide `_provider` field in the cache entry (not in frontmatter, just in `.analysis_cache.json`). This lets debugging and future quality analysis trace which provider produced each result.

### A4. The `api_key_env` indirection doesn't validate the key format (Minor)

§3.3 says API keys come from env vars. The spec validates that the env var *name* exists in config, but never validates that the env var is *set* before starting a batch. The current code ([analysis.py:230](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L230)) checks `ANTHROPIC_API_KEY` at the top of `analyze_slides()`. With profiles, this check must move to route resolution time and must be provider-aware.

§6.4 does specify the pending message for a missing key. Good. But the spec doesn't say *when* this check happens. If it happens per-slide, the user gets 50 identical warnings. If it happens once at route resolution, they get one clean error.

**Recommendation:** State explicitly: key presence is validated once during route resolution (before the first slide), not per-slide. A missing key on the primary profile triggers the pending path immediately, with no fallback attempt (consistent with §2.5: "not used for auth failures").

### A5. `_profile_version` was removed by Round 2 N8 — but the spec still has it (Clarification)

Round 2 (N8) recommended considering whether `_profile_version` over-scopes cache invalidation. The spec at §7.1 says `_profile_version is intentionally omitted from cache invalidation scope` (good), but §7.1 still shows only `_provider_version` and `_model_version` in the example JSON, without `_profile_version`. This is correct — but the spec should explicitly state that `_profile_version` is omitted *from the cache file entirely*, not just from the invalidation check, to avoid confusing implementers who read Round 2's N8 discussion.

## R2: Alignment Reviewer

**Mandate:** Ensure the spec serves the product vision, preserves architectural invariants, and doesn't create technical debt that contradicts the Ontology or PRD.

### L1. Prompt rewrite is the highest-risk item and gets a subsection, not a plan (Should-Fix)

§5.5.1 acknowledges the prompt migration from prose to JSON. It requires "a quality comparison on a representative 10-20 slide sample." This is necessary but insufficient:

- **The current prose prompt** ([analysis.py:23-59](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L23-L59)) has been tested and refined across the entire development history. It produces evidence in a specific format that the regex parser depends on.
- **A JSON prompt** will produce structurally different outputs. Framework detection, slide type, and especially evidence *quality* — not just format — may change.
- **The DEPTH_PROMPT** ([analysis.py:493-521](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L493-L521)) also requests prose output and must be rewritten for JSON.

The spec treats this as a sub-bullet. In practice, this is the single item most likely to cause a quality regression visible to users. A bad prompt rewrite degrades every future conversion for every user.

**Recommendation:** Elevate §5.5.1 to a standalone gated milestone in the delivery slices (§10). v0.4.0-a should include the prompt rewrite + quality validation *before* any new provider work begins. If the JSON prompt quality is worse, you need a decision point, not a surprise.

### L2. The PRD deltas are well-scoped but FR-606 conflates two concerns (Minor)

FR-606 says: "Folio SHALL record internal execution metadata for LLM analysis AND SHALL degrade to pending analysis without failing conversion when analysis cannot run."

These are separate requirements:
1. Recording metadata (observability)
2. Graceful degradation (resilience)

Combining them makes acceptance testing ambiguous — you could satisfy "metadata is recorded" but fail "degradation works" and a tester might miss it because they're checking one FR.

**Recommendation:** Split into FR-606 (metadata recording) and FR-607 (graceful degradation). This aligns with how the spec already separates these concerns: §7.4 (metadata) vs. §6.2/§6.4 (degradation).

### L3. Reserved routes in config may mislead users (Minor)

§3.1 shows `ingest`, `enrich`, and `query_answer` routes in the config example. §2.2 says "v0.4.0 only executes `convert`." But if these routes are in the *default config that ships with v0.4.0*, users will expect them to work.

**Recommendation:** Either:
- Ship v0.4.0 config with only `default` and `convert` routes, documenting reserved future routes in comments
- Or show them in the config with inline `# Reserved for future use` comments

This is a documentation/packaging concern, not a code concern, but it affects user expectations.

### L4. Ontology boundary is well-maintained (Positive)

§7.4's `_llm_metadata` approach is architecturally sound. The underscore prefix convention is consistent with the "internal, non-schema" pattern. The ontology v2 ([Folio_Ontology_Architecture_v2.md:733-797](file:///Users/jonathanoh/Dev/folio.love/docs/architecture/Folio_Ontology_Architecture_v2.md#L733-L797)) is untouched. The decision to avoid extending §12.4 (Source Fields) for v0.4.0 is correct — LLM metadata is runtime provenance, not ontological structure.

### L5. Version semantics decision is correct but should reference the invariant (Minor)

§2.6 says provider changes don't bump version. §6.5 elaborates. This is the right call — version tracking is content-based per [FR-301](file:///Users/jonathanoh/Dev/folio.love/docs/product/02_Product_Requirements_Document.md#L161-L175). But the spec should cite FR-301 explicitly to create a traceability link. Future maintainers who see `_llm_metadata` changing without a version bump will wonder if it's a bug unless the spec grounds the decision.

## R3: Code Quality Reviewer

**Mandate:** Assess implementability, refactoring complexity, test coverage, and whether the spec will produce clean, maintainable code.

### C1. The refactor boundary (§5.9) is well-defined but the seam is large (Should-Fix)

§5.9 lists what moves behind the provider interface vs. what stays in `analysis.py`. Looking at the actual code:

| Responsibility | Current location | Lines affected |
|---|---|---|
| Client creation | [analysis.py:242-243](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L242-L243), [652-653](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L652-L653) | 4 lines, but 2 separate functions |
| Image payload | [analysis.py:299-331](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L299-L331) | ~33 lines in `_analyze_single_slide()` |
| Response extraction | [analysis.py:336](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L336) | 1 line, but deeply embedded after retry logic |
| Error handling | [analysis.py:351-357](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L351-L357) | Generic `Exception` catch |
| Prose parsing | [analysis.py:360-451](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L360-L451) | ~91 lines of regex parsing |

The refactor touches **both** `analyze_slides()` and `analyze_slides_deep()`, which duplicate the Anthropic client creation and SDK-specific code. The `_run_depth_pass()` function ([analysis.py:793-862](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L793-L862)) also has the same pattern.

**Recommendation:** The spec should note that `_analyze_single_slide()` and `_run_depth_pass()` share the same provider interface, and that the adapter's `analyze()` method must support both pass-1 and pass-2 invocations (same method, different prompts). This prevents implementers from creating separate adapter methods for each pass.

### C2. Prose parsing code (91 lines) becomes dead code with JSON output (Should-Fix)

The switch to JSON output (§2.4) means:
- `_parse_analysis()` (L360-384) — dead
- `_parse_evidence()` (L387-419) — dead
- `_parse_single_evidence()` (L422-451) — dead
- `_is_valid_pass1_response()` (L191-201) — dead
- `_is_valid_pass2_response()` (L204-208) — dead
- `_parse_depth_reassessment()` (L770-790) — dead

That's **~130 lines** of regex parsing code that becomes obsolete. The spec doesn't say whether to remove it, keep it behind a feature flag, or deprecate it.

**Recommendation:** The delivery slice v0.4.0-a (§10) should explicitly state: "Remove prose parsing code; replace with JSON parsing. The old parsing functions are deleted, not deprecated." This is a clean break because `_prompt_version` hashing already ensures caches are invalidated when the prompt changes.

### C3. `ProviderInput.prompt` assumes a single string — but the depth prompt uses `Template.safe_substitute()` (Minor)

§5.2 defines `ProviderInput.prompt: str`. The orchestrator is responsible for prompt construction (§5.3), so by the time `ProviderInput` is created, the prompt is already a rendered string. This is correct. But it means the orchestrator must call `Template.safe_substitute()` *before* creating `ProviderInput`, and the adapter never sees the template.

This is fine for v0.4.0, but note: if a future version wants provider-specific prompt variants (different system prompts per provider), this design puts the variant logic in the orchestrator, not the adapter. That's actually the right place — but the spec should note this design intent for traceability.

### C4. Parametrized test strategy is right but needs a mock response contract (Minor)

§9.2 says core tests run against all providers via `pytest.mark.parametrize`. For this to work, each mock response factory must produce output in the *provider's native format* (not the normalized JSON), so the adapter's normalization code is exercised.

Example: the Anthropic mock must return `MockResponse(content=[MockContent(text='{"slide_type": "data", ...}')])`, while the OpenAI mock must return `MockChoice(message=MockMessage(content='{"slide_type": "data", ...}'))`.

**Recommendation:** Add a note in §9.1 that mock factories must simulate provider-native response shapes, not pre-normalized JSON. Otherwise tests will pass with mocks but fail with real providers.

### C5. The `ErrorDisposition.original` field was added per Round 2 N4 — good (Positive)

§5.2 now includes `original: Exception | None = None` in `ErrorDisposition`. This addresses Round 2's N4. The orchestrator can log the full traceback before classifying the error.

### C6. Cache version should increment from 1 to 2 (Minor)

Current cache version is `_ANALYSIS_CACHE_VERSION = 1` ([analysis.py:21](file:///Users/jonathanoh/Dev/folio.love/folio/pipeline/analysis.py#L21)). The spec adds `_provider_version` to cache metadata (§7.1) and changes the cache format. This should trigger `_ANALYSIS_CACHE_VERSION = 2`, which the spec notes as `"_cache_version": 2` in the example. Good — already covered.

## Overall Assessment

Rev 2 is a substantial improvement over rev 1. The Technical Design section gives a developer a clear protocol, type system, and refactor boundary. The Round 1 blockers are genuinely resolved. The remaining issues are implementation-readiness refinements, not architectural gaps.

The biggest risk is **L1 (prompt rewrite)** — a JSON prompt that degrades analysis quality would affect every user. This should be the **first** thing validated, not a footnote in §5.5.1.

Second biggest risk is **A1 (JSON reliability)** — the assumption that all providers return clean JSON is optimistic. A defensive extraction utility is cheap insurance.

**Verdict: Approve with conditions.** Address A1, A3, L1, C1, and C2 before implementation begins. The rest can be resolved during implementation.
