### 2.5 Non-goals (this slice)

Per parent Â§15.6, Â§12, and Phase-A narrowing decision:

- **LLM-backed semantic search.** v0.9.0 is structural-only. No `folio.llm` imports, no `openai` / `anthropic` calls. Embedding-based search deferred to v0.9.1+.
- **Regex QUERY syntax.** v0.9.0 uses `str` substring matching only. Regex (`--regex` flag or field-scoped syntax) deferred to v0.9.1. Rationale: substring is deterministic, trivially testable, and closes ReDoS surface at the spec level.
- **Ranking / relevance scoring.** v0.9.0 preserves the shared collector's iteration order (stable; deterministic per `collect_pending_relationship_proposals` contract). Score-based ranking deferred.
- **Field-scoped query syntax** (`producer:enrich draws_from`). Over-scoped for v0.9.0; unbounded grammar. Deferred to v0.9.1+ if operator demand surfaces.
- **Multi-lifecycle scope.** Search only finds `lifecycle_state == "queued"` proposals because the shared collector hard-filters there. - **`--relation` / `--lifecycle` filter flags.** Redundant (`--relation`) or footgun (`--lifecycle`) at v0.9.0 per operator-confirmed scope shape. Additive in v0.9.1.
- **New doc artifacts.** Search is read-only; no `analysis` doc creation, no registry rows written, no markdown files created.
- **Cross-engagement scope.** `<scope>` resolves to a single engagement subtree or document ID (or omitted = library-wide).
- **Per-finding match annotation** (e.g., `[match: reason_summary]` on each stdout line). Adds a field not present in the shared envelope and invites "why isn't that in `--json`?" review churn. Deferred to v0.9.1.

### 3.6 QUERY matching semantics (CB-2 closure)

**Normalization helper.** All QUERY and field-value comparisons go through a single module-level helper:

```python
import unicodedata
from typing import Any, Optional

def _normalize_search_text(value: Any) -> Optional[str]:
    """Normalize a value for search comparison.

    Returns ``None`` for non-string, ``None``, empty, or whitespace-only
    values (those cannot match any QUERY). Otherwise applies NFC
    canonical composition (so canonically equivalent sequences like
    "Ă©" / "e\\u0301" compare equal) and then casefold (so Unicode case
    folding behaves correctly for eszett, Turkish dotless-I, etc.).
    """
    if not isinstance(value, str):
        return None
    if not value or not value.strip():
        return None
    return unicodedata.normalize("NFC", value).casefold()
```

`search()` rejects empty / whitespace-only QUERY at function entry by raising `ValueError`; the CLI layer rejects the same at flag-parse time via a Click `callback=` validator that raises `click.UsageError` (exit 2). This is CB-2 closure: the contract is **total** (no crash on malformed frontmatter) and **Unicode-safe** (NFC+casefold is defined).

**Matching rule.** A view matches QUERY if `needle := _normalize_search_text(query)` is non-`None` AND `needle` is a substring of the normalized form of **any** of the following six field values:

| Field | Source | Match expression | Notes |
|---|---|---|---|
| `source_id` | `view.source_id` | `needle in (_normalize_search_text(view.source_id) or "")` | Always set in collector output; fallback `""` is defensive against unexpected `None`. |
| `target_id` | `view.proposal.target_id` | `needle in (_normalize_search_text(view.proposal.target_id) or "")` | May be `None`; helper returns `None` â†’ `""` substring-never-matches. |
| `relation` | `view.proposal.relation` | `needle in (_normalize_search_text(view.proposal.relation) or "")` | Always in `SUPPORTED_RELATIONS` in practice; helper is defensive. |
| `producer` | `view.producer` | `needle in (_normalize_search_text(view.producer) or "")` | Always set. |
| `reason_summary` | `view.proposal.rationale` | `needle in (_normalize_search_text(view.proposal.rationale) or "")` | Always set when frontmatter is well-formed; helper handles malformed gracefully. |
| `evidence_bundle` | `view.proposal.signals` | `any(needle in (_normalize_search_text(s) or "") for s in (view.proposal.signals or []))` | **Any-element rule.** Empty list never matches. Non-string elements skipped via `_normalize_search_text` returning `None`. |

**Non-searchable fields (explicit exclusion):** `proposal_id`, `subject_id`, `input_fingerprint` / `basis_fingerprint`, `schema_gate_result`, `lifecycle_state`, `flagged_inputs`, any field of `SuppressionCounts`. Rationale: these are identifier / metadata fields, not operator-facing content; searching them couples the search UX to internal representation details.

**Empty evidence_bundle.** `view.proposal.signals == []` is a surface-time possibility â€” the collector (`folio/links.py`) does not mandate non-empty signals, so malformed / incomplete frontmatter can produce an empty list. The any-element rule correctly evaluates to `False` in that case. Test SRC-11 exercises this explicitly.

**Short-circuit.** Iteration over the six fields uses Python `or` short-circuiting (first match wins; rest not evaluated). Correctness-irrelevant but minor-performance-win; `SearchReport.findings` iteration order mirrors the collector's order (insertion order across managed documents).

**Unicode discipline.** NFC normalization + casefold handles:
- Canonically equivalent sequences: `"Ă©"` (U+00E9) and `"e\u0301"` (U+0065 + U+0301) normalize to the same NFC form â†’ match.
- Eszett / Turkish dotless-I: casefold correctly handles `Ăź` â†’ `ss`, `Ä°` â†’ `iĚ‡` per Unicode rules.
- Zero-width characters: NOT stripped. A QUERY containing a zero-width joiner matches field values containing the same sequence; zero-width chars alone do not qualify as whitespace (`str.strip()` ignores them). This is intentional â€” operators copying ZWJ from rich-text paste get literal matching.
- Lone surrogates / malformed Unicode: Python strings cannot contain lone surrogates outside test fixtures; if encountered (e.g., via a test exploit), `unicodedata.normalize` raises â€” Phase C must either catch and treat as non-matching, or propagate. Test SRC-40 documents the chosen behavior.

Tests SRC-12 (eszett), SRC-36 (NFC/NFD equivalence), SRC-37 (zero-width), SRC-40 (surrogate) verify.

**Operator-facing over-match guidance (PROD-SF-001 closure).** Because `relation` is one of the six searchable fields, a QUERY like `folio search draws_from` will match every `draws_from`-typed proposal in scope â€” that's ~1/6 of the pending queue for a typical library. Operators who want to narrow relation-typed search should use `--scope ENGAGEMENT` to bound, or wait for v0.9.1's `--relation` filter. `--help` tip mirrors this. In practice most operator QUERY strings are domain-specific terms (e.g., `"framework"`, `"pilot"`, `"stakeholder"`) where over-match is natural and useful, not a footgun.

**Empty / whitespace-only QUERY (CB-2 closure).** `search(query="")` raises `ValueError("QUERY must contain at least one non-whitespace character.")`. `search(query="   ")` same. CLI: Click `callback=_validate_query` validator at `folio/cli.py` decorator raises `click.UsageError` â†’ exit 2. Tests SRC-38, SRC-39 (function-level), SRC-CLI-25, SRC-CLI-26 (CLI-level).
