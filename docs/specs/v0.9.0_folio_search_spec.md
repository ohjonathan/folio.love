# v0.9.0 — `folio search` (Shared-Consumer Expansion Sub-Slice 3 of 3)

## 1. Overview

**Delivers:** the third and final sub-slice of Shipping Plan §15.6 (shared-consumer expansion) from `docs/specs/tier4_discovery_proposal_layer_spec.md` rev 5. Introduces a new CLI command, `folio search <QUERY>`, that consumes the §5 Proposal Object Contract as a **QUERY lens over the pending proposal queue** — a read-only, substring-based search surface that honors §11 trust-gating and emits the shared payload-level `--json` envelope at the first minor-version increment (`schema_version: "1.1"`) via the addition of a `query` top-level key.

**One-sentence summary:** `folio search` narrows the pending relationship-proposal queue by substring-matching a required QUERY string across six proposal text fields, uses the shared `folio.tracking.trust.derive_trust_status` helper (proving the shared-consumer pattern holds at N=3 surfaces), honors `--include-flagged` trust-override, and emits a shared envelope whose `schema_version` steps from "1.0" to "1.1" to accommodate the new `query` key.

**Target version:** v0.9.0 (new minor; new command).

**Theme:** Tier 4 proposal-layer governance — shared-contract consumption on a third greenfield CLI surface; first production exercise of the shared-envelope minor-version bump policy.

**Risk level:** `low` (feature) + `low` (migration). Structural-only MVP; no LLM, no registry mutations, no writes; iteration and query-matching are deterministic. Primary risks: (1) envelope-versioning-policy reinterpretation at B.1 — mitigated by adopting the stricter reading of synthesize §3.3 ("Producers MUST bump minor") and shipping "1.1" from v1.0 of this spec; (2) render_envelope duplication across graph/synthesize/search — mitigated by explicit §3.3 "acceptable duplication, lift-later" disposition; (3) QUERY-matching rules under-specified — mitigated by exhaustive §3.6 field-list + casefold rule + evidence_bundle any-element rule.

### 1.1 Core behaviors

1. `folio search <QUERY>` emits a deterministic search report listing §5 relationship proposals whose text fields substring-match QUERY (case-insensitive via `str.casefold()`). Default output is a human-readable stdout summary.
2. `folio search <QUERY> --json` emits the **shared payload-level envelope** (§3.3) with `schema_version: "1.1"`, `command: "search"`, and a new `query` top-level key carrying the verbatim QUERY input.
3. `<QUERY>` is a **required** positional argument. `folio search` (no arg) exits with Click usage error (exit 2).
4. `--scope SCOPE` resolves to an engagement subtree, document ID, or library-wide (omitted or `-`); same scope-resolution contract as synthesize §3.4. In JSON, library-wide is `"scope": null`.
5. `--producer P` is the only structured filter in v0.9.0; it conjunctively narrows findings to those whose `view.producer == P` (exact-string match, NOT casefold). `--producer enrich` ≠ `--producer ENRICH`.
6. Trust gating: default excludes proposals whose source OR target input is flagged (§11). `--include-flagged` widens the predicate and the envelope's `trust_override_active` field is set to `true`. `excluded_flagged_count` is surfaced in both stdout and JSON per parent §11 rule 4 and is routed from `SuppressionCounts.flagged_input` (CB-1 carry-forward from sub-slice 2).
7. Exit codes: `0` on success (findings ≥ 0); `1` on scope-resolution failure or registry-missing; `2` on Click usage error (missing QUERY); other non-zero on internal error. Command never shells out; no LLM calls in v0.9.0.
8. Search is **read-only** — no registry mutations, no doc creation, no side effects.
9. Zero-results diagnostics: stdout always prints the `Excluded (flagged inputs): N` line (even when N=0). When findings==0 AND excluded==0, a next-action line is printed: `No matches for '<QUERY>'. Try a broader scope, check spelling, or run \`folio ingest\` / \`folio enrich\` if no proposals have been emitted yet.` (search-specific adaptation of sub-slice 2 CB-5 discipline.)
10. `--limit N` caps findings (parity with synthesize SF-C8). Default unbounded.

### 2.5 Non-goals (this slice)

Per parent §15.6, §12, and Phase-A narrowing decision:

- **LLM-backed semantic search.** v0.9.0 is structural-only. No `folio.llm` imports, no `openai` / `anthropic` calls. Embedding-based search deferred to v0.9.1+.
- **Regex QUERY syntax.** v0.9.0 uses `str` substring matching only. Regex (`--regex` flag or field-scoped syntax) deferred to v0.9.1. Rationale: substring is deterministic, trivially testable, and closes ReDoS surface at the spec level.
- **Ranking / relevance scoring.** v0.9.0 preserves the shared collector's iteration order (stable; deterministic per `collect_pending_relationship_proposals` contract). Score-based ranking deferred.
- **Field-scoped query syntax** (`producer:enrich draws_from`). Over-scoped for v0.9.0; unbounded grammar. Deferred to v0.9.1+ if operator demand surfaces.
- **Multi-lifecycle scope.** Search only finds `lifecycle_state == "queued"` proposals because the shared collector hard-filters there. Searching rejected/accepted/suppressed proposals would require a collector-layer extension crossing forbidden `folio/links.py`. Reserved for a dedicated collector-expansion slice (v0.9.1 or later).
- **`--relation` / `--lifecycle` filter flags.** Redundant (`--relation`) or footgun (`--lifecycle`) at v0.9.0 per operator-confirmed scope shape. Additive in v0.9.1.
- **New doc artifacts.** Search is read-only; no `analysis` doc creation, no registry rows written, no markdown files created.
- **Cross-engagement scope.** `<scope>` resolves to a single engagement subtree or document ID (or omitted = library-wide).
- **Per-finding match annotation** (e.g., `[match: reason_summary]` on each stdout line). Adds a field not present in the shared envelope and invites "why isn't that in `--json`?" review churn. Deferred to v0.9.1.
- **Graph envelope migration to schema_version "1.1".** v0.9.0 does NOT modify `folio/cli.py` graph doctor's `--json` payload. After v0.9.0 ships, graph's envelope lacks both `schema_version` and `command` (sub-slice 1 shipped before the shared envelope was designed) — graph migration is still a follow-up slice. v0.9.0 bumps to "1.1" only for search's own envelope; synthesize remains at "1.0" until it adds a top-level key of its own.
- **`render_envelope` hoist to a shared module.** v0.9.0's `folio.search.render_envelope` is byte-for-byte parallel to `folio.synthesize.render_envelope` (only the constants differ). Hoisting to `folio/_shared_envelope.py` (or similar) is deferred to a follow-up slice that can touch graph + synthesize + search together. See §3.3 duplication disposition.

    ...

    """Build the shared payload-level JSON envelope for --json output.

    Top-level keys at schema_version "1.1":
      schema_version, command, scope, query, trust_override_active,

        "schema_version": SCHEMA_VERSION,
        "command": COMMAND_NAME,
        "scope": report.scope,
        "query": report.query,
        "trust_override_active": report.trust_override_active,
        "excluded_flagged_count": report.excluded_flagged_count,
        "findings": [asdict(f) for f in report.findings],
    }

### 3.2 Shared-consumer contract wiring

`folio.search.search()` MUST call `folio.tracking.trust.derive_trust_status(view)` directly on each iterated view, **not** via a re-exported alias. `tests/test_trust.py::TestSharedConsumerInvariant::test_search_module_uses_shared_helper` asserts `folio.search.derive_trust_status is folio.tracking.trust.derive_trust_status` (extension of the existing graph + synthesize invariants), proving all three consumers call the same helper.

**§5 contract read path.** Search consumes `collect_pending_relationship_proposals(config, scope=, doc_id=, include_flagged=)` — the identical entry point graph + synthesize use. This is the load-bearing shared-consumer invariant: the function name is the contract boundary. Tests MUST assert the import + call exist (`tests/test_search.py::test_consumes_shared_collect_function`).

**Deferral for future `proposal_type` values.** v0.9.0 handles `proposal_type: "relationship"` only because the read boundary is `collect_pending_relationship_proposals` (the name describes the contract). Entity-merge and diagram-archetype proposals require a new collector or a generalized proposal iterator — neither exists today. `SearchFinding` is forward-compatible (subject_id / source_id / target_id / schema_gate_result / input_fingerprint all accept the needed values).

### 3.3 Shared payload-level `--json` envelope (first minor-version bump)

Parent carry-forward from synthesize v0.8.0 §3.3: this sub-slice owns the **first minor-version increment** of the shared envelope. Synthesize established `schema_version: "1.0"` with 6 required top-level keys `{schema_version, command, scope, trust_override_active, excluded_flagged_count, findings}` and a semver-esque versioning policy that states explicitly:

> "1.x — additive top-level keys permitted. Consumers MUST tolerate unknown top-level keys at the same major version. **Producers MUST bump minor (1.0 → 1.1) when adding a top-level key.**"

Search adds `query` as a seventh top-level key. Per the versioning policy, this triggers `schema_version: "1.1"` — the first production exercise of the minor-bump mechanic.

@click.argument("query", required=True)

    type=int,
    default=None,
    help="Maximum number of findings to emit (default: unbounded).",
)
@click.pass_context
def search(
    ctx,
    query: str,
    scope: Optional[str],
    producer: Optional[str],
    include_flagged: bool,
    json_output: bool,
    limit: Optional[int],
):
    """Search pending relationship proposals by QUERY substring (v0.9.0 structural MVP).

    QUERY is matched case-insensitively against six text fields on each
    pending proposal: source_id, target_id, relation, producer,
    reason_summary, and any element of evidence_bundle. Matching is a
    plain substring check via str.casefold(); regex and semantic search
    are not available in v0.9.0 (planned for v0.9.1+).

    Read-only: no LLM calls, no registry mutations. Emits the shared
    payload-level --json envelope at schema_version 1.1 with a new
    `query` top-level key (first minor-version bump of the shared
    envelope; see CHANGELOG v0.9.0).
    """
    from .search import search as run_search, render_envelope

    config = ctx.obj["config"]
    try:
        report = run_search(
            config,
            query=query,
            scope=scope,
            producer=producer,
            include_flagged=include_flagged,
            limit=limit,
        )
    except ScopeResolutionError as exc:
        click.echo(f"Error: {exc}", err=True)
        ctx.exit(1)

    if json_output:
        payload = render_envelope(report)
        click.echo(json.dumps(payload, indent=2))
        return

Prevents first-touch operator expectation mismatch. Parity with synthesize CB-4 discipline.

**Exit codes:**
- `0` — success (findings ≥ 0; zero matches is success).
- `1` — scope-resolution error.
- `2` — Click usage error (missing QUERY).
- Other non-zero — unexpected internal error.

### 3.5 Stdout renderer (`_render_search_stdout`)

Defined in `folio/search.py` (not `cli.py`, to keep CLI narrow). Mirrors synthesize §3.5 discipline with search-specific copy:

```python
def _render_search_stdout(report: SearchReport) -> None:
    """Compact human-readable summary: header + per-finding lines + footer.

    Always prints the excluded-flagged-count line (even at 0) to honor
    parent §11 rule 5 purposively. Zero-findings + zero-exclusions case
    adds a search-specific next-action breadcrumb.
    """
    scope_label = report.scope if report.scope is not None else "(library-wide)"
    click.echo(f"Search for '{report.query}' in scope: {scope_label}")
    click.echo(f"Matches: {len(report.findings)}")
    click.echo(
        f"Excluded (flagged inputs): {report.excluded_flagged_count}"
        + (
            " (use --include-flagged to include)"
            if report.excluded_flagged_count
            else ""
        )
    )
    if report.trust_override_active:
        click.echo("Trust override active: --include-flagged")
    if not report.findings and report.excluded_flagged_count == 0:
        click.echo(
            f"No matches for '{report.query}'. Try a broader scope, check "
            f"spelling, or run `folio ingest` / `folio enrich` if no "
            f"proposals have been emitted yet."
        )
    click.echo("")
    for finding in report.findings:
        flag = " [flagged]" if finding.trust_status == "flagged" else ""
        rel = finding.relation or "<n/a>"
        click.echo(
            f"[{finding.lifecycle_state}{flag}] "
            f"{finding.source_id} --{rel}--> "
            f"{finding.target_id or '—'}  ({finding.producer})"
        )
        click.echo(f"  {finding.reason_summary}")
```

Per-finding match-annotation (e.g., `[match: reason_summary]`) is deferred per §2.5.

### 3.6 QUERY matching semantics

**Normalization:** `needle = query.casefold()` computed once at `search()` entry. Field values are casefold'd at match-time (NOT pre-normalized on ingest; this slice adds no persistence).

**Matching rule:** A view matches QUERY if `needle` is a substring of **any** of the following six field values:

| Field | Type | Match rule | Notes |
|---|---|---|---|
| `view.source_id` | `str` | `needle in view.source_id.casefold()` | Always set. |
| `view.proposal.target_id` | `Optional[str]` | `view.proposal.target_id is not None and needle in view.proposal.target_id.casefold()` | Skip if None. |
| `view.proposal.relation` | `Optional[str]` | `view.proposal.relation is not None and needle in view.proposal.relation.casefold()` | Skip if None. |
| `view.producer` | `str` | `needle in view.producer.casefold()` | Always set. |
| `view.proposal.rationale` (→ `reason_summary`) | `str` | `needle in view.proposal.rationale.casefold()` | Always set. |
| `view.proposal.signals` (→ `evidence_bundle`) | `list[str]` | `any(needle in s.casefold() for s in view.proposal.signals)` | **Any-element rule.** Empty list never matches. |

**Non-searchable fields (explicit exclusion):** `proposal_id`, `subject_id`, `input_fingerprint` / `basis_fingerprint`, `schema_gate_result`, `lifecycle_state`, `flagged_inputs`, any field of `SuppressionCounts`. Rationale: these are identifier / metadata fields, not operator-facing content; searching them produces noise and couples the search UX to internal representation details.

**Short-circuit:** iteration over the six fields uses Python `or` short-circuiting (first match wins; rest not evaluated). This is a correctness non-issue (the view either matches or it doesn't) but a minor performance/determinism consideration — matches are determined, not ranked. `SearchReport.findings` iteration order mirrors the collector's order (currently insertion order across managed documents).

**Unicode discipline:** `str.casefold()` (NOT `.lower()`) is specified to handle Unicode case-folding correctly. Edge cases (`ß` → `ss`, Turkish dotless-I) are expected to work correctly because both QUERY and field values go through casefold. Tests SRC-8 + SRC-9 verify.

**Empty QUERY:** `search(query="")` raises `ValueError` (defensive guard). CLI layer should never reach this path because Click's `required=True` triggers exit code 2 on missing arg; but the function is defensive. Test SRC-CLI-query-required covers the CLI path.

## 5. §11 Trust Gating

Search honors parent §11 rules 1–5. Rule 6 (`extraction_confidence`) is not a search concern.

| Rule | How honored |
|---|---|
| 1. `review_status: flagged` inputs excluded by default | `collect_pending_relationship_proposals(include_flagged=False)` excludes flagged source/target; search passes the flag through. |
| 2. Trust posture surfaces explicitly | `trust_status` field on every finding. |
| 3. Explicit operator override `--include-flagged` | CLI flag; passed to collector; envelope `trust_override_active` reflects. |
