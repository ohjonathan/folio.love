# v0.6.3 Emission-Time Rejection Memory + Queue-Cap Enforcement Spec

## 1. Overview

This spec implements Slice 3 of the Tier 4 Shipping Plan
(`tier4_discovery_proposal_layer_spec.md` §15.3). Two features:

1. **Emission-time rejection memory** (§10.1): move the rejection-memory
   filter from review-time (`folio/links.py`, Slice 1) to emission-time
   (`folio/enrich.py`), so the enrichment pipeline marks proposals matching
   a previously rejected `basis_fingerprint` as `lifecycle_state: "suppressed"`
   instead of `"queued"`.

2. **Pending-queue cap enforcement** (§9.1): no producer may hold more than
   20 proposals in `lifecycle_state: "queued"` for a single engagement
   (= one folio library). Excess proposals are marked `"suppressed"`.

Both features write `lifecycle_state: "suppressed"` -- the first usage of
this state value, which entered the `PROPOSAL_LIFECYCLE_STATES` enum in
Slice 2 (v0.6.1) but had no writers until now.

**Critical fix included:** `enrich_note()` line 860 replaces
`fm["_llm_metadata"]["enrich"]` wholesale each run. Rejected proposals set
by `folio links reject` are lost on re-enrichment. This spec adds
rejected-proposal preservation across enrich runs.

## 3. Critical Fix: Rejected-Proposal Preservation

### 3.1 Problem

`enrich_note()` builds a fresh `enrich_block` dict (line 789) and writes
it to `fm["_llm_metadata"]["enrich"]` (line 860), replacing the entire
producer block. Any `lifecycle_state: "rejected"` entries previously set by
`folio links reject` are overwritten.

Currently masked: `input_fingerprint` idempotency prevents re-enrichment
when inputs haven't changed. But any input change (LLM profile update,
spec version bump, entity resolution drift) triggers re-enrichment,
and the rejection entry vanishes. On the *next* re-enrichment after that,
the LLM re-proposes the same relationship, and with no rejection entry
to match against, the proposal passes through -- violating §10.1's
zero-defect invariant.

### 3.2 Design

Before building the new proposals list, read existing proposals from
`_llm_metadata.enrich.axes.relationships.proposals` and preserve entries
with `lifecycle_state == "rejected"`. The final proposals list written
to frontmatter is:

```
preserved_rejected + new_proposals (queued or suppressed)
```

**Why only `"rejected"`, not `"suppressed"`?** Suppressed proposals from
a prior run have no value on re-enrichment: the LLM will re-propose
(or not) based on current inputs. Preserving old suppressed entries would
cause unbounded list growth. Rejected entries, by contrast, represent
deliberate operator decisions that must persist per §10.1.

### 3.3 Stale rejection cleanup

A rejected entry becomes stale when its `basis_fingerprint` no longer
matches any proposal the LLM would emit. This happens naturally:
when inputs change enough that the LLM stops proposing the same
relationship, the rejected entry lingers harmlessly. No active cleanup
is needed -- the entry is inert (it suppresses nothing) and small.

Future work (not this slice): a `folio links gc` command could prune
rejected entries whose `basis_fingerprint` is unreachable.

### 3.4 `_preserve_rejected_proposals` specification

```python
def _preserve_rejected_proposals(existing_meta: dict) -> list[dict]:
    """Return raw dicts for rejected proposals from existing frontmatter.

    Args:
        existing_meta: Output of ``_get_enrich_meta(fm)`` — the current
            ``_llm_metadata.enrich`` block before this enrich run overwrites it.

    Returns:
        List of raw proposal dicts where ``lifecycle_state == "rejected"``
        (with ``is None`` fallback to ``status`` via ``_STATUS_TO_LIFECYCLE``
        for legacy compat). Returns raw dicts (not ``RelationshipProposal``
        objects) because they are appended directly to the output list.
    """
```

Implementation:
1. Read `existing_meta.get("axes", {}).get("relationships", {}).get("proposals", [])`.
2. For each raw dict, compute effective lifecycle state (same `is None`
   fallback pattern as `_suppress_rejected_proposals`).
3. If state == `"rejected"`, include in output.
4. Return list of raw dicts (no deserialization needed).

## 4. Feature 1: Emission-Time Rejection Memory

### 4.1 Current behavior (Slice 1)

`_suppress_rejected_proposals()` (enrich.py lines 969-1003) runs
post-emission. It reads rejected proposals from existing frontmatter,
builds a map of `(relation, target_id) -> basis_fingerprint`, and
**drops** new proposals whose key matches with identical
`basis_fingerprint`. Dropped proposals vanish -- no audit trail.

### 4.2 New behavior

The function changes from "drop matching proposals" to "mark matching
proposals as suppressed":

**Before signature:**
```python
def _suppress_rejected_proposals(
    new_proposals: list[RelationshipProposal],
    existing_meta: dict,
    force: bool,
) -> list[RelationshipProposal]:
```

**After signature:**
```python
def _suppress_rejected_proposals(
    new_proposals: list[RelationshipProposal],
    existing_meta: dict,
    force: bool,
) -> tuple[list[RelationshipProposal], int]:
```

Returns `(proposals_with_state_set, suppression_count)`. Proposals
matching a rejected basis have `lifecycle_state` set to `"suppressed"`;
non-matching proposals remain `"queued"`. The `force` flag continues to
have no effect on rejection memory (§10.1 -- rejection memory is a
zero-defect product invariant).

### 4.3 Matching rules (unchanged from Slice 1)

- Match key: `(relation, target_id)` -- same as current.
- Basis comparison: exact string equality on `basis_fingerprint`.
- Changed basis: if `basis_fingerprint` differs, the proposal is
  **not** suppressed (it represents new evidence). This is the
  "revived" case from Slice 1's review-time filter.
- Empty `basis_fingerprint`: never suppresses (defensive).

### 4.4 Interaction with review-time filter

The Slice 1 review-time filter in `folio/links.py`
(`collect_pending_relationship_proposals`) remains as defense-in-depth.
It operates on `lifecycle_state == "queued"` entries, so suppressed
entries are already invisible to it. No changes needed.

## 5. Feature 2: Queue-Cap Enforcement

### 5.1 Spec requirement (§9.1)

> No producer may hold more than 20 proposals in lifecycle state
> `queued` for a single engagement at any time.

- **Engagement** = one folio library (one `folio` root).
- **Cap** = 20 proposals with `lifecycle_state == "queued"`.
- **State cap**, not rate cap -- no run counter or timestamps needed.
- **Per producer** -- currently only `"enrich"` producer exists, but
  the implementation is producer-aware for forward compatibility.

### 5.2 Counting method

Count `lifecycle_state == "queued"` proposals across ALL documents in
the library for the producer. The count includes:

1. Existing queued proposals in other documents' frontmatter (read from
   registry scan).
2. New queued proposals from the current enrich run (already marked
   queued after the rejection-memory filter).

The count does NOT include:
- Rejected entries (preserved but not queued).
- Suppressed entries (from rejection memory or prior cap enforcement).
- Proposals in documents outside the current library.

### 5.3 Implementation

New function `_count_library_queued_proposals(config, producer, exclude_doc_id)`:

1. Resolve `config.library_root / "registry.json"`.
2. Iterate `reg_data["decks"]` entries.
3. For each, read frontmatter and count proposals in
   `_llm_metadata.<producer>.axes.relationships.proposals`
   where `lifecycle_state == "queued"` (with `is None` fallback to
   `status` via `_STATUS_TO_LIFECYCLE` for legacy compat).
4. Skip the document currently being enriched (`exclude_doc_id`) --
   its proposals are being rebuilt this run and the old count is stale.
5. Return total count.

**Failure policy (L2 — catch+log+fallback):** The function reads
frontmatter from every registered document, which may include corrupted
or missing files. Wrap the per-document read in a try/except; log a
warning for any unreadable document and skip it. If the registry itself
is unreadable, return 0 (fail-open: cap is not enforced). This prevents
a single corrupted document from aborting the entire relationship
pipeline for an unrelated document.

New function `_enforce_queue_cap(proposals, existing_queued_count, cap=20)`:

1. Separate proposals into `queued` (lifecycle_state == "queued") and
   `non_queued` (already suppressed by rejection memory or in any other
   non-queued state). Only proposals with `lifecycle_state == "queued"`
   are eligible for cap suppression; proposals already in any other
   lifecycle state pass through unchanged.
2. Compute `remaining_capacity = cap - existing_queued_count`.
3. If `remaining_capacity <= 0`, mark ALL queued proposals as suppressed.
4. If `remaining_capacity > 0` but `< len(queued)`:
   - Stable sort by confidence (`"high"` before `"medium"`), preserving
     original emission order for equal confidence (Python's `sorted` is
     stable). This resolves PR-A-13 (within-batch tie-breaking).
   - First `remaining_capacity` proposals keep `"queued"`.
   - Remaining proposals set to `"suppressed"`.
5. Return `(non_queued + sorted_queued, cap_suppression_count)`.

### 5.4 Units clarification

Per §9: "engagement = one folio library." The library is identified by
`config.library_root`. All documents registered in `registry.json` under
that root are in the same engagement. The cap applies library-wide,
not per-document and not per-client/engagement frontmatter field.

## 6. Emission Flow (Revised)

### 6.1 Current flow (enrich.py lines 717-738)

```
for raw_p in raw_proposals:
    ... validate, compute fingerprint ...
    proposal = RelationshipProposal(..., lifecycle_state="queued")
    proposals.append(proposal)                                    # line 717

proposals = _enforce_singular_supersedes(proposals)               # line 720
proposals = _suppress_rejected_proposals(                         # line 723
    new_proposals=proposals,
    existing_meta=_get_enrich_meta(fm),
    force=force,
)
proposals = _remove_promoted_proposals(proposals, fm)             # line 730

if proposals:                                                     # line 732
    relationships_axis = EnrichAxisResult(
        status="proposed",
        proposals=[p.to_dict() for p in proposals],
    )
```

### 6.2 New flow

```python
existing_meta = _get_enrich_meta(fm)  # read BEFORE overwrite

for raw_p in raw_proposals:
    ... validate, compute fingerprint ...
    proposal = RelationshipProposal(..., lifecycle_state="queued")
    proposals.append(proposal)

proposals = _enforce_singular_supersedes(proposals)

# Step 10a: mark rejected-basis matches as suppressed (not dropped)
proposals, rejection_suppressed = _suppress_rejected_proposals(
    new_proposals=proposals,
    existing_meta=existing_meta,
    force=force,
)

# Step 10b: remove proposals already promoted to canonical fields
proposals = _remove_promoted_proposals(proposals, fm)

# Step 10c: enforce queue cap (NEW)
existing_queued = _count_library_queued_proposals(
    config, "enrich", exclude_doc_id=plan_entry.entry.id,
)
proposals, cap_suppressed = _enforce_queue_cap(
    proposals, existing_queued, cap=QUEUE_CAP,
)

# Step 10d: preserve rejected entries from previous runs (NEW)
preserved_rejected = _preserve_rejected_proposals(existing_meta)

# Build output: preserved rejected + new proposals
all_proposal_dicts = (
    preserved_rejected + [p.to_dict() for p in proposals]
)

# Suppression diagnostics
suppression_counts = {}
if rejection_suppressed:
    suppression_counts["rejection_memory"] = rejection_suppressed
if cap_suppressed:
    suppression_counts["queue_cap"] = cap_suppressed

# Operator feedback via warnings (no cli.py change needed)
if suppression_counts:
    parts = []
    for reason, count in suppression_counts.items():
        parts.append(f"{reason}: {count}")
    warnings.append(
        f"{sum(suppression_counts.values())} proposal(s) suppressed "
        f"({', '.join(parts)})"
    )

# Use "proposed" even when all new proposals are suppressed,
# as long as proposals exist (suppressed or preserved rejected).
# "no_change" means genuinely zero proposals generated.
if all_proposal_dicts:
    relationships_axis = EnrichAxisResult(
        status="proposed",
        proposals=all_proposal_dicts,
        suppression_counts=suppression_counts or None,
    )
else:
    relationships_axis = EnrichAxisResult(status="no_change")
```

### 6.3 Ordering rationale

1. **Singular supersedes** first -- reduces the candidate set before
   more expensive checks.
2. **Rejection memory** second -- marks suppressed before cap counting
   (suppressed proposals don't count against the cap).
3. **Remove promoted** third -- canonical-field promotions reduce queued
   count.
4. **Queue cap** fourth -- operates on the final queued set.
5. **Preserve rejected** last -- appends historical entries that don't
   interact with the filtering pipeline.

## 7. Suppression Diagnostics

### 7.1 `EnrichAxisResult` extension

Add optional `suppression_counts` field:

```python
@dataclass
class EnrichAxisResult:
    status: str
    # ... existing fields ...
    proposals: Optional[list[dict]] = None
    suppression_counts: Optional[dict[str, int]] = None  # NEW

    def to_dict(self) -> dict:
        d: dict = {"status": self.status}
        # ... existing serialization ...
        if self.suppression_counts is not None:
            d["suppression_counts"] = dict(self.suppression_counts)
        return d
```

### 7.2 Diagnostic output

The `suppression_counts` dict appears in frontmatter under
`_llm_metadata.enrich.axes.relationships.suppression_counts`:

```yaml
_llm_metadata:
  enrich:
    axes:
      relationships:
        status: proposed
        suppression_counts:
          rejection_memory: 2
          queue_cap: 5
        proposals:
          - lifecycle_state: rejected    # preserved from prior run
            relation: supersedes
            ...
          - lifecycle_state: queued      # new, passed both filters
            relation: impacts
            ...
          - lifecycle_state: suppressed  # rejected-basis match
            relation: supersedes
            ...
```

`folio enrich diagnose` and `folio enrich` summary output can read these
counts. No CLI changes in this slice (cli.py is forbidden path); the
counts are available for future slices to surface.

### 7.3 `EnrichResult.proposals_count` (NOT CHANGED)

`proposals_count` (enrich_data.py line 179) is set by the caller in
`enrich_note()`, not by `EnrichAxisResult`. Changing its semantics
(from all proposals to queued-only) would be a silent scope expansion
not authorized by §15.3/§15.7. The count continues to reflect all
proposals in the list (including suppressed and preserved rejected).
A future slice may revisit this with proper scope authorization.

### Changed
- **Emission-time rejection memory:** proposals matching a previously
