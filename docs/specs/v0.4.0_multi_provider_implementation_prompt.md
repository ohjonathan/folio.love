## Task Context

### What to Build

Build v0.4.0 multi-provider LLM support for Folio conversion. This is not just “add OpenAI and Gemini.” It is a controlled refactor of the current Anthropic-only analysis path into a provider-agnostic architecture with:

- named LLM profiles in config
- route-based profile selection for `convert`
- `--llm-profile` CLI override for `convert` and `batch`
- Anthropic, OpenAI, and Google Gemini provider adapters
- transient-only fallback chains
- JSON-only response contract
- provider/model-aware cache invalidation
- internal `_llm_metadata` frontmatter provenance
- provider-aware pending-analysis messaging

### Scope Boundaries

Current runtime scope is only `folio convert` and `folio batch`, and only the `convert` LLM route is active in v0.4.0.

Future routes such as `ingest`, `enrich`, and `query_answer` must be supported by the config shape, but they are not implemented in this work.

### What Not to Build

- No dynamic model auto-routing.
- No `--provider` or `--model` CLI flags.
- No raw API keys in `folio.yaml`.
- No mixed route selection between Pass 1 and Pass 2. Both passes use the same resolved `convert` route in v0.4.0.
- No ontology schema expansion for provider/model metadata. Use `_llm_metadata` only.
- No changes to extraction stages in `folio/pipeline/normalize.py`, `folio/pipeline/images.py`, or `folio/pipeline/text.py`.
- No changes to version tracking semantics beyond documenting that provider/model changes do not create a new source-content version.
- No prose-response parsing kept around “for compatibility.” The old prose parsing functions are removed, not deprecated.
- No text-only fallback behavior for a provider that lacks image support. All three supported providers must run multimodal analysis.

### Rollout Constraint

Do not start with “just add provider adapters.” The approved spec requires a prompt migration and JSON parsing quality gate first. Implementation order below is mandatory for regression control.

## Target Architecture

### Provider Abstraction

Create `folio/llm/` and keep the provider abstraction outside `analysis.py`. `analysis.py` remains the orchestrator.

Create these files:

- `folio/llm/__init__.py`
- `folio/llm/types.py`
- `folio/llm/providers.py`
- `folio/llm/registry.py`

### Required Core Types

Implement these exact contract types in `folio/llm/types.py`:

```python
from dataclasses import dataclass
from typing import Literal, Protocol


@dataclass
class ProviderInput:
    model: str
    prompt: str
    image_bytes: bytes | None
    media_type: str | None
    max_tokens: int


@dataclass
class ProviderOutput:
    text: str
    truncated: bool


@dataclass
class ErrorDisposition:
    kind: Literal["transient", "permanent"]
    message: str
    original: Exception | None = None


class AnalysisProvider(Protocol):
    provider_name: str

    def create_client(self, api_key: str) -> object: ...

    def analyze(self, client: object, request: ProviderInput) -> ProviderOutput: ...

    def classify_error(self, exc: Exception) -> ErrorDisposition: ...
```

Additional implementation-only dataclasses are allowed and recommended. Use them to avoid passing loose dicts between config, converter, and analysis. At minimum, add a resolved profile/route type and a stage metadata type. If you name them differently, preserve these fields:

- route name
- requested profile name
- primary profile name
- provider
- model
- api key env var
- fallback profile names
- whether CLI override disabled fallback
- stage status
- reason
- whether fallback was used

### Provider Responsibilities

Provider adapters own only provider-specific behavior:

- client creation
- provider-specific multimodal request payload construction
- provider-specific image encoding or image part construction
- provider-specific response text extraction
- provider-specific truncation detection
- provider-specific exception classification

Provider adapters do not own:

- prompt construction
- retry policy
- fallback policy
- JSON extraction
- JSON normalization
- evidence validation
- cache load/save
- density scoring
- frontmatter metadata assembly

### Client Lifetime Rule

The orchestrator owns retry/fallback policy. Disable SDK-level automatic retries where the SDK supports them so Folio’s explicit “one retry, then fallback” contract remains true. This applies to Anthropic and OpenAI at minimum; check Google’s client behavior and avoid hidden automatic retries there as well.

Create each provider client once per pass invocation and reuse it across all slides in that pass.

If a transient failure triggers fallback during a pass:

- instantiate the fallback client once
- reuse that fallback client for the remaining slides in that pass
- record per-slide `_provider` / `_model` provenance in cache entries

Pass 2 uses the same resolved route as Pass 1, but it starts from the route’s primary profile again. Do not carry a pass-1 fallback choice into pass 2 as sticky route state.

## Common Output Contract

### Pass 1 JSON Contract

Every provider must return one JSON object with this shape:

```json
{
  "slide_type": "data",
  "framework": "tam-sam-som",
  "visual_description": "string",
  "key_data": "string",
  "main_insight": "string",
  "evidence": [
    {
      "claim": "string",
      "quote": "string",
      "element_type": "title",
      "confidence": "high"
    }
  ]
}
```

### Pass 2 JSON Contract

Every provider must return one JSON object with this shape:

```json
{
  "slide_type_reassessment": "unchanged",
  "framework_reassessment": "unchanged",
  "evidence": [
    {
      "claim": "string",
      "quote": "string",
      "element_type": "body",
      "confidence": "medium"
    }
  ]
}
```

### Normalization Rules

After JSON parse:

- lowercase-hyphenate `slide_type`, `framework`, `slide_type_reassessment`, `framework_reassessment`
- normalize `element_type` to `title`, `body`, or `note`; default invalid values to `body`
- normalize `confidence` to `high`, `medium`, or `low`; default invalid values to `medium`
- cap evidence to 10 items and log the original count when truncating
- reject Pass 1 payloads with zero evidence items
- reject Pass 2 payloads with zero evidence items

### Mapping to `SlideAnalysis`

Pass 1 mapping:

- `slide_type` -> `SlideAnalysis.slide_type`
- `framework` -> `SlideAnalysis.framework`
- `visual_description` -> `SlideAnalysis.visual_description`
- `key_data` -> `SlideAnalysis.key_data`
- `main_insight` -> `SlideAnalysis.main_insight`
- each evidence item becomes a dict with:
  - `claim`
  - `quote`
  - `element_type`
  - `confidence`
  - `validated=False` before source-text validation
  - `pass=1`

Pass 2 mapping:

- evidence items become pass-2 evidence dicts with `pass=2`
- `slide_type_reassessment != "unchanged"` -> `SlideAnalysis.pass2_slide_type`
- `framework_reassessment != "unchanged"` -> `SlideAnalysis.pass2_framework`
- do not overwrite pass-1 `slide_type` or `framework`

### JSON Extraction Algorithm

Add this exact helper to `folio/pipeline/analysis.py`:

```python
def _extract_json(raw_text: str) -> str | None:
    """Return a JSON object string or None."""
```

Behavior:

1. attempt `json.loads(raw_text)` directly
2. if that fails, strip one surrounding markdown code fence pair plus optional `json` language tag and retry
3. if that still fails, return `None`

Do not add regex-based fuzzy scraping for arbitrary JSON substrings.

### Prompt Rewrite Requirement

Current `ANALYSIS_PROMPT` and `DEPTH_PROMPT` are prose prompts. Replace both with JSON-contract prompts before adding non-Anthropic providers.

This is not optional. Prompt rewrite is the first implementation slice because:

- it changes the output contract
- it invalidates existing caches through `_prompt_version`
- it removes the old prose parser path
- it is the only way to compare Anthropic quality before multiplying the blast radius across more providers

Anthropic parity in this project means:

- same required fields are produced
- evidence validation still works
- pass selection and caching still work
- frontmatter and markdown structure remain correct

Anthropic parity does not mean byte-for-byte identical LLM wording, because the prompt format is changing from prose to JSON.

## Downstream Interaction Map

| System | What it depends on today | Risk if provider output differs | What developers must preserve |
|---|---|---|---|
| Evidence parsing | `analysis.py` regex parser expects prose `Evidence:` blocks | Non-Anthropic providers will not reliably match prose markers | Replace prose parsing with JSON normalization. The orchestrator, not providers, must produce the final evidence dict shape |
| Evidence validation | `_validate_evidence()` matches `quote` against extracted text | Provider quoting may be shorter, fenced, or differently normalized | Preserve exact `quote`-based validation path; do not move validation into adapters |
| Density scoring | `_compute_density_score()` reads `SlideAnalysis.evidence`, `framework`, `slide_type`, `key_data` | Missing or differently normalized fields will break pass-2 eligibility | Ensure all providers populate the same `SlideAnalysis` fields before density scoring |
| Multi-pass | `DEPTH_PROMPT` interpolates pass-1 fields into pass 2 | If pass-1 structure changes, pass 2 breaks | Keep pass-2 input fields identical from the orchestrator’s perspective. Pass 1 and Pass 2 use the same resolved route in v0.4.0 |
| Caching | `.analysis_cache*.json` currently validate by prompt + model + extraction version | Cross-provider cache hits would silently reuse wrong analysis | Add provider metadata at cache root and per-slide provenance; do not change per-entry keys |
| Version tracking | `versions.compute_version()` and FR-301 are text-content-based | Provider/model changes could accidentally look like a version bump if wired incorrectly | Do not alter version tracking. Provider/model changes only affect analysis cache and `_llm_metadata` |
| Frontmatter | `frontmatter.generate()` aggregates `frameworks`, `slide_types`, `grounding_summary` | Pending or malformed provider outputs can pollute ontology fields | Keep ontology fields unchanged; add only `_llm_metadata` as an internal block |
| Markdown rendering | Pending message is currently Anthropic-specific | Users will see wrong key/setup guidance for OpenAI or Google | Render pending reason from the analysis object, not a hardcoded env var |

Behavior requirements:
