# v0.6.5 Entity-Merge Rejection Memory Spec

## 1. Overview

Implements **Shipping Plan §15.6** (first half, entity-merge) of `docs/specs/tier4_discovery_proposal_layer_spec.md` (rev 5). Applies the rejection-memory filter pattern from slice 1 (v0.6.0, `folio links review`) to entity-merge candidates surfaced by `folio entities suggest-merges` and `folio graph doctor`.

**Key design decision:** entity-merge candidates today are **deterministic heuristic suggestions** produced by `EntityRegistry.suggest_person_merges()` (no LLM producer, no stored proposal object, no lifecycle_state). This slice does NOT introduce an `EntityMergeProposal` dataclass. Instead, it layers **rejection records** on top of the heuristic surface. Parent §5 proposal-object contract is for LLM-emitted proposals; heuristic suggestions use the parent §7 entity-merge fingerprint clause directly.

**Operator upgrade path:** no action required. `entities.json` schema gains a `rejected_merges` array; missing-on-load is treated as empty.

## 3. Storage Schema

### 3.1 `entities.json` gains a top-level `rejected_merges` key

```json
{
  "_schema_version": 1,
  "updated_at": "<iso>",
  "entities": { ... },
  "rejected_merges": [
    {
      "subject_type": "person",
      "entity_keys": ["alice_chen", "chen_alice"],
      "basis_fingerprint": "sha256:...",
      "reasons_at_rejection": ["last_first_transpose"],
      "rejected_at": "2026-04-16T..."
    }
  ]
}
```

`entity_keys` is stored lexicographically sorted.

**No schema-version bump** (B.1 R1 codex ADV-001 resolution). `_ENTITY_SCHEMA_VERSION` remains `1`. Rationale: (a) adding a top-level key is additive; (b) `EntityRegistry.save()` preserves unknown top-level keys via `self._data` write-through, so pre-v0.6.5 code that reads/writes the file won't strip `rejected_merges`; (c) the existing loader at `entities.py:555-561` rejects newer versions with `EntitySchemaVersionError`, so bumping to 2 would break downgrade (new-writer / old-reader) — a real regression for any adopter who reverts. Missing `rejected_merges` on load is handled by `setdefault("rejected_merges", [])` post-load; fully backward-compat in both directions.

### 3.2 `basis_fingerprint` derivation (per parent §7)

```python
import json
import hashlib

subject_type = "person"
sorted_keys  = sorted([left_key, right_key])
sorted_reasons = sorted(reasons)  # heuristic labels are deterministic from current alias state
payload = {
    "subject_type": subject_type,
    "entity_keys": sorted_keys,
    "reasons": sorted_reasons,
}
serialized = json.dumps(payload, sort_keys=True, separators=(",", ":"), ensure_ascii=False)
basis_fingerprint = "sha256:" + hashlib.sha256(serialized.encode("utf-8")).hexdigest()
```

Canonical JSON serialization (B.1 R1 codex ADV-004 resolution) ensures injective encoding: `reasons = ["a", "b"]` and `reasons = ["a|b"]` now produce distinct hashes. This matches the "no ambiguous string parsing" standard expected for a persisted fingerprint.

The `reasons` list acts as the "supporting input state" clause from parent §7 ("each supporting input's current version or source identifiers"): if entity aliases change such that a new heuristic fires (e.g., a new alias adds `alias_overlap` to the reasons set), `basis_fingerprint` changes → the suggestion revives.

### 3.3 Rejection-key tuple

Analog of slice 1's 4-tuple `(source_id, target_id, relation, basis_fingerprint)`:

```
(subject_type, sorted_keys[0], sorted_keys[1], basis_fingerprint)
```

4 components. Identity by sorted pair + subject_type; basis by fingerprint. Two records with matching identity but different fingerprints = revival case.

## 4. Technical Design

### 4.1 `folio/tracking/entities.py`

1. **`_empty_entity_registry()`** (line 420): add `"rejected_merges": []` to returned dict. **No schema-version bump** (see §3.1 rationale).

2. **`EntityRegistry.load()`** (line 528): after successful load, apply `self._data.setdefault("rejected_merges", [])`. Legacy files without the key become `[]`; new files retain their records. Placed at the end of `load()` after schema-version validation and entity-dict normalization so it runs exactly once per load.

3. **New function `_compute_entity_merge_basis_fingerprint(subject_type, left_key, right_key, reasons) -> str`**: implements §3.2 canonical-JSON hashing. Unit-testable.

4. **`EntityRegistry.suggest_person_merges(apply_rejection_memory: bool = True) -> list[PersonMergeSuggestion]`**:
   - Compute raw suggestions as today.
   - For each suggestion, compute `basis_fingerprint`.
   - Build rejected-key index: `{(subject_type, lk, rk): set(fps)}` from `self._data["rejected_merges"]`.
   - If `apply_rejection_memory=True`: drop suggestions whose `(subject_type, sorted_keys, basis_fingerprint)` matches a stored rejection.
   - Annotate revived suggestions: if `(subject_type, sorted_keys)` has ANY rejection records but current `basis_fingerprint` doesn't match any, mark `revived=True` on the suggestion.
   - Return filtered + annotated list, preserving the existing sort order (left_name, right_name, left_key, right_key). **Revived suggestions do NOT float to the top** — they appear in their natural alphabetical position with the inline annotation; the suppression-count and revival surfaces are what signal operator attention.

5. **`PersonMergeSuggestion` dataclass** (line 479): add fields `basis_fingerprint: str = ""` and `revived: bool = False`. Defaults preserve existing call sites.

6. **New method `EntityRegistry.reject_person_merge(left_key, right_key) -> tuple[bool, str]`**:
   - Validate both keys exist under `entities["person"]`; raise `EntityRegistryError(f"...'{key}' no longer exists (possibly already merged)")` if not. (B.1 R1 ADV-005 resolution — explicit stale-candidate error.)
   - Compute current reasons for the pair (from same heuristic bucketing as `suggest_person_merges`).
   - Compute `basis_fingerprint`.
   - Check for existing exact match: if a record with same `(subject_type, sorted_keys, basis_fingerprint)` exists, return `(False, "already rejected")` — idempotent no-op. Otherwise, append rejection record and return `(True, "rejected")`.
   - Update `updated_at` only on real append.

7. **New method `EntityRegistry.count_entity_merge_suppressions() -> int`**: returns the count of currently-stored rejection records (renamed from `collect_entity_merge_suppression_counts` per B.1 R1 peer S-4; scalar return, clean naming). Used by CLI for `M total rejections recorded` disclosure. Called separately from `suggest_person_merges` to report total rather than filter-delta.

### 4.2 `folio/cli.py`

1. **`suggest_merges_cmd`** (line 2473): Required behavior at command entry:
   - Wrap load + registry work in `library_lock(library_root, "entities")` (read path — consistent with existing entity commands).
   - Even when the suggestion list is empty, or the `--type` filter produces zero results, or paging returns an empty page: render the disclosure line at the END of the command flow. The disclosure is NEVER bypassed by early returns.
   - Revived candidates: inline suffix `(revived — basis changed)` on their line (matches slice 1 verbatim).
   - Three-branch disclosure copy (B.1 R1 peer S-2 + product PR-SF-1):
     - N=0: `No merge candidates suppressed by rejection memory.`
     - N=1: `1 merge candidate suppressed by rejection memory.`
     - N>1: `{N} merge candidates suppressed by rejection memory.`
   - Additionally render total-rejections disclosure (B.1 R1 product PR-SF-4): `({M} total rejections recorded.)` where `M = registry.count_entity_merge_suppressions()`. Rendered on a line immediately after the suppression-count line. When `M == 0`, render `(0 total rejections recorded.)` rather than suppressing — operator sees the surface even when pristine.

2. **New command `folio entities reject-merge <left_key> <right_key>`**:
   - Click command on `entities` group.
   - Docstring: `"""Reject a merge candidate so it stops surfacing in suggest-merges.\n\nExample: folio entities reject-merge alice_chen chen_alice\n"""` (B.1 R1 product PR-MN-1).
   - Wrap the entire load-mutate-save in `library_lock(library_root, "entities")` (B.1 R1 codex ADV-002 resolution — prevents lost-update race with concurrent rejects/merges).
   - Loads registry, invokes `reject_person_merge`, saves only on real append.
   - Output (B.1 R1 product PR-SF-2 + PR-SF-3):
     - First reject: `"✓ Rejected merge candidate: {left_name} [{left_key}] <> {right_name} [{right_key}] (reasons: {', '.join(reasons)})"`, exit 0.
     - Idempotent reject (same fingerprint already recorded): `"= Already rejected: {left_name} <> {right_name} (no change)"`, exit 0.
     - Unknown / stale key: `"✗ Merge candidate is stale: '{key}' no longer exists (possibly already merged)"` on stderr, exit non-zero.

**Locking scope decision (B.1 R1 codex ADV-002):** This slice locks only the new `reject-merge` command plus the existing `suggest-merges` read path. Broader entity-family locking across `merge`, existing `reject`, `import` is OUT OF SCOPE and recorded as a follow-up. The spec notes that operators running `merge` and `reject-merge` in parallel terminals can still race in the merge vs reject-merge direction; documented as a known limitation that surfaces as an `EntityRegistryError` ("key no longer exists") rather than silent corruption, because merge removes the loser key.

### 4.3 `folio/graph.py`

Two call sites to `suggest_person_merges()` — enumerated explicitly (B.1 R1 codex ADV-003 + peer S-1 resolution):

1. **`graph_status()` line 126-134**: aggregate `duplicate_person_candidates` count rendered by `folio graph status`. **Filtered**: invoke with `apply_rejection_memory=True` (the default). Semantic change: the count now reports "reviewable duplicates" (not latent raw duplicates). Operator-visible change on the `graph status` surface.
2. **`graph_doctor()` line 224-260**: per-pair `duplicate_person_candidate` finding loop. **Filtered**: same default. Same semantic ("reviewable").

Both sites MUST pass `apply_rejection_memory=True` **explicitly**, even though it matches the default (B.2 R2 codex R2-ADV-003). Rationale: `graph_status` and `graph_doctor` are operator surfaces; making the filter-semantics visible in code prevents silent drift if a future slice changes the default.

Documentation: update `folio graph status` output label if the change is observable to the operator — if the CLI currently renders `Duplicate person candidates: {N}`, change label to `Reviewable duplicate person candidates: {N}` to disclose the filtering semantics. **Unchanged behavior when no rejections exist** — the filtered count equals the raw count. Tested at T-11b.

## 6. Risk Assessment

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| Old-writer / new-reader compatibility — pre-v0.6.5 code saves `entities.json` and loses `rejected_merges` | Low | Low | `save()` preserves unknown top-level keys via `self._data` write-through; no bump means no version check fires. Regression test T-14 covers this. Downgrade path works because v0.6.5 files use version 1 (no bump). |
| Concurrent `reject-merge` lost-update race | High | Medium | `library_lock(library_root, "entities")` wraps the load-mutate-save critical section per §4.2. Test T-15 verifies existing lock blocks. |
| `basis_fingerprint` encoding ambiguity | Medium | Low | Canonical JSON serialization (sort_keys, fixed separators, ensure_ascii=False) per §3.2; injective encoding even if reason labels later contain `|` or unicode. |
| `basis_fingerprint` derivation changes break rejection memory | Medium | Low | Hash inputs pinned by spec §3.2; any future change is an explicit migration (covered when the entity-merge proposal object ships). |
| Revival false-positive from non-deterministic reasons | Medium | Low | `reasons` always sorted before hashing (§3.2). Test T-4. |
| `folio graph status` count-semantics surprise — filter changes reported value | Medium | Medium | Label changed to "Reviewable duplicate person candidates" per §4.3 to disclose filtering; CHANGELOG entry calls out the semantic change. |
| Cross-family lock races (merge vs reject-merge) | Low | Low | Out of scope; documented as known limitation in §4.2. Surfaces as `EntityRegistryError`, not silent corruption. |
| `folio graph doctor` latency from fingerprint computation | Low | Low | sha256 of short canonical-JSON string; negligible. |
| `rejected_merges` list growth unbounded | Low | Medium | ~200 bytes/record. Future housekeeping slice can prune/archive. |
