# v0.5.1 — Tier 3 Entity System Kickoff Spec

## 1. Overview

This spec defines the **minimum viable entity system** for folio.love: a
lightweight entity registry, a CLI for viewing and importing entities, and
ingest-time name resolution that turns unresolved `[[wikilinks]]` into
canonical graph nodes.

The entity system is the second Tier 3 slice (Week 16–18), immediately
following the `folio ingest` slice (v0.5.0, PR #32, merged to `main`).

**Output path:** `docs/specs/v0.5.1_tier3_entity_system_spec.md`

**Version label rationale:** v0.5.1 is correct. The entity system builds
directly on the v0.5.0 ingest baseline without introducing a new top-level
command family outside the `folio entities` subcommand. It is an incremental
capability extension, not a new product surface. PR A and PR B are both
covered under v0.5.1.

**Shipped-baseline note (2026-04-15):** `folio entities suggest-merges` and
`folio entities merge` later shipped as Tier 4 graph-ops identity hygiene.
They are outside the v0.5.1 scope of this spec, but they are no longer future
product gaps.

---

## 2. Why This Slice Comes Next

`folio ingest` (v0.5.0) ships entity extraction: it identifies people,
departments, systems, and processes from interview transcripts and renders them
as unresolved Obsidian wikilinks. But those wikilinks are names floating in
markdown — there is no canonical registry, no alias resolution, no import path
for known org data, and no way to determine whether `Jane`, `Jane Smith`, and
`the CTO` refer to the same person.

The entity system makes those names real. It is the smallest slice that
materially improves the shipped ingest baseline by:

1. Giving users a place to declare known entities before ingest
2. Resolving extracted names against that registry during ingest
3. Surfacing unconfirmed entities for human review

Without this slice, every interaction note is an island. With it, cross-note
queries like "show me all interviews mentioning Engineering" become reliable
because the names are canonical.

---

## 3. Goals

1. **Entity registry** — a machine-readable, human-editable store of known
   entities with canonical names, aliases, and type-specific metadata.
2. **CSV import** — bulk-load people and departments from an org chart export.
3. **CLI** — `folio entities` for listing, viewing, confirming, and rejecting
   entities.
4. **Ingest-time resolution** — exact-match and alias-match resolution against
   the registry during `folio ingest`, replacing raw extracted names with
   canonical names in rendered wikilinks.
5. **LLM-proposed soft match** — when exact/alias match fails, the LLM proposes
   a match against the registry, flagged `needs_confirmation`. Required for
   PR B.
6. **Human confirmation boundary** — clear mechanism for users to confirm,
   reject, or defer auto-created and LLM-proposed entities.

---

## 4. Non-Goals

These are explicitly out of scope for v0.5.1:

| Item | Why deferred |
|------|-------------|
| `folio enrich` | Separate Tier 3 slice (Week 19–20) |
| Retroactive rewrite of existing notes | Re-ingest is the mechanism; batch rewrite is an enrich concern |
| Relationship graph authoring | Beyond name resolution; requires enrichment pass |
| Semantic search over entities | Tier 4 capability |
| Recursive org traversal queries | Exceeds Dataview; deferred to Tier 4 per ontology §5 |
| Entity markdown stub files | Nice-to-have for Obsidian navigation; deferred unless trivial |
| `folio entities edit` | v0.5.1 is read + import + confirm + reject; direct edit UX remains outside this slice |
| Algorithmic fuzzy matching | No Levenshtein / Jaro-Winkler / phonetic matching. LLM soft match is the approved non-exact mechanism. |
| Cross-library entity sharing | Libraries are independent; no federation in v1 |
| Entity confidence scoring | Unlike findings, entities have no per-item confidence in v0.5.0 |
| Cross-type resolution fallback | Resolution is type-strict in v1 (see §10.2) |

## 5. Current Shipped Baseline

These facts were verified against the `main` branch as of PR #32 merge.

### 5.1 What exists

1. **`folio ingest`** creates ontology-native `type: interaction` documents.
2. **`folio/pipeline/interaction_analysis.py`** extracts four entity types via
   LLM: `people`, `departments`, `systems`, `processes`. Returns
   `InteractionAnalysisResult.entities: dict[str, list[str]]`.
3. **`_coerce_entities()`** deduplicates case-insensitively within a single
   note, preserving original case of first occurrence.
4. **`folio/output/interaction_markdown.py`** renders entities as unresolved
   wikilinks in an `## Entities Mentioned` section with per-type subsections.
5. **`folio/output/frontmatter.py` `generate_interaction()`** does NOT include
   entities in YAML frontmatter — entities live only in the markdown body.
6. **`--participants`** CLI flag stores user-provided participants in
   frontmatter, orthogonal to LLM-extracted entities.
7. **`registry.json`** tracks interaction entries alongside evidence entries
   using the same `RegistryEntry` schema. No entity fields exist in the
   registry.
8. **`folio status`** includes interaction documents in counts.
9. **`folio scan`** includes `.txt`/`.md` transcript sources.
10. **`folio refresh`** skips interaction entries (tells user to re-run ingest).
11. **`folio promote`** works on interactions; gates on `review_status`.

### 5.2 What does NOT exist

1. No entity registry file.
2. No `folio entities` CLI command.
3. No `folio entities import` command.
4. No entity resolution during ingest.
5. No alias system.
6. No `needs_confirmation` flag on any entity.
7. No entity fields in `RegistryEntry` or `registry.json`.

### 5.3 LLM prompt constraint

The current analysis prompt instructs:
> "Use unresolved entity names only; do not invent IDs."

This instruction will need to be extended (not replaced) when registry-backed
resolution is added.

## 6. Key Decisions

### D1: JSON registry, not markdown directory

**Decision:** Entities are stored in a single `entities.json` file alongside
`registry.json` in the library root.

**Rationale:**
- Consistent with the existing `registry.json` pattern.
- Entities are graph nodes, not documents (ontology §5). A markdown directory
  would conflate entities with Folio documents.
- Atomic read/write avoids partial-update bugs.
- Human-editable in any text editor.
- Obsidian wikilink resolution does NOT require target files — wikilinks still
  function as search anchors. Entity stub files can be generated later if
  needed.

**Trade-off acknowledged:** Without entity markdown files, `[[Jane Smith]]`
shows as an unresolved link in Obsidian's graph view. This is acceptable for
v1 because the primary value is cross-note search (`[[Jane Smith]]` is still
searchable across all files), not graph visualization.

### D2: Type-namespaced, slug-keyed

Entities are namespaced by type to prevent collisions (e.g., "Operations" as
both a department and a process). Keys are slugified canonical names.

### D3: Library-scoped, client-optional

One `entities.json` per library. The `client` field on each entity is optional.
No global entity federation in v1. A consultant who works across clients keeps
all entities in one library, tagged by client when relevant.

**Multi-client same-name policy (S4):** Two entities of the same type with the
same canonical name but different clients are NOT supported in v1. The slug key
is derived from the canonical name alone, so `Jane Smith` at ClientA and
`Jane Smith` at ClientB would collide. If a user has same-name entities across
clients, they must disambiguate the canonical name (e.g., "Jane Smith (Acme)"
vs "Jane Smith (Globex)"). This is documented as a known limitation.

### D4: Auto-create with `needs_confirmation`, excluded from resolution

When ingest encounters an entity not in the registry:
- A new entity entry is auto-created with `needs_confirmation: true`.
- **Unconfirmed entities do NOT participate in future resolution.** They exist
  in the registry for visibility and confirmation workflow, but the resolution
  algorithm (§10.2) only matches against confirmed entities.
- The user confirms via `folio entities confirm` or rejects via
  `folio entities reject`.

This prevents bad extractions from becoming self-reinforcing graph state. A
typo like "Jnae Smith" gets auto-created but never drives resolution of future
ingests until a human confirms it.

### D5: LLM soft match is required for PR B

The roadmap requires "LLM-proposed soft match with human confirmation." This is
part of the approved v1 resolution model and is **required** for PR B, not a
stretch goal. The full confirmation/rejection flow is specified in §10.2 and
§11.

### D6: Re-ingest is the sole update mechanism

Existing interaction notes are NOT retroactively rewritten. To resolve
previously unresolved entities, the user re-ingests the source transcript.
Re-ingest detects the existing note via `source_hash` identity and updates it
in place. There is no lighter-weight re-resolve path in v0.5.1 — a batch
re-resolve command would constitute retroactive note mutation, which is
deferred to enrich per §4 and the governing roadmap/ontology docs.

### D7: Type-strict resolution, no cross-type fallback

Resolution searches only within the extracted entity's type namespace. If the
LLM extracts "Operations" as a system, resolution searches only
`entities.system`, not `entities.department`. This prevents silent wrong-type
matches. If the LLM miscategorizes an entity, the correct fix is to re-ingest
or improve the extraction prompt — not to paper over the error with cross-type
guessing.

**Rationale:** All three reviewers flagged cross-type fallback as unsafe and
outside the approved v1 resolution model. The extraction already assigns types;
silently overriding that assignment at resolution time adds complexity and
hides errors.

## 7. Entity Registry Model

### 7.1 File location

```
<library_root>/entities.json
```

### 7.2 Top-level schema

```json
{
  "_schema_version": 1,
  "updated_at": "2026-03-22T14:30:00Z",
  "entities": {
    "person": { ... },
    "department": { ... },
    "system": { ... },
    "process": { ... }
  }
}
```

### 7.3 Entity entry schema (all types)

```json
{
  "canonical_name": "Jane Smith",
  "type": "person",
  "aliases": ["Jane", "the CTO", "J. Smith"],
  "needs_confirmation": false,
  "source": "import",
  "first_seen": "2026-03-22T14:30:00Z",
  "created_at": "2026-03-22T14:30:00Z",
  "updated_at": "2026-03-22T14:30:00Z",
  "proposed_match": null
}
```

The `first_seen` field records the timestamp of the earliest ingest or import
that introduced this entity. It is never overwritten on subsequent updates.
This provides provenance for cleanup decisions.

The `proposed_match` field is non-null only for auto-created entities where LLM
soft match proposed a candidate. Value is the entity key of the proposed match
(e.g., `"jane_smith"`). Set to `null` on confirmation or rejection.

### 7.4 Type-specific fields

| Type | Additional fields |
|------|------------------|
| `person` | `title`, `department` (→ department key), `reports_to` (→ person key), `client` |
| `department` | `head` (→ person key), `client` |
| `system` | `owner_dept` (→ department key), `status` |
| `process` | `owner_dept` (→ department key), `status` |

Cross-references use entity keys (slugified canonical names within the same
type namespace). Unresolvable references are stored as-is and flagged in
`folio entities` output.

### 7.5 Entity key derivation and slug collision policy

```
slug("Jane Smith") → "jane_smith"
slug("SAP ERP") → "sap_erp"
slug("Incident Triage") → "incident_triage"
```

Rule: lowercase, replace non-alphanumeric runs with `_`, strip leading/trailing
underscores.

**Slug collision policy:** If two distinct canonical names produce the same slug
within a type namespace (e.g., `"Jane Smith"` and `"Jane  Smith"`), the second
write is **rejected with an error** identifying the existing entity. The user
must disambiguate canonical names before retrying. This applies to both import
and auto-creation. During import, the colliding row is skipped with a warning.
During ingest auto-creation, the extracted name is left as an unresolved
wikilink with a warning logged.

### 7.6 `source` field values

| Value | Meaning |
|-------|---------|
| `"import"` | Created via `folio entities import` |
| `"extracted"` | Auto-created during ingest (not in registry at extraction time) |
| `"manual"` | Created by direct edit of `entities.json` |

### 7.7 `needs_confirmation` semantics

- `false` — entity is confirmed (imported, manually verified, or explicitly
  confirmed via CLI).
- `true` — entity was auto-created or LLM-proposed and awaits human review.
- **Unconfirmed entities do NOT participate in resolution.** The resolution
  algorithm (§10.2) only matches against entities where
  `needs_confirmation == false`. This prevents junk entities from reinforcing
  themselves across ingests.
- Unconfirmed entities are visible in `folio entities` and
  `folio entities --unconfirmed` for review.
- Unconfirmed entities can be confirmed (`folio entities confirm`) or rejected
  (`folio entities reject`).

### 7.8 Durability model

`entities.json` follows the same durability contract as `registry.json`:

1. **Atomic writes with advisory locking:** All writes use the same
   `_atomic_write_json()` pattern shipped in `folio/tracking/registry.py`:
   acquire an exclusive advisory lock via `fcntl.flock` on a `.lock` file,
   write to a `.tmp` file, `os.rename` into place, release lock. This
   prevents concurrent writers from racing and matches the shipped
   `registry.json` write-safety model exactly.
2. **Write failure cleanup:** If the write fails (disk full, read-only
   filesystem), the `.tmp` file is cleaned up and an `OSError` is raised with
   a diagnostic message. The `.lock` file is always released.
3. **Bootstrap:** If `entities.json` does not exist, the first write creates it
   with `_schema_version: 1` and an empty entities dict. No bootstrap from
   library files — entity data does not exist in notes.
4. **Rebuild:** Unlike `registry.json` (which can be rebuilt from library
   frontmatter), `entities.json` cannot be rebuilt from notes because entities
   are not stored in frontmatter. `entities.json` is the sole source of truth
   for entity data. It should be committed to version control.
5. **Schema migration:** If `_schema_version` is less than the current code's
   expected version, apply forward migrations. If greater, refuse to load and
   emit an error asking the user to upgrade.

### 7.9 Corruption recovery

Because `entities.json` is human-editable (and manual editing is part of the
v0.5.1 workflow), parse failures are expected:

1. **JSON parse error:** Emit a clear error with the parse exception message
   and the file path. Do not attempt to auto-repair. Suggest the user fix the
   JSON syntax or restore from version control.
2. **Schema validation error** (missing `_schema_version`, unknown type
   namespace, missing required fields on an entry): Emit a warning per invalid
   entry. Load all valid entries and skip invalid ones. Report skipped entries
   to the user.
3. **Read-only fallback:** If `entities.json` exists but cannot be written
   (permissions), entity resolution still works for reading but auto-creation
   and import are disabled with a clear error.

## 8. CLI Contract

### 8.1 `folio entities`

List all entities in the library, grouped by type.

```
$ folio entities

People (12 total, 3 unconfirmed)
  Jane Smith          CTO, Engineering         confirmed
  John Doe            VP Operations            confirmed
  the intern          (unconfirmed)            unconfirmed
  ...

Departments (4 total, 0 unconfirmed)
  Engineering         Head: Jane Smith         confirmed
  Operations          Head: John Doe           confirmed
  ...

Systems (2 total, 1 unconfirmed)
  ServiceNow          Owner: Operations        confirmed
  Legacy CRM          (unconfirmed)            unconfirmed

Processes (1 total, 0 unconfirmed)
  Incident Triage     Owner: Operations        confirmed
```

**Flags:**
- `--type <type>` — filter to one entity type.
- `--unconfirmed` — show only `needs_confirmation: true` entities.
- `--json` — output raw JSON for scripting.

### 8.2 `folio entities show <name>`

Show detail for a single entity, matched case-insensitively against canonical
names and aliases.

**Disambiguation:** If the name matches multiple entities (e.g., an alias
shared across types, or a canonical name in one type and an alias in another):
- List all matches with type and canonical name.
- Prompt the user to re-run with `--type <type>` to disambiguate.
- Do NOT silently pick one.

```
$ folio entities show "Operations"

Multiple matches for "Operations":
  1. Operations (department) — canonical name
  2. Operations (process) — canonical name

Use --type to disambiguate:
  folio entities show "Operations" --type department
```

When a single match is found:

```
$ folio entities show "Jane Smith"

Jane Smith (person) confirmed
  Title:       Chief Technology Officer
  Department:  Engineering
  Reports to:  CEO
  Aliases:     Jane, the CTO, J. Smith
  Client:      AcmeCorp
  Source:       import
  First seen:  2026-03-22
  Created:     2026-03-22
  Mentioned in: 7 interaction notes
```

**"Mentioned in" semantics (S2):** Counts the number of library markdown files
under `library_root` whose body contains the wikilink `[[<canonical_name>]]`.
Alias wikilinks are NOT counted (they would have been canonicalized during
ingest). This is a read-time `grep -rl` equivalent, not a stored value. For
libraries with >500 notes, this scan may be slow; a future optimization can
cache counts, but v0.5.1 uses the simple scan.

### 8.3 `folio entities import <csv>`

Bulk import entities from a CSV file.

```
$ folio entities import org_chart.csv
Imported 45 people, 8 departments.
Created 3 new departments from unique department values.
Skipped 2 duplicate people (existing, no changes).
Updated 1 person (Jane Smith: added title).
⚠ 1 warning: unrecognized column "location" (ignored)
```

See §9 for the full import contract.

### 8.4 `folio entities confirm <name>`

Confirm a `needs_confirmation: true` entity.

```
$ folio entities confirm "the intern"
Confirmed: the intern (person)
```

**Disambiguation:** Same rules as `show` — if the name matches multiple
entities, require `--type` flag.

**If already confirmed:** No-op with message: "Already confirmed: the intern
(person)".

**Proposed match handling:** If the entity has a `proposed_match`, confirming
it means accepting the entity as-is (a distinct entity from the proposed
match). To accept the proposed match instead, reject this entity and manually
add the extracted name as an alias on the proposed match entity.

### 8.5 `folio entities reject <name>`

Reject an unconfirmed entity, removing it from the registry.

```
$ folio entities reject "Jnae Smith"
Rejected and removed: Jnae Smith (person)
```

**Behavior:**
- Removes the entity entry from `entities.json`.
- Only works on entities with `needs_confirmation: true`. Attempting to reject
  a confirmed entity emits an error: "Cannot reject confirmed entity. Edit
  entities.json directly to remove confirmed entities."
- Disambiguation via `--type` if needed.

**Effect on existing notes:** Rejection does not rewrite notes. Existing
`[[Jnae Smith]]` wikilinks remain in previously ingested notes. On re-ingest,
the name will be re-extracted and (if still not in the registry) auto-created
again. To prevent recurrence, the user should fix the source transcript or add
the correct name as an alias on the intended entity.

### 8.6 Commands explicitly NOT in v0.5.1

- `folio entities create` — use import or let ingest auto-create.
- `folio entities edit` — edit `entities.json` directly.
- `folio entities merge` — not part of v0.5.1. It later shipped as Tier 4
  graph-ops identity hygiene and should not be read as a current product gap
  from this historical slice.
- `folio entities delete` — use `reject` for unconfirmed; edit file directly
  for confirmed.

## 9. Import Contract

### 9.1 CSV format for org chart import

**Required columns:**

| Column | Description |
|--------|-------------|
| `name` | Full name of the person. Becomes `canonical_name`. |

**Optional columns:**

| Column | Description |
|--------|-------------|
| `title` | Job title (e.g., "Chief Technology Officer") |
| `department` | Department name. Auto-creates department entity if not in registry. |
| `reports_to` | Name of the person this person reports to. Must match another row's `name` or an existing person entity. |
| `aliases` | Semicolon-separated list of alternate names (e.g., `Jane;the CTO;J. Smith`) |
| `client` | Client code/name |

**Unrecognized columns (S3):** Emit a warning listing each unrecognized column
name. Continue import — do not fail. This prevents silent data loss when a user
expects a column to be processed.

### 9.2 Normalization and encoding rules

- Leading/trailing whitespace stripped from all values.
- Empty cells treated as absent (not empty string).
- Names are stored with original case; matching is case-insensitive.
- Alias list delimiter is semicolon (`;`), not comma (to avoid CSV escaping
  issues).

**Encoding (B3):** The CSV parser accepts UTF-8 (with or without BOM). A
leading UTF-8 BOM (`\xef\xbb\xbf`) is stripped before parsing. Non-UTF-8
files produce a clear error: "Cannot read <file>: expected UTF-8 encoding."

**Malformed rows (B3):** Rows with fewer columns than headers are padded with
empty values. Rows with more columns than headers are truncated to the header
count with a per-row warning. Completely unparseable rows (broken quoting) cause
the import to abort with an error identifying the line number.

### 9.3 Duplicate handling

- **Exact name match to existing confirmed entity:** Update fields that are
  currently `null`/absent; do NOT overwrite existing non-null values. Report as
  "updated" if any field changed, "skipped" if no changes.
- **Exact name match to existing unconfirmed/extracted entity (B3):** The
  import **upgrades** the entity: set `source: "import"`,
  `needs_confirmation: false`, and fill in any fields from the CSV row. This is
  the expected path when a user imports an org chart after ingest has already
  auto-created entities from transcripts.
- **Alias collision:** If an imported `name` matches an existing entity's
  alias, report a warning and skip the row. Do not silently merge.
- **Duplicate rows in CSV:** First occurrence wins; subsequent duplicates
  skipped with warning.

### 9.4 Department auto-creation

When a `department` column value does not match any existing department entity:
- Auto-create a department entity with `source: "import"`,
  `needs_confirmation: false`.
- The `head` field is left `null` in v1 — automatic head inference is deferred.

### 9.5 `reports_to` resolution

Import processes rows in **two passes** to eliminate row-order dependence (B3):

1. **Pass 1:** Create/update all person entities from all rows (without
   resolving `reports_to`).
2. **Pass 2:** Resolve `reports_to` for each row against the now-complete
   person entity set. Store as entity key (slug of referenced person's
   canonical name). If the referenced person is not found, store the raw name
   as-is and emit a warning.

This ensures that `reports_to` references work regardless of CSV row ordering.

### 9.6 Imported entities are confirmed by default

All entities created via `folio entities import` have
`needs_confirmation: false` and `source: "import"`. The assumption is that
org-chart data is authoritative.

### 9.7 Batch semantics and rollback (B3)

Import is **all-or-nothing at the file level:**

- The entire CSV is parsed and validated before any writes to `entities.json`.
- If parsing fails (encoding error, missing `name` column, unparseable row),
  no entities are created or modified.
- If parsing succeeds, all valid rows are applied and `entities.json` is
  written atomically.
- Per-row warnings (duplicates, alias collisions, unresolved `reports_to`) do
  NOT abort the import — they are collected and reported at the end.

### 9.8 Wikilink sanitization

Canonical names and aliases are sanitized before use in wikilinks:
- Characters not allowed in Obsidian wikilinks (`[`, `]`, `|`, `#`, `^`) are
  stripped.
- Leading/trailing whitespace is stripped.
- This sanitization is applied at import time and at auto-creation time.

## 10. Ingest-Time Resolution Contract

This is the most important design boundary in the spec.

### 10.1 What ingest continues to do (unchanged)

- LLM extracts entity names from source text.
- `_coerce_entities()` deduplicates case-insensitively within the note.
- Entities are rendered in the `## Entities Mentioned` section.
- Entities remain absent from YAML frontmatter (body-only).

### 10.2 What changes when entity registry exists

When `entities.json` exists in the library root, ingest adds a **resolution
pass** after extraction and before markdown rendering.

#### Resolution algorithm (per extracted entity name)

```
1. Normalize: strip whitespace, collapse internal spaces.
2. Determine type namespace from extraction category:
   people → person, departments → department,
   systems → system, processes → process.
3. Search CONFIRMED entities in the type namespace only (case-insensitive):
   a. Match canonical_name → RESOLVED.
   b. Match any alias → RESOLVED.
4. If exactly one match → use canonical_name for wikilink rendering.
5. If multiple matches within the type → AMBIGUOUS:
   a. Keep extracted name as-is in wikilink.
   b. Log warning with candidate list.
   c. Do NOT auto-create.
6. If no match → attempt LLM soft match (§10.3).
7. If soft match returns a proposed match → AUTO-CREATE with proposed_match:
   a. Auto-create entity with:
      - canonical_name = extracted name
      - type = mapped extraction category
      - needs_confirmation = true
      - source = "extracted"
      - proposed_match = <matched entity key>
      - aliases = []
   b. Render wikilink with extracted name (NOT the proposed match's
      canonical name — the match is unconfirmed).
8. If soft match returns no match → AUTO-CREATE without proposed_match:
   a. Auto-create entity with same fields as step 7, but
      proposed_match = null.
   b. Render wikilink with extracted name.
```

**Type-strict resolution (D7/B1):** Resolution never searches across type
boundaries. If the LLM extracts "Operations" as a system and the registry has
"Operations" only as a department, that is a miss — not a cross-type match.
This prevents silent wrong-type resolution.

**Confirmed-only resolution (D4/B4):** Only entities with
`needs_confirmation == false` participate in step 3. Auto-created entities
from previous ingests cannot drive resolution until a human confirms them.

### 10.3 LLM soft match

When step 3 above yields no match and the registry has confirmed entities of
the same type:

```
1. Collect confirmed entities of the same type (canonical names + aliases).
2. If zero candidates → skip soft match, go to step 8 above.
3. Cap candidates at 50. If more than 50, select the 50 most recently
   updated.
4. Present the LLM with:
   - the extracted name
   - the source sentence containing the mention
   - the candidate list (canonical name + aliases for each)
5. LLM responds with one of:
   - { "match": "<canonical_name>" } → proposed match
   - { "match": null } → no match
6. If proposed match: store proposed_match = entity key of the matched
   candidate on the auto-created entity.
7. Proposed matches are surfaced in `folio entities --unconfirmed` output
   with the proposed match shown.
```

**LLM soft match is bounded:**
- Runs only for unresolved entities (after exact + alias miss).
- Runs only when the registry has confirmed entities of the same type.
- Runs only once per unique extracted entity name per ingest invocation.
- Capped at 50 candidates per query.
- Uses a single LLM call per unresolved entity (not per mention).

**Confirmation/rejection flow for proposed matches:**
- `folio entities confirm <name>` — accepts the auto-created entity as a
  distinct entity. Clears `proposed_match`. Entity becomes confirmed and
  participates in future resolution.
- `folio entities reject <name>` — removes the auto-created entity. User
  should then add the extracted name as an alias on the correct entity (if the
  LLM proposal was correct but the entity should not exist separately).
- Neither action rewrites existing notes. Re-ingest to update wikilinks.

### 10.4 Post-resolution deduplication (S1)

After resolution, alias-to-canonical replacement can produce duplicate
wikilinks in the same entity list (e.g., "Jane" and "Jane Smith" both
resolve to `[[Jane Smith]]`). A **second deduplication pass** runs after
canonicalization:

- Within each entity type list, deduplicate by resolved canonical name
  (case-insensitive).
- Preserve the first occurrence's position.
- This ensures the `## Entities Mentioned` section has no duplicate wikilinks.

### 10.5 When entity registry does NOT exist

Behavior is identical to v0.5.0: extract, deduplicate, render as unresolved
wikilinks. No error, no warning. The entity system is additive.

### 10.6 Resolution does NOT modify frontmatter

Resolved entity names appear only in the markdown body (`## Entities
Mentioned` section). Frontmatter remains unchanged. This preserves the v0.5.0
contract that entities are body-only.

### 10.7 Re-ingest semantics (B6)

When re-ingesting an existing source (matched by `source_hash` or canonical
source path via `folio ingest --source <transcript>`):

**What is preserved:**
- `id` — reused from existing note.
- `version` — incremented.
- `status`, `authority`, `curation_level` — preserved from existing frontmatter
  if previously promoted beyond L0.
- `client`, `engagement` — preserved from existing frontmatter.
- `participants` — preserved from existing frontmatter.

**What is overwritten:**
- LLM analysis results (summary, findings, quotes, entities) — re-extracted
  from source text using current LLM.
- `## Entities Mentioned` section — rebuilt with current resolution state.
- `review_status`, `review_flags`, `extraction_confidence` — recomputed from
  fresh analysis.
- `grounding_summary` — recomputed.

Re-ingest is a full LLM re-analysis. This is the correct behavior when the
source transcript may have changed and ensures the entity resolution reflects
the current registry state.

**No lighter-weight re-resolve path in v0.5.1.** A batch command that rewrites
entity sections across existing notes without re-running LLM analysis would
constitute retroactive note mutation, which the non-goals (§4), the
Implementation Roadmap, and the Ontology Architecture all defer to the enrich
tier. The v0.5.1 upgrade path after importing an org chart is to re-ingest the
transcripts that need resolution.

### 10.8 `--participants` remains orthogonal

The `--participants` CLI flag populates frontmatter `participants: [...]`.
These are user-declared, not LLM-extracted. Resolution does NOT apply to
participants. If a participant name happens to match a registry entity, that is
coincidental — no linking occurs in v0.5.1.

## 11. Human Confirmation / Ambiguity Handling

### 11.1 What "human confirmation" means in v0.5.1

Confirmation is **registry-flag-driven, not interactive**.

- Ingest never prompts for confirmation. It runs to completion and logs
  warnings for ambiguous/unresolved entities.
- `folio entities --unconfirmed` shows all entities awaiting review.
- `folio entities confirm <name>` sets `needs_confirmation: false` and clears
  `proposed_match`.
- `folio entities reject <name>` removes unconfirmed entities from the
  registry.
- Users may also edit `entities.json` directly.

### 11.2 What the user sees when ambiguity exists

During ingest:
```
⚠ Ambiguous entity: "Jane" matches Jane Smith (person), Jane Doe (person)
  → Keeping unresolved wikilink: [[Jane]]
```

After ingest, `folio entities --unconfirmed`:
```
Unconfirmed entities (5):
  the intern         person      source: extracted     no proposed match
  Legacy CRM         system      source: extracted     no proposed match
  Jane               person      source: extracted     proposed: Jane Smith
  Jnae Smith         person      source: extracted     no proposed match
  ops team           department  source: extracted     proposed: Operations
```

### 11.3 Confirmation lifecycle (B4)

The lifecycle for auto-created entities is:

```
EXTRACTED (needs_confirmation: true)
  │
  ├── folio entities confirm → CONFIRMED (needs_confirmation: false)
  │     Now participates in future resolution.
  │
  ├── folio entities reject → REMOVED from entities.json
  │     Wikilinks in existing notes unchanged.
  │     Will be re-created on next ingest of same source.
  │
  └── (no action) → remains EXTRACTED
        Does NOT participate in resolution.
        Visible in --unconfirmed listing.
        May accumulate over time — periodic cleanup recommended.
```

**Cleanup guidance:** Users should periodically run
`folio entities --unconfirmed` and confirm or reject entries. Unconfirmed
entities are inert (they don't affect resolution or note content) but they do
accumulate. A future `folio entities prune` command may automate cleanup, but
it is not in scope for v0.5.1.

### 11.4 Confirmation does NOT trigger re-rendering

Confirming an entity updates `entities.json` only. Existing interaction notes
are not rewritten. To update wikilinks in existing notes, the user re-ingests
the source transcript.

### 11.5 Entity uncertainty and note trust (S6)

Entity resolution uncertainty does NOT affect note-level `review_status` or
`extraction_confidence`. These fields reflect the quality of the LLM's analysis
of the source text (findings, quotes, grounding), not entity identity.

**Rationale:** Entity resolution is a cross-note concern (the same entity
appears in many notes), while `review_status` is a per-note concern (how
trustworthy is this specific note's analysis). Coupling them would cause
cascading status changes across notes when an entity is confirmed or rejected.

If a note's entity section contains only unresolved wikilinks, that is visible
in the note body and in `folio entities --unconfirmed`. It does not constitute
a "flagged" review status because the note's analysis (summary, findings,
quotes) is unaffected by entity identity.

### 11.6 Deferred to later UX pass

- Interactive confirmation during ingest (`--interactive` flag).
- Bulk confirmation (`folio entities confirm --all`).
- Merge workflow for duplicate entities.
- Conflict resolution UI.
- `folio entities prune` for batch cleanup of stale unconfirmed entities.

## 12. Mixed-Library / Migration Behavior

### 12.1 Coexistence with existing commands

| Command | Impact |
|---------|--------|
| `folio status` | Add entity count summary line: "Entities: 45 (3 unconfirmed)" |
| `folio scan` | Unchanged — entities are not source files. |
| `folio refresh` | Unchanged — entities managed separately from source reconversion. |
| `folio promote` | Unchanged — entity confirmation is a separate workflow, not a promotion gate. |
| `folio ingest` | Extended with resolution pass (§10). |

### 12.2 Backward compatibility

- If `entities.json` does not exist, all commands behave identically to v0.5.0.
- `entities.json` is created on first `folio entities import` or first
  `folio ingest` that auto-creates an entity.
- No migration of existing `registry.json` is required.
- No migration of existing interaction notes is required.

### 12.3 Existing interaction notes

Already-created interaction notes contain unresolved wikilinks. The entity
system does NOT backfill or rewrite these notes. The upgrade path is:

1. Import org chart → populate `entities.json`.
2. Re-ingest transcripts that need resolution → updated notes get canonical
   wikilinks via current resolution state.
3. Optionally confirm auto-created entities from step 2.

### 12.4 `entities.json` and `.gitignore`

`entities.json` should be committed to the library repo (like `registry.json`).
It is part of the library's knowledge graph, not a cache. Because it cannot be
rebuilt from notes (§7.8 point 4), version control is the recovery mechanism
for data loss.

## 16. Challenge to Current Assumptions

### C1: "No fuzzy matching algorithm" — clarified

The roadmap constraint means "no Levenshtein / Jaro-Winkler / phonetic
matching algorithm." LLM soft match is the approved non-exact mechanism and is
now required for PR B (not optional). The constraint language should be updated
in the roadmap to say "no algorithmic fuzzy matching" to avoid confusion.

### C2: Entities absent from frontmatter may limit Dataview queries

The v0.5.0 decision to keep entities in the markdown body (not frontmatter)
means Dataview cannot query entities directly. The roadmap deliverable says
"Show me all interviews mentioning Engineering works via Obsidian search" —
which works via full-text search of `[[Engineering]]`. Dataview-based entity
queries are deferred to enrich or a fast follow-on.

`folio enrich` is the NEXT slice after entity system (Week 19–20). It will:
- Backfill entity extraction across existing evidence notes.
- Add relationship fields (`depends_on`, `draws_from`) to frontmatter.
- Potentially generate entity markdown stub files.
- Potentially add entity arrays to frontmatter for Dataview queries.
- Potentially add a `folio entities resolve` command for batch re-resolution of
  wikilinks in existing notes without full LLM re-analysis.

None of the above is in scope for v0.5.1. This spec does not pre-commit to any
enrich design. The entity registry and resolution contract defined here are
designed to be consumed by enrich without modification, but enrich may extend
the entity schema if needed.
