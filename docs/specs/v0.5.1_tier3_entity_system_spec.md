# v0.5.1 — Tier 3 Entity System Kickoff Spec

## 1. Overview

This spec defines the **minimum viable entity system** for folio.love: a
lightweight entity registry, a CLI for viewing and importing entities, and
ingest-time name resolution that turns unresolved `[[wikilinks]]` into
canonical graph nodes.

The entity system is the second Tier 3 slice (Week 16–18), immediately
following the `folio ingest` slice (v0.5.0, PR #32, merged to `main`).

**Output path:** `docs/specs/v0.5.1_tier3_entity_system_spec.md`

**Version label rationale:** v0.5.1 is correct. The entity system builds
directly on the v0.5.0 ingest baseline without introducing a new top-level
command family outside the `folio entities` subcommand. It is an incremental
capability extension, not a new product surface. PR A and PR B are both
covered under v0.5.1.

---

## 2. Why This Slice Comes Next

`folio ingest` (v0.5.0) ships entity extraction: it identifies people,
departments, systems, and processes from interview transcripts and renders them
as unresolved Obsidian wikilinks. But those wikilinks are names floating in
markdown — there is no canonical registry, no alias resolution, no import path
for known org data, and no way to determine whether `Jane`, `Jane Smith`, and
`the CTO` refer to the same person.

The entity system makes those names real. It is the smallest slice that
materially improves the shipped ingest baseline by:

1. Giving users a place to declare known entities before ingest
2. Resolving extracted names against that registry during ingest
3. Surfacing unconfirmed entities for human review

Without this slice, every interaction note is an island. With it, cross-note
queries like "show me all interviews mentioning Engineering" become reliable
because the names are canonical.

---

## 3. Goals

1. **Entity registry** — a machine-readable, human-editable store of known
   entities with canonical names, aliases, and type-specific metadata.
2. **CSV import** — bulk-load people and departments from an org chart export.
3. **CLI** — `folio entities` for listing, viewing, and confirming entities.
4. **Ingest-time resolution** — exact-match and alias-match resolution against
   the registry during `folio ingest`, replacing raw extracted names with
   canonical names in rendered wikilinks.
5. **LLM-proposed soft match** — when exact/alias match fails, the LLM may
   propose a match against the registry, flagged `needs_confirmation`.
6. **Human confirmation boundary** — clear mechanism for users to confirm or
   reject auto-created and LLM-proposed entities.

---

## 4. Non-Goals

These are explicitly out of scope for v0.5.1:

| Item | Why deferred |
|------|-------------|
| `folio enrich` | Separate Tier 3 slice (Week 19–20) |
| Retroactive rewrite of existing notes | Re-ingest is the mechanism; batch rewrite is an enrich concern |
| Relationship graph authoring | Beyond name resolution; requires enrichment pass |
| Semantic search over entities | Tier 4 capability |
| Recursive org traversal queries | Exceeds Dataview; deferred to Tier 4 per ontology §5 |
| Entity markdown stub files | Nice-to-have for Obsidian navigation; deferred unless trivial |
| `folio entities edit` / `folio entities merge` / `folio entities delete` | v1 is read + import + confirm; mutation is a future UX pass |
| Fuzzy matching algorithm | Explicitly excluded per roadmap constraint |
| Cross-library entity sharing | Libraries are independent; no federation in v1 |
| Entity confidence scoring | Unlike findings, entities have no per-item confidence in v0.5.0 |

## 5. Current Shipped Baseline

These facts were verified against the `main` branch as of PR #32 merge.

### 5.1 What exists

1. **`folio ingest`** creates ontology-native `type: interaction` documents.
2. **`interaction_analysis.py`** extracts four entity types via LLM:
   `people`, `departments`, `systems`, `processes`. Returns
   `InteractionAnalysisResult.entities: dict[str, list[str]]`.
3. **`_coerce_entities()`** deduplicates case-insensitively within a single
   note, preserving original case of first occurrence.
4. **`interaction_markdown.py`** renders entities as unresolved wikilinks in an
   `## Entities Mentioned` section with per-type subsections.
5. **`frontmatter.py` `generate_interaction()`** does NOT include entities in
   YAML frontmatter — entities live only in the markdown body.
6. **`--participants`** CLI flag stores user-provided participants in
   frontmatter, orthogonal to LLM-extracted entities.
7. **`registry.json`** tracks interaction entries alongside evidence entries
   using the same `RegistryEntry` schema. No entity fields exist in the
   registry.
8. **`folio status`** includes interaction documents in counts.
9. **`folio scan`** includes `.txt`/`.md` transcript sources.
10. **`folio refresh`** skips interaction entries (tells user to re-run ingest).
11. **`folio promote`** works on interactions; gates on `review_status`.

### 5.2 What does NOT exist

1. No entity registry file.
2. No `folio entities` CLI command.
3. No `folio entities import` command.
4. No entity resolution during ingest.
5. No alias system.
6. No `needs_confirmation` flag on any entity.
7. No entity fields in `RegistryEntry` or `registry.json`.

### 5.3 LLM prompt constraint

The current analysis prompt instructs:
> "Use unresolved entity names only; do not invent IDs."

This instruction will need to be extended (not replaced) when registry-backed
resolution is added.

## 6. Key Decisions

### D1: JSON registry, not markdown directory

**Decision:** Entities are stored in a single `entities.json` file alongside
`registry.json` in the library root.

**Rationale:**
- Consistent with the existing `registry.json` pattern.
- Entities are graph nodes, not documents (ontology §5). A markdown directory
  would conflate entities with Folio documents.
- Atomic read/write avoids partial-update bugs.
- Human-editable in any text editor.
- Obsidian wikilink resolution does NOT require target files — wikilinks still
  function as search anchors. Entity stub files can be generated later if
  needed.

**Trade-off acknowledged:** Without entity markdown files, `[[Jane Smith]]`
shows as an unresolved link in Obsidian's graph view. This is acceptable for
v1 because the primary value is cross-note search (`[[Jane Smith]]` is still
searchable across all files), not graph visualization.

### D2: Type-namespaced, slug-keyed

Entities are namespaced by type to prevent collisions (e.g., "Operations" as
both a department and a process). Keys are slugified canonical names.

### D3: Library-scoped, client-optional

One `entities.json` per library. The `client` field on each entity is optional.
No global entity federation in v1. A consultant who works across clients keeps
all entities in one library, tagged by client when relevant.

### D4: Auto-create with `needs_confirmation`

When ingest encounters an entity not in the registry:
- A new entity entry is auto-created with `needs_confirmation: true`.
- It is immediately usable for wikilink resolution.
- The user confirms or rejects via `folio entities confirm`.

This avoids blocking ingest on human review while keeping the confirmation
boundary visible.

### D5: LLM soft match is in scope but optional for PR B

The roadmap requires "LLM-proposed soft match with human confirmation." This
is defined in the spec and included in PR B scope, but marked as a stretch
goal. PR B must ship with at least exact + alias match. LLM soft match may
land in the same PR or a fast follow-on.

### D6: Re-ingest is the update mechanism

Existing interaction notes are NOT retroactively rewritten. To resolve
previously unresolved entities, the user re-ingests the source transcript.
Re-ingest detects the existing note via `source_hash` identity and updates it
in place.

## 7. Entity Registry Model

### 7.1 File location

```
<library_root>/entities.json
```

### 7.2 Top-level schema

```json
{
  "_schema_version": 1,
  "updated_at": "2026-03-22T14:30:00Z",
  "entities": {
    "person": { ... },
    "department": { ... },
    "system": { ... },
    "process": { ... }
  }
}
```

### 7.3 Entity entry schema (all types)

```json
{
  "canonical_name": "Jane Smith",
  "type": "person",
  "aliases": ["Jane", "the CTO", "J. Smith"],
  "needs_confirmation": false,
  "source": "import",
  "created_at": "2026-03-22T14:30:00Z",
  "updated_at": "2026-03-22T14:30:00Z"
}
```

### 7.4 Type-specific fields

| Type | Additional fields |
|------|------------------|
| `person` | `title`, `department` (→ department key), `reports_to` (→ person key), `client` |
| `department` | `head` (→ person key), `client` |
| `system` | `owner_dept` (→ department key), `status` |
| `process` | `owner_dept` (→ department key), `status` |

Cross-references use entity keys (slugified canonical names within the same
type namespace). Unresolvable references are stored as-is and flagged in
`folio entities` output.

### 7.5 Entity key derivation

```
slug("Jane Smith") → "jane_smith"
slug("SAP ERP") → "sap_erp"
slug("Incident Triage") → "incident_triage"
```

Rule: lowercase, replace non-alphanumeric runs with `_`, strip leading/trailing
underscores.

### 7.6 `source` field values

| Value | Meaning |
|-------|---------|
| `"import"` | Created via `folio entities import` |
| `"extracted"` | Auto-created during ingest (not in registry at extraction time) |
| `"manual"` | Created by direct edit of `entities.json` |

### 7.7 `needs_confirmation` semantics

- `false` — entity is confirmed (imported or manually verified).
- `true` — entity was auto-created or LLM-proposed and awaits human review.
- Confirmation is not a gate: unconfirmed entities still participate in
  resolution. The flag is informational, surfaced in `folio entities` output.

## 8. CLI Contract

### 8.1 `folio entities`

List all entities in the library, grouped by type.

```
$ folio entities

People (12 total, 3 unconfirmed)
  Jane Smith          CTO, Engineering         ✓
  John Doe            VP Operations            ✓
  the intern          (unconfirmed)            ?
  ...

Departments (4 total, 0 unconfirmed)
  Engineering         Head: Jane Smith         ✓
  Operations          Head: John Doe           ✓
  ...

Systems (2 total, 1 unconfirmed)
  ServiceNow          Owner: Operations        ✓
  Legacy CRM          (unconfirmed)            ?

Processes (1 total, 0 unconfirmed)
  Incident Triage     Owner: Operations        ✓
```

**Flags:**
- `--type <type>` — filter to one entity type.
- `--unconfirmed` — show only `needs_confirmation: true` entities.
- `--json` — output raw JSON for scripting.

### 8.2 `folio entities show <name>`

Show detail for a single entity, matched case-insensitively against canonical
names and aliases.

```
$ folio entities show "Jane Smith"

Jane Smith (person) ✓
  Title:       Chief Technology Officer
  Department:  Engineering
  Reports to:  CEO
  Aliases:     Jane, the CTO, J. Smith
  Client:      AcmeCorp
  Source:       import
  Created:     2026-03-22
  Mentioned in: 7 interaction notes
```

The "Mentioned in" count is computed by scanning library markdown files for
`[[Jane Smith]]` (and alias wikilinks). This is a read-time computation, not
stored.

### 8.3 `folio entities import <csv>`

Bulk import entities from a CSV file.

```
$ folio entities import org_chart.csv
Imported 45 people, 8 departments.
Created 3 new departments from unique department values.
Skipped 2 duplicate people (existing, no changes).
Updated 1 person (Jane Smith: added title).
```

See §9 for the full import contract.

### 8.4 `folio entities confirm <name>`

Confirm a `needs_confirmation: true` entity.

```
$ folio entities confirm "the intern"
Confirmed: the intern (person)
```

If the name matches multiple entities across types, prompt for disambiguation
(or accept a `--type` flag).

### 8.5 Commands explicitly NOT in v0.5.1

- `folio entities create` — use import or let ingest auto-create.
- `folio entities edit` — edit `entities.json` directly.
- `folio entities merge` — manual merge is a future UX pass.
- `folio entities delete` — edit `entities.json` directly.

## 9. Import Contract

### 9.1 CSV format for org chart import

**Required columns:**

| Column | Description |
|--------|-------------|
| `name` | Full name of the person. Becomes `canonical_name`. |

**Optional columns:**

| Column | Description |
|--------|-------------|
| `title` | Job title (e.g., "Chief Technology Officer") |
| `department` | Department name. Auto-creates department entity if not in registry. |
| `reports_to` | Name of the person this person reports to. Must match another row's `name` or an existing person entity. |
| `aliases` | Semicolon-separated list of alternate names (e.g., `Jane;the CTO;J. Smith`) |
| `client` | Client code/name |

Unrecognized columns are silently ignored.

### 9.2 Normalization rules

- Leading/trailing whitespace stripped from all values.
- Empty cells treated as absent (not empty string).
- Names are stored with original case; matching is case-insensitive.
- Alias list delimiter is semicolon (`;`), not comma (to avoid CSV escaping
  issues).

### 9.3 Duplicate handling

- **Exact name match to existing entity:** update fields that are currently
  `null`/absent; do NOT overwrite existing non-null values. Report as "updated"
  if any field changed, "skipped" if no changes.
- **Alias collision:** if an imported name matches an existing entity's alias,
  report a warning and skip the row. Do not silently merge.
- **Duplicate rows in CSV:** first occurrence wins; subsequent duplicates
  skipped with warning.

### 9.4 Department auto-creation

When a `department` column value does not match any existing department entity:
- Auto-create a department entity with `source: "import"`,
  `needs_confirmation: false`.
- The `head` field is populated later if a person with matching department is
  the most senior (by `reports_to` chain). In v1, `head` is left `null` unless
  explicitly set — automatic head inference is deferred.

### 9.5 `reports_to` resolution

- Resolved against person entities in the registry (including entities created
  earlier in the same import batch).
- Stored as an entity key (slug of the referenced person's canonical name).
- If the referenced person is not found, store the raw name as-is and emit a
  warning. The reference will resolve on future import or manual correction.

### 9.6 Imported entities are confirmed by default

All entities created via `folio entities import` have
`needs_confirmation: false` and `source: "import"`. The assumption is that
org-chart data is authoritative.

## 10. Ingest-Time Resolution Contract

This is the most important design boundary in the spec.

### 10.1 What ingest continues to do (unchanged)

- LLM extracts entity names from source text.
- `_coerce_entities()` deduplicates case-insensitively within the note.
- Entities are rendered in the `## Entities Mentioned` section.
- Entities remain absent from YAML frontmatter (body-only).

### 10.2 What changes when entity registry exists

When `entities.json` exists in the library root, ingest adds a **resolution
pass** after extraction and before markdown rendering.

#### Resolution algorithm (per extracted entity name)

```
1. Normalize: strip whitespace, collapse internal spaces.
2. Search registry (case-insensitive):
   a. Match canonical_name in the entity's type namespace → RESOLVED.
   b. Match any alias in the entity's type namespace → RESOLVED.
   c. Match canonical_name or alias in ANY type namespace → RESOLVED
      (cross-type fallback, same priority as within-type).
3. If exactly one match → use canonical_name for wikilink.
4. If multiple matches → AMBIGUOUS:
   a. Keep extracted name as-is in wikilink.
   b. Log warning with candidate list.
   c. Do NOT auto-create.
5. If no match → UNRESOLVED:
   a. Auto-create entity with:
      - canonical_name = extracted name
      - type = extraction category (people→person, departments→department, etc.)
      - needs_confirmation = true
      - source = "extracted"
      - aliases = []
   b. Render wikilink with extracted name (now the canonical name).
```

#### LLM soft match (stretch goal for PR B)

When step 2 above yields no match and the registry is non-empty for the
relevant type:

```
6. Present the LLM with:
   - the extracted name
   - the source sentence containing the mention
   - the list of registry entries for that type (canonical names + aliases)
7. LLM responds with one of:
   - "match:<entity_key>" → proposed match, stored as:
     proposed_match = entity_key on the auto-created entity
     needs_confirmation = true
   - "no_match" → proceed with step 5 (auto-create, no proposed match)
8. Proposed matches are surfaced in `folio entities --unconfirmed` output.
```

LLM soft match is bounded: it runs only for unresolved entities, only when the
registry has entries of the same type, and only once per entity per ingest.

### 10.3 When entity registry does NOT exist

Behavior is identical to v0.5.0: extract, deduplicate, render as unresolved
wikilinks. No error, no warning. The entity system is additive.

### 10.4 Resolution does NOT modify frontmatter

Resolved entity names appear only in the markdown body (`## Entities
Mentioned` section). Frontmatter remains unchanged. This preserves the v0.5.0
contract that entities are body-only.

### 10.5 `--participants` remains orthogonal

The `--participants` CLI flag populates frontmatter `participants: [...]`.
These are user-declared, not LLM-extracted. Resolution does NOT apply to
participants. If a participant name happens to match a registry entity, that is
coincidental — no linking occurs in v0.5.1.

## 11. Human Confirmation / Ambiguity Handling

### 11.1 What "human confirmation" means in v0.5.1

Confirmation is **registry-flag-driven, not interactive**.

- Ingest never prompts for confirmation. It runs to completion and logs
  warnings for ambiguous/unresolved entities.
- `folio entities --unconfirmed` shows all entities awaiting review.
- `folio entities confirm <name>` sets `needs_confirmation: false`.
- Users may also edit `entities.json` directly.

### 11.2 What the user sees when ambiguity exists

During ingest:
```
⚠ Ambiguous entity: "Jane" matches Jane Smith (person), Jane Doe (person)
  → Keeping unresolved wikilink: [[Jane]]
```

After ingest, `folio entities --unconfirmed`:
```
Unconfirmed entities (5):
  the intern         person      source: extracted     no proposed match
  Legacy CRM         system      source: extracted     no proposed match
  Jane               person      source: extracted     proposed: jane_smith
  ...
```

### 11.3 Confirmation does NOT trigger re-rendering

Confirming an entity updates `entities.json` only. Existing interaction notes
are not rewritten. To update wikilinks in existing notes, the user re-ingests
the source transcript.

### 11.4 Deferred to later UX pass

- Interactive confirmation during ingest (`--interactive` flag).
- Bulk confirmation (`folio entities confirm --all`).
- Merge workflow for duplicate entities.
- Conflict resolution UI.

## 12. Mixed-Library / Migration Behavior

### 12.1 Coexistence with existing commands

| Command | Impact |
|---------|--------|
| `folio status` | Add entity count summary line: "Entities: 45 (3 unconfirmed)" |
| `folio scan` | Unchanged — entities are not source files. |
| `folio refresh` | Unchanged — entities managed separately from source reconversion. |
| `folio promote` | Unchanged — entity confirmation is a separate workflow, not a promotion gate. |
| `folio ingest` | Extended with resolution pass (§10). |

### 12.2 Backward compatibility

- If `entities.json` does not exist, all commands behave identically to v0.5.0.
- `entities.json` is created on first `folio entities import` or first
  `folio ingest` that auto-creates an entity.
- No migration of existing `registry.json` is required.
- No migration of existing interaction notes is required.

### 12.3 Existing interaction notes

Already-created interaction notes contain unresolved wikilinks. The entity
system does NOT backfill or rewrite these notes. The upgrade path is:

1. Import org chart → populate `entities.json`.
2. Re-ingest transcripts that need resolution → updated notes get canonical
   wikilinks.
3. Optionally confirm auto-created entities from step 2.

This is deliberate: batch rewrite of existing notes is an `enrich`-tier
operation. The entity system provides the registry; re-ingest provides the
update mechanism.

### 12.4 `entities.json` and `.gitignore`

`entities.json` should be committed to the library repo (like `registry.json`).
It is part of the library's knowledge graph, not a cache.

## 14. Acceptance Criteria

### PR A: Entity registry + CLI + import

- [ ] `entities.json` created with `_schema_version: 1` on first entity write.
- [ ] All four entity types supported: person, department, system, process.
- [ ] Entity keys are deterministic slugs of canonical names.
- [ ] Aliases stored and used for lookup.
- [ ] `folio entities` lists entities grouped by type with confirmation status.
- [ ] `folio entities --type <t>` filters by type.

## 15. Open Questions for Reviewers

### Q1: Entity type namespace for resolution lookup

The resolution algorithm (§10.2) searches within the extraction category first
(e.g., a name extracted as "people" searches person entities), then falls back
to cross-type search. Should the cross-type fallback exist at all? It adds
complexity but handles cases where the LLM miscategorizes an entity.

**Recommendation:** Include cross-type fallback. The LLM sometimes extracts a
system name under departments or vice versa. Removing fallback would cause
avoidable misses.

### Q2: Entity auto-creation threshold

Should ingest auto-create entities for EVERY unresolved name, or only when a
minimum number of mentions exist? Auto-creating for every mention could flood
the registry with noise from one-off LLM hallucinations.

**Recommendation:** Auto-create all. The `needs_confirmation` flag is the noise
filter. Users confirm what's real and can ignore or delete the rest. Thresholds
add complexity without clear benefit in v1.

### Q3: `entities.json` write atomicity

`registry.json` uses atomic write (write to temp file, then rename). Should
`entities.json` follow the same pattern?

**Recommendation:** Yes. Use the same atomic-swap pattern from `registry.py`.

### Q4: Should the spec include entity markdown stubs?

Generating a minimal `_entities/Jane_Smith.md` file per entity would make
Obsidian wikilinks resolve to a clickable page. This is additive and relatively
simple.

**Recommendation:** Defer to a fast follow-on. The primary value of v0.5.1 is
the registry and resolution, not Obsidian navigation. Stub generation can be
added without schema changes.

### Q5: LLM soft match token budget

The soft match step (§10.2 step 6–8) sends the full registry of a given type
to the LLM. For large registries (100+ entities), this could be expensive.
Should there be a cap?

**Recommendation:** Cap at 50 candidates per type. If the registry exceeds 50
entities of one type, use only the 50 most recently updated. Document the cap
in the implementation prompt.

### Q6: Version label

The prompt suggested v0.5.1. This is correct for an incremental slice building
on v0.5.0 ingest. An alternative would be v0.6.0 to signal a new capability
surface (`folio entities`). The author recommends v0.5.1 because the entity
system is architecturally an extension of ingest, not a standalone product
surface.

**Decision requested from reviewers:** Confirm v0.5.1 or override to v0.6.0.

## 16. Challenge to Current Assumptions

### C1: "No fuzzy matching algorithm" may be too restrictive

The roadmap says v1 resolution is "exact match + LLM-proposed soft match with
human confirmation. No fuzzy matching algorithm." This is sound for v1. But the
LLM soft match IS a form of fuzzy matching — it's just that the fuzzy logic
lives in the LLM rather than a string-distance algorithm. The constraint should
be read as "no Levenshtein / Jaro-Winkler / phonetic matching algorithm," not
"no non-exact matching whatsoever."

**Recommendation:** Clarify the constraint language in the roadmap to say
"no algorithmic fuzzy matching" to avoid confusion with LLM-based soft match.

### C2: Entities absent from frontmatter may limit Dataview queries

The v0.5.0 decision to keep entities in the markdown body (not frontmatter)
means Dataview cannot query entities directly. The roadmap deliverable says
"Show me all interviews mentioning Engineering works via Obsidian search" —
which works via full-text search of `[[Engineering]]`. But Dataview-based
entity queries (e.g., `TABLE WHERE contains(entities, "Engineering")`) are not
possible without frontmatter fields.

**Recommendation:** Accept this for v0.5.1. Full-text search for `[[canonical
name]]` is sufficient for the stated deliverable. Adding entity arrays to
frontmatter is a natural extension in the enrich slice or a fast follow-on.
