# v0.6.4 Trust-Gated Surfacing for `folio links review`

## 1. Overview

This spec implements Slice 4 of the Tier 4 Shipping Plan
(`tier4_discovery_proposal_layer_spec.md` §15.4). It applies the §11
trust-gated surfacing rules to the `folio links review` surface AND its
companion commands `folio links status`, `folio links confirm`,
`folio links reject`, `folio links confirm-doc`, `folio links reject-doc`.

(The parent spec §15.4 uses the phrase "re-apply" because §11 already governs
`folio digest`; this is the first application to the `folio links` command
family, which had no trust-gating prior to v0.6.4.)

**Core behavior:** Proposals whose source document or target document has
`review_status: flagged` are excluded from default surfaces. Operators can
override with `--include-flagged`, which surfaces flagged-input proposals
with explicit trust-posture annotation. All surfaces must disclose flagged
exclusions -- no silent empty results.

**§11 rules implemented:**

| Rule | Description | Implementation |
|------|-------------|----------------|
| 1 | `review_status: flagged` inputs excluded by default | Filter in `collect_pending_relationship_proposals`, applied transitively to status / confirm / reject surfaces |
| 2 | Flagged-input proposals surface trust posture explicitly when included | `flagged_inputs` field on `RelationshipProposalView` |
| 3 | Committed surfaces provide explicit `--include-flagged` override | Flag on `review`, `status`, `confirm`, `reject`, `confirm-doc`, `reject-doc` |
| 4 | Surfaces disclose excluded flagged-input counts | Disclosure line on every surface |
| 5 | No surface may silently report empty when flagged inputs are the real cause | Silent-empty protection on review, status, confirm-doc, reject-doc |
| 6 | `extraction_confidence` MUST NOT be used as a hard exclusion filter criterion | No code path reads `extraction_confidence` for filtering; regression test enforces |

**Not in scope:** `folio digest` (command does not exist; separate feature).
`folio graph doctor` (informational diagnostic, not a review surface).
`folio enrich` (emission-time, not a query-time surface).

## 3. Data Model Changes

### 3.1 `RelationshipProposalView` -- add `flagged_inputs` field

```python
from dataclasses import dataclass, field  # field import added

@dataclass
class RelationshipProposalView:
    source_id: str
    source_path: Path
    source_markdown_path: str
    producer: str
    proposal: RelationshipProposal
    revived: bool = False
    flagged_inputs: list[str] = field(default_factory=list)  # ["source"|"target"] or both
```

**Semantics:** Empty when neither source nor target is flagged. Populated with
`"source"` and/or `"target"` when the corresponding document has
`review_status: flagged`.

### 3.2 New `SuppressionCounts` structure

```python
@dataclass
class SuppressionCounts:
    rejection_memory: dict[str, int] = field(default_factory=dict)  # keyed by producer
    flagged_input: int = 0

    def total(self) -> int:
        return sum(self.rejection_memory.values()) + self.flagged_input
```

**Rationale (CB-3 resolution):** The v0.6.0 / v0.6.3 shape `dict[str, int]`
conflates per-producer rejection-memory counts with trust-gate counts.
Separating them prevents the producer-name collision bug and lets renderers
display the two reasons independently.

### 3.3 Backwards compatibility

Existing callers of `collect_pending_relationship_proposals` destructure the
return as `views, counts` and either ignore `counts` or sum its values.
Under v1.1:
- `views` is unchanged.
- `counts` becomes a `SuppressionCounts` dataclass. Callers that did
  `counts.values()` must migrate to `counts.rejection_memory.values()`.
  Callers that did `sum(counts.values())` migrate to `counts.total()` (for a
  grand total) or `sum(counts.rejection_memory.values())` (for rejection-only).
  The migration is straightforward and covered by tests.

## 4. Function Specification

### 4.1 `collect_pending_relationship_proposals` -- signature change

```python
def collect_pending_relationship_proposals(
    config: FolioConfig,
    *,
    scope: Optional[str] = None,
    doc_id: Optional[str] = None,
    target_id: Optional[str] = None,
    include_flagged: bool = False,
) -> tuple[list[RelationshipProposalView], SuppressionCounts]:
```

New keyword-only parameter `include_flagged` (default `False`).
Return type changes to `tuple[list[RelationshipProposalView], SuppressionCounts]`.

### 4.2 Flagged-input detection

For each proposal:

1. **Source flagged:** `_is_flagged(fm.get("review_status"))`.
   `fm` is the source document frontmatter, already loaded.

2. **Target flagged:** read target frontmatter from disk, cache per-call.

   ```python
   def _resolve_target_flagged(
       target_id: str,
       registry_data: dict,
       library_root: Path,
       cache: dict[str, bool],
   ) -> bool:
       if target_id in cache:
           return cache[target_id]
       target_entry = registry_data.get("decks", {}).get(target_id)
       if not target_entry:
           cache[target_id] = False  # Unknown target — can't determine trust
           return False
       target_path = library_root / target_entry.get("markdown_path", "")
       target_fm = registry_mod._read_frontmatter(target_path)
       flagged = bool(target_fm and _is_flagged(target_fm.get("review_status")))
       cache[target_id] = flagged
       return flagged
   ```

   **Rationale (CB-1 resolution):** The registry dict's `review_status` is a
   sync-time snapshot and can be stale. Frontmatter is authoritative. The
   per-call cache keyed by target_id prevents re-reading the same target when
   multiple proposals reference it. File I/O is bounded at O(unique_targets).

### 4.3 `_is_flagged` helper (SF-E resolution)

```python
def _is_flagged(value) -> bool:
    """Normalize review_status to a flagged boolean.

    Conservative policy:
      - None or missing -> False (clean fail-open for unset)
      - String exactly "flagged" (after strip + lower) -> True
      - Other string values -> False (unknown status, not flagged)
      - Non-string values (list, dict, bool, int) -> False (treated as
        malformed; caller may emit a diagnostic if desired)
    """
    if value is None:
        return False
    if not isinstance(value, str):
        return False
    return value.strip().lower() == "flagged"
```

Non-string / unexpected values are treated as NOT flagged to preserve
fail-open behavior. A diagnostic warning MAY be emitted in future work but
is out of scope for this slice -- treating malformed values as flagged would
make unrelated frontmatter errors gate reviews, which is a separate policy
question.

### 4.4 Filter + annotate logic

Ordering inside the per-proposal loop:

1. Lifecycle-state check (existing): skip if not `"queued"`.
2. target_id filter (existing).
3. Rejection-memory check (existing): if matched, bump
   `counts.rejection_memory[producer]`, skip.
4. **Flagged-input check (new):**
   - Compute `flagged_list`.
   - If non-empty and `not include_flagged`: bump `counts.flagged_input`, skip.
   - If non-empty and `include_flagged`: set `view.flagged_inputs`, append.

Rejection-memory takes precedence over flagged-input as a suppression reason
(an explicit operator rejection is a stronger signal than a trust gate).

### 4.5 `relationship_status_summary` -- signature change (CB-2)

```python
@dataclass
class RelationshipStatusRow:
    source_id: str
    pending: int
    confirmed: int
    flagged_excluded: int = 0  # per-source flagged-input suppression count

def relationship_status_summary(
    config: FolioConfig,
    *,
    scope: Optional[str] = None,
    include_flagged: bool = False,
) -> tuple[list[RelationshipStatusRow], int]:
    """Returns rows and total flagged-excluded proposals across all sources."""
```

The function runs `collect_pending_relationship_proposals` once with
`include_flagged=True` to count per-source flagged exclusions, then again
with `include_flagged=False` for reviewable pending counts. OR, preferably,
it performs a single pass that tracks both. Implementation detail: single-pass
is cleaner and cheaper.

Single-pass implementation sketch:

```python
views_all, _ = collect_pending_relationship_proposals(
    config, scope=scope, include_flagged=True
)
# Partition by source_id; count flagged vs. reviewable.
```

Rows are emitted whenever `pending`, `confirmed`, OR `flagged_excluded` is
nonzero. A source with only flagged exclusions still appears.

### 4.6 `confirm_doc` / `reject_doc` -- signature change (CB-2)

```python
def confirm_doc(
    config: FolioConfig,
    doc_id: str,
    *,
    include_flagged: bool = False,
) -> tuple[int, int]:
    """Returns (acted_count, flagged_excluded_count)."""

def reject_doc(
    config: FolioConfig,
    doc_id: str,
    *,
    include_flagged: bool = False,
) -> tuple[int, int]:
    """Returns (acted_count, flagged_excluded_count)."""
```

Both functions pass `include_flagged` through to
`collect_pending_relationship_proposals`, then act on the resulting views.
The `flagged_excluded` return is taken from `counts.flagged_input` (zero when
`include_flagged=True`).

### 4.7 `_find_pending_view` / `confirm_proposal` / `reject_proposal` (CB-4)

`_find_pending_view` gains `include_flagged: bool = False`. `confirm_proposal`
and `reject_proposal` pass it through. When called with `include_flagged=True`,
the lookup searches the unfiltered collection, so an operator who saw a
flagged proposal via `folio links review --include-flagged` can act on it
with `folio links confirm <id> --include-flagged`.

## 5. CLI Specification

### 5.1 `--include-flagged` on all mutating and surfacing commands

Added to:
- `folio links review` (surface)
- `folio links status` (surface)
- `folio links confirm` (mutating)
- `folio links reject` (mutating)
- `folio links confirm-doc` (mutating)
- `folio links reject-doc` (mutating)

All commands pass the flag through to the underlying function.

### 5.2 Trust-posture tag in `links review` output

When `view.flagged_inputs` is non-empty, append to the proposal line:

| flagged_inputs | Tag |
|----------------|-----|
| `["source"]` | `(flagged: source)` |
| `["target"]` | `(flagged: target)` |
| `["source", "target"]` | `(flagged: source, target)` |

If the proposal is both revived AND flagged, tags render in this order:

```
[<proposal_id>] source -> target (relation, confidence, producer) (revived - basis changed) (flagged: source, target)
```

### 5.3 Flagged-input disclosure in `links review` (SF-G)

After any revived / rejection-memory disclosure, print the flagged breakdown:

```python
if counts.flagged_input:
    click.echo(
        f"{counts.flagged_input} proposal(s) excluded (flagged inputs). "
        "Use --include-flagged to review."
    )
```

(The simpler message above is acceptable for v1.1. A full source/target
breakdown is a should-fix but not a blocker -- operators can read per-proposal
tags after running `--include-flagged`.)

### 5.4 Silent-empty protection in `links review` (SF-H)

When `len(views) == 0` AND `counts.flagged_input > 0`:

```
Pending (0 items -- N excluded because source or target has review_status: flagged; use --include-flagged to review)
```

Replaces the default `Pending (0 item(s))` header.

### 5.5 `links status` disclosure (CB-2)

The status table adds a `Flagged Excluded` column. Footer includes total
flagged exclusions. When ALL sources are filtered out:

```
No document-level relationship data to review. N proposal(s) excluded due to flagged inputs; use folio links status --include-flagged to inspect.
```

### 5.6 `confirm-doc` / `reject-doc` disclosure (CB-2, DC-2)

Whenever `flagged_excluded > 0 and not include_flagged` (regardless of
`acted`), emit a diagnostic after the success line:

```
Confirmed K proposal(s).
N proposal(s) excluded from <doc_id> (flagged inputs); use --include-flagged to act on them.
```

This covers both the all-flagged case (`K == 0`) and the mixed case
(`K > 0` with `N > 0`). The mixed case is the dangerous one: without the
disclosure, the operator sees a clean success and has no signal that flagged
proposals were silently skipped.

Non-zero exit code for this diagnostic is NOT required.

### 5.7 `confirm` / `reject` consent path (CB-4)

When an operator supplies `--include-flagged`, the lookup falls back to the
unfiltered collection. Without the flag, a flagged proposal ID returns the
existing `Unknown proposal_id` error -- closing the bypass surface and
preventing accidental confirmation.

## 6. Failure Policy

- Target frontmatter read failure (missing file, malformed YAML): treat as
  NOT flagged (cannot determine trust state). Fail-open consistent with
  registry-unknown-target behavior. Diagnostics logged via existing
  `registry_mod._read_frontmatter` return (returns None on malformed).

- `review_status` malformed (non-string, weird casing): `_is_flagged` returns
  False. Not flagged.

- Target not in registry: `_resolve_target_flagged` returns False. Not flagged.
  Consistent with a target document being unknown -- the proposal itself
  should probably be rejected, but that is separate from trust gating.

No new exception classes are introduced. No new write paths are introduced.
