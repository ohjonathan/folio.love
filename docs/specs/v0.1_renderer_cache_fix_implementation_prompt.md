## Task Context

### What to Build

Build the approved renderer reliability mitigation in two parts:

- `Phase 0`: improve the automated PPTX path on managed macOS by removing the PowerPoint temp-dir sandbox write problem, adding batch observability, and adding dedicated-session restart resilience for AppleScript fatigue.
- `Phase 1`: document and surface the existing PDF-first path as a mitigation workflow for files that remain unconvertible after `Phase 0`.

### What Is Broken

Tier 1 validation ran 50 real consulting decks through Folio:

- 32 succeeded
- 18 failed
- all 18 failures were in normalization (`PPTX -> PDF`)
- 0 silent failures
- 612 slides processed successfully
- 2,904 evidence claims extracted
- 93.4% average evidence validation rate

Observed failure modes from the validation run:

- `-9074` on 17 specific files, consistently
- `-1712` timeout / AppleScript unresponsiveness
- `-1728` intermittent AppleScript object-missing failure
- cumulative AppleScript fatigue after roughly 19-31 conversions
- Terminal permission/TCC differences between Cursor and Terminal.app
- managed-mac sandbox dialog when PowerPoint writes to an arbitrary temp dir
- LibreOffice blocked by MDM (`SIGKILL`, exit code 137)

### Tier 1 Boundary

Tier 1 requires fully automated conversion. Manual/operator-exported PDFs are mitigation-only and do **not** count toward the 50-deck Tier 1 gate.

### Cache Boundary

- Automated-PPTX reconversion cache persistence remains deferred to `Approach J` as a separate `v0.5.1` follow-on.
- The older validation report suggestion to use `text hash + slide position` as the primary key is superseded by the approved proposal and must not be implemented.

### Current Normalization Flow

`FolioConverter.convert()` currently does this:

1. Resolve `deck_dir` and markdown target.
2. Create a temporary directory with `tempfile.TemporaryDirectory()`.
3. Call:

```python
pdf_path = normalize.to_pdf(
    source_path,
    tmpdir,
    timeout=self.config.conversion.libreoffice_timeout,
    renderer=self.config.conversion.pptx_renderer,
)
```

4. Call `images.extract_with_metadata(pdf_path, deck_dir, ...)`, which writes persistent slide PNGs into `deck_dir/slides` via an atomic temp-dir swap.
5. Call `text.extract_structured(source_path)`, which reads:
   - PPTX/PPT directly via MarkItDown
   - PDF directly via pdfplumber
6. Reconcile slide text count against authoritative image count.
7. Run `analysis.analyze_slides()` and optional `analysis.analyze_slides_deep()` using `deck_dir` as the persistent cache directory.

`normalize.to_pdf()` currently does this:

1. Validate the source file.
2. If suffix is `.pdf`, copy it directly into `output_dir` and return the copied PDF path.
3. If suffix is `.pptx` or `.ppt`, select a renderer with `_select_renderer()`.
4. Compute a single `expected_pdf = output_dir / f"{source_path.stem}.pdf"`.
5. In `auto`, prefer LibreOffice and fall back to PowerPoint if LibreOffice exists but conversion fails.
6. PowerPoint conversion currently uses this AppleScript path:

```applescript
tell application "Microsoft PowerPoint"
    launch
    with timeout of <timeout> seconds
        open POSIX file "<source>"
        save presentation "<name>" in POSIX file "<output>" as save as PDF
        close presentation "<name>" saving no
    end timeout
end tell
```

## Proposed Implementation

### Phase 0a: PowerPoint-Only Output Path Seam

Modify `normalize.to_pdf()` to take this exact signature:

```python
def to_pdf(
    source_path: Path,
    output_dir: Path,
    *,
    pptx_output_dir: Path | None = None,
    timeout: int = 60,
    renderer: str = "auto",
) -> Path:
```

Required behavior:

- Keep the direct PDF copy path unchanged. PDF input still copies to `output_dir`.
- Keep LibreOffice output behavior unchanged. LibreOffice still writes to `output_dir`.
- PowerPoint writes to `pptx_output_dir or output_dir`.
- In `auto`, if LibreOffice fails and PowerPoint is used as fallback, the fallback PowerPoint PDF must use the PowerPoint path, not the LibreOffice path.
- The final existence check must validate the path for the renderer that actually ran.
- `converter.py` must pass `pptx_output_dir=deck_dir`.
- After image extraction succeeds, `converter.py` must delete only the intermediate PowerPoint PDF written into `deck_dir`.
- Do not delete:
  - the original source PDF
  - the temp-dir PDF used by the PDF copy path
  - the temp-dir PDF used by LibreOffice

### Phase 0b: `open -a` Spike

Treat this as a spike, not a guaranteed production change.

Manually test `open -a "Microsoft PowerPoint"` plus export automation against 3-5 known `-9074` files from Terminal.app.

Success criteria:

- If `open -a` opens **and exports** 3 or more of the known failing files from Terminal.app, it is acceptable to replace the current `open POSIX file` step with a two-step PowerPoint runtime path.
- If `open -a` opens but export still fails, it fixes only TCC and must **not** replace the current runtime path.
- If `open -a` fails on the same files, treat the failures as file-intrinsic and leave the current runtime path in place.

Do not land an `open -a` production refactor unless the spike proves it resolves Terminal.app `-9074`, not just Cursor TCC.

### Phase 0c: Batch Observability

Add observable batch instrumentation only.

Required per-file data:

- file name
- renderer label available at the batch layer
- duration
- outcome bucket

Allowed outcome buckets:

- `success`
- `applescript_<code>` when an AppleScript code can be parsed from the exception text, such as `applescript_-9074`, `applescript_-1712`, or `applescript_-1728`
- `timeout`
- `unknown`

Rules:

- No direct `sandbox` outcome bucket
- No GUI-dialog detection claims
- No broad structured normalization return contract in this slice
- It is acceptable in this slice for the batch summary to use the effective renderer label available at the orchestration layer rather than widening the normalization API solely for telemetry

### Phase 0d: Dedicated-Session Restart Resilience

Implement restart behavior in `cli.py`, not `normalize.py`.

Approved operating assumption:

- `folio batch` runs in a dedicated PowerPoint session with no unrelated presentations open.
- In that dedicated session, Folio may own and restart the PowerPoint app process.

Required behavior:

- Count automated PPTX/PPT conversions only; do not count PDF files toward restart cadence.
- Preemptively restart PowerPoint every `N=15` automated PPTX/PPT conversions.
- Restart flow:
  - quit PowerPoint
  - wait 5 seconds
  - relaunch PowerPoint
- On unexpected `applescript_-9074` during batch:
  - restart PowerPoint
  - retry the same file once
  - if retry still fails, record it as failed and route it to manual/PDF mitigation handling
- Do not add restart behavior for the PDF mitigation path.
- If the dedicated-session invariant is explicitly disabled inside the batch helper logic, fall back to manual batch boundaries and skip app-level restart automation.

### Phase 1: PDF-First Mitigation

Do not build a new PDF pipeline. Use the existing one.

Required behavior:

- Keep `folio convert deck.pdf` as the direct mitigation path.
- Keep `folio batch <dir> --pattern "*.pdf"` as the batch mitigation path.
- Do not add Tier 1 counting language for operator-assisted PDFs except explicit exclusion.
- Add warnings/reporting only:
  - portrait PDF warning for likely notes-page exports
  - scanned-PDF warning for low-text PDFs
- Do not reject portrait PDFs by default.
- Do not add frontmatter fields for mitigation quality or warnings.
- Do not change source/version/frontmatter semantics for PDF input.

## Cache Map and Scenarios

### Cache Map

| Component | Current Key | Proposed Key | Why Changed |
|---|---|---|---|
| Pass 1 lookup | `_hash_image(slide_png)` | No change | Approved proposal keeps visual identity as cache identity |
| Pass 1 entry validation | `_text_hash(slide_text)` stored per entry | No change | Already implemented and must remain intact |
| Pass 2 lookup | `f"{image_hash}_deep"` | No change | Same reason |
| Pass 2 entry validation | `_text_hash(slide_text)` + `_pass1_context_hash(pass1_result)` | No change | Already implemented and must remain intact |
| Root invalidation | `_cache_version`, `_prompt_version`, `_model_version`, `_provider_version`, `_extraction_version` | No change | Existing cache invalidation remains correct |

Exact formulas already present in `analysis.py` and to be treated as unchanged:

- `image_hash = SHA256(slide_png_bytes)[:16]`
- `_text_hash = SHA256((slide_text.full_text or "").encode())[:16]`
- `_pass1_context_hash = SHA256(f"{slide_type}|{framework}|{key_data}|{main_insight}".encode())[:16]`
- `_prompt_version = SHA256(prompt_text.encode())[:8]`

### Scenario A: Same Manual-Exported PDF, Reconversion Without Changes

- Same PDF bytes
- Same slide PNG bytes after `images.extract_with_metadata()`
- Same `image_hash`
- Same `_text_hash`
- Same `_provider_version`, `_model_version`, `_prompt_version`, `_extraction_version`
- Result: cache hit

### Scenario B: Slide Content Edited, Then PDF Re-Exported

- Changed PDF page bytes
- Changed slide PNG bytes
- Changed `image_hash` on the edited slide
- Result: cache miss on the edited slide

### Scenario C: Provider or Model Changed

- Image bytes may be unchanged
- `image_hash` may be unchanged
- `_provider_version` or `_model_version` mismatch invalidates the cache file on load
- Result: cache miss even with identical slide bytes

### Explicit Non-Goal

Same PPTX reconverted through PowerPoint with no logical changes may still miss cache because PowerPoint PDF output is non-deterministic. This remains deferred to `Approach J` and is not part of this implementation.
