# Multi-Provider LLM Support Spec (v0.4.0, rev 2)

**Status:** Ready for re-review
**Type:** Feature addition
**Estimated effort:** 3-4 days
**Scope:** Product proposal plus implementation design for `folio convert`

---

## 1. Overview

Folio currently supports slide analysis through Anthropic only:

- `folio/pipeline/analysis.py` is hardcoded to the Anthropic SDK
- Credential lookup is hardcoded to `ANTHROPIC_API_KEY`
- Config exposes only one `llm.provider` / `llm.model`
- README and PRD describe Anthropic as the only supported provider

This spec expands Folio to support:

- Anthropic
- OpenAI
- Google Gemini

The proposal goes beyond "add more providers" and defines a complete runtime model:

- bring-your-own credentials
- named LLM profiles
- task-based routing
- optional transient fallback chains
- provider-safe cache scoping
- internal LLM execution metadata

This revision resolves the review board's main concern: the earlier draft described product behavior but did not define the technical design closely enough for implementation.

## 2. Key Decisions

### 2.1 Definitions

- **Provider** = external SDK/service (`anthropic`, `openai`, `google`)
- **Profile** = named config entry containing `provider + model + api_key_env`
- **Route** = task-to-profile mapping, optionally with fallbacks

These terms are not interchangeable in this spec.

### 2.2 Current runtime scope

v0.4.0 changes the runtime for `folio convert` only.

- Current implemented task: `convert`
- Reserved future tasks: `ingest`, `enrich`, `query_answer`

Reserved future task routes may exist in config, but v0.4.0 only executes `convert`.
Rev 1 used separate route names `convert_pass1` and `convert_pass2`; this revision intentionally collapses them into a single `convert` route because v0.4.0 requires both passes to use the same effective profile.

### 2.3 Pass-1 / Pass-2 profile policy

For v0.4.0, Pass 1 and Pass 2 must use the same effective profile.

Reason:

- Pass 2 consumes Pass 1 outputs
- cross-provider pass mixing adds response-shape and prompt-portability risk
- same-profile multi-pass is sufficient for the current product need

Cross-provider pass mixing is deferred to a later version after response normalization is proven across providers.

### 2.4 Structured output strategy

v0.4.0 chooses **JSON structured output** as the sole supported response contract.

The implementation must not rely on provider-specific prose formatting or Claude-specific field order. Each provider adapter must return normalized JSON payloads that are then converted into the existing `SlideAnalysis` shape.

### 2.5 Fallback policy

Fallback is:

- opt-in per route
- transient-only
- bounded to one retry on the active profile before moving to the next fallback profile

Fallback is not used for auth/config/request/policy failures.

### 2.6 Version semantics

Changing provider, profile, or model does **not** increment Folio document version.

Version history remains source-content-based per [FR-301](/Users/jonathanoh/Dev/folio.love/docs/product/02_Product_Requirements_Document.md#L162). LLM execution changes are recorded only in analysis cache metadata and frontmatter `_llm_metadata`.

## 3. Configuration Model

### 3.1 Target schema

```yaml
llm:
  profiles:
    high_quality_anthropic:
      provider: anthropic
      model: claude-sonnet-4-20250514
      api_key_env: ANTHROPIC_API_KEY

    backup_google:
      provider: google
      model: gemini-2.5-pro
      api_key_env: GEMINI_API_KEY

    fast_openai:
      provider: openai
      model: gpt-5-mini
      api_key_env: OPENAI_API_KEY

  routing:
    default:
      primary: high_quality_anthropic
      fallbacks: []

    convert:
      primary: high_quality_anthropic
      fallbacks: [backup_google]
```

Future routes such as `ingest`, `enrich`, and `query_answer` remain reserved for later releases and are intentionally omitted from the shipping v0.4.0 config example so the example does not imply runtime support.

### 3.2 Validation rules

Config validation happens in `FolioConfig.load()` / `FolioConfig.__post_init__()`.

Rules:

- `profiles` keys must match `^[a-z][a-z0-9_]*$`
- each profile must define `provider`, `model`, and `api_key_env`
- `provider` must be one of `anthropic`, `openai`, `google`
- `routing.<task>.primary` must reference an existing profile
- every fallback in `routing.<task>.fallbacks` must reference an existing profile
- undefined task routes fall back to `routing.default`
- missing `routing.default` is invalid

### 3.3 Secret handling

- API keys are supplied via environment variables only
- `folio.yaml` stores env-var names, never raw keys
- default env vars:
  - `ANTHROPIC_API_KEY`
  - `OPENAI_API_KEY`
  - `GEMINI_API_KEY`

### 3.4 Legacy schema support

Existing config remains valid:

```yaml
llm:
  provider: anthropic
  model: claude-sonnet-4-20250514
```

Legacy config must be loaded by this explicit migration path:

1. create synthetic profile `default_<provider>`
2. set `routing.default.primary = default_<provider>`
3. set `routing.convert.primary = default_<provider>`
4. set all fallback arrays to `[]`
5. derive `api_key_env` from provider default if omitted

### 3.5 CLI override semantics

Add:

```bash
folio convert deck.pptx --llm-profile fast_openai
folio batch ./materials --llm-profile high_quality_anthropic
```

Semantics:

- `--llm-profile X` applies to all LLM work in that command invocation
- for `convert`, both Pass 1 and Pass 2 use profile `X`
- `--llm-profile` disables task-specific routing and disables configured fallbacks for that invocation
- if `X` does not exist in config, fail fast before conversion starts

This keeps CLI override deterministic and easy to reason about.
The fallback disable is intentional: explicit CLI override is treated as an operator-control/debugging mode where deterministic single-provider behavior is preferable to automatic resilience.

## 5. Technical Design

### 5.1 Module layout

Refactor Anthropic-specific logic out of `folio/pipeline/analysis.py` into provider adapters.

Recommended layout:

- `folio/llm/types.py`
- `folio/llm/providers.py`
- `folio/llm/registry.py`
- `folio/pipeline/analysis.py` remains the orchestration layer

### 5.2 Core data types

```python
@dataclass
class ProviderInput:
    model: str
    prompt: str
    image_bytes: bytes | None
    media_type: str | None
    max_tokens: int


@dataclass
class ProviderOutput:
    text: str
    truncated: bool


@dataclass
class ErrorDisposition:
    kind: Literal["transient", "permanent"]
    message: str
    original: Exception | None = None
```

### 5.3 Provider protocol

```python
class AnalysisProvider(Protocol):
    provider_name: str

    def create_client(self, api_key: str) -> object: ...

    def analyze(self, client: object, request: ProviderInput) -> ProviderOutput: ...

    def classify_error(self, exc: Exception) -> ErrorDisposition: ...
```

Responsibilities:

- `analysis.py` builds prompts, manages retries, caching, validation, fallback, and parsing
- provider adapters build provider-specific request payloads and normalize raw SDK responses to plain text
- provider adapters own provider-specific image payload construction
- provider adapters own provider-specific exception classification
- `ProviderInput.prompt` is a fully rendered string built by the orchestrator before adapter invocation; provider-specific prompt variants, if ever needed later, still belong in the orchestrator layer
- `analyze()` is pass-agnostic and must serve both Pass 1 and Pass 2 invocations using the same method with different prompt strings

Type-safety note:

- `create_client() -> object` is a pragmatic protocol compromise
- each adapter must validate the incoming client instance type at the start of `analyze()` and raise a permanent configuration error if the wrong client type is supplied

Client lifetime:

- `create_client()` is called once per provider/profile per conversion invocation
- the created client is reused across all slides for that invocation
- if fallback activates, the fallback provider client is created once and then reused for the remaining slides

### 5.4 Provider registry / factory

`registry.py` exposes a simple mapping:

```python
PROVIDERS = {
    "anthropic": AnthropicAnalysisProvider(),
    "openai": OpenAIAnalysisProvider(),
    "google": GoogleAnalysisProvider(),
}
```

Selection flow:

1. resolve profile from config route
2. look up provider adapter by `profile.provider`
3. create client using `profile.api_key_env`
4. invoke adapter with `ProviderInput`
5. adapter returns normalized `ProviderOutput.text`

### 5.5 Response contract

All providers must be prompted to return JSON only.

Pass 1 schema:

```json
{
  "slide_type": "data",
  "framework": "tam-sam-som",
  "visual_description": "string",
  "key_data": "string",
  "main_insight": "string",
  "evidence": [
    {
      "claim": "string",
      "quote": "string",
      "element_type": "title",
      "confidence": "high"
    }
  ]
}
```

Pass 2 schema:

```json
{
  "slide_type_reassessment": "unchanged",
  "framework_reassessment": "unchanged",
  "evidence": [
    {
      "claim": "string",
      "quote": "string",
      "element_type": "body",
      "confidence": "medium"
    }
  ]
}
```

Rules:

- adapters use provider-native structured output / JSON mode when available
- if provider-native JSON mode is unavailable or unreliable, adapters request a single raw JSON object and the orchestrator uses `_extract_json(raw_text)` before parsing
- prose-mode parsing is out of scope for v0.4.0
- a response that cannot be parsed as valid JSON is treated as pending / discarded, not fuzzily guessed

### 5.5.1 Prompt migration from prose to JSON

The current `ANALYSIS_PROMPT` and `DEPTH_PROMPT` in `folio/pipeline/analysis.py` request prose-formatted output. v0.4.0 requires both prompts to be rewritten to request JSON matching the schemas above.

Implementation notes:

- changing the prompt automatically invalidates existing caches via `_prompt_version`
- this one-time cache wipe is expected and should be called out in release notes
- before shipping the JSON prompt, run a quality comparison on a representative 10-20 slide sample:
  - current prose prompt output
  - new JSON prompt output
  - compare framework detection, evidence quality, and overall slide usefulness

This quality comparison is required in the first implementation slice before the JSON prompt is considered production-ready.

### 5.6 JSON normalization rules

After JSON parse:

- normalize `slide_type` and `framework` to lowercase hyphenated strings
- normalize `element_type` to one of `title`, `body`, `note`
- normalize `confidence` to one of `high`, `medium`, `low`
- cap evidence items at **10** per slide
- reject Pass 1 responses with zero evidence items
- reject Pass 2 responses with zero evidence items
- when evidence count exceeds 10, keep the first 10 items in provider order and log a warning with the original count

### 5.7 Image payload abstraction

The caller passes:

- raw image bytes
- media type (`image/png`)

Each adapter constructs the correct payload:

- Anthropic: base64 `image` content block
- OpenAI: image input using the provider's multimodal request format
- Google: binary/image part using the provider's multimodal request format

This logic must not remain in `analysis.py`.

### 5.8 Parsing and validation flow

The orchestrator owns a small JSON extraction utility:

```python
def _extract_json(raw_text: str) -> str | None:
    """Return a JSON object string or None."""
```

Required behavior:

1. attempt `json.loads(raw_text)` directly
2. if that fails, strip one surrounding markdown code-fence pair plus optional `json` language tag and retry
3. if that still fails, return `None`

This utility is intentionally narrow. It does not regex-scrape arbitrary substrings that merely look like JSON.

Pass 1:

1. build prompt
2. provider adapter returns normalized JSON text
3. if `ProviderOutput.truncated` is true, log a truncation-specific warning and treat the slide as pending
4. call `_extract_json()`
5. parse extracted JSON if present
6. validate required keys
7. convert to `SlideAnalysis`
8. run evidence validation against extracted slide text

Pass 2:

1. build depth prompt from Pass 1 result
2. provider adapter returns normalized JSON text
3. if `ProviderOutput.truncated` is true, log a truncation-specific warning and treat the slide as pending
4. call `_extract_json()`
5. parse extracted JSON if present
6. validate reassessment keys + evidence
7. merge deduplicated evidence into Pass 1 result

### 5.8.1 `ProviderOutput.truncated` semantics

`ProviderOutput.truncated` is retained because it lets the orchestrator distinguish "provider hit output limit" from generic malformed JSON.

Adapters set `truncated=True` when the provider explicitly signals max-token truncation, for example:

- Anthropic: stop reason indicates max-token termination
- OpenAI: finish reason indicates output-length termination
- Google: finish reason indicates max-token termination

Orchestrator behavior:

- log a truncation-specific warning
- do not attempt fuzzy recovery
- treat the result as pending / discarded for that slide
- allow retry/fallback logic to proceed as normal when truncation is surfaced through the provider invocation path

### 5.9 Refactor boundary in `analysis.py`

The following Anthropic-only responsibilities move behind the provider interface:

- client creation
- request payload construction
- image encoding specifics
- raw response text extraction
- provider-specific error typing

The following stay in `analysis.py`:

- prompt construction
- cache load/save
- cache invalidation
- evidence validation
- Pass 2 density scoring
- deduplication
- fallback orchestration
- `_analyze_single_slide()` and `_run_depth_pass()` both call the same provider interface; no separate pass-specific adapter methods are introduced

## 6. Runtime Behavior

### 6.1 Route resolution

For v0.4.0:

- `folio convert` resolves the `convert` route
- the resolved profile is used for both Pass 1 and Pass 2
- if `--llm-profile` is provided, route resolution is bypassed
- API key presence and SDK availability are validated once during route resolution before the first slide is processed
- missing API key or missing SDK on the selected primary profile immediately triggers the pending-analysis path and does not attempt fallback

### 6.2 Retry and fallback

Retry budget:

- 1 retry on the active profile
- exponential backoff: `1s`, then fallback if configured
- at most one pass through each fallback profile in order

Fallback chain exhaustion degrades to pending analysis and does not fail conversion.
When `--llm-profile` is used, fallback is intentionally disabled for that invocation.

### 6.3 Provider-specific exception mapping

Classify errors in provider adapters using current official SDK exception types.

| Provider | Transient | Permanent |
|----------|-----------|-----------|
| Anthropic | `RateLimitError`, `APITimeoutError`, `APIConnectionError`, `InternalServerError` | `AuthenticationError`, `PermissionDeniedError`, `BadRequestError` |
| OpenAI | `RateLimitError`, `APITimeoutError`, `APIConnectionError`, `InternalServerError` | `AuthenticationError`, `BadRequestError`, `PermissionDeniedError` |
| Google | `ResourceExhausted`, `DeadlineExceeded`, `ServiceUnavailable`, `InternalServerError` | `PermissionDenied`, `InvalidArgument`, `Unauthenticated` |

If an SDK exposes only status codes, map:

- transient: `408`, `429`, `500`, `502`, `503`, `504`
- permanent: `400`, `401`, `403`, `404`, `422`

### 6.4 Pending-analysis messages

User-facing pending-analysis messaging must be provider-aware and actionable.

Examples:

- missing key:
  `Analysis pending - profile 'high_quality_anthropic' requires ANTHROPIC_API_KEY`
- SDK missing:
  `Analysis pending - install the OpenAI SDK to use profile 'fast_openai'`
- permanent provider error:
  `Analysis pending - provider 'google' rejected the request for profile 'backup_google'`
- fallback exhausted:
  `Analysis pending - all configured providers for route 'convert' failed transiently`

### 6.5 Version tracking semantics

Provider/model changes, and profile-selection changes that imply different provider/model/prompt identity:

- do not increment `version_history.json`
- do not change `VersionInfo.version`
- do update analysis cache metadata relevant to provider/model/prompt identity
- do update `_llm_metadata` in frontmatter

This follows the content-based versioning invariant in [FR-301](/Users/jonathanoh/Dev/folio.love/docs/product/02_Product_Requirements_Document.md#L162).

## 7. Cache and Metadata Design

### 7.1 Cache file metadata

Both `.analysis_cache.json` and `.analysis_cache_deep.json` must store:

```json
{
  "_cache_version": 2,
  "_prompt_version": "abc12345",
  "_provider_version": "anthropic",
  "_model_version": "claude-sonnet-4-20250514",
  "_extraction_version": "v0.3.1"
}
```

`_profile_version` is intentionally omitted from the cache file entirely. Profile names are configuration labels; renaming a profile without changing provider/model/prompt should not force re-analysis.

### 7.2 Cache validation

On load, invalidate cache if any of these differ from the active runtime:

- `_cache_version`
- `_prompt_version`
- `_provider_version`
- `_model_version`
- `_extraction_version`

Backward compatibility:

- caches missing `_provider_version` are invalidated automatically
- this is a one-time re-analysis, same migration pattern as previous cache-version upgrades

### 7.3 Cache keys

Per-slide entries remain keyed by image hash plus pass-specific suffixes:

- Pass 1: `image_hash`
- Pass 2: `f"{image_hash}_deep"`

Provider/model identity belongs in cache metadata, not in per-entry keys.

Per-slide cache entries also record the actual provider/model that produced that entry for provenance when fallback activates mid-batch. Recommended internal fields:

```json
{
  "_provider": "anthropic",
  "_model": "claude-sonnet-4-20250514"
}
```

These provenance fields are for debugging and quality analysis only. They do not participate in cache keying or invalidation.

### 7.4 Frontmatter metadata

Do **not** add new ontology fields for provider/model/profile in v0.4.0.

Store internal execution metadata under a nested underscore-prefixed block:

```yaml
_llm_metadata:
  convert:
    requested_profile: high_quality_anthropic
    profile: high_quality_anthropic
    provider: anthropic
    model: claude-sonnet-4-20250514
    fallback_used: false
    status: executed
    pass2:
      status: executed
```

If Pass 2 is disabled or skipped:

```yaml
_llm_metadata:
  convert:
    requested_profile: high_quality_anthropic
    profile: high_quality_anthropic
    provider: anthropic
    model: claude-sonnet-4-20250514
    fallback_used: false
    status: executed
    pass2:
      status: skipped
      reason: pass_disabled
```

This keeps ontology v2 unchanged while still exposing runtime provenance.

### 9.1 Provider mock infrastructure

Add provider-specific mock helpers:

- mocked Anthropic response factory
- mocked OpenAI response factory
- mocked Google response factory

Mock factories must simulate provider-native SDK response shapes, not pre-normalized orchestrator strings, so adapter normalization paths are exercised in tests.

No real API keys are required for test execution.

### 9.2 Parametrized provider tests

Core pass-1 tests should run against all three provider adapters with `pytest.mark.parametrize`.

Coverage:

- valid JSON response parses correctly
- markdown code-fenced JSON parses correctly through `_extract_json()`

- truncated response becomes pending
- missing required keys becomes pending

### 9.3 Cache tests

Add tests for:

- provider change invalidates cache
- model change invalidates cache
- legacy cache missing `_provider_version` invalidates
- no cross-provider cache reuse
- renaming a profile with the same provider/model does **not** invalidate cache reuse

### 9.4 Fallback tests

Add tests for:

- transient failure on primary triggers retry then fallback
- permanent failure on primary does not fallback
- missing API key does not trigger fallback
- fallback-chain exhaustion degrades gracefully to pending
- mid-batch fallback writes correct per-slide `_provider` / `_model` cache provenance

### 9.5 CLI and config tests

Add tests for:

- legacy config load path
- new `profiles` + `routing` config
- invalid profile reference fails on config load
- undefined route falls back to `default`
- `--llm-profile` applies to both Pass 1 and Pass 2
- unknown `--llm-profile` fails before conversion work starts

### 9.6 Metadata tests

Add tests for:

- `_llm_metadata` written to frontmatter
- provider-aware pending messages rendered in markdown
- provider/model changes do not bump version

## 11. Acceptance Criteria

- Folio loads both legacy and profile-based LLM config
- `folio convert` can run analysis through Anthropic, OpenAI, or Google Gemini
- all provider adapters return normalized JSON that maps to the existing `SlideAnalysis` structure
- the orchestrator uses `_extract_json()` and accepts fenced JSON but not arbitrary fuzzy extraction
- Pass 1 and Pass 2 use the same effective profile in v0.4.0
- fallback occurs only for transient failures
- provider/model changes invalidate cache reuse without bumping document version
- renaming a profile without changing provider/model/prompt does not invalidate cache reuse
- per-slide cache entries record actual provider/model provenance when fallback switches mid-batch
- `_llm_metadata` is written to frontmatter without extending ontology v2
- user-facing pending-analysis copy is no longer Anthropic-specific
- all tests run with mocked providers and no real API keys

## Appendix A: Implementation Deviations

The following intentional deviations from the spec were made during implementation. Each is a pragmatic improvement over the spec's design and is documented here for traceability.

| ID | Spec says | Implementation does | Rationale |
|----|-----------|---------------------|-----------|
| M1 | `ErrorDisposition` is a dataclass with `kind`, `message`, `original` | Enum with `TRANSIENT`/`PERMANENT` values | Simpler. The orchestrator (`_analyze_with_fallback`, `_run_depth_with_fallback`) builds its own provider-aware messages independently, making the dataclass fields redundant. |
| M2 | `ProviderInput` has `image_bytes: bytes` + `media_type: str` | `image_path: Path` | Each adapter reads bytes in its own format (base64 for Anthropic/OpenAI, raw for Google). Passing a `Path` avoids unnecessary memory copy and double-buffering. |
| M3 | `create_client(api_key: str)` receives the key value | `create_client(api_key_env: str)` receives the env var name | Adapter resolves its own env var internally. Cleaner for BYO credential wiring and avoids the orchestrator needing to know env var semantics. |
| M5 | `ProviderOutput.text: str` | `ProviderOutput.raw_text: str` + `provider_name`/`model_name` fields | `raw_text` is more descriptive. Extra provenance fields enable per-slide cache tracking without a separate return value. |
