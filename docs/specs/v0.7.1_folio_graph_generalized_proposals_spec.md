# Spec v1.1 — folio-graph-generalized-proposals-v0-7-1

## 1. Overview

**Delivers:** the first sub-slice of Shipping Plan §15.6 (shared-consumer expansion) from `docs/specs/tier4_discovery_proposal_layer_spec.md` rev 5. Specifically, `folio graph doctor` is retrofit to formally consume the **§5 Proposal Object Contract** on relationship-proposal findings — every §5 field that applies to relationship proposals appears in the finding dict under the §5 name; a minimal today-computable relation-rule gate (target exists + supported relation kind) populates `schema_gate_result`; and the default CLI stdout row is updated to preserve operator-facing proposal identity (via a new `proposal_id` key) while freeing `subject_id` for its §5 semantics (the entity/document subject of the proposal — applicable to entity-merge, `None` for relationship).

**One-sentence summary:** `folio graph doctor` findings now carry the nine parent §5 contract rows (represented as 11 emitted keys after the §5 ID-triad split) with a minimal today-computable schema gate, a breaking-but-mitigated `subject_id` → `proposal_id` rename, and a narrowed §7 `input_fingerprint` disposition pending parent §7 revision.

**Target version:** v0.7.1.

**Theme:** Tier 4 proposal-layer governance — shared-contract consumption on the existing `folio graph` surface.

**Risk level:** `medium` (feature) + `medium` (migration). v1.1 raises this from `low` because (a) adding `folio/cli.py` to scope-lock and the `subject_id` → `proposal_id` back-compat migration touches the stdout rendering path that operators rely on daily, and (b) the minimal schema-gate addition, while small, introduces a new source of review-noise if relation-rule misclassification occurs. Primary risks: silent-field-drop regression on future `proposal_type` values (mitigated by §4 inventory test); CLI stdout regression during the key rename (mitigated by §3.4 and `G-test-3`).

## 2. Scope

**B2.** The `--json` output of `folio graph doctor` MUST serialize all 11 emitted keys. No key may be silently dropped. Serialization of `schema_gate_result` is either `None` or a dict with keys `{"status": "fail", "rule": "<rule_id>"}` (see §3.3).

**B3.** A new test `test_graph_doctor_surfaces_shared_proposal_contract` enumerates the 11 emitted keys and asserts every one is present on each pending-proposal finding.

**B4.** Existing `graph doctor` findings that are NOT proposals (`orphaned_canonical_relation`, `protected_enrich_body`, `stale_analysis_artifact`, `unconfirmed_entity`, `missing_entity_stub`, `duplicate_person_candidate`) are unchanged by this slice. Only `pending_relationship_proposal` findings get the §5 shape.

**B5.** The default CLI stdout rendering for pending-proposal findings (not `--json`) MUST be updated to use `proposal_id` in place of `subject_id`, and MUST include `trust_status` when `"flagged"` and `schema_gate_result` summary when not `None`. Details in §3.4. `folio/cli.py` is added to manifest allowed_paths.

## 3. Changes

### 3.1 `folio/graph.py` — `graph_doctor()` renderer extension

**File:** `folio/graph.py` (target lines 164–178 + new module-level constants + new helpers).

**Canonical §5 field mapping:** see the table in §2.1 (B1). This §3.1 anchor references the same canonical table; the implementation reader should find every emitted key + source + disposition at §2.1.

**Change:** Extend the `findings.append({...})` call in the pending-proposal loop to include all 11 emitted keys. Add module-level constant `_SHARED_PROPOSAL_CONTRACT_EMITTED_KEYS` and helpers `_derive_trust_status(view)` and `_compute_relationship_schema_gate(view, all_ids)`.

**Pseudocode:**

```python
# Module-level: exported under test but not import *
_SHARED_PROPOSAL_CONTRACT_EMITTED_KEYS: tuple[str, ...] = (
    "proposal_type",
    "source_id",
    "target_id",
    "subject_id",
    "evidence_bundle",
    "reason_summary",
    "trust_status",
    "schema_gate_result",
    "producer",
    "input_fingerprint",   # legacy basis_fingerprint alias; see §7 row of §2.2
    "lifecycle_state",
)

_SUPPORTED_RELATIONS = frozenset({"supersedes", "impacts", "draws_from", "depends_on"})


def _derive_trust_status(view) -> str:
    # §5 row 5. Explicit set intersection avoids truthiness edge-case (ADV-SF5).
    flagged = set(view.flagged_inputs or [])
    return "flagged" if ({"source", "target"} & flagged) else "ok"


def _compute_relationship_schema_gate(view, all_ids: set[str]) -> dict | None:
    # §5 row 6, minimal today-computable gate (ADV-B1 closure).
    # None = all today-computable rules passed.
    # Follow-up slices add richer rules per parent §13 validation tracks.
    if view.proposal.target_id not in all_ids:
        return {"status": "fail", "rule": "target_exists"}
    if view.proposal.relation not in _SUPPORTED_RELATIONS:
        return {"status": "fail", "rule": "supported_relation"}
    return None


def graph_doctor(config, *, scope=None, limit=None) -> list[dict]:
    ...
    # all_ids is already loaded earlier in graph_doctor (graph.py:162).
    for view in pending_views:
        findings.append({
            # Legacy keys (backward-compat; subject_id value changes per §5 semantics).
            "code": "pending_relationship_proposal",
            "severity": "medium",
            "detail": (f"{view.source_id} has pending {view.proposal.relation} -> "
                       f"{view.proposal.target_id} from {view.producer}"),
            "recommended_action": "Review with `folio links review` and confirm or reject it.",
            # Legacy identifier preservation (v1.1 — mitigates subject_id → None break).
            "proposal_id": view.proposal.proposal_id,
            # Shared proposal contract (§5 of tier4_discovery_proposal_layer_spec).
            "proposal_type": "relationship",
            "source_id": view.source_id,
            "target_id": view.proposal.target_id,
            "subject_id": None,  # §5 semantics — applicable to entity-merge, not relationship.
            "evidence_bundle": list(view.proposal.signals),
            "reason_summary": view.proposal.rationale,
            "trust_status": _derive_trust_status(view),
            "schema_gate_result": _compute_relationship_schema_gate(view, all_ids),
            "producer": view.producer,
            # Legacy basis_fingerprint alias; does NOT satisfy full parent §7.
            # Contract-complete derivation deferred to a §7 parent-spec revision.
            "input_fingerprint": view.proposal.basis_fingerprint,
            "lifecycle_state": view.proposal.lifecycle_state,
        })
    ...
```

**Note on `subject_id` / `proposal_id`:** Pre-v0.7.1, `subject_id` carried `view.proposal.proposal_id`. This was inconsistent with §5 semantics (subject_id applies to entity-merge proposals; for relationship proposals the §5 ID triad is `source_id` + `target_id`). v1.1 surfaces the proposal identifier under a new top-level `proposal_id` key and sets `subject_id` to `None` per §5 semantics. This is a breaking change at the key level — documented in CHANGELOG per §6 OQ-3. The CLI stdout renderer is updated in §3.4 to read `proposal_id`, so the operator-facing default output is preserved.

**Proposal-ID recovery tuple (OQ-1 v1.1 resolution):** Consumers that need the proposal identifier should read the top-level `proposal_id` key directly. If reconstruction is required (stateless derivation), use the tuple `(source_id, relation, target_id, basis_fingerprint)` per `folio/pipeline/enrich_data.py:120-126` (`compute_relationship_proposal_id`). `producer` is NOT part of the identity tuple despite the v1.0 spec's incorrect claim (P-4 closure).

### 3.2 Helper-divergence disclosure + promotion path (F-050)

`trust_status` as surfaced on findings is a **derived** label, not a contract field on the underlying `RelationshipProposalView` dataclass. The view retains its `flagged_inputs: list[str]` shape (slice 4 contract unchanged).

v1.0 kept `_derive_trust_status` private in `graph.py`, deferring promotion to sub-slice 2. ADV-SF6 (B.1 Round 1) flagged this as a predictable duplicate-implementation / bad-import trap when `folio synthesize` lands. v1.1 resolution: **keep the helper private in `graph.py` for v0.7.1**, but commit sub-slice 2 (`folio-synthesize-v0-8-0`) to **promote `_derive_trust_status` to a shared module (`folio/proposals.py` or equivalent) BEFORE synthesize implementation begins**, with graph tests proving both consumers call the shared helper. The sub-slice 2 manifest MUST include this as an acceptance criterion in its `pre_a.entry` narrowing.

Rationale: promoting in this slice would expand scope to a new module creation; deferring to sub-slice 2 (which is greenfield and will own the shared-module creation anyway) is less risky.

### 3.3 Minimal relation-rule gate (`schema_gate_result`)

**Rationale (ADV-B1 closure):** parent §5 defines `schema_gate_result` as "result of relation-rule checks needed before review." The v1.0 spec locked `schema_gate_result = None` for all relationship proposals, claiming "no gate runs today." ADV-B1 reproduced that a proposal pointing at a missing `target_id` still surfaces as pending, and today's `graph_doctor()` already loads `all_ids` from `registry.json` (graph.py:162) — target-existence IS computable at the renderer layer.

**v1.1 adds two today-computable rules**:

1. **`target_exists`** — `view.proposal.target_id` must be a key in `registry.json`'s deck IDs. Failure → `schema_gate_result: {"status": "fail", "rule": "target_exists"}`.
2. **`supported_relation`** — `view.proposal.relation` must be in `_SUPPORTED_RELATIONS` (matching `folio/links.py:21`). Failure → `schema_gate_result: {"status": "fail", "rule": "supported_relation"}`.

Both checks use data already loaded in `graph_doctor()`; no new registry reads or imports. Failure return-shape is a 2-key dict `{"status", "rule"}`. Pass (all rules OK) returns `None`. Dispatch hook for future `proposal_type` values: see §3.5.

**Out of scope for this slice:** relation cardinality (e.g., `supersedes` is singular), target-type validation (e.g., `draws_from` targets must be `evidence` type), ontology-term validation. These land when validation-track producers ship per parent §13.

### 3.4 CLI stdout renderer update (`folio/cli.py`)

**File:** `folio/cli.py` (target line 2019 — pending-finding stdout template).

**Change:** Update the stdout template to:
1. Read `finding.get("proposal_id")` (fallback: `finding.get("subject_id")` for non-proposal findings, preserving existing behavior).
2. When `finding["code"] == "pending_relationship_proposal"` AND `finding.get("trust_status") == "flagged"`, append `" [flagged]"` to the severity marker.
3. When `finding.get("schema_gate_result")` is not `None`, append `" [schema-gate: <rule>]"` to the first line.

**Pseudocode:**

```python
# Replaces the current line 2019 (human-readable rendering loop):
for finding in findings:
    subject_or_proposal = (
        finding.get("proposal_id")
        or finding.get("subject_id")
        or "—"
    )
    severity_suffix = ""
    if finding.get("trust_status") == "flagged":
        severity_suffix += " [flagged]"
    gate = finding.get("schema_gate_result")
    if gate is not None:
        severity_suffix += f" [schema-gate: {gate.get('rule', 'unknown')}]"
    click.echo(
        f"[{finding['severity']}{severity_suffix}] {finding['code']} "
        f"{subject_or_proposal}: {finding['detail']}"
    )
    click.echo(f"  Action: {finding['recommended_action']}")
```

The `subject_or_proposal` fallback chain preserves existing rendering for non-proposal findings that still carry `subject_id` (orphaned relations, stale analysis, etc. — those findings are unchanged per §2.3 B4).

### 3.5 Dispatch hook for future `proposal_type` values (advisory)

Per §15.6 and §12, future `proposal_type` values (entity-merge, diagram-archetype) will surface via the same renderer. The `findings.append({...})` dict literal pattern is intentionally NOT abstracted into a factory in this slice — per OQ-2 v1.0 resolution, premature abstraction is avoided. When sub-slice 2's greenfield synthesize arrives, `_derive_trust_status` promotion (§3.2) will likely trigger a broader factory refactor. For v0.7.1, the dispatch hook is documented-only: future proposal_type renderers must emit the same 11 `_SHARED_PROPOSAL_CONTRACT_EMITTED_KEYS` (with the literal `proposal_type` value swapped and the ID triad re-applied — e.g., entity-merge sets `subject_id` to the entity ID and `source_id / target_id` to `None`).

### OQ-4 (v1.1 new): schema_gate_result return shape stability

Future slices may add more rules (e.g., relation-cardinality, ontology-term validation). The v1.1 return shape `{"status": "fail", "rule": "<rule_id>"}` is a 2-key dict. Should we commit to a shape that's forward-compatible with multi-rule-failure (e.g., a list)?

Deferring the shape evolution keeps the v0.7.1 surface minimal.

**Disposition:** resolved-during-A v1.1.
