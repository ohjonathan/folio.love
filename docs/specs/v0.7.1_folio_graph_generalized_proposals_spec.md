# Spec v1.0 — folio-graph-generalized-proposals-v0-7-1

## 1. Overview

**Delivers:** the first sub-slice of Shipping Plan §15.6 (shared-consumer expansion) from `docs/specs/tier4_discovery_proposal_layer_spec.md` rev 5. Specifically, `folio graph doctor` is retrofit to formally consume the **§5 Proposal Object Contract** on relationship-proposal findings — every §5 field that exists on the underlying `RelationshipProposalView` must appear in the finding dict under the §5 name, and §5 fields that do not apply to relationship proposals (today, `schema_gate_result`) must be surfaced explicitly as `null` with a one-line contract comment, not silently omitted. After this slice ships, the finding shape is stable across future `proposal_type` values (entity-merge, diagram-archetype) without further renderer edits.

**One-sentence summary:** when `folio graph doctor` yields a pending-proposal finding, the finding dict now includes all nine §5 contract fields keyed by their §5 names — producing a stable, contract-complete surface that operators and downstream tools can rely on regardless of `proposal_type`.

**Target version:** v0.7.1 (patch release after v0.7.0 digest).

**Theme:** Tier 4 proposal-layer governance — shared-contract consumption.

**Risk level:** `low` (narrow retrofit; no new machinery; no migration). The §5 contract fields already exist on the underlying dataclasses; this slice is renderer-layer wiring plus conformance tests. Primary risk is silent-field-drop regression (a future proposal_type adds a §5 field and the renderer forgets to surface it) — mitigated by an explicit §5-field-inventory test that enumerates the nine required fields.

## 2. Scope

### 2.1 In-scope (behavioral criteria)

**B1.** `graph_doctor()` relationship-proposal findings MUST include these §5 field keys at top level:

| §5 field | Source | Present on RelationshipProposalView | Notes |
|---|---|---|---|
| `proposal_type` | literal `"relationship"` | N/A — derived | Hardcoded for this slice; dispatches by view type in future slices. |
| `source_id` | `view.source_id` | yes | Already surfaced via `subject_id` → renamed to `source_id` per §5. |
| `target_id` | `view.proposal.target_id` | yes | Newly surfaced. |
| `subject_id` | N/A | N/A | Omitted (value `null`) for relationship proposals; applicable to entity-merge in later slices. |
| `evidence_bundle` | `view.proposal.signals` | yes | Surfaced as list. |
| `reason_summary` | `view.proposal.rationale` | yes | Surfaced as string. |
| `trust_status` | derived from `view.flagged_inputs` | yes (as `flagged_inputs`) | Derived: `"flagged"` if any of `["source","target"]` in `flagged_inputs`, else `"ok"`. Helper-divergence disclosure per F-050: `trust_status` is a **derived** label on the finding; the raw `flagged_inputs` list remains the underlying contract on `RelationshipProposalView` and is unchanged by this slice. |
| `schema_gate_result` | literal `null` | N/A | No schema gate runs against relationship proposals today. Surfaced as `null` with an inline comment in the renderer that documents this is a deliberate contract hole, not a bug. When schema-gated producers ship (§13 validation tracks), the renderer will read the gate result off the view. |
| `producer` | `view.producer` | yes | Newly surfaced. |
| `input_fingerprint` | `view.proposal.basis_fingerprint` | yes | Surfaced under §5 name. Per parent §7 implementation invariant, `basis_fingerprint` **is** `input_fingerprint` for relationship proposals. |
| `lifecycle_state` | `view.proposal.lifecycle_state` | yes | Newly surfaced. |

Legacy finding keys (`code`, `severity`, `subject_id`, `detail`, `recommended_action`) are retained unchanged for backward-compatibility with existing CLI output.

**B2.** The `--json` output of `folio graph doctor` MUST serialize the above nine §5 fields by the §5 names; no field may be silently dropped. An explicit `null` (or the string `"ok"`/`"flagged"` for `trust_status`) is acceptable; absence of a key is not.

**B3.** A new test `test_graph_doctor_surfaces_shared_proposal_contract` MUST enumerate the §5 field names and assert every one is present on each pending-proposal finding.

**B4.** Existing `graph doctor` findings that are NOT proposals (e.g., `orphaned_canonical_relation`, `enrich_protected_note`, `stale_analysis_artifact`) are unchanged by this slice. Only `pending_relationship_proposal` findings get the §5 shape.

## 3. Changes

### 3.1 `folio/graph.py` — `graph_doctor()` renderer extension

**File:** `folio/graph.py` (existing; target lines 164–178).

**Change:** Extend the `findings.append({...})` call in the pending-proposal loop to include all nine §5 field keys. Add an inline helper `_derive_trust_status(view)` that maps `view.flagged_inputs` to `"flagged"`/`"ok"`. Add a module-level constant `_SHARED_PROPOSAL_CONTRACT_FIELDS` enumerating the nine §5 field names, used by the conformance test.

**Pseudocode:**

```python
_SHARED_PROPOSAL_CONTRACT_FIELDS: tuple[str, ...] = (
    "proposal_type",
    "source_id",
    "target_id",
    "subject_id",
    "evidence_bundle",
    "reason_summary",
    "trust_status",
    "schema_gate_result",
    "producer",
    "input_fingerprint",
    "lifecycle_state",
)


def _derive_trust_status(view) -> str:
    return "flagged" if view.flagged_inputs else "ok"


def graph_doctor(config, *, scope=None, limit=None) -> list[dict]:
    ...
    for view in pending_views:
        findings.append({
            # Legacy keys (backward-compat)
            "code": "pending_relationship_proposal",
            "severity": "medium",
            "subject_id": None,          # §5 semantic subject_id is N/A for relationship proposals
            "detail": (f"{view.source_id} has pending {view.proposal.relation} -> "
                       f"{view.proposal.target_id} from {view.producer}"),
            "recommended_action": "Review with `folio links review` and confirm or reject it.",
            # Shared proposal contract (§5 of tier4_discovery_proposal_layer_spec)
            "proposal_type": "relationship",
            "source_id": view.source_id,
            "target_id": view.proposal.target_id,
            "evidence_bundle": list(view.proposal.signals),
            "reason_summary": view.proposal.rationale,
            "trust_status": _derive_trust_status(view),
            "schema_gate_result": None,  # §5 contract hole — no gate for relationship proposals
            "producer": view.producer,
            "input_fingerprint": view.proposal.basis_fingerprint,
            "lifecycle_state": view.proposal.lifecycle_state,
        })
    ...
```

**Note on `subject_id` conflict:** The legacy finding key `subject_id` carried `view.proposal.proposal_id` before this slice. Per §5, `subject_id` denotes the document/entity subject of the proposal (applicable to entity-merge), not the proposal ID. This slice renames the legacy use: the proposal ID is no longer written to `subject_id`; it is not required by §5 and is not re-surfaced. Consumers that needed the proposal ID should read `input_fingerprint` (stable across reruns) or derive from `(source_id, target_id, producer)`. This is a deliberate breaking change to the `--json` output shape — the pre-slice `subject_id` value semantics differed from §5 and were wrong to begin with.

### 3.2 Helper-divergence disclosure (per F-050)

`trust_status` as surfaced on findings is a **derived** label, not a contract field on the underlying `RelationshipProposalView` dataclass. The view retains its `flagged_inputs: list[str]` shape (slice 4 contract). This slice does not propose moving the derivation into `links.py` — keeping it in the renderer localizes the §5-contract-specific mapping to the consumer (graph.py) rather than polluting the producer (`links.py`). If future consumers (`folio synthesize`, `folio search`) need the same derivation, they will call the same `_derive_trust_status` helper, which may then be promoted to a shared module. Not promoting speculatively avoids premature abstraction.

### 3.3 No schema-gate source

`schema_gate_result` is surfaced as literal `None` with an inline comment explaining why (no gate runs against relationship proposals in v0.7.1). A future slice that introduces schema-gated producers (parent §13 validation tracks) will read the gate result off the view and populate this field. The renderer is ready for that change; no further renderer edit is needed when the source becomes available, only the field assignment flips from `None` to `view.proposal.schema_gate_result`.

## 6. Open Questions

### OQ-1: should `subject_id` carry `view.proposal.proposal_id` for back-compat?

**Recommendation:** no. §5 defines `subject_id` as a proposal-about-entity field (applicable to entity-merge), not as the proposal's own ID. Surfacing `proposal_id` as `subject_id` would mean the key carries different semantics per `proposal_type`, defeating the shared-contract goal. Consumers that need the proposal ID can reconstruct it from `(source_id, relation, target_id, input_fingerprint)` — the same tuple used by the lifecycle-state machinery.

**Disposition:** resolved-during-A. Change documented in §3.1 "Note on `subject_id` conflict." Breaking change acknowledged in the CHANGELOG entry.

### OQ-2: should `_derive_trust_status` live in `links.py` instead of `graph.py`?

**Recommendation:** no, not yet. Keeping the derivation in the consumer (graph.py) avoids introducing a §5-specific surface into the producer module. When `folio synthesize` and `folio search` adopt the same derivation (sub-slices 2 and 3), we will promote the helper to a shared module (likely `folio/proposals/` — TBD by sub-slice 2).

**Disposition:** resolved-during-A. `_derive_trust_status` is a module-private helper in `graph.py` for v0.7.1.

### OQ-3: does this slice require a migration note in CHANGELOG?

**Recommendation:** yes. The `subject_id` semantic change in `graph doctor --json` output is a breaking change per §3.1. CHANGELOG v0.7.1 entry documents this under "Breaking" with the replacement guidance.

**Disposition:** resolved-during-A. CHANGELOG entry is part of Phase C.
