# Spec Review Consolidation: v0.4.0 Multi-Provider LLM Support

**Spec under review:** `docs/specs/v0.4.0_multi_provider_llm_support_spec.md`
**Date:** 2026-03-07
**Reviewers:** 3 independent (Peer, Alignment, Adversarial)
**Verdict:** Request Changes (0/3 Approve)

---

## Verdict Summary

All three reviewers converged on the same core finding: the spec defines *what* to build (config model, routing, fallback policy) but critically omits *how* to build it (provider abstraction, response parsing, prompt portability). As written, it is a product proposal, not an implementation spec.

| Reviewer | Role | Verdict | Blocking Issues |
|----------|------|---------|-----------------|
| R1 | Peer | Request Changes | No provider abstraction, no response parsing strategy, no test plan |
| R2 | Alignment | Request Changes | Evidence parsing fragility, cache key scoping undefined, frontmatter schema breaks ontology |
| R3 | Adversarial | Request Changes | Response parsing hardcoded for Anthropic, evidence format assumes Claude, cache lacks provider identity |

---

## Blocking Issues (Critical — must resolve before implementation)

### B1. Provider abstraction undefined

**Flagged by:** R1, R2, R3

The spec has no Technical Design section. No interface/protocol, no method signatures, no factory pattern. A developer cannot implement without designing the architecture themselves.

**What to add:** A new "Technical Design" section defining:
- An `AnalysisProvider` protocol with methods for `create_message()`, response text extraction, error classification
- How providers are registered and selected (factory pattern keyed on `provider` string)
- How the existing Anthropic code in `pipeline/analysis.py` gets refactored behind this interface

### B2. Response parsing hardcoded for Anthropic SDK

**Flagged by:** R1, R2, R3

The current code at `pipeline/analysis.py:336` does:
```python
raw_text = response.content[0].text
```
And the image payload at `analysis.py:311-332` uses Anthropic's specific base64 message structure:
```python
response = client.messages.create(
    model=model,
    max_tokens=2048,
    timeout=120.0,
    messages=[{"role": "user", "content": [
        {"type": "image", "source": {"type": "base64", "media_type": media_type, "data": image_data}},
        {"type": "text", "text": full_prompt},
    ]}],
)
```
OpenAI returns `response.choices[0].message.content`. Gemini has yet another structure. The first non-Claude call will crash or silently return `pending()`.

**What to add:** Each provider adapter must:
1. Accept a common input (prompt text + optional base64 image bytes)
2. Build provider-specific request payload (including image encoding format)
3. Normalize the response to a plain `str`

This belongs in the Technical Design section as concrete method signatures.

### B3. Evidence format parsing assumes Claude's instruction-following

**Flagged by:** R1, R2, R3

`_parse_evidence()` at `analysis.py:387-419` uses strict regex patterns:
```
r"^-\s*Claim:\s*"
r'Quote:\s*"?([^"]*?)"?\s*(?:\n|$)'
r"Element:\s*(\w+)"
r"Confidence:\s*(\w+)"
```
These rely on Claude producing exact field names in exact order. Other providers may return different field names, reorder fields, use narrative prose, or omit delimiters. Evidence will silently drop to zero claims.

**What to add:** Choose one strategy and document it:
- **(a)** Provider-specific prompt variants tuned to each provider's formatting tendencies
- **(b)** Switch to JSON-based structured output (all three major providers support structured/JSON mode)
- **(c)** Add fuzzy parsing with quality gates (fallback regex patterns, minimum claim threshold)

Recommendation: **(b)** is the most portable and testable. Document expected accuracy per provider.

## Should-Fix Issues (Major — high risk if deferred)

### S1. Cache scoping strategy undefined

**Flagged by:** R1, R2, R3

Spec Section 6 says "cache scoped by profile/provider/model" but doesn't specify how. Current implementation at `analysis.py:974-977`:
```python
cache["_cache_version"] = _ANALYSIS_CACHE_VERSION
cache["_prompt_version"] = _prompt_version(ANALYSIS_PROMPT)
cache["_model_version"] = model
cache["_extraction_version"] = _EXTRACTION_VERSION
```
Cache keys are `image_hash` (SHA256[:16]) with no provider qualifier. `_model_version` stores the model name but not the provider. Two providers using similarly-named models could cause cross-provider cache reuse.

**Recommendation:** Add `_provider_version` to cache metadata alongside `_model_version`. On load, validate both match current config. Backward compat: old caches missing `_provider_version` auto-invalidate (one-time re-analysis, same pattern as existing `_cache_version` migration).

### S2. Frontmatter LLM metadata fields not in ontology

**Flagged by:** R2

Spec Section 5.4 says to store provider/model/profile/fallback in frontmatter. But Ontology v2 Section 12 (`Folio_Ontology_Architecture_v2.md:733-797`) has no such fields. Adding unschematized fields breaks the established contract.

**Recommendation:** Either:
- Extend Ontology Section 12.4 (Source Fields) with `llm_provider`, `llm_model`, `llm_profile`, `llm_fallback_used`
- Or use a nested `_llm_metadata` block (underscore prefix = internal/non-ontology), keeping core schema clean

The nested approach is safer for v0.4.0 since the ontology is shared across all space types and LLM metadata only applies to converted evidence.

### S3. Multi-pass with different providers not addressed

**Flagged by:** R1, R2

Spec Section 4.2 allows separate routing for `convert_pass1` and `convert_pass2`. But Pass 2's depth prompt (DEPTH_PROMPT) interpolates Pass 1 results. If Pass 1 is Claude and Pass 2 is GPT, response format incompatibility may break depth analysis.

**Recommendation:** For v0.4.0, explicitly lock Pass 1 and Pass 2 to the same provider profile. Document cross-provider pass mixing as a future enhancement requiring format normalization between passes.

### S4. Fallback exception mapping not specified

**Flagged by:** R1, R2, R3

Spec Section 5.3 lists transient vs. permanent failures conceptually but doesn't map to actual SDK exception types. Current code catches generic `Exception`.

**Recommendation:** Add a provider-specific exception classification table:

| Provider | Transient | Permanent |
|----------|-----------|-----------|
| Anthropic | `RateLimitError`, `APITimeoutError`, `APIConnectionError`, `InternalServerError` | `AuthenticationError`, `PermissionDeniedError`, `BadRequestError` |
| OpenAI | `RateLimitError`, `APITimeoutError`, `APIConnectionError`, `InternalServerError` | `AuthenticationError`, `BadRequestError` |
| Google | `ResourceExhausted`, `DeadlineExceeded`, `Unavailable` | `PermissionDenied`, `InvalidArgument`, `Unauthenticated` |

### S5. Image input format is provider-specific

**Flagged by:** R1, R3

`_analyze_single_slide()` at `analysis.py:299-332` hardcodes Anthropic's base64 image encoding in the message payload. OpenAI uses a different `image_url` structure. Gemini has its own format.

**Recommendation:** Include image encoding in the provider abstraction. Each adapter's `create_message()` handles its own image payload format. The caller passes raw image bytes + media type; the adapter builds the provider-specific structure.

### S6. Test strategy entirely missing

**Flagged by:** R1, R2, R3

No test plan exists. Current test suite (11 test files in `tests/`) mocks Anthropic only. Multi-provider support multiplies the test surface significantly.

**Recommendation:** Add a Test Strategy section covering:
- **Provider-specific mocks:** Each adapter gets its own mock response factory
- **Parametrized tests:** Core analysis tests run against all provider mocks via `@pytest.mark.parametrize`
- **Cache tests:** Cross-provider cache invalidation, provider version mismatch detection
- **Fallback tests:** Transient failure triggers fallback, permanent failure does not, chain exhaustion degrades gracefully
- **No real API keys required:** All tests use mocked SDK clients

### S7. CLI override semantics (`--llm-profile`) undefined

**Flagged by:** R2

Spec Section 5.2 shows usage but doesn't define behavior edges:
- Does `--llm-profile` override both passes or just the primary route?
- Does it disable fallbacks?
- What happens if the profile name doesn't exist in config?

**Recommendation:** Clarify: `--llm-profile X` sets the effective profile for all tasks in that command invocation, disabling task-specific routing. Fallbacks from that profile's route still apply. If `X` is not defined in config, raise an error at startup (not mid-conversion).

### S8. Version tracking semantics for provider changes undefined

**Flagged by:** R2

If a user re-converts the same deck with a different provider, should the version increment? Current version tracking in `tracking/versions.py` is content-based (text/image changes drive version bumps).

**Recommendation:** State explicitly: provider/model changes do NOT bump version (version is content-based). LLM metadata changes are recorded in frontmatter only. This preserves the existing version semantics.

### S9. Config validation for profile references missing

**Flagged by:** R2

If routing references a nonexistent profile name, behavior is undefined. No validation exists in `FolioConfig.load()` (config.py:59-94).

**Recommendation:** Add validation: all profiles referenced in routing must exist in `profiles`. Error on config load, not at runtime. This is consistent with the existing `_validate()` pattern in `FolioConfig.__post_init__()`.

### S10. Roadmap placement unclear

**Flagged by:** R2

The roadmap (`04_Implementation_Roadmap_v2.md:188-190`) mentions "LLM settings" as part of Tier 2 Week 9-10 config work. It does not scope multi-provider as a standalone feature. The v0.4.0 framing implies a major feature release, but the roadmap doesn't account for it.

**Recommendation:** Either revise the roadmap to include multi-provider as explicit Tier 2 work, or reposition this as a v0.3.x incremental enhancement. The scope (3 providers, fallback chains, task routing) is substantial enough to warrant its own roadmap line item.

## Recommended Structural Changes to the Spec

The current spec has 10 sections focused on config, runtime behavior, and doc updates. The reviewers agree it needs at minimum:

1. **New Section: Technical Design** (between current Sections 5 and 6)
   - `AnalysisProvider` protocol definition
   - Provider adapter implementations (Anthropic, OpenAI, Gemini)
   - Factory/registry pattern for provider selection
   - Response normalization contract
   - Image payload abstraction
   - Evidence parsing strategy (structured output vs. provider-specific prompts)

2. **New Section: Test Strategy** (after current Section 9)
   - Provider mock infrastructure
   - Parametrized test approach
   - Cache cross-contamination tests
   - Fallback chain tests
   - No-API-key test execution

3. **Expanded Section 6 (Cache Behavior):**
   - Concrete cache key format including provider identity
   - Backward compatibility migration (old caches auto-invalidate)
   - Cache file format changes

4. **Expanded Section 5.3 (Fallback Policy):**
   - Provider-specific exception classification table
   - Retry budget / backoff strategy

---

## Recommendation: Split the Implementation

All three reviewers implicitly suggest this. The spec covers three largely separable concerns:

| Phase | Scope | Risk |
|-------|-------|------|
| **v0.4.0-a** | Config model + extract provider interface from current Anthropic-hardcoded code. Anthropic remains the only adapter. | Low — refactor only, no new providers |
| **v0.4.0-b** | Add OpenAI adapter + cache scoping + frontmatter LLM metadata | Medium — first real multi-provider test |
| **v0.4.0-c** | Add Gemini adapter + fallback chains + exception mapping | Higher — fallback logic is the riskiest part |

This lets you ship and validate incrementally. Phase A is testable immediately with existing tests (refactor, no behavior change). Phase B is the real proof point. Phase C adds the remaining provider and the fallback complexity.
