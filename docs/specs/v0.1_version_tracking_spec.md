## 5. Key Design Decisions

### D1: Change Detection vs Analysis Cache Hashing (G1)

**Decision: Accept the difference.** The concerns are distinct:

- `versions.py:_normalize_text()` (line 243) — whitespace-insensitive comparison for **semantic content** changes. A reformatted speaker note should NOT create a new version.
- `analysis.py:_text_hash()` (line 134) — exact text hash for **API input reproduction**. Different whitespace = different API input = cache miss.

**User-facing scenario:** A whitespace-only edit (e.g., reformatting speaker notes) produces a conversion where version history says "no changes" but analysis re-runs. This is correct, not incoherent: version history tracks semantic content (the words didn't change), while the analysis cache tracks API input fidelity (the prompt bytes changed, so the LLM should re-evaluate). A user viewing the output sees the same version number with fresh analysis — which is exactly the right behavior for a formatting-only edit.

**Cross-reference:** See analysis caching spec (v0.1) Section 4.2–4.3 for `_text_hash()` storage details and per-slide cache validation flow.

### D2: No `_EXTRACTION_VERSION` in Texts Cache (G2)

**Decision: Not needed.** The texts cache stores raw extracted text for change detection via `_normalize_text()` comparison. If extraction logic changes, the new text will differ from the old text, and `detect_changes()` will correctly report the difference. Unlike the analysis cache (where identical inputs can produce stale outputs after a prompt change), the texts cache is self-correcting — honest diffs require no version marker.

**Extraction version bump scenario:** If `_EXTRACTION_VERSION` changes (e.g., extraction becomes more aggressive, pulling text from previously-skipped elements), the first post-upgrade conversion will see different text for every affected slide. Version history will show a one-time spike of "all slides modified" — this honestly reflects that the extracted content changed, even though the source file didn't. This is expected behavior, not a bug.

**Cross-reference:** The analysis cache handles `_EXTRACTION_VERSION` differently — it stores the version as a cache-level marker and invalidates on mismatch (see analysis caching spec v0.1, Section 4.5). The two approaches are complementary: the analysis cache needs explicit version tracking because prompt changes can produce different outputs from identical inputs. The texts cache doesn't have this problem because it stores raw extraction output, not derived results.

### D3: Slide Reorder Deferral (G3)

**Decision: Defer to Tier 2+.** Positional identity (slide number = identity) is the Tier 1 approach. Reorders appear as multiple "modified" slides — conservative, never misses a change. Content-based reorder detection introduces complexity (what if reordered AND modified?) without Tier 1 benefit.

**Known limitation — insertion/deletion scenarios:** Because identity is position-based, inserting or deleting a slide in the middle of a deck causes all subsequent slides to shift position. Example: deleting slide 3 from a 5-slide deck means old slide 4 is now compared to new slide 3, and old slide 5 to new slide 4. Both are reported as "modified" even though their content is unchanged — it simply moved. This produces inflated "modified" counts in the ChangeSet and version history table.

This is an inherent trade-off of positional identity. The behavior is conservative (no changes are missed) but not precise (unchanged-but-shifted content is reported as modified). Users should interpret "modified" counts with this in mind. The `_format_version_history()` output includes a comment noting that detection is position-based.

Content-based identity (e.g., slide content hashing or fingerprinting) would resolve this but introduces its own edge cases (duplicate slides, slides modified AND moved). Deferred to Tier 2+.

### D4: Write Failure Behavior (G8)

**Decision: Let it fail.** Version history (`version_history.json`) and texts cache (`.texts_cache.json`) are correctness-critical, not performance caches. If `version_history.json` can't be written, the next conversion would start at v1, losing all provenance. Analysis cache failure is tolerable because results are in memory and returned regardless; version tracking failure is data corruption. The current behavior — `_atomic_write_json()` at `versions.py:251` raises OSError on failure — is correct.

**Concurrency note:** Version tracking is single-process. `_atomic_write_json()` provides atomicity against partial writes (crash safety), not mutual exclusion. Parallel conversion of the same deck (e.g., two `folio convert` processes targeting the same output directory) is unsupported and may produce corrupted version history. This is acceptable for Tier 1 — the CLI is inherently single-user.

### D5: Version on No-Change Reconversion

**Decision: Always increment (existing behavior).** Every `folio convert` invocation calls `compute_version()` (converter.py:154), which increments the version number. The `changes` field captures "no changes" via empty `added`, `removed`, `modified` lists and a populated `unchanged` list. This preserves a complete audit trail.

### D6: Status Field Semantics (G7)

**Decision: Document the distinction.** Two unrelated enums exist:
- `frontmatter.py:71` — `status: "active"` is a **lifecycle** state (active vs archived). Set at conversion time, doesn't change.
- `sources.py:check_staleness()` (line 62) — returns `current|stale|missing`, which is a **staleness** check computed dynamically by the `folio status` CLI command. Not stored in frontmatter.

These are different dimensions. No unification needed.

### G6: Unbounded Version History Table (Code)

Full history remains in `version_history.json` (written by `save_version_history()` at `versions.py:203`). Only the markdown display is truncated.

**Files:** `folio/output/markdown.py`

**Resolution:** Add comment to `frontmatter.py` at the `status` field explaining the distinction. Reference D6.

**Files:** `folio/output/frontmatter.py`

### G8: Write Failure Behavior (Documentation)

**Current behavior:** `_atomic_write_json()` at `versions.py:251` does not catch OSError. A write failure during `save_version_history()` or `save_texts_cache()` propagates up and crashes the conversion.

**Resolution:** Add docstring to `_atomic_write_json()` explaining this is intentional: version data is correctness-critical, not a cache. Compare to `_save_cache()` in `analysis.py:969` which catches OSError because analysis results are already in memory. Reference D4.

**Files:** `folio/tracking/versions.py`

## 7. Technical Design

### 7.1 G6: Version History Table Truncation

**File: `folio/output/markdown.py`**
**Function: `_format_version_history()` (line 177)**

```python
def _format_version_history(history: list[dict], max_display: int = 10) -> str:
    """Format the version history table.

    Args:
        history: Full version history list.
        max_display: Maximum number of versions to show in the table.
            Full history is preserved in version_history.json.
    """
    lines = [
        "## Version History",
        "",
    ]

    total = len(history)
    if total > max_display:
        lines.append(f"*Showing last {max_display} of {total} versions.*")
        lines.append("")
        display_history = history[-max_display:]
    else:
        display_history = history

    lines.extend([
        "| Version | Date | Changes | Note |",
        "|---------|------|---------|------|",
    ])

    for v in reversed(display_history):
        # ... existing row formatting unchanged ...
```

**Truncation ordering:** History is truncated to the last N entries chronologically (`history[-max_display:]`), then reversed for display (newest first). This ensures the most recent versions are always visible.

**Call site** at `markdown.py:75` (`assemble()`) — no change needed. The call `_format_version_history(version_history)` uses the default `max_display=10`.

**~8 lines changed.**

### 7.1.1 Version History Entry Schema

Each entry in `version_history.json` is produced by `VersionInfo.to_dict()`:

```json
{
  "version": 3,
  "timestamp": "2026-03-06T12:00:00+00:00",
  "source_hash": "a1b2c3d4e5f6",
  "source_path": "../../sources/deck.pptx",
  "note": "Updated financials",
  "slide_count": 12,
  "changes": {
    "added": [],
    "removed": [5],
    "modified": [3, 7],
    "unchanged": [1, 2, 4, 6, 8, 9, 10, 11, 12]
  }
}
```

The `_format_version_history()` function receives `history: list[dict]` where each dict conforms to this schema. The `changes` sub-dict drives the "Changes" column rendering.

## 9. Migration and Compatibility

**No migration needed.** All changes are:
- Documentation additions (docstrings, comments)
- Additive parameter with default (`max_display=10`)
- New test file

Existing `version_history.json` and `.texts_cache.json` files are fully compatible. No format changes.

**Pre-version-tracking decks:** Decks converted before version tracking was implemented will have no `version_history.json` or `.texts_cache.json`. On first re-conversion with version tracking active, `load_version_history()` returns an empty list and `load_texts_cache()` returns an empty dict. `compute_version()` creates v1 with all slides marked as "added". No migration step is needed — the system bootstraps naturally from the absence of prior state.
