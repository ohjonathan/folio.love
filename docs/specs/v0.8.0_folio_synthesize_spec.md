# v0.8.0 — `folio synthesize` (Shared-Consumer Expansion Sub-Slice 2 of 3)

## 1. Overview

**Delivers:** the second sub-slice of Shipping Plan §15.6 (shared-consumer expansion) from `docs/specs/tier4_discovery_proposal_layer_spec.md` rev 5. Introduces a new CLI command, `folio synthesize <scope>`, that consumes the §5 Proposal Object Contract and surfaces proposal-linked cross-references across a scoped set of documents as a **structural synthesis report** (JSON and human-readable stdout).

**One-sentence summary:** `folio synthesize` is a read-only reporting surface that groups §5 shared-contract proposal findings by scope, honors §11 trust-gating with `--include-flagged`, and emits a shared payload-level `schema_version` envelope that graph and future search will also adopt.

**Target version:** v0.8.0 (new minor; new command).

**Theme:** Tier 4 proposal-layer governance — shared-contract consumption on a greenfield CLI surface.

**Risk level:** `low` (feature) + `low` (migration). Structural-only MVP; no LLM, no registry mutations, no writes. Primary risks: (1) envelope-design churn if future search diverges — mitigated by designing the envelope conservatively (§3.3) and forward-declaring graph migration; (2) scope drift toward LLM synthesis — mitigated by explicit §2.5 non-goals and F-058 mitigation (author scope-line-caps after the test plan is frozen).

### 1.1 Core behaviors

1. `folio synthesize <scope>` emits a deterministic synthesis report listing §5 proposal-linked cross-references grouped by source document within the scope. Default output is a human-readable stdout summary.
2. `folio synthesize <scope> --json` emits the **shared payload-level envelope** (§3.3) with `schema_version: "1.0"` and `command: "synthesize"`.
3. `<scope>` resolves to an engagement subtree, document ID, or `-` for library-wide (same scope-resolution pattern as `folio links` / `folio graph doctor`).
4. Trust gating: default excludes proposals whose source OR target input is flagged (§11). `--include-flagged` widens the predicate and the envelope's `trust_override_active` field is set to `true`. Excluded count is surfaced in both stdout and JSON per parent §11 rule 4.
5. Exit codes: `0` on success (findings ≥ 0); non-zero only on scope-resolution failure or internal error. The command never shells out; no LLM calls in v0.8.0.
6. Synthesis is **read-only** — no registry mutations, no doc creation, no side effects on the library.

### 3.2 Shared-consumer contract wiring

`folio.synthesize.synthesize()` MUST call `folio.tracking.trust.derive_trust_status(view)` directly on each iterated view, **not** via a re-exported alias. `tests/test_trust.py::TestSharedConsumerInvariant::test_synthesize_module_uses_shared_helper` asserts `folio.synthesize.derive_trust_status is folio.tracking.trust.derive_trust_status` (extension of the existing graph invariant), proving both consumers call the same helper.

**§5 contract read path.** Synthesize consumes `collect_pending_relationship_proposals(config, scope=scope, include_flagged=include_flagged)` — the identical entry point `folio.graph.graph_doctor` uses. This is the load-bearing shared-consumer invariant: the function name is the contract boundary, not a string grep. Tests MUST assert the import + call exist (`tests/test_synthesize.py::test_consumes_shared_collect_function`).

**Dispatch hook for future `proposal_type` values.** v0.8.0 surfaces `proposal_type: "relationship"` only. When entity-merge / diagram-archetype producers emit, synthesize renders them via the same finding shape — `subject_id` carries the entity ID, `source_id`/`target_id` are `None`. This is documented behavior (per v0.7.1 §3.5 dispatch-hook advisory, carried forward); no code change required until a non-relationship producer ships.

### 3.4 CLI contract (`folio/cli.py` narrow addition)

Register a new Click command group following the `graph` pattern at `folio/cli.py:1954-1958`:

```python
# New in v0.8.0 — added near folio/cli.py:1954 (after graph registrations, before any tail imports):

@cli.command()
@click.argument("scope", required=False, default=None)
@click.option("--json", "json_output", is_flag=True, default=False, help="Output as shared envelope JSON.")
@click.option(
    "--include-flagged",
    is_flag=True,
    default=False,
    help="Include proposals whose source or target document has `review_status: flagged`.",
)
@click.pass_context
def synthesize(ctx, scope: Optional[str], json_output: bool, include_flagged: bool):
    """Structural synthesis report of §5 proposal-linked cross-references within a scope.

    Read-only: no LLM calls, no registry mutations. See CHANGELOG v0.8.0 for details.
    """
    from .synthesize import synthesize as run_synthesize, render_envelope

    config = ctx.obj["config"]
    report = run_synthesize(config, scope=scope, include_flagged=include_flagged)

    if json_output:
        payload = render_envelope(report, scope_arg=scope or "")
        click.echo(json.dumps(payload, indent=2))
        return

    _render_synthesis_stdout(report)
```

**Choice of `@cli.command()` vs `@cli.group()`:** v0.8.0 ships a single top-level command (no subcommands). If v0.8.1 adds subcommands (`folio synthesize doctor`, `folio synthesize daily`, etc.) the migration to `@cli.group()` is mechanical. This is consistent with how `folio links` evolved.

**`<scope>` resolution:** use the same `ScopeOrCommandGroup`-compatible scope argument pattern as `folio graph doctor`. `None` (no arg) ⇒ library-wide. Explicit `-` ⇒ library-wide. String ⇒ engagement subtree or document ID match.

### 3.5 Stdout renderer (`_render_synthesis_stdout`)

Defined in `folio/synthesize.py` (not `cli.py`, to keep CLI narrow):

```python
def _render_synthesis_stdout(report: SynthesisReport) -> None:
    """Compact human-readable summary: one line per finding + header + footer."""
    click.echo(f"Synthesis for scope: {report.scope or '(library-wide)'}")
    click.echo(f"Findings: {len(report.findings)}")
    if report.excluded_flagged_count:
        click.echo(
            f"Excluded (flagged inputs): {report.excluded_flagged_count} "
            f"(use --include-flagged to include)"
        )
    if report.trust_override_active:
        click.echo("Trust override active: --include-flagged")
    click.echo("")
    for finding in report.findings:
        flag = " [flagged]" if finding.trust_status == "flagged" else ""
        rel = finding.relation or "<n/a>"
        click.echo(
            f"[{finding.lifecycle_state}{flag}] {finding.source_id} "
            f"--{rel}--> {finding.target_id or '—'}  ({finding.producer})"
        )
        click.echo(f"  {finding.reason_summary}")
```

This honors parent §8 (compact summary + trust label + schema-gate-compatible line) — partially. Full evidence-bundle rendering (1–3 locator lines) deferred to v0.8.1 per §1.3 row / §2.5.

## 5. §11 Trust Gating

Synthesize honors parent §11 rules 1–5 (rule 6 — `extraction_confidence` — is not a synthesize concern).

| Rule | How honored |
|---|---|
| 1. `review_status: flagged` inputs excluded by default | `collect_pending_relationship_proposals(include_flagged=False)` already excludes flagged source/target; synthesize passes the flag through. |
| 2. Flagged-dependent proposals may exist but trust posture surfaces explicitly | `trust_status` field on every finding; "flagged" inputs surface "flagged" status; tests verify. |
| 3. Explicit operator override `--include-flagged` | CLI flag; passed to `collect_pending_relationship_proposals`; envelope `trust_override_active` reflects. |
| 4. Surfaces disclose excluded flagged-input counts | `excluded_flagged_count` in envelope + "Excluded (flagged inputs): N" line in stdout. |
| 5. No surface silently reports empty when exclusion is the cause | Stdout always prints the excluded count (even when 0); tests verify empty-findings-with-exclusions path shows the count message. |

## 7. `--include-flagged` behavior

```
$ folio synthesize engagement-acme --include-flagged
Synthesis for scope: engagement-acme
Findings: 14
Trust override active: --include-flagged

[queued] doc_a --draws_from--> doc_b  (enrich)
  Both docs reference the shared framework.
[queued [flagged]] doc_c --impacts--> doc_d  (enrich)
  Source document has review_status: flagged.
```

## 8. Failure Modes

| Failure | Behavior |
|---|---|
| Invalid scope string (no matching engagement/doc) | Exit 1; stderr: `Error: scope '<arg>' does not resolve to an engagement or document.` |
| Registry missing | Exit 1; stderr: `Error: registry.json not found. Run \`folio ingest\` first.` |
| `collect_pending_relationship_proposals` raises | Exit non-zero; propagate the exception; no partial JSON output. |
| Zero findings | Exit 0; stdout prints header + "Findings: 0" + trailing newline. |
| Zero findings with `N` excluded flagged | Exit 0; stdout prints header + "Findings: 0" + "Excluded (flagged inputs): N" + footer. |
