# v0.8.0 — `folio synthesize` (Shared-Consumer Expansion Sub-Slice 2 of 3)

## 1. Overview

**Delivers:** the second sub-slice of Shipping Plan §15.6 (shared-consumer expansion) from `docs/specs/tier4_discovery_proposal_layer_spec.md` rev 5. Introduces a new CLI command, `folio synthesize <scope>`, that consumes the §5 Proposal Object Contract and surfaces proposal-linked cross-references across a scoped set of documents as a **structural synthesis report** (JSON and human-readable stdout).

**One-sentence summary:** `folio synthesize` is a read-only reporting surface that groups §5 shared-contract proposal findings by scope, honors §11 trust-gating with `--include-flagged`, and emits a shared payload-level `schema_version` envelope that graph and future search will also adopt.

**Target version:** v0.8.0 (new minor; new command).

**Theme:** Tier 4 proposal-layer governance — shared-contract consumption on a greenfield CLI surface.

**Risk level:** `low` (feature) + `low` (migration). Structural-only MVP; no LLM, no registry mutations, no writes. Primary risks: (1) envelope-design churn if future search diverges — mitigated by designing the envelope conservatively (§3.3) and forward-declaring graph migration; (2) scope drift toward LLM synthesis — mitigated by explicit §2.5 non-goals and F-058 mitigation (author scope-line-caps after the test plan is frozen).

### 1.1 Core behaviors

1. `folio synthesize <scope>` emits a deterministic synthesis report listing §5 proposal-linked cross-references grouped by source document within the scope. Default output is a human-readable stdout summary.
2. `folio synthesize <scope> --json` emits the **shared payload-level envelope** (§3.3) with `schema_version: "1.0"` and `command: "synthesize"`.
3. `<scope>` resolves to an engagement subtree, document ID, or `-` for library-wide (same scope-resolution pattern as `folio links` / `folio graph doctor`). In the JSON envelope, library-wide is represented as `"scope": null` (MIN-1 closure).
4. Trust gating: default excludes proposals whose source OR target input is flagged (§11). `--include-flagged` widens the predicate and the envelope's `trust_override_active` field is set to `true`. `excluded_flagged_count` is surfaced in both stdout and JSON per parent §11 rule 4 and is routed from `SuppressionCounts.flagged_input` (the integer field on the dataclass returned by `collect_pending_relationship_proposals` at `folio/links.py:203-210` — CB-1 closure). The `SuppressionCounts.rejection_memory` dict is NOT used for this count (rejection-memory suppressions are a different concern; v0.8.0 does not surface per-producer suppression counts).
5. Exit codes: `0` on success (findings ≥ 0); non-zero only on scope-resolution failure or internal error. The command never shells out; no LLM calls in v0.8.0.
6. Synthesis is **read-only** — no registry mutations, no doc creation, no side effects on the library.
7. Zero-findings diagnostics (CB-5 closure): stdout always prints an `Excluded (flagged inputs): N` line (even when N=0). When findings==0 AND excluded==0, a next-action line is printed: `Next: run `folio ingest` or `folio enrich` if no proposals have been emitted yet`. This prevents parent §11 rule 5 purposive violation (silent empty results).

## 3. Module Structure

### 3.1 Public surface (`folio/synthesize.py`)

```python
# folio/synthesize.py
"""Synthesis reporting for Tier 4 proposal-layer findings.

Consumes the §5 shared proposal contract from
docs/specs/tier4_discovery_proposal_layer_spec.md and renders a structural
synthesis report across a scope. Read-only; no LLM calls in v0.8.0.
"""

from __future__ import annotations

from dataclasses import dataclass
from typing import Optional

from .config import FolioConfig
from .links import collect_pending_relationship_proposals
from .tracking.trust import derive_trust_status


SCHEMA_VERSION = "1.0"  # Shared envelope version — see §3.3 of the v0.8.0 spec.
COMMAND_NAME = "synthesize"  # Envelope `command` field — see §3.3 E3.


@dataclass(frozen=True)
class SynthesisFinding:
    proposal_id: str
    proposal_type: str                # literal "relationship" for v0.8.0
    source_id: str
    target_id: Optional[str]
    subject_id: Optional[str]         # None for relationship; set for entity-merge in v0.8.1+
    relation: Optional[str]
    evidence_bundle: list[str]
    reason_summary: str
    trust_status: str                 # "ok" | "flagged"
    producer: str
    lifecycle_state: str
    # Parent §5 minimum-contract fields added per CB-2 closure:
    schema_gate_result: Optional[dict]  # None in v0.8.0 — synthesize does not run gate rules
    input_fingerprint: str              # legacy alias for basis_fingerprint (parity w/ graph v0.7.1 §3.1 row 8)


@dataclass(frozen=True)
class SynthesisReport:
    scope: Optional[str]              # None for library-wide (MIN-1); JSON serializes as null
    findings: list[SynthesisFinding]
    excluded_flagged_count: int       # Routes from SuppressionCounts.flagged_input (CB-1)
    trust_override_active: bool


def synthesize(
    config: FolioConfig,
    *,
    scope: Optional[str] = None,
    include_flagged: bool = False,
    limit: Optional[int] = None,
) -> SynthesisReport:
    """Produce a structural synthesis report for the given scope.

    Read-only. Consumes the §5 shared proposal contract via
    `folio.links.collect_pending_relationship_proposals`, derives trust
    posture via the shared `folio.tracking.trust.derive_trust_status`, and
    groups findings by source document.

    Return type:
      SynthesisReport with `excluded_flagged_count` routed from
      `SuppressionCounts.flagged_input` (the int field on the dataclass
      returned by the collector). `SuppressionCounts.rejection_memory` is
      NOT used — it is a producer-attributed suppression concern that
      v0.8.0 does not surface.

    Implementation sketch:
      views, counts = collect_pending_relationship_proposals(
          config, scope=scope, include_flagged=include_flagged
      )
      if limit is not None:
          views = views[:limit]
      findings = [
          SynthesisFinding(
              proposal_id=v.proposal.proposal_id,
              proposal_type="relationship",
              source_id=v.source_id,
              target_id=v.proposal.target_id,
              subject_id=None,
              relation=v.proposal.relation,
              evidence_bundle=list(v.proposal.signals),
              reason_summary=v.proposal.rationale,
              trust_status=derive_trust_status(v),
              producer=v.producer,
              lifecycle_state=v.proposal.lifecycle_state,
              schema_gate_result=None,
              input_fingerprint=v.proposal.basis_fingerprint,
          )
          for v in views
      ]
      return SynthesisReport(
          scope=scope,
          findings=findings,
          excluded_flagged_count=counts.flagged_input,
          trust_override_active=include_flagged,
      )


def render_envelope(report: SynthesisReport) -> dict:
    """Build the shared payload-level JSON envelope.

    Derivation (SF-C1 closure):
      - `schema_version` <- module-level SCHEMA_VERSION constant ("1.0")
      - `command` <- module-level COMMAND_NAME constant ("synthesize")
      - `scope` <- report.scope (None serializes to JSON null per MIN-1)
      - `trust_override_active` <- report.trust_override_active
      - `excluded_flagged_count` <- report.excluded_flagged_count
      - `findings` <- [asdict(f) for f in report.findings] (ordered per §3.3)
    """
    ...
```

### 3.2 Shared-consumer contract wiring

`folio.synthesize.synthesize()` MUST call `folio.tracking.trust.derive_trust_status(view)` directly on each iterated view, **not** via a re-exported alias. `tests/test_trust.py::TestSharedConsumerInvariant::test_synthesize_module_uses_shared_helper` asserts `folio.synthesize.derive_trust_status is folio.tracking.trust.derive_trust_status` (extension of the existing graph invariant), proving both consumers call the same helper.

**§5 contract read path.** Synthesize consumes `collect_pending_relationship_proposals(config, scope=scope, include_flagged=include_flagged)` — the identical entry point `folio.graph.graph_doctor` uses. This is the load-bearing shared-consumer invariant: the function name is the contract boundary, not a string grep. Tests MUST assert the import + call exist (`tests/test_synthesize.py::test_consumes_shared_collect_function`).

**Deferral for future `proposal_type` values (SF-C12 closure).** v0.8.0 handles `proposal_type: "relationship"` only because the read boundary is `collect_pending_relationship_proposals` (the name describes the contract). Entity-merge and diagram-archetype proposals require (a) producers that actually emit those types, AND (b) a new collector (e.g., `collect_pending_proposals(type="merge", ...)`) or a generalized proposal iterator. Neither exists today. When those producers ship, a follow-up slice adds the collector + extends synthesize's render loop; `SynthesisFinding` is forward-compatible (subject_id / source_id / target_id / schema_gate_result / input_fingerprint all accept the needed values).     scope: Optional[str],
    json_output: bool,
    include_flagged: bool,

### 3.5 Stdout renderer (`_render_synthesis_stdout`)

Defined in `folio/synthesize.py` (not `cli.py`, to keep CLI narrow). Closes CB-5 (zero-findings diagnostic breadcrumb) and SF-C6 (count-disclosure wording alignment):

```python
def _render_synthesis_stdout(report: SynthesisReport) -> None:
    """Compact human-readable summary: one line per finding + header + footer.

    Always prints the excluded-flagged-count line (even at 0) to honor parent
    §11 rule 5 purposively (no silent-empty results). Zero-findings +
    zero-exclusions case adds a next-action diagnostic breadcrumb (CB-5).
    """
    scope_label = report.scope if report.scope is not None else "(library-wide)"
    click.echo(f"Synthesis for scope: {scope_label}")
    click.echo(f"Findings: {len(report.findings)}")
    # CB-5 / SF-C6: always print excluded count (even when 0).
    click.echo(
        f"Excluded (flagged inputs): {report.excluded_flagged_count}"
        + (
            " (use --include-flagged to include)"
            if report.excluded_flagged_count
            else ""
        )
    )
    if report.trust_override_active:
        click.echo("Trust override active: --include-flagged")
    # CB-5: zero-findings + zero-exclusions diagnostic breadcrumb.
    if not report.findings and report.excluded_flagged_count == 0:
        click.echo(
            "Next: run `folio ingest` or `folio enrich` if no proposals "
            "have been emitted yet, or check that the scope resolves."
        )
    click.echo("")
    for finding in report.findings:
        flag = " [flagged]" if finding.trust_status == "flagged" else ""
        gate_suffix = ""
        if finding.schema_gate_result is not None:
            rule = finding.schema_gate_result.get("rule", "unknown")
            gate_suffix = f" [schema-gate: {rule}]"
        rel = finding.relation or "<n/a>"
        click.echo(
            f"[{finding.lifecycle_state}{flag}{gate_suffix}] "
            f"{finding.source_id} --{rel}--> "
            f"{finding.target_id or '—'}  ({finding.producer})"
        )
        click.echo(f"  {finding.reason_summary}")
```

**Parent §8 honor (CB-6 closure).** This renderer implements parent §8 "compact summary + trust label + schema warning" partially. It does NOT implement "1–3 evidence locator lines per finding" — deferred to v0.8.1 per §2.5. §1.2 §8 row accurately describes this partial status in v1.1; v1.0's "full evidence-bundle rendering" claim was wrong and is corrected.

## 5. §11 Trust Gating

Synthesize honors parent §11 rules 1–5 (rule 6 — `extraction_confidence` — is not a synthesize concern).

| Rule | How honored |
|---|---|
| 1. `review_status: flagged` inputs excluded by default | `collect_pending_relationship_proposals(include_flagged=False)` already excludes flagged source/target; synthesize passes the flag through. |
| 2. Flagged-dependent proposals may exist but trust posture surfaces explicitly | `trust_status` field on every finding; "flagged" inputs surface "flagged" status; tests verify. |
| 3. Explicit operator override `--include-flagged` | CLI flag; passed to `collect_pending_relationship_proposals`; envelope `trust_override_active` reflects. |
| 4. Surfaces disclose excluded flagged-input counts | `excluded_flagged_count` in envelope (always present) + "Excluded (flagged inputs): N" line in stdout (always printed per CB-5). Routes from `SuppressionCounts.flagged_input`, NOT `SuppressionCounts.total()` (which would conflate rejection-memory suppressions). |
| 5. No surface silently reports empty when exclusion is the cause | Stdout always prints the excluded count (even when 0). Zero-findings + zero-exclusions case also prints a next-action breadcrumb (CB-5) so the operator can diagnose empty-scope vs producers-haven't-run. Tests: SYN-CLI-6 (zero findings with exclusions) + SYN-CLI-14 (zero findings / zero exclusions diagnostic). |
