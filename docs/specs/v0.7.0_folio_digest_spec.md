# v0.7.0 Folio Digest — First Implementation Slice

## 1. Overview

This spec implements the first slice of the `folio digest` command per
`docs/specs/tier4_digest_design_spec.md` (rev 4) — the **daily + weekly,
engagement-scoped digest** described in design §1.

This is the implementation referenced by the parent Tier 4 Shipping Plan
(`docs/specs/tier4_discovery_proposal_layer_spec.md` §15.4 "Slice 4 —
trust-gated surfacing + `--include-flagged`"), narrowed per §15.7 contract
(*slice manifests narrow, they don't reinterpret*). v0.6.4 already shipped
the trust-gating primitives (`_is_flagged`, `--include-flagged` propagation,
suppression disclosure) for the `folio links` family; this slice is the
**first implementation of the `folio digest` command itself**, the consumer
the §11 contract was originally written for. v0.7.0 wires `folio digest`
through those primitives without re-litigating §11 semantics.

This slice is also the first external consumer of the Tier 4 graph-ops +
trust-gated surfacing contract, unblocking parent §15.6 Slice 6b
shared-consumer expansion (`folio synthesize` / `folio search` will follow
the same source-less analysis-doc + `draws_from` pattern this slice
establishes).

**Core behavior:**

1. `folio digest <scope>` produces a deterministic daily synthesis note from
   eligible evidence/interaction docs in the engagement scope.
2. `folio digest <scope> --week` produces a deterministic weekly synthesis
   note from previously-generated daily digests in the same scope.
3. Both outputs register as **source-less managed `analysis` docs** in
   `registry.json` and appear under
   `<engagement-root>/analysis/digests/<digest-id>/<digest-id>.md`.
4. Daily input selection excludes flagged source-backed inputs by default;
   `--include-flagged` widens the predicate and the trust note discloses
   the override.
5. LLM failure after input selection is fatal (exit non-zero, no partial
   output, existing digest preserved).

## 3. Module Structure

### 3.1 `folio/digest.py` public surface

```python
from dataclasses import dataclass
from pathlib import Path
from typing import Literal, Optional

@dataclass
class DigestFlaggedCounts:
    """SF-108: structured counts to prevent positional-tuple swap bugs.
    Mirrors v0.6.4 `SuppressionCounts` rationale (folio/links.py)."""
    excluded: int = 0
    included: int = 0

@dataclass
class DailyInputSelection:
    """SF-108: structured return type for `_collect_daily_inputs`."""
    eligible: list  # list[RegistryEntry]
    counts: DigestFlaggedCounts

@dataclass
class DigestResult:
    status: Literal["written", "rerun", "empty", "error"]
    digest_id: Optional[str]
    path: Optional[Path]
    flagged_counts: DigestFlaggedCounts        # SF-108: replaces v1.1's two adjacent ints
    draws_from_count: int                      # MN-9: structured count for stdout disclosure
    message: str
    exit_code: int  # 0 for written/rerun/empty; non-zero for error

# All §11 failure rows (LLM, write, registry-upsert, parse, scope) collapse into
# status="error" + non-zero exit_code; the `message` field is the discriminator
# for the user-facing diagnostic. (MN-6)

def generate_daily_digest(
    config: FolioConfig,
    *,
    scope: str,
    date: Optional[str] = None,           # YYYY-MM-DD; None → local current date
    include_flagged: bool = False,
    llm_profile: Optional[str] = None,    # routing override per design §4 rule 6
) -> DigestResult: ...

def generate_weekly_digest(
    config: FolioConfig,
    *,
    scope: str,
    date: Optional[str] = None,           # YYYY-MM-DD anchor; None → local current date

    llm_profile: Optional[str] = None,
) -> DigestResult: ...
```

### 3.3 `folio/cli.py` additions

Two narrow modifications to `folio/cli.py`:

**(a) Register the `digest` group** (matches the `provenance` pattern at cli.py:1174-1187 — `ScopeOrCommandGroup` rewrites the first positional into a `--scope` option, so the `@click.option` form is mandatory; see B-001 below):

```python
from . import digest as digest_module

@cli.group(cls=ScopeOrCommandGroup, invoke_without_command=True)
@click.option("--scope", default=None, hidden=True)
@click.option("--date", "date_str", default=None, help="Digest date (YYYY-MM-DD); default local today.")
@click.option("--week", "week_mode", is_flag=True, default=False, help="Generate weekly digest for the ISO week containing --date.")
@click.option(
    "--include-flagged",
    is_flag=True,
    default=False,
    help="Include source-backed inputs whose review_status is flagged (daily mode only; no-op in --week).",
)
@click.option("--llm-profile", default=None, help="Override routing.digest for this invocation.")
@click.pass_context
def digest(ctx, scope, date_str, week_mode, include_flagged, llm_profile):
    """Generate a daily or weekly digest for a single engagement scope.

    Examples:

        folio digest ClientA/DD_Q1_2026
        folio digest ClientA/DD_Q1_2026 --date 2026-04-04
        folio digest ClientA/DD_Q1_2026 --week --date 2026-04-04
        folio digest ClientA/DD_Q1_2026 --include-flagged

    Re-run after upstream notes change to refresh the digest.
    """
    if ctx.invoked_subcommand is not None:
        return
    if scope is None:
        raise click.UsageError("scope is required (e.g. `folio digest ClientA/DD_Q1_2026`)")
    config = ctx.obj["config"]
    fn = digest_module.generate_weekly_digest if week_mode else digest_module.generate_daily_digest

    # SF-103: serialize digest mutations via the existing library lock. Matches
    # the `refresh`, `enrich`, and `convert` patterns. Prevents concurrent
    # `folio digest <same-scope> --date <same-day>` runs from clobbering each
    # other's writes or losing the §10.4 self-heal upsert race.
    from .lock import library_lock, LibraryLockError
    try:
        with library_lock(config.library_root, "digest"):
            result = fn(
                config,
                scope=scope,
                date=date_str,
                include_flagged=include_flagged,
                llm_profile=llm_profile,
            )
    except LibraryLockError as e:
        click.echo(f"✗ {e}")
        ctx.exit(1)

    _emit_digest_result(result, week_mode=week_mode, include_flagged=include_flagged)
    ctx.exit(result.exit_code)
```

`_emit_digest_result(result, *, week_mode, include_flagged)` is a small
local renderer inside the cli.py block (not exported). It owns all
user-facing message rendering per §4 contract — including the
three-branch pluralization (SF-12), `✓`/`✗` prefixes (SF-13), the
`--week --include-flagged` advisory (SF-14), and the
`--include-flagged` echo on empty-no-inputs (SF-15). The renderer
receives the structured `DigestResult` plus the flags so it can compose
the right disclosure suffixes (MN-7).

**Pattern reference correction (B-002, PR-2):** The pattern citation is
`provenance` at cli.py:1174 — `@cli.group(cls=ScopeOrCommandGroup,
invoke_without_command=True) + @click.option("--scope", default=None,
hidden=True)`. Also see `links` at cli.py:1580 for the
`invoke_without_command=False` variant. The v1.0 spec wrongly cited
`enrich` at cli.py:1174; `enrich` is a flat `@cli.command()` at cli.py:1132
and does NOT use `ScopeOrCommandGroup`.

**Why `@click.option("--scope")` not `@click.argument("scope")`
(B-001, PR-1):** `ScopeOrCommandGroup.parse_args` (cli.py:30-33) rewrites
the first non-flag positional into `--scope <arg>` *before* calling
`super().parse_args`. With `@click.argument("scope", required=True)`, the
rewrite leaves the positional empty and click's required-validation fails.
Both existing `ScopeOrCommandGroup` consumers (`provenance` at cli.py:1175,
`links` at cli.py:1581) use `@click.option("--scope", default=None,
hidden=True)` for this reason; required-validation is deferred to the
function body (`if scope is None: raise click.UsageError(...)`).

**(b) Patch `cli.py:1053-1056` for digest-specific refresh guidance (SF-2,
CF-1 closure):** The existing message at cli.py:1053-1056 emits a generic
`"... rerun the originating analysis workflow instead"` for any

## 4. CLI Contract (narrows design §4)

Command surface:

```bash
folio digest <scope> [--date YYYY-MM-DD] [--week] [--include-flagged] [--llm-profile <profile>]
```

| Aspect | Spec |
|---|---|
| `<scope>` | Required positional. Must resolve under exactly one engagement subtree (`library_root/<client>/<engagement>/...` or a path inside that subtree). Error on missing/empty/multi-engagement. |
| `--date` | Optional; format `YYYY-MM-DD`. Omitted = local current date. Invalid format → exit non-zero with explicit error citing `YYYY-MM-DD`. |
| `--week` | Boolean flag. Without it = daily mode. With it = weekly mode anchored on the ISO week containing `--date`. |
| `--include-flagged` | Boolean flag. Daily mode only consults this; weekly mode accepts but does not act on it (daily inputs already filtered at write time). When `--week` AND `--include-flagged` are both passed, `_emit_digest_result` emits a one-line advisory after the success/empty line: `Note: --include-flagged has no effect in --week mode; weekly inputs are existing daily digests, which are already filtered at write time.` (SF-14). The runtime advisory and the static help-text mode-dependency disclosure (`--include-flagged` Click help string per §3.3) MUST agree. |
| `--llm-profile` | Optional. Forwards to `_call_llm` as the `llm_profile` override; existing `LLMConfig.resolve_profile(override=..., task="digest")` precedence applies (config.py:107-162; covers override-bypass → task-route → default-route → first-available fallback). |

**Exit codes:**

- `0` — digest written or rerun-updated, OR no eligible inputs (empty)
- non-zero — invalid input (scope/date), LLM call failed, write error, or
  scope outside library root

**Stdout examples** — pluralization follows the slice 1 / slice 6a three-branch
ladder (SF-12; reference: `cli.py:2536-2541`). Success/update lines use `✓`,
errors use `✗`, advisories use no prefix (SF-13). Continuation lines indent two
spaces (matches `cli.py:358-371` convert/ingest convention).

```
# Successful daily, fresh write, N>1
✓ Wrote daily digest: usbank_techresilience2026_analysis_20260404_daily-digest
  Path: usbank/techresilience2026/analysis/digests/usbank_techresilience2026_analysis_20260404_daily-digest/usbank_techresilience2026_analysis_20260404_daily-digest.md
  Drawn from 7 inputs.

# Successful daily, fresh write, N==1
✓ Wrote daily digest: usbank_techresilience2026_analysis_20260404_daily-digest
  Path: ...
  Drawn from 1 input.
```

## 5. Daily Input Predicate (narrows design §5)

`_collect_daily_inputs(config, scope, day, include_flagged)` returns
`(eligible: list[RegistryEntry], excluded_flagged_count: int, included_flagged_count: int)`.
Algorithm:

1. Load registry via `registry.load_registry(library_root / "registry.json")`.
   On `_corrupt`, call `registry.rebuild_registry` and persist.
2. Resolve `(client, engagement, engagement_root)` from `<scope>` via
   `_resolve_engagement_scope`.
3. Iterate `registry["decks"].values()`. For each entry:
   1. Skip unless `_matches_scope(entry.markdown_path, scope)` returns `True`
      (uses the existing `cli._matches_scope` at cli.py:36-43, which handles
      trailing-slash normalization to prevent prefix collisions like
      `ClientA` matching `ClientA2/...`).
   2. Skip unless `entry.type in {"evidence", "interaction"}` (excludes
      `context` and `analysis` per design §5 to prevent recursion).
   3. Compute the **effective activity date** per design §5:
      - `entry.modified` if non-empty
      - else `entry.converted` if non-empty
      - else skip (ineligible)
      Compare against `day` (local date string `YYYY-MM-DD`).
   4. Read the entry's frontmatter via `registry._read_frontmatter` to
      get the authoritative `review_status` (per v0.6.4 CB-1: registry
      snapshots can be stale; frontmatter is authoritative).
      **Fail-open contract (SF-3 / ADV-001):** if `_read_frontmatter`
      returns `None` (missing file, malformed YAML), or returns a non-dict,
      treat the input as **not flagged** for trust-gating decisions
      (matches v0.6.4 CB-1 fail-open at links.py:113 and `_resolve_target_flagged`).
      The input remains eligible if other predicates pass; the orchestrator
      MAY emit a warning to stderr but does not exclude on read failure.
      Otherwise, if `_is_flagged(fm.get("review_status"))` returns `True`:
      - If `not include_flagged`: increment `excluded_flagged_count` and skip.
      - If `include_flagged`: increment `included_flagged_count` and include.
4. Return `(sorted_by_id(eligible), excluded_flagged_count, included_flagged_count)`.

**Date semantics (SF-5 / ADV-003):**

- Activity-date comparison uses **string equality on `YYYY-MM-DD`**, not
  timestamp parsing. The design spec defines `modified`/`converted` as
  date-only fields. If a value contains a `T` (full ISO timestamp), the
  date portion (`split("T")[0]`) is used **regardless of any timezone
  offset suffix**. If the value cannot be parsed as `YYYY-MM-DD`, the
  entry is skipped (treated as ineligible).
- The `--date` argument and the default-current-date are interpreted as
  the operator's **local-system date** (`datetime.date.today()`). No
  conversion to UTC.
- Cross-timezone notes whose local date differs from their stored UTC
  date may match the digest of a different day (e.g., `modified:
  2026-04-04T23:00:00Z` matches `--date 2026-04-04` because
  `.split("T")[0]` returns `2026-04-04` from the literal string). This is
  intentional for v0.7.0 (single-engagement, single-operator workflow);
  multi-timezone semantics deferred to a later slice if multi-operator
  use surfaces.
- Same date semantics apply to the ISO-week boundary in §6: the week
  range is computed from the literal `--date` string, not from a
  timezone-converted timestamp.

**Other notes:**

- `extraction_confidence` is **not** consulted as a hard exclusion (design
  §5 final paragraph; v0.6.4 contract §11 rule 6).
- File mtime is **not** consulted (design §5 explicit non-goal).
- The empty-result message MUST distinguish the all-empty case from the
  flagged-only-exclusion case per design §5 final paragraph; this distinction
  is materialized in the CLI output (§4 above) using
  `excluded_flagged_count > 0 and len(eligible) == 0` as the discriminator.

## 6. Weekly Discovery Predicate (narrows design §6)

`_collect_weekly_inputs(config, scope, week_monday)` returns
`list[RegistryEntry]`. Algorithm:

1. Load registry; resolve engagement scope (as in §5 step 1–2).
2. Iterate registry entries. For each:
   1. Skip unless `_matches_scope(entry.markdown_path, scope)` returns
      `True` (same helper as §5; MN-5).
   2. Skip unless ALL of:
      - `entry.type == "analysis"`
      - `entry.subtype == "digest"`
      - frontmatter `digest_type == "daily"` (read via
        `registry._read_frontmatter`; if `_read_frontmatter` returns
        `None`/non-dict, skip with diagnostic — weekly inputs MUST
        positively prove `subtype: digest, digest_type: daily,
        digest_period: <valid YYYY-MM-DD>` per ADV-001 second policy)
      - frontmatter `digest_period` is a `YYYY-MM-DD` string in the ISO
        week beginning at `week_monday` (i.e., `week_monday <=
        digest_period <= week_monday + 6 days`)
3. Return `sorted_by_digest_period(eligible)`.

The weekly digest's `digest_period` MUST equal `week_monday` (Monday of the
ISO week, design §6 last sentence).

Empty result: exit `0`, write nothing, message
`No daily digests found for ISO week starting <week_monday> in scope <scope>.`

### 9.4 LLM retry policy (SF-8 / ADV-006)

| Failure class | Retry policy | Implementation |
|---|---|---|
| **Validation retry** — LLM-returned body fails `_validate_body_sections` (missing **LLM-owned** heading OR duplicate **LLM-owned** heading) | One **corrective re-prompt** that names the missing/duplicate heading explicitly. If the second response still fails validation, exit per §11. | Caller (`generate_*_digest`) orchestrates: call `_call_llm`, validate, on failure re-call with corrective prompt, validate again, fail or proceed. |
| **Transient-error retry** — `_call_llm` raises (timeout, rate-limit, 5xx, profile not configured) | **No retry** in v0.7.0 — fail-fast per §11. Carry-forward CF-8 (timeout/backoff in v0.7.x or later). | `_call_llm` raises; caller catches and routes to §11 row 5. |

Per §9.6 ownership split (B-101 fix), `_validate_body_sections` only
validates LLM-owned headings — a compliant LLM that omits `## Trust Notes`
or `## Documents Drawn From` (system-owned) DOES NOT trigger a retry.

The corrective re-prompt for missing-heading is constructed by the caller:
prepend `"Your previous response was missing the following required
section(s): {missing_llm_owned_headings}. Please regenerate the full
digest body with all of these sections (the system will append the
Documents-Drawn-From and Trust Notes sections; you should not emit
them)."` to the original prompt. The corrective re-prompt for
duplicate-heading: `"Your previous response contained more than one
'## {heading}' section. Please emit each required section exactly once."`

`_validate_body_sections` returns both missing AND duplicate (LLM-owned)
headings; the caller decides which corrective prompt applies (or both, if
both kinds of defect occurred — concatenate corrective preambles).

### 9.5 System-owned section scrubbing (SF-7 / ADV-005 / B-101)

Even after the LLM is instructed not to emit system-owned headings, defensive
behavior MUST scrub stray occurrences before system-rendered blocks are
appended. The caller calls:

- `_strip_section(llm_body, "Trust Notes")` (renamed from
  `_strip_duplicate_section` per MN-102 — removes ALL occurrences, not just
  duplicates)
- `_strip_section(llm_body, "Documents Drawn From")` (daily) or
  `_strip_section(llm_body, "Daily Digests Drawn From")` (weekly)

Both `_validate_body_sections` and `_strip_section` are **fence-aware**
(SF-104): they track ` ``` ` and `~~~ ` code-fence boundaries and skip
heading matches inside fenced blocks. Headings with trailing whitespace
or trailing closing-`#` characters (`## Trust Notes ##`, `## Trust Notes  `)
are matched correctly per ATX heading semantics.

This guarantees the final digest body has exactly one of each system-owned
section, in the deterministic position the system places them, regardless
of LLM compliance with §9.6 ownership rules.

| `## What Changed This Week` | Weekly | LLM | Yes |
| `## Cross-Cutting Themes` | Weekly | LLM | Yes |

The LLM prompt (§3.2 `_build_daily_prompt` / `_build_weekly_prompt`)

a compliant LLM that omits Trust Notes no longer triggers a missing-heading
retry, because Trust Notes is not in the validation set.

## 10. Rerun, Atomicity, And Correction Path (narrows design §10)

### 10.1 Rerun semantics

For the same `(scope, period, digest_type)` triple:

1. `_compute_digest_id` and `_compute_digest_path` resolve to the same
   `(digest_id, path)`.
2. If the path exists, `_load_existing_digest` reads the existing
   frontmatter; the new frontmatter inherits `created` and increments
   `version`.
3. The registry entry is upserted (overwrite), with new `modified`.

Result `status` returns `"rerun"` for an updated existing digest;
`"written"` for a fresh path.

### 10.2 Write atomicity (SF-17 / PR-7 / SF-101 — tempfile capture ordering fix)

`_atomic_write(path, content)`:

```python
import tempfile, os
from pathlib import Path

path.parent.mkdir(parents=True, exist_ok=True)
tmp_path: Path | None = None
try:
    # SF-101 (round 2): capture tmp_path BEFORE the write call, so cleanup
    # in the except branch handles write-mid-stream failures (the most common
    # case the cleanup was meant to handle). v1.1's ordering placed
    # `tmp_path = Path(tmp.name)` AFTER `tmp.write()` and inside the `with`,
    # leaving cleanup unreachable when write raised.
    tmp = tempfile.NamedTemporaryFile(
        mode="w", encoding="utf-8", dir=str(path.parent), delete=False, suffix=".tmp"
    )
    tmp_path = Path(tmp.name)
    try:
        tmp.write(content)
    finally:
        tmp.close()
    os.replace(tmp_path, path)
    tmp_path = None  # ownership transferred to target — skip cleanup below
except Exception:
    if tmp_path is not None and tmp_path.exists():
        tmp_path.unlink(missing_ok=True)
    raise
```

Guarantees:
- Same-directory tempfile so `os.replace` is atomic on POSIX (rename within
  filesystem).
- Failure mid-write leaves the existing target untouched (design §10 rule 2).
- No partial / truncated digest can land at the target path (design §10
  rule 3).
- Failure mid-write also leaves no orphan `.tmp` file in the digest folder
  — `tmp_path` is captured BEFORE the write so the except branch can
  always clean up.

Test coverage: DG-20 (LLM failure preserves existing); DG-20b (write-failure
leaves no tempfile orphan — MUST mock `tmp.write` specifically, not the
outer flow, so the test catches the SF-101 regression class).

### 10.3 Correction path

- Operators rerun via `folio digest <scope> [--week] [--date]` after
  upstream notes change. New version increments, audit trail visible in
  frontmatter.
- Operators may delete the digest file and registry entry manually; next
  rerun starts at `version: 1` again.
- Manual edits to digest body are NOT preserved across rerun in v0.7.0
  (design §10 explicit; carry-forward CF-2 below).

### 10.4 Rerun corruption fallback (SF-6 / ADV-004)

| Pre-rerun state | Behavior |
|---|---|
| File exists, frontmatter parses cleanly, `version` is integer | Standard rerun: `new_version = existing_version + 1`, preserve `created`. |
| File exists, frontmatter is malformed YAML (`yaml.safe_load` raises) | Log warning to stderr (`↷ Existing digest at {path} has malformed frontmatter; treating as fresh write.`); proceed as fresh write (`version: 1`, `created = today`). Rationale: digests are regenerable; data loss is bounded to the prior synthesis text and the rerun produces an audit trail via `git diff`. |
| File exists, frontmatter has non-integer `version` | Coerce via `int(value) + 1`; on `ValueError`/`TypeError`, fall back to `version: 1` with stderr warning. |
| File exists, frontmatter missing `created` | Use `today` as `created`. |
| **Orphan: file exists, registry row absent (SF-4 self-heal)** | Before LLM call, build `RegistryEntry` from existing-file frontmatter and call `_register_digest` to upsert. Then proceed with rerun. Eliminates the orphan state. |
| **Stale: registry row exists, file absent** | Treat as fresh write (`version: 1`); registry will be overwritten by the new entry. Stderr warning (`↷ Registry referenced missing file at {path}; regenerating.`). |
| Both file and registry row absent (normal first run) | Fresh write (`version: 1`, `created = modified = today`). |

Test coverage: DG-19 (standard rerun), DG-19b (corrupt YAML rerun), DG-19c
(stale registry — file absent), DG-20c (orphan self-heal).

### 10.5 Concurrency (SF-103 / ADV-103)

The §3.3 CLI handler wraps the entire generate/write/register sequence —
including the §10.4 self-heal step — in `library_lock(library_root,
"digest")` (existing helper at `folio/lock.py`). Matches the `refresh`
(cli.py:978), `enrich` (cli.py:1132), and `convert` patterns. Two
concurrent `folio digest <same-scope>` runs serialize: the second waits
for the first to release, or fails fast with `LibraryLockError` if the
lock is non-blocking (depends on `library_lock` semantics).

Rationale: without the lock, the §10.4 self-heal sequence
(observe-orphan → register → load-existing-version → synthesize → write
→ upsert) is a read-modify-write that two processes can race. Both
observe `version: 2`, both synthesize, both write `version: 3` — last
writer wins, one synthesis is lost. The library-wide lock prevents this
class of race for all digest-mutation paths (fresh write AND rerun AND
self-heal).

Test coverage: DG-CLI-18 — concurrent `folio digest <same-scope>` raises
`LibraryLockError` on the second invocation (or serializes, depending on
the lock helper's blocking semantics; assertion is "second invocation
does not corrupt registry/file state").

## 11. Failure Behavior (narrows design §11)

All error rows render with `✗ ` prefix (SF-13). All failure cases collapse
to `DigestResult.status = "error"` with non-zero `exit_code`; the
`message` field is the user-facing discriminator (MN-6, PR-14).

| Failure point | Behavior | Exit code |
|---|---|---|
| Invalid `<scope>` (out of library root, multi-engagement) | `ValueError` from `_resolve_engagement_scope`; caught at CLI; user-facing error message: `✗ Invalid scope '{scope}': {reason}` | non-zero |
| Invalid `--date` format | `ValueError` from `_parse_date`; caught at CLI; explicit `YYYY-MM-DD` guidance: `✗ Invalid --date '{value}': expected YYYY-MM-DD format.` | non-zero |
| Empty input set (no eligible) | Successful empty result per design §5 / §6; `✓` prefix omitted (informational, not a write success); `--include-flagged` echo per SF-15 if applicable | 0 |
| Empty input set due to flagged exclusion only | Successful empty result with explicit `--include-flagged` guidance per design §5; three-branch pluralization per SF-12 | 0 |
| LLM call raises (timeout, API error, profile not configured) | Exit non-zero; existing digest at target path preserved unchanged; error message: `✗ LLM synthesis failed for {type} digest on {period} in scope {scope} (existing digest, if any, preserved). Cause: {upstream}` | non-zero |
| LLM returns malformed body (missing/duplicate heading after one retry per §9.4) | Exit non-zero; existing digest preserved; error message: `✗ LLM-returned body failed validation after retry: {missing_or_duplicate_headings}. Existing digest, if any, preserved.` | non-zero |
| Atomic write failure (disk full, permission) | Exit non-zero; existing digest preserved (atomic rename never executed); orphan tempfile cleaned up per §10.2; error message: `✗ Atomic write failed at {path}: {os_error}` | non-zero |
| Registry upsert failure (corrupt registry) | Exit non-zero; digest file may exist on disk but registry was not updated; error message: `✗ Registry upsert failed for {digest_id}: {cause}. Run \`folio digest <scope> [--week] [--date]\` again to self-heal (the rerun re-registers any orphan digest, see §10.4). If the registry itself is corrupt, run \`folio status --refresh\` to trigger \`rebuild_registry\` (auto-invoked on _corrupt detection at registry.py:121).` (SF-4 + SF-106: corrected guidance — the prior v1.0 "use folio refresh" guidance was wrong, AND the v1.1 "folio rebuild-registry" CLI does not exist. The actual recovery surfaces are (a) self-heal on rerun and (b) `folio status --refresh` which calls `rebuild_registry` internally. PR-103 round-2 fix.) | non-zero |

Rationale: per design §11, "no digest" is safer than a half-generated
synthesis artifact. The atomic-write contract makes "preserve existing"
trivially true (replace happens last). The self-heal on rerun (§10.4)
makes orphan digest files recoverable without operator intervention
beyond rerunning the same command.

## 12. Trust & Reviewability (narrows design §12)

The first slice's trust posture is hardcoded (not configurable):

| Field | Value | Rationale |
|---|---|---|
| `review_status` | `"flagged"` | Design §12 rule 1: conservative interim posture |
| `review_flags` | `["synthesis_requires_review"]` | Design §12 rule 2: explicit signal to downstream review surfaces |
| Daily input filter | Excludes `review_status: flagged` source-backed inputs by default | Design §12 rule 4 + parent §11 rule 1 |
| `--include-flagged` override | Widens daily input predicate; weekly mode accepts but ignores (no daily-input filter at weekly synthesis time, since weekly inputs are dailies) | Design §12 rule 4 caveat |
| Weekly intermediate | Consumes existing daily digests, not raw notes | Design §12 rule 5; design §6 |
| Trust Notes section | Rendered programmatically (not LLM-trusted), discloses flagged exclusion state, count, and override status | Design §9 + §12 |

`folio digest` does **not** introduce a new trust model. It inherits the
v0.6.4 trust-gating contract (parent §11) for source-backed inputs, and
leaves digest-output trust to the existing `review_status: flagged` +
`synthesis_requires_review` carry-forward. A dedicated Tier 4 synthesis
review model is parent §15.6+ work, deferred from this slice.
