## 5. ID Convention Decision

**Decision: DEFER auto-abbreviation.**

Rationale:

1. Ontology Section 8 shows `ddq126` as abbreviation of "DD Q1 2026" â€” a human judgment call with no deterministic algorithm
2. Current `_generate_id()` (L289-310) uses `_sanitize_name(engagement)` producing `dd_q1_2026` â€” correct, deterministic, reversible
3. Auto-abbreviation would require either a lookup table (config burden) or heuristic compression (fragile)
4. Users who want short IDs should pass `--engagement ddq126`

**Guidance:** define the engagement-short convention once in the engagement Context document and use it consistently across all CLI invocations.

## 6. Backward Compatibility

1. **Additive only.** New fields (`industry`, parameterized `subtype`/`source_type`) are additions. No fields removed or renamed.
2. **Old docs still valid.** Existing frontmatter parses correctly. Missing fields are simply absent.
3. **Dataview queries must be null-safe.** Use `WHERE industry` guard, not bare `contains(industry, ...)`.
4. **Reconversion overwrites frontmatter.** `folio convert` on an existing doc regenerates all fields except `id` and `created` (preserved via `frontmatter.py` L61-66). **Any manually added fields â€” including relationship fields like `relates_to` or `supersedes` â€” will be lost on reconversion.** This is a known limitation: the ontology designates frontmatter as the single source of truth for metadata (Section 2.6, 6.3), but the current pipeline has no mechanism to preserve manually curated fields. Mitigation: back up or extract manual fields before reconverting. A future `folio enrich` command should read and re-inject relationship fields.
5. **No migration tool.** Batch backfill deferred. Users can reconvert via `folio convert` / `folio batch`, or add fields manually. (`folio refresh` is not yet implemented.)

## 7. Deferred Items

| Item | Rationale |
|------|-----------|
| `source_type: report` | Semantic classification, not file-format. Cannot auto-detect from extension. Future: `--source-type` CLI override or inference from `subtype=external_report`. |
| L1+ field enforcement | L0 auto-conversion is intentionally permissive. Enforcement belongs in `folio promote` (not yet built). |
| Relationship fields (`relates_to`, `supersedes`, `instantiates`) | Applicable to evidence but L2+ feature requiring `folio enrich` or manual curation. `depends_on`, `draws_from`, `impacts` are N/A for evidence. |
| Reconversion field preservation | Reconversion overwrites all fields except `id` and `created`. Manually added metadata (including relationships) is lost. Requires `folio enrich` or a preservation mechanism. |
| ID auto-abbreviation | See Section 5. User-input concern. |
| Tag vocabulary validation | Deferred to `folio vocab` command. |
| Frontmatter migration tool | No batch backfill for existing docs. Deferred. |
