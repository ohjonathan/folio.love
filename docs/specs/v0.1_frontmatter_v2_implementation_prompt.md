## Task Context

### What to Build

Close 5 gaps between the Folio Ontology v2 Schema (Section 12) and the frontmatter output generated by `frontmatter.py`. All changes are **additive** ‚Äî no fields are removed or renamed.

| Gap | Summary |
|-----|---------|
| G1 | `source_type` ‚Äî auto-detect `"deck"` or `"pdf"` from file extension instead of hardcoding `"deck"` |
| G2 | `subtype` ‚Äî make configurable via `--subtype` CLI arg instead of hardcoding `"research"` |
| G3 | `industry` ‚Äî add optional `--industry` CLI arg (comma-separated list) |
| G4 | Tag override ‚Äî add `--tags` CLI arg for manual tags merged with auto-generated |
| G5 | Field ordering ‚Äî reorder frontmatter dict by semantic groups |

Additionally, create `tests/test_frontmatter.py` with 25 tests covering all 5 gaps plus existing untested functions (`_collect_unique`, `_generate_tags`).

### What NOT to Build

- No changes to `analysis.py`, `normalize.py`, `images.py`, `text.py` (pipeline stages unchanged)
- No changes to `versions.py`, `sources.py` (tracking systems unchanged)
- No changes to `config.py` (no new config keys)
- No changes to `markdown.py` (receives pre-generated YAML string, does not reference individual fields)
- No new external dependencies
- No relationship fields (`depends_on`, `relates_to`, etc.) ‚Äî L2+ / deferred
- No `source_type: report` detection (semantic classification, cannot auto-detect from extension)
- No L1+ field enforcement (deferred to `folio promote`)
- No tag vocabulary validation (deferred to `folio vocab`)
- No frontmatter migration tool for existing documents
- No ID auto-abbreviation

### YAML Safety

User-provided strings from `--industry` and `--tags` flow through PyYAML and are quoted as needed. No manual escaping required.

The `title` field comes from `converter._title_from_name()` which produces simple title-case words from sanitized names ‚Äî no special characters possible.

## Backward Compatibility

### Fields That Change

| Field | Before | After | Risk |
|-------|--------|-------|------|
| `source_type` | Always `"deck"` | `"deck"` or `"pdf"` (auto-detected) | PDFs previously got `source_type: deck`. Now correctly `pdf`. Dataview queries `WHERE source_type = "deck"` no longer match PDFs. Correctness fix. |
| `subtype` | Always `"research"` | Configurable, default `"research"` | No change for existing workflows (default preserved). |
| Field order | Ad-hoc | Semantic groups | YAML is unordered; Dataview ignores order. Cosmetic diffs only. |

### New Fields

| Field | When Present | Risk |
|-------|-------------|------|
| `industry` | Only when `--industry` provided | Absent on old docs. Dataview queries must guard: `WHERE industry` before `contains()`. |

### No-Change Fields (Must NOT Break)

`id`, `created` (reconversion preservation), `modified`, `converted`, `version`, `grounding_summary`, all reconciliation fields, `tags` (auto-generation still works), `frameworks`, `slide_types`, `client`, `engagement`.

### `generate()` Signature Break

Adding `source_type` as a required parameter breaks callers that don't pass it. Only 5 call sites exist (1 in converter.py, 4 in test_pipeline_integration.py). Step 3 fixes the test calls; Step 1c fixes the converter call.

### Reconversion Warning

Reconversion regenerates ALL fields except `id` and `created`. This is an existing behavior, not new. Manually added fields (including future relationship fields) are lost on reconversion. Known limitation documented in spec Section 6.
