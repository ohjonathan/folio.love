# v0.6.1 Proposal Lifecycle Rename Spec

## 1. Overview

This spec implements **Shipping Plan §15.2** of `docs/specs/tier4_discovery_proposal_layer_spec.md` (rev 5). It renames the persisted **relationship proposal** schema field `status` to `lifecycle_state` and expands the value vocabulary from two states (`pending_human_confirmation`, `rejected`) to six states (`queued`, `accepted`, `rejected`, `suppressed`, `stale`, `superseded`), honoring §6 of the parent proposal.

**Migration-only; no behavior change in isolation.** The four new states enter the enum but gain no writers until later slices. Existing rejection-memory, acceptance-rate, and review-rendering behavior is preserved identically.

**Operator upgrade path:** no action required. `from_dict` reads legacy `status` transparently; `to_dict` emits only `lifecycle_state`. Old and new frontmatter coexist seamlessly. Raw-dict readers in scope all compare to `"rejected"` (unchanged value), so legacy data is handled correctly without value mapping.

## 3. Storage Schema Migration

### 3.1 Field rename

| Layer | Before | After |
|-------|--------|-------|
| Persisted frontmatter key | `status` | `lifecycle_state` |
| Dataclass field | `RelationshipProposal.status` | `RelationshipProposal.lifecycle_state` |
| `to_dict()` output key | `"status"` | `"lifecycle_state"` |

### 3.2 Value mapping

| Old value | New value | Semantics (§6) |
|-----------|-----------|-----------------|
| `"pending_human_confirmation"` | `"queued"` | Eligible for operator review |
| `"rejected"` | `"rejected"` | Explicitly declined by operator |
| *(new)* | `"accepted"` | Promoted to canonical graph state (enum-only; no writer this slice) |
| *(new)* | `"suppressed"` | Hidden by rejection memory or policy (enum-only; no writer this slice) |
| *(new)* | `"stale"` | Input basis changed; prior decision untrustworthy (enum-only; no writer this slice) |
| *(new)* | `"superseded"` | Deterministic rewrite replaced proposal (enum-only; no writer this slice) |

### 3.3 Constant

Defined in `folio/pipeline/enrich_data.py`, exported from the module.

### 3.4 Backward-compatibility read strategy (dataclass path)

`from_dict` reads `lifecycle_state` first; falls back to `status` with value mapping:

```python
_STATUS_TO_LIFECYCLE: dict[str, str] = {
    "pending_human_confirmation": "queued",
    "rejected": "rejected",
}

@classmethod
def from_dict(cls, d: dict) -> RelationshipProposal:
    lifecycle_state = d.get("lifecycle_state")
    if lifecycle_state is None:
        old_status = d.get("status", "pending_human_confirmation")
        lifecycle_state = _STATUS_TO_LIFECYCLE.get(old_status, old_status)
    ...
    return cls(..., lifecycle_state=lifecycle_state, ...)
```

Edge cases:
- Both `lifecycle_state` and `status` present: `lifecycle_state` wins (explicit `is None` check — a present-but-empty-string `lifecycle_state` is treated as valid, not as absent).
- Neither key present: defaults through `d.get("status", "pending_human_confirmation")` → maps to `"queued"`.
- `lifecycle_state` key present with value `None` (YAML `null`): `d.get("lifecycle_state")` returns `None` → falls through to `status` fallback. This is the correct behavior.
- Unknown `lifecycle_state` value (e.g., typo `"queeud"`): passes through as-is. Validation is deferred — this is a migration-only slice and adding input validation would be a behavior change.

`to_dict` emits `lifecycle_state` only. The old `status` key is never written by new code.

### 3.5 Raw-dict reader pattern

In-scope raw-dict readers (`graph.py` line 320, `enrich.py` line 985) both compare to `"rejected"`. Since `"rejected"` maps to `"rejected"` (unchanged value), the fallback pattern handles both old and new frontmatter correctly:

```python
state = raw.get("lifecycle_state")
if state is None:
    state = raw.get("status")
```

This uses the same `is None` guard as `from_dict` (§3.4), avoiding the falsy-coalescing inconsistency that `or` would introduce (where `lifecycle_state: ""` would incorrectly fall through to `status`).

No value mapping is needed in raw-dict readers because the only compared value (`"rejected"`) is unchanged. If a future slice adds a raw-dict reader that compares to `"queued"`, it must apply `_STATUS_TO_LIFECYCLE` mapping or use `from_dict` instead.

### 3.6 Provenance proposal namespace separation

`folio/provenance.py` defines a parallel `ProvenanceProposal` dataclass (line 56) with its own `status` field using the same vocabulary (`"pending_human_confirmation"`, `"rejected"`). Provenance proposals are stored under provenance-pair metadata (`pair.proposals[]`), NOT under `_llm_metadata.<producer>.axes.relationships.proposals[]`. The two proposal types occupy separate frontmatter namespaces and never flow through the same read/write paths.

The relationship-proposal rename in this slice does not affect provenance proposals. The provenance migration is deferred to a follow-up slice (§8, PROV-1).

## 4. Technical Design

### 4.1 `folio/pipeline/enrich_data.py`

1. Add `PROPOSAL_LIFECYCLE_STATES` frozenset after imports (before dataclasses section).

3. Rename `RelationshipProposal.status` field to `lifecycle_state` with default `"queued"`.
4. Update `to_dict()`: emit `"lifecycle_state": self.lifecycle_state` instead of `"status": self.status`.
5. Update `from_dict()`: read `lifecycle_state` with `status` fallback per §3.4.

### 4.2 `folio/enrich.py`

1. **Line 715 (emission):** change `status="pending_human_confirmation"` to `lifecycle_state="queued"`.
2. **Line 985 (emission-time rejected-key filter):** update the raw-dict read inside the existing `isinstance` guard:
   ```python
   if isinstance(p, dict) and (p.get("lifecycle_state") if p.get("lifecycle_state") is not None else p.get("status")) == "rejected":
   ```
   Or equivalently, extract a local for readability:
   ```python
   if isinstance(p, dict):
       state = p.get("lifecycle_state")
       if state is None:
           state = p.get("status")
       if state == "rejected":
           ...
   ```
   The `isinstance(p, dict)` guard MUST remain before any `.get()` call to prevent `AttributeError` on non-dict entries.

### 4.3 `folio/links.py`

1. **Line 169:** `proposal.status != "rejected"` → `proposal.lifecycle_state != "rejected"`.
2. **Line 184:** `proposal.status != "pending_human_confirmation"` → `proposal.lifecycle_state != "queued"`.
3. **Line 371:** `raw_copy["status"] = new_status` → `raw_copy["lifecycle_state"] = new_status`. Callers pass `new_status="rejected"` (from `reject_proposal`) or `new_status=None` (from `confirm_proposal`, which removes the entry). No caller passes `"pending_human_confirmation"`, so no value mapping is needed at this call site.
