# Folio Ontology Architecture

**folio.love**

---

## 1. The Core Problem: Knowledge Decay in Consulting

Consulting knowledge fails in three ways:

| Problem | Description |
|---------|-------------|
| **Engagement Amnesia** | You staffed a similar DD eighteen months ago. The frameworks, hypotheses, and interview guides exist somewhere across old decks, OneNote pages, and email threads. You rebuild from scratch. |
| **Interaction Loss** | You conduct 30 interviews over 6 weeks. The synthesis lives in your head and a final deck. The raw signal, the contradictions, the evolution of your hypothesis — gone. |
| **Format Prison** | Your best thinking is locked in PowerPoint slides that no tool can search, no AI can read, and no graph can connect. |

**The Common Thread:** Consulting knowledge is created in high-density bursts (engagements) but retrieved across long time horizons (careers). The creation format (PPTX, meetings, notes) is optimized for production, not retrieval.

## 2. Design Philosophy

### 2.1 Curation Over Capture

Borrowed directly from Project Ontos. Folio demands deliberate curation but eliminates mechanical ceremony.

| Curation (Human Judgment) | Ceremony (Folio Automates) |
|---------------------------|---------------------------|
| "This deck is about market sizing for retail" | Extract text, generate images, scaffold frontmatter |
| "This interview changed our hypothesis" | Link the interaction to the analysis document |
| "These three decks form a single deliverable" | Maintain bidirectional links, update indexes |
| "Tag this as a reusable framework" | Validate tags against vocabulary, warn on unrecognized terms |

**The Librarian's Wager, Folio Edition:** A curated library of 500 well-tagged assets beats 5,000 untagged files in a shared drive. Every time.

### 2.2 Structure AND Search

Ontos rejects probabilistic retrieval. Folio embraces both:

- **Structure for navigation.** When you know what you're looking for: "show me all DD frameworks I've used for retail clients." Deterministic. Exact. Powered by tags, links, and Dataview.
- **Search for discovery.** When you don't know what you're looking for: "what have I written that's relevant to supply chain resilience?" Probabilistic. Approximate. Powered by semantic search over content + metadata.

Structure is the skeleton. Search is the nervous system. You need both.

### 2.3 Portability Over Platform

Folio outputs are markdown files with images. The viewer is Obsidian today, but the data is yours:

- No database required (flat files + frontmatter)
- Works with any markdown tool
- Travels via git, OneDrive, Dropbox
- AI-native by default (LLMs read markdown natively)

### 2.4 Progressive Formalization

Not everything needs to be perfect on day one. Knowledge matures:

| Level | Name | How It Gets Here | What's Required |
|-------|------|-------------------|-----------------|
| **L0** | Raw | Auto-generated by `folio convert` or `folio ingest` | `id`, `type`, auto-extracted metadata |
| **L1** | Contextualized | Human adds engagement context, purpose | `client`, `engagement` (for engagement-scoped types), basic `tags` |
| **L2** | Connected | Claude enrichment or manual curation | Full tags, relationship links, entity resolution |
| **L3** | Synthesized | Human review + cross-asset linking | Part of synthesis documents, hypothesis chains, verified provenance |

Most assets will live at L1-L2. L3 is reserved for high-value synthesis work.

### 2.5 Authority Tiers

Curation level measures document completeness. Authority measures epistemic weight. They're orthogonal.

| Tier | Name | Meaning | Example |
|------|------|---------|---------|
| **T1** | Captured | One person said it | Interview note, raw meeting note |
| **T2** | Analyzed | Team processed it | Synthesis doc, hypothesis tracker |
| **T3** | Aligned | Team + client agreed | SteerCo deck, approved recommendation |
| **T4** | Decided | Formally signed off | Final deliverable, board presentation |

A perfectly curated (L3) interview note is still T1. A rough (L1) SteerCo deck is T3. Authority is about what the organization has validated, not how complete the metadata is.

**Use case:** When an AI assistant finds conflicting information in the library, authority tier tells it which source wins.

### 2.6 Frontmatter as Source of Truth

**Design decision:** Frontmatter is the single source of truth for all structured metadata, including relationships.

Wikilinks in document bodies are **derived** from frontmatter, not the other way around. `folio enrich` generates a "Related" section from frontmatter fields. If a user manually adds a wikilink in the body, `folio enrich` detects it and offers to promote it to frontmatter.

This prevents sync drift between two representations. One source, one direction.

### 2.7 Trust & Reviewability

Folio's output must be trustworthy enough for direct use in active engagements serving Fortune 100 clients. This introduces a third orthogonal dimension alongside curation level and authority:

| Dimension | Measures | Set By | Example |
|-----------|----------|--------|---------|
| **Curation Level** (L0-L3) | Completeness of human curation | Pipeline default + human promotion | L0 = raw conversion, L2 = enriched with links |
| **Authority** (T1-T4) | Organizational validation weight | Human judgment | T1 = one person said it, T4 = board signed off |
| **Review Status** | Whether the system trusts its own output | Machine + human | `flagged` = system detected issues, `reviewed` = human confirmed |

A document can be L0/T1/clean (raw conversion, no issues detected), L2/T3/flagged (enriched and client-aligned, but a specific data point has low-confidence extraction), or L1/T1/overridden (contextualized interview note where a human corrected an entity name).

**Design principles:**
- The system must know what it doesn't know. Every LLM extraction carries a confidence level. Uncertainty is surfaced, not hidden.
- Every extracted fact traces to its source: which page, which text span, which model, which pass, which extraction method.
- Human corrections persist across re-processing. The system never silently overwrites human judgment.
- Flagged documents are visible in both CLI (`folio status`) and Obsidian (Dataview queries on `review_status`).

See PRD FR-700 for the full requirements specification.

## 3. The Dual Ontology: Space & Time

Adapted from Project Ontos. Consulting knowledge exists in two dimensions:

```
┌─────────────────────────────────────────────────────────────────┐
│                     FOLIO DUAL ONTOLOGY                         │
│                                                                 │
│   SPACE (Knowledge Graph)          TIME (Interaction History)   │
│   "What do we know?"               "What happened?"             │
│                                                                 │
│   ┌────────────┐                   ┌────────────┐               │
│   │  Context   │ Engagement,       │ Meeting    │ Client sync   │
│   │            │ client, SOW       │            │ on 2/13       │
│   └─────┬──────┘                   └─────┬──────┘               │
│         │                                │                      │
│         ▼                                │ impacts              │
│   ┌────────────┐                         │                      │
│   │  Analysis  │ Issue trees,            ▼                      │
│   │            │ hypotheses        ┌────────────┐               │
│   └─────┬──────┘                   │ Hypothesis │ Updated       │
│         │                          │ Doc v3     │ after new     │
│         ▼                          └────────────┘ evidence      │
│   ┌────────────┐                                                │
│   │  Evidence  │ Data, research,   ┌────────────┐               │
│   │            │ external sources  │ Interview  │ Expert call   │
│   └─────┬──────┘                   │            │ with CTO      │
│         │                          └─────┬──────┘               │
│         ▼                                │                      │
│   ┌────────────┐                         │ impacts              │
│   │ Deliverable│ Decks, memos,           ▼                      │
│   │            │ recommendations   ┌────────────┐               │
│   └────────────┘                   │ Market     │ Added new     │
│                                    │ Sizing v2  │ data point    │
│   ┌────────────┐                   └────────────┘               │
│   │ Reference  │ Templates,                                     │
│   │            │ firm frameworks                                │
│   └────────────┘                                                │
│                                                                 │
│   ┌────────────────────────────────────────────────────────┐    │
│   │ ENTITIES (Graph Nodes)                                  │    │
│   │ Person, Department, System, Process                     │    │
│   │ Referenced by documents, not documents themselves       │    │
│   └────────────────────────────────────────────────────────┘    │
│                                                                 │
│   Navigation: depends_on, draws_from    Bridge: impacts         │
│   Changes: Deliberate updates           Changes: Accumulation   │
└─────────────────────────────────────────────────────────────────┘
```

### Why Two Dimensions?

| Aspect | Space | Time |
|--------|-------|------|
| **Contains** | What we know right now | What happened and when |
| **Document types** | Context, Analysis, Evidence, Deliverable, Reference | Interaction (meetings, interviews, calls) |
| **Mutability** | Updated as understanding evolves | Immutable records of events |
| **Relationships** | `depends_on`, `draws_from`, `relates_to` | `impacts` (interaction → space document) |
| **Query pattern** | "What informs this deliverable?" | "What meetings shaped this hypothesis?" |
| **Consulting analogy** | The current state of the case | The interview log and meeting history |

## 4. Document Types

### 4.1 Space Types (The Knowledge Graph)

**Context** — The engagement scaffolding. Why are we here, for whom, doing what.

```yaml
---
id: clienta_ddq126_context_20260120_engagement
type: context
subtype: engagement
status: active
authority: aligned
curation_level: L1
client: ClientA
engagement: Due Diligence Q1 2026
industry: [retail, ecommerce]
service_line: due_diligence
team: [Johnny, Sarah, Mike]
sow_reference: "SOW-2026-0142"
tags: [dd, retail, ecommerce, growth-equity]
---
```

Subtypes: `engagement`, `client_profile`, `workstream`

`engagement` field: Optional for Context. A `client_profile` spans engagements; an `engagement` context obviously has one.

---

**Analysis** — The thinking work. Hypotheses, issue trees, frameworks applied, synthesis.

```yaml
---
id: clienta_ddq126_analysis_market-hypothesis
type: analysis
subtype: hypothesis
status: active
authority: analyzed
curation_level: L2
client: ClientA
engagement: Due Diligence Q1 2026
depends_on: [clienta_ddq126_context_20260120_engagement]
draws_from: [clienta_ddq126_interview_20260213_01, retail_market_sizing_2025]
tags: [market-sizing, tam, growth-hypothesis, competitive-dynamics]
frameworks: [tam-sam-som, porter-five-forces]
---
```

Subtypes: `hypothesis`, `issue_tree`, `synthesis`, `framework_application`, `digest`

The `digest` subtype covers temporal roll-ups: daily digests, weekly digests, and SteerCo input documents. A digest `draws_from` the interactions and analyses it summarizes. See Section 9 for the roll-up pipeline.

---

**Evidence** — What we found. Data, research, external inputs. Converted decks land here.

```yaml
---
id: clienta_ddq126_evidence_market-sizing
type: evidence
subtype: research
status: active
authority: captured
curation_level: L1
client: ClientA
engagement: Due Diligence Q1 2026
source_type: deck
source: ../sources/ClientA/DD_Q1/market_sizing_v3.pptx
source_hash: abc123def456
industry: [retail]
tags: [market-sizing, retail, tam, competitive-landscape]
frameworks: [tam-sam-som, porter-five-forces]
slide_count: 24
---
```

Subtypes: `research`, `data_extract`, `external_report`, `benchmark`

---

**Deliverable** — What we said. The output artifacts.

```yaml
---
id: clienta_ddq126_deliverable_final-deck
type: deliverable
subtype: deck
status: active
authority: decided
curation_level: L2
client: ClientA
engagement: Due Diligence Q1 2026
depends_on: [clienta_ddq126_context_20260120_engagement]
draws_from:
  - clienta_ddq126_analysis_market-hypothesis
  - clienta_ddq126_analysis_ops-assessment
  - clienta_ddq126_evidence_market-sizing
tags: [dd, final-deliverable, board-presentation, investment-thesis, risk-assessment]
frameworks: [scr, mece]
---
```

Subtypes: `deck`, `memo`, `model`, `recommendation`

---

**Reference** — Reusable firm knowledge. Templates, methodologies, evergreen frameworks.

```yaml
---
id: ref_dd-interview-guide-template
type: reference
subtype: template
status: active
authority: aligned
curation_level: L2
service_line: due_diligence
tags: [dd, interview-guide, template, reusable, hypothesis-testing]
---
```

Subtypes: `template`, `methodology`, `firm_framework`, `playbook`

`engagement` field: Not applicable. Reference docs are engagement-independent by definition.

---

### 4.2 Time Type (The Interaction History)

**Interaction** — What happened. Meetings, interviews, calls. Immutable once created.

```yaml
---
id: clienta_ddq126_interview_20260213_01
type: interaction
subtype: expert_interview
status: complete
authority: captured
curation_level: L0
date: 2026-02-13
client: ClientA
engagement: Due Diligence Q1 2026
participants: [Johnny Oh, Jane Smith (CTO)]
duration_minutes: 45
impacts: []
tags: [interview, expert, technology, infrastructure]
source_transcript: ../transcripts/2026-02-13_expert_interview_01.md
source_recording: ../recordings/2026-02-13_recording_22.wav
---

## Summary
[LLM-generated narrative summary of what happened and key takeaways]

## Key Findings
[Structured extraction: claims with attribution, data points, decisions, open questions]

## Quotes / Evidence
[Notable statements with timestamps]

## Impact on Hypotheses
[STUB at L0 — filled during L0→L1 promotion by human, refined at L2 by enrichment]
[Prompt: Did this interaction change, support, or challenge any active hypotheses?]
```

Subtypes: `client_meeting`, `expert_interview`, `internal_sync`, `partner_check_in`, `workshop`

**L0 behavior:** `folio ingest` generates Summary, Key Findings, and Quotes automatically. The "Impact on Hypotheses" section is created as an empty stub. The LLM doesn't have library context at ingest time to judge hypothesis impact reliably. That judgment happens at L1 (human) or L2 (enrichment with full library context).

**Immutability:** Once an interaction is created, the record of what happened doesn't change. What changes is the metadata: tags get refined, `impacts` links get added, curation level increases. The content sections are append-only after L0.

## 5. Entity Graph

Documents reference people, departments, systems, and processes. These entities appear across multiple documents and connect them. Entities are **graph nodes, not documents.** They exist in the type system but follow different rules.

### 5.1 Entity Types

| Type | Examples | Key Properties |
|------|----------|---------------|
| **Person** | Jane Smith (CTO), Mike Chen (Partner) | name, title, department, reports_to, client |
| **Department** | Engineering, Finance, Operations | name, head, client |
| **System** | SAP ERP, Salesforce CRM, Legacy Billing | name, owner_dept, status |
| **Process** | Order-to-Cash, Monthly Close, Hiring Pipeline | name, owner_dept, status |

### 5.2 Implementation

Entities are stored in a lightweight registry (JSON or markdown directory). Not full Folio documents with frontmatter. Their purpose is name resolution and cross-document linking, not standalone knowledge.

**Person node example:**
```yaml
# entities/people/jane_smith.yml (or registry.json entry)
name: Jane Smith
aliases: [Jane, the CTO, J. Smith]
title: Chief Technology Officer
department: Engineering
reports_to: Bob Johnson (CEO)
client: ClientA
first_seen: clienta_ddq126_interview_20260213_01
```

### 5.3 Entity Extraction

During `folio ingest` and `folio enrich`, the LLM:
1. Identifies all people, departments, systems, and processes mentioned
2. Resolves names against the entity registry ("Jane," "Jane Smith," "the CTO" → `[[Jane Smith]]`)
3. Creates wikilinks in the document body
4. Adds new entities to the registry with a `needs_confirmation` flag for ambiguous matches

**Name resolution approach (v1):** Exact match against registry + LLM-proposed soft match with human confirmation. No fuzzy matching algorithm. Keep it simple, let humans resolve ambiguity.

### 5.4 Entity Queries

Flat entity queries are supported from Tier 3:
- "All interviews mentioning Engineering department" (tag/text search)
- "All interactions with Jane Smith as participant" (frontmatter query)
- "What did Operations people say about the ERP migration?" (entity + text search)

Recursive queries ("everyone reporting to the CFO, two levels deep") require org traversal that exceeds Dataview's capabilities. This is deferred to Tier 4 and may require a custom query engine. Flat queries cover 80% of real use cases.

### 5.5 Org Chart Import

```bash
folio entities import org_chart.csv
```

Expected CSV format: `name, title, department, reports_to`. Populates the entity registry with `reports_to` hierarchy. Enables the `reports_to` traversal queries if/when that infrastructure is built.

## 6. Relationship Types

### 6.1 Structural Relationships (Space → Space)

| Relationship | Meaning | Example |
|-------------|---------|---------|
| `depends_on` | Cannot exist without this context | Analysis depends on engagement context |
| `draws_from` | Uses as input or evidence | Deliverable draws from analysis and evidence |
| `relates_to` | Topically connected, no dependency | Two analyses across different clients on same topic |
| `supersedes` | Replaces a previous version | Hypothesis v2 supersedes hypothesis v1 |
| `instantiates` | Applies a reference template | Interview guide instantiates DD template |

### 6.2 Temporal Relationships (Time → Space)

| Relationship | Meaning | Example |
|-------------|---------|---------|
| `impacts` | This interaction changed this document | Interview impacts hypothesis document |

### 6.3 Implementation

**Frontmatter is the source of truth.** Relationships are stored as ID references:

```yaml
depends_on: [clienta_ddq126_context_20260120_engagement]
draws_from: [clienta_ddq126_evidence_market-sizing, clienta_ddq126_interview_20260213_01]
relates_to: [clientb_ddq425_analysis_market-hypothesis]
```

**Wikilinks are derived.** `folio enrich` generates a "Related" section in the document body from frontmatter:

```markdown
## Related
- Depends on: [[clienta_ddq126_context_20260120_engagement|ClientA DD Engagement Context]]
- Draws from: [[clienta_ddq126_evidence_market-sizing|Retail Market Sizing]]
- See also: [[clientb_ddq425_analysis_market-hypothesis|Similar hypothesis for ClientB]]
```

If a user manually adds a wikilink in the body, `folio enrich` detects it and offers to promote it to frontmatter. The reverse is never true: frontmatter always wins.

### 6.4 v1 Recommendation

Start with `depends_on`, `draws_from`, and `impacts`. Add `relates_to`, `supersedes`, and `instantiates` when real usage creates the need.

Actual v1 sequencing is corpus-driven. PR C shipped `supersedes` first because
the registry-managed corpus currently contains evidence and interaction notes,
and `supersedes` is the only provenance-relevant relationship that is legal on
evidence notes. PR D therefore seeds provenance from confirmed `supersedes`
pairs in v1; `depends_on` and `draws_from` provenance remains v2 work when
analysis and deliverable notes enter the registry.

## 7. Tag Vocabulary

Single unified field. No separate `concepts` and `tags`. One field, soft validation.

**Design decision:** The original ontology had two fields: `tags` for retrieval-oriented keywords and `concepts` for vocabulary-controlled semantic terms. In practice, the distinction doesn't survive engagement pressure. At 11pm between calls, no one pauses to ask "is market-sizing a tag or a concept?" Merged into `tags` with tool-level vocabulary validation.

### 7.1 Vocabulary Domains

**Industry:** `retail`, `ecommerce`, `healthcare`, `financial-services`, `technology`, `manufacturing`, `energy`, `media`, `telecom`, `logistics`

**Methodology:** `due-diligence`, `market-sizing`, `competitive-dynamics`, `operational-assessment`, `digital-transformation`, `cost-optimization`, `growth-strategy`, `org-design`, `tech-strategy`, `post-merger-integration`

**Framework:** `mece`, `scr`, `pyramid-principle`, `porter-five-forces`, `tam-sam-som`, `value-chain`, `3-horizons`, `bcg-matrix`, `jobs-to-be-done`

**Analytical:** `hypothesis-testing`, `issue-tree`, `root-cause`, `benchmarking`, `scenario-analysis`, `sensitivity`, `cohort-analysis`, `unit-economics`

### 7.2 Governance

- Prefer existing terms over new synonyms
- Compound terms use hyphens: `market-sizing` not `market sizing`
- `folio` CLI warns on unrecognized tags (soft validation, not blocking)
- Vocabulary is additive; removal requires manual cleanup of existing assets
- No hierarchy among tags; all are flat for maximum queryability

## 8. ID Convention

**Design decision:** IDs are date-based and mechanical. Human-readable names go in `title`, not `id`.

### 8.1 Pattern

```
{client}_{engagement-short}_{type-short}_{date}_{descriptor-or-sequence}
```

**Components:**

| Component | Format | Example |
|-----------|--------|---------|
| client | lowercase, short | `clienta` |
| engagement-short | abbreviated engagement + period | `ddq126` (DD Q1 2026) |
| type-short | abbreviated doc type | `interview`, `evidence`, `analysis`, `deliverable`, `context` |
| date | YYYYMMDD | `20260213` |
| descriptor or sequence | short label or zero-padded number | `cto-jane`, `01`, `market-sizing` |

### 8.2 Examples

```
clienta_ddq126_interview_20260213_01          # First interview on Feb 13
clienta_ddq126_interview_20260213_02          # Second interview that day
clienta_ddq126_evidence_20260210_market-sizing # Converted deck
clienta_ddq126_analysis_20260215_investment-thesis # Hypothesis doc
clienta_ddq126_deliverable_20260301_final-deck # Final deliverable
clienta_ddq126_context_20260120_engagement     # Engagement context
ref_dd-interview-guide-template                # Reference (no client, no date)
```

### 8.3 Rules

- IDs are **immutable** once created. Renaming breaks every link in the graph.
- Dates use creation date, not the date of the event (for interactions, `date` field in frontmatter captures the event date).
- Engagement-short should be consistent within an engagement. Define it once in the Context document.
- Reference docs use `ref_` prefix, no client or engagement component.
- Client profiles use `{client}_profile_{descriptor}`.
- `folio convert` and `folio ingest` auto-generate IDs following this convention.

## 9. The Temporal Pipelines

### 9.1 Interaction Pipeline

How a meeting becomes a knowledge asset:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   CAPTURE    │     │   CONVERT    │     │ CONTEXTUALIZE│     │   CONNECT    │
│              │     │              │     │              │     │              │
│ Record audio │────►│ Transcribe   │────►│ Add client,  │────►│ Link to      │
│ Take notes   │     │ Summarize    │     │ engagement,  │     │ hypotheses,  │
│ in OneNote   │     │ (LLM)       │     │ participants │     │ analyses,    │
│              │     │ Structure    │     │ Tag          │     │ evidence     │
│              │     │ output .md   │     │ Fill impact  │     │              │
│ [Manual]     │     │ [folio       │     │  stubs       │     │ [Claude      │
│              │     │  ingest]     │     │ [Human       │     │  enrichment] │
│              │     │              │     │  curation]   │     │              │
│              │     │  ► L0        │     │  ► L1        │     │  ► L2        │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

#### folio ingest

```bash
# Ingest a transcript + optional raw notes
folio ingest transcript.md --notes raw_notes.md --type expert_interview

# Ingest from audio (transcribe + summarize) — future capability
folio ingest recording.wav --type client_meeting --client ClientA

# Batch ingest meeting notes
folio ingest ./meeting_notes/ --pattern "*.md" --type internal_sync
```

Output: Structured interaction markdown at L0. Summary, Key Findings, and Quotes are auto-generated. Impact on Hypotheses is an empty stub.

### 9.2 Temporal Roll-Up Pipeline

How daily activity becomes strategic synthesis:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   NOTE       │     │   DAILY      │     │   WEEKLY     │     │   STEERCO    │
│   LEVEL      │     │   DIGEST     │     │   DIGEST     │     │   INPUT      │
│              │     │              │     │              │     │              │
│ "What was    │────►│ "What moved  │────►│ "Where do we │────►│ "What does   │
│  said"       │     │  forward     │     │  stand and   │     │  leadership  │
│              │     │  today"      │     │  what's      │     │  need to     │
│              │     │              │     │  changed"    │     │  decide"     │
│ Interactions │     │              │     │              │     │              │
│ & evidence   │     │ folio digest │     │ folio digest │     │ Human +      │
│              │     │              │     │ --week       │     │ folio assist │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

**Each level is a different analytical altitude, not compression.** A weekly digest doesn't summarize daily digests. It synthesizes what changed at the engagement level across a week.

Roll-ups are Analysis documents with subtype `digest`:

```yaml
---
id: clienta_ddq126_analysis_20260214_daily-digest
type: analysis
subtype: digest
status: complete
authority: analyzed
curation_level: L1
client: ClientA
engagement: Due Diligence Q1 2026
draws_from:
  - clienta_ddq126_interview_20260214_01
  - clienta_ddq126_interview_20260214_02
  - clienta_ddq126_evidence_20260214_financials
tags: [digest, daily, dd]
digest_period: 2026-02-14
digest_type: daily
```

## 11. Query Patterns

### 11.1 Structural Queries (Dataview)

```
# All evidence for a specific engagement
TABLE source_type, frameworks, status
FROM "ClientA/DD_Q1"
WHERE type = "evidence"
SORT modified DESC

# All hypotheses across clients
TABLE client, status, authority, modified
WHERE type = "analysis" AND subtype = "hypothesis"
SORT modified DESC

# Reusable frameworks applied to retail
TABLE client, engagement, frameworks
WHERE contains(tags, "retail") AND type = "evidence"
FLATTEN frameworks

# All T3+ authority documents (team/client validated)
TABLE type, client, authority
WHERE authority = "aligned" OR authority = "decided"
SORT modified DESC
```

### 11.2 Temporal Queries (Interaction History)

```
# All interactions for an engagement, chronological
TABLE date, subtype, participants
WHERE type = "interaction" AND engagement = "Due Diligence Q1 2026"
SORT date ASC

# What changed this hypothesis?
TABLE date, subtype, summary
WHERE contains(impacts, "clienta_ddq126_analysis_20260120_investment-thesis")
SORT date ASC

# This week's digests
TABLE digest_type, digest_period
WHERE subtype = "digest" AND digest_period >= date(2026-02-17)
SORT digest_period ASC
```

### 11.3 Entity Queries (Flat)

```
# All interactions with a specific participant
TABLE date, subtype, participants
WHERE type = "interaction" AND contains(participants, "Jane Smith")
SORT date ASC

# All documents mentioning Engineering (requires tag or full-text search)
# Via Obsidian search: tag:engineering OR "Engineering department"
```

### 11.4 Semantic Queries (Search — Tier 4)

These go beyond what Dataview can handle:

- "What have I seen about supply chain resilience across all clients?"
- "Find frameworks similar to this issue tree"
- "What interview findings contradict our current hypothesis?"

Semantic search operates over full text content + metadata, complementing structural queries. Architecture decisions (embedding model, vector store, query interface) are deferred until query patterns are clear from real usage.

### 12.7 Trust & Reviewability Fields

| Field | Type | Applies To | Required |
|-------|------|------------|----------|
| `review_status` | enum | all | Auto (`clean` default) |
| `review_flags` | list[string] | all | Auto (empty default) |
| `extraction_confidence` | float or null | all with LLM analysis | Auto |
| `grounding_summary` | object | evidence, interaction, analysis | Auto |

`review_flags` may include provenance-specific additive signals such as
`provenance_link_stale`. These flags do not by themselves require
`review_status: flagged`.

**`review_status` values:**

| Value | Meaning | Set By |
|-------|---------|--------|
| `clean` | No issues detected | Pipeline (default) |
| `flagged` | System detected issues requiring human attention | Pipeline (automatic) |
| `reviewed` | Human has reviewed and confirmed content | Human |
| `overridden` | Human has corrected system output | Human (via `.overrides.json`) |

**Auto-flagging triggers** (pipeline sets `review_status: flagged` when any apply):
- Any extraction claim has an unvalidated quoted span (not found in source text)
- Any extraction claim has low confidence
- LLM analysis was unavailable (degraded conversion)
- Aggregate `extraction_confidence` is below the configured threshold (default: 0.6)

**`grounding_summary` structure:**

```yaml
grounding_summary:
  total_claims: 12
  high_confidence: 8
  medium_confidence: 3
  low_confidence: 1
  validated: 10
  unvalidated: 2
  pass_1_claims: 9
  pass_2_claims: 3
  pass_2_slides: 2
```

This is a queryable summary only. Full per-claim evidence (quoted spans, element types, confidence levels, pass numbers) lives in the markdown body alongside each slide's analysis section.

**`extraction_confidence` computation:**

The aggregate score is derived from per-claim evidence:
- All claims high-confidence and validated: approaches 1.0
- Mix of high and medium: 0.6-0.8 range
- Any low-confidence or unvalidated: below 0.6
- No LLM analysis: `null`

The exact formula is an implementation detail subject to empirical calibration. The requirement is that the score exists, is queryable, and is directionally meaningful.

**Interaction with `folio promote`:**

Promotion to L1+ may require `review_status != flagged`. A document with unresolved quality issues should not be promoted to a higher curation level without the human explicitly reviewing the flagged items first.

## 14. What Folio Borrows from Ontos

| Ontos Concept | Folio Adaptation |
|---------------|------------------|
| Dual Ontology (Space + Time) | Space = Knowledge Graph (5 types), Time = Interactions |
| Curation vs Ceremony | Same philosophy, applied to consulting workflows |
| Progressive Formalization (L0-L2) | Extended to L0-L3 with consulting-specific gates |
| Concept Vocabulary | Simplified to unified `tags` with soft vocabulary validation |
| `impacts` bridge | Interactions impact Space documents identically |
| `depends_on` structural | Adapted as multiple relationship types for consulting's looser hierarchy |
| Deterministic Purity | **Rejected.** Folio uses structure + semantic search |

## 15. What Folio Does Differently

| Dimension | Ontos | Folio |
|-----------|-------|-------|
| **Domain** | Software projects | Consulting engagements |
| **Type hierarchy** | kernel → strategy → product → atom (strict rank) | context, analysis, evidence, deliverable, reference (flexible graph) |
| **Relationships** | `depends_on` only (acyclic by rank) | Multiple typed relationships (`depends_on`, `draws_from`, `relates_to`, `supersedes`, `instantiates`) |
| **Search** | Rejected | First-class feature alongside structure |
| **Time dimension** | Session logs | Interactions with structured output + temporal roll-ups |
| **Synthesis** | Not addressed | Core feature: hypothesis tracking, interview synthesis, cross-asset patterns |
| **Source materials** | Code files | PPTX, PDF, transcripts, audio |
| **Scale** | Per-project | Cross-client, cross-engagement, career-spanning |
| **Authority** | Not addressed | Four-tier epistemic weight system |
| **Entities** | Not addressed | People, departments, systems as graph nodes |
