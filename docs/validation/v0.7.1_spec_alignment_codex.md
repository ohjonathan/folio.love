# Alignment Review — folio-graph-generalized-proposals-v0-7-1 / B.1 / codex-as-alignment

## 1. Architecture compliance

The artifact is directionally aligned with the parent architecture boundary: it keeps proposal findings non-canonical and does not propose canonical graph writes. However, it does not faithfully carry forward every parent contract it touches. The parent §15.7 requires each slice Phase A spec to declare which parent §§ it implements and defers; the spec's §2.2 table omits §11 and §12 even though the artifact derives trust posture and exists specifically to make `folio graph` consume the shared proposal contract. It also omits the `folio graph` row of §15.6 while deferring only the other two shared-consumer sub-slices.

## 2. Diagram-architecture cross-reference

No diagrams are present in the artifact under review, and none are required by the v0.7.1 spec. No diagram/prose mismatch was found.

## 3. Roadmap alignment

The manifest and parent Shipping Plan identify this deliverable as Shipping Plan §15.6 slice 6b sub-slice 1: `folio graph` consumes the shared proposal contract, while `folio synthesize` and `folio search` follow later. The spec correctly defers `folio synthesize` and `folio search`, but its carry-forward table does not explicitly disposition the in-scope `fully generalized folio graph` consumer named by parent §15.6. That is a narrowing-table defect, not a product-roadmap halt.

## 5. Backward compatibility

The spec acknowledges a breaking `graph doctor --json` semantic change for `subject_id`, but it contradicts itself. §2.1 says legacy keys including `subject_id` are retained unchanged, while §3.1 says legacy `subject_id` stops carrying `proposal_id` and becomes `None` for relationship proposals. The latter may be the right §5 alignment, but the artifact should not call the change purely additive or unchanged.

## 6. Consistency check

The main consistency defect is the §7 mapping. Parent §7 defines `input_fingerprint` as the material proposal identity and invalidation basis. The v0.7.1 spec maps it directly to `view.proposal.basis_fingerprint`, but current relationship proposal mechanics keep claim identity components outside `basis_fingerprint` in `_build_rejection_key`. This silently narrows the parent contract into a smaller field without authority.

The §8 consistency issue is also material: the spec says it implements the Review Rendering Contract, but its implementation section only extends the finding dict / JSON payload. The default operator-facing CLI row remains outside the edit plan and does not render the trust label, schema warning, or evidence locators.
