# v0.6.4 Trust-Gated Surfacing Spec Adversarial Review

## Verdict

Needs Fixes. The spec correctly identifies the `review_status: flagged` trust
gate, but it leaves several operator-visible paths where flagged proposals can
be silently hidden, misreported, or accidentally acted on when registry state is
stale. The failures are not edge-polish: they hit the parent contract's rules
that flagged inputs are excluded by default, excluded counts are disclosed, and
surfaces do not silently report empty when flagged inputs are the cause.

## Blockers

### ADV-001: Target gating uses stale registry state while source gating uses live frontmatter

The spec defines source flagged detection from the source document frontmatter
and target flagged detection from `registry_data["decks"]`:
`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:114-117`. That creates two
different sources of truth in one trust decision.

This is not safe in the current codebase. `registry_mod.load_registry()` only
loads JSON and does not reconcile frontmatter before returning
(`folio/tracking/registry.py:81-101`). The registry module explicitly treats
`review_status` as frontmatter-authoritative during sync/reconciliation
(`folio/tracking/registry.py:244-255`). Therefore the spec can observe:

- Source status: current frontmatter from `registry_mod._read_frontmatter()`.
- Target status: stale registry row from an earlier sync.

Break cases:

- Target frontmatter is changed to `review_status: flagged`, but registry still
  says clean. Default `folio links review` surfaces the proposal, and
  `folio links confirm <proposal_id>` or `folio links confirm-doc <doc_id>` can
  confirm it without any `--include-flagged` consent path.
- Target frontmatter is changed back to clean, but registry still says flagged.
  The proposal is suppressed by default and, when included, is annotated as
  `(flagged: target)` even though the target is no longer flagged.
- A missing target registry row fails open because the lookup defaults to `{}`.
  If the target markdown exists and is flagged but the registry is stale or
  incomplete, the trust gate misses it.

This directly violates the parent rule that `review_status: flagged` inputs are
excluded by default (`docs/specs/tier4_discovery_proposal_layer_spec.md:305`) and
creates the accidental confirm bypass the slice is supposed to prevent.

Required fix: define one authoritative source for target trust state. Given the
existing registry contract, `collect_pending_relationship_proposals` should use
the registry only to locate the target markdown path, then read the target
frontmatter for `review_status`. If the spec intentionally wants registry
authority, it must first change the registry contract and require a sync or
freshness guarantee before trust-gated surfaces run. Add stale-registry tests for
both directions: registry clean / target frontmatter flagged, and registry
flagged / target frontmatter clean.

### ADV-002: `folio links status` silently reports empty or zero when flagged proposals exist

The spec's carry-forward says no additional work is needed for
`folio links status` because flagged proposals are excluded from pending counts:
`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:241-244`. That violates the
parent contract, which requires excluded flagged-input counts and forbids silent
empty results (`docs/specs/tier4_discovery_proposal_layer_spec.md:310-312`).

Current status code discards the suppression counts returned by
`collect_pending_relationship_proposals` (`folio/links.py:236`) and only creates
a row when `pending` or `confirmed` is nonzero (`folio/links.py:251-253`). The
CLI then prints `No document-level relationship data found.` when there are no
rows (`folio/cli.py:1639-1642`).

Break case:

1. A source document has three queued relationship proposals.
2. All three target a flagged document, or the source itself is flagged.
3. Under the spec, `relationship_status_summary()` receives zero pending views
   and throws away `suppression_counts["flagged_input"] == 3`.
4. `folio links status` exits 0 with either no data or a table showing
   `0 pending`, depending on confirmed relationship count.

The operator is told the queue is empty when the real cause is trust gating.
That is the exact silent-empty failure rule 5 forbids.

Required fix: extend the status summary model to carry flagged-excluded counts
per source and in total. The CLI can keep the existing `Pending` count as
"reviewable pending", but it must disclose hidden flagged proposals, for example
`3 proposal(s) excluded from status counts because source or target has
review_status: flagged. Use folio links review --include-flagged to inspect.`

### ADV-003: `confirm-doc` and `reject-doc` return successful zero counts when flagged inputs are the real cause

The spec says `confirm_doc` and `reject_doc` should call
`collect_pending_relationship_proposals()` with default `include_flagged=False`
and calls this intentional:
`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:193-199` and
`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:246-249`.

The implementation shape means bulk commands will silently succeed with zero
work. `confirm_doc()` and `reject_doc()` collect default pending views, loop over
the resulting proposal IDs, and return the count (`folio/links.py:426-441`).
The CLI prints a success checkmark with that count (`folio/cli.py:1705-1711` and
`folio/cli.py:1723-1729`).

Break case:

- `doc_a` is flagged and has five queued proposals.
- Operator runs `folio links confirm-doc doc_a`.
- The trust gate filters all five out.
- CLI prints `Confirmed 0 proposal(s)` and exits 0.

The command did not accidentally confirm in the happy path, but it still
violates the silent-empty rule. Worse, if ADV-001 is present and the flagged
state is only visible in target frontmatter, bulk confirmation can actually
confirm a flagged-target proposal because target trust came from stale registry
state.

Required fix: bulk write commands need a flagged-excluded diagnostic. Minimum:
return both the acted count and the flagged-excluded count, and print a
non-success diagnostic when the acted count is zero but flagged exclusions are
nonzero. Stronger: add explicit `--include-flagged` to bulk write commands so
operators must opt in before acting on flagged-input proposals.

### ADV-004: The new `flagged_input` suppression key corrupts rejection-memory output

The spec adds `suppression_counts["flagged_input"]` to the same dict used for
per-producer rejection-memory suppression:
`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:85-89` and
`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:127-129`.

Existing `links review` output computes `total_suppressed =
sum(suppression_counts.values())` and labels that total as rejection memory
(`folio/cli.py:1622-1628`). The spec says to add the flagged disclosure "after
the existing suppression disclosure" (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:168-179`),
but it never says to exclude the new sentinel key from the rejection-memory
total.

Break case:

- There are zero rejection-memory suppressions.
- There are three flagged-input suppressions.
- `suppression_counts == {"flagged_input": 3}`.
- The CLI says `3 proposals suppressed by rejection memory.` and then says
  `3 proposal(s) excluded (flagged inputs).`

That is two contradictory explanations for the same proposals. It also makes
the ordering rule ambiguous: the spec says rejection memory takes precedence,
but the display layer can no longer distinguish rejection-memory counts from
trust-gate counts by summing the shared dict.

There is also a collision bug: producer names come from arbitrary
`_llm_metadata` keys except `links` (`folio/links.py:105-123`). A producer named
`flagged_input` would collide with the sentinel and make rejection-memory and
trust-gate counts indistinguishable.

Required fix: either split the return shape into structured categories, for
example `{ "rejection_memory": dict[str, int], "flagged_input": int }`, or
explicitly reserve a non-producer namespace and update every renderer to exclude
policy counts from rejection-memory totals. Add tests for flagged-only,
rejection-only, mixed flagged plus rejection, and producer-name collision.

## Should-Fix

### ADV-SF-001: Malformed `review_status` fails open too broadly

The failure policy says missing or non-string `review_status` is treated as not
flagged (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:201-209`). Missing or
`None` as non-flagged is reasonable. Treating every malformed value as clean is
not.

Break cases:

- `review_status: [flagged]` in YAML loads as a list and bypasses the gate.
- `review_status: true` loads as boolean `True` and bypasses the gate.
- `review_status: " flagged "` or `"Flagged"` bypasses the gate even though the
  operator intent is obvious.

The spec should not make malformed trust metadata silently equivalent to clean
trust metadata. At minimum, define a helper such as `is_flagged_review_status()`
with explicit normalization and invalid-value handling. A conservative policy:
`None` and missing are non-flagged; normalized string `"flagged"` is flagged;
containers containing the string `"flagged"` are flagged or diagnosed; other
non-string truthy values produce an invalid-trust-metadata warning and are not
silently called clean.

Add tests for `None`, missing, unknown string, list containing `flagged`, boolean
true, and case/whitespace variants.

### ADV-SF-002: `--include-flagged` review can surface proposals that no action command can act on

The only explicit override in the spec is on `folio links review`
(`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:146-154`). But normal operator
workflow after reviewing a proposal is `folio links confirm <proposal_id>` or
`folio links reject <proposal_id>`, and both use `_find_pending_view()` with the
default filtered collection (`folio/links.py:329-333`, `folio/links.py:381-423`).

Result: an operator can run `folio links review --include-flagged`, copy a
proposal ID, then run `folio links confirm <id>` and receive `Unknown
proposal_id` even though the proposal was just displayed by the official review
surface. That is not an accidental confirm bypass, but it is a broken consent
workflow.

The spec needs to choose one behavior:

- Flagged proposals shown by `--include-flagged` are read-only diagnostics until
  the document is unflagged. If so, the output must say they are blocked from
  confirm/reject.
- Or action commands accept their own `--include-flagged` override and use it as
  explicit consent for the write.

### ADV-SF-003: Revived plus flagged tag composition needs an exact contract and a test

The spec says the flagged tag renders after the revived tag and both may
co-occur (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:156-166`), but it
does not show the exact composite line and the test plan does not include this
case (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:215-222`).

The likely output is understandable:

```text
[proposal_id] source -> target (relation, confidence, producer) (revived - basis changed) (flagged: source, target)
```

But without an exact copy fixture, implementers can accidentally reorder tags,
merge meanings into one parenthetical, or drop one tag when both conditions are
true. Add a test like `test_revived_and_flagged_tags_render_together` with the
full expected line.

## Informational

### ADV-I-001: Rejection-memory precedence is defensible, but only if counts remain separated

The spec intentionally runs the flagged-input check after rejection-memory
suppression (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:134-142`). That is
defensible: an operator's explicit rejection can be treated as the stronger
reason to suppress. The problem is not the ordering itself; it is the later
mixing of flagged counts into the same aggregate used for rejection-memory
rendering. Fixing ADV-004 preserves the intended precedence.
