## 3. Quality assessment

**Design quality is good.** The three new helpers (`_derive_trust_status`, `_compute_relationship_schema_gate`, `_derive_recommended_action`) are well-factored — each has a single responsibility, none exceed 30 lines, and the `graph_doctor` pending-loop simply composes them (`trust = ...; gate = ...; findings.append({...})`). DRY is respected: `trust` and `gate` are computed once and reused both in the `recommended_action` derivation and in the finding dict. This matches the spec's "compute once, reuse in action derivation + finding dict" annotation at spec §3.1.

**Naming is consistent** with the spec's ID conventions — `target_registered` (not `target_exists`, per v1.2 P-R2-1 closure), `supported_relation`, `_SUPPORTED_RELATIONS` (frozenset of 4 relations matching links.py:21). The spec's renames (`target_exists → target_registered`) are reflected everywhere consistently (graph.py, test names, CHANGELOG, action strings).

**Readability is fine.** The finding dict in `graph_doctor` orders keys legacy-first (code, severity, subject_id, detail, recommended_action), then proposal_id, then the 10 other §5 keys. This ordering differs from the spec §3.1 pseudocode (which places proposal_id directly after recommended_action and then the §5 block) — the difference is stylistic; Python dict ordering is preserved but not load-bearing for any test or CLI consumer. Minor (see §5 M-1).

**Backward-compat path is correctly handled.** The cli.py renderer uses `finding.get("proposal_id") or finding.get("subject_id") or "—"`, which:
- For pending-proposal findings: `proposal_id` is truthy → renders it. `subject_id` (which is `None`) is never consulted.
- For non-proposal findings: `proposal_id` absent → falls back to `subject_id` (truthy) → renders it.
- For pathological findings where both are missing: renders `"—"`.

This satisfies §3.4's "fallback chain preserves existing rendering for non-proposal findings." **Direct-run evidence**: `test_graph_doctor_json_uses_fixed_finding_schema` (tests/test_graph_cli.py:244-421) exercises a mixed-findings fixture that includes both pending_relationship_proposal AND five non-proposal kinds (orphaned_canonical_relation, protected_enrich_body, stale_analysis_artifact, unconfirmed_entity, missing_entity_stub, duplicate_person_candidate); all pass.

## 4. UX review

**CLI output is correctly updated.** When a pending proposal is flagged AND has a missing target, stdout will render:

```
[medium [flagged] [schema-gate: target_registered]] pending_relationship_proposal <proposal_id>: ...
  Action: Target not in registry. Run `folio refresh` or `folio ingest` upstream; if the target is intentionally missing, reject via `folio links review`.
```

The bracket-within-bracket rendering (`[medium [flagged] [schema-gate: ...]]`) is syntactically valid but visually unusual — an operator scanning output may parse it as nested. Per PRD-8 (v1.1 review disposition "accepted as author's-call"), this is acknowledged upstream and not a defect. Noted.

**`--help` copy update** is placed as the second paragraph of the docstring (folio/cli.py:2000-2002), not "at the end of the existing help string" as §3.4 specifies. Since the docstring before this change was a single sentence "Emit actionable graph-health findings.", placement is effectively at the end — the spec's placement direction is honored in spirit. Minor cosmetic (see §5 M-2).

**Breaking change is appropriately loud.** CHANGELOG entry is verbatim per OQ-3 v1.1 closure, `--help` references CHANGELOG, and consumers that read JSON will see `subject_id: null` immediately.

## 6. Positive observations

- **Set-intersection pattern for `_derive_trust_status`** (graph.py:67-69) is exactly what spec §3.1 prescribes — `{"source","target"} & flagged`. This correctly handles the `flagged_inputs=["both"]` pathological case (covered by test `TestDeriveTrustStatus::test_unexpected_value_is_ok`), and gracefully degrades on `flagged_inputs=None` via `or []` (covered by `test_none_input_is_ok`). This is a clean Pythonic idiom.
- **Short-circuit rule ordering in the schema gate** is deterministic and matches spec §3.3: `target_registered` is checked before `supported_relation`. Test `test_target_missing_short_circuits_before_relation_check` (tests/test_graph.py:142-148) pins this ordering.
- **The stdout fallback chain** (cli.py:2035-2039) uses truthy-falsy short-circuiting with an em-dash sentinel — correctly handles both the pending-proposal case (proposal_id truthy) and the non-proposal case (proposal_id absent → subject_id truthy), plus the P-R2-2 blank-proposal_id edge (both falsy → `"—"`, tested at tests/test_graph_cli.py:1042-1047).
- **CHANGELOG entry is verbatim** to the spec §6 OQ-3 v1.1 script, including the "Known gaps" section that discloses `input_fingerprint` legacy-alias, deferred evidence_bundle rendering, deferred `schema_version` payload field, and deferred non-deck-target validity.
- **No deprecation warnings, no test skips, no xfails.** 54 tests pass cleanly.
- **Helpers are independently testable** — 18 of the 19 test_graph.py unit tests exercise the helpers directly via `SimpleNamespace` views without touching the filesystem. This is the right layering.
- **`include_flagged` threading is at least complete** — even though P-2 flags the scope issue, the flag is correctly threaded from CLI → graph_doctor → collect_pending_relationship_proposals, and the test matrix exercises both flagged-source and flagged-target variants (`trust_status_flagged_when_source_flagged`, `trust_status_flagged_when_target_flagged`, `_recommended_action_notes_flagged_input`, `_cli_stdout_renders_flagged_annotation`).

## 7. Verdict

**Needs Fixes.**

Rationale: the implementation is well-structured and faithful to spec §3.1 / §3.3 / §3.4 / §3.6 on substance, but it ships an un-spec'd `--include-flagged` flag (P-2) and has three spec-enumerated test names missing (P-1). Either the spec needs a Phase-D.4 update documenting these, or the code needs trimming. The `_SUPPORTED_RELATIONS` duplication (P-3) is a latent drift trap worth closing now rather than carrying forward. None of these rise to blocker severity — no observed misbehavior, no broken contract — but all three are should-fix defects that D.3 and D.4 can close in the same pass.
