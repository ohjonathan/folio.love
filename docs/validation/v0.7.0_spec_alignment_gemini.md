# Alignment Review — folio-digest-v0-7-0 / B.1 / gemini

## Verdict
**Needs Fixes**

The spec does an excellent job of adhering to the shared Tier 4 contracts and remaining within its tight manifest constraints. The deferral of the `cli.refresh` message is correctly handled via the §15.7 narrowing contract to avoid modifying out-of-scope paths. However, there is a logic flaw in the Trust Notes rendering that produces factually inaccurate trust disclosures when `--include-flagged` is used on an all-clean input set, which violates the strict trust signaling requirements.

## Validation Findings

1. **Design spec citation (sections 1-13):** Passed, with one minor omission. The spec maps sections 1-13 cleanly to implementation or deferral, but misses "N-way synthesis" from the Design §3 deferral list in the Spec §1 table.
2. **Parent specification citations (§11, §15.4, §15.6, §15.7):** Passed. Citations are explicit and deferrals are enumerated using the §15.7 worked-example pattern in §14.

The core skipping behavior is already present, making the deferral safe.
4. **Backward compatibility & Registry shape:** Passed. The registry entry initialization in §8.2 perfectly matches the source-less analysis-doc shape established in `analysis_docs.py:181-197`.
5. **Naming conventions & Path layout:** Passed. The divergence from `analysis_docs.resolve_analysis_path` to `analysis/digests/<id>/<id>.md` is explicitly noted and accurately reflects the requirements of Design §7.

## Detailed findings

### ALIGN-001: Trust Notes block hallucinates flagged input inclusion when none were present
* **Authority Citation:** `tier4_digest_design_spec.md` §12 rule 3 ("remind the reader that digest notes remain review-required synthesis artifacts") and §9 rule 1 ("state whether flagged source-backed inputs were excluded or included via override").
* **Artifact Location:** `docs/specs/v0.7.0_folio_digest_spec.md` §9.3 (Trust Notes block) and §5 (Daily Input Predicate).
* **Finding:** The Trust Notes template `{flagged_disclosure}` maps the condition `included_flagged_via_override and excluded_flagged_count == 0` to the text: *"Daily input selection ran with --include-flagged; flagged source-backed inputs were included in this synthesis."* However, `_collect_daily_inputs` in §5 does not track `included_flagged_count`—it only tracks exclusions. Thus, if a user passes `--include-flagged` on a scope containing *only* clean inputs, the spec dictates rendering a statement claiming flagged inputs were included. This is factually incorrect and undermines the trust posture of the artifact.
* **Recommendation:** Modify `_collect_daily_inputs` to track and return `included_flagged_count`, or change the rendered disclosure text to accurately reflect the possibility of inclusion (e.g., *"flagged source-backed inputs (if any) were permitted in this synthesis"*).

### ALIGN-002: Design Spec §3 "N-way synthesis" deferral is missing from Spec §1 deferral table
* **Authority Citation:** `tier4_digest_design_spec.md` §3 ("N-way synthesis" listed as deferred to `folio synthesize`).
* **Artifact Location:** `docs/specs/v0.7.0_folio_digest_spec.md` §1 (Not in scope table).
* **Finding:** The spec's "Not in scope" table explicitly lists items deferred from Design §3 and §13 (e.g., `--steerco`, Automatic file-watcher, Cross-engagement, FR-700 grounding parity), but it omits "N-way synthesis" which was explicitly listed in Design §3. 
* **Recommendation:** Add "N-way synthesis" to the "Not in scope" table in §1 to perfectly match the Design §3 exclusions.
