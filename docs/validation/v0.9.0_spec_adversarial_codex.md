# v0.9.0 Spec Adversarial Review (codex)

**Phase:** B.1
**Role:** adversarial
**Family:** codex
**Date:** 2026-04-17
**Target:** docs/specs/v0.9.0_folio_search_spec.md v1.0
**Verdict:** BLOCK

## Blockers (ADV-B1-NNN)

### ADV-B1-001 Manifest cardinality assertions are false at Phase A/B.1
**Severity:** blocker
**Evidence:** frameworks/manifests/folio-search-v0-9-0.yaml:141-168 says the cardinality assertions "dry-run to exit-0 TODAY" and then runs commands that require `cli.commands["search"]` and `folio.search.SCHEMA_VERSION`. The same manifest declares `folio/search.py` as a greenfield create path at frameworks/manifests/folio-search-v0-9-0.yaml:72-87. In the current workspace, `importlib.util.find_spec("folio.search")` returns `None`, and `from folio.cli import cli; "search" in cli.commands` returns `False`.
**Why it blocks:** The manifest's own F-056 mitigation is self-contradictory: it encodes post-implementation assertions as if they are pre-implementation invariants. A B.1 reviewer or framework gate that evaluates these cardinality assertions before Phase C will fail for expected greenfield absence, not for a real implementation defect. This is a protocol-level manifest defect, not a code issue.
**Proposed closure:** Move the future `folio.search` / CLI registration checks out of pre-implementation `cardinality_assertions` or guard them so they pass before Phase C. Keep the post-implementation checks in `gate_prerequisites` / D-phase verification, where `folio/search.py` is expected to exist.

### ADV-B1-002 QUERY matching is not a total, Unicode-safe contract
**Severity:** blocker
**Evidence:** docs/specs/v0.9.0_folio_search_spec.md:589-610 defines matching as direct `.casefold()` substring checks and claims Unicode discipline is covered by casefold. The field table calls `view.proposal.rationale` and each `view.proposal.signals` element `str` at docs/specs/v0.9.0_folio_search_spec.md:601-602. Current deserialization does not enforce those runtime types: `RelationshipProposal.from_dict()` preserves explicit `None` or non-string values for `rationale` / `signals` at folio/pipeline/enrich_data.py:101-119, and `folio/links.py` filters `relation` / `target_id` but not `rationale` or every signal element at folio/links.py:187-200. The spec also does not define NFC/NFD behavior, zero-width character behavior, lone-surrogate handling, or whitespace-only QUERY behavior.
**Why it blocks:** This is the core behavior of `folio search`. As written, the implementation can crash on malformed-but-parseable frontmatter (`None.casefold()` or `int.casefold()`), silently miss canonically equivalent Unicode text (`"é"` vs `"e\u0301"`), and accept invisible or whitespace-only queries with surprising match results. The acceptance criterion at docs/specs/v0.9.0_folio_search_spec.md:851 says "Unicode-safe", but the test plan only names an eszett case at docs/specs/v0.9.0_folio_search_spec.md:744 and even mis-cites SRC-8/SRC-9 as Unicode tests at docs/specs/v0.9.0_folio_search_spec.md:608 while those IDs are producer/reason-summary tests at docs/specs/v0.9.0_folio_search_spec.md:740-741.
**Proposed closure:** Define a single helper such as `_normalize_search_text(value) -> str | None` and specify it normatively: reject empty/whitespace-only QUERY at CLI and function level; normalize match operands with `unicodedata.normalize("NFC", value).casefold()` or explicitly document codepoint-exact matching; skip or coerce non-string field values consistently; and add tests for `None`, empty strings, non-string signal elements, NFC/NFD equivalence, zero-width characters, and surrogate/no-crash behavior.

### ADV-B1-003 Failure-mode contract is contradicted by the specified dependencies
**Severity:** blocker
**Evidence:** docs/specs/v0.9.0_folio_search_spec.md:86 and docs/specs/v0.9.0_folio_search_spec.md:616-620 require exit 1 on registry-missing; docs/specs/v0.9.0_folio_search_spec.md:721 specifies `Error: registry.json not found. Run \`folio ingest\` first.` The read-only dependency `registry_mod.load_registry` is explicitly listed at docs/specs/v0.9.0_folio_search_spec.md:181, but the implementation returns an empty registry when the file is absent at folio/tracking/registry.py:81-89. The CLI sketch also catches `ScopeResolutionError` at docs/specs/v0.9.0_folio_search_spec.md:524 without importing it from `.search` at docs/specs/v0.9.0_folio_search_spec.md:512.
**Why it blocks:** Following the spec literally will not produce the specified registry-missing behavior. A missing registry will look like an empty library unless search adds its own existence check, but the spec does not require that check and the test plan / acceptance criteria omit registry-missing coverage. The invalid-scope handler is also incomplete as sketched, making a required exit-code path ambiguous.
**Proposed closure:** Choose one registry contract and make it testable. Either revise the failure-mode table to say missing registry is a successful zero-result search with the no-matches breadcrumb, or require an explicit `registry.json.exists()` check in `folio.search` before `load_registry()`. Add unit and CLI tests plus an acceptance criterion for the chosen behavior. Import `ScopeResolutionError` in the CLI sketch.

## Should-fix items (ADV-SF-NNN)

### ADV-SF-001 `--limit` accepts negative values as written
**Severity:** should-fix
**Evidence:** docs/specs/v0.9.0_folio_search_spec.md:483-487 uses `type=int`; docs/specs/v0.9.0_folio_search_spec.md:333-335 slices `matched_views[:limit]`. The acceptance criterion only says `--limit N` caps findings at docs/specs/v0.9.0_folio_search_spec.md:857.
**Why it blocks:** This does not rise to blocker because it is an edge case, but `--limit -1` would silently drop the last finding rather than fail usage validation. That is not a cap in the operator sense.
**Proposed closure:** Use `click.IntRange(min=0)` or an explicit validation guard. Add CLI tests for negative and non-integer limits.

### ADV-SF-002 Shared-helper invariant wording contradicts the search pseudocode
**Severity:** should-fix
**Evidence:** docs/specs/v0.9.0_folio_search_spec.md:379 says search must call `derive_trust_status(view)` "on each iterated view"; SRC-20 says "called exactly once per view pre-filter" at docs/specs/v0.9.0_folio_search_spec.md:752. But the pseudocode filters by query, producer, and limit before converting to findings at docs/specs/v0.9.0_folio_search_spec.md:328-336, so the helper would only run for emitted findings if `_view_to_finding()` follows the synthesize pattern.
**Why it blocks:** The identity assertion alone is not the runtime invariant; SRC-20 is the runtime guard. Today the spec leaves implementers choosing between the pseudocode and the test wording.
**Proposed closure:** Define the invariant as either "once per collector view before search filtering" or "once per emitted finding after filtering." I recommend "once per emitted finding" unless the product needs trust derivation for suppressed non-matches. Update SRC-20 and the acceptance criterion at docs/specs/v0.9.0_folio_search_spec.md:848 accordingly.

### ADV-SF-003 `--scope -` is specified but not covered
**Severity:** should-fix
**Evidence:** docs/specs/v0.9.0_folio_search_spec.md:83 and docs/specs/v0.9.0_folio_search_spec.md:462 say `-` means library-wide. The search CLI tests at docs/specs/v0.9.0_folio_search_spec.md:771-793 and acceptance criteria at docs/specs/v0.9.0_folio_search_spec.md:831-861 cover omitted scope but not `--scope -`. The direct template did cover the dash alias for synthesize at docs/specs/v0.8.0_folio_synthesize_spec.md:721.
**Why it blocks:** Operators will reasonably try `--scope -` because the help text promises it. Without a test, this can drift or be forgotten during implementation.
**Proposed closure:** Add `SRC-CLI-24 test_search_envelope_scope_null_for_dash_alias` and an acceptance criterion: `folio search foo --scope - --json` yields `"scope": null`.

### ADV-SF-004 Regex/ReDoS guard is behaviorally underspecified
**Severity:** should-fix
**Evidence:** docs/specs/v0.9.0_folio_search_spec.md:166 forbids `re.compile`, `re.match`, and `re.search`; SRC-23 at docs/specs/v0.9.0_folio_search_spec.md:755 checks forbidden symbols. The spec does not forbid `re.fullmatch`, `re.findall`, `fnmatch`, `pathlib.Path.match`, or another pattern engine, and there is no behavior test proving regex metacharacters are treated literally.
**Why it blocks:** The prose is clear enough that this is not a blocker, but the test guard is weaker than the stated no-regex security posture.
**Proposed closure:** Add a behavior test where QUERY contains regex metacharacters such as `.*`, `[a-z]+`, and `(?=x)` and only literal substring matches succeed. Consider forbidding `import re`, `fnmatch`, and `Path.match` in `folio/search.py` for this slice.

### ADV-SF-005 Synthesize's `payload_type` revisit was not explicitly disposed
**Severity:** should-fix
**Evidence:** docs/specs/v0.8.0_folio_synthesize_spec.md:465 says `payload_type` should be revisited at sub-slice 3, and docs/specs/v0.8.0_folio_synthesize_spec.md:755 carries it as a deferred item. The search spec keeps `command` at docs/specs/v0.9.0_folio_search_spec.md:431 and docs/specs/v0.9.0_folio_search_spec.md:445, but does not explicitly close the revisit.
**Why it blocks:** Keeping `command` is defensible, but the direct template left an explicit follow-up marker for this exact slice. Silent non-disposition creates review churn.
**Proposed closure:** Add a short disposition under search section 3.3: `payload_type` remains deferred or rejected for v0.9.0 because the shared envelope now has two producers and `command` remains the agreed discriminator.

## Minor items (ADV-MIN-NNN)

### ADV-MIN-001 `--producer` exact case-sensitive match is defensible but easy to misread
**Severity:** minor
**Evidence:** docs/specs/v0.9.0_folio_search_spec.md:84 and docs/specs/v0.9.0_folio_search_spec.md:713 require exact `view.producer == P`. Current relationship proposals default to producer `"enrich"` at folio/pipeline/enrich_data.py:72-83, and `folio/links.py` treats `_llm_metadata` keys as producer names while skipping only reserved `"links"` at folio/links.py:173-190.
**Why it blocks:** It does not block. Producer is an internal workflow identifier, so exact matching is coherent. The footgun is operator ergonomics: `--producer Enrich` silently returns zero results for the common `"enrich"` producer.
**Proposed closure:** Keep exact match, but add one example to help text or zero-result diagnostics when `--producer` is present. Do not casefold producer IDs unless the storage contract changes.

### ADV-MIN-002 Truncation fields are internal but unobservable
**Severity:** minor
**Evidence:** `SearchReport` carries `total_available` and `truncated` at docs/specs/v0.9.0_folio_search_spec.md:296-297, but the stdout renderer at docs/specs/v0.9.0_folio_search_spec.md:549-585 does not print a truncation footer and the envelope intentionally emits only seven top-level keys at docs/specs/v0.9.0_folio_search_spec.md:345-374.
**Why it blocks:** This is not a correctness defect if v0.9.0 only promises capping. It is still surprising because synthesize's shipped implementation exposes a limit footer, while search stores the metadata and drops it.
**Proposed closure:** Either remove `total_available` / `truncated` from v0.9.0 or print a stdout footer when `--limit` truncates. Do not add a top-level JSON key unless the schema version policy is applied again.

## Accept-as-is / no issues

- Envelope versioning is compliant as written. Synthesize section 3.3 says verbatim at docs/specs/v0.8.0_folio_synthesize_spec.md:422-426:

  > **Envelope versioning policy (CB-3 closure).** The envelope uses semver-esque ordering at the `schema_version` string:
  >
  > - `"1.0"` — v0.8.0 shipping shape.
  > - `"1.x"` — additive top-level keys permitted (e.g., future `findings_truncated: bool`, `producer_acceptance_rates: [...]`). Consumers MUST tolerate unknown top-level keys at the same major version. Producers MUST bump minor (1.0 → 1.1) when adding a top-level key.
  > - `"2.x"` — reserved for breaking changes (field rename, type change, removal). Consumers SHOULD check `schema_version.startswith("1.")` before parsing.

  Search adds `query` as a top-level key at docs/specs/v0.9.0_folio_search_spec.md:391 and emits `schema_version == "1.1"` at docs/specs/v0.9.0_folio_search_spec.md:397. That is a clean application of the producer-side minor-bump rule, not a reinterpretation. The fact that `query` is command-specific does not violate the quoted rule; the rule says "when adding a top-level key," not "when adding a top-level key shared by every command."
- The render-envelope duplication disposition is acceptable. docs/specs/v0.9.0_folio_search_spec.md:447 correctly rejects importing `folio.synthesize.render_envelope`, and the manifest forbids edits to `folio/synthesize.py` at frameworks/manifests/folio-search-v0-9-0.yaml:107-114. No G-scope-1 violation is required here.
- The hard `lifecycle_state == "queued"` collector scope is honestly disclosed. `folio/links.py` filters at folio/links.py:251-253, and search defers multi-lifecycle behavior at docs/specs/v0.9.0_folio_search_spec.md:193 and docs/specs/v0.9.0_folio_search_spec.md:819.
- CB-1 carry-forward is correctly routed in the spec: `excluded_flagged_count` comes from `SuppressionCounts.flagged_input`, not `total()` or `rejection_memory`, at docs/specs/v0.9.0_folio_search_spec.md:649 and docs/specs/v0.9.0_folio_search_spec.md:750-751.
- Parent section 5 minimum shared-contract parity is preserved: the finding fields include all 11 shared keys at docs/specs/v0.9.0_folio_search_spec.md:652-667 and the acceptance criterion repeats that at docs/specs/v0.9.0_folio_search_spec.md:844.
- ReDoS/injection risk is directionally controlled by the prose: regex query syntax is a non-goal at docs/specs/v0.9.0_folio_search_spec.md:189-190, matching is specified as substring-only at docs/specs/v0.9.0_folio_search_spec.md:593-603, and the command never shells out per docs/specs/v0.9.0_folio_search_spec.md:86.

## Summary

Verdict is BLOCK. The envelope bump itself is compliant, but the manifest currently encodes impossible pre-implementation cardinality assertions, QUERY matching is not robust enough for a search command, and the registry/invalid-scope failure-mode contract is contradicted by the read-only dependencies and missing tests. Close those three before Phase C; the rest can be handled as should-fix cleanup.
