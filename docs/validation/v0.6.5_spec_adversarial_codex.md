# B.1 Adversarial Review - entity-merge-rejection-memory-v0-6-5

Loaded: `ontos_context_map`, `entity-merge-rejection-memory-v0-6-5`, `tier4_discovery_proposal_layer_spec`, `v0.6.0_proposal_review_hardening_spec`, `folio/tracking/entities.py`, `folio/cli.py`, `folio/graph.py`.

## Verdict

Needs Fixes.

The spec has the right product shape: persist explicit merge rejections, filter deterministic entity-merge suggestions, disclose suppression counts, and revive when the material basis changes. However, the current draft has two correctness blockers:

- The schema-version compatibility claim is false for downgrade/new-writer-old-reader cases.
- The new reject path is specified as an unlocked load-mutate-save operation, so concurrent rejections can erase each other and violate rejection-memory invariants.

Three should-fix/minor items cover call-site inventory, fingerprint encoding hardening, and stale already-merged keys.

## Findings

| ID | Severity | Description |
|----|----------|-------------|
| ADV-001 | blocker | Schema bump creates an undeclared downgrade break despite the spec claiming old and new files are tolerated |
| ADV-002 | blocker | `reject-merge` has a lost-update race and does not inherit an existing entity-registry lock |
| ADV-003 | should-fix | `suggest_person_merges()` call-site inventory misses `graph status`, so the default-filter behavior changes an operator surface without a contract or test |
| ADV-004 | should-fix | `basis_fingerprint` input encoding is delimiter-ambiguous outside today's hardcoded reason labels and generated slug keys |
| ADV-005 | minor | `reject-merge` behavior for already-merged loser keys is unspecified and will currently fail as unknown-key |

## Detailed Findings

### ADV-001 (blocker): Schema bump creates an undeclared downgrade break

Spec references: §3.1, §6.

Source evidence:

- Spec §3.1 requires `"_schema_version": "<current version, bumped by +1>"` and says missing `rejected_merges` plus the schema bump "tolerates both old and new files" (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:55`, `:70`).
- Spec §6 lists "Schema-version mismatch on load" as mitigated by defaulting `rejected_merges` to `[]` with "no hard migration" (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:176`).
- Current entity registry code defines `_ENTITY_SCHEMA_VERSION = 1` (`folio/tracking/entities.py:16`) and rejects any loaded file with a newer version (`folio/tracking/entities.py:555-561`).
- Existing tests codify this behavior: `_schema_version: 999` raises `EntitySchemaVersionError` (`tests/test_entities.py:363-369`).

Failure mode:

If v0.6.5 writes `entities.json` with schema version 2, then a v0.6.4 reader does not merely ignore `rejected_merges`; it fails before loading the registry. That breaks all entity commands that load `entities.json`, including read-only surfaces such as `folio entities list`, because the old reader exits on the version check.

This is not covered by the spec's backward-compat statement. The default-empty migration only handles old-writer/new-reader files. It does not handle new-writer/old-reader downgrade.

Required fix:

Choose one explicit compatibility policy:

1. Do not bump `_schema_version` for this additive top-level key. Current old readers would preserve unknown top-level keys on save because `save()` writes `self._data` and only uses `setdefault("_schema_version", _ENTITY_SCHEMA_VERSION)` (`folio/tracking/entities.py:611-613`).
2. Keep the schema bump, but declare v0.6.5 a one-way entity-registry migration and update §3.1, §6, CHANGELOG, and tests to say old readers fail with `EntitySchemaVersionError`.

The current text claims both a schema bump and downgrade tolerance; those cannot both be true against the current loader.

### ADV-002 (blocker): `reject-merge` has a lost-update race and no inherited entity lock

Spec references: §4.2, parent §10.1.

Source evidence:

- Spec §4.2 defines `folio entities reject-merge <left_key> <right_key>` as: load registry, invoke `reject_person_merge`, save registry (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:131-136`).
- Parent §10.1 says rejected proposals must stay suppressed and exact rejected resurfacing is a product defect (`docs/specs/tier4_discovery_proposal_layer_spec.md:266-274`).
- Current entity commands in `folio/cli.py` load and save directly. For example, existing entity reject loads at `folio/cli.py:2433-2436` and saves at `folio/cli.py:2459-2460`; entity merge loads at `folio/cli.py:2526-2529` and saves after mutation at `folio/cli.py:2551-2552`.
- No `library_lock` is used in the entity command block (`folio/cli.py:2098-2570`). The lock exists separately as a library-wide `.folio.lock` context manager (`folio/lock.py:58-83`) and is used by other mutation families, but the spec does not require it for the new entity rejection mutation.
- `EntityRegistry.save()` atomically writes the current in-memory dict, but it does not reload or merge concurrent updates (`folio/tracking/entities.py:601-613`).

Failure mode:

Two terminal sessions reject different pairs concurrently:

1. Process A loads `entities.json` with no `rejected_merges`.
2. Process B loads the same file with no `rejected_merges`.
3. A appends rejection record `R1` and saves.
4. B appends rejection record `R2` to its stale in-memory data and saves.

Final file contains only `R2`. `R1` is lost, so the first rejected merge can resurface. Atomic write prevents torn JSON, not lost read-modify-write updates.

Required fix:

Mandate locking for the new command's load-mutate-save critical section:

```python
with library_lock(library_root, "entities"):
    reg.load()
    reg.reject_person_merge(left_key, right_key)
    reg.save()
```

The spec should also state whether this slice locks only `reject-merge` or all entity-registry mutating CLI commands. Locking only `reject-merge` still allows races with `merge`, `import`, `confirm`, and existing entity `reject`; locking the whole entity mutation family is cleaner but broader.

Add a regression test that an existing `.folio.lock` blocks `reject-merge`, plus either a concurrency test or a documented single-writer invariant.

### ADV-003 (should-fix): Call-site inventory misses `graph status`

Spec references: §4.1, §4.3, §7.2; pattern source v0.6.0 §7.

Source evidence:

- Spec §4.1 changes `EntityRegistry.suggest_person_merges()` to default `apply_rejection_memory=True` (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:104-110`).
- Spec §4.3 names only the `duplicate_person_candidate` finding loop in `folio graph doctor` and says to pass `apply_rejection_memory=True` there (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:138-140`).
- Current `graph_status()` also calls `reg.suggest_person_merges()` for the aggregate duplicate count (`folio/graph.py:126-134`).
- The CLI renders that aggregate as `Duplicate person candidates: ...` in `folio graph status` (`folio/cli.py:1850-1865`).
- Current `graph_doctor()` has a separate duplicate finding loop at `folio/graph.py:224-260`.
- The pattern source explicitly treated caller inventory as a compatibility requirement: v0.6.0 §7 lists every caller found by `rg` and requires regression coverage (`docs/specs/v0.6.0_proposal_review_hardening_spec.md:407-428`).

Failure mode:

Because the default is `True`, `graph_status()` will silently switch from raw duplicate count to filtered visible duplicate count after any rejection exists. That may be the desired behavior, but the spec never states it, never adds a suppression disclosure for `graph status`, and does not add a test analogous to T-11 for status.

This is exactly the class of call-site undercount that the slice 1 pattern hardened against with a verified inventory.

Required fix:

Update §4.3 and §7 to include `graph_status()`:

- If status should count only reviewable duplicates, state that `Duplicate person candidates` is filtered by rejection memory and add a test after rejection.
- If status should report raw latent duplicates, pass `apply_rejection_memory=False` explicitly and document why status differs from `graph doctor`.
- If status remains filtered, consider a disclosure count or a renamed label such as `Reviewable duplicate person candidates`.

### ADV-004 (should-fix): Fingerprint input encoding is delimiter-ambiguous outside today's constants

Spec references: §3.2, parent §7.

Source evidence:

- Spec §3.2 hashes a flat delimiter string: `f"{subject_type}|{sorted_keys[0]}|{sorted_keys[1]}|{'|'.join(sorted_reasons)}"` (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:72-80`).
- Parent §7 requires normalized claim identity for entity-merge proposals and calls out trailing-whitespace normalization and ID-prefix handling in the shared fingerprint contract (`docs/specs/tier4_discovery_proposal_layer_spec.md:171-180`).
- Current generated entity keys are safe because `slugify()` lowercases and emits only `[a-z0-9_]`-style keys (`folio/tracking/entities.py:96-105`).
- Current reason labels are also safe hardcoded ASCII tokens (`folio/tracking/entities.py:960-964`).
- But current `EntityRegistry.load()` accepts arbitrary JSON object keys under `entities[etype]`; it validates entry values, not key character sets or Unicode normalization (`folio/tracking/entities.py:563-596`).

Failure mode:

With today's generated keys and reason labels, a realistic collision is unlikely. The problem is that the spec pins a delimiter-based hash format as a versioned compatibility contract (§6 says future changes require a breaking schema bump), while the loader allows states outside the assumed alphabet.

The delimiter encoding is not injective if reason labels ever contain `|` or empty strings:

- `reasons = ["a", "b"]`
- `reasons = ["a|b"]`

Both produce the same reason suffix after `'|'.join(sorted_reasons)`. For the same entity pair, that yields the same `basis_fingerprint` for two materially different bases and can false-suppress a legitimate revived merge. Key delimiter ambiguity is less directly suppressive because suppression also compares the sorted key pair, but it still makes the fingerprint contract harder to reason about and debug.

Required fix:

Use canonical structured serialization for the hash input, for example:

```python
payload = {
    "subject_type": subject_type,
    "entity_keys": sorted_keys,
    "reasons": sorted_reasons,
}
input = json.dumps(payload, sort_keys=True, separators=(",", ":"), ensure_ascii=False)
```

Alternatively, explicitly constrain entity keys and reason labels to a delimiter-free alphabet on load and in tests. Structured serialization is simpler and matches the "no ambiguous string parsing" standard expected for a persisted fingerprint.

### ADV-005 (minor): Already-merged loser keys are unspecified

Spec references: §4.1, §4.2, §5 OQ-1.

Source evidence:

- Spec §4.1 says `reject_person_merge(left_key, right_key)` validates both keys exist under `entities["person"]` and raises `EntityRegistryError` if not (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:114-119`).
- Spec §4.2 says the CLI exits non-zero on unknown key (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:131-136`).
- Spec §5 OQ-1 resolves only the `--force` question, saying no force is needed because the operator typed both keys (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:163-168`).
- Current `merge_person_entities()` deletes the loser key from `entities["person"]` (`folio/tracking/entities.py:1051`).
- `get_entity()` returns `None` when a key is absent (`folio/tracking/entities.py:768-776`).

Failure mode:

If an operator has a stale `suggest-merges` result and another process already merged `left_key` into `right_key`, `reject-merge left_key right_key` will fail as an unknown key because the loser entry no longer exists. That is probably acceptable, but it should be explicit because it is a common operator race around the exact command this spec adds.

Required fix:

Specify one behavior and test it:

- Unknown/merged key is a non-zero stale-candidate error with clear copy such as `Merge candidate is stale: left_key no longer exists`.
- Or, if one key resolves as an alias on the surviving entity, treat it as an idempotent no-op and print that the pair is already merged.

The first option is smaller and aligns with the existing validation rule.

## Targeted Failure-Mode Notes

### Revival Trap

No blocker found.

Spec §4.1 says exact fingerprint matches are dropped, while same-pair different-fingerprint records only annotate the returned suggestion as revived (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:104-110`). The same method also deduplicates exact `(subject_type, sorted_keys, basis_fingerprint)` rejections (`docs/specs/v0.6.5_entity_merge_rejection_memory_spec.md:114-119`).

Given that ordering, the v1 -> v2 -> v1 scenario behaves correctly:

1. Reject fingerprint A.
2. Reasons change to fingerprint B; suggestion revives.
3. Reject fingerprint B.
4. Reasons revert to fingerprint A; exact A record suppresses it, so it is not double-revived.

The spec would be stronger if T-6 added a revert-to-prior-basis assertion, but this does not block approval by itself.

### Parameter Default

The default `apply_rejection_memory=True` does not break any in-repo call site that clearly requires raw unfiltered suggestions. The problem is narrower: the spec inventory misses one current caller (`graph_status()`), so the behavior change is undocumented there. See ADV-003.

### Key Sorting And Unicode

No current-code false suppression was found for identical-prefix keys or Unicode-normalization variants when generated through `slugify()`, because current generated keys are ASCII slug keys. Manual or legacy JSON keys remain unconstrained on load, which is why ADV-004 recommends structured fingerprint serialization or explicit key validation.

Static review only. No tests were run because this phase is a spec adversarial review and no implementation patch was requested.
