## Rationale

- **B3R2-2 incomplete:** §7 added the breaking-change declaration as required, but the spec's §2 and §4.4 still contain the phrase "additive" describing the JSON migration (notably §4.4: "Schema is additive: existing `findings` consumers unaffected"). Codex reads this as an internal contradiction: §7 calls it a breaking change; §4.4 calls it additive. Both statements refer to the same migration.

The spec *intent* was that the `findings` key's internal shape is additive (findings list unchanged) while the wrapping top-level shape is a breaking change. The wording does not communicate that distinction; codex is correct that a reader would encounter the contradiction. This is a wording-precision defect, fixable in ~5 minutes of search-and-replace, but it remains the Round-3 preserved blocker.
