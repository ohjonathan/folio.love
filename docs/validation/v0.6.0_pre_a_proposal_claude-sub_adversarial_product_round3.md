## 2. B-4 closure assessment (THE scope)

### Rev 4 change summary (direct quotes from §9.2)

> "Cumulative acceptance-rate gate. Any producer must sustain a cumulative acceptance rate of at least 50% across all its *reviewed* proposals (reviewed = lifecycle state moved out of `queued` to `accepted` / `rejected`; `suppressed` and `stale` transitions are excluded from the denominator). The rate is computed at query time by aggregating `lifecycle_state` values from all entries of `_llm_metadata.<producer>.axes.relationships.proposals` across all managed documents — **no `rejected_at` timestamp, no global chronological ordering, and no new per-producer persistence is required**. A 10-reviewed-proposal warm-up applies: the gate does not restrict default surfacing until the producer has accumulated at least 10 reviewed proposals cumulatively, so newly-introduced producers are not gated on small-sample noise."

> "The cumulative framing is a deliberate tradeoff against the rolling-window framing of prior revisions: cumulative is less responsive to recent trend improvement (a producer that fixed a prior bad patch still carries the tail of old rejections in its rate) but is operationally simpler and requires no infrastructure the current code does not already provide. If field evidence shows the tail drag produces unfair gating for recovered producers, a follow-up spec can reintroduce a time-windowed variant once timestamping is available."

### Closure verdict: **closed**

### Evidence (cross-check against folio/links.py)

**Cumulative framing avoids the `rejected_at` dependency?** Yes — verifiably.

- The rate is a count ratio: `accepted_count(producer) / (accepted_count(producer) + rejected_count(producer))`. A count ratio does not require ordering; it requires only discoverable state at query time.
- Under gemini PR-T-1-R2 the rolling window needed to identify "the 20 most recently reviewed," which required ordering, which required `rejected_at`. The cumulative variant walks the full set — there is nothing to order.

**Aggregation computable from existing frontmatter?** Yes — verifiably.

Cross-checking against `folio/links.py`:

- Accepted-path counting: `confirm_proposal` (line 321) invokes `_remove_or_update_proposal(..., new_status=None)` which *removes* the proposal from `_llm_metadata.<producer>.axes.relationships.proposals` (lines 296–317) and writes a confirmation record into `_llm_metadata.links.confirmed_relationships` via `_upsert_confirmation_record` (lines 213–237). Each record carries `proposal_id`, `producer`, and `basis_fingerprint`. Aggregating accepted-count-per-producer is a filter over `_llm_metadata.links.confirmed_relationships` across all managed docs.
- Rejected-path counting: `reject_proposal` (line 349) invokes `_remove_or_update_proposal(..., new_status="rejected")` which *keeps* the proposal in `_llm_metadata.<producer>.axes.relationships.proposals` with `status = "rejected"` (line 311). Aggregating rejected-count-per-producer is a filter over `_llm_metadata.<producer>.axes.relationships.proposals` where `status == "rejected"` across all managed docs.
- Cross-document traversal is already done by `collect_pending_relationship_proposals` (lines 107–155) via `registry_data.get("decks", {}).items()`. The same traversal pattern works for both counts — walk the registry, open each doc's frontmatter, sum counts.
- One minor notational wrinkle: the spec says the rate is aggregated "from all entries of `_llm_metadata.<producer>.axes.relationships.proposals` across all managed documents." That is exactly right for the *rejected* half, but *accepted* proposals have been moved out of that tree into `_llm_metadata.links.confirmed_relationships` by `confirm_proposal`. A strict literal reading of the spec line would only count rejections. However, §6 lifecycle states `accepted = promoted to canonical graph state or accepted as the winning identity action`, and §9.2 defines "reviewed" as `lifecycle state moved out of queued to accepted / rejected`. A Phase A spec author will naturally read `_llm_metadata.<producer>.axes.relationships.proposals` as the producer's logical proposal history including the accepted-and-moved records (resolvable either by reading both trees or by adjusting `confirm_proposal` to keep the proposal in-list with `status="accepted"` alongside the canonical-field write). This is a Phase A implementation-mapping concern, not a blocker — the cumulative rate is computable; only the specific tree-path description needs a light edit. Recording as PR-A-15 (should-fix, not blocking).

**Warm-up handles cold-start without gaming attack surface?** Partially.

- The warm-up eliminates small-sample gating noise — a producer that makes 3 reviewed proposals with 1 acceptance (33%) would fail the 50% gate under a naive cumulative rule; the 10-proposal warm-up lets the producer accumulate evidence before gating activates. That is good-faith product behavior.
- But: a producer can *exploit* the warm-up by emitting 9 intentionally trivially-acceptable proposals (e.g., redundant `depends_on` edges that are obvious from citation data, or a `draws_from` edge to a document the operator already manually linked) to pad 9 easy acceptances into its denominator, then emit real noise. After the padding, the producer has 9 accepts and can absorb up to 9 rejections before its rate drops below 50%. This is real attack surface, not a theoretical concern — producers are a shared-contract consumer and the shared contract gives each producer independent admission to the queue.
- **Severity judgment:** should-fix, not a new blocker. Three reasons:
  1. Producers in Folio are a small trusted set added via code changes to the enrich pipeline, not a user-extensible plugin surface. The threat model is operator-accidental drift, not adversarial third-party code.
  2. The `confirm_proposal` trail in `folio/links.py` keeps `basis_fingerprint` per confirmation, so a Phase A spec could require warm-up acceptances to have distinct `basis_fingerprint` values and/or require `evidence_bundle` diversity, closing the trivially-acceptable-padding loophole at the contract level.
  3. The §9.1 pending-queue cap of 20 proposals already bounds the blast radius — even a warm-up-exploited producer can emit at most 20 items into the queued state at any time, and the operator still has to explicitly accept each padding proposal. The warm-up is gameable only to the extent operators rubber-stamp trivially-acceptable proposals, which is a review-quality concern independent of this gate.
- Recording as PR-A-16 (should-fix, not blocking). A Phase A spec author can address with a one-sentence addition: "warm-up acceptances require distinct `basis_fingerprint` values and non-redundant evidence locators."

### New adversarial hypotheses (two, both should-fix)

- **PR-A-15** (aggregation tree-path notation): §9.2 instructs aggregation "across all entries of `_llm_metadata.<producer>.axes.relationships.proposals`." In `folio/links.py`, `confirm_proposal` removes confirmed proposals from that tree and writes them to `_llm_metadata.links.confirmed_relationships`. A strict literal reading of §9.2 would therefore only count rejections. Two resolutions: (a) amend §9.2 to read "aggregating across both `_llm_metadata.<producer>.axes.relationships.proposals` (for rejected) and `_llm_metadata.links.confirmed_relationships` filtered by `producer` (for accepted)"; (b) adjust `confirm_proposal` in folio/links.py (Phase C work) to retain the proposal in-list with `lifecycle_state == "accepted"` alongside the canonical write. Either way the cumulative rate is computable; the spec only needs to pin which approach. Severity: should-fix, resolvable in Phase A spec pass. Location: §9.2, Evidence: static-inspection of folio/links.py:296–345.

- **PR-A-16** (warm-up padding attack): A producer can emit 9 trivially-acceptable proposals to pad the warm-up denominator, absorbing up to 9 rejections afterward before the 50% gate engages. Mitigation is straightforward (require warm-up acceptances to have distinct `basis_fingerprint` values), but the current spec text does not name this constraint. Severity: should-fix, not blocking given Folio's closed-set producer threat model. Location: §9.2 warm-up clause, Evidence: static-inspection.

**Tail-drag tradeoff (Q4 adversarial check):** The spec names this tradeoff explicitly in the second paragraph of §9.2 ("cumulative is less responsive to recent trend improvement … a follow-up spec can reintroduce a time-windowed variant once timestamping is available"). This is adequate scope handling: the spec does not pretend cumulative is neutral, it names the failure mode, commits a fallback escape hatch, and defers the fix until the prerequisite (timestamping) exists. Not a new product-scope risk.

## 3. Scope sanity

Rev 4's §9.2 rewrite did **not** grow scope. It is strictly simpler than Rev 3's rolling-window framing:
- Rev 3 implied a per-review ordering requirement (a chronologically sorted "last 20 reviewed" window).
- Rev 4 requires only set aggregation across an existing tree, computable at query time.
- No new persistence, no new timestamps, no new CLI surface, no new lifecycle state.

The tradeoff paragraph is scope-shrinking, not scope-expanding: it pre-emptively defers the time-windowed variant to a future spec rather than committing infrastructure now.

## 5. Verdict

**Proceed to Phase A.**

Justification: Rev 4's §9.2 rewrite closes B-4 with direct citation evidence against `folio/links.py`. The cumulative acceptance rate is genuinely computable from current frontmatter state without any new persistence, timestamps, or per-producer state files — `confirm_proposal` and `reject_proposal` already write the state needed for accepted-count-per-producer and rejected-count-per-producer aggregation respectively. The `rejected_at` dependency gemini PR-T-1-R2 raised is eliminated by removing ordering from the gate semantics. The 10-reviewed-proposal warm-up handles cold-start reasonably though it introduces a should-fix padding-attack surface (PR-A-16). One tree-path notational wrinkle around where confirmed proposals live (PR-A-15) is also should-fix, resolvable in Phase A. The tail-drag tradeoff is named explicitly with a committed fallback, not smoothed over. No scope growth. No reviewer-diversity halt.

Gemini's PR-T-1-R2 caught what I missed. Round 3's narrow scope forced me to verify the specific `reject_proposal` / `confirm_proposal` storage behavior, and Rev 4's cumulative reframe is now verifiably operationalizable against that actual storage shape. Neither PR-A-15 nor PR-A-16 rises to blocker; both are Phase-A-resolvable.
