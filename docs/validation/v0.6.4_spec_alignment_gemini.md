# v0.6.4 Spec Alignment Review (Round 1)

**Spec under review:** `docs/specs/v0.6.4_trust_gated_surfacing_spec.md` (v1.0)
**Parent proposal:** `docs/specs/tier4_discovery_proposal_layer_spec.md` (rev 5)
**Parent sections in scope:** SS 11 (trust-gated surfacing rules), SS 15.4 (Slice 4 scope)
**Reviewer role:** alignment (gemini family)

---

## Methodology

Each of the six SS 11 rules and the SS 15.4 scope envelope were checked against
the spec's stated implementation, data model, function specification, CLI
specification, test plan, and carry-forward disposition. Findings are
classified as Blocker (misalignment that must be fixed before Phase C),
Should-Fix (minor drift worth correcting), or Informational (observations
that do not block).

---

## Blockers

### AL-1: SS 11 rule 3 is under-scoped -- only `folio links review` provides `--include-flagged`

**Parent SS 11 rule 3** says: "committed Tier 4 surfaces must provide an
explicit operator override for including flagged inputs where that surface
is in scope."

**Parent SS 11 final paragraph** says: "For `folio digest`, the committed
override is `--include-flagged`. Equivalent explicit overrides are part of
the requirement for `folio synthesize`, semantic search, and traversal when
those surfaces ship."

**Parent SS 15.4** scopes Slice 4 to `folio/digest.py (if present)`,
`folio/cli.py`, and `folio/links.py`. The spec correctly defers `folio digest`
(the file does not exist). However, the spec does NOT evaluate whether other
committed surfaces accessible through the scoped files (`folio/cli.py` in
particular) qualify as "committed Tier 4 surfaces" that need the override.

Specifically: `folio links status` is a committed review surface exposed
through `folio/cli.py`. It renders pending-proposal counts. The spec's
SS 5.5 notes that `links status` "gets the default False behavior" and
calls the count "accurate." But SS 11 rule 3 requires an explicit override
on committed surfaces, and SS 11 rule 4 requires disclosure of excluded
counts. `folio links status` currently provides neither.

The spec's SS 9 carry-forward 2 argues no additional work is needed because
the count "accurately reflects reviewable proposals." This is a
reinterpretation of the parent: the parent says surfaces must *disclose*
excluded flagged-input counts (rule 4) and provide an override (rule 3).
Silently returning a filtered count without disclosure is exactly what
rule 5 is designed to prevent.

**Required fix:** Either (a) add `--include-flagged` and a flagged-exclusion
disclosure line to `folio links status`, or (b) explicitly cite which SS 11
rules are deferred for `links status` and add a carry-forward entry that
names the deferral, per SS 15.7's "slice manifests narrow, they don't
reinterpret" contract.

### AL-2: SS 11 rule 5 silent-empty protection does not cover `folio links confirm-doc` / `reject-doc`

**Parent SS 11 rule 5** says: "no surface may silently report an empty
result when excluded flagged inputs are the real cause."

The spec's SS 5.5 notes that `confirm_doc` and `reject_doc` call
`collect_pending_relationship_proposals` with the default `include_flagged=False`.
If all proposals for a given document are flagged-input proposals, these
commands will see an empty list and presumably report "nothing to confirm/reject"
without disclosing that proposals exist but were excluded.

This is a silent empty caused by flagged-input exclusion -- exactly the
scenario rule 5 prohibits.

**Required fix:** Either (a) add flagged-input disclosure to the empty-result
path of `confirm_doc` and `reject_doc`, or (b) document this as a known
deferral with an explicit carry-forward citing the rule 5 gap and the
rationale for deferral.

## Should-Fix

### AL-SF-1: Spec says `extraction_confidence` is "NOT CHANGED" but SS 11 rule 6 requires an affirmative statement

**Parent SS 11 rule 6** says: "`extraction_confidence` remains surfaced trust
metadata, not a second hard exclusion rule in v1."

The spec's SS 1 rule-implementation table says "NOT CHANGED -- informational
only." This is directionally correct but does not constitute an affirmative
commitment. Rule 6 exists precisely because a future implementer might add a
hard-exclusion threshold on `extraction_confidence`. The spec should include a
brief normative statement (one sentence in SS 6 or SS 4) making it explicit
that `extraction_confidence` must not be used as a filtering criterion in the
flagged-input filter or anywhere in the `collect_pending_relationship_proposals`
path. "Not changed" is a description of the diff, not a design constraint.

### AL-SF-2: Test plan lacks a negative test for `extraction_confidence` non-exclusion

The test plan (SS 7) has 8 tests covering rules 1-5. There is no test
asserting that a proposal with low `extraction_confidence` is NOT excluded
by the flagged-input filter. This would be a regression guard for rule 6.

**Suggested addition:** Test S4-9: `test_low_extraction_confidence_not_excluded` --
a proposal from a document with low `extraction_confidence` but
`review_status: clean` passes through the filter unchanged.

### AL-SF-3: SS 15.4 says "re-apply" trust-gating to `folio links review`; spec does not acknowledge this as new behavior

**Parent SS 15.4** says the purpose is to "honor SS 11 ... and *re-apply*
the same trust-gating rules to `folio links review` output." The word
"re-apply" implies trust-gating already existed somewhere and is being
extended. The spec should clarify that this is the *first* implementation
of trust-gating on `folio links review`, not a re-application of an
existing mechanism. This prevents confusion about whether Slice 1 or 3
already shipped partial trust-gating that this slice extends.

## Informational

### AL-I-1: Digest deferral is justified

`folio/digest.py` does not exist on disk. The parent SS 15.4 says
"folio/digest.py (if present)." The spec's deferral (SS 1 "Not in scope"
and SS 9 carry-forward 1) is correctly conditioned on the file's absence
and names the future obligation. No action needed.

### AL-I-2: `folio graph doctor` exclusion is reasonable

The spec excludes `folio graph doctor` as "informational diagnostic, not a
review surface." The parent SS 11 targets "committed Tier 4 surfaces" and
SS 15.4 does not list `folio/graph.py` in the Slice 4 scope. The exclusion
is consistent.

### AL-I-3: Rejection-memory precedence over flagged-input is a design choice, not a parent requirement

The spec's SS 4.4 establishes that rejection-memory suppression takes
precedence over flagged-input suppression (no double-counting). The parent
is silent on interaction ordering between these two filters. The spec's
choice is operationally sensible (rejection is a stronger signal) and does
not contradict any parent rule, but it is a design decision the spec is
making, not one the parent mandates.

### AL-I-4: CHANGELOG.md is in scope but not in forbidden paths

The spec's SS 2.1 lists `CHANGELOG.md` as an allowed path. SS 2.2 does not
list it as forbidden. This is fine, but the parent SS 15.4 does not mention
`CHANGELOG.md`. Minor scope addition for hygiene; no concern.

**Verdict: Needs Fixes.** The two blockers represent genuine gaps where
committed surfaces under the spec's own scope envelope (`folio links status`,
`folio links confirm-doc`, `folio links reject-doc`) either violate SS 11
rules 3-5 or silently reinterpret them as non-applicable. Per SS 15.7,
deferrals must be named explicitly -- implicit exclusion-by-omission is
itself an alignment defect. The should-fix items strengthen the spec's
rule 6 coverage and test plan but are not blockers on their own.
