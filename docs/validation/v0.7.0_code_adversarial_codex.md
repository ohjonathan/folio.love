## Detailed findings

### ADV-201 - Orphan self-heal is missing before the LLM call

**Severity:** Blocker  
**Evidence:** static-inspection, direct-run  
**Files:** `folio/digest.py:970-1034`, `folio/digest.py:1181-1242`

Spec Section 10.4 requires the orphan case to self-heal before invoking the LLM:

> File exists, registry row absent: before LLM call, build `RegistryEntry` from existing-file frontmatter and call `_register_digest`.

The implementation loads existing frontmatter and computes rerun version, then immediately builds the prompt and calls `_build_body_with_validation_retry`. There is no registry lookup for `digest_id` and no `_register_digest` call until after a successful new write (`folio/digest.py:1074-1078`, weekly equivalent `folio/digest.py:1281-1285`). This means the advertised recovery message after registry upsert failure is only true when the next rerun also survives the LLM call.

Reproduction:

1. Create an eligible evidence row.
2. Pre-create the expected digest file with valid frontmatter.
3. Leave the digest row absent from `registry.json`.
4. Patch `_call_llm` to raise.
5. Run `generate_daily_digest`.

Observed output:

```text
result.status= error
digest_id= clienta_ddq126_analysis_20260404_daily-digest
registry_has_digest_after_failed_rerun= False
```

Expected: `registry_has_digest_after_failed_rerun=True` before the LLM failure returns, because the existing file was valid and recoverable.

Impact:

- A registry-upsert failure can create an orphan digest file.
- A later rerun during an LLM outage does not self-heal the orphan despite the spec and user-facing recovery message saying it will.
- Weekly digest has the same structure and the same gap.

Required fix:

- After computing `digest_id` and `path`, load the registry and detect `path.exists() and digest_id not in registry["decks"]`.
- If `_load_existing_digest(path)` returned a dict, call `_register_digest` before prompt construction / `_call_llm`.
- Keep this inside the CLI lock-protected generate call.
- Add the missing tests promised by the spec: daily orphan self-heal before LLM, weekly orphan self-heal before LLM, and stale-registry file-absent warning behavior.

### ADV-202 - Empty `--include-flagged` message is not spec-exact

**Severity:** Should Fix  
**Evidence:** static-inspection, direct-run  
**Files:** `folio/digest.py:937-952`, `folio/cli.py:1248-1256`

Spec Section 4 gives this exact empty override example:

```text
No eligible inputs for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026 (--include-flagged was honored; no flagged inputs found either).
```

The implementation builds the base message with a period, then appends the parenthetical without final punctuation:

```python
message = f"No eligible inputs for daily digest on {day} in scope {scope}."
message += " (--include-flagged was honored; no flagged inputs found either)"
```

Reproduction:

```text
exit_code= 0
No eligible inputs for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026. (--include-flagged was honored; no flagged inputs found either)
```

Impact: the CLI contract is visibly different from the v1.2 example, and the existing CLI test only checks the substring. The spec specifically called out this echo wording in SF-15.

Required fix:

- Render the override-empty branch as one full sentence with the parenthetical before the period.
- Tighten `DG-CLI-10b` to assert the exact line.

### ADV-203 - Valid indented ATX headings evade validation and scrubbing

**Severity:** Should Fix  
**Evidence:** static-inspection, direct-run  
**Files:** `folio/digest.py:456-478`, `folio/digest.py:501-535`, `folio/digest.py:839-864`

The implementation handles fenced headings, trailing `#`, and trailing whitespace, but it only matches headings beginning at column 0:

```python
pattern = re.compile(rf"^##\s+{re.escape(heading)}(?:\s+#+)?\s*$")
any_heading = re.compile(r"^##\s+\S")
```

Markdown ATX headings may be indented by up to three spaces. A model output containing ` ## Trust Notes` is still a heading in common Markdown parsers, but `_heading_positions` returns no match and `_strip_section` leaves it in place.

Reproduction:

```text
positions_for_leading_space_trust= []
strip_changed= False
valid_markdown_one_space_summary_seen_as_missing= ['Summary']
```

Impact:

- A valid Markdown rogue system-owned heading can survive `_strip_section`.
- Because `_extract_section` also uses a column-0 `##` boundary, the rogue indented section can be folded into the prior LLM-owned section, then the system appends a second Trust Notes block.
- Valid LLM-owned headings with 1-3 leading spaces are treated as missing, causing unnecessary retry or failure.

Required fix:

- Use a shared ATX-H2 matcher that allows `[ \t]{0,3}` before `##`.
- Apply it consistently in `_heading_positions`, `_strip_section`, and `_extract_section`.
- Add regression tests for leading-space LLM-owned headings and leading-space system-owned sections outside fences.

### ADV-204 - Corrective retry loses the original validation failure on second-call exceptions

**Severity:** Should Fix  
**Evidence:** static-inspection, direct-run  
**Files:** `folio/digest.py:782-816`, `folio/digest.py:1010-1034`, `folio/digest.py:1217-1242`

The retry loop performs the second `_call_llm` directly. If that corrective call raises, the exception escapes and `generate_*_digest` reports only the upstream exception. The original validation defect that triggered the corrective retry is not preserved.

Reproduction:

1. First LLM response: only `## Summary`, missing the other daily LLM-owned headings.
2. Corrective re-call raises `RuntimeError("corrective timeout")`.

Observed:

```text
status= error
message= LLM synthesis failed for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026 (existing digest, if any, preserved). Cause: corrective timeout
contains_original_missing= False
```

Expected: the failure should retain the original missing headings as context, for example `corrective retry failed after initial validation failure: missing ## What Moved Today, ...; cause: corrective timeout`.

The corrective prompt also names headings as Python list reprs (`['What Moved Today']`, `['Summary']`) rather than the spec's exact `## Heading` wording. It is still understandable, but it is weaker than the v1.2 contract and harder for an LLM to follow precisely.

Required fix:

- Store the first validation failure details before the corrective call.
- Wrap exceptions from the corrective call with both the original validation details and the corrective-call cause.
- Format missing/duplicate headings as exact Markdown headings, e.g. `## What Moved Today`.
