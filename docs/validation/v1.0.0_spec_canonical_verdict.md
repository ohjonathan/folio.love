## Closure inventory

### Canonical Blockers (CB-1..CB-4)

**CB-1 - ADV-B1-001: invalid YAML frontmatter.** [closed-in-v1.1]
Spec frontmatter now quotes the colon-bearing parent and roadmap references (`consumes_parent`, `consumes_roadmap` in lines 10-11). Direct-run in this B.3 session: `ontos map` exited 0 and regenerated `Ontos_Context_Map.md` despite unrelated metadata warnings.

**CB-2 - ADV-B1-002 + PEER-B1-001: Click group reshape regresses legacy `folio enrich SCOPE` parsing.** [needs-rework]
v1.1 correctly replaces the group-level `@click.argument("scope")` with hidden `@click.option("--scope", default=None, hidden=True)` in spec §3.3, and explains why `ScopeOrCommandGroup` rewrites first-position scope tokens. That closes only part of the blocker. The original codex blocker also identified option-before-scope legacy forms as flat-command-compatible. Direct-run minimal reproduction using existing `ScopeOrCommandGroup` plus the v1.1 hidden-option shape:

| Invocation | Result |
|---|---|
| `["ClientA"]` | exit 0, scope parsed |
| `["ClientA", "--dry-run"]` | exit 0, scope parsed |
| `["--dry-run", "ClientA"]` | exit 2, `No such command 'ClientA'` |
| `["--llm-profile", "X", "ClientA"]` | exit 2, `No such command 'ClientA'` |
| `["--force", "ClientA"]` | exit 2, `No such command 'ClientA'` |

Spec tests ED-CLI-2 / ED-CLI-3 do not cover scope after group options: ED-CLI-3 exercises options with no scope. Required B.2 fix: either extend `ScopeOrCommandGroup.parse_args` for post-option bare scope tokens or otherwise preserve all previously valid flat-command forms, then add explicit regression tests for `folio enrich --dry-run ClientA`, `folio enrich --llm-profile X ClientA`, and `folio enrich --force ClientA`.

**CB-3 - ADV-B1-003: invalid scope / unreadable-registry behavior violates parent §7.7.** [closed-in-v1.1]
Parent §7.7 requires invalid scope and unreadable registry to be fatal. v1.1 adds `_resolve_scope_or_raise` in spec §3.2, invoked before `plan_enrichment` in §3.1, and updates §9 so typo scopes and corrupt registries exit 1 rather than returning false-clean empty findings. Tests ED-22, ED-23, ED-CLI-21, and ED-CLI-22 cover this closure. Manifest G-cardinality-7 also checks `ScopeResolutionError` export.

**CB-4 - ADV-B1-004: Sub-item B firewall misses CLI path for `--include-flagged`.** [needs-rework]
The manifest now has the right gate categories: G-scope-5 scans `folio/enrich.py`, G-scope-6 scans the marker-bounded diagnose block in `folio/cli.py`, and G-scope-7 checks `folio enrich diagnose --help` for absence of the literal `--include-flagged`. However, the v1.1 spec's own §3.3 diagnose docstring includes the literal string `--include-flagged` inside the `# v1.0.0 diagnose CLI: BEGIN/END` marker block, while §6.3 and G-scope-6 forbid that same literal in the marker-bounded block. If implemented from the spec, the G-scope-6 awk/grep gate and G-scope-7 help gate would fail. Required B.2 fix: reconcile PROD-SF-005 with CB-4 by either avoiding the literal in help prose (for example, "the sibling flagged override does not apply") or changing the gate/test to distinguish option declarations from explanatory help copy.
