# D.2 Product Review — trust-gated-surfacing-v0-6-4

## Verdict

**Needs Fixes.** The core trust-gating UX is cleanly wired: operators can see flagged proposals opt-in, confirm them opt-in, and every default surface discloses the exclusion count with an actionable `--include-flagged` pointer. One spec-divergence blocker (`folio links status` is missing the `--include-flagged` flag that spec §5.1 mandates and spec §5.5's own disclosure line points operators at). Two should-fix UX improvements around confirm/reject-doc success messaging when operating in `--include-flagged` mode. Several informationals (dead branch, unknown-id ambiguity) are intentional design tradeoffs worth recording. All seven operator workflows I simulated end-to-end complete without dead-ends once the blocker is resolved.

## Scope & method

Static inspection of `folio/cli.py:1580-1818` (links command group), `folio/links.py:1-598` (behavior layer), and `tests/test_links_cli.py:886-1337` (v0.6.4 fixtures), cross-referenced against `docs/specs/v0.6.4_trust_gated_surfacing_spec.md` §5. Direct-run simulation of 9 operator scenarios via `CliRunner` covering: clean review, flagged-source review (default + `--include-flagged`), status with mixed sources, status with all-flagged source, status with empty library, status with unmatched scope, confirm-doc (default + `--include-flagged`), reject-doc (default), confirm flagged id (with + without flag), `--doc` + `--include-flagged` cross-command workflow.

## Findings

### Blockers

| ID | Severity | Surface | Issue | Evidence | Suggested fix |
|---|---|---|---|---|---|
| D-PR-001 | blocker | `folio links status` | **Spec §5.1 mandates `--include-flagged` on `folio links status`, but the flag is absent on the command.** Spec §5.5 compounds the operator confusion by emitting a disclosure `use folio links status --include-flagged to inspect` — though the implementation quietly rewrote that disclosure to point at `links review --include-flagged`. An operator who reads the spec, or who copy-pastes one of the several disclosure sentences that reference `folio links status --include-flagged` (the spec template appears verbatim in the v1.1 spec block at §5.5), will hit `Usage: cli links status [OPTIONS] [SCOPE]` with no such option. Additionally, the implementation's disclosure inside status itself (`cli.py:1672-1675`) redirects the operator to `folio links review --include-flagged` — which is the right escape hatch, but the absence of the flag on status is still a spec-contract violation and a discoverability hit (`links status --help` shows no hint that trust gating is even a concept on this command). | `cli.py:1661-1699` (no `@click.option("--include-flagged"...)`); spec `§5.1` line 303 explicitly lists `folio links status (surface)`; `links status --help` output confirms only `--help` present; `relationship_status_summary` in `links.py:314-318` also takes no `include_flagged` parameter. | Add `--include-flagged` to `links_status_cmd` and thread through `relationship_status_summary`. At minimum, even if the flag is a no-op on status (rows are always summarised with `include_flagged=True` internally), its presence lets operators copy-paste the disclosure sentence verbatim and satisfies spec §5.1. If the flag is intentionally cut, the spec must be revised and the disclosure reworded before merge. |

### Should-fix

| ID | Severity | Surface | Issue | Evidence | Suggested fix |
|---|---|---|---|---|---|
| D-PR-SF-001 | should-fix | `folio links confirm-doc --include-flagged` / `reject-doc --include-flagged` success line | When the operator consents to act on flagged proposals (`--include-flagged`), the success message is indistinguishable from a clean action: `✓ Confirmed 1 proposal(s)`. There's no trust-posture feedback to remind the operator they just promoted a flagged-input relationship to canonical frontmatter. Compare `links review --include-flagged`, which tags each surfaced proposal `(flagged: source)` — the operator had visibility at review time. After `confirm-doc --include-flagged` the trust posture is invisible. If the operator typos their way from `confirm-doc <clean_source>` to `confirm-doc <flagged_source> --include-flagged`, the success receipt looks identical. | Direct-run scenario: `confirm-doc srcflag --include-flagged` on a source with `review_status: flagged` yields `✓ Confirmed 1 proposal(s)` with no mention of trust state. | Append a trust-posture clause when `include_flagged=True` AND at least one acted proposal had flagged inputs: `✓ Confirmed 1 proposal(s) (1 involved flagged-input documents)`. Requires `confirm_doc` / `reject_doc` to return a third counter (`acted_flagged`) or to compute it from the view list before acting. Low complexity; high trust-surface value. |
| D-PR-SF-002 | should-fix | `folio links status` footer | The footer `Use folio links review --include-flagged to inspect excluded proposals.` is only emitted when `total_flagged_excluded > 0`. But the table body already renders a `Flagged Excluded` column full of zeros on clean libraries. The column header is unconditionally present — which is the right call for consistency — but on a fully clean library the operator sees a "Flagged Excluded" column they may not understand without context, and the footer that explains it never fires. A first-time user reading the status table on a clean library has no way to learn what flagged-exclusion *is* from this surface. | `cli.py:1683-1699`: header always emitted (line 1683); disclosure only on non-zero (line 1696). | Option A (minimal): suppress the `Flagged Excluded` column when `total_flagged_excluded == 0`. Option B (explanatory): always render a one-line footer clarifier like `Flagged Excluded = proposals excluded because source or target has review_status: flagged`, placed once below the total row. Option A is cleanest; Option B preserves column stability for scripted consumers. |

### Informational

| ID | Severity | Surface | Issue | Evidence |
|---|---|---|---|---|
| D-PR-I-001 | informational | `folio links status` silent-empty branch | The `cli.py:1670-1678` silent-empty branch for status (`not rows` + `total_flagged_excluded > 0`) is unreachable in practice. `relationship_status_summary` appends a row whenever `pending + confirmed + flagged_excluded > 0` (`links.py:357`); if `flagged_excluded > 0` for any source, a row is always emitted. Total flagged > 0 implies at least one source has `flagged_counts[src] > 0`, which passes the row predicate. The branch is harmless defensive code but redundant. Not a bug — just noting the "silent-empty protection for status" is structurally unreachable because the column-in-table implementation supersedes the empty-header protection. If spec §5.5 anticipated a different code structure (one that could produce no rows despite exclusions), this divergence is worth recording. | `links.py:331-365` (row-emission predicate); `cli.py:1670-1678` (branch). |
| D-PR-I-002 | informational | `folio links confirm <flagged_id>` without flag | Returns `✗ Unknown proposal_id 'rprop-xxxx'` with exit 1. Per spec §5.7 CB-4 this is the designed consent path — the error does not leak that the ID exists but is gated. From an operator's perspective the message is three-way-ambiguous: "typo vs. already-acted-on vs. flagged-behind-gate." Recovery path is not dead-end (operator can reach `links status`, see the `Flagged Excluded` column with non-zero on a source, try `--include-flagged`), but it's a multi-hop diagnostic chain. Accepting this as an intentional tradeoff: leaking "this ID exists but is gated" would itself be a bypass surface per CB-4 rationale. Worth recording that this is a known friction, not an unintended UX bug. | `links.py:451-453`; spec §5.7 lines 373-378. |
| D-PR-I-003 | informational | Consistency of `1 proposal(s)` pluralization | `links_review_cmd` disclosure (`cli.py:1654`) uses explicit `plural = "proposal" if counts.flagged_input == 1 else "proposals"`, yielding `1 proposal excluded` / `2 proposals excluded`. Every other disclosure in the links command family (`status` disclosure line 1674, `confirm-doc` line 1784, `reject-doc` line 1815, base-confirm success 1781) uses `N proposal(s)` without pluralization. The review disclosure is the only place that pluralises properly. Not a bug — the `(s)` form is established folio-cli idiom — but the review surface is the odd one out. Pick one convention and apply consistently (pluralising everywhere is more polished but breaks existing tests that assert substring `proposal(s)`). | `cli.py:1654-1657` vs. `cli.py:1781, 1784, 1812, 1815`. |
| D-PR-I-004 | informational | `links review` disclosure ordering | When both rejection-memory AND flagged-input suppressions exist, the review surface prints the rejection-memory line first (`cli.py:1646-1652`), then the flagged-input line (`cli.py:1653-1658`). Spec §5.3 says "after any revived / rejection-memory disclosure, print the flagged breakdown" — correctly implemented. This is the right ordering because rejection-memory is a more specific per-producer signal; flagged-input is a blanket trust gate. Recording the ordering decision explicitly so future edits don't silently swap them. | `cli.py:1646-1658`. |
| D-PR-I-005 | informational | Help-text vocabulary consistency across `--include-flagged` | Four different help strings: review says `Include proposals involving flagged-input documents (review_status: flagged).`; confirm says `Allow confirmation of a flagged-input proposal.`; reject says `Allow rejection of a flagged-input proposal.`; confirm-doc / reject-doc both say `Act on proposals involving flagged-input documents.` All four are meaningful and context-appropriate — review (surfacing language), confirm/reject (action-consent language), confirm-doc/reject-doc (bulk-action language). Only observation: review is the sole surface that names the frontmatter key `review_status: flagged` in the help text. Operators who help-poke at confirm/reject-doc first will have to cross-reference to learn what "flagged-input" means at the frontmatter level. Minor discoverability nit; not a blocker because the spec document and the disclosure lines make the vocabulary explicit. | Help output of all six commands, captured via `links <cmd> --help`. |

## End-to-end workflow trace

Simulated the spec's implied primary workflow:

1. Operator runs `folio links review` in a library with a flagged source → sees `Pending (0 items — 1 excluded because source or target has review_status: flagged; use --include-flagged to review)` header and a duplicate `1 proposal excluded (flagged inputs). Use --include-flagged to review.` disclosure. Both point at the same escape hatch. Not wrong — defense in depth — just worth noting the operator sees the exclusion signaled twice in close succession.
2. Operator runs `folio links review --include-flagged` → sees the proposal with `(flagged: source)` annotation. Clear trust posture.
3. Operator runs `folio links confirm <id>` → hits `Unknown proposal_id` (the CB-4 consent gate).
4. Operator runs `folio links confirm <id> --include-flagged` → `✓ Confirmed <id>: impacts -> tgt`. Frontmatter updated correctly.
5. Operator runs `folio links status` → sees source row with `Flagged Excluded = 1` and footer hint. Clear.
6. Operator attempts `folio links confirm-doc <flagged_source>` → `✓ Confirmed 0 proposal(s). 1 proposal(s) excluded from <src> (flagged inputs); use --include-flagged to act on them.` Actionable.
7. Operator runs `folio links confirm-doc <flagged_source> --include-flagged` → `✓ Confirmed 1 proposal(s).` No trust-posture annotation (see D-PR-SF-001).

The only surface operators cannot reach via help-text exploration is `folio links status --include-flagged` — because the flag doesn't exist (D-PR-001).

## Failure visibility

| Failure path | User-visible signal | Recovery? |
|---|---|---|
| Flagged source, default review | Silent-empty header + disclosure line, both naming `--include-flagged` | yes |
| Flagged target, default review | Same as above (source/target collapsed into "source or target") | yes |
| Flagged id on `confirm` / `reject` without flag | `Unknown proposal_id` | indirect (via `links status` cross-reference) |
| All-flagged source on `confirm-doc` / `reject-doc` | `Confirmed 0 proposal(s). N excluded... use --include-flagged` | yes |
| `--include-flagged` on clean-only library | Flag accepted, behaves identically to default | yes (no-op, not a trap) |
| Scope that matches no library paths | `No document-level relationship data found.` | yes |

No surface produces a dead-end state once D-PR-001 is resolved. The flagged-id→`Unknown proposal_id` path is the one indirect recovery; it's a conscious CB-4 tradeoff, not a UX bug.

## Test-coverage corroboration

The v0.6.4 test suite (`tests/test_links_cli.py:943-1337`) exercises the exact copy strings of the disclosure and silent-empty sentences (`test_links_review_silent_empty_protection` asserts `"0 items"`, `"excluded"`, `"--include-flagged"`; `test_links_status_discloses_flagged_excluded` asserts `"Flagged Excluded"` and `"flagged-excluded"`; `test_links_confirm_doc_discloses_flagged_excluded` asserts `"Confirmed 0 proposal(s)"`, `"flagged inputs"`, `"--include-flagged"`). Copy drift will fail tests. However, there is **no test for `folio links status --include-flagged`** — because the flag is not implemented. This absence is consistent with D-PR-001: the missing flag is missing end-to-end.

## Rationale

The core trust-gating loop (review → consent → act) is sound from the operator perspective. Spec §5 is mostly satisfied, copy strings are actionable, pluralization is mostly consistent, and the double-disclosure on review (header + trailing line) is a deliberate defense-in-depth choice that matches the spec literally. The single blocker is a spec-contract violation on `links status` — not an implementation bug, but an unfulfilled spec surface. The two should-fix items (confirm-doc trust-posture feedback, all-zeros column visibility) are UX polish that would bring the surface from "correct" to "helpful on first encounter." The informationals document intentional design tradeoffs that reviewers at canonical-verdict time should not mistake for defects.

Ship once D-PR-001 is resolved — either by adding the flag, or by revising spec §5.1 and §5.5 to match the implementation and removing the `folio links status --include-flagged` reference from the disclosure.
