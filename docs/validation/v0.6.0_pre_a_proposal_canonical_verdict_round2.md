## Rationale

Round 2's split mirrors Round 1's structurally but with the reviewers *swapped*: Round 1 had claude-sub raising blockers / gemini approving; Round 2 has gemini raising a blocker / claude-sub approving. In both cases the rule that decides consolidation is P5 (evidence-weighted consensus): a blocker with direct-citation evidence against the code preserves over an opposing family's Approve verdict when the opposing family did not engage with the specific code-behavior cited.

Gemini's Round 2 blocker PR-T-1-R2 is:

> "Rolling acceptance-rate gate requires ordering reviewed proposals chronologically. `folio/links.py` `reject_proposal` does not write a `rejected_at` timestamp, making it technically impossible to establish a global rolling window of the last 20 reviews without new persistence."

This is a fact about `folio/links.py`, verifiable by direct inspection:

- `folio/links.py` `confirm_proposal` records `confirmed_at` in `links.confirmed_relationships` — **timestamp present for accepted**.
- `folio/links.py` `reject_proposal` marks the proposal `status: "rejected"` but does **not** record a `rejected_at` timestamp (the proposal's `proposed_at` is the enrichment-run emission time, not the rejection decision time).

Claude-sub's Round 2 closure verdict for B-2 checked the *pending-queue-size* part of §9 carefully against `folio/links.py:107-155` and found it operationalizable (correctly). But claude-sub did not separately verify whether §9.2's "20 most recently *reviewed* proposals" rolling window was operationalizable — that check requires examining `reject_proposal`'s storage shape, not `collect_pending_relationship_proposals`'s filter. Gemini did that check; claude-sub did not. Per P5, the evidenced blocker preserves.

## Decision

**Round 2 canonical verdict = Revise and re-review.** Lifecycle does not advance to Phase A on this run.

Under the operator brief's halt list ("Pre-A verdict ≠ Proceed to Phase A → HALT") this is the second halt in two rounds. Framework default (playbook.md §13.5) allows a Round 3 only by explicit escalation (split, timebox with different reviewer, or strategic-decision pass) — not a routine continuation. Combined with the operator's strict halt rule, the session stops here.

If the operator elects to continue in a future session, the remaining work to reach Proceed is genuinely small:

- **Option (b) from the B-4 resolution table is the lowest-risk path:** a single-paragraph rewrite of §9.2 to frame the rolling window as *per-producer* over its own proposal history (which ordering *is* stable in `folio/links.py` via the in-list order of `_llm_metadata.<producer>.axes.relationships.proposals`). No code change, no new persistence, one proposal-doc edit. A third round of Pre-A on Rev 4 with just this fix should Proceed cleanly.
- **Option (a)** (add `rejected_at` in `folio/links.py`) is also small but is a code change, which blurs the proposal-revision / implementation boundary. Better to hold for Phase C.
- **Option (c)** (drop §9.2 entirely) is the cleanest scope reduction — acceptance-rate gating is arguably a producer-admission policy question, not a proposal-lifecycle-contract question, and may belong in a follow-up spec.
