# Product Review — folio-graph-generalized-proposals-v0-7-1 / B.1 / claude-sub

## 1. User-value assessment

The primary operator is Johnny, preparing SteerCo material. Today he runs
`folio graph doctor` to find pending relationship proposals and then
flips to `folio links review` to act on them. The only triage signal the
finding carries today is the free-text `detail` string:
`"{source_id} has pending {relation} -> {target_id} from {producer}"`
(folio/graph.py:171–174). That string is human-parseable but not
machine-parseable, and it omits three fields operators actually use to
sort the queue: trust (has an upstream flag been raised on source/target?),
rationale (why did the producer suggest this?), and evidence (what signals
back the claim?).

This spec surfaces `proposal_type`, `source_id`, `target_id`,
`evidence_bundle`, `reason_summary`, `trust_status`, `schema_gate_result`,
`producer`, `input_fingerprint`, `lifecycle_state` on the finding dict
(spec §2.1 B1, §3.1). That materially improves triage when the operator
consumes `--json`: a downstream script or digest can now filter on
`trust_status == "flagged"`, sort by `producer` acceptance rate, and show
`reason_summary` alongside the edge without dereferencing the underlying
`RelationshipProposalView`. That is a real UX win for the scripted /
digest consumer path.

However, the spec's user-value story has a gap on the **human-readable**
stdout path. The spec does not touch `folio/cli.py`'s text rendering
(folio/cli.py:2016–2023), which still echoes only
`[{severity}] {code} {subject_id}: {detail}` + `recommended_action`. After
this slice, `subject_id` on pending-proposal findings is `None` (§3.1
"Note on `subject_id` conflict"), so the operator running plain
`folio graph doctor` in a terminal will see the literal string
`"None"` where the proposal ID used to appear, and will not see any of
the new `proposal_type` / `trust_status` / `reason_summary` fields on
screen. The enriched JSON surface improves downstream tooling; the
stdout surface degrades for the operator-at-terminal. That's a
problem-solution-fit gap worth flagging at spec time.

The slice is otherwise well-calibrated to the job-to-be-done: it delivers
a stable shared-contract shape for the queue triage workflow without
forcing a new subcommand, without migration, and without churn on
`links.py` or `enrich_data.py`.

## 8. Positive observations

- **Contract names are used verbatim.** The spec commits to the §5 field
  names (`proposal_type`, `trust_status`, `evidence_bundle`,
  `reason_summary`, etc.) without renaming them for CLI convenience.
  That's the right call: a shared contract only delivers value if
  consumers agree on the names, and the graph renderer is one of three
  planned consumers.
- **Explicit `null` over silent omission** (§2.1 B2, §3.1 for
  `subject_id` and `schema_gate_result`). This is the operator-friendly
  choice: a consumer that checks `"schema_gate_result" in finding`
  works regardless of proposal_type, and a `null` value carries a clear
  "N/A today, may be populated later" semantic.
- **Legacy keys preserved.** Existing stdout consumers that parse
  `code`, `severity`, `detail`, `recommended_action` continue to work
  unchanged (§2.1 B1 trailing paragraph). The breaking change is
  isolated to the `subject_id` value semantic on the pending-proposal
  finding class — not a wholesale shape change.
- **OQ-3 explicitly commits to a CHANGELOG entry** for the breaking
  `subject_id` semantic change. Many past slices have left this
  implicit; calling it out at spec time is a real improvement.
- **Narrowness check** (test 12 in §4) — asserting other finding kinds
  do NOT gain the §5 shape — shows the author thought about blast
  radius and is protecting non-proposal findings from accidental
  shape creep.
- **OQ-2 resists premature abstraction** by keeping
  `_derive_trust_status` module-private until a second consumer shows
  up. That discipline is exactly right for a first sub-slice.

## 9. Verdict

Needs Fixes

Blocking: PRD-1 (stdout regression on literal `"None"`). Should-fix:
PRD-2 (breaking-change signaling), PRD-3 (`--help` copy), PRD-4
(trust-aware `recommended_action`).

The underlying product direction is sound and the JSON surface is a
real improvement for the scripted-triage workflow. The stdout path
regresses, the breaking change is under-signaled for JSON consumers,
and flagged trust status is informational rather than actionable. All
four are solvable with spec-level copy and rendering decisions before
Phase C.
