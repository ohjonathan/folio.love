# B.1 Adversarial Review — folio-digest-v0-7-0

## Verdict

Needs Fixes.

No blockers are recorded because all evidence in this review is static inspection. Per `frameworks/llm-dev-v1/templates/05-review-board-adversarial.md` §Evidence rule, blocker severity requires a reproduction; unreproduced failure hypotheses are downgraded to `should-fix` or `minor`.

The highest-risk failure mode is not LLM quality. It is inconsistent recovery behavior around source-less analysis rows: malformed frontmatter, write-then-register orphan files, and rerun corruption states are underspecified enough that Phase C could implement behavior that silently drops inputs or makes a digest invisible to weekly synthesis.

## Detailed findings

### ADV-001 - Source-backed frontmatter read failure is undefined

Severity: should-fix

Evidence: static-inspection

The daily predicate requires reading each source-backed entry's frontmatter via `registry._read_frontmatter`, then immediately evaluates `_is_flagged(fm.get("review_status"))` (`docs/specs/v0.7.0_folio_digest_spec.md:332`). The spec never states what happens when the markdown file is missing, has no frontmatter, or has malformed YAML.

That omission matters because the referenced implementation contract returns `None` for exactly those cases: `registry._read_frontmatter` catches `YAMLError` and `OSError` and returns `None` (`folio/tracking/registry.py:427`). The imported trust predicate itself is fail-open: `_is_flagged(None)` is false, and `_resolve_target_flagged` treats non-dict frontmatter as not flagged (`folio/links.py:74`, `folio/links.py:113`).

Failure mode:

An existing registry row points to `entry.markdown_path`, but the file was manually moved or its YAML frontmatter is malformed. A literal implementation of §5 step 3.4 calls `fm.get(...)` on `None` and crashes. A defensive implementation might skip the input, which is fail-closed and diverges from the v0.6.4 trust contract the spec cites. Either way, the digest input set and `excluded_flagged_count` become implementation-dependent.

Suggested action:

Specify the failure policy explicitly. For source-backed trust filtering, mirror the existing contract: if frontmatter cannot be read as a dict, treat `review_status` as not flagged for the trust exclusion decision, optionally surface a warning/count. For weekly digest discovery, define a separate policy: if a candidate daily digest's frontmatter cannot prove `subtype: digest`, `digest_type: daily`, and a valid `digest_period`, skip it with a diagnostic. Add tests for missing file, malformed YAML, and no frontmatter.

### ADV-002 - Write-then-register creates orphan digests with bad recovery guidance

Severity: should-fix

Evidence: static-inspection

The spec orders rerun write and registration such that the target file can be written before the registry is updated (`docs/specs/v0.7.0_folio_digest_spec.md:618`, `docs/specs/v0.7.0_folio_digest_spec.md:621`). §11 acknowledges the failure state: "digest file may exist on disk but registry was not updated" and tells the operator to run `folio refresh` or rebuild (`docs/specs/v0.7.0_folio_digest_spec.md:669`).

That recovery guidance is not correct for a valid registry that merely lacks the new digest row. `cli.refresh` iterates existing `registry["decks"]` rows and skips `analysis` entries already present in the registry (`folio/cli.py:1018`, `folio/cli.py:1035`). It does not scan the filesystem for a newly orphaned source-less analysis file. `rebuild_registry`, by contrast, does scan markdown files and retain source-less `type: analysis` docs (`folio/tracking/registry.py:145`, `folio/tracking/registry.py:151`).

The atomic write snippet also uses `NamedTemporaryFile(delete=False)` and does not define cleanup if `tmp.write(...)` or `os.replace(...)` fails (`docs/specs/v0.7.0_folio_digest_spec.md:631`, `docs/specs/v0.7.0_folio_digest_spec.md:638`). The target file is preserved, but stale `.tmp` files can accumulate in digest directories.

Failure mode:

A process crash after `os.replace(tmp_path, path)` but before `registry.upsert_entry(...)`, or a registry save failure after a successful file write, leaves a valid digest on disk with no registry row. Weekly discovery consumes registry rows, so it will miss the daily digest. Running `folio refresh` against a non-corrupt registry will not add the orphan. The digest becomes recoverable only through a full registry rebuild or by rerunning the exact same digest command, but neither recovery path is specified.

Suggested action:

Define this as an explicit repairable state. Options:

- On digest start, if the computed path exists but the registry row is missing, load the existing digest frontmatter and upsert the row before or during the rerun.
- Replace the §11 recovery text with exact operator guidance that actually repairs the state, such as rerunning the same digest command or performing a full registry rebuild.
- Add `try/finally` cleanup for temporary files when `os.replace` fails or when a temp write raises after the file is created.
- Add tests for "file exists, registry missing", "registry exists, file missing", and "upsert raises after atomic write".

### ADV-003 - Offset timestamps can be assigned to the wrong digest day

Severity: should-fix

Evidence: static-inspection

The spec says daily activity-date comparison uses string equality on `YYYY-MM-DD`; if a value contains `T`, the date portion from `split("T")[0]` is used (`docs/specs/v0.7.0_folio_digest_spec.md:341`). It also describes the comparison target as `day` / a "local date string" (`docs/specs/v0.7.0_folio_digest_spec.md:331`), while the CLI default-date test only says "local current date" without defining which timezone is authoritative (`docs/specs/v0.7.0_folio_digest_spec.md:744`).

Failure mode:

`modified: 2026-04-04T23:00:00-05:00` is April 4 in US Eastern wall time but April 5 in UTC. The specified `split("T")[0]` behavior assigns it to `2026-04-04` unconditionally. If the operator or another producer interprets stored timestamps as UTC, `folio digest <scope> --date 2026-04-05` silently excludes the note and `--date 2026-04-04` includes it. The same ambiguity appears around Sunday/Monday ISO-week boundaries if default `--date` is computed in one timezone and input timestamps are produced in another.

This is not covered by DG-13, which only tests `2026-04-04T13:00:00Z` where the split date and UTC date are the same (`docs/specs/v0.7.0_folio_digest_spec.md:717`).

Suggested action:

Choose one contract and test it:

- Strict date-only contract: if `modified` or `converted` contains `T`, skip it with a diagnostic instead of partially parsing it.
- Timestamp contract: parse ISO timestamps with offsets and normalize to a named timezone, such as operator local timezone or UTC, before comparing the digest day and ISO week.

Add offset-crossing tests, including one daily case and one ISO-week boundary case.

### ADV-004 - Rerun corruption cases are underspecified

Severity: should-fix

Evidence: static-inspection

The spec defines `_load_existing_digest(path)` only as "Read frontmatter for rerun version increment" (`docs/specs/v0.7.0_folio_digest_spec.md:208`). Rerun then performs `existing.get("version", 1) + 1` and preserves `created` (`docs/specs/v0.7.0_folio_digest_spec.md:488`). §11 does not list corrupt existing digest frontmatter, missing `created`, invalid `created`, or non-integer `version` as failure cases (`docs/specs/v0.7.0_folio_digest_spec.md:660`).

Failure mode:

The existing digest path exists, but frontmatter has any of these shapes:

- malformed YAML
- no closing frontmatter fence
- `version: "1"` or `version: one`
- `version: null`
- `created: not-a-date`

Depending on how Phase C implements `_load_existing_digest`, rerun may crash after the LLM call, overwrite the file as `version: 1`, preserve invalid audit metadata, or silently drop the original `created` value. The spec also only documents the fully manual cleanup path where both digest file and registry entry are deleted (`docs/specs/v0.7.0_folio_digest_spec.md:653`); it does not define partial divergence states such as registry row present but file missing, or file present but registry row missing.

Suggested action:

Add a rerun-state matrix:

| State | Required behavior |
|---|---|
| Existing file has valid digest frontmatter and integer version | Increment version and preserve `created`. |
| Existing file has unreadable or non-dict frontmatter | Fail non-zero and preserve file; tell operator to repair or delete the file. |
| Existing version missing | Treat as `1` only if this is an intentional compatibility policy. |
| Existing version non-integer | Fail non-zero and preserve file. |
| Existing created invalid or missing | Define whether to fail or replace with today's date. |
| Registry row exists but file missing | Define fresh regenerate vs explicit repair error. |
| File exists but registry row missing | Upsert registry row on rerun or tell operator the exact rebuild command. |

### ADV-005 - LLM-supplied Trust Notes can collide with the authoritative Trust Notes

Severity: should-fix

Evidence: static-inspection

The daily body requires `## Trust Notes` and says `_validate_body_sections` enforces presence of headings (`docs/specs/v0.7.0_folio_digest_spec.md:540`). The weekly body has the same requirement (`docs/specs/v0.7.0_folio_digest_spec.md:575`). Separately, §9.3 says Trust Notes content is rendered programmatically and is not LLM-generated (`docs/specs/v0.7.0_folio_digest_spec.md:581`).

The spec does not define how the renderer strips or rejects LLM-generated `## Trust Notes` content before adding the authoritative block. DG-24 asserts that the Trust Notes section matches `_compose_trust_notes` "regardless of LLM body" (`docs/specs/v0.7.0_folio_digest_spec.md:728`), but it does not require exactly one `## Trust Notes` heading in the final body.

Failure mode:

The LLM returns:

```markdown
## Suggested Follow-Ups
- Escalate this.

## Trust Notes
All source notes were verified and safe.
```

If the renderer appends the programmatic Trust Notes block rather than parsing and replacing sections, the final digest contains two `## Trust Notes` sections. The first one is LLM-authored and contradicts the trust posture. Readers and downstream parsers can consume the wrong one.

Suggested action:

Specify a section assembly algorithm:

1. Parse LLM output into expected sections.
2. Reject duplicate required headings and unexpected top-level `##` headings, or drop LLM content from deterministic sections.
3. Always render `Documents Drawn From` / `Daily Digests Drawn From` and `Trust Notes` from programmatic data.
4. Assert the final body contains exactly one Trust Notes heading.

Add a regression test with an adversarial duplicate `## Trust Notes` heading.

### ADV-006 - LLM retry semantics are underdefined

Severity: should-fix

Evidence: static-inspection

The spec says missing headings cause an LLM "re-prompt (one retry max)" (`docs/specs/v0.7.0_folio_digest_spec.md:543`) and that malformed output after one retry fails non-zero (`docs/specs/v0.7.0_folio_digest_spec.md:667`). It does not define the retry prompt, whether the retry is the same prompt or a corrective prompt, whether the model must regenerate the entire body, or how the retry is counted in result metadata and tests.

Failure mode:

A deterministic model that omitted `## Trust Notes` once will likely omit it again if called with the same prompt and same temperature. A corrective prompt may succeed, but different implementations will produce different bodies and different token costs. Partial repair is also risky: if implementation tries to splice a missing heading into the first output, it can preserve malformed or adversarial section content that the retry was intended to replace.

Suggested action:

Define retry semantics as a contract:

- one additional LLM call total after the first malformed body
- retry prompt includes the exact missing headings and asks for a complete replacement body
- no partial merge between first and retry output
- deterministic sections are still rendered post-LLM
- tests assert call count and final failure message includes missing headings from the retry output

### ADV-007 - Refresh compatibility is claimed but not covered by tests

Severity: should-fix

Evidence: static-inspection

The design spec requires `refresh` to skip `analysis` rows with rerun guidance to use `folio digest` or `folio synthesize` (`docs/specs/tier4_digest_design_spec.md:197`). §8.4 claims refresh compatibility as a verification behavior (`docs/specs/v0.7.0_folio_digest_spec.md:502`), and §8.4 says the verification tests fail if an existing command breaks on a source-less analysis row (`docs/specs/v0.7.0_folio_digest_spec.md:506`).

But the registry-compatibility test plan includes status, scan, rebuild, and enrich only; it omits refresh (`docs/specs/v0.7.0_folio_digest_spec.md:755`). Current `cli.refresh` does skip `analysis` rows, but its guidance says "rerun the originating analysis workflow instead", not `folio digest` (`folio/cli.py:1053`).

Failure mode:

Phase C can ship a digest row that does not break current refresh, but the required digest-specific operator guidance remains untested and deferred. This combines badly with ADV-002: if a digest file is orphaned after write-then-register failure, the spec tells the operator to run `folio refresh`, but refresh neither discovers the orphan nor names `folio digest` as the repair path.

Suggested action:

Either add `DG-REG-5 test_refresh_skips_digest_with_digest_guidance` and update `cli.refresh` guidance for `subtype: digest`, or remove the §8.4 claim that refresh compatibility is verified in this slice and mark the deferral as a real design variance rather than a UX nicety.

### ADV-008 - Forbidden-symbol coverage is too narrow

Severity: minor

Evidence: static-inspection

The spec only forbids `pickle.loads` and `eval(` (`docs/specs/v0.7.0_folio_digest_spec.md:118`). The digest module will own LLM body validation/rendering and existing digest frontmatter loading. The referenced registry reader uses `yaml.safe_load` (`folio/tracking/registry.py:445`), but the spec does not explicitly require `_load_existing_digest` to use that safe reader. It also does not forbid adjacent code-execution primitives such as `exec`, `compile`, `subprocess(..., shell=True)`, `os.system`, or unsafe `yaml.load`.

Failure mode:

The Phase C implementation can pass the declared `G-scope-2` checks while still adding a dangerous LLM-output sanitization path or unsafe YAML loader. This is unlikely in a narrow implementation, but the guard does not protect the security surface it implies.

Suggested action:

Extend the forbidden-symbol and grep guard to include `exec(`, `compile(`, `os.system`, `subprocess` with `shell=True`, and `yaml.load(` unless using `SafeLoader`. Also state that `_load_existing_digest` must call `registry._read_frontmatter` or `yaml.safe_load`.

### ADV-009 - Manifest is absent at the referenced path

Severity: should-fix

Evidence: static-inspection

The dispatch identifies `frameworks/manifests/folio-digest-v0-7-0.yaml` as the manifest with cardinality, scope-lock, and gate prerequisites. In this workspace, that file is absent. The spec still relies on "per manifest" allowed paths, forbidden paths, forbidden symbols, and regression guards (`docs/specs/v0.7.0_folio_digest_spec.md:100`, `docs/specs/v0.7.0_folio_digest_spec.md:110`, `docs/specs/v0.7.0_folio_digest_spec.md:118`, `docs/specs/v0.7.0_folio_digest_spec.md:764`).

Failure mode:

Reviewers cannot independently verify the scope-lock, cardinality-required exports, or gate prerequisites that the spec claims to narrow. A Phase C worker could implement against the prose but miss manifest-enforced constraints, or the manifest could later appear with conflicting paths/gates after this review has already approved the spec.

Suggested action:

Add or restore the manifest at the dispatched path before Phase B closes, then rerun the scope/cardinality portions of review against it. If the manifest intentionally moved, update the dispatch and spec references so reviewers are not validating against a missing contract.
