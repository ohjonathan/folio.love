# Product Review — proposal-review-hardening-v0-6-0 / B.1 / claude-sub

## Verdict

**Needs Fixes**

The spec unquestionably targets the right user problem and picks a well-
scoped slice. It also already resolves — via its Phase-A §4 aggregator
and §2 out-of-scope carve — several product traps that a less disciplined
spec would have fallen into (enforcement scope creep; unobservable
enforcement without review-surface feedback; requiring timestamps Johnny
can't see). But the spec under-commits on **user-facing copy inventory**
for the two new output strings it adds (suppression line and revival
annotation), under-specifies the **doctor-report empty state / sort-order
/ warm-up copy clarity**, and leaves the **operator mental model for
"below-gate → what do I do"** unstated. All three are Phase-B.1 copy-
inventory gaps per Template 19 §2.1, which classifies "spec prose without
exact copy strings" as blocking when the copy IS the contract. I have
scoped these to should-fix rather than blocker, because the spec
otherwise delivers a coherent, observable, operator-legible slice and
the missing copy is narrow and easy to inventory in a revision pass.

## User-value assessment

The spec cleanly ties to the concrete operator pain from the proposal
doc §1 and PRD FR-813: Johnny, mid-SteerCo prep, rejects a machine
suggestion and then sees the same suggestion again on the next review
pass. Rev 4's committed slice — rejection-memory filtering at
`collect_pending_relationship_proposals` — is the minimum observable
fix: after ship, a rejected proposal with an unchanged `basis_fingerprint`
never returns to the review list, and the suppression-count disclosure
proves the filter is working rather than silently dropping results.
The user-value claim holds. The doctor acceptance-rate diagnostic is a
secondary value surface (operator introspection into producer health
without firing any test suite) and is genuinely additive — the existing
`folio graph doctor` already speaks the "actionable findings" idiom, and
the new table extends that idiom rather than inventing a parallel one.

One caveat on user-value: the revival path (proposal P3 with new
`basis_fingerprint` surfacing with a "revived (basis changed)"
annotation per §10.2 sequence diagram) is the *most operator-impactful*
copy string in the entire slice — it's the moment Johnny has to decide
whether a proposal he *already rejected* deserves review again because
the evidence changed. That copy does not appear in §4 and is only
mentioned in §5 OQ-2 as "Proposed". It is the load-bearing failure-
visibility string for the whole feature. Leaving it in OQ for Phase B
resolution is acceptable *only if* Phase B commits before implementation;
it cannot resolve silently at code time.

## Failure-visibility findings

| ID | Description | Evidence |
|----|-------------|----------|
| FV-1 | The spec's §8 adversarial-notes row 4 ("What if `_llm_metadata.links.confirmed_relationships` is missing the `producer` field on older accepted entries?") says the aggregator "must tolerate missing `producer` and exclude those entries from per-producer rates **with a warning**". That warning is a new failure-visibility surface but has no copy inventory. Johnny runs doctor, sees a table, and also sees … a warning? An aside? An entry in a separate "data-integrity issues" list? This is an operator-facing surface spoken of in §8 but not surfaced in §4 or §10 as a rendered artifact. | static-inspection, spec §8 adversarial row 4 |
| FV-2 | Section 10.2 diagram describes: "if `basis_fingerprint` is empty-string or missing on a rejected entry … Producers with empty fingerprints cannot participate in rejection memory; documented as a producer defect." The operator observable for this defect is nowhere in §4. Johnny's intended behavior (rejection-memory) silently no-ops for affected producers, and the only evidence is a log warning (?) or a post-hoc test. This is the exact "filter false-positive that defeats the feature" risk the spec itself names in §8 as severity High. Mitigation: surface a count in the suppression-count dict, e.g., `dict[str, int]` gains a sentinel entry like `{"<producer>": 0, "__malformed__": <n>}`, OR the doctor report gains a data-integrity line. As currently speced, operator discovers the defect only by reading code or running tests. | static-inspection, spec §10.2 error-path note and §8 risk row 2 |
| FV-3 | The diagnostic-only scoping (§2 out-of-scope row 2, OQ-3 recommendation a) means when a producer drops to `below-gate`, **nothing happens to Johnny's review queue**. The operator could plausibly assume the diagnostic means "this producer is being throttled" and wonder why they keep seeing its noisy proposals. The spec should add one sentence to the doctor-table render note clarifying: "Status column is informational only; `below-gate` producers continue to surface proposals at normal priority in this slice." Without that, the status label promises more than the feature delivers. | static-inspection, spec §2 out-of-scope row 2 |
| FV-4 | When `folio links review` is run in a library with **zero producers that have ever rejected** (a common state for newly-adopted Folio), the new suppression-count line either renders as `"0 suppressed"` or is elided. Both are defensible choices; spec doesn't pick one. Quiet-by-default (elide when count is zero) matches terminal-output conventions; explicit (always render, even as `"0 suppressed"`) makes the feature's existence discoverable. No inventory in spec. | static-inspection, spec §4.2 |

## Scope / calibration findings

| ID | Description | Rationale |
|----|-------------|-----------|
| SC-1 | The choice to scope this slice to **surfacing-time** filtering (not emission-time) is the right product call for v0.6.0. The spec §2 rationale ("Surface-time filtering fixes the operator-observable symptom … with a bounded change") tracks: Johnny cannot distinguish "rejected proposal re-emitted but hidden before I see it" from "rejected proposal never re-emitted"; both read as "I don't see the ghost anymore". Deferring emission-time hardening is correct; no calibration issue. | static-inspection, spec §2 out-of-scope row 1 |
| SC-2 | The "diagnostic only, no enforcement" scoping for the acceptance-rate gate (§2 out-of-scope row 2) is a product choice worth pushing on. Rev 4 of the proposal doc §9.2 describes the gate as restricting default surfacing ("producers below the gate after warm-up continue to emit proposals, but those proposals land in a non-default-surfaced state visible only via explicit operator inspection"). The spec under review downgrades this to pure diagnostic. This is defensible for v0.6.0 (enforcement needs override semantics), but it **creates a gap between what the proposal-doc describes and what v0.6.0 ships** — Johnny reading the proposal expects surfacing restriction; v0.6.0 only gives him a table row. The spec should explicitly note this gap in the doctor-table render note so operators do not misread the diagnostic as active enforcement. See FV-3 for the copy fix. | static-inspection, spec §2 row 2 vs proposal rev 4 §9.2 |
| SC-3 | The slice's test plan (§4.5 / §4.6) covers the filter and the aggregator comprehensively, but does not propose a test for the *product* invariant: "after a reject + rerun, Johnny sees the suppression-count line". The integration test (§6 `test_links_review_workflow_respects_rejection_memory`) implicitly covers this but asserts the rejected proposal isn't re-appearing; it doesn't assert Johnny sees the disclosure proving the filter ran. Product-lens gap — the disclosure IS the feature (per §2 out-of-scope's reasoning "fixes the operator-observable symptom"); the test should assert the observable. | static-inspection, spec §4.5 + §6 |
| SC-4 | The 10-reviewed warm-up threshold is specified once (§4.3, proposal rev 4 §9.2). No test calibrates it against a realistic Johnny-library size. The test §4.6.3 uses "5 reviewed" and §4.6.1 uses "16 reviewed" — both synthetic. Not a blocker (the threshold is a single constant, reviewable in isolation), but a product note: on a fresh engagement library with one producer just enabled, all producers render `warmup` for the first several weeks. That's the right behavior (suppresses small-sample noise) but means the doctor table is nearly empty of signal during the most fragile period. No remediation; just noting that Phase-C manual-test §6 step 2 should explicitly validate the warm-up-state operator experience, not just the post-warm-up state. | static-inspection, spec §6 manual testing |
| SC-5 | Scope calibration is clean: single-slice, reverts cleanly (§7 rollback plan), touches two files, adds 10 tests. This is the right shape for a v0.6.0 increment. Praise noted. | static-inspection, spec §2 + §7 |

## Notes

- The spec is well-written and has clearly absorbed the three rounds of
  Pre-A review. User-facing surfaces are named correctly and the
  operator persona is concrete. The "Needs Fixes" verdict rests entirely
  on copy inventory gaps per Template 19 §2.1 — fixable in a narrow
  revision without re-running Pre-A.
- The most consequential single fix is UX-2 (revival annotation). If
  only one copy fix lands in Phase B, this is the one.
- Operator-mental-model recommendation for Phase B: add a one-line
  "what does this mean for me?" note under the doctor table that maps
  each status to operator action. Example: `below-gate: informational
  in v0.6.0 — review the producer's recent rejections manually; no
  surfacing change yet.` This eliminates FV-3 and SC-2 simultaneously.
- Accessibility is a non-issue for this slice: CLI-only, plain-text,
  pipe-table, no color dependency, no unicode traps. CPY-4 confirms
  parity with existing idioms. No remediation needed.
- The spec correctly defers emission-time rejection memory and gating
  enforcement. Both deferrals have defensible product rationales in §2.
  Do not re-open them in Phase B absent new evidence.
- Positive observation: the spec names Johnny explicitly (user-facing
  surface definition in the task brief, proposal rev 4 §1). Many specs
  abstract the operator into "the user"; naming the persona keeps the
  product lens concrete.
