# Adversarial Review — proposal-review-hardening-v0-6-0 / D.1 / claude-sub

## Verdict

**Needs Fixes.** One blocker reproduced by direct run: malformed proposal dicts with missing `relation` / `target_id` keys crash `folio links review`, `folio links status`, `folio graph status`, `folio graph doctor` (all seven call sites). This is a regression versus v0.5.2-era behavior — the spec's §4.5 test 13 (`test_filter_tolerates_malformed_proposal_entries`) only tolerates non-dict / null entries; dicts missing required scalar fields are not guarded. Plus several should-fix and minor notes below.

## 1. Assumption attack

| Assumption | Why it might be wrong | Impact if wrong |
|------------|------------------------|-----------------|
| Every frontmatter proposal dict has `relation` and `target_id` keys. | `RelationshipProposal.from_dict` hard-indexes `d["relation"]` and `d["target_id"]`. A pre-v0.6.0 file, a hand-edit, or a partially-written proposal breaks this. | Blocker — `KeyError` bubbles up through `collect_pending_relationship_proposals` → seven commands crash, not degrade. |
| Producer identity = outer key in `_llm_metadata`. | `_upsert_confirmation_record` writes the `producer` field from `proposal.producer` (which defaults to the outer key via `raw.setdefault`, but a producer *could* emit a proposal with a different producer field). | Acceptance (from confirmed-record `producer` field) and rejection (from outer frontmatter key) can disagree, producing two phantom producers with complementary bars. |
| `_iter_producer_proposals` structurally rejects the reserved `"links"` key because it lacks an `axes` child. | That's accidental filtering, not intentional — a future edit that adds `_llm_metadata.links.axes.x.y` would start leaking `links` into the producer axis. | Silent regression — tests wouldn't catch it; spec says skip `links`, code only skips in `_aggregate`. |
| `_aggregate` always produces deterministic output across Python versions. | `_aggregate_producer_acceptance_rates` iterates `llm_meta.items()` in dict-insertion order for per-producer counting, which is fine; however, `producers = sorted(set(accepted) | set(rejected))` is only stable after status-bucket sort. `_STATUS_SORT_RANK.get(..., 9)` means unknown statuses tie at 9 and then fall back to alphabetic — arbitrary status strings would mix into one bucket. | Minor. |
| `basis_fingerprint` field is always a string. | The code trusts YAML to round-trip it as `str`. A YAML-malformed `basis_fingerprint: 0x42` would parse as int → `_build_rejection_key` tuple contains int → equality still works for matching entries but mixed-type keys would surprise downstream observability. | Minor. |

## 2. Failure mode analysis

| Failure | How it happens | Would we notice? |
|---------|----------------|-------------------|
| `KeyError('relation')` at `pipeline/enrich_data.py:87` propagates through `_iter_producer_proposals`, crashing ALL seven call sites (review, status, confirm, reject, confirm-doc, reject-doc, doctor). | One bad legacy proposal dict is sufficient. **Reproduced** below. | Only via crash stack; `folio links review` becomes unavailable until the operator manually edits frontmatter. |
| Filter double-iterates `_iter_producer_proposals` per document (once for rejection-key build, once for pending views). | Implementation choice. | Minor perf only; hidden until libraries grow. |
| Within-bucket sort is `_STATUS_SORT_RANK.get(r.status, 9)` — any misspelled or future status lands in bucket 9 and sorts after "warmup". | If a future writer adds `"conflicted"` status, it appears at bottom, confusing operators. | Yes (visible in table) — low severity. |
| Revival marker depends on `rejected_fps_by_prefix` being populated in the first sweep. A proposal missing `target_id` would KeyError before the map is built → no revival data produced. | Same KeyError root cause. | No. |

## 3. Diagram completeness attack

§10.2 sequence shows error/retry paths for "malformed YAML" → existing `ValueError` flow. The diagram does **not** cover a well-formed YAML with partial proposal dicts (missing `relation` or `target_id`), which the code does not handle. This is a prose/diagram gap that matches the implementation gap (blocker X-1).

## 4. Edge case inventory

- **Dict proposal missing `relation` / `target_id`** — crashes (blocker X-1).
- **Duplicate rejected entries with identical fingerprint** — counted once in `rejected_keys_by_producer` set (fine). Identical fingerprint rejected twice under same producer is a no-op for suppression count.
- **Same pending fp emitted twice by producer in one sweep** — both suppressed under rejection memory; `suppression_counts[producer]` increments twice. Counter semantics match spec's "aggregated across call" wording; verified by trace.
- **Rejected fp=X and fp=Y, pending fp=Z (same prefix)** — direct-run confirms `revived=True`. Correct.
- **Pending with NO prior rejection under same prefix** — direct-run confirms `revived=False`. Correct.
- **Empty-string `basis_fingerprint`** — §10.2 calls this a "producer defect"; code allows it. Collisions only match other empty-fp entries, so behavior is bounded.
- **Empty-string producer key (`_llm_metadata[""]`)** — would be enumerated as a producer with empty-string name; no guard. Minor.
- **Producer whose confirmed-record `producer` field disagrees with outer key** — acceptance rate bifurcates between the two identities.
- **`--limit` applied to findings does not apply to acceptance-rate table** — matches spec §4.4, not a bug.

## 5. Security surface

No authN/authZ; no network; filesystem writes gated by `library_lock` on confirm/reject. `graph doctor` / `links review` are read-only, do not take the lock; a concurrent `reject` while doctor is reading could show a stale rate row — tolerable. No injection surfaces. `--json` output uses `json.dumps`, safe. No secret exposure.

## 6. Blind-spot identification

- Author treats "malformed" as "not a dict" (§4.5 test 13) but skips "dict with missing scalar keys". Legacy files predating schema tightening, user hand-edits, or interrupted writers produce exactly this shape.
- Author relies on `_proposal_from_raw` normalization, but that function only sets defaults for `producer`, `source_id`, `proposal_id` — NOT `relation`/`target_id`. The guard the spec pseudo-code at §4.2 lines 156-160 claims (`_proposal_from_raw`-style safe access) is not actually present for the mandatory fields that crash.
- Test `test_filter_handles_missing_source_id_target_id` was listed in spec §4.5 test 11 but I see no matching test name in `tests/test_links_cli.py`; the analogous test is `test_filter_tolerates_malformed_proposal_entries` with only `None` / string entries — **the SF-2 safety claim is partially missed**.

## 7. Risk-assessment override

Author's §8 `medium` rating understates the blast radius of H1. A single malformed proposal dict crashes seven CLI surfaces, and this is a realistic shape (legacy notes, hand-edits). Recommend upgrading the **filter-correctness risk** severity to `high` until H1 is fixed; feature-level risk stays `medium`.
