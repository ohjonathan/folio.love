# D.2 Adversarial Code Review - `folio synthesize` v0.8.0

## Verdict

BLOCK.

The CB-1/CB-2 shared-envelope regressions appear closed in the implementation: `excluded_flagged_count` is routed from `counts.flagged_input`, all 11 shared-contract finding keys are emitted through `asdict()`, `schema_gate_result` is literal `None`, and `input_fingerprint` is sourced from `basis_fingerprint`.

However, the code violates the v1.1 CLI scope contract in two ways that the supplied tests can pass: document-ID scopes are not implemented, and invalid scopes return success despite the spec requiring a non-zero scope-resolution failure.

## Blocking Findings

### ADV-D2-001 - Document-ID scope contract is not implemented

The spec says `<scope>` resolves to an engagement subtree, document ID, or `-` for library-wide (`docs/specs/v0.8.0_folio_synthesize_spec.md:153-156`, `docs/specs/v0.8.0_folio_synthesize_spec.md:530`). The collector has an explicit document-ID path: `doc_id` is accepted at `folio/links.py:203-210` and applied as `entry.id != doc_id` at `folio/links.py:217-220`.

`synthesize()` never uses that path. It normalizes only `None` and `"-"` and always passes the user string as `scope`:

- `folio/synthesize.py:62-65`

The collector's `scope` branch only checks `entry.markdown_path` and `entry.deck_dir`:

- `folio/links.py:221-225`

So `folio synthesize <document_id>` will be treated as a path scope, not a document ID. Unless a document ID accidentally equals a markdown path or deck directory, the command returns zero findings. That is a direct contract violation and can also trigger the misleading zero-findings breadcrumb.

Required fix: resolve the incoming argument before collection, or teach the collector/synthesize wrapper to route exact registry IDs through `doc_id=...` while preserving subtree/path scopes through `scope=...`.

### ADV-D2-002 - Invalid scope exits successfully, and the CLI test blesses the violation

The spec still requires invalid scope to be a scope-resolution failure:

- Exit-code contract: `docs/specs/v0.8.0_folio_synthesize_spec.md:586-590`
- Failure mode: `docs/specs/v0.8.0_folio_synthesize_spec.md:663-666`

The CLI does no validation. It calls `run_synthesize(...)` and renders whatever report comes back:

- `folio/cli.py:1984-1991`

The implementation likewise does no validation and returns a report for any string:

- `folio/synthesize.py:62-91`

The collector silently returns an empty list for a scope that matches no entries:

- `folio/links.py:217-225`
- `folio/links.py:303`

The test named `test_synthesize_invalid_scope_exits_nonzero` now asserts the opposite of its name and the spec: it documents current behavior and accepts exit code `0`:

- `tests/test_cli_synthesize.py:308-318`

This is not a spec issue; it is a code/test mismatch. The implementation can report "Findings: 0" for typos or nonexistent scopes, which violates the stated CLI contract and hides user error.

Required fix: add an explicit scope-resolution step that can distinguish "valid but empty" from "unmatched scope", return/raise a Click error for unmatched scope, and restore SYN-CLI-7 to assert non-zero exit.

## Nonblocking Should-Fix Items

### ADV-D2-SF-001 - Negative `--limit` has Python slice semantics

The CLI accepts any integer (`folio/cli.py:1963-1968`), and `synthesize()` applies `views[:limit]` directly (`folio/synthesize.py:66-67`). A negative limit emits "all but N from the tail" instead of rejecting the value or producing zero findings. The sibling graph code clamps negative limits with `max(0, limit)` (`folio/graph.py:350-351`).

Fix with `click.IntRange(min=0)` or an explicit validation error. Add a CLI test for `--limit -1`.

### ADV-D2-SF-002 - Empty-string scope is treated library-wide but reported as `""`

Programmatic `synthesize(scope="")`, and shell invocation with an explicit empty argument, pass `""` through `resolved_scope` (`folio/synthesize.py:62`) and store it in the report (`folio/synthesize.py:87`). The collector treats empty string as falsy and therefore library-wide (`folio/links.py:221-225`), but the JSON envelope reports `"scope": ""` instead of `null` (`folio/synthesize.py:98`). This violates the library-wide `null` convention in edge cases.

Fix by normalizing empty/blank scope consistently, or rejecting it as invalid input.

### ADV-D2-SF-003 - Version bump enforcement is test-only, not code-level

`render_envelope()` hard-codes the v1.0 shape (`folio/synthesize.py:94-102`), and `SCHEMA_VERSION` is an independent constant (`folio/synthesize.py:26`). The current unit test would catch an added top-level key at v0.8.0 because it asserts exact keys (`tests/test_synthesize.py:207-219`), but production code does not couple envelope keys to schema minor version.

This is acceptable for v0.8.0 if tests remain authoritative, but future v0.8.1 work can silently add a key without a runtime guard. Consider centralizing `ENVELOPE_KEYS_V1_0` and adding an explicit "if keys change, schema version must change" test.

### ADV-D2-SF-004 - Frozen dataclasses are shallow; nested lists remain mutable

`SynthesisFinding` and `SynthesisReport` are frozen dataclasses (`folio/synthesize.py:30-52`), but they contain mutable `list` fields (`folio/synthesize.py:38`, `folio/synthesize.py:50`). The implementation does correctly copy upstream signals with `list(view.proposal.signals)` (`folio/synthesize.py:76`), and `asdict()` emits the keys (`folio/synthesize.py:101`), but callers can still mutate `report.findings` and `finding.evidence_bundle` after creation.

Not a blocker unless immutability is part of the public API, but the current type shape should not be treated as deeply immutable.

### ADV-D2-SF-005 - SYN-15 does not catch several forbidden-symbol bypasses

The forbidden-symbol test combines substring checks and a narrow AST check:

- `tests/test_synthesize.py:278-309`

It would not catch patterns such as `getattr(os, "system")(...)`, `getattr(module, "system")(...)`, `subprocess.Popen(..., shell=True)`, or aliases that hide `subprocess.run`. Current production code is clean, but the guard is weaker than the spec-side forbidden-symbol intent.

### ADV-D2-SF-006 - SYN-24 read-only invariant has false negatives

The spec describes registry and library-file read-only behavior, but SYN-24 only snapshots `registry.json` bytes:

- `tests/test_synthesize.py:448-458`

It would miss document mutations elsewhere in the library, new-file create/delete sequences, and registry writes that restore identical bytes before assertion. Current production code appears read-only, but the test is not a complete mutation guard.

## Attack Surface Results

- CB-1 flagged-count routing: PASS. `synthesize()` returns `excluded_flagged_count=counts.flagged_input` (`folio/synthesize.py:86-90`) and does not use `SuppressionCounts.total()` or `rejection_memory`.
- CB-2 shared finding keys: PASS. `SynthesisFinding` declares all 11 shared-contract keys plus `proposal_id` and `relation` (`folio/synthesize.py:30-44`); `schema_gate_result` is literal `None` and `input_fingerprint` is `basis_fingerprint` (`folio/synthesize.py:69-83`); `render_envelope()` uses `asdict(f)` and does not drop fields (`folio/synthesize.py:94-102`).
- CB-3 producer-exact versioning: PARTIAL. Current tests catch v0.8.0 top-level drift (`tests/test_synthesize.py:207-219`), but code does not enforce version/key coupling.
- CB-4 help copy: PASS. The Click docstring contains "v0.8.0 structural MVP", "Narrative synthesis", and "planned for a future version" (`folio/cli.py:1971-1976`).
- CB-5 zero-findings diagnostic: PASS for the specified zero-findings/zero-exclusions path. `_render_synthesis_stdout()` prints the excluded count always and prints `Next: run...` only when findings are empty and excluded count is zero (`folio/synthesize.py:105-124`). The invalid-scope bug above can still produce a misleading successful empty report.
- Shared-helper call: PASS. Production calls the module-level `derive_trust_status(view)` in the finding loop (`folio/synthesize.py:78`), so the SYN-19 monkeypatch on `folio.synthesize.derive_trust_status` observes the production path.
- `--limit` count ordering: PASS for positive limits. The collector runs before slicing (`folio/synthesize.py:63-67`), and `counts.flagged_input` is used afterward (`folio/synthesize.py:89`). Negative limits remain a should-fix.
- Scope resolution: BLOCK. Document-ID routing and invalid-scope failure are not implemented.
- JSON serialization: PASS for `scope is None -> null` through `render_envelope()` (`folio/synthesize.py:98`) and for upstream signal copying (`folio/synthesize.py:76`). Shallow mutability remains a should-fix note.
