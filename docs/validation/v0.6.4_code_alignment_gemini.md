### D-AL-002 — `folio links status` missing `--include-flagged` flag promised by §5.1 and §5.5

**Severity:** Alignment blocker (code does not implement the signature the
spec committed to).

**Spec commits (v1.1):**

- §5.1: "`--include-flagged` on all mutating and surfacing commands. Added to:
  `folio links review` (surface), **`folio links status` (surface)**, …"
- §4.5: `def relationship_status_summary(config, *, scope=None, **include_flagged: bool = False**) -> …`
- §5.5: empty-status fallback reads `"use folio links status --include-flagged
  to inspect"`.

**Code reality:**

- `links_status_cmd` (`folio/cli.py:1661–1699`) declares no `--include-flagged`
  option. Its signature is `def links_status_cmd(ctx, scope)`.
- `relationship_status_summary` (`folio/links.py:314–369`) has no
  `include_flagged` parameter.
- The empty-status fallback (`folio/cli.py:1675`) directs operators to
  `folio links review --include-flagged`, not `folio links status --include-flagged`
  as §5.5 specifies.

**Why this is a blocker, not a should-fix:** §5.1 is the normative list of
commands that must receive the flag, and §5.5 explicitly points operators at
`folio links status --include-flagged`. Shipping v0.6.4 without the flag
leaves `links status` unable to honor the `--include-flagged` surface
contract at all; an operator who follows the empty-status disclosure text
receives `Error: no such option: --include-flagged`.

This is distinguishable from a CLI-wording drift (which is should-fix) because
the *promised capability* is absent, not just mislabeled. The spec's B.3
canonical verdict elevated CB-2 to a blocker on the grounds that silent-empty
reporting violates §11 rule 5. The `folio links status` surface is one of
the three surfaces §5.1 enumerates, and without `--include-flagged` the
surface cannot fully honor rule 5 (operator cannot reach the flagged rows
through `links status` at all).

**Resolution options:**

1. Add `--include-flagged` to `links_status_cmd` and accept `include_flagged`
   on `relationship_status_summary`. When the flag is set, the renderer
   could replace the `Flagged Excluded` column with `Flagged Included` and
   surface per-source flagged rows individually (or simply document that
   status is always include-aware and the flag's presence is a no-op
   preserved for CLI symmetry).
2. If the decision is that `links status` already discloses flagged-excluded
   counts unconditionally and the flag is unneeded, revise spec §5.1 and §5.5
   to remove the commitment. But that is a spec change, not an implementation
   change — and would need to route through B.3 meta-consolidation, since it
   reopens CB-2's resolution plan.

Recommended: option (1). Mechanical addition; does not affect existing tests;
closes the §5.5 wording drift (D-AL-SF-001) at the same time.

### D-AL-SF-002 — `links review` disclosure message does not match §5.3 template verbatim

Spec §5.3 template:

```
{counts.flagged_input} proposal(s) excluded (flagged inputs). Use --include-flagged to review.
```

Code (`folio/cli.py:1653–1658`):

```python
if counts.flagged_input and not include_flagged:
    plural = "proposal" if counts.flagged_input == 1 else "proposals"
    click.echo(
        f"{counts.flagged_input} {plural} excluded (flagged inputs). "
        "Use --include-flagged to review."
    )
```

Semantic equivalence: the code renders "1 proposal excluded" / "3 proposals
excluded" where the spec template uses invariant `proposal(s)`. This is a
copy-edit enhancement, not a spec violation — "proposal(s)" is shorthand for
pluralization and the code implements real pluralization. Treat as
informational (D-AL-I-001), not a drift, unless strict template compliance is
required.

Flagging here because the spec template uses a literal format, and the
canonical spec verdict explicitly includes CLI-string compliance in Phase D.2
alignment scope. Recommend either (a) relax the spec template note to accept
real pluralization, or (b) switch the code back to the literal `proposal(s)`.
No test currently pins the specific wording, so either path is safe.

### D-AL-SF-003 — `links review` disclosure fires only when `not include_flagged`

Code (`folio/cli.py:1653`):

```python
if counts.flagged_input and not include_flagged:
    …
```

Spec §5.3 shows the disclosure gated only on `counts.flagged_input`, with no
`not include_flagged` conjunct:

```python
if counts.flagged_input:
    …
```

Since the collector zeroes `counts.flagged_input` when `include_flagged=True`
is passed in (`folio/links.py:276–278` skips the increment path entirely),
both gate conditions produce identical runtime behavior today: when
`include_flagged=True`, `counts.flagged_input == 0`, and neither gate prints
the disclosure. The code's extra `not include_flagged` guard is defensive
redundancy, not a behavior change.

Flagging as should-fix because the redundant guard creates a maintenance
footgun: if a future change causes `flagged_input` to be non-zero under
`include_flagged=True` (e.g., counting flagged-but-surfaced proposals for
metrics), the spec's single-conjunct gate would surface them and the code's
double-conjunct gate would silently swallow them. Recommend removing
`and not include_flagged` to match §5.3.

Low priority; no test currently exercises the difference.

### D-AL-I-002 — `confirm_doc` / `reject_doc` pass `include_flagged` into each
per-proposal `confirm_proposal` call

Code pattern in `folio/links.py:569–576`:

```python
for proposal_id in proposals:
    confirm_proposal(
        config,
        proposal_id,
        confirmation_source="folio links confirm-doc",
        include_flagged=include_flagged,
    )
```

Spec §4.6 says `confirm_doc` passes `include_flagged` through to
`collect_pending_relationship_proposals`. The code's additional propagation
into `confirm_proposal` (which calls `_find_pending_view` again) ensures that
the re-lookup at confirmation time uses the same trust posture as the initial
collection. This is a correctness enhancement — without it, the second
`_find_pending_view` call would default to `include_flagged=False` and the
just-collected flagged proposals would all fail with `Unknown proposal_id`
during bulk confirm of a flagged doc. Tests likely don't exercise this
because `confirm_proposal` is called with the same proposal just collected,
so the window between collect and find is narrow — but the defensive
parameter propagation is correct.

Not a spec drift. Call it out here for traceability: the implementation does
more than §4.6 strictly specifies, but in the direction of greater
consent-workflow integrity. The spec is slightly under-specified on this
plumbing detail and the code's behavior is defensible.
