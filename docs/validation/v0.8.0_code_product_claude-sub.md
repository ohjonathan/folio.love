# D.2 Product Code Review — `folio synthesize` v0.8.0

**Scope:** operator-facing UX end-to-end at commit 2d4b56c (post B.1). Spec v1.1
closed the help-copy framing (PROD-B1 / CB-4) and zero-findings breadcrumb
(PROD-B2 / CB-5). This review stress-tests the shipped implementation.

## Verdict rationale

No BLOCKERS. B.1 blockers are genuinely closed in code: the `--help` copy leads
with structural framing and foreshadows narrative (SYN-CLI-17 enforces the
exact strings); the zero-findings/zero-exclusions case prints the next-action
breadcrumb (SYN-CLI-14). The JSON envelope honors §5 minimum-shared-contract
parity (11 keys per finding, verified by SYN-CLI-18/19/20). Arrow rendering is
readable, `--limit` works, and `-` alias normalizes to `null` in JSON.

I file six should-fix items below. None are spec drift; all are polish gaps
that a first-time operator will trip over.

## Should-fix items

**PROD-D2-SF1 — `[flagged]` suffix erases source-vs-target distinction.**
`view.flagged_inputs` is already a list like `["source"]`, `["target"]`, or
both (folio/links.py:33, :292). The renderer collapses all of these to a bare
`[flagged]` label (synthesize.py:126). With `--include-flagged` the operator
sees "why was this surfaced?" but cannot tell whether the source deck or the
target deck triggered the flag without re-running `folio links review` or
opening the source YAML. Propose rendering `[flagged:source]` /
`[flagged:target]` / `[flagged:both]`. JSON envelope would benefit from
propagating `flagged_inputs: list[str]` on the finding too (trivial additive
key, bumps `schema_version` to "1.1" per §3.3 policy — can defer to v0.8.1 but
stdout fix is decoupleable and operator-visible today).

**PROD-D2-SF2 — `--limit N` gives operators no truncation signal.** For a
scope with 500 findings, `folio synthesize my-engagement --limit 10` prints
"Findings: 10" + "Excluded (flagged inputs): 0" and renders 10 rows. A
first-time operator reads "Findings: 10" as "my scope has 10 findings." Spec
§4 (line 602) acknowledges `findings_truncated: bool` is deferred to v0.8.1,
but the stdout gap is immediately confusing. Cheap fix: when `limit is not
None and len(report.findings) == limit`, print
`Showing first N (use --limit to adjust or remove)` under the Findings line.
No envelope bump required; stdout-only. Also worth distinguishing "Findings:"
(displayed) from "Total in scope:" when truncation is active.

**PROD-D2-SF3 — Invalid scope silently becomes empty scope.** Spec §8 row 1
commits to "Exit 1; stderr: Error: scope '<arg>' does not resolve …" — but
`collect_pending_relationship_proposals` does NOT raise on nonexistent scope
(it returns empty). Synthesize therefore exits 0 with zero findings, and the
CB-5 breadcrumb fires the same way it would for a truly-empty-but-valid scope.
test_synthesize_invalid_scope_exits_nonzero (SYN-CLI-7) even bakes this in,
commenting: "Current collect_pending_relationship_proposals does NOT raise on
bad scope (it returns empty). Synthesize therefore exits 0 …" This is spec
drift that was tested-around rather than fixed. For a first-time operator
typing `folio synthesize engaement-acme` (typo), they see zero findings and
the "run `folio ingest`" breadcrumb, which points them at the wrong remediation.
Either (a) implement scope validation in synthesize (light scope-resolver call
against the registry before invoking the collector), or (b) rewrite spec §8 to
match reality. Current state — spec promises one behavior, code does another,
test locks in the mismatch — is the genuinely problematic combination.

**PROD-D2-SF4 — Breadcrumb wording omits "check your scope spelling."** The
breadcrumb currently reads: "Next: run `folio ingest` or `folio enrich` if no
proposals have been emitted yet, or check that the scope resolves." The last
clause helps, but given PROD-D2-SF3 (invalid scope is ambiguous with empty
scope), the ordering is wrong: typos are more likely than producers-haven't-run
for a library that's been set up. Reorder to lead with scope-spelling:
"Next: verify the scope string resolves, or run `folio ingest` / `folio enrich`
if no proposals exist yet." Minor but materially changes first-touch diagnosis.

**PROD-D2-SF5 — CHANGELOG omits the `-` dash alias.** The v0.8.0 entry lists
`[SCOPE]` as the positional arg and shows `--json`, `--include-flagged`,
`--limit` — but doesn't document that `-` is accepted as a library-wide alias.
Operators scripting around synthesize (e.g., piping to `jq`) will want to know
they can write `folio synthesize - --json` explicitly for library-wide.
SYN-CLI-16 tests the behavior; CHANGELOG should surface it.

**PROD-D2-SF6 — Arrow direction is fine for relationship proposals; flag for
v0.8.1.** The `source --relation--> target` rendering reads naturally for
`draws_from` and `impacts` but is awkward for `supports` / `refines` where the
target is the primary subject and source is peripheral. Not a blocker for
v0.8.0 (single proposal_type). When entity-merge / diagram-archetype dispatch
lands in v0.8.1, add per-type renderers rather than one arrow string.

## What's genuinely good

- Help-copy framing (SYN-CLI-17) actually does land — I traced Click's
  auto-format and the key phrases survive wrapping.
- `schema_version: "1.0"` + `command: "synthesize"` at envelope top makes
  dashboard dispatch clean; `schema_gate_result: null` per finding is honestly
  distinguished from "gate passed" because a passed gate would be a dict, not
  null — downstream consumers can branch on `is None`.
- `folio synthesize --include-flagged -` works (positional scope,
  unambiguous).
- CHANGELOG v0.8.0 entry's "Known gaps" section is unusually operator-helpful:
  it tells adopters what NOT to rely on (`findings_truncated`, gate
  computation, graph envelope parity) without requiring them to open the spec.
