   114	## Summary
   115	
   116	`folio` does not natively handle `.eml` files. Email threads — a critical class of consulting engagement evidence — currently require a custom parsing step (extract headers, reconstruct quoted thread, strip Outlook noise, render markdown) before they can be fed into `folio ingest --type internal_sync`. A first-class email pipeline would close a significant gap in the format coverage story alongside #56.
   117	
   118	## Today's workaround
   119	
   120	On a live engagement we wrote a ~370-line `scripts/parse_emails.py` to:
   121	- Parse `.eml` via Python's `email` stdlib
   122	- Extract envelope headers (From, To, CC, Date, Subject)
   123	- Reconstruct the quoted thread by splitting on Outlook's `From: ...` / `Date: ...` blocks
   124	- Sort messages chronologically (oldest-first), with the outermost message dated from the EML envelope
   125	- Strip noise: `<tel:>` / `<mailto:>` / `<https://disclaimer-link>` wrappers, Zoom dial-in tables, SIP/H.323 lines, `____________` separators, repeated signatures, external-sender warnings, scotiabank/firm-wide disclaimers
   126	- Collapse Outlook's split bullets (`*` on one line, content on the next) into proper markdown bullets
   127	- Catalog attachments by filename + MIME type (no binary extraction)
   128	
   129	The resulting markdown then ingests cleanly via:
   130	
   131	```bash
   132	folio ingest <thread.md> \
   133	  --type internal_sync \
   134	  --date <YYYY-MM-DD> \
   135	  --client <Client> \
   136	  --engagement <Engagement> \
   137	  --participants \"<comma-separated-names>\"
   138	```
   139	
   140	For a 3-thread, 11-attachment Scotiabank corpus this produced 3 `review_status: clean` interaction notes with 28 total claims extracted via the Anthropic fallback (Sonnet 4 took over after the OpenAI route 403'd at the AI Gateway).
   141	
   142	## Suggested behavior
   143	
   144	### 1. New document type: `correspondence`
   145	
   146	Email threads aren't really meetings. Add a sibling to `interaction` and `evidence`:
   147	
   148	```yaml
   149	type: correspondence
   150	subtype: email_thread  # or email_single
   151	```
   152	
   153	This avoids overloading `internal_sync` and gives consumers (e.g., the Folio Library cursor rule) a cleaner lookup vocabulary.
   154	
   155	### 2. New CLI: `folio ingest-email <path.eml>`
   156	
   157	Or extend `folio ingest` with `--type email_thread` and accept `.eml` directly. Required behavior:
   158	
   159	- Parse all `text/plain` parts; fall back to `html` -> markdown via `markitdown`
   160	- Reconstruct the thread (oldest-first); preserve per-message From/To/CC/Date in frontmatter
   161	- Strip Outlook/Zoom/disclaimer noise (a curated rules list will get most of it)
   162	- Resolve participants from To/CC/From across the thread (deduplicated, normalized)
   163	- Catalog attachments in frontmatter:
   164	  ```yaml
   165	  attachments:
   166	    - filename: SOW_v9.docx
   167	      content_type: application/vnd.openxmlformats-officedocument.wordprocessingml.document
   168	      size_bytes: 2140160
   169	      sha256: <hash>
   170	  ```
   171	- Optionally extract attachment bodies into the registry (gated by `--extract-attachments`) so that PDFs and images on the thread become first-class documents and can be `impacts:`-linked back to the email
   172	
   173	### 3. Registry / frontmatter additions
   174	
   175	```yaml
   176	sender: Mishal Desai <Mishal_Desai@mckinsey.com>
   177	recipients_to: [Jake Itzkowitz, Tim Vogeler, ...]
   178	recipients_cc: [Pankaj Kumar, ...]
   179	thread_message_count: 11
   180	attachment_count: 3
   181	external_thread: false   # true if any party is non-McKinsey
   182	```
   183	
   184	### 4. Enrichment hints
   185	
   186	The `enrich` step would benefit from email-aware heuristics:
   187	- Quoted blocks (markdown `>`) should be lower-weight than primary author content
   188	- Signatures should be detected and excluded from claim extraction
   189	- Decisions that show up in the most recent message should be boosted
   190	
   191	## Why this matters
   192	
   193	For a typical engagement, the email thread is where decisions are actually made and where vendor positions are first surfaced. The Scotia corpus we just ingested includes a Microsoft solutions-engineer pitch on Azure vs GCP that doesn't appear in any deck or meeting note, but is essential context for the architecture decision. Without first-class email support, this evidence either gets dropped or gets shoehorned into `internal_sync` with awkward titles like \"Re: [External] AI Platform Vendor Decision Points\".

   208	
   209	- URL: https://github.com/ohjonathan/folio.love/issues/62
   210	
   211	- State at triage: OPEN
   212	
   213	- Created: 2026-04-17T05:03:55Z

   215	- Updated: 2026-05-12T19:23:19Z

   221	## Summary
   222	
   223	Add `folio watch <dir>` — a long-running daemon (or `--once` mode for cron / launchd) that monitors a directory for new files and routes each to the appropriate Folio pipeline based on file extension. Closes the "manual CLI invocation per file" gap that today makes Folio feel like a tool you reach for rather than a system that just runs.
   224	
   225	## User pain (current state)
   226	
   227	To save an email, transcript, deck, or PDF into Folio today, every artifact has the same 4-7 ceremony steps:
   228	
   229	1. Get the file into the right subdirectory (`library/<client>/<engagement>/<doctype>/`)
   230	2. Open terminal, `cd` to workspace, `source .env`
   231	3. Remember which Folio verb (`ingest` vs `convert`)
   232	4. Type the right flags (`--type`, `--client`, `--engagement`, `--date`, `--participants`, `--target`)
   233	5. Wait for processing
   234	6. Verify the registry entry was written
   235	
   236	Steps 2-6 are 100% mechanical given the file. The user always has the same intent ("this is engagement evidence, file it") but has to translate that into Folio's verb-and-flag vocabulary every single time.
   237	
   238	For a solo consultant on a live engagement, this friction is the dominant reason artifacts *don't* end up in Folio — the ceremony cost beats the marginal value of any single file.
   239	
   240	## Concrete example from a live engagement
   241	
   242	Today, ingesting one email thread on the Scotiabank AI Platform engagement looks like this (from our actual workflow):
   243	
   244	```bash
   245	folio ingest \"library/scotiabank/aiplatform/correspondence/2026-04-17_scotia_ai_platform_azure_gcp.md\" \\
   246	  --type internal_sync \\
   247	  --date 2026-04-16 \\
   248	  --client Scotiabank \\
   249	  --engagement \"AI Platform\" \\
   250	  --participants \"Chi Wai Cheung, Jonathan Oh, Bishal Dasgupta, Will Lau\" \\
   251	  --target library/scotiabank/aiplatform/correspondence/ \\
   252	  -n \"Initial ingest from email thread (Azure/GCP agentic stack discussion)\"
   253	```
   254	
   255	Seven flags. Six of them have the same value across every email on this engagement, or could be derived from the email's headers. Only `-n` is genuinely per-invocation.
   256	
   257	Multiply by a dozen artifacts a day and the reason Folio adoption stalls becomes obvious.
   258	
   259	## Proposed feature
   260	
   261	### CLI
   262	
   263	```bash
   264	folio watch <dir> [--config folio.yaml] [--once] [--dry-run] [--quiet]
   265	```
   266	
   267	| Flag | Behavior |
   268	|---|---|
   269	| (default) | Long-running daemon. Watches `<dir>` via `watchdog` (cross-platform: kqueue on macOS, inotify on Linux, ReadDirectoryChangesW on Windows) |
   270	| `--once` | Single pass: process anything currently in `<dir>` that hasn't been seen, then exit. Suitable for cron / launchd. |
   271	| `--dry-run` | Log what would be processed but don't call the underlying pipeline. |
   272	| `--quiet` | Suppress per-file stdout; only log errors. |
   273	
   274	### Routing rules
   275	
   276	By file extension, with overrides via `folio.yaml`:
   277	
   278	| Extension | Default pipeline | CLI equivalent |
   279	|---|---|---|
   280	| `.eml` | EML thread ingest (#61) | `folio ingest --type email_thread` |
   281	| `.pptx`, `.ppt` | Deck conversion | `folio convert` |
   282	| `.pdf` | Document conversion | `folio convert` |
   283	| `.docx` | Document conversion (#56) | `folio convert` |
   284	| `.md`, `.txt` | Interaction ingest | `folio ingest` |
   285	| Other | Skip with warning |
   286	
   287	### Configuration
   288	
   289	```yaml
   290	# folio.yaml
   291	watch:
   292	  - path: ~/Folio-Inbox
   293	    defaults:
   294	      client: Scotiabank
   295	      engagement: AI Platform
   296	    routing:
   297	      - match: \"*.eml\"
   298	        action: ingest
   299	        type: email_thread
   300	      - match: \"*.pptx\"
   301	        action: convert
   302	      - match: \"*.md\"
   303	        action: ingest
   304	        type: internal_sync   # or auto-derive (see related FR for defaults)
   305	    after_process:
   306	      action: archive          # or \"delete\", \"leave\"
   307	      to: ~/Folio-Inbox/_archive/
   308	    notify:
   309	      backend: macos_notification  # or \"stdout\", \"slack\", \"none\"
   310	```
   311	
   312	### Behavior
   313	
   314	1. New file appears in watched directory
   315	2. Folio matches extension against routing rules
   316	3. Folio waits until file size is stable for N seconds (configurable; default 3) — guards against partial drops while drag is in progress
   317	4. Folio runs the appropriate pipeline (calls the existing `folio convert` or `folio ingest` internally), with engagement defaults applied
   318	5. On success: archive (or delete, or leave) the original; post a notification
   319	6. On failure: write `<filename>.error.log` to `<dir>/_failed/`; post a notification with the error
   320	
   321	### Edge cases
   322	
   323	- **Partial files / drag in progress**: wait for file size to be stable for N seconds before processing. Configurable via `watch.stability_seconds`.
   324	- **Duplicates** (same file dropped twice): hash content; if already in registry, log \"already ingested as <id>\" and skip (or archive directly without re-processing).
   325	- **Concurrent files dropped together**: process serially to avoid LLM rate limits and to make registry writes atomic.
   326	- **Daemon hygiene**: log rotation under `~/.folio/logs/`, restart on `folio.yaml` change (config watcher), `folio watch --healthcheck` command for monitoring.
   327	- **Missing target subdirectory**: if `library/scotiabank/aiplatform/correspondence/` doesn't exist, create it (parent dir derived from `defaults.client` + `defaults.engagement` + routed `type`).
   328	
   329	### Composition with other features
   330	
   331	This feature depends on or composes with:
   332	
   333	- **#56** — `.docx` normalization gate. Without this, the watcher can't route `.docx` properly.
   334	- **#61** — Native `.eml` support. Without this, the watcher needs an intermediate parser for emails (which we've written as a workaround in this engagement — temporary scaffolding to be deleted when #61 ships).
   335	- **[FR-defaults]** — Engagement defaults in `folio.yaml`. Without this, the watcher would still need to know `--client`, `--engagement`, `--type` etc. for each file. This is the feature that makes `folio watch` truly zero-ceremony. To be filed as a sibling issue.
   336	- **[FR-thread-dedup]** — Message-ID-based thread dedup. Without this, dragging a thread continuation creates a duplicate registry entry. With it, thread continuations bump the existing entry's version. To be filed as a sibling issue.
   337	
   338	### Why not just a shell wrapper around `fswatch` / `inotifywait`?
   339	
   340	We considered this. We declined because:
   341	
   342	1. **Routing rules need to evolve with Folio's own format support** (today PPTX/PDF; tomorrow DOCX/EML/CSV/etc.). A shell wrapper has to be re-edited every time Folio adds a format.
   343	2. **Defaults need to compose with Folio's existing config** (`folio.yaml`). A shell wrapper invents its own config or hardcodes values.
   344	3. **Dedup needs registry-aware logic**, which a shell wrapper would have to reimplement against `library/registry.json`.
   345	4. **Notification format / failure handling should match Folio's existing UX**, not diverge across N user-built wrappers.
   346	5. **The Folio user community would build N divergent shell scripts**; one Folio feature replaces them all.
   347	
   348	The same argument holds for asking users to write a Hazel rule, Dropzone action, or macOS Folder Action. They all work, but they all reinvent Folio-aware logic outside of Folio.
   349	
   350	### Why this matters
   351	
   352	Folio today is a CLI-of-verbs (`convert`, `ingest`, `enrich`). For solo consultants doing live engagements, the cognitive overhead of \"which verb, which flags, where does it go\" is the dominant friction — and it's the reason artifacts don't get filed.
   353	
   354	A drop-zone collapses that friction to \"save this thing\" — same gesture for any artifact, no taxonomy decisions at the moment of saving. The taxonomy work happens in `folio.yaml` once and is reused forever.
   355	
   356	This pattern is well-established in adjacent tools (Hazel, Dropzone, DropBox folder upload, OneDrive auto-sort). What's missing is something that knows the Folio pipeline.

   426

   468	      - rule: type_is
   469	        type: email_thread
   470	        target: \"{target_root}/{client_slug}/{engagement_slug}/correspondence/\"
   471	      - rule: type_is
   472	        type: client_meeting
   473	        target: \"{target_root}/{client_slug}/{engagement_slug}/interactions/\"
   474	```
   475	
   476	### CLI behavior
   477	
   478	When a flag is omitted, Folio resolves in this precedence order:
   479	
   480	1. **Explicit CLI flag** (highest precedence)
   481	2. **Derivation rule** matching the source file
   482	3. **`defaults:` block** in `folio.yaml`
   483	4. **Interactive prompt** (only if `--strict` is set; default behavior is to use defaults silently)
   484	5. **Error** (if still unresolved and `--strict` is set, OR if the field is required and unresolvable)
   485	
   486	Folio always logs the derived value so the user can see what was used:
   487	
   488	```bash
   489	$ folio ingest thread.eml
   490	[derive] client: Scotiabank (defaults.client)
   491	[derive] engagement: AI Platform (defaults.engagement)
   492	[derive] type: internal_sync (rule: all_participants_match_domain @mckinsey.com)
   493	[derive] date: 2026-04-16 (eml.date)
   494	[derive] participants: [Chi Wai Cheung, Jonathan Oh, Bishal Dasgupta, Will Lau] (eml.headers)
   495	[derive] target: ./library/scotiabank/aiplatform/correspondence/ (rule: type_is email_thread)
   496	✓ Ingested as scotiabank_aiplatform_sync_20260416_re_scotia_azure_gcp
   497	```
   498	
   499	### CLI overrides still work
   500	
   501	```bash
   502	# Override just the type — everything else is derived
   503	folio ingest thread.eml --type workshop
   504	
   505	# Strict mode: prompt if anything's unresolvable
   506	folio ingest thread.eml --strict
   507	```
   508	
   509	### Why this matters
   510	
   511	Without this, **`folio watch` (related FR) is impossible** without per-file metadata sidecar files or some other awkward UX. With this, the watcher just calls `folio ingest <file>` and the right thing happens.
   512	
   513	This also reduces error: today, a typo like `--client \"Scotabank\"` creates a phantom client in the registry that's hard to clean up afterward. With config-defined defaults, the value is right once and right forever.
   514	
   515	### Composition
   516	
   517	- **#62 (folio watch)** depends on this. The watcher cannot apply per-file flags via shell — it needs config-driven defaults.
   518	- **#61 (EML support)** is the highest-value beneficiary of derivation: date, participants, and type all come \"for free\" from RFC 5322 headers.
   519	- **#56 (DOCX support)** benefits from this. DOCX ingest could derive type and date from the document itself.
   520	
   521	### Edge cases
   522	
   523	- **Multiple defaults per workspace** (e.g., user works on multiple engagements simultaneously): support `defaults` per directory tree. `folio.yaml` in a subdirectory overrides the parent. Or: `--profile <name>` flag to select among named default profiles.
   524	- **Missing required field** (e.g., file has no extractable date): error with a clear message naming the field, the source file, and the rules that were tried.
   525	- **Conflicting derivations** (e.g., filename says 2026-04-16 but EML header says 2026-04-15): use first-match-wins per the order in `derive:`. Log both.
   526	- **Backward compatibility**: when `defaults` is absent, behavior is unchanged. Existing scripts continue to work.
   527	
   528	### Why not \"just write a shell alias\"?
   529	
   530	A user could `alias fingest='folio ingest --client Scotiabank --engagement \"AI Platform\"'`. We considered this and declined because:
   531	
   532	1. Aliases don't compose with `folio watch` — daemons don't read shell aliases.
   533	2. Aliases can't do file-content derivation (date, participants, type from headers).
   534	3. Aliases are per-user; Folio config is per-workspace and survives onboarding new team members.
   535	4. Aliases proliferate (`fingest`, `fingest_internal`, `fingest_client`, ...) where one config block would do.

   551	### Issue #64: Email thread continuation / Message-ID-based dedup for EML ingest
   552	
   553	- URL: https://github.com/ohjonathan/folio.love/issues/64
   554	
   555	- State at triage: OPEN
   556	
   557	- Created: 2026-04-17T05:05:33Z
   558	
   559	- Updated: 2026-05-12T19:23:23Z
   560	
   561	- Labels: none
   562	
   563	#### Body
   564	
   565	## Summary
   566	
   567	When ingesting an `.eml` (post-#61) or an EML-derived markdown file, use the email's RFC 5322 `Message-ID`, `In-Reply-To`, and `References` headers to detect when the new file is a **continuation** of a previously-ingested thread. Treat continuations as new versions of the same registry entry rather than as independent entries.
   568	
   569	## User pain (anticipated; we'll hit this within a week of using #61)
   570	
   571	A typical engagement email thread grows over time:
   572	
   573	| Day | What happened | Action |
   574	|---|---|---|
   575	| Mon | \"Re: Vendor Decision\" — 3 messages so far | Drag .eml to Folio → ingested as `entry_v1` |
   576	| Tue | Same thread, now 5 messages (2 new replies) | Drag updated .eml to Folio |
   577	| Wed | Same thread, now 8 messages | Drag updated .eml to Folio again |
   578	
   579	Today, Folio (and our parser as a temporary workaround) hashes the full body for each ingest. Different bodies = different `source_hash` = different registry entry IDs.
   580	
   581	Result:
   582	
   583	```
   584	scotiabank_aiplatform_sync_20260415_re_vendor_decision (3 msgs, v1)
   585	scotiabank_aiplatform_sync_20260416_re_vendor_decision (5 msgs, v1)
   586	scotiabank_aiplatform_sync_20260417_re_vendor_decision (8 msgs, v1)
   587	```
   588	
   589	All three look like distinct interactions. None are marked stale. Claims are extracted three times — once for each version — and surfaced redundantly in any consumer that filters by `review_status: clean`. Search returns three hits for what the user mentally models as one conversation.
   590	
   591	This violates the Folio promise of \"one document, versioned correctly.\"
   592	
   593	## Why this is solvable
   594	
   595	Email already carries the relationship metadata. Every reply chain has:
   596	
   597	- **`Message-ID:`** — unique identifier for this specific message
   598	- **`In-Reply-To:`** — Message-ID of the parent
   599	- **`References:`** — full ancestry of Message-IDs in the thread
   600	
   601	So a thread is **definitively identified** by the set of Message-IDs in its body. If two ingested files share any Message-ID, they're the same thread. RFC 5322 has guaranteed this since 1982.
   602	
   603	## Proposed feature
   604	
   605	### Detection
   606	
   607	For email-sourced documents, Folio extracts the **set of Message-IDs** in the thread:
   608	
   609	- The outermost `Message-ID:` header (most recent message in this ingest)
   610	- All Message-IDs in `References:` (the full ancestry)
   611	- Optionally: parse quoted-message blocks for embedded `Message-ID:` lines (Outlook sometimes preserves them)
   612	
   613	Store this set in the registry entry:
   614	
   615	```yaml
   616	id: scotiabank_aiplatform_sync_20260415_re_vendor_decision
   617	message_ids:
   618	  - \"<CAJ8z...mck@mail.gmail.com>\"
   619	  - \"<MX2P234...scotia@scotiabank.com>\"
   620	  - \"<DM6PR07...mck@mckinsey.com>\"
   621	```
   622	
   623	### Behavior on ingest
   624	
   625	When ingesting a new EML:
   626	
   627	1. Extract the new file's Message-ID set
   628	2. Query the registry for any existing entry whose `message_ids` set intersects
   629	3. **If intersection is non-empty:**
   630	   - This is a thread continuation
   631	   - Compute the union of old + new Message-IDs
   632	   - Bump the existing entry's `version` from N → N+1
   633	   - Mark prior version as `staleness_status: stale`
   634	   - Update `message_ids` to the union
   635	   - Re-run enrichment on the new (larger) body
   636	4. **If intersection is empty:**
   637	   - This is a new thread; create a new entry as today
   638	
   639	### CLI output
   640	
   641	```bash
   642	$ folio ingest 2026-04-17_re_vendor_decision.eml
   643	[detect] Message-ID set: {<msg-1@>, <msg-2@>, <msg-3@>, <msg-4@>, <msg-5@>}
   644	[detect] Continuation of: scotiabank_aiplatform_sync_20260415_re_vendor_decision (v1, 3 msgs)
   645	[action] Bumping to v2 (5 msgs); marking v1 as stale
   646	✓ Updated scotiabank_aiplatform_sync_20260415_re_vendor_decision (v2)
   647	```
   648	
   649	### Behavior on subsequent ingests of the same content
   650	
   651	If the same file is dropped again with no new messages (identical Message-ID set + identical body hash):
   652	
   653	```bash
   654	$ folio ingest same_thread.eml
   655	[detect] Message-ID set: {<msg-1@>, <msg-2@>, <msg-3@>}
   656	[detect] Identical to: scotiabank_aiplatform_sync_20260415_re_vendor_decision (v1)
   657	[action] No-op (no new content)
   658	✓ Skipped (already current)
   659	```
   660	
   661	### Edge cases
   662	
   663	- **Branch in thread**: someone replies-all and creates a fork (different recipient list, but shares early Message-IDs). Folio detects divergence (the new branch has Message-IDs the existing thread doesn't, AND the existing thread has Message-IDs the new branch doesn't). Configurable behavior:
   664	  - `branch_strategy: new_entry` (default) — create a separate registry entry, link via `branched_from:`
   665	  - `branch_strategy: merge` — merge into existing entry as a v2 with both branches in the body
   666	- **Forward-as-new-thread**: someone forwards the thread to a new audience, generating a new outer Message-ID but preserving old ones in References. Configurable:
   667	  - `forwarded_as_continuation: true` — continuation of the original thread
   668	  - `forwarded_as_continuation: false` (default) — new entry, link via `forwarded_from:`
   669	- **Lost Message-IDs**: Outlook sometimes strips or mangles Message-IDs in deeply quoted thread blocks. Fall back to body-hash similarity (e.g., longest common substring of body content) as a secondary signal. Threshold configurable.
   670	- **Manual override**: `folio ingest --as-new-entry` to force creation of a new entry even when continuation would be detected.
   671	
   672	### Composition
   673	
   674	- **#61** — depends on this. EML ingest without dedup creates the duplication problem; EML ingest with dedup is the polished feature. Worth landing #61 first without this, then layering dedup on; but it should be in the same release.
   675	- **#62 (folio watch)** — critical for usability. Without dedup, a watched folder polluted with re-dragged thread updates becomes useless after a week.
   676	- **#63 (defaults)** — pairs naturally; once defaults remove flag friction, dedup removes registry friction.
   677	- **Existing version semantics** — this generalizes the existing `version: N` field to be aware of inherent document identity (Message-ID for emails) rather than just `source_hash`. Other document types (PPTX with version-suffixed filenames, e.g., `proposal_v6.pptx` → `proposal_v7.pptx`) could similarly benefit from filename-based identity rules in a future generalization.
   678	
   679	### Why this matters
   680	
   681	Email threads are **inherently cumulative**: every reply is a superset of the previous body. Treating each version as a fresh document violates the user's mental model and pollutes the registry. Other Folio document types (PPTX, PDF) have explicit version tracking (via filename or `--version`). EML needs the same, but derived automatically from headers that already exist in every email since 1982.
   682	
   683	Without this, `folio watch` becomes a registry-pollution machine.
   684

   686	
   687	- folio-love 0.6.4
   688	- macOS 25.4.0
   689	- Python 3.12

   692	
   693	- #61 — Native EML support (this feature is meaningless without it)
   694	- #62 — `folio watch <dir>` (this feature is critical to make watching usable)
   695	- #63 — Engagement defaults

   698	### Issue #69: Support transcript-native ingest formats like .vtt and .srt
   699	
   700	- URL: https://github.com/ohjonathan/folio.love/issues/69
   701	
   702	- State at triage: OPEN
   703	
   704	- Created: 2026-04-22T23:14:08Z
   705	
   706	- Updated: 2026-05-12T19:23:25Z
   707	
   708	- Labels: none
   709	
   710	#### Body
   711	
   712	## Problem
   713	
   714	`folio ingest` currently only accepts `.txt` and `.md` sources.
   715	
   716	That is too restrictive for the actual transcript artifacts people get from meeting tools. In practice, raw exports often come as:
   717	
   718	- `.vtt` / `WEBVTT`
   719	- `.srt`
   720	- plain UTF-8 transcript files that may not have a `.txt` extension
   721	
   722	In a real workflow today, I had a meeting transcript that Folio should absolutely be able to ingest directly, but instead I had to:
   723	
   724	1. move the raw transcript out of the workspace root
   725	2. rename/re-home it
   726	3. create a separate intermediate `.md` note
   727	4. run `folio ingest` on that surrogate file instead of the actual source transcript
   728	
   729	That is a lot of avoidable friction for a transcript-ingestion command.
   730	
   731	## Current behavior
   732	
   733	`folio ingest` rejects the raw transcript source based on extension.
   734	
   735	Example failure:
   736	
   737	```text
   738	Unsupported source extension ''. v0.5.0 supports .txt and .md only.
   739	```
   740	
   741	And `.vtt` is likewise not supported even though it is a very common transcript format.
   742	
   743	## Requested behavior
   744	
   745	`folio ingest` should support transcript-native input formats directly, at minimum:
   746	
   747	- `.vtt`
   748	- `.srt`
   749	- `.txt`
   750	- `.md`
   751	
   752	Nice-to-have:
   753	
   754	- accept extensionless plaintext files when the content is clearly transcript-like / UTF-8 text
   755	- or provide an explicit override flag such as `--format vtt|srt|plain|markdown`
   756	
   757	## Why this matters
   758	
   759	These formats are not edge cases. They are standard outputs from common meeting / transcription systems. If Folio is going to be the interaction-ingestion layer, it should ingest the raw artifacts users actually have, not force a manual conversion to markdown first.
   760	
   761	## Suggested implementation
   762	
   763	For `.vtt` / `.srt` inputs:
   764	
   765	- parse the transcript structure instead of rejecting by suffix
   766	- strip / normalize transport markup (cue numbers, timestamps, `WEBVTT` header, etc.)
   767	- preserve speaker attribution and timestamps where possible
   768	- feed the normalized transcript into the existing interaction-analysis path
   769	- keep the original source path in provenance/frontmatter
   770	
   771	For extensionless plaintext:
   772	
   773	- either content-sniff UTF-8 text and allow ingest
   774	- or require an explicit `--format plain` / `--format transcript` override
   775	
   776	## Acceptance criteria
   777	
   778	- `folio ingest meeting.vtt --type internal_sync ...` succeeds
   779	- `folio ingest meeting.srt --type internal_sync ...` succeeds
   780	- extracted interaction notes still preserve grounded findings, quotes, participants, and raw transcript sections
   781	- provenance/frontmatter still points back to the original raw source file
   782	- docs/help text for `folio ingest` mention the supported transcript formats explicitly
   783	
   784	## Extra note
   785	
   786	The current docs describe interaction ingestion as taking a transcript/note file, which is exactly why the current extension restriction is surprising. Supporting raw transcript formats would make the CLI match the real user workflow much better.

   790	
   791	- URL: https://github.com/ohjonathan/folio.love/issues/70
   792

   800	
   801	#### Body
   802	
   803	## Problem
   804	
   805	`folio ingest` extracts interaction findings into four buckets — `claims`, `data_points`, `decisions`, `open_questions`. The subtype prompt hints in [`folio/pipeline/interaction_analysis.py`](https://github.com/ohjonathan/folio.love/blob/main/folio/pipeline/interaction_analysis.py) explicitly tell the LLM to surface **next steps**, **actions**, **owners**, **commitments**, and **asks**:
   806	
   807	```python
   808	_SUBTYPE_PROMPT_HINTS = {
   809	    "client_meeting":    "Emphasize decisions, asks, owners, and next steps.",
   810	    "expert_interview":  "Emphasize interview insights, expert observations, and cited datapoints.",
   811	    "internal_sync":     "Emphasize updates, blockers, dependencies, and internal decisions.",
   812	    "partner_check_in":  "Emphasize partner asks, commitments, risks, and coordination topics.",
   813	    "workshop":          "Emphasize workshop outputs, open questions, decisions, and actions.",
   814	}
   815	```
   816	
   817	But the JSON schema in `_ANALYSIS_SYSTEM_PROMPT_TEMPLATE` has no slot for them:
   818	
   819	```json
   820	"findings": {
   821	  "claims": [...],
   822	  "data_points": [],
   823	  "decisions": [],
   824	  "open_questions": []
   825	}
   826	```
   827	
   828	Result: the LLM either drops next-step content entirely or squashes it into `decisions` with the wrong framing and timestamp.
   829

   831	
   832	I just ingested an `expert_interview` transcript that closed with three distinct forward-looking commitments:
   833	
   834	| Time | Speaker | Commitment |
   835	|---|---|---|
   836	| 00:23:30 | Jed | "Let me digest the conversation and come back to you with some next steps." |
   837	| 00:23:35 | Jed | "Get some more information and hopefully be in a position to come back to you with some thinking on an offer soon, like in the next couple of days, at least maybe the next couple of business days." |
   838	| 00:24:05 | Jonathan | "Let me know if you have any other questions. I'll do vice versa if I have any more questions." |

   841	
   842	```yaml
   843	- Jed will follow up with offer details within next couple of business days
   844	  - quote: "hopefully be in a position to come back to you with some thinking on an offer soon..."
   845	  - timestamp: 00:23:56     # wrong — actual sentence starts at 00:23:35
   846	```
   847	
   848	Problems:
   849	
   850	1. Only **1 of 3** commitments was captured.
   851	2. The captured one's timestamp is wrong (00:23:56 is "and we go from there", not where the commitment starts).
   852	3. No `owner`, no `due`, no signal that this is an action vs. a decision.
   853	4. The first Jed commitment ("digest and come back with next steps") was missed entirely — there was no schema bucket to hold it.
   854	
   855	I had to hand-add a `### Next Steps` subsection to the L0 markdown post-hoc, with `owner` and `due` fields invented locally, to make the doc actionable.
   856	
   857	## Current behavior
   858	
   859	- The four-bucket schema forces every action-shaped finding into `decisions` (or drops it).
   860	- No `owner` / `due` / `status` metadata on findings, even though `_SUBTYPE_PROMPT_HINTS` references "owners".

   862	- Markdown renderer emits `### Claims / ### Data Points / ### Decisions / ### Open Questions` under `## Key Findings` — no `### Next Steps` or `### Action Items` section.
   863	
   864	## Requested behavior
   865	
   866	Add a fifth finding bucket to the schema and surface it in the rendered markdown.
   867	
   868	### Schema additions
   869	
   870	```python
   871	# folio/pipeline/interaction_analysis.py — _ANALYSIS_SYSTEM_PROMPT_TEMPLATE
   872	"findings": {

   874	  "data_points":    [],
   875	  "decisions":      [],
   876	  "open_questions": [],
   877	  "action_items":   [                                     # NEW
   878	    {
   879	      "statement":    "<who will do what>",
   880	      "quote":        "<supporting quote>",
   881	      "element_type": "action",                           # NEW enum value
   882	      "confidence":   "<high|medium|low>",
   883	      "speaker":      "<speaker if known>",
   884	      "timestamp":    "<timestamp if available>",
   885	      "attribution":  "<attribution if available>",
   886	      "owner":        "<owner if identifiable>",          # NEW field
   887	      "due":          "<ISO date or natural-language window>"  # NEW field
   888	    }
   889	  ]
   890	}
   891	```
   892

   894	
   895	```python
   896	# folio/pipeline/interaction_analysis.py
   897	@dataclass
   898	class InteractionFinding:
   899	    statement: str
   900	    quote: str

   924	| Subtype | Header |
   925	|---|---|
   926	| `client_meeting`, `expert_interview`, `partner_check_in` | `### Next Steps` |

   940	Files that need to change (probably all in `folio/pipeline/interaction_analysis.py` plus rendering downstream):
   941

   943	2. **`_REDUCE_SYSTEM_PROMPT`** — already says "same schema as before"; verify reducer dedupes action_items by `(owner, statement)` not just `(quote)`.
   944	3. **`InteractionFinding`** — add `owner` / `due` optional fields.
   945	4. **`InteractionAnalysisResult`** — add `action_items` list; update `all_findings()` to include it.
   946	5. **Provenance / grounding** (`folio/pipeline/provenance_analysis.py`) — extend the validation walk to cover `action_items`.
   947	6. **Markdown rendering** (likely in `folio/analysis_docs.py` or `folio/output/`) — emit `### Next Steps` or `### Action Items` subsection under `## Key Findings`, with `owner:` and `due:` shown when present.
   948	7. **Registry `grounding_summary`** — count action_items in totals (or expose them as a separate count if maintainers prefer).

   977	### Issue #71: Interaction analysis: speaker analytics (talk time, word share, turn balance)
   978	
   979	- URL: https://github.com/ohjonathan/folio.love/issues/71
   980	
   981	- State at triage: OPEN
   982	
   983	- Created: 2026-04-24T17:34:54Z
   984	
   985	- Updated: 2026-05-12T19:23:28Z
   986	
   987	- Labels: enhancement
   988	
   989	#### Body
   990	
   991	## Problem
   992	
   993	Every transcript Folio ingests already contains the raw signal needed to answer *who talked, how much, and for how long* — speaker labels and second-resolution timestamps on every turn. Today we throw that signal away during ingest. The analysis pipeline only feeds the text into the LLM and never computes the deterministic per-speaker stats that fall out of the structure for free.
   994	
   995	This is a missed opportunity because:
   996	
   997	1. **It's almost free to compute** — a regex over the timestamp/speaker prefix lines, no LLM call needed.
   998	2. **It enables a class of self-coaching questions Folio is uniquely positioned to answer**, e.g. *"in my last 5 client meetings, what % of words did I speak?"* — useful for consultants, salespeople, interviewers, facilitators.
   999	3. **It surfaces conversation imbalance instantly** — for an `expert_interview`, the SME should be doing >70% of the talking; if I'm doing 60%, that's a quality flag I'd want to see in the registry.
  1000	
  1001	## Concrete evidence
  1002	
  1003	Today I ingested an `expert_interview` (Altitude workspace, `library/altitude/spinoff/interactions/2026-04-24_20260424_jed_cairo_*`) — a ~24-minute Jed Cairo / Jonathan Oh conversation. Every turn in the source transcript looks like:
  1004	
  1005	```
  1006	00:00:04 Jed Cairo
  1007	Yeah, you know, today is beautiful.
  1008	00:00:09 Jed Cairo
  1009	I like warm but not hot, and I feel like we only have a couple of weeks every year of that...
  1010	00:00:19 Jonathan Oh
  1011	That's true, but that four weeks is the nicest place on earth, right?
  1012	```
  1013	
  1014	Speaker labels and timestamps on every turn. After ingest, the rendered note has:
  1015	
  1016	- `## Summary` (LLM paragraph)
  1017	- `## Key Findings` (claims / data_points / decisions / open_questions)
  1018	- `## Entities Mentioned`
  1019	- `## Quotes / Evidence`
  1020	- raw transcript callout
  1021	
  1022	Nowhere does it say:
  1023	
  1024	- *"Jed spoke 38% of the words / Jonathan spoke 62%"*
  1025	- *"Jed had 47 turns averaging 22 words; Jonathan had 51 turns averaging 38 words"*
  1026	- *"Jed's longest contiguous monologue: 142 words at 00:18:30"*
  1027	- *"Total duration: 24m 14s; Jed talk time: ~9m; Jonathan talk time: ~15m"*
  1028	
  1029	For an `expert_interview` where the goal was to absorb Jed's perspective on AGTI, that 62/38 split would be a useful red flag I could act on next time.
  1030	
  1031	## Current behavior
  1032	
  1033	- `folio/pipeline/interaction_analysis.py` parses the transcript only as text input to the LLM.
  1034	- No deterministic speaker-stats pass exists.
  1035	- `InteractionAnalysisResult` has no speaker-stats fields.
  1036	- The rendered markdown has no speaker-analytics section.
  1037	- `registry.json` entries carry no speaker-balance data, so corpus queries like *"interactions where I spoke <30%"* are impossible without re-parsing every raw transcript.
  1038	
  1039	## Requested behavior
  1040	
  1041	Add a deterministic speaker-stats pass at ingest time. Render in the markdown, lift summary metrics into frontmatter, and surface a compact subset in `registry.json` for corpus queries.
  1042	
  1043	### Per-speaker metrics (computed deterministically)
  1044	
  1045	| Metric | Notes |
  1046	|---|---|
  1047	| `word_count` | `len(text.split())` summed across the speaker's turns |
  1048	| `word_share` | `word_count / total_words`, as a float 0-1 |
  1049	| `turn_count` | number of contiguous speaking blocks |
  1050	| `avg_words_per_turn` | mean turn length |
  1051	| `longest_turn_words` | longest single contiguous monologue (words) |
  1052	| `longest_turn_timestamp` | when the longest turn started |
  1053	| `talk_time_seconds` | sum of turn durations (next_turn.start − this_turn.start; last turn estimated from words / WPM) |
  1054	| `talk_share` | `talk_time_seconds / total_duration` |
  1055	| `first_turn_timestamp` / `last_turn_timestamp` | engagement window |
  1056	
  1057	### Aggregate metrics
  1058	
  1059	| Metric | Notes |
  1060	|---|---|
  1061	| `total_words` | all words across all speakers |
  1062	| `total_duration_seconds` | last_turn.start − first_turn.start (+ trailing turn estimate) |
  1063	| `speaker_count` | number of distinct speakers detected |
  1064	| `balance_score` | normalized 0-1 metric of how balanced the conversation was; e.g. `1 - gini(word_shares)` so 1.0 = perfectly even, 0.0 = monologue |
  1065	| `dominant_speaker` | speaker with highest `word_share` |
  1066	
  1067	### Frontmatter (proposed)

  1132	## Why this matters
  1133	
  1134	- **Self-coaching loop for the user.** *"In my last 5 client meetings, my average word_share was 47% — too high; aim for <35% next month."* Folio becomes a feedback mechanism, not just a record.
  1135	- **Quality flag on expert interviews.** If `dominant_speaker` is the interviewer (not the SME), the doc is suspect for that purpose. Could even auto-emit a `review_flag: interview_balance_inverted` when an `expert_interview` has interviewer word_share > 50%.
  1136	- **Engagement health signal.** Across a portfolio of meetings with one stakeholder, a sudden drop in their `talk_share` over time is a leading indicator of disengagement.
  1137	- **Workshop facilitation diagnostics.** For workshops, `balance_score` directly answers *"did everyone participate?"* without re-listening.
  1138	- **Free of charge.** Pure regex + arithmetic. No incremental LLM tokens.
  1139	
  1140	## Suggested implementation
  1141	
  1142	New deterministic pass in the ingest pipeline, runs **before** the LLM analysis call so its output is available regardless of LLM status (analysis_unavailable runs still get speaker stats):
  1143	
  1144	1. **New module `folio/pipeline/speaker_analytics.py`** — pure-function parser + analyzer.
  1145	   - `parse_turns(source_text: str) -> list[Turn]` — regex over the standard `HH:MM:SS Speaker Name` line pattern; supports the existing transcript convention (one timestamp + speaker per turn block).
  1146	   - `compute_stats(turns: list[Turn]) -> SpeakerStats` — emits the per-speaker + aggregate metrics above.
  1147	   - `last-turn duration estimate`: words / 150 WPM (configurable).
  1148	
  1149	2. **Hook into `folio/pipeline/interaction_analysis.py`** — call `speaker_analytics.compute_stats()` early in the ingest path; attach result to `InteractionAnalysisResult.speaker_stats`.
  1150	
  1151	3. **Dataclass additions:**
  1152	   ```python
  1153	   @dataclass
  1154	   class TurnStats:
  1155	       speaker: str
  1156	       word_count: int
  1157	       word_share: float
  1158	       turn_count: int
  1159	       avg_words_per_turn: float
  1160	       longest_turn_words: int
  1161	       longest_turn_timestamp: str
  1162	       talk_time_seconds: int
  1163	       talk_share: float
  1164	       first_turn_timestamp: str
  1165	       last_turn_timestamp: str
  1166	
  1167	   @dataclass
  1168	   class SpeakerStats:
  1169	       per_speaker: dict[str, TurnStats]
  1170	       total_words: int
  1171	       total_duration_seconds: int
  1172	       speaker_count: int
  1173	       balance_score: float
  1174	       dominant_speaker: str
  1175	   ```
  1176	
  1177	4. **Render** in the existing markdown writer (probably `folio/analysis_docs.py` or `folio/output/`) — emit `## Speaker Analytics` section + `speakers:` frontmatter block.
  1178	
  1179	5. **Registry surface** — extend interaction entries with the compact `speaker_summary` block in `folio/tracking/registry.py`.
  1180	
  1181	6. **Speaker-name normalization** — defer to existing entity resolution where possible; canonical name takes precedence over transcript label, but `aliases` are accepted as same-speaker (so "Jonathan" and "Jonathan Oh" don't show up as two speakers).
  1182	
  1183	7. **Graceful degradation** — if the transcript has no parseable timestamps/speakers (e.g. a free-form interview note), skip the pass silently and emit `review_flag: speaker_analytics_unavailable`.
  1184

  1189	- `registry.json` entries without `speaker_summary` remain valid — consumers should treat the field as optional.
