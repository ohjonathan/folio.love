# Alignment Review — folio-graph-generalized-proposals-v0-7-1 / D.2 / codex-as-alignment

## 1. Architecture compliance

The Phase C implementation mostly stays inside the approved retrofit boundary: `folio/graph.py` owns the pending-proposal finding shape, `folio/cli.py` owns the `graph doctor` rendering path, tests land in the two graph test files, and forbidden implementation paths are untouched. Static inspection and scope commands found no edits to the parent Tier 4 spec, `folio/links.py`, `folio/pipeline/`, `folio/tracking/entities.py`, `frameworks/llm-dev-v1/`, `AGENTS.md`, or `Ontos_Context_Map.md` in `main..HEAD`.

The main architecture mismatch is not the emitted code path. It is the manifest gate anchors: the runtime cardinality tests that v1.2/B.3 made load-bearing exist in `tests/test_graph_cli.py`, but the manifest still points `G-cardinality-1/2` at `tests/test_graph.py`. Those exact manifest commands fail and would break later approval gates even though the behavior-level contract test passes at the actual test location.

## 2. Diagram-architecture cross-reference

No diagrams are present in the Phase C implementation artifact, and none are required by the v0.7.1 spec or B.3 canonical verdict. No diagram/prose mismatch was found.

## 3. Roadmap alignment

The code implements the `folio graph` sub-slice of parent §15.6 and does not attempt to ship the deferred `folio synthesize` or `folio search` consumers. The helper-promotion carry-forward remains consistent with B.3: `_derive_trust_status` is still private in `folio/graph.py` for v0.7.1, while sub-slice 2 is left to promote it before synthesize design.

The added `--include-flagged` option is a scope-sensitive change. It is not named in v0.7.1 spec §3.4, but it is an understandable consequence of parent §11's explicit-override requirement and of the flagged-output tests: without an override, flagged proposals remain filtered by `collect_pending_relationship_proposals` and the `[flagged]` renderer path is not reachable. The manifest's D.1 revision now names the flag in `G-scope-3`, and the changelog documents it. This is not a silent product expansion, but the spec should be backfilled to make the authority explicit.

## 5. Backward compatibility

The behavior-level backward-compatibility story matches v1.2. Pending relationship proposals now carry `subject_id: null` for §5 semantics, preserve the proposal identifier under `proposal_id`, and render `proposal_id` in the default CLI row. The `graph doctor --json` breaking change is documented in `CHANGELOG.md`, and non-proposal findings retain their legacy fixed shape in the migrated schema test.

The `--include-flagged` flag is additive and defaults to the prior behavior, so it does not break existing callers. The compatibility risk is documentation alignment rather than runtime behavior: the spec should name the flag the same way the manifest and changelog now do.

## 6. Consistency check

Spec v1.2 and B.3 closed the earlier §3.3 naming/scope blockers correctly, and the code follows that closure. `target_registered` is honest registry-deck membership, not target activity or non-deck validity. The code also honors the documented-only dispatch hook by leaving future proposal types unimplemented.

The unresolved consistency problem is the manifest/test anchor mismatch. B.3 says the cardinality gates are runtime fixture gates to be verified later, but the manifest's runnable commands do not point to the test file that actually contains those tests. This is an approval-pipeline mismatch, not an emitted-finding mismatch.
