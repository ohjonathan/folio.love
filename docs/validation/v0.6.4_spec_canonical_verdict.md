## Consolidated Blockers

### CB-1 (Target trust state is read from stale registry while source is read from live frontmatter)
*Sources:* B-003 (peer), ADV-001 (adversarial)

Spec reads source `review_status` from live frontmatter (already loaded by
`registry_mod._read_frontmatter`) but reads target `review_status` from the
registry dict (populated by an earlier sync). The registry contract treats
`review_status` as frontmatter-authoritative and does NOT reconcile on load.
This creates a window during which target state disagrees with registry:

- Target flagged in frontmatter, clean in registry → proposal not filtered
  by default; `folio links confirm <id>` proceeds without operator consent path.
- Target clean in frontmatter, flagged in registry → proposal filtered;
  `--include-flagged` annotates as `(flagged: target)` despite target being clean.

**Resolution:** Change the spec to read target `review_status` from the target
document's frontmatter (not registry). Use the registry only to locate the
target markdown path (which it already does for all other fields). Add
regression tests for both directions of staleness.

### CB-2 (Status / confirm-doc / reject-doc silently report empty when flagged inputs are the real cause)
*Sources:* AL-1 (alignment), AL-2 (alignment), PR-1 (product), PR-2 (product), ADV-002 (adversarial), ADV-003 (adversarial)

Three reviewers independently identified that §11 rule 5 applies to every
surface operated by `folio/cli.py`, not just `folio links review`:

- **`folio links status`:** `relationship_status_summary` discards
  `suppression_counts`. When all proposals for a source are flagged-input,
  the status row shows `pending: 0` (or is omitted), violating rule 5.
- **`folio links confirm-doc` / `reject-doc`:** These bulk commands call
  `collect_pending_relationship_proposals` with default filtering and print
  `Confirmed 0 proposal(s)` / `Rejected 0 proposal(s)` when all proposals
  are flagged-input — indistinguishable from a document with no proposals.

**Resolution:**
- Extend `relationship_status_summary` to return `tuple[list[RelationshipStatusRow], int]`
  with a total flagged-excluded count. CLI discloses the count.
- `confirm_doc` and `reject_doc` return `tuple[int, int]` (acted, flagged-excluded).
  When acted == 0 and flagged_excluded > 0, CLI prints a non-success diagnostic
  directing the operator to `folio links review --include-flagged`.

### CB-3 (`suppression_counts["flagged_input"]` collides with rejection-memory rendering)
*Sources:* B-001 (peer), ADV-004 (adversarial)

Existing CLI computes `total_suppressed = sum(suppression_counts.values())` and
labels it as rejection memory. Adding a `"flagged_input"` sentinel key to the
same dict causes:

- Double-counting: flagged-input suppressions show up in both the rejection-memory
  disclosure and the new flagged-input disclosure.
- Producer-name collision: a producer literally named `flagged_input` would
  be indistinguishable from the sentinel.

**Resolution:** Change the return shape of `collect_pending_relationship_proposals`
to `tuple[list[RelationshipProposalView], SuppressionCounts]` where
`SuppressionCounts` is a `TypedDict` or dataclass with two fields:
`rejection_memory: dict[str, int]` (per-producer) and `flagged_input: int`.
Update every renderer and caller.

### CB-4 (`--include-flagged` creates orphaned proposals that action commands cannot touch)
*Sources:* PR-SF-4 (product, elevated to blocker), ADV-SF-002 (adversarial)

Originally classified as should-fix by product but elevated here because it
creates a broken consent workflow: operator runs `folio links review --include-flagged`,
sees a flagged-input proposal, copies the ID, then `folio links confirm <id>`
fails with `Unknown proposal_id`. The proposal was real — it just isn't reachable
from the default-filtered collection used by `_find_pending_view`.

**Resolution:** Either
- (a) Add `--include-flagged` to `folio links confirm`, `reject`, `confirm-doc`,
  `reject-doc`; OR
- (b) Document in the `--include-flagged` review output that flagged-input
  proposals are read-only diagnostics until the document is unflagged.

**Recommendation:** Option (a). Explicit consent per command matches §11 rule 3
(operator override) and closes the silent-empty holes in CB-2 with the same
mechanism.
