## 5. New-issue scan (introduced by v1.1)

### N-1 — `reject_person_merge` return type widened to `tuple[bool, str]`

**Observation:** Round 1 method returned implicit success. v1.1 §4.1 item 6
(line 126) now returns `tuple[bool, str]` — `(True, "rejected")` on append,
`(False, "already rejected")` on no-op.

**Concern:** does any other code path call this method today? Check:

- grep: no current call sites of `reject_person_merge` (the method is new
  in this slice; nothing to break).
- The CLI is the only caller and §4.2 line 154 handles both return tuples.

**Verdict:** NOT a defect. New API; tuple shape is fine and makes the
idempotent-vs-real distinction machine-readable. The second element
("rejected" / "already rejected") is effectively a tag, not a free-form
message — worth noting in a docstring but not worth another revision.

### N-2 — "always rendered" `(0 total rejections recorded.)` contract

**Observation:** v1.1 §4.2 line 145 commits to `(0 total rejections recorded.)`
literally when `M == 0`. Slice 1 parallel check: v0.6.0 §4.4 /
`folio/cli.py:1648-1652` does NOT emit a "total rejections" line at all —
only the suppression-delta line. So "total rejections recorded" is a
NEW surface introduced in this slice, not an import from slice 1.

**Test-lock check:** T-10b (line 234) says:
"Output contains `({M} total rejections recorded.)` (including
`(0 total rejections recorded.)` when pristine)." The pristine case IS
explicitly asserted, which locks the "always" commitment.

**Verdict:** NOT a defect. The zero-case literal is unambiguous and
test-locked. Minor cosmetic nit: parentheses around the total-rejections line
aren't echoed in slice 1's surface (slice 1 suppression line is unparenthesized).
This is an intentional visual separator between the suppression-delta line
and the registry-total line; acceptable. No action needed.

### N-3 — Lock scope leak between `merge` and `reject-merge`

**Observation:** §4.2 line 157 explicitly scopes locking to `reject-merge` +
`suggest-merges` read path. `merge` / existing `reject` / `import` are NOT
locked. Spec acknowledges this with a written limitation —
"operators running `merge` and `reject-merge` in parallel terminals can
still race in the merge vs reject-merge direction."

**Concern check:** is the surfaced error (`EntityRegistryError` on stale key)
actually what happens? Yes — `reject_person_merge` (§4.1 item 6) validates
both keys exist under `entities["person"]` and raises the error if not;
§4.2 line 155 surfaces this as `✗ Merge candidate is stale …` with non-zero
exit.

**Verdict:** NOT a defect — limitation is explicitly scoped, documented in
§4.2 + §6, and degrades safely (error, not corruption). T-15 covers only
the in-scope case. Broader locking is listed under §8 carry-forward.

### N-4 — `setdefault` inside `load()` runs AFTER per-etype validation

**Observation:** §4.1 item 2 now pins insertion point: "end of `load()` after
schema-version validation and entity-dict normalization." Current `load()`
(entities.py:528-599) ends at line 599 with `self._loaded = True`. The
`setdefault("rejected_merges", [])` must be inserted between line 598
(`self._data = data`) and the return, OR on `data` before the assignment.
Either works. Spec pins "end of `load()`" which is unambiguous enough for
Phase C.

**Verdict:** NOT a defect; M-5 closed cleanly.

### N-5 — T-10c scope vs. `suggest_merges_cmd` control-flow rewrite

**Observation:** T-10c asserts disclosure renders on empty + paged + filtered
paths. Current cli.py:2488-2495 has two early `return`s; Phase C must rewrite
to emit disclosure at the tail in all cases. This is a structural rewrite,
not a one-line patch. Whether §4.2's "render … at the END of the command
flow" is enough guidance is an implementation-review question, not a spec
defect — the binding contract is clear.

**Verdict:** NOT a defect. Flagged as implementation-effort note for Phase C.
