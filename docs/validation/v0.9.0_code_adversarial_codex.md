## Should-fix (ADV-SF-NNN)

### ADV-SF-001 - `--limit 0` produces false zero-match diagnostics

**Severity:** should-fix

`search()` computes `total_available` before slicing (`folio/search.py:260`), applies `producer_matched = producer_matched[:limit]` (`folio/search.py:261`), then renders diagnostics based on `not report.findings` (`folio/search.py:347`, `folio/search.py:359`) before printing the truncation footer (`folio/search.py:379`).

That means `folio search framework --limit 0` with real matches prints both:

- `No matches for 'framework'...`
- `(showing 0 of 1; use --limit to adjust)`

Those statements contradict each other. With `--producer enrich --limit 0`, the renderer additionally prints the producer exact-match hint even though producer matching succeeded; the zero findings are solely a limit artifact (`folio/search.py:266`, `folio/search.py:347`).

The test suite misses this because `test_search_limit_caps_findings` checks only JSON finding count for `--limit 0` (`tests/test_cli_search.py:430`) and the stdout footer test uses a positive limit (`tests/test_cli_search.py:460`).

**Required fix:** drive zero-match diagnostics from pre-limit availability, not post-limit emitted findings. In practice: use `report.total_available == 0` for the no-match breadcrumb and producer hint; reserve `not report.findings and report.truncated` for the truncation footer.

### ADV-SF-002 - Producer zero-match hint can falsely claim there are no proposals in scope

**Severity:** should-fix

`producer_hint_names` is populated from `query_matched`, i.e. after QUERY filtering but before producer filtering (`folio/search.py:255`, `folio/search.py:266`). When `--producer` is supplied and QUERY matches nothing, `producer_hint_names=[]`; the renderer then prints:

`Hint: no proposals in scope for any producer. Run \`folio ingest\` / \`folio enrich\`...`

That message is false when the scope has proposals but none match QUERY (`folio/search.py:355`). The same renderer labels non-empty hints as "Observed producers in scope" (`folio/search.py:350`) even though the list is actually "observed producers among QUERY-matched views."

`SRC-CLI-30` only covers the wrong-case producer path where QUERY did match a proposal (`tests/test_cli_search.py:543`); it does not cover `--producer` plus a query miss.

**Required fix:** either collect producer names from the full scoped view set when the message says "in scope", or keep the current data source and change the text/control flow to say "among QUERY matches." If `query_matched` is empty, fall back to the normal "No matches for QUERY" breadcrumb rather than the producer-population hint.

### ADV-SF-003 - `_view_to_finding()` is less total than `_view_matches()` for malformed views

**Severity:** should-fix

The matcher is defensive for non-string field values through `_normalize_search_text()` (`folio/search.py:145`) and uses `(view.proposal.signals or [])` for evidence matching (`folio/search.py:184`). The conversion path is not equally defensive:

- `evidence_bundle=list(view.proposal.signals)` raises `TypeError` when `signals is None` (`folio/search.py:204`).
- `flagged_inputs=list(view.flagged_inputs)` raises `TypeError` when `flagged_inputs is None` even though `derive_trust_status()` treats `None` as `"ok"` (`folio/search.py:206`, `folio/search.py:212`; helper behavior at `folio/tracking/trust.py:17`).
- direct attribute access throughout `_view_matches()` and `_view_to_finding()` raises `AttributeError` for missing `proposal` attributes (`folio/search.py:169`, `folio/search.py:172`, `folio/search.py:181`, `folio/search.py:198`).

The current malformed-fixture tests cover non-string signal elements and `None` target/relation (`tests/test_search.py:712`, `tests/test_search.py:744`), but they still provide `signals` and `flagged_inputs` as lists. Real deserialization also preserves explicit `None` for fields such as `rationale` and `basis_fingerprint` because `RelationshipProposal.from_dict()` uses `d.get(...)` without coercing explicit nulls (`folio/pipeline/enrich_data.py:103`, `folio/pipeline/enrich_data.py:115`).

**Required fix:** add a small coercion layer in `_view_to_finding()` matching the matcher discipline: tolerate `None`/missing `signals` and `flagged_inputs`, coerce payload text fields to strings or safe defaults where the shared contract requires strings, and add regression tests using `SimpleNamespace` fixtures with `signals=None`, `flagged_inputs=None`, and missing optional proposal attributes.

### ADV-SF-004 - SRC-23 does not enforce the full forbidden-symbol contract

**Severity:** should-fix

The ratified spec says SRC-23 covers all §2.3 forbidden entries, including `compile(`, `subprocess.*shell=True`, unsafe dynamic imports, `globals()[`, `locals()[`, `Path.match`, `.glob` on QUERY inputs, and pattern engines (`docs/specs/v0.9.0_folio_search_spec.md:247`, `docs/specs/v0.9.0_folio_search_spec.md:992`, `docs/specs/v0.9.0_folio_search_spec.md:1119`).

The implemented test only blocks AST imports of `re` and `fnmatch`, `re.<func>` calls, and substring checks for `pickle.loads`, `eval(`, `exec(`, `os.system`, `yaml.load(`, and `__import__(` (`tests/test_search.py:501`). It does catch `from re import search as _s`, but it would not catch `import regex`, `import re2`, `from regex import search`, `compile(...)`, `importlib.import_module(...)`, `globals()[...]`, `locals()[...]`, `subprocess.run(..., shell=True)`, `Path(...).match(...)`, or `Path(...).glob(...)`.

The current production code is clean on this point: `folio/search.py` imports only stdlib `json`/`unicodedata`/dataclasses/typing, `click`, and the allowed Folio dependencies (`folio/search.py:22`, `folio/search.py:29`). This is therefore not a blocker, but the guard is materially weaker than the spec and the adversarial examples named in B.1.

**Required fix:** extend SRC-23's AST walk to block third-party regex modules (`regex`, `re2`) for v0.9.0, alias calls to pattern APIs, `compile`, dynamic import helpers, `globals`/`locals` subscripting, `subprocess` calls with `shell=True`, and `Path.match`/`Path.glob`. Add behavior tests for at least `[a-z]+` and `(?=x)`, since the spec says SRC-41 covers those but the implementation currently tests only `.*` (`tests/test_search.py:854`).

### ADV-SF-005 - Programmatic `search(limit=-1)` still has Python slice semantics

**Severity:** should-fix

The CLI closure is correct: `--limit` uses `click.IntRange(min=0)` (`folio/cli.py:2026`) and negative CLI input exits 2 (`tests/test_cli_search.py:534`). The public function does not enforce the same invariant. A direct caller can pass `search(..., limit=-1)`, and `producer_matched[:limit]` will emit all but the final match (`folio/search.py:261`).

This is not a CLI blocker, but `folio.search.search()` is a public module surface used by tests and future callers (`folio/search.py:216`). Function-level parity with empty QUERY validation would avoid a quiet API footgun.

**Required fix:** reject negative `limit` in `search()` with `ValueError`, or clamp to zero if the intended public API mirrors graph behavior. Add a unit test for `search(config, query="x", limit=-1)`.

### ADV-MIN-003 - SRC-33 is a weak read-only test
