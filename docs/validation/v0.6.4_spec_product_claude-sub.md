## Detailed Findings

### PR-1 (blocker): `folio links status` silently misreports pending counts for flagged documents

Section 9.2 states: "Flagged-input proposals are already excluded from counts. No additional work needed for this surface -- the count accurately reflects reviewable proposals." This is technically correct but produces a misleading operator experience.

Consider: a document has 5 pending proposals, all involving a flagged target. `folio links status` shows `| doc_abc | 0 | 3 |`. The operator believes no proposals need review. They never learn that 5 proposals exist but were filtered. This is the same class of "silent empty" problem that section 5.4 solves for `folio links review`, but `folio links status` gets no equivalent protection.

The `relationship_status_summary` function (links.py line 228) calls `collect_pending_relationship_proposals` with default `include_flagged=False` and discards the `suppression_counts` dict (line 236: `pending_views, _ = ...`). The flagged-input count is computed but thrown away.

**Minimum fix:** `relationship_status_summary` should propagate the `suppression_counts["flagged_input"]` value so the CLI can append a footnote: "N proposal(s) excluded from counts (flagged inputs). Use `folio links review --include-flagged` to see them." This does not require changing the table format -- a footer line after "Total: X pending, Y confirmed" is sufficient.

### PR-2 (blocker): `confirm-doc` / `reject-doc` silently skip flagged proposals with no operator feedback

Section 5.5 states that `confirm_doc` and `reject_doc` call `collect_pending_relationship_proposals` without `include_flagged`, so flagged-input proposals are excluded. This is described as "intentional: operators should not be able to confirm or reject proposals from flagged documents without explicitly opting in."

The problem is the operator experience of this intention. Current code (links.py lines 426-443):

```python
def confirm_doc(config, doc_id):
    views, _ = collect_pending_relationship_proposals(config, doc_id=doc_id)
    ...
    return count
```

If a document has 5 proposals but the document is flagged, the operator runs `folio links confirm-doc doc_abc` and sees "Confirmed 0 proposal(s)." There is no message explaining that 5 proposals were skipped because the document is flagged. The operator will think the document has no proposals, not that the proposals are trust-gated.

This violates the spirit of section 11 rule 5 (no surface may silently report empty when flagged inputs are the real cause). `confirm-doc` and `reject-doc` are surfaces that report counts to the operator.

**Minimum fix:** When `confirm-doc` or `reject-doc` returns 0 confirmed/rejected AND the source document has `review_status: flagged`, emit a diagnostic message: "0 proposals confirmed. Document 'doc_abc' has review_status: flagged. Use `folio links review --include-flagged` to see its proposals." This requires checking the document's `review_status` in the CLI layer -- no changes to `links.py` needed.

### PR-SF-1 (should-fix): Disclosure line does not distinguish source-flagged from target-flagged exclusions

Section 5.3 specifies:

```
3 proposal(s) excluded (flagged inputs). Use --include-flagged to review.
```

This tells the operator that proposals were excluded but not why. If 2 proposals were excluded because their source is flagged and 1 because its target is flagged, the operator cannot tell. This matters because the remediation is different:

- Source flagged: the operator needs to address the source document's review flags before trusting its proposals.
- Target flagged: the proposals themselves may be fine, but they reference a document that needs attention.

**Suggested fix:** Split the disclosure or add a breakdown:

```
3 proposal(s) excluded (flagged inputs: 2 source, 1 target). Use --include-flagged to review.
```

This requires tracking source-only vs. target-only vs. both counts in `suppression_counts`, e.g., using keys `"flagged_source"`, `"flagged_target"`, `"flagged_both"` instead of or in addition to `"flagged_input"`.

### PR-SF-2 (should-fix): Silent-empty header does not explain what `--include-flagged` does

Section 5.4 specifies:

```
Pending (0 items -- N excluded due to flagged inputs; use --include-flagged to review)
```

The `--include-flagged` mention is good for discoverability. However, the header does not explain what "flagged inputs" means. An operator encountering this for the first time will see the flag name but may not know what makes an input "flagged" or what they should do about it.

**Suggested fix:** Expand the hint slightly:

```
Pending (0 items -- N excluded because source or target has review_status: flagged; use --include-flagged to review)
```

Alternatively, add a one-line explanation after the header: "Flagged documents have unresolved review issues. See `folio provenance review` to address them."

### PR-SF-3 (should-fix): `folio links status` gives no indication that flagged-input proposals exist

Related to PR-1 but distinct: even if `folio links status` correctly excludes flagged-input proposals from the "Pending" column (which is the spec's intent), there is no column, footnote, or annotation indicating that hidden proposals exist. An operator who only uses `folio links status` to decide whether to run `folio links review` will never discover proposals gated behind flagged inputs.

**Suggested fix:** Add a "(N flagged-gated)" annotation to the Pending column when the excluded count is nonzero, or add a summary footer line similar to the one proposed for `folio links review`. This is lower priority than PR-1 because `folio links review` is the primary triage surface.

### PR-SF-4 (should-fix): Partial-flagging semantics deserve explicit operator guidance

The spec handles partial flagging correctly at the data level: if the source is clean but the target is flagged, the proposal is still excluded (section 4.2-4.3). The trust-posture tag in section 5.2 shows `(flagged: target)` when `--include-flagged` is used.

However, the spec does not describe the operator's workflow for this case. When the operator sees `(flagged: target)` and wants to act on the proposal:

1. Should they fix the target document first? The target's flagged status may be unrelated to the proposal's validity.
2. Can they confirm the proposal anyway via `folio links confirm` while the target is flagged?
3. If they confirm, does the canonical relationship get written even though the target is flagged?

Currently, `folio links confirm <proposal_id>` calls `_find_pending_view`, which calls `collect_pending_relationship_proposals` with default `include_flagged=False`. So the operator cannot confirm a flagged-target proposal even if they saw it via `--include-flagged`. This is a dead end: the review surface shows proposals that cannot be acted upon through the normal workflow.

**Suggested fix:** Document the intended workflow explicitly. Either: (a) `folio links confirm` should also accept `--include-flagged` to allow confirmation of flagged-target proposals, or (b) the spec should state that the operator must resolve the target's flagged status before confirming, and the `(flagged: target)` tag serves as a "blocked" indicator rather than an actionable item.

### PR-I-1 (informational): Consider supporting `--include-flagged` on `confirm-doc` / `reject-doc`

Section 5.5 intentionally excludes flagged proposals from bulk operations. This is a reasonable safety default. However, if the operator has triaged proposals via `folio links review --include-flagged` and decided they want to confirm all proposals for a flagged document, they cannot use `confirm-doc` -- they must confirm each proposal individually. For a document with many proposals, this is tedious.

A follow-up could add `--include-flagged` to `confirm-doc` / `reject-doc` for operators who have already reviewed via the flagged surface.

### PR-I-2 (informational): Disclosure line placement relative to pagination

Section 5.3 places the flagged-input disclosure "after the existing suppression disclosure." The existing suppression disclosure (rejection memory) is at the bottom of the review output (cli.py lines 1622-1628), after the paginated proposal list. On a full page of 20 proposals, the disclosure is several screens below the header. If the operator is paging through a large review queue, the flagged-input count at the bottom of the first page may not be noticed.

This is acceptable for the current design but worth noting: the silent-empty protection in section 5.4 covers the critical case (0 visible proposals), and the bottom-of-page disclosure covers the partial case (some visible, some hidden).

### PR-I-3 (informational): Behavior when `review_status` has an unexpected value

Section 6 states that if `review_status` is missing or not a string, the document is treated as NOT flagged (fail-open). This is correct. The spec does not discuss what happens when `review_status` is a string but not one of the expected values (e.g., `review_status: "pending"` or `review_status: "quarantined"`). The fail-open behavior means any non-`"flagged"` string is treated as clean, which is reasonable. No change needed, but the spec could include a one-sentence note: "Only the literal value `'flagged'` triggers trust-gating; all other `review_status` values, including unknown future values, are treated as non-flagged."

## Verdict

Needs Fixes

## Rationale

Two blockers prevent approval.

PR-1 identifies a silent-misreport bug in `folio links status`: the status table throws away the flagged-input suppression count and shows a pending count of 0 for documents whose proposals are all gated behind flagged inputs. This directly contradicts section 11 rule 5 ("no surface may silently report empty when flagged inputs are the real cause"). The spec acknowledges `folio links status` in section 9.2 but incorrectly concludes "no additional work needed."

PR-2 identifies the same class of silent-empty problem in `confirm-doc` / `reject-doc`: an operator running `folio links confirm-doc doc_abc` on a flagged document sees "Confirmed 0 proposal(s)" with no explanation. The operator cannot distinguish "no proposals exist" from "proposals exist but are trust-gated." The spec's section 5.5 acknowledges the exclusion but does not address the operator feedback gap.

The four should-fix items address operator experience gaps:
- PR-SF-1: the disclosure line lumps source-flagged and target-flagged exclusions together, losing diagnostic signal the operator needs for remediation.
- PR-SF-2: the silent-empty header mentions `--include-flagged` but does not explain what "flagged inputs" means, reducing first-time discoverability.
- PR-SF-3: `folio links status` gives no indication that hidden proposals exist, so operators who use status as their triage entry point will miss them.
- PR-SF-4: the operator workflow for partial flagging (clean source, flagged target) is a dead end -- proposals are visible via `--include-flagged` but cannot be confirmed through the normal `folio links confirm` path.

The informational items are suggestions for follow-up work and documentation clarity.
