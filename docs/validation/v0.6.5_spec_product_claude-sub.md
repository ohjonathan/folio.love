# B.1 Product Review -- entity-merge-rejection-memory-v0-6-5

## Summary

Spec is well-scoped and consistent with slice 1's architecture. The operator-experience surface is small (one new command, one disclosure line, one doctor filter) and the design decisions (pair-sorted identity, fingerprint revival, idempotent reject, always-rendered disclosure) all track slice 1 exactly. However, four operator-facing wording and feedback problems should be fixed before implementation: the suppression-count copy diverges from slice 1 in a subtly-parallel way that the sole operator will notice, the `reject-merge` success message is less informative than its slice 1 analog, idempotent re-reject gives silent success when slice 1's pattern admits an ambiguity the operator cannot diagnose, and `graph doctor` silently hides rejected merges with no counterpart surface for "did my reject stick?" verification. One minor: help text omits an example.

Verdict: **Needs Fixes** (4 should-fix items; no blockers).

## Detailed findings

### PR-SF-1 (should-fix): Suppression-count copy diverges from slice 1

**Slice 1 copy** (verified in `folio/cli.py:1646-1652`):

```
"No proposals suppressed by rejection memory."         # N == 0
"1 proposal suppressed by rejection memory."           # N == 1
"{N} proposals suppressed by rejection memory."        # N > 1
```

**Spec §4.2 item 1 copy:**

```
"{N} merge candidates suppressed by rejection memory."   # always, even N == 0
```

Two divergences the sole operator will notice when scanning output:

1. **N == 0 wording.** Slice 1 says `"No proposals suppressed..."` (natural-language negative). Spec says `"0 merge candidates suppressed..."` (numeric zero). The "N=0 uses 'No' prefix" pattern is established in slice 1; spec should match it.

2. **Pluralization.** Slice 1 has a three-branch ladder (`No` / singular `1 proposal` / plural `{N} proposals`). Spec's always-plural `"merge candidates"` collapses the N=1 case to `"1 merge candidates suppressed by rejection memory."` which reads wrong.

**Recommended rewrite for §4.2 item 1:**

```
N == 0: "No merge candidates suppressed by rejection memory."
N == 1: "1 merge candidate suppressed by rejection memory."
N >  1: "{N} merge candidates suppressed by rejection memory."
```

This matches slice 1's three-branch structure and preserves parallel reading across `folio links review` and `folio entities suggest-merges`. Nothing about "merge candidates" vs "proposals" is negotiable (they are different surfaces), but the "No / 1 / N" structure should be identical.

### PR-SF-2 (should-fix): `reject-merge` success message drops reasons context

**Slice 1 reject output** (verified at `folio/cli.py:1775-1778`):

```
"✓ Rejected {proposal_id}: {relation} -> {target_id}"
```

Note: slice 1 echoes the *semantic content* of what was rejected (relation + target), not just the IDs. The operator who typed `folio links reject prop_abc123` gets a confirmation that reads back the meaning.

**Spec §4.2 item 2 success copy:**

```
"Rejected merge candidate: {left_name} <> {right_name}"
```

This is better than nothing (names not keys), but it drops the *reasons* context that the operator just saw in `suggest-merges` output. `suggest-merges` renders each candidate with its `Reasons: last_first_transpose, alias_overlap` line; the rejection `basis_fingerprint` is derived from exactly those reasons. If the operator rejects a candidate and sees only `"Rejected merge candidate: Alice Chen <> Chen Alice"`, they have no echo of "which heuristic bundle they're rejecting." When the pair revives later with `(revived — basis changed)`, the operator has to remember or look up what reasons changed.

**Recommended rewrite for §4.2 item 2:**

```
"✓ Rejected merge candidate: {left_name} [{left_key}] <> {right_name} [{right_key}] (reasons: {reasons})"
```

Match slice 1's leading `✓` prefix for symmetry. Include both names and keys (matches `suggest-merges` output row format exactly). Include `reasons` so that a future revival's `(revived — basis changed)` annotation ties back to a specific earlier reason set.

**Unknown-key error path (spec §4.2 item 2 "Exits non-zero on unknown key"):** spec does not specify the message copy. Slice 1 errors are prefixed with `✗` (see `folio/cli.py:1770`, `1485`, etc.). Recommended copy:

```
"✗ No person found matching '{left_key}'."          # mirrors existing lookup error at cli.py:2541
```

Plus a follow-up hint when relevant: if one key exists and the other does not, name which one is unknown. Spec should commit to this in §4.2.

### PR-SF-3 (should-fix): Idempotent re-reject is silently success

**Spec §4.1 item 6** says second-rejection of the same `(subject_type, sorted_keys, basis_fingerprint)` is a "no-op." Combined with §4.2 item 2's success copy, the operator-visible behavior is:

```
$ folio entities reject-merge alice_chen chen_alice
Rejected merge candidate: Alice Chen <> Chen Alice     # first call: stored

$ folio entities reject-merge alice_chen chen_alice
Rejected merge candidate: Alice Chen <> Chen Alice     # second call: SAME output, no record written
```

For a solo operator running the command twice (maybe by muscle memory, or because shell history replayed it, or because they're not sure whether the first call landed), the two outputs are indistinguishable. This is the ambiguity slice 1 also has in some places, but slice 1's `links reject` actually errors on a second call because the proposal transitioned out of `pending_human_confirmation` — so the silent-same-output pattern doesn't manifest there.

**Recommended behavior:** When `reject_person_merge` detects an identical rejection record already exists, emit a distinct message:

```
# first call
"✓ Rejected merge candidate: Alice Chen [alice_chen] <> Chen Alice [chen_alice] (reasons: last_first_transpose)"

# second call (no-op)
"Already rejected: Alice Chen [alice_chen] <> Chen Alice [chen_alice] (no change)"
```

This preserves idempotency (exit 0, no error) while giving the operator explicit feedback. The spec should add a third bullet under §4.2 item 2:

> Emits `"Already rejected: {left_name} [{left_key}] <> {right_name} [{right_key}] (no change)"` on idempotent re-reject; exits 0.

### PR-SF-4 (should-fix): `graph doctor` silently hides rejected merges; no verification surface

**Spec §4.3** says `graph_doctor` passes `apply_rejection_memory=True` when invoking `suggest_person_merges()`. The consequence: rejected merges stop appearing as `duplicate_person_candidate` findings.

For a solo operator, this raises a real question: **"I rejected Alice <> Chen yesterday. Did it stick? Is the record still there?"** Under the spec as written, there is *no CLI surface* that answers this question.

- `folio entities suggest-merges` — hides it (filter applied)
- `folio graph doctor` — hides it (filter applied, per §4.3)
- `entities.json` — inspectable but requires JSON-diving

Contrast slice 1's `folio links review` pattern: rejected proposals are suppressed from review, but the `folio links status` surface and the frontmatter itself still let the operator verify `status: rejected` on the original proposal record. For entity-merge, the rejection record is in `entities.json`'s `rejected_merges` array — no command surfaces it.

Three options, in increasing scope:

1. **Minimum (recommended):** Add one line to `folio entities suggest-merges` output (alongside the existing suppression count) listing the total rejections on record:
   ```
   N merge candidates suppressed by rejection memory.
   (M total rejections recorded in entities.json.)
   ```
   Zero new commands; cost is one integer read.

2. **Medium:** Add `--include-rejected` flag to `folio entities suggest-merges` mirroring slice 1's `--include-flagged` ergonomic pattern. When set, rejected merges appear in the output with a `(rejected at {date})` annotation, visually distinct.

3. **Maximum:** A new `folio entities list-rejected-merges` command. Deferrable to a follow-up slice; option 1 is sufficient for "did my reject stick?" verification this slice.

The spec should commit to at least option 1 in §4.2 or add a §4.4 CHANGELOG line. Without any verification surface, the solo operator's mental model of rejection memory is "I hope it worked" rather than "I can check."

Alternative framing for §4.3: rather than silently filtering in `graph doctor`, fold rejected candidates into a collapsed summary row like `"{M} merge candidates suppressed by rejection memory (use 'folio entities suggest-merges' for details)"` within the findings table. This keeps the symmetry with the CLI suggest-merges surface. Either approach resolves the issue; spec should pick one.

### PR-MN-1 (minor): Help text for `reject-merge` should include an example

Spec §4.2 item 2 lists the behavior but not the Click docstring. The spec should commit to docstring copy. Comparing existing `entities merge` command (`folio/cli.py:2514-2519`):

```python
@entities.command("merge")
@click.argument("winner")
@click.argument("loser")
@click.pass_context
def merge_entities_cmd(ctx, winner: str, loser: str):
    """Merge one person entity into another."""
```

One-line docstring. Terse. The `reject-merge` command's semantics are more novel (the operator may not know what "basis_fingerprint" means in `--help` output), so a one-line docstring is marginal. Recommended:

```python
def reject_merge_cmd(ctx, left_key: str, right_key: str):
    """Reject a suggested person-merge candidate.

    Records a rejection keyed on the sorted pair + current heuristic
    basis so the candidate stays hidden from `suggest-merges` unless
    the alias set later shifts the basis fingerprint.

    Example: folio entities reject-merge alice_chen chen_alice
    """
```

Short help stays on the summary line (shown in `folio entities --help`); the body helps the operator who runs `folio entities reject-merge --help` directly.

Spec should add this docstring verbatim to §4.2 item 2.

### PR-I-1 (informational): Revival annotation verbatim match confirmed

Spec §4.2 item 1 uses `(revived — basis changed)` as the revival annotation. Slice 1 uses `(revived — basis changed)` (verified `folio/cli.py:1634`). Exact match including the em-dash. No action needed; flagged here so reviewers can quickly confirm. The OQ-3 recommendation (inline suffix) also matches slice 1's placement.

### PR-I-2 (informational): OQ-2 "no reason field" is the right call; add carry-forward

Spec §5 OQ-2 defers a free-text rejection reason field. Correct call — slice 1 does not have one either, and the rejection-reason-field idea is meaningfully different in cost (schema field) from the "operator-wanted" improvements above (all copy-only). Recommended: add a row to §8 carry-forward disposition:

> | This slice | Rejection-reason free-text field | Defer — operator friction has not surfaced need; add when first concrete use case appears |

This makes the decision traceable if a future reviewer asks "why no reason field?"

## Verdict rationale

All four should-fix items are operator-experience copy or feedback-surface issues, not architectural problems. None of them block implementation; they require a spec addendum (roughly §4.2 item 1 copy rewrite, §4.2 item 2 docstring + error message + idempotent-case message specification, and a §4.3 verification-surface addition or cross-reference). Post-rewrite the spec is Approve-quality.

The spec's core design — pair-sorted rejection key, fingerprint-based revival, doctor-filter consistency, idempotent no-op, no `--force` requirement — all track slice 1 correctly and should not change.
