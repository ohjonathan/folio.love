## Findings

### D-COD-1: Suppressed proposal re-queue on subsequent run (informational)

**Severity:** informational

**Question:** Can a suppressed proposal get re-queued on a subsequent run?

**Analysis:** Yes, by design. On each enrich run, `_suppress_rejected_proposals`
operates on freshly constructed `RelationshipProposal` objects (line 705-720),
all initialized with `lifecycle_state="queued"`. The suppression filter then
checks each against the rejected-basis map. Old suppressed entries are NOT
preserved by `_preserve_rejected_proposals` (which only preserves `"rejected"`).

This means: if the LLM re-proposes the same relationship on a subsequent run,
it starts as `"queued"` again. If the rejection entry still exists with matching
`basis_fingerprint`, it gets re-suppressed. If the rejection entry was removed
(e.g., operator un-rejected), the proposal passes through as `"queued"`.

This is correct behavior per spec section 3.2: "Suppressed proposals from a
prior run have no value on re-enrichment: the LLM will re-propose (or not)
based on current inputs." No bug here.

### D-COD-2: `_preserve_rejected_proposals` legacy `status` field handling (should-fix)

**Severity:** should-fix

**Question:** Does `_preserve_rejected_proposals` correctly handle legacy `status` field?

**Analysis:** The function uses `_STATUS_TO_LIFECYCLE` for the `is None` fallback
(lines 1059-1061):

```python
state = p.get("lifecycle_state")
if state is None:
    old_status = p.get("status")
    state = _STATUS_TO_LIFECYCLE.get(old_status, old_status)
```

Meanwhile, `_suppress_rejected_proposals` uses a simpler pattern (lines 1023-1025):

```python
state = p.get("lifecycle_state")
if state is None:
    state = p.get("status")
```

These two functions use different legacy-compat patterns for the same purpose.
`_suppress_rejected_proposals` does a raw `p.get("status")` without mapping
through `_STATUS_TO_LIFECYCLE`. For the `"rejected"` case this is
coincidentally correct because `_STATUS_TO_LIFECYCLE` maps `"rejected"` to
`"rejected"` (identity mapping). However, the inconsistency is a maintenance
hazard: if someone adds a new status-to-lifecycle mapping that affects rejection
detection, only one function would pick it up.

**Recommendation:** Unify both functions to use `_get_lifecycle_state()` or at
minimum have `_suppress_rejected_proposals` use `_STATUS_TO_LIFECYCLE` the same
way `_preserve_rejected_proposals` does. Not a functional bug today, but a
should-fix for consistency.

### D-COD-3: `_count_library_queued_proposals` edge cases (informational)

**Severity:** informational

**Question:** What about empty registry, missing frontmatter, concurrent writes?

**Analysis:**

1. **Empty registry / no `registry.json`:** Handled. Line 1088 checks
   `registry_path.exists()` and returns 0. Empty `reg_data.get("decks", {})`
   produces an empty iterator, returning 0. Correct.

2. **Missing frontmatter:** Handled. Line 1107 checks `_read_frontmatter(md_path)`
   returns a truthy dict; if `None` or empty, continues. Correct.

3. **Corrupted frontmatter / parse failures:** Handled. The inner try/except
   (lines 1104-1125) catches any exception per L2 failure policy and skips the
   document with a warning log. Correct.

4. **Concurrent writes:** The function reads frontmatter files without locking.
   If another process writes a frontmatter file while this function reads it,
   the count could be wrong. However, folio is explicitly a solo-operator tool
   (no concurrent enrich runs expected). The fail-open policy (returns 0 on
   registry failure) provides additional safety. Acceptable for the current
   architecture.

5. **`exclude_doc_id` matching:** Compares `deck_id == exclude_doc_id` against
   registry keys. These are the same namespace (`plan_entry.entry.id`).
   Correct.

No bugs found. The L2 failure policy implementation is thorough.

### D-COD-4: `_enforce_queue_cap` stable sort correctness (informational)

**Severity:** informational

**Question:** Is the stable sort correct?

**Analysis:** The sort key maps `"high"` to 0 and everything else to 1:

```python
key=lambda p: 0 if p.confidence == "high" else 1,
```

Python's `sorted()` is guaranteed stable, so proposals with equal sort keys
retain their original (emission) order. The spec (section 5.3 step 4) requires
"sort by confidence (high before medium), preserving original emission order
for equal confidence." This is satisfied.

One subtlety: the spec mentions only `"high"` and `"medium"` confidence
values. If a proposal somehow had `confidence="low"` (not currently possible
per the validation at emission time), it would be treated identically to
`"medium"`. This is acceptable -- the sort key degrades gracefully.

The function returns `non_queued + queued_sorted` (line 1162). This means the
output order is: all non-queued proposals first, then sorted queued proposals.
This changes the relative ordering of non-queued vs queued proposals compared
to the input. This is acceptable because the output proposals list order is
not semantically meaningful (proposals are identified by `proposal_id`).

No bug.

### D-COD-5: In-place mutation of proposals (should-fix)

**Severity:** should-fix

**Question:** Does the emission flow mutate proposals in-place? Could this
cause aliasing bugs?

**Analysis:** Yes, multiple functions mutate proposals in-place:

1. `_suppress_rejected_proposals` sets `proposal.lifecycle_state = "suppressed"`
   (line 1038) on the input `RelationshipProposal` objects.

2. `_enforce_queue_cap` sets `p.lifecycle_state = "suppressed"` (lines 1155,
   1159) on the input objects.

The `proposals` list is built fresh each enrich run (lines 705-721), so aliasing
across runs is not a concern. Within a single run, the mutation chain is:

```
proposals → _enforce_singular_supersedes (returns new list, no mutation)
          → _suppress_rejected_proposals (mutates lifecycle_state in-place)
          → _remove_promoted_proposals (returns new list, no mutation)
          → _enforce_queue_cap (mutates lifecycle_state in-place)
```

The critical question: does `_remove_promoted_proposals` (step 10b) drop
proposals that were already mutated to `"suppressed"` by step 10a? It could,
because `_remove_promoted_proposals` checks canonical fields regardless of
lifecycle state. A suppressed proposal whose `(relation, target_id)` matches
a canonical field is silently dropped.

This is actually correct behavior: if the relationship is already in canonical
frontmatter, neither the queued nor the suppressed proposal adds value.
However, the suppression count (`rejection_suppressed`) was already incremented
in step 10a, so the `suppression_counts` diagnostic could report a higher
count than the number of suppressed proposals actually in the output.

**Example scenario:** proposal A is marked suppressed by rejection memory
(count = 1), then removed by `_remove_promoted_proposals`. The output reports
`rejection_memory: 1` but contains zero suppressed proposals.

This is a minor diagnostic inaccuracy. The count reflects "proposals the
rejection memory filter acted on" rather than "suppressed proposals in the
final output." The spec (section 7.2) says `suppression_counts` "reports
per-cause suppression totals" which is ambiguous about whether it means
"suppression actions taken" or "suppressed proposals in output."

**Recommendation:** Document that `suppression_counts` reflects actions taken,
not final output counts. Or move `_remove_promoted_proposals` before
`_suppress_rejected_proposals` in the pipeline. This is should-fix for
diagnostic clarity, not a functional bug.

### D-COD-6: `proposals_count` correctness with new flow (should-fix)

**Severity:** should-fix

**Question:** Does `proposals_count` still work correctly?

**Analysis:** Line 937 sets `proposals_count=len(proposals)`, where `proposals`
is the `list[RelationshipProposal]` after all filtering steps (10a-10c) but
BEFORE the preserved rejected dicts are prepended.

The `all_proposal_dicts` list (line 749-751) is:

```python
preserved_rejected + [p.to_dict() for p in proposals]
```

So `proposals_count` counts only new proposals (queued + suppressed), excluding
preserved rejected entries. Meanwhile, the `proposals` field in the
`EnrichAxisResult` contains all of them (preserved rejected + new).

This means `proposals_count` (used for CLI output, line 1857) understates the
total proposal count when there are preserved rejected entries. The spec
(section 7.3) explicitly says: "The count continues to reflect all proposals
in the list (including suppressed and preserved rejected)."

The code does NOT match this spec requirement. `len(proposals)` counts only
the `RelationshipProposal` objects, not the preserved rejected dicts.

**Recommendation:** Change line 937 to `proposals_count=len(all_proposal_dicts)`
or, if the intent is to count only new proposals, document the divergence
from the spec. Per the spec's own caveat that changing `proposals_count`
semantics is a scope expansion issue (section 7.3), this may be intentionally
left as-is, but the code and spec text are in conflict.

### D-COD-7: `existing_meta` staleness window (informational)

**Severity:** informational

**Analysis:** `existing_meta = _get_enrich_meta(fm)` is read at line 726,

The `fm` dict was loaded at the beginning of `enrich_note()` and is not

Both `_suppress_rejected_proposals` (line 729) and `_preserve_rejected_proposals`

modifies `existing_meta`, there is no staleness within a single run.

The `existing_meta` assignment was moved earlier per spec section 6.2 (P-6
fix), which is correct -- it must be read before the enrich block is
overwritten.

No staleness bug.

### D-COD-10: Duplicate rejected entry accumulation over many runs (informational)

**Severity:** informational

**Analysis:** `_preserve_rejected_proposals` returns raw dicts from existing
frontmatter. These are prepended to `all_proposal_dicts` on every run. Since
rejected entries are preserved unchanged and the same entry is read back and
re-preserved on each subsequent run, there is no duplication -- the same dict
is read and written back each time.

However, if an operator rejects the same `(relation, target_id)` with a
different `basis_fingerprint` (e.g., after the LLM re-proposes with changed
evidence, the operator rejects again), the preserved list would contain two
rejected entries for the same key with different fingerprints. The
`_suppress_rejected_proposals` function builds its rejected map with
`rejected[key] = p.get("basis_fingerprint", "")`, so the last rejected entry
wins (dict overwrite). If there are two rejected entries for the same key with
different fingerprints, only the last one's fingerprint is used for suppression.

This is a latent edge case but not currently reachable in a single run (you
can't reject twice in one operation). Over multiple operator rejections across
runs, the behavior is: the most recently preserved rejected entry's fingerprint
is used. This is reasonable -- the latest rejection should be authoritative.

No bug, but worth noting as carry-forward context.
