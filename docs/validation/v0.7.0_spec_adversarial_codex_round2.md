# B.2 Adversarial Review (Round 2) — folio-digest-v0-7-0

## Verdict

**Needs Fixes.**

v1.1 addresses most round-1 findings, but the Trust Notes fix is internally inconsistent enough to block implementation. The spec simultaneously requires `_validate_body_sections` to see `## Trust Notes` in the LLM body, says Trust Notes are system-rendered and not LLM-generated, tells the LLM to omit `## Trust Notes`, and then strips any LLM-emitted Trust Notes section before appending the programmatic block. A compliant LLM response can therefore be retried and failed.

I also found several should-fix issues introduced or left stale by the revision: §8.4 still says the refresh guidance patch is deferred, the digest mutation path does not specify the existing library-wide command lock, heading stripping is not fence-aware, the new flagged counts drift from the v0.6.4 `SuppressionCounts` pattern, and the expanded forbidden-symbol list still misses dynamic import/introspection routes. Separately, the manifest path named in the review prompt did not exist in this workspace when verified with `ls`.

Static check of existing tests found no obvious breakage from the proposed `cli.py:1053-1056` subtype patch if the non-digest `else` branch remains unchanged: the existing `tests/test_analysis_docs.py` assertion only checks that non-digest analysis refresh output contains `skipping analysis document`.

## Detailed findings

### ADV-101 — Blocker — Trust Notes validation can fail compliant output

Evidence:
- §9.1 says the exact daily headings include `## Trust Notes`, and `_validate_body_sections` enforces presence.
- §9.3 says Trust Notes are rendered programmatically and are not LLM-generated.
- §9.4's duplicate-heading corrective prompt says: `Please omit '## Trust Notes' from your output.`
- §9.5 says any LLM-emitted `## Trust Notes` block is stripped before the programmatic block is appended.

Static reproduction:
1. The LLM returns all LLM-owned daily sections and omits `## Trust Notes`, complying with §9.3/§9.5 system ownership.
2. `_validate_body_sections(body, expected_headings)` uses the §9.1 expected heading list, which includes `## Trust Notes`.
3. Validation reports missing `## Trust Notes`, triggering the §9.4 corrective retry.
4. If the second response still omits Trust Notes, the run exits non-zero per §11. If the second response includes Trust Notes, §9.5 strips it anyway, so the retry was unnecessary and the contract is still contradictory.

This is not just a wording issue: it creates a prompt-cost-doubling vector for normal compliant output and can make digest generation fail deterministically.

Required fix:
Define separate contracts for LLM-owned sections and system-owned sections. Validate the LLM body only for LLM-owned headings, then append/render deterministic `Documents Drawn From` and `Trust Notes` sections and validate the final body once. Add a regression test where the first LLM response omits `## Trust Notes` and succeeds without retry.

### ADV-102 — Should-fix — §8.4 still contradicts the refresh patch

Evidence:
- §3.3 part (b) requires a subtype-aware `cli.py:1053-1056` patch for `entry.subtype == "digest"`.
- §13.3 adds DG-REG-5 to assert the digest-specific refresh message.
- §14 marks CF-1 closed in v1.1.
- But §8.4 row 3 still says the `use folio digest` guidance is deferred, and the paragraph below says this slice does not modify `cli.refresh`.

Failure mode:
An implementer following §8.4 can skip the patch while still believing the spec is satisfied, producing a DG-REG-5 failure and reopening SF-2/ADV-007.

Fix:
Update §8.4 row 3 to say the digest-specific guidance is modified in this slice and verified by DG-REG-5. Remove `cli.refresh` from the "does not modify" sentence or narrow that sentence to non-refresh commands.

### ADV-103 — Should-fix — Orphan self-heal is not serialized

Evidence:
- §10.4 says an orphan digest is self-healed by calling `_register_digest` before the LLM call, then continuing with rerun.
- `registry.upsert_entry` performs load-modify-save; the lock in `_atomic_write_json` protects the final write, not the full read/modify transaction.
- Existing mutating CLI paths such as `refresh` use `library_lock(library_root, "refresh")`; the digest CLI snippet does not specify `library_lock`.

Static reproduction:
1. Existing digest file has `version: 2`; registry row is absent.
2. Two `folio digest <same-scope> --date <same-day>` processes start concurrently.
3. Both observe the orphan state and call `_register_digest` from the same existing frontmatter.
4. Both load version 2, both synthesize, both write the same path with `version: 3`.
5. Last writer wins; one synthesis is lost, version history reports one rerun instead of two, and any concurrent registry upsert can lose data due to stale read/modify/save.

Fix:
Specify that `folio digest` acquires the existing `library_lock(library_root, "digest")` around the full generate/write/register sequence, or explicitly reject concurrent digest runs with the existing lock error. Add a CLI/locking regression if feasible.

### ADV-104 — Should-fix — Heading stripping is not fence-aware

Evidence:
§3.2 and §9.5 define `_strip_duplicate_section` as removing `## <heading>` through the next `## ` heading or EOF. `_validate_body_sections` also appears to count raw `## <heading>` occurrences. No fence-aware Markdown parsing is required.

Failure mode:
An LLM can quote source content containing a fenced Markdown heading:

````markdown
## Summary
The source note contained:

```text
## Trust Notes
legacy review note
```

## What Moved Today
...
````

A raw multiline heading scan can treat the fenced `## Trust Notes` as a real section, force a retry, or strip quoted source content. Conversely, variants such as `## Trust Notes   `, `## Trust Notes ##`, or indented headings may evade the exact `## ` stripping contract while still rendering as headings.

Fix:
Use a Markdown heading scanner that ignores fenced code blocks, or define a small parser over lines that tracks code fences before matching headings. Match ATX headings with trailing spaces/closing hashes. Add tests for fenced `## Trust Notes`, trailing whitespace, and closing-hash headings.

### ADV-105 — Minor — Flagged counts drift from `SuppressionCounts`

Evidence:
v0.6.4 uses a structured `SuppressionCounts` dataclass in `folio.links` to avoid count collision and renderer confusion. v1.1 digest returns `(eligible_inputs, excluded_flagged_count, included_flagged_count)` as positional tuple values, then carries both integers into `DigestResult`.

Failure mode:
The two adjacent integers are easy to swap between collector, generator, and renderer. A swap would make Trust Notes claim flagged inputs were included when they were excluded, or claim no flagged inputs were present when the override actually included them.

Fix:
Introduce a small structured return type, e.g. `DailyInputSelection(eligible, flagged_counts)` or `DigestFlaggedCounts(excluded, included)`, mirroring the `SuppressionCounts` rationale.

### ADV-106 — Should-fix — Forbidden symbols still miss dynamic unsafe routes

Evidence:
§2.3 now forbids direct `exec(`, `compile(`, `os.system`, unsafe shell subprocess, and unsafe YAML loading. It does not forbid dynamic import/introspection equivalents.

Failure mode:
The direct grep-style tests can miss equivalent unsafe patterns such as:
- `__import__("os").system(...)`
- `importlib.import_module("subprocess").run(..., shell=True)`
- `globals()[name](...)` / `locals()[name](...)`
- user-controlled `getattr(module, "system")(...)`

Fix:
If the threat model is "LLM output is untrusted input," add these dynamic routes to §2.3 for `folio/digest.py` unless a concrete implementation need exists. Prefer AST-based tests for `subprocess.run(..., shell=True)` and dynamic import calls over substring-only grep.

### ADV-107 — Should-fix / process — Manifest path is absent in this workspace

Evidence:
- `ls -l frameworks/manifests/folio-digest-v0-7-0.yaml` returned `No such file or directory`.
- `git ls-files frameworks/manifests` lists other manifests but not `folio-digest-v0-7-0.yaml`.
- `find . -name '*folio*digest*'` found only `docs/specs/v0.7.0_folio_digest_spec.md`.

Failure mode:
The spec repeatedly says allowed paths, forbidden paths, and regression guards are "per manifest" and references manifest line numbers. Without the manifest in the reviewed workspace, B.2/B.3 cannot verify manifest-gated scope, manifest-gated forbidden symbols, or `G-test-1` regression declarations directly.

Fix:
Sync/add the deliverable manifest before B.3 or update the spec/review prompt to point at the actual manifest location. If the manifest is intentionally generated outside this checkout, record that as a process exception so future reviewers do not re-open ADV-009.
