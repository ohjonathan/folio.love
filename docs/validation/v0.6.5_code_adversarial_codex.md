## Targeted Failure-Mode Review

### 1. Lock coverage

Pass.

- `folio entities suggest-merges` wraps `reg.load()` plus the suggestion reads and disclosure computation inside `with library_lock(library_root, "entities suggest-merges")` at `folio/cli.py:2487-2528`.
- `folio entities reject-merge` wraps `reg.load()`, `reg.reject_person_merge(...)`, and `reg.save()` on real append inside `with library_lock(library_root, "entities reject-merge")` at `folio/cli.py:2567-2591`.
- Error-path release is handled by the `library_lock` context manager's `finally` unlink at `folio/lock.py:90-98`. Early returns and `sys.exit(1)` inside the `with` body still unwind the context manager. Acquisition failure raises before ownership, so there is no new lock to release.

Note: the command name strings are more specific than the spec examples' `"entities"` string. That does not weaken mutual exclusion because all commands share the same `.folio.lock`; the string is only owner metadata in the lock payload.

### 2. `basis_fingerprint` injective encoding

Pass.

`_compute_entity_merge_basis_fingerprint` builds a structured payload with `subject_type`, sorted `entity_keys`, and sorted `reasons`, then serializes with:

```python
json.dumps(payload, sort_keys=True, separators=(",", ":"), ensure_ascii=False)
```

at `folio/tracking/entities.py:443-450`.

The ambiguous delimiter case is safe. `reasons=["a", "b"]` serializes as a JSON array with two string elements, while `reasons=["a|b"]` serializes as a JSON array with one string element. These payloads are byte-distinct before hashing, so the hashes differ. T-4 covers this shape at `tests/test_entities.py:507-519`.

### 3. Revival edge cases with malformed records

Pass.

The rejection index builder in `suggest_person_merges` skips non-dict records and records missing a string `subject_type`, two-item list `entity_keys`, or string `basis_fingerprint` at `folio/tracking/entities.py:1008-1020`. Since skipped records never enter `rejected_index`, they cannot produce either suppression or `revived=True`.

The `revived` flag is set only when a valid prior record exists for the pair and the current fingerprint is not in that pair's valid fingerprint set at `folio/tracking/entities.py:1037-1044`. A malformed record missing `entity_keys` or `basis_fingerprint` therefore does not accidentally revive a candidate.

Non-blocking edge: `count_entity_merge_suppressions()` counts dict-shaped records even if they are semantically malformed (`folio/tracking/entities.py:1066-1071`). That affects only the total-record disclosure, not filtering or revival. It is outside the targeted spec contract for malformed-record revival.

### 4. `reject_person_merge` when pair is not currently suggested

Pass with operator-surprise note.

`reject_person_merge` recomputes current unfiltered suggestions and, if the pair is absent from all heuristics, stores a well-defined rejection with `reasons=[]` and the empty-reasons fingerprint at `folio/tracking/entities.py:1104-1119`.

This is semantically consistent with the v1.2 fingerprint model: the rejection records the current basis. If aliases later change enough to create a heuristic reason, the basis changes and the pair revives rather than being silently suppressed. That is the same revival rule used for ordinary suggestion rejections.

Possible surprise: the CLI prints `Rejected merge candidate` with `reasons: none` even though the pair was not currently a suggestion. I do not consider this a spec violation because the implementation preserves the invariant that only the rejected basis is suppressed.

### 5. Idempotency under basis changes

Pass.

The idempotency check matches exact `(subject_type, sorted entity_keys, basis_fingerprint)` and returns `(False, "already rejected")` only for an exact fingerprint match at `folio/tracking/entities.py:1121-1134`.

If aliases or heuristic reasons shift between rejects, the new fingerprint does not match and a new rejection record is appended at `folio/tracking/entities.py:1136-1144`. That is the correct behavior for revival and re-rejection of a new basis. T-7 covers same-basis idempotency at `tests/test_entities.py:563-573`; T-6b covers multiple fingerprints for the same pair at `tests/test_entities.py:635-665`.
