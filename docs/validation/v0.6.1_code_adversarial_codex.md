## Rationale
The implementation code matches the v1.1 spec on the required paths:

- `folio/pipeline/enrich_data.py` defines the six-member `PROPOSAL_LIFECYCLE_STATES`, maps legacy `"pending_human_confirmation"` to `"queued"` in `_STATUS_TO_LIFECYCLE`, renames the dataclass field to `lifecycle_state`, reads legacy `status` only as a backward-compat fallback, and `to_dict()` emits only `"lifecycle_state"`.
- `folio/enrich.py` emits new proposals with `lifecycle_state="queued"` and the rejection-memory raw-dict filter keeps the `isinstance(p, dict)` guard before every `.get()` call, with an `is None` fallback to legacy `"status"`.
- `folio/links.py` consumes `proposal.lifecycle_state` for rejected/queued filtering and writes `"lifecycle_state"` when rejecting a proposal.
- `folio/graph.py` uses the required raw-dict `lifecycle_state` read with `is None` fallback to legacy `"status"` for rejection counts.

I did not find a missed relationship-proposal raw-dict `status` read in implementation code. Remaining `status` reads are enrich/body axis status, graph producer health status, CLI/provenance namespace handling, or the intentional legacy fallbacks in `enrich_data.py`, `enrich.py`, and `graph.py`. `folio/provenance.py` is completely untouched by the diff (`git diff --quiet main..HEAD -- folio/provenance.py` exit 0), as required by spec §3.6.

Direct verification passed for the requested subset:

```text
python3 -m pytest tests/test_enrich_data.py tests/test_enrich.py tests/test_links_cli.py tests/test_graph_cli.py -q
```

The only blocker to approval is the stale proposal-status assertion outside that requested subset. It should be fixed before this review is approved because it preserves an old-field regression guard that no longer guards the renamed schema.
