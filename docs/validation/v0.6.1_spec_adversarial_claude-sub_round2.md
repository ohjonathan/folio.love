## New Issue Scan (v1.1)

Checked for new issues introduced by the v1.1 revisions:

1. **Section 3.5 raw-dict pattern and section 4.2/4.4 implementation consistency.** The `is None` guard in section 3.5 matches sections 4.2 and 4.4 implementation templates. No inconsistency.

2. **Section 2.2 scope-expansion justification for graph.py.** The justification cites section 15.7 of the parent spec. The reasoning is sound: `graph.py` line 320 reads raw proposal dicts from the relationship-proposal frontmatter path; omitting it would silently break rejection counting. Verified against source: `graph.py` lines 317-322 iterate `proposals` from `axes.get("relationships")`, which is the relationship-proposal namespace.

3. **Section 3.6 namespace-separation claim.** Verified against source. `ProvenanceProposal` dicts are stored under `pair.proposals[]` within provenance-pair metadata. `RelationshipProposal` dicts are stored under `_llm_metadata.<producer>.axes.relationships.proposals[]`. No cross-reading occurs. The claim is accurate.

4. **Section 4.2 inline code template.** The inline form `(p.get("lifecycle_state") if p.get("lifecycle_state") is not None else p.get("status"))` calls `p.get("lifecycle_state")` twice. This is a minor inefficiency but not a correctness issue -- the value is read from a dict (deterministic). The expanded form immediately below avoids this. This is a style preference, not a defect.

5. **Test T-7 coverage.** New test `test_from_dict_neither_key_defaults_to_queued` covers the case where neither `lifecycle_state` nor `status` is present in the input dict. This exercises the default path `d.get("status", "pending_human_confirmation")` followed by `_STATUS_TO_LIFECYCLE` mapping. Good addition.

6. **Test T-8 coverage.** New test `test_graph_acceptance_rate_reads_legacy_rejected_status` verifies that `graph.py` acceptance-rate aggregation counts legacy `"status": "rejected"` proposals correctly after the rename. This closes the gap for the scope-expanded file.

7. **Section 4.3 links.py line 371 write.** The spec renames the write from `raw_copy["status"]` to `raw_copy["lifecycle_state"]`. This is correct -- once `to_dict` emits `lifecycle_state`, writes to in-memory proposal dicts should also use the new key. No backward-compat concern here because writes create new state (not reading legacy data).

No new blockers, should-fix, or minor issues found.

**Approve**

## Rationale

All nine Round 1 findings are closed:

- **Blockers A-1 and A-2** are resolved by the explicit provenance exclusion with namespace-separation justification (section 3.6) and the PROV-1 carry-forward (sections 2.3, 8). Source verification confirms the two proposal types occupy separate frontmatter paths and never flow through the same read/write code.

- **Should-fix items A-3 through A-6** are all addressed: the raw-dict pattern uses `is None` consistently (A-3), unknown-value pass-through is explicitly acknowledged and justified (A-4), the `isinstance` guard ordering is correct in all code templates (A-5), and test T-3b covers the `lifecycle_state=None` edge case (A-6).

- **Minor items A-7 through A-9** are corrected: the grep-audit claim is now accurate (A-7), caller values for `new_status` are documented (A-8), and the CHANGELOG no longer claims a breaking change (A-9).

No new issues were introduced by the v1.1 revisions. The spec is complete, internally consistent, and implementable.
