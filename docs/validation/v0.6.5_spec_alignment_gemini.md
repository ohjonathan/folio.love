# Alignment Review: v0.6.5 Entity-Merge Rejection Memory Spec

## Validation Findings

### 1. Scope Narrowing from §15.6
**Pass.** The spec explicitly implements the first half of Tier 4 §15.6 ("Slice 6a — entity-merge rejection memory") and defers the second half ("shared-consumer expansion"). This is a valid and well-defined narrowing of the shipping plan without reinterpreting its goals.

### 2. Scope Expansion to cli.py and graph.py
**Pass.** The expansion to `folio/cli.py` and `folio/graph.py` is justified per parent §15.7. Since these modules are the primary display surfaces and consumers of `suggest_person_merges()`, failing to update them would result in rejected merges continuing to be surfaced to the operator, defeating the purpose of the feature.

### 3. No-EntityMergeProposal-Dataclass Design Decision
**Pass.** The decision to avoid introducing an `EntityMergeProposal` dataclass is consistent with parent §5. Parent §5 governs LLM-emitted proposal objects that undergo a full lifecycle. Since the current entity-merge candidates are deterministic heuristic suggestions computed on the fly, layering rejection records directly over the heuristic surface cleanly satisfies the rejection-memory requirement without over-engineering an unused proposal state machine. This design properly defers the full proposal object until an LLM producer for entity-merges is introduced.

### 4. `basis_fingerprint` Derivation (§3.2)
**Pass.** The fingerprint derivation in §3.2 (`subject_type`, sorted `left_key` and `right_key`, and sorted `reasons`) accurately adheres to the parent §7 clause. It correctly applies the "lexicographically-sorted set of candidate entity IDs together with `subject_type`" rule, and effectively uses the `reasons` list as the "supporting input state" to allow for proposal revival if the underlying entity aliases (and thus the heuristics) materially change.

### 5. Deferrals Matching the Shipping Plan
**Pass.** The deferrals listed in §8 align with the constraints and sequencing of the parent Shipping Plan. Deferring the shared-consumer expansion, LLM-emitted proposals, and organization/product merges logically follows the current state of the codebase and limits the slice to a cohesive, testable increment.
