# Product Review — proposal-review-hardening-v0-6-0 / D.1 / claude-sub

## Verdict

**Approve.** The six user-facing surfaces at Phase C commit `3cc99be` render exactly as spec v1.3 §4.3 / §4.4 specifies; copy, ordering, empty-state, threshold disclosure, JSON schema, and CHANGELOG breaking-change notice all match. No blockers; two minors noted.

## 1. User-value assessment

The target user is the operator ("Johnny") who runs `folio links review` mid-engagement to triage LLM-generated relationship proposals. The pre-v0.6.0 pain: once Johnny rejected a proposal, the next enrich run re-emitted it and he saw it again in the very next `review`. That is the exact proxy failure the rejection-memory filter eliminates. The complementary diagnostic, `folio graph doctor`'s "Producer acceptance rates" section, extends the same triage instinct: Johnny can now see which producer is generating the noise he keeps rejecting, without waiting for a silent gate to throttle it. The spec correctly flags the slice as diagnostic-only in v0.6.0 — promoting to gating would be a policy change Johnny doesn't yet have an override for, and the copy ("Status column is informational only in v0.6.0; `low-acceptance` producers continue to surface proposals at normal priority in this slice.") makes that tradeoff visible. User-value claim holds: Johnny's observable rejection-loss symptom is gone, and his diagnostic visibility into producer health is a clean additive gain.

## Scope calibration

Diagnostic-only in v0.6.0 is the correct calibration. Three reasons:

1. Spec §2 out-of-scope row 2 and the scope-disclosure footer both name this explicitly — no operator will be surprised.
2. Gating on `low-acceptance` without an `--include-flagged` override would create a dead-end state Johnny can't escape. The slice correctly defers gating to v0.7.x trust-gating.
3. The warmup threshold (10 reviewed) is conservatively low; producers won't sit in warmup for meaningful engagements. This prevents a "diagnostic is always warmup → operators never see acceptance rates" failure mode.

The breaking change on `--json` shape is correctly scoped: no known scripted consumers per CHANGELOG `### Notes`, and the migration path ("extract `findings` from the new top-level object") is exactly one line of code for any consumer. Pre-1.0 minor-version break is proportionate.

The pipe-table syntax for the acceptance-rate table (`| ... | ... |`) and right-aligned numeric columns (`|---:|---:|`) match the pattern used in `links_status_cmd` (`folio/cli.py:1643–1648`) and other tabular surfaces. The check/cross glyphs (`✓` / `✗`) are used by `confirm` / `reject` subcommands but intentionally absent from `review` and `doctor` (these are read-only diagnostic surfaces, not action surfaces) — correct idiomatic use.

This is an internal API boundary violation by Python convention but matches the spec's "`graph_doctor` returns data, `cli.py` renders" decision (§4.4). A follow-up slice could rename to public or introduce a façade — noted as minor architectural nit; not a product-lens finding.
- Positive observation: the test suite explicitly asserts the exact user-facing copy strings (`tests/test_graph_cli.py:608–609`, `625–626`), including the scope-disclosure footer and empty-state line. Copy-drift regressions will fail tests, not silently corrupt UX.
