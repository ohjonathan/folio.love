## Blockers (DCB-NNN)

### DCB-001 - Library-wide diagnose silently false-cleans corrupt or unreadable registry

**Category:** failure-policy regression / false-clean diagnostic

**Evidence:** The approved spec requires registry and I/O errors to be fatal (`docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:312`) and explicitly says a corrupt registry exits 1 (`docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:1250`, `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:1256`). The implementation only runs `_resolve_scope_or_raise()` when `scope is not None` (`folio/enrich.py:2123`), while the corrupt-registry check lives inside that helper (`folio/enrich.py:1973`). With no scope, `diagnose_notes()` falls through to `plan_enrichment()`, which rebuilds corrupt registry data under dry-run instead of raising (`folio/enrich.py:254`, `folio/enrich.py:256`).

**Direct reproduction:**

```bash
python3 - <<'PY'
import json
import tempfile
from pathlib import Path
import yaml
from click.testing import CliRunner
from folio.cli import cli

root = Path(tempfile.mkdtemp()) / "lib"
root.mkdir()
(root / "registry.json").write_text(json.dumps({"_corrupt": True, "decks": {}}))
config_path = root / "folio.yaml"
config_path.write_text(yaml.dump({
    "library_root": str(root),
    "llm": {
        "profiles": {"default": {"name": "default", "provider": "anthropic", "model": "test"}},
        "routing": {"default": {"primary": "default"}, "enrich": {"primary": "default", "fallbacks": []}},
    },
}))
result = CliRunner().invoke(cli, ["--config", str(config_path), "enrich", "diagnose"])
print("exit", result.exit_code)
print(repr(result.output))
PY
```

Observed:

```text
exit 0
'Registry bootstrapped: 0 entries\nNo enrichment hygiene findings.\n'
```

The same bug reproduces with an unreadable registry file (`chmod 000 registry.json`): observed exit 0 with "Permission denied ... marking corrupt", "Registry bootstrapped: 0 entries", and "No enrichment hygiene findings."

**Expected:** Exit 1 with a fatal registry error, not a healthy empty report.

**Proposed fix:** Make registry readability/corruption preflight independent of scope. Before calling `plan_enrichment()`, if `registry.json` exists and `registry_mod.load_registry()` returns `_corrupt`, raise `ScopeResolutionError`. Better: add a `plan_enrichment(..., fail_on_corrupt_registry=True)` or shared loader path for diagnose so the second registry read cannot silently rebuild after a successful preflight.

### DCB-002 - Package metadata still reports v0.9.0, not the contracted v1.0.0 release

**Category:** release identity / packaging regression

**Evidence:** The implementation contract says this slice targets v1.0.0 and specifically calls out the pyproject version bump (`docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:297`). The changelog declares v1.0.0 (`CHANGELOG.md:4`, `CHANGELOG.md:8`). Actual package metadata remains `version = "0.9.0"` (`pyproject.toml:7`).

**Direct reproduction:**

```bash
python3 -c 'import pathlib, tomllib; data=tomllib.loads(pathlib.Path("pyproject.toml").read_text()); print(data["project"]["version"])'
```

Observed:

```text
0.9.0
```

**Expected:** `1.0.0`.

**Proposed fix:** Set `[project].version = "1.0.0"` in `pyproject.toml` and add a small release-identity gate so the manifest/changelog/spec version cannot drift from packaging metadata again.

## Should-fix (DSF-NNN)

### DSF-001 - Truncation footer cannot report the promised unfiltered count

**Category:** spec/code drift / misleading CLI output

**Evidence:** The spec promises `(showing N of M; raise --limit to see more)` when output is truncated (`docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:310`). The JSON contract also says `summary.total` is post-truncation (`docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:1125`), so `_render_diagnose_text()` cannot derive `M`; it sets `unfiltered = result.summary.total` and prints "showing N of more" (`folio/cli.py:1326`).

**Direct reproduction:** A library with 3 findings and `--limit 1` produced:

```text
exit 0
... (showing 1 of more; raise --limit to see more)
1 findings: 1 warning.
```

**Proposed fix:** Preserve `unfiltered_count` on `DiagnoseResult` for renderer use, or change the approved text contract to avoid claiming `M`. If the JSON envelope must remain unchanged, keep the extra count internal and do not render it into JSON.

### DSF-002 - Suggested shell commands interpolate unquoted user-controlled scope or subject IDs

**Category:** security / copy-paste command injection

**Evidence:** The enrich breadcrumb interpolates `scope` directly into a backtick-wrapped command (`folio/cli.py:1250`). Recommended actions do the same with `subject_id` (`folio/enrich.py:2050`, `folio/enrich.py:2062`). There is no shell execution inside Folio, but these strings are explicitly presented as commands to copy.

**Direct reproduction:**

```text
scope = 'ClientA`touch /tmp/folio_pwned`'
observed output:
  Tip: run `folio enrich diagnose ClientA`touch /tmp/folio_pwned`` to see why these notes stalled.
```

**Proposed fix:** Use `shlex.quote()` for scope and subject IDs inside command suggestions, or stop rendering exact shell snippets and instead say "run diagnose for this scope" without embedding untrusted tokens.

### DSF-003 - Missing `library_root` reports a healthy empty library

**Category:** failure-policy edge case

**Evidence:** With a config pointing at a nonexistent library root, library-wide diagnose exits 0 and prints "Registry bootstrapped: 0 entries" plus "No enrichment hygiene findings." That is indistinguishable from a healthy empty library and conflicts with the spec's fatal config / I/O error posture (`docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:312`). The path comes from `config.library_root.resolve()` (`folio/enrich.py:250`, `folio/enrich.py:1969`) and `rebuild_registry()` tolerates nonexistent roots by returning an empty registry.

**Direct reproduction:** A temp config with `library_root` set to a path that did not exist produced:

```text
exists_before False
exit 0
output 'Registry bootstrapped: 0 entries\nNo enrichment hygiene findings.\n'
exists_after False
```

**Proposed fix:** For diagnose, fail fast if `config.library_root` does not exist or is not a directory. If bootstrap-on-missing-root is intentionally supported elsewhere, keep that behavior out of the diagnostic health check.

## Minor (DMIN-NNN)

### DMIN-001 - Legacy `--` option terminator no longer works for `folio enrich -- ClientA`

**Category:** parser edge-case regression

**Evidence:** The parser treats literal `--` as an option-like token (`folio/cli.py:73`) and then rewrites the following scope token into `--scope` after the option terminator. Click then sees no group argument and exits with usage error.

**Direct reproduction:** Against the actual `ScopeOrCommandGroup` with `enrich_batch` patched to avoid LLM calls:

```text
['enrich', '--', 'ClientA'] exit 2
Usage: cli enrich [OPTIONS] COMMAND [ARGS]...
```

The canonical 10 v1.3 patterns pass, so this is not a CB-2 blocker. It is still a narrow backward-compatibility loss versus a flat Click command with a positional scope.

**Proposed fix:** Special-case literal `--` in `ScopeOrCommandGroup.parse_args()` for group-body legacy invocation, or document that the custom scope/subcommand disambiguator does not support `--` before a legacy scope.
