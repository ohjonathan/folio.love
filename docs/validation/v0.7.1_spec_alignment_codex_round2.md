# Alignment Review — folio-graph-generalized-proposals-v0-7-1 / B.2 / codex-as-alignment

## 1. Architecture compliance

v1.1 materially fixes the B.1 alignment blockers. The spec now treats `folio graph` as the in-scope §15.6 consumer, explicitly defers `folio synthesize` and `folio search`, narrows §7 to a legacy alias instead of claiming parent-complete fingerprint implementation, and narrows §8 to partial default-row rendering instead of full review-rendering implementation.

No blocking architecture violation was found. The remaining concerns are authority-wording issues: v1.1 cites §15.8 as "revision authority" for a future §7 parent-spec update even though §15.8 is specifically about revising the Shipping Plan sequencing, and v1.1 uses MUST language to bind the future synthesize sub-slice to an acceptance criterion. Both are explicit, not silent over-claims, so they are should-fix rather than blockers.

## 2. Diagram-architecture cross-reference

No diagrams are present in the v0.7.1 spec, and none are required for this closure-verification review. No diagram/prose mismatch was found.

## 3. Roadmap alignment

The deliverable is still aligned to parent §15.6 shared-consumer expansion: this slice handles `folio graph`, while `folio synthesize` and `folio search` remain future sub-slices. The manifest header continues to identify the deliverable as §15.6 sub-slice 1 of 3 and cites the inherited parent anchors by number and name.

The added `proposal_id` key does not reinterpret §5 because v1.1 explicitly labels it "not a §5 contract field" and uses it for backward-compatible operator identity. The `schema_gate_result` shape is also acceptable as a slice-defined implementation detail because parent §5 names the field but does not prescribe a wire shape.

## 5. Backward compatibility

v1.1 now handles the `subject_id` compatibility issue honestly. It declares that `subject_id` remains present but changes value to `None` for relationship proposals, adds a top-level `proposal_id` to preserve the prior operator-facing identifier, updates CLI stdout to read `proposal_id`, and supplies CHANGELOG language for JSON consumers.

The `proposal_id` key is an extra compatibility field, not a §5 field. Because §5 defines the minimum shared contract rather than a closed object shape, this is acceptable as long as future proposal consumers do not treat `proposal_id` as a substitute for any required §5 row.

## 6. Consistency check

R1 A-1, A-2, A-3, SF-1, SF-2, SF-3, and MN-1 are closed on spec v1.1. The narrowing language is internally consistent across §1, §2.1, §2.2, §3.1, §7 references, and the CHANGELOG block.

Two consistency issues remain. First, the §15.8 reference should be tightened so it does not imply that §15.8 directly authorizes a semantic rewrite of §7. Second, the future synthesize helper-promotion requirement should be recorded as a carry-forward requirement for the sub-slice 2 author/manifest, not as a current-slice MUST that preempts that future slice's own narrowing authority.
