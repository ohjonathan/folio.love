## Findings

### D-ADV-001 - Blocker - `confirm-doc` / `reject-doc` silently hide flagged exclusions in mixed documents

The bulk commands compute `flagged_excluded`, but the CLI only discloses it when
`acted == 0`.

Evidence:

- `confirm_doc()` returns `(acted, counts.flagged_input)` from
  `collect_pending_relationship_proposals()` (`folio/links.py:551-577`).
- `reject_doc()` does the same (`folio/links.py:580-598`).
- `links_confirm_doc_cmd()` prints the flagged diagnostic only under
  `if acted == 0 and flagged_excluded > 0` (`folio/cli.py:1781-1785`).
- `links_reject_doc_cmd()` has the same condition (`folio/cli.py:1812-1816`).

This means a document with one clean proposal and one flagged-input proposal
reports a clean success and suppresses the flagged skip:

```text
exit 0
✓ Confirmed 1 proposal(s)
```

The same direct probe for `reject-doc` reports:

```text
exit 0
✓ Rejected 1 proposal(s)
```

No line tells the operator that one proposal was excluded because source or
target has `review_status: flagged`.

Why this matters:

- v1.1 says all surfaces must disclose flagged exclusions and no surface may
  silently report incomplete results when flagged inputs are the cause
  (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:36-50`).
- CB-2 was explicitly about status / confirm-doc / reject-doc disclosing
  flagged-excluded counts.
- The all-flagged case is covered, but the mixed case is the more dangerous
  operator path: the command succeeds, so the operator has less reason to
  inspect for skipped proposals.

Required fix:

- Print the flagged-excluded diagnostic whenever
  `flagged_excluded > 0 and not include_flagged`, regardless of `acted`.
- Add mixed clean+flagged regression tests for both `confirm-doc` and
  `reject-doc`.

### D-ADV-SF-001 - Should-fix - `links status --include-flagged` is missing from the CLI and function signature

The v1.1 contract lists `status` among the committed surfaces that provide an
explicit `--include-flagged` override:

- Rule table: `review`, `status`, `confirm`, `reject`, `confirm-doc`,
  `reject-doc` (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:42-50`).
- CLI spec repeats the same command list
  (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:299-309`).
- `relationship_status_summary()` is specified with
  `include_flagged: bool = False`
  (`docs/specs/v0.6.4_trust_gated_surfacing_spec.md:237-243`).

Implementation drift:

- `relationship_status_summary()` accepts only `config` and `scope`
  (`folio/links.py:314-318`).
- `links_status_cmd()` exposes only positional `scope`; there is no
  `@click.option("--include-flagged")` (`folio/cli.py:1661-1669`).

Direct probe:

```text
ARGS links status --help
exit 0
Usage: cli links status [OPTIONS] [SCOPE]

  Summarize canonical and pending document-level relationships.

Options:
  --help  Show this message and exit.

ARGS links status --include-flagged
exit 2
Error: No such option: --include-flagged
```

This is not a consent bypass by itself: status is read-only and currently
returns source-level rows, not proposal IDs. But it is a contract miss on a
named committed surface. Either implement the flag and define whether flagged
proposals move from `Flagged Excluded` into `Pending`, or amend the spec to
state that `links status` is disclosure-only and points operators to
`links review --include-flagged`.

### D-ADV-SF-002 - Should-fix - S4-11 only covers one stale-registry direction

CB-1 required target review status to come from target frontmatter, not the
registry, with tests for both directions of staleness.

The current S4-11 test only covers:

- registry target row has no `review_status`
- target frontmatter has `review_status: flagged`
- expected result: proposal is filtered

Evidence: `tests/test_links_cli.py:1144-1181`.

Missing direction:

- registry target row says `review_status: flagged`
- target frontmatter says `review_status: clean`
- expected result: proposal surfaces and `flagged_inputs == []`

Static inspection of `_resolve_target_flagged()` suggests the implementation is
correct because it ignores the registry's `review_status` and reads the target
markdown frontmatter (`folio/links.py:90-117`). The gap is test coverage, not
an observed implementation failure.

### D-ADV-SF-003 - Should-fix - CB-4 tests do not fully exercise the reviewed bypass workflow

S4-17 verifies that `confirm <flagged_id>` without `--include-flagged` fails
when the test already knows the generated ID (`tests/test_links_cli.py:1267-1285`).
That is useful, but it does not exercise the full bypass workflow that drove
CB-4:

1. operator runs `links review --include-flagged`
2. output discloses a flagged proposal ID
3. operator tries `links confirm <that_id>` without `--include-flagged`
4. command refuses
5. operator retries with `--include-flagged` and succeeds

Related gaps:

- No equivalent single-proposal `reject` tests for without/with
  `--include-flagged`.
- No bulk `confirm-doc --include-flagged` or `reject-doc --include-flagged`
  test proves that the propagated flag actually acts on flagged-input
  proposals.
- S4-16 covers only `confirm-doc` default-mode all-flagged disclosure; there is
  no `reject-doc` disclosure test and no mixed clean+flagged bulk test, which is
  how D-ADV-001 escaped.

### D-ADV-I-001 - Informational - Target frontmatter authority is implemented on the reviewed code paths

The implementation reads target trust state from target frontmatter in the
shared collection path:

- `_resolve_target_flagged()` loads the target entry only to find
  `markdown_path`, then calls `registry_mod._read_frontmatter(target_path)`
  and checks `review_status` from that dict (`folio/links.py:90-117`).
- `collect_pending_relationship_proposals()` uses `_resolve_target_flagged()`
  for every queued proposal after rejection-memory suppression
  (`folio/links.py:251-278`).
- `_find_pending_view()` delegates to `collect_pending_relationship_proposals()`
  and passes `include_flagged` through (`folio/links.py:441-453`).
- `confirm_proposal()` and `reject_proposal()` delegate to `_find_pending_view()`
  (`folio/links.py:500-548`).
- `confirm_doc()` and `reject_doc()` also collect through the same path
  (`folio/links.py:551-598`).

I did not find a remaining code path that authorizes a proposal using target
`review_status` from the registry.

### D-ADV-I-002 - Informational - `_resolve_target_flagged` does not leak cache state across collection calls

The target flagged cache is local to a single call:

```python
target_flagged_cache: dict[str, bool] = {}
```

It is created inside `collect_pending_relationship_proposals()`
(`folio/links.py:203-216`) and passed into `_resolve_target_flagged()` per
proposal (`folio/links.py:268-270`). There is no module-level cache and no
default mutable argument.

Per-call staleness is possible if files change during one collection pass, but
that is expected for this slice. The failure mode requested in the prompt,
global cross-call state leakage, is not present.

### D-ADV-I-003 - Informational - `relationship_status_summary()` does not create a consent bypass

`relationship_status_summary()` does call
`collect_pending_relationship_proposals(..., include_flagged=True)` internally
(`folio/links.py:329-333`), but it only returns aggregate
`RelationshipStatusRow` objects with `source_id`, `pending`, `confirmed`, and
`flagged_excluded` (`folio/links.py:52-58`, `314-369`).

The CLI status output does not print proposal IDs (`folio/cli.py:1681-1699`).
An operator cannot copy a flagged proposal ID from `links status` and confirm
it without having first obtained the ID elsewhere. The missing status flag is
tracked separately in D-ADV-SF-001.

### D-ADV-I-004 - Informational - `SuppressionCounts.total()` is not production-used, but it is not harmful dead code

`SuppressionCounts.total()` exists in the dataclass (`folio/links.py:36-50`) and
is exercised by the S4-12 test (`tests/test_links_cli.py:1184-1194`).
Production renderers currently use the two fields independently:

- rejection-memory total: `sum(counts.rejection_memory.values())`
  (`folio/cli.py:1646`)
- flagged exclusions: `counts.flagged_input` (`folio/cli.py:1653-1658`)

That is the right rendering behavior because the two reasons should not be
conflated. The method is future-proofing for callers that need a grand total,
not a current bug.

### D-ADV-I-005 - Informational - Other checked failure modes look correct

- Rejection-memory filtering still precedes flagged-input filtering
  (`folio/links.py:257-278`), so proposals that match rejection memory are
  counted under `counts.rejection_memory`, not `counts.flagged_input`.
- `links review --include-flagged` does not trigger the silent-empty
  flagged-exclusion header because included flagged proposals are surfaced as
  views and `counts.flagged_input` remains zero (`folio/links.py:276-278`,
  `folio/cli.py:1621-1627`).
- A producer literally named `flagged_input` no longer collides with the
  trust-gate counter because producer counts live under
  `counts.rejection_memory` and trust-gate counts live under
  `counts.flagged_input` (`folio/links.py:36-50`, `folio/cli.py:1646-1658`).
- Malformed or unreadable target frontmatter fails open as specified:
  `registry_mod._read_frontmatter()` returns `None` for invalid frontmatter
  (`folio/tracking/registry.py:427-450`), and `_resolve_target_flagged()`
  requires `isinstance(target_fm, dict)` before reading `review_status`
  (`folio/links.py:114-116`).

## Verdict

Needs Fixes.

No target-frontmatter authorization bypass was found in the core
confirm/reject lookup paths, and the cache is correctly scoped per collection
call. The blocker is the mixed bulk-command disclosure hole: `confirm-doc` and
`reject-doc` can partially act while silently omitting flagged-input proposals.
That violates the v1.1 trust-gated surfacing contract and should be fixed before
approval.
