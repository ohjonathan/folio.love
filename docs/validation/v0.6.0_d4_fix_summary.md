## Spec deviations declared

**DB-3: `_aggregate_producer_acceptance_rates` signature**

- **Blocker addressed:** DB-3
- **Spec section affected:** `docs/specs/v0.6.0_proposal_review_hardening_spec.md` §4.3 ("New helper" paragraph specifies `_aggregate_producer_acceptance_rates(config: FolioConfig) -> list[ProducerAcceptanceRate]`)
- **Deviation:** Implementation uses signature `_aggregate_producer_acceptance_rates(config: FolioConfig, *, scope: Optional[str] = None) -> tuple[list[ProducerAcceptanceRate], int]`. The second tuple element is the `missing_producer_count` diagnostic from §4.3's missing-producer-handling paragraph.
- **Authority:** Orchestrator directive. The user directed continuation of Phase C implementation ("Then let's do it"); the fix-author is exercising reasonable design judgment within that scope. The alternative (module-level counter, or smuggling `missing_producer_count` into the result list via a sentinel entry) was judged worse-designed than the explicit tuple return. A `scope: Optional[str]` kwarg is a non-behavioral extension aligned with `graph_status` and `graph_doctor`'s own existing `scope` kwargs.
- **Knock-on effects:** `folio/cli.py:~1768` destructures the tuple as `rates, missing_producer_count = _aggregate_producer_acceptance_rates(config, scope=scope)`. No other spec clauses are affected. The spec's §4.4 JSON schema already anticipates the `missing_producer_count` — it appears as `producer_acceptance_rates_data_integrity.missing_producer_count`. The implementation simply plumbs that count through the return tuple instead of a globals-backed counter, which matches §4.3's "no hidden module-level state" rationale.
