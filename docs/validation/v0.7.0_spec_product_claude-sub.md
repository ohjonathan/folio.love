# B.1 Product Review — folio-digest-v0-7-0

## Summary

The spec's overall operator surface is small and coherent — one new command group, a four-branch trust-notes matrix, two distinct empty-result paths, and a handful of stdout success lines. The architectural choices (programmatic Trust Notes, deterministic wikilink section, atomic write, exit 0 on empty) all read correctly for the sole-operator profile. However, applying the same skepticism slice 6a's product reviewer applied (the PR-SF-1..4 pluralization / context-completeness / verifiability / silent-acceptance pattern), this spec carries forward several of the same operator-experience defects:

1. **Pluralization regression on the success line.** "Drawn from N input(s)." reuses the bracket-suffix pattern slice 6a's PR-SF-1 explicitly flagged as wrong — the operator running a single-input rerun sees "Drawn from 1 input(s)." which the prior review's three-branch fix was specifically meant to eliminate.
2. **The "Updated daily digest: X (version 3 → 4)" rerun line lacks a success indicator** (no `✓` prefix) while every other CLI write surface uses one (cli.py:357, 457, 1110, 1360, 2607, 2712, 2764). Operators scanning a terminal for "did this succeed" rely on that visual marker.
3. **`--week --include-flagged` accepts silently** with no operator-visible feedback that the flag is a no-op, the same category of silent-acceptance issue slice 4's PR-SF-3 flagged as a should-fix.
4. **Empty-result message for "no inputs at all" lacks the `--include-flagged` no-op clarification.** When the operator runs `folio digest <scope> --include-flagged` and gets the "No eligible inputs" message because there genuinely are none, they cannot tell whether their override was even consulted. Symmetric problem to PR-SF-3 of slice 4.
5. **Trust Notes copy uses always-plural `source-backed input(s)`** in the disclosure-line column — same pluralization bug as the success line, in user-visible markdown that ships into the digest body.
6. **CF-1's discoverability story is wrong.** The spec assumes operators learn the rerun path via the message; in reality, `cli.refresh` already emits a generic "rerun the originating analysis workflow instead" line (cli.py:1054-1056) which doesn't name `folio digest`. Either fix the message in this slice, or the spec must commit to a discoverability surface (e.g., `folio digest --help` mentions it under examples).

These are all should-fix items (no blockers). The core design — predicate, ordering, atomicity, registry contract, body template — is correct and tracks slice 6a's lessons cleanly in those areas. Verdict: **Needs Fixes** (5 should-fix; 2 minor; 2 informational).

## Detailed findings

### PROD-SF-1 (should-fix): Pluralization bug on `Drawn from N input(s).` and Trust Notes disclosure line

**Spec stdout examples (§4):**

```
Drawn from 7 input(s).        # OK for N>1
Drawn from 8 input(s).        # OK for N>1
Drawn from 1 input(s).        # WRONG for N=1
```

**Spec Trust Notes branch 2 (§9.3):**

```
Daily input selection excluded N source-backed input(s) with review_status: flagged. Run with --include-flagged to widen the input set.
```

This `input(s)` parenthetical-suffix shortcut is exactly the pattern slice 6a's PR-SF-1 product reviewer flagged as a should-fix and slice 6a v1.2 fixed by adopting a three-branch ladder (`No / 1 / N`). The fixed slice-6a code lives at `folio/cli.py:2536-2541`:

```python
if suppressed_count == 0:
    click.echo("No merge candidates suppressed by rejection memory.")
elif suppressed_count == 1:
    click.echo("1 merge candidate suppressed by rejection memory.")
else:
    click.echo(f"{suppressed_count} merge candidates suppressed by rejection memory.")
```

Slice 1's `links_review_cmd` followed the same pattern (cli.py:1647-1652, 1653-1658). The current spec regresses to the always-plural form for both the daily success line and the Trust Notes branch 2 disclosure.

The Trust Notes case is *worse* than the stdout case because the bug ships into the *digest body file itself* — the operator (and any future reader of the markdown artifact) sees `excluded 1 source-backed input(s)` as a permanent record, not a transient terminal line.

**Recommended rewrite for §4 success line:**

```
N == 0: (not applicable — the empty-result paths are separate; success line is only emitted when N >= 1)
N == 1: "Drawn from 1 input."
N >  1: "Drawn from {N} inputs."
```

**Recommended rewrite for §9.3 Trust Notes branch 2:**

```
N == 1: "Daily input selection excluded 1 source-backed input with review_status: flagged. Run with --include-flagged to widen the input set."
N >  1: "Daily input selection excluded {N} source-backed inputs with review_status: flagged. Run with --include-flagged to widen the input set."
```

(Branches 1 and 3 don't need pluralization; branch 4 is impossible.)

Also affects the empty-due-to-flagged stdout line in §4: `3 source-backed input(s) excluded` → spec must commit to the same three-branch wording. (For N=0, this branch can't fire either — it's gated by `excluded_flagged_count > 0`.)

The spec should add a one-line implementation note in §4 (and §9.3) directing the implementer to follow the slice 1 / slice 6a three-branch pattern verbatim, with a link to `_render_merge_disclosure` (cli.py:2531-2543) as the reference style.

### PROD-SF-2 (should-fix): Success / update lines lack `✓` prefix

**Spec stdout examples (§4):**

```
Wrote daily digest: usbank_techresilience2026_analysis_20260404_daily-digest
Updated daily digest: usbank_techresilience2026_analysis_20260404_daily-digest (version 3 → 4)
```

**Existing folio CLI convention (verified across cli.py):**

| Line | Surface |
|---|---|
| `cli.py:357` | `✓ {Path(source).name}` (convert) |
| `cli.py:457` | `✓ {source.name}` (ingest) |
| `cli.py:582,619,1110` | `✓ {f.name}` and `✓ {entry.id}` (batch convert) |
| `cli.py:1360` | `✓ Confirmed {proposal_id} -> {link_id}` |
| `cli.py:1378,1452,1576` | `✓ Rejected ...`, `✓ Removed ...` etc. |
| `cli.py:1744,1776` | `✓ Confirmed {proposal_id}: ...`, `✓ Rejected {proposal_id}: ...` |
| `cli.py:2607` | `✓ Rejected merge candidate: ...` |
| `cli.py:2712,2764` | `✓ Created context document: ...`, `✓ Created analysis document: ...` |

Every write/mutate surface uses the `✓` prefix. The digest spec is the only outlier. For Jonathan's terminal-scanning workflow ("did the command actually do the thing"), the `✓` is the load-bearing visual signal.

**Recommended rewrite for §4 stdout examples:**

```
# Successful daily, fresh write
✓ Wrote daily digest: usbank_techresilience2026_analysis_20260404_daily-digest
  Path: usbank/techresilience2026/analysis/digests/...
  Drawn from 7 inputs.

# Successful daily, rerun
✓ Updated daily digest: usbank_techresilience2026_analysis_20260404_daily-digest (version 3 → 4)
  Drawn from 8 inputs.
```

Two-space indent on continuation lines also matches the convert/ingest convention (cli.py:358-371). The spec should commit to either the indented or non-indented continuation-line style explicitly to prevent implementer drift.

On the question raised in the review prompt — "is the version 3 → 4 useful or noise?" — **useful, keep it**. The version delta is what tells the operator a rerun was idempotent-with-new-content rather than a no-op. Without the delta, the operator who runs the same command twice in five minutes can't distinguish "the second invocation re-synthesized fresh inputs" from "the second invocation was wasted." Slice 6a's `(no change)` suffix in `Already rejected` (cli.py:2587) demonstrates the same value pattern. Keep `(version 3 → 4)`. Consider whether to also emit a `(no change to inputs)` annotation when `version` increments but `draws_from` is unchanged — but that's a minor enhancement and not required for this slice.

### PROD-SF-3 (should-fix): `--week --include-flagged` accepts silently

**Spec §4 row 4:**

> The `--week --include-flagged` combination is accepted without warning to avoid surprising operators who default-pass the flag.

**Spec §13.2 DG-CLI-14:**

> `--week --include-flagged` accepts both flags, exits 0; `--include-flagged` is no-op for weekly

This is exactly the silent-acceptance pattern slice 4's PR-SF-3 (`v0.6.4_spec_product_claude-sub.md`) flagged as a should-fix in a different surface. The operator running `folio digest <scope> --week --include-flagged` (perhaps because their muscle-memory reaches for `--include-flagged` on every digest invocation) gets *no signal* that one of their flags did nothing. They might come back the next morning believing the weekly digest had widened input semantics that it never had.

The spec's stated rationale ("avoid surprising operators who default-pass the flag") is reasonable for the *minimal* case but fails the *high-information* case: one terse advisory line is not "surprising," it is informative. Compare slice 4's solution: `folio links status --include-flagged` produces a footer line explaining what the override changed.

**Recommended rewrite for §4 row 4:**

```
--include-flagged | Boolean flag. Daily mode only consults this; weekly mode emits an advisory line ("--include-flagged has no effect in --week mode; weekly inputs are existing daily digests, which are already filtered at write time.") and proceeds without error.
```

Or, alternatively, surface the no-op in the weekly Trust Notes block (since the trust posture is the actual semantic question being asked). Either resolves the silent-acceptance issue. The spec should commit to one.

DG-CLI-14 should then assert presence of the advisory line, not just exit-0 silent acceptance.

### PROD-SF-4 (should-fix): "No eligible inputs" doesn't echo whether `--include-flagged` was consulted

**Spec stdout (§4):**

```
# Empty (no inputs at all)
No eligible inputs for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026.
```

The operator who runs `folio digest <scope> --date 2026-04-04 --include-flagged` (because they want to be sure to catch flagged work) and gets this exact message has no way to tell whether their override was honored. Two equally-plausible operator interpretations:

1. "There genuinely are no inputs, including flagged ones." (correct)
2. "Did `--include-flagged` even work? Is there a CLI bug? Did I typo the flag?"

Worse, this reads identically whether the operator used the flag or not. The slice 4 disclosure pattern (`folio links status --include-flagged` echoes the override status in the footer) explicitly avoids this.

**Recommended rewrite for §4 empty-no-inputs case:**

```
# Empty (no inputs at all, no override)
No eligible inputs for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026.

# Empty (no inputs at all, --include-flagged was passed)
No eligible inputs for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026 (--include-flagged was honored; no flagged inputs found either).
```

The conditional clause confirms the override took effect. Same pattern applies to the empty-due-to-flagged case (which currently is also silent on whether the operator's override was the *reason* the count is what it is).

This is the same "no surface may silently report empty when flagged inputs are the real cause" principle slice 4's section 11 rule 5 codified — applied here to the *override-side* of the same axis.

### PROD-SF-5 (should-fix): CF-1 deferral leaves discoverability gap

**Spec §14 CF-1:**

> CF-1: `cli.refresh` rerun-guidance message for `analysis` rows (design §8 rule 3) — Existing `refresh` already skips source-less analysis rows (no error); the spec asks for an explicit "use folio digest" guidance message which is a UX nicety, not a correctness requirement.

**Verified at cli.py:1053-1056:**

```python
for entry in skipped_analysis:
    click.echo(
        f"↷ {entry.id}: skipping analysis document (source-less); rerun the originating analysis workflow instead"
    )
```

The existing message says "rerun the originating analysis workflow instead" — *generic*, not naming `folio digest`. For a sole operator who has never used `folio digest` before, this surface is the first place they encounter "this row exists, refresh skipped it, what now?" — and the answer is not in the message.

CF-1's reasoning ("UX nicety, not a correctness requirement") is correct but incomplete. The discoverability question is: **without the message, where else does the operator learn `folio digest` is the right command?** The spec doesn't name a substitute surface. Three options:

1. **Fix in this slice.** Modify cli.py:1054-1056 to detect `subtype == "digest"` and say `... rerun 'folio digest' instead` for digest rows. ~5 lines of change. Inside the spec's allowed-paths (cli.py is "Modify (narrow addition)"). Recommended.
2. **Defer but commit to a help-text discoverability surface.** Spec adds: `folio digest --help` examples block names the rerun case explicitly (e.g., "Re-run after upstream notes change: `folio digest <scope>`"). Operators who hit the generic refresh message at least know to try `folio digest --help` next.
3. **Leave purely deferred.** Operator must discover `folio digest` via release notes / CHANGELOG.

Option 1 is the discoverable one for a sole operator. The spec should choose between 1 and 2 explicitly. Option 3 is below the bar for a "one operator, no team to ask" workflow.

The CF-1 row in §14 should be updated either (a) to remove deferral and add the cli.py:1054-1056 diff to the spec, or (b) to commit to option 2 in §3.3's `--help` text.

### PROD-MN-1 (minor): `--include-flagged` Click help text doesn't disclose `--week` no-op

**Spec §3.3:**

```python
@click.option("--include-flagged", is_flag=True, default=False, help="Include source-backed inputs whose review_status is flagged.")
```

The help text is accurate-and-terse. But given:

- The flag's behavior differs by mode (daily: widens predicate; weekly: no-op)
- This complexity is the substance of PROD-SF-3 above

The help text should mention the mode dependency. Recommended:

```python
help="Include source-backed inputs whose review_status is flagged (daily mode only; no-op in --week)."
```

This is one extra clause in the existing help string — minimal cost. Pairs with PROD-SF-3's runtime advisory line; the static help text and the runtime advisory should agree.

### PROD-MN-2 (minor): Error line lacks `✗` prefix

**Spec §4 LLM failure example:**

```
Error: LLM synthesis failed for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026 (existing digest, if any, preserved). Cause: <upstream message>
```

**Existing folio CLI error convention (verified):**

| Line | Surface |
|---|---|
| `cli.py:374,377` | `✗ Error: {e}`, `✗ Conversion failed: {e}` |
| `cli.py:1208` | `✗ {e}` (provenance) |
| `cli.py:2443,2447,2453,2492,2572,2578` | `✗ ...` (entities errors) |

Replace `Error:` with `✗ ` prefix to match. Same scanning-affordance argument as PROD-SF-2.

**Recommended rewrite for §4:**

```
✗ LLM synthesis failed for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026 (existing digest, if any, preserved). Cause: <upstream message>
```

Same change to all error lines in §11 (the table column "user-facing error message" should specify `✗` prefix uniformly).

### PROD-I-1 (informational): Trust Notes "impossible" branch confirmed impossible

Spec §9.3 row 4 marks `included_flagged_via_override and excluded_flagged_count > 0` as impossible. Verified by static inspection of §5 step 3.4: `excluded_flagged_count` increments only when `not include_flagged`. With `include_flagged=True`, no increment, so `excluded_flagged_count == 0` always.

No race risk: `_collect_daily_inputs` is single-threaded and the flag value is fixed for the duration of the call. The branch is genuinely impossible. No action.

(Minor stylistic note: the spec could replace the `(impossible)` row with a one-line comment outside the table — `# Branches above are exhaustive given the §5 invariant include_flagged → excluded_flagged_count == 0` — to avoid suggesting a fourth code path needs implementation.)

### PROD-I-2 (informational): Disclosure precedence not enumerated

The spec doesn't name a stdout precedence order when multiple disclosure conditions trigger. The only realistic combinations are:

1. Empty-due-to-flagged + `--include-flagged` was *not* passed → covered by §4 empty-due-to-flagged case (single message).
2. Successful write + `--include-flagged` was passed + flagged inputs were included → §9.3 Trust Notes branch 3 covers the body; the stdout success line is `Wrote/Updated daily digest:` plus `Drawn from N inputs.` — no separate `--include-flagged` echo on stdout.

The "successful write + override echoed to stdout" case is missing from §4. After PROD-SF-2 / PROD-SF-4 fixes, the spec should specify: when `include_flagged=True` AND inputs were synthesized, emit a one-line stdout disclosure like `Note: --include-flagged was honored; flagged inputs included.` This mirrors the body Trust Notes content for operators who don't open the file immediately after the run.

Not a blocker — the body Trust Notes section catches it eventually — but worth a one-line addition in §4. Marked informational because the body disclosure is the canonical record; stdout is a convenience.
