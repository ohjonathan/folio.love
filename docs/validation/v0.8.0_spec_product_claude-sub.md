# Summary

The structural-only MVP is a defensible narrowing (F-058 applies; v0.8.0's load-bearing value is the shared-consumer envelope + helper invariants, not LLM synthesis). But the command name "synthesize" overpromises relative to what v0.8.0 actually ships, and several §11 rule 5 / operator-diagnosis gaps in the zero-findings case mean operators will silently misread empty output. Two blockers on naming + rule-5 compliance; three should-fix on help copy, `--limit`, and envelope dispatch; minor notes on deferred-item priority.

# Blockers

**PROD-B1: `--help` copy misrepresents the command; operators will expect narrative.**
- Claim: spec §4 help copy ("Structural synthesis report of §5 proposal-linked cross-references within a scope") mirrors the command docstring verbatim. The word "synthesis" colloquially means narrative integration. An operator reading `folio synthesize --help` will reasonably expect prose output, not an arrow-rendered list of relationship proposals.
- Evidence: spec lines 317-320 (docstring) + 378-382 (help block) are identical and both lead with "Structural synthesis report". No sentence clarifies that v0.8.0 emits a list, not prose; no sentence names the equivalent of `folio links --pending` / `folio graph doctor` that already produce similar output shapes. Contrast with §15.6 parent line 462 ("shared-consumer expansion") which is the framing this slice actually honors.
- Impact: first-touch operator disappointment; support-channel noise ("where is the synthesis?"); makes v0.8.1 LLM addition a breaking UX change rather than an additive one, because operators will have internalized the wrong mental model.
- Fix: rewrite help to lead with what the command DOES, not with a category label. Proposed: "List §5 proposal-linked cross-references within SCOPE, grouped by source document. Read-only; no narrative generation in v0.8.0 (deferred to v0.8.1). Use --json for machine-readable envelope output." Add one sentence naming the deferred narrative work so operators know to expect it.

**PROD-B2: Zero-findings / zero-exclusions case violates parent §11 rule 5 spirit.**
- Claim: spec §8 failure-modes row "Zero findings" prints only `Findings: 0 + trailing newline`. The adjacent row "Zero findings with N excluded flagged" adds the excluded-count line. But the zero-exclusions case gives the operator no information to distinguish "scope resolved, no proposals exist" from "scope resolved but is empty of documents" from "producers haven't run yet."
- Evidence: spec lines 447-449 (§8) and §3.5 renderer (lines 347-351) only emit the "Excluded (flagged inputs):" line when `excluded_flagged_count` is truthy. Parent rule 5 says "no surface may silently report empty when excluded flagged inputs are the real cause" — the 0-exclusions zero-findings path doesn't *silently* hide exclusions (there are none), but the operator still cannot diagnose the cause. Parent rule 5 read purposively (operator cannot be left with unexplained emptiness) is not satisfied.
- Impact: the discoverability question (item 2 of the product brief) is the operator's main failure mode for a brand-new command. Empty output with no diagnostic breadcrumb is the classic Unix-tool UX failure.
- Fix: always print a one-line diagnostic when `len(findings) == 0`. Suggested shape: `No pending relationship proposals in scope (0 excluded). Run \`folio enrich <scope>\` to generate proposals, or \`folio links --pending\` for a full listing.` This satisfies rule 5's read-purposively intent and front-loads the operator's next action. Add SYN-CLI-14 test.

# Should-fix

**PROD-SF1: Arrow rendering doesn't scale; ship `--limit N` in v0.8.0.**
- Claim: §3.5 renders 2 stdout lines per finding (arrow line + reason summary). "Typical ≤1000 findings" (§2.5 line 172) = 2000 lines of scrollback on an unbounded library-wide invocation. Operators will pipe to `less` or `head -N`, but piped output is not composable with `--json`, and the absence of a canonical truncation flag forces every operator to invent one.
- Evidence: spec §2.5 line 169 defers `--relation`/`--producer` flags to v0.8.1 but says nothing about pagination. §11 acceptance criteria don't mention limit. `folio links --pending` pattern (per spec §3.4 scope-resolution reference) is the analog — worth checking if it already has a limit.
- Fix: add `--limit N` (default unlimited, but documented as "consider --limit 50 for library-wide") to v0.8.0 CLI surface. Cost is ~3 lines in `_render_synthesis_stdout` and 1 CLI option. Keeps the v0.8.1 narrowing flags as the real filtering story, but gives v0.8.0 a graceful degrade path.

**PROD-SF2: `command` is the wrong JSON discriminator for the three-command envelope.**
- Claim: spec §3.3 uses `"command": "synthesize"` as the dispatch field. A downstream consumer reading mixed output (dashboard, log aggregator) keys on the shape of `findings`, not the command that produced it. Graph's findings will have different semantics (traversal results) than synthesize's (proposal list); search's will be ranked hits. All three will carry `"findings"` as the payload key but mean different things.
- Evidence: §3.3 envelope table rows E1+E3 lock the key name. E6 allows finding-level additive extension, but the top-level `findings` label is shared across all three commands. This is a payload-type conflation: one word, three meanings.
- Fix: either (a) rename top-level key per command (`proposals` / `nodes` / `hits`) while keeping `command` as convenience metadata, OR (b) add `payload_type: "synthesis_report"` alongside `command` so dashboards can dispatch on payload type. (a) is the cleaner long-run shape; (b) is the additive-compatible mitigation. Pick one and document as envelope invariant E7. Bump to schema_version 1.1 when graph/search ship so the three don't silently re-use key names.

**PROD-SF3: Deferred `schema_gate_result` is a user-visible regression vs. graph's v0.7.1 output.**
- Claim: spec §6 row 6 defers schema_gate_result from the envelope. Graph's v0.7.1 `--json` (per v0.7.1 spec shipped at PR #60) surfaces it. When the same operator reads a graph payload (with schema-gate info) and a synthesize payload (without), the envelope parity story breaks — one of the spec's two load-bearing goals (shared envelope across commands).
- Evidence: §6 row 6 marks it "defer-to-v0.8.1" with rationale "synthesize is surface-time only". But §3.3 claims the envelope is shared. Shared-envelope with differing per-command finding keys is not parity; it's coexistence.
- Fix: either surface schema_gate_result as a passthrough (value taken from the proposal; synthesize doesn't validate, just reports) in v0.8.0, OR revise §3.3 to document the divergence as intentional and add envelope invariant "finding-level key presence is per-command, schema_version tracks envelope shape only." Option 1 is 1 extra dict key and preserves parity.

# Minor

**PROD-M1: Deferred-item pain ranking (product-brief item 6).** Operator pain from highest to lowest: (1) no `--limit` — blocks every library-wide invocation (elevated to SF1). (2) graph envelope migration — creates two-shape world for JSON consumers until the follow-up ships (parity regression noted in SF3). (3) `--relation`/`--producer` narrowing — operators will work around with `jq` on `--json` output; painful but doable. (4) full evidence-bundle rendering — compact one-line + reason is actually more scannable than 1-3 locator lines at scale; defer is correct. (5) LLM narrative — the naming problem (B1) is the real issue; if help copy is honest, the absence is fine for a v0.8.0.

**PROD-M2: Alternate MVP shape (product-brief item 7).** Structural-only IS the right narrowing given the shared-contract-first framing. An "ID-only synthesis report piped to other tools" variant is essentially what the `--json` envelope already is minus the stdout human-readable — not enough delta to reframe the MVP. But consider adding a `folio synthesize --format ids` mode (one proposal_id per line, pipe-friendly) to make the "feed to other tools" workflow a first-class shape without waiting for v0.8.1. Low cost; high composability.

**PROD-M3: Spec §3.4 choice-of-@cli.command-vs-@cli.group note is correct but the line "consistent with how `folio links` evolved" (line 334) should cite a commit or date so reviewers can verify the claim. Minor — not a blocker.

**PROD-M4: Acceptance criteria (§11) don't include a `--help` copy check. If PROD-B1 fix lands, add a test asserting the rewritten help string contains "list" (or similar) and NOT lead with "Structural synthesis report" as the sole framing.
