### D-2: Test coverage — missing legacy `status` test for `_preserve_rejected_proposals` (should-fix)

**Severity:** should-fix

`_preserve_rejected_proposals` has a legacy fallback path using
`_STATUS_TO_LIFECYCLE` (lines 1059-1061). The test
`test_rejected_preserved` uses `lifecycle_state` directly. There is no test
that exercises the legacy `status: "rejected"` path through
`_preserve_rejected_proposals`.

By contrast, `_suppress_rejected_proposals` has explicit legacy coverage in
`test_enrich_data.py` (`test_enrich_emission_time_rejection_filter_reads_legacy_status`).

**Recommendation:** Add a test like:

```python
def test_preserve_rejected_reads_legacy_status(self):
    from folio.enrich import _preserve_rejected_proposals
    existing_meta = {
        "axes": {
            "relationships": {
                "proposals": [
                    {"relation": "supersedes", "target_id": "t1",
                     "status": "rejected", "basis_fingerprint": "sha256:a"},
                ]
            }
        }
    }
    preserved = _preserve_rejected_proposals(existing_meta)
    assert len(preserved) == 1
```

### D-3: Test coverage — `_count_library_queued_proposals` not unit tested (should-fix)

**Severity:** should-fix

`_count_library_queued_proposals` is a non-trivial function (40+ lines) that
reads the registry, iterates documents, and parses frontmatter. It has
filesystem dependencies that make testing harder, but there are no tests for
it at all -- not even for the fail-open behavior (registry missing returns 0)
or the legacy status counting path.

The queue cap enforcement tests in `TestQueueCapEnforcement` only test
`_enforce_queue_cap` in isolation with `existing_queued_count` passed directly.
The counting function itself is untested.

**Recommendation:** Add at least:
1. A test with a mock registry that verifies the count.
2. A test that verifies registry-not-found returns 0.
3. A test that verifies a corrupted document is skipped (not fatal).

This is should-fix because a bug in counting would silently disable the queue
cap (returning 0 means the cap is never reached).

### D-10: Error handling — `_count_library_queued_proposals` L2 policy (informational)

**Severity:** informational

The function implements L2 failure policy correctly:

- Missing registry: early return 0.

### D-12: `_get_lifecycle_state` default differs from `_suppress_rejected_proposals` (should-fix)

**Severity:** should-fix

`_get_lifecycle_state` (lines 1067-1073) uses a default value for the
`status` key:

```python
old_status = raw.get("status", "pending_human_confirmation")
```

Meanwhile, `_suppress_rejected_proposals` (line 1025) does:

```python
state = p.get("status")
```

No default. If a legacy proposal has neither `lifecycle_state` nor `status`,
`_get_lifecycle_state` returns `"queued"` (via `_STATUS_TO_LIFECYCLE`
mapping of `"pending_human_confirmation"`), while `_suppress_rejected_proposals`
returns `None`, which fails the `!= "rejected"` check and skips the entry.

Both behaviors are effectively correct for their respective use cases (the
counting function treats missing state as queued, the suppression function
treats missing state as non-rejected), but the inconsistency could confuse
future maintainers.

**Recommendation:** Use `_get_lifecycle_state` in `_suppress_rejected_proposals`
for consistency. This would also resolve the adversarial review finding
about inconsistent legacy fallback patterns.

### D-13: `_enforce_queue_cap` early return preserves original order (informational)

**Severity:** informational

When `remaining_capacity >= len(queued)` (all proposals fit), line 1144
returns `proposals, 0` -- the original list, unsorted. When the cap is
exceeded, it returns `non_queued + queued_sorted`. This means the output
order differs depending on whether the cap was hit. This is acceptable (as
noted in the adversarial review) but worth flagging for completeness.
