### PEER-SF-D2-001 — CSF-3 producer-hint branch emits misleading message when QUERY matches zero views

**Severity:** should-fix.
**Evidence:** `folio/search.py:265-269` + `folio/search.py:347-358`.

```python
# search.py:265-269
producer_hint_names: Optional[list[str]] = None
if producer is not None and not findings:
    producer_hint_names = [v.producer for v in query_matched]
```

```python
# search.py:347-358
if not report.findings and producer_hint_names is not None:
    if producer_hint_names:
        observed = sorted(set(producer_hint_names))
        click.echo(
            "Hint: --producer matches exactly (case-sensitive). "
            f"Observed producers in scope: {observed}"
        )
    else:
        click.echo(
            "Hint: no proposals in scope for any producer. "
            "Run `folio ingest` / `folio enrich` to populate the queue."
        )
```

**Reasoning:** `producer_hint_names` is populated from `query_matched` (post-QUERY, pre-producer filter). If the scope has 50 proposals from various producers but none match QUERY, `query_matched` is empty → `producer_hint_names = []` → stdout prints "no proposals in scope for any producer." But that is factually wrong: there ARE proposals in scope; QUERY simply did not match any of them. The correct hint in that situation is the generic "No matches for '<QUERY>'" zero-results breadcrumb, not the producer-scoped hint. The CSF-3 spec text (§3.5: "producer names observed in the un-filtered view set (pre-producer-filter); non-None only when the caller passed --producer and got zero matches. Empty list is a valid signal (scope has zero proposals of any producer).") anticipates this — it was written assuming `producer_hint_names` would be sourced from `views` (pre-QUERY-match), not from `query_matched` (post-QUERY-match).

Note: the two specs lines above are slightly in tension (§3.5's "un-filtered view set" is ambiguous about whether QUERY is applied), but the operator-facing message "no proposals in scope for any producer" is only true if computed from `views`, not `query_matched`. Test SRC-CLI-30 happens to use a library with exactly one proposal matching QUERY but with wrong-case `--producer`, so it exercises the "observed list is non-empty" branch and misses this edge.

**Proposed closure:** change `producer_hint_names = [v.producer for v in query_matched]` to `[v.producer for v in views]` (matches §3.5 spec text semantically). Add a CLI test: library has proposals from producer X but QUERY doesn't match any of them, `--producer Y` → expect the `No matches for '<QUERY>'` breadcrumb path (i.e., producer hint suppressed), OR an updated hint that distinguishes "QUERY matched zero views" from "producer matched zero views." Either is spec-consistent if the observed behavior changes to match.

### PEER-SF-D2-002 — Lambda callback indirection for `_search_validate_query` adds three hops without payoff

**Severity:** should-fix (cosmetic + maintainability).
**Evidence:** `folio/cli.py:2000` + `folio/cli.py:2081-2085` + `folio/search.py:302-313`.

```python
# cli.py:2000
@click.argument("query", required=True, callback=lambda ctx, p, v: _search_validate_query(ctx, p, v))
```

```python
# cli.py:2081
def _search_validate_query(ctx, param, value):
    """Thin wrapper delegating to folio.search._validate_query (lazy import)."""
    from .search import _validate_query
    return _validate_query(ctx, param, value)
```

**Reasoning:** Three levels of indirection to invoke one validator: (1) the lambda at decoration time, (2) the module-level `_search_validate_query` wrapper, (3) the actual `_validate_query` in `folio.search`. The spec §3.4 sketch uses `callback=_validate_query` directly on an imported name. If the goal is CLI-level lazy import parity with the `from .search import (...)` block inside the command body, it's already satisfied by the `_search_validate_query` wrapper alone — the lambda adds nothing. Options: (a) drop the lambda; use `callback=_search_validate_query` directly. (b) Move `_validate_query` up to a top-level eager import in cli.py (would break lazy-import pattern parity with synthesize). Option (a) is strictly better: same behavior, two fewer hops, one fewer line of code. The `lambda ctx, p, v: _search_validate_query(ctx, p, v)` wrapper is a pointless eta-expansion — it's literally equivalent to the function object itself.

**Proposed closure:** change line 2000 to `@click.argument("query", required=True, callback=_search_validate_query)`. Keep `_search_validate_query` (lazy-import preservation). Tests unaffected.

### PEER-SF-D2-003 — `SearchReport.scope` stores double-normalization; the second call is wasted

**Severity:** should-fix (minor defect, no observable regression).
**Evidence:** `folio/search.py:248-279`.

```python
doc_id, subtree_scope = _resolve_scope(config, scope)  # calls _normalize_scope_arg internally
...
return SearchReport(
    scope=_normalize_scope_arg(scope),  # SAME normalization, second call
    ...
)
```

**Reasoning:** `_resolve_scope` calls `_normalize_scope_arg` at line 126 to compute `normalized`; then `search()` at line 271 calls `_normalize_scope_arg(scope)` AGAIN to set `report.scope`. Two identical normalizations of the same input. This is a carry-forward from synthesize (`folio/synthesize.py:178` exhibits the same pattern), so it's not a search-specific defect — but this is a fresh greenfield opportunity to fix it. Proposed: return the normalized value from `_resolve_scope` as a third tuple element, OR use it directly. Cleaner:

```python
normalized_scope = _normalize_scope_arg(scope)
doc_id, subtree_scope = _resolve_scope_from_normalized(config, normalized_scope)
# ...
return SearchReport(scope=normalized_scope, ...)
```

The wasted call is cheap (string strip + comparison), so this is "should-fix" only for maintainability. If the hoist slice for `render_envelope` lifts `_resolve_scope` too, fixing it globally there is equivalent.

**Proposed closure:** either defer to the shared-envelope hoist slice (noting carry-forward from synthesize) OR refactor `_resolve_scope` to accept and return the normalized scope. Not a blocker since synthesize has the same pattern and ships.

### PEER-SF-D2-004 — SRC-20 trust-helper call-count regression: fails silently under `--limit`

**Severity:** should-fix.
**Evidence:** `tests/test_search.py:416-447` + `folio/search.py:261-264`.

```python
# search.py — findings computed AFTER limit slicing
if limit is not None:
    producer_matched = producer_matched[:limit]
...
findings = [_view_to_finding(v) for v in producer_matched]  # trust called N times for N views
```

```python
# test_search.py — SRC-20 does NOT pass limit=
...
report = search(config, query="match_me")
assert len(report.findings) == 5
assert call_count["n"] == 5
```

**Reasoning:** The spec §3.1 pseudocode slices `matched_views = matched_views[:limit]` BEFORE `_view_to_finding` runs — confirming "post-limit" trust invocation. CSF-4's "once per matched view (post-QUERY/producer filter, inside _view_to_finding)" is satisfied, but only because limit is applied before the finding-construction loop. This is correct behavior, but SRC-20 is limit-unaware: it never exercises the limit path. A downstream refactor that moved `_view_to_finding` before the limit slice would pass SRC-20 (5 == 5) while silently regressing the call count invariant (would be 5, but for 10 views if `[:5]` were removed, it'd be 10). The spec text "once per matched view (post-filter, inside _view_to_finding)" makes no explicit claim about post-limit behavior; one could argue calling trust for all `producer_matched` views then discarding N+ of them would still be "once per matched view."

**Proposed closure:** add a second call-count test: 10 views matching QUERY, limit=3 → spy count == 3 (post-limit invariant). Or rewrite SRC-20 to exercise both branches. Low-risk addition; pins the current behavior.

### PEER-SF-D2-005 — `folio search --limit -1` stderr text surface not asserted

**Severity:** should-fix (narrow).
**Evidence:** `tests/test_cli_search.py:534-538`.

```python
def test_search_limit_negative_cli_exits_2(tmp_path):
    config_path = _setup_empty_library(tmp_path)
    r = _invoke(config_path, ["search", "foo", "--limit", "-1"])
    assert r.exit_code == 2
```

**Reasoning:** SRC-CLI-29 asserts exit code but not the Click `IntRange` diagnostic text. The spec §8 failure-modes table (row: "--limit -1 or non-integer") specifies stderr: `"Invalid value for '--limit': N is not in the range x>=0."` Adding a substring check on `r.output` for `"--limit"` or `"range"` would prevent a silent regression where the negative value is caught by a wrong validator (e.g., if someone switches to `type=int` with post-parse bounds-check). Parity: SRC-CLI-25 already asserts `"non-whitespace"` substring.

**Proposed closure:** add `assert "--limit" in r.output or "range" in r.output` to SRC-CLI-29.
