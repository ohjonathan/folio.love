# B.1 Product Review -- emission-time-rejection-v0-6-3

## Findings

| ID | Severity | Description |
|----|----------|-------------|
| PR-1 | blocker | **No operator-visible feedback when proposals are suppressed.** The spec writes `suppression_counts` into frontmatter (section 7.2) and defers CLI rendering to a future slice (section 2.2: "`--show-suppressed` CLI flag (SF-5) deferred"). The spec also notes "cli.py is forbidden path" for this slice. The result: when an operator runs `folio enrich`, proposals are silently suppressed. The operator sees a lower `proposals_count` (per section 7.3's semantic change to count only queued proposals) but receives no message explaining why the count dropped. If the queue cap suppresses 5 out of 7 proposals, the operator sees "2 proposals" with no indication that 5 more existed. Before Slice 3, the operator would have seen "7 proposals." This is a user-experience regression: the operator loses information about what the system did. **Minimum fix:** even without modifying `cli.py`, the enrich summary output (which already logs warnings via `warnings.append`) can emit a log-level message such as "5 proposals suppressed (queue cap: 3, rejection memory: 2)." This uses the existing warning path, does not require `cli.py` changes, and provides basic diagnostic visibility. |
| PR-2 | should-fix | **Operator cannot distinguish "no proposals generated" from "all proposals suppressed."** After Slice 3, `folio enrich` may produce `relationships_axis = EnrichAxisResult(status="no_change")` in two scenarios: (a) the LLM returned zero raw proposals, or (b) the LLM returned proposals but all were suppressed by rejection memory and/or the queue cap. Section 6.2 shows the flow: if `all_proposal_dicts` is empty (no preserved rejected, no new proposals), the result is `status="no_change"`. But if all new proposals were suppressed AND there are no prior rejected entries to preserve, `all_proposal_dicts` is empty, so the status is `"no_change"` -- identical to the genuine no-proposals case. The operator has no way to tell these apart without reading raw frontmatter. Suggested fix: when proposals were generated but all suppressed, use `status="proposed"` with `suppression_counts` populated (the proposals list would contain only suppressed entries). This preserves the audit trail and lets the operator know that the LLM did produce proposals. |
| PR-3 | should-fix | **`proposals_count` semantic change (section 7.3) alters existing operator expectations without announcement.** The spec changes `proposals_count` from "all proposals" to "only queued proposals." This means an operator who runs `folio enrich` on two successive days may see the count drop from 7 to 2 (because 5 are now suppressed), even if no operator action occurred between runs. The count drop looks like a regression or data loss. The CHANGELOG entry (section 10) describes the new features but does not explicitly call out that the "N proposals" count shown by the CLI now has a different denominator. Add a CHANGELOG line under `### Changed`: "The proposals count shown by `folio enrich` now reflects only actionable (`queued`) proposals; suppressed and rejected proposals are excluded from the count." |
| PR-4 | should-fix | **Reject-all-then-re-enrich workflow produces confusing output.** Consider: operator runs `folio links reject` on all 5 proposals for a document, then re-enriches with changed inputs. The LLM re-proposes the same 5 relationships (same `basis_fingerprint`). All 5 are marked `"suppressed"` by rejection memory. The 5 original `"rejected"` entries are preserved. The output is: 5 rejected + 5 suppressed = 10 proposals in frontmatter, 0 queued. The operator sees "0 proposals" and may think the enrichment failed or the LLM stopped finding relationships. There is no signal that the system is working correctly -- it found relationships but suppressed them because the operator previously rejected them. This is the intended behavior per the spec, but the operator experience is confusing without a message like "5 proposals suppressed (rejection memory); run `folio links review` to see revived proposals if inputs changed." |
| PR-5 | should-fix | **Queue-cap enforcement creates a non-obvious ordering dependency across documents.** The queue cap is library-wide (section 5.4). If an operator enriches documents A, B, C in sequence and the cap is hit during B, documents C's proposals may all be suppressed. But if the operator enriches in order C, A, B, document C's proposals survive and B's are suppressed. The spec does not discuss whether this ordering sensitivity is acceptable or whether the enrichment plan should sort documents to produce a stable outcome. For a solo operator this is manageable (re-enrich after reviewing), but the non-determinism could be confusing if the operator re-enriches the same library twice and gets different per-document results because registry iteration order changed. |
| PR-6 | informational | **Upgrade path requires zero operator action, but the spec does not state this explicitly.** The spec introduces new frontmatter fields (`suppression_counts`) and a new lifecycle state writer (`"suppressed"`). Both are additive. Old frontmatter without these fields continues to work. Old `lifecycle_state` values are unaffected. The operator does not need to run a migration, re-enrich, or modify any configuration. This is good, but the spec should include a one-sentence statement in section 1 or a new section: "No operator action required on upgrade. Existing frontmatter is forward-compatible." This prevents implementation uncertainty and reassures operators reading the CHANGELOG. |
| PR-7 | informational | **Stale rejected entries accumulate indefinitely.** Section 3.3 acknowledges that rejected entries with unreachable `basis_fingerprint` values linger harmlessly and defers cleanup to a future `folio links gc` command. For a solo operator with a small library this is fine. For larger libraries over time, the proposals list in frontmatter could grow with stale rejected entries that never suppress anything. The spec's deferral is reasonable, but it would be helpful to state an expected upper bound (e.g., "a document with 10 rejected entries adds approximately 2KB to frontmatter; this is negligible for libraries under 1000 documents"). |
| PR-8 | informational | **`suppression_counts` is not surfaced by `folio links review`.** The review-time filter in `collect_pending_relationship_proposals` (links.py lines 139-208) already filters on `lifecycle_state == "queued"`, so suppressed proposals are invisible to review. But the `suppression_counts` dict written to frontmatter is also not read or displayed by any review command. An operator who wants to understand why their review queue is smaller than expected must read raw frontmatter. This is acceptable for now (the spec defers CLI surface), but the diagnostic value of `suppression_counts` is limited until a consumer exists. |

## Verdict

Needs Fixes

## Rationale

One blocker prevents approval.

PR-1 identifies a user-experience regression: the spec introduces suppression behavior that silently reduces the proposal count shown to operators, with no diagnostic feedback in the current CLI output. The suppression is working as designed, but "working silently" is a product defect when the operator's mental model depends on the proposal count to understand what the system did. The fix is minimal -- a warning-level log message using the existing `warnings.append` path -- and does not require touching `cli.py`.

The four should-fix items address operator experience gaps that would cause confusion in normal workflows:
- PR-2: the "no_change" status conflates two distinct scenarios (no proposals vs. all suppressed), losing diagnostic signal.
- PR-3: the `proposals_count` denominator change is a silent behavioral shift that should be announced in the CHANGELOG.
- PR-4: the reject-all-then-re-enrich workflow produces a "0 proposals" result with no explanation, which looks like a system failure to the operator.
- PR-5: library-wide queue-cap enforcement makes enrichment results depend on document processing order, which is non-obvious.

The informational items (PR-6, PR-7, PR-8) are documentation and future-proofing suggestions that do not block implementation.
