## 3. Quality assessment

The implementation is tidy and reads directly from the spec. The six-way decision tree inside `_entry_to_finding` (`folio/enrich.py:1994-2092`) is flat, sequentially ordered by disposition, and cheap to audit — clearer than a dispatch table would be for six cases. `diagnose_notes` (`folio/enrich.py:2095-2162`) is a thin filter+sort+envelope over `plan_enrichment`; the two preflight checks (scope, limit) run before the plan call, which is correct. `_resolve_scope_or_raise` correctly mirrors `plan_enrichment`'s deck-type filter and uses the local `_matches_scope` so diagnose does not import `folio.links` (as §2.2 forbids).

The `ScopeOrCommandGroup` rewrite walk (`folio/cli.py:43-97`) is the subtlest piece. Given the pressure of three v1.x iterations on this code path, the final shape is surprisingly readable: one flag (`saw_subcommand`) plus two state variables (`rewrote`, `skip_next`) compose cleanly. `saw_subcommand` short-circuits correctly — once a registered subcommand is emitted, all remaining tokens pass verbatim. Value-taking options are detected by introspecting `self.params`, which is the right place to look.

The two concerns that offset the otherwise-clean impression:

1. The truncation footer is textually broken — the renderer emits `... (showing N of more; raise --limit to see more)` because the pre-truncation total is not available on `DiagnoseResult`. This is not "the operator sees a mildly wrong number" — the literal word "more" appears where the count should be. See PEER-DCB-001.
2. `DiagnoseResult.findings` is declared as bare `tuple` instead of `tuple[DiagnoseFinding, ...]` (spec §3.1 promises the parameterized form). See PEER-DMIN-001.

Abstraction depth is appropriate. Render helpers are the right size and the right location (cli.py, outside the marker block, mirroring graph doctor at `folio/cli.py:2365-2384`). No premature abstraction into a shared envelope module.

## 4. UX review

`--help` output is operator-friendly (PROD-MIN-002 closure is visible in the docstring wording). The `Notes:` paragraph inside `enrich_diagnose_cmd` threads the deliberate-omission posture for `--include-flagged` without embedding the forbidden literal. Breadcrumb text `Tip: run \`folio enrich diagnose ...\`` matches PROD-SF-006.

The truncation footer bug (above) is the only operator-facing defect. Users who supply `--limit` small enough to truncate will see a garbled English sentence. The summary line "1 findings" has a pluralization nit (grammatical only; not spec-violating).

Exit semantics look correct: exit 0 on findings, exit 1 on `ScopeResolutionError` / generic Exception, Click usage error (exit 2) on `--limit` ≤ 0 via `IntRange(min=1)`.

## 6. Positive observations

- Field-order regression is locked tight. ED-14 + G-cardinality-5 both assert exact field lists on all three dataclasses; impossible to silently drift a positional-tuple consumer.
- `MappingProxyType` wrapping at `enrich.py:2150` is the right stdlib choice for value-level immutability on a frozen dataclass. ED-24 asserts the type AND the `TypeError` on mutation — covers both surface and invariant.
- `ScopeOrCommandGroup.parse_args` (cli.py:43-97) survived three revisions (v1.1 bare-positional, v1.2 option-value awareness, v1.3 subcommand-args verbatim) with the final shape staying linear and debuggable. The `saw_subcommand` flag is the cleanest possible encoding of the "stop rewriting once a subcommand is taken" rule.
- Defensive-default coverage is exhaustive and mirrors the spec's ADV-SF-003 table: protect/conflict/analyze-non-eligible all fall through to `managed_sections_unidentified` rather than silently omit. Matches ED-15a/b/c exactly.
- Marker comments `# v1.0.0 diagnose CLI: BEGIN/END` bound the diagnose block so G-scope-6 firewall can grep cli.py without false positives from digest/search/graph-doctor's legitimate `--include-flagged`.
- Renderer separation (`_render_diagnose_text` vs `_render_diagnose_json`) is right-sized; both are local, both do one thing. No premature abstraction into a shared envelope module despite the §11 carry-forward flirting with it.
- Exception-to-exit mapping in `enrich_diagnose_cmd` (cli.py:1292-1303) cleanly routes `ScopeResolutionError → exit 1`, `ValueError → exit 1`, everything else → `exit 1 with logger.exception`. Mirrors the enrich-group body style.
- Breadcrumb (cli.py:1248-1252) respects the empty-scope case with `.rstrip()` — no trailing backtick-space-backtick.

## 7. Verdict

**Needs Fixes.**

One blocking peer finding (PEER-DCB-001: truncation footer is textually broken). Two major (PEER-DSF-001 weak test that hides DCB-001, PEER-DSF-002 library-wide corrupt-registry bypass). The rest are minor and can be bundled with the D.4 fix pass or deferred.

PEER-DCB-001 and PEER-DSF-001 should ship together: fixing the renderer without strengthening the test leaves the regression surface open. PEER-DSF-002 is the only other finding that touches a spec-promised failure policy (§9) and should not be deferred.

## 8. Notes

- The `DiagnoseResult` model not carrying an unfiltered count is a spec-shape concern as well as an impl concern — the §7 envelope doesn't expose M either, so JSON consumers also can't reconstruct "showing N of M" semantics. §11 carry-forwards list `summary.by_code_unfiltered` as a deferred v1.x key; a `summary.unfiltered_total` sibling would be the natural shape for PEER-DCB-001's resolution and could ship in the same minor bump. If the slice-local fix is code-only (compute M at render time by re-running the unfiltered tail), the envelope-key work can still defer — but the code change should land now.
- PEER-DSF-002 (library-wide corrupt-registry bypass) is on the Alignment lens's edge too; flag it once, don't double-charge. Including here because the fix is a two-line code move, not a spec edit.
- All ED-15d / ED-21 / ED-22 / ED-23 defensive-default and error-path tests are genuinely covered; the limited `pytest.warns` assertion is correct for `RuntimeWarning` scoping.
- No `folio.tracking.trust`, `folio.links`, or LLM-related imports land in the diagnose code path — G-scope-4/5 firewalls would pass. ED-12 and ED-16 enforce this at test time.
