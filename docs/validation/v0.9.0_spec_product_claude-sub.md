## Product should-fix (PROD-SF-NNN)

### PROD-SF-001 QUERY over-match on common relation / producer strings is a first-touch footgun — §3.6 needs an operator-facing "good-example" callout

**Severity:** should-fix (operator-ergonomics; high first-touch confusion probability)

**Evidence:**
- §3.6 (lines 589–608) specifies 6 searchable fields with casefold substring match. All of `source_id`, `target_id`, `relation`, `producer`, `reason_summary`, and `evidence_bundle` are searched conjunctively-via-OR.
- `relation` values are drawn from the 6-element `SUPPORTED_RELATIONS` enum (`depends_on`, `draws_from`, `impacts`, `relates_to`, `supersedes`, `instantiates` per `folio/cli.py:47-50`). A library with N pending proposals will typically have ~N/6 per relation.
- `producer` values are drawn from a tiny set (`enrich`, `provenance`, etc.). Searching `folio search enrich` returns every proposal in the library.
- §3.4 help copy lists "six text fields" but does not warn that QUERY will match relation / producer strings — an operator typing `folio search draws_from` will be surprised the result set is "every draws_from proposal" rather than "proposals that specifically mention draws_from in their reasoning".

**Why it matters (operator-impact reasoning):** A first-touch operator's mental model for `folio search X` is "find proposals about X" (content search) not "find proposals whose metadata contains the substring X" (metadata-inclusive search). The first time a consultant types `folio search draws_from` expecting to find specific proposals and gets their entire pending queue back, they will either (a) assume search is broken, or (b) lose trust in the ranking. Because v0.9.0 has no ranking (explicit §2.5 non-goal), this is specifically the case where an operator needs to understand up-front that QUERY is metadata-wide. A single §3.6 callout + one line in the help copy closes this at ~30 lines of spec.

**Proposed closure:** Add to §3.6 a short "Good examples vs over-match examples" block and mirror one sentence into the §3.4 help copy:

```
Good queries (narrow): "shared framework" (matches reason_summary), "doc_acme_2024" (matches source_id/target_id)
Over-match queries (broad; consider --producer / --scope): a bare relation name like "draws_from", a bare producer name like "enrich"
```

And in help copy add one line: `Tip: QUERY matches metadata fields (relation, producer) as well as content. Use --producer and --scope to narrow when searching common terms.`

Rationale: The spec explicitly lists `relation` and `producer` as searchable fields in §3.6; at least one acceptance criterion should guarantee operator-facing awareness.

### PROD-SF-002 `--producer` case-sensitivity is documented in help copy but zero-result case silently misses "did you mean" — §3.5 + §8 should surface a hint

**Severity:** should-fix (operator-ergonomics; invisible footgun on a flag the spec itself flags as case-sensitive)

**Evidence:**
- §1.1 rule 5 (line 84): `--producer enrich ≠ --producer ENRICH`.
- §3.4 help copy (line 468): "Restrict to findings emitted by this exact producer name (case-sensitive)." — documented, good.
- But §3.5 renderer (lines 549–585) and §8 failure modes (lines 717–725) say nothing about "operator typed `--producer Enrich` and got zero results; producer names are case-sensitive; known producers are {enrich, provenance, …}". The stdout just says `Matches: 0` and prints the generic breadcrumb from §1.1 rule 9.
- This is the specific footgun the product-focus prompt named: operators WILL try `--producer Enrich` (sentence-case is a natural habit in English) and get zero matches with no hint they hit a case-sensitivity trap.

**Why it matters (operator-impact reasoning):** This is a near-universal footgun: Click flag names are case-sensitive but most CLI tools case-fold their string values. An operator who has been using `folio links` (no producer filter) will have no intuition that `--producer` is case-sensitive. Discovering this costs them a support-loop round-trip ("why doesn't my filter work?"). Surfacing a one-line hint on zero-results when `--producer` is set is ~10 lines of stdout logic.

**Proposed closure:** Add to §3.5 a `--producer`-specific hint in the zero-findings case:

```python
if not report.findings and report.excluded_flagged_count == 0:
    if producer is not None:
        click.echo(
            f"No matches for '{report.query}' with --producer={producer}. "
            f"Note: --producer is case-sensitive (try lowercase). "
            f"Or try a broader scope / check spelling / run `folio ingest` first."
        )
    else:
        click.echo(
            f"No matches for '{report.query}'. Try a broader scope, "
            f"check spelling, or run `folio ingest` / `folio enrich`…"
        )
```

And add a matching acceptance criterion: `[ ] Zero-findings with --producer set prints a "--producer is case-sensitive" hint (SRC-CLI-NEW)`. Pair with a new test SRC-CLI-24 that asserts the hint fires on `--producer Enrich` when `enrich` proposals exist.

### PROD-SF-003 Scope-resolution error in §8 should suggest `folio status` (or equivalent) the way "registry missing" suggests `folio ingest`

**Severity:** should-fix (error-message actionability; operator-onboarding friction)

**Evidence:**
- §8 line 720: `Error: scope '<arg>' does not resolve to an engagement or document.` — states the fact, suggests nothing.
- §8 line 721 (peer line): `Error: registry.json not found. Run folio ingest first.` — states the fact AND suggests the next action. Good pattern.
- The asymmetry is visible and is the exact kind of first-touch confusion that lowers trust. An operator typing `folio search foo --scope client-acm` (typo) gets told "not resolved" but has no affordance to discover valid scopes without opening the registry manually.

**Why it matters (operator-impact reasoning):** The consultant persona reviewing AI-generated proposals will frequently mistype scope strings (engagement slugs are long). The spec already concedes there's no cross-engagement scope (§2.5), so scope typos are the #1 likely input error. The "registry missing → ingest" pattern proves the spec knows how to suggest next actions; applying it to scope resolution is trivial.

**Proposed closure:** Rewrite §8 row 2 to:

```
| Invalid --scope string | Exit 1; stderr: `Error: scope '<arg>' does not resolve to an engagement or document. Try `folio status` to list registered scopes.` |
```

And add an acceptance criterion that the error message includes `folio status` (or whichever command is canonical for listing scopes). If `folio status` doesn't cleanly list scopes, `folio links` or registry-grep instructions are acceptable substitutes — whichever the spec author confirms is current. (Suggest a Phase B clarification AskUserQuestion if uncertain.)

### PROD-SF-004 CHANGELOG v0.9.0 entry should explain the `schema_version "1.1"` asymmetry — operators will notice `synthesize --json` shows 1.0 while `search --json` shows 1.1

**Severity:** should-fix (downstream-consumer clarity; release-notes legibility)

**Evidence:**
- §3.3 lines 440–445 ("Graph + synthesize migration disposition") already explains this clearly inside the spec: each producer bumps its own envelope minor at its own pace; consumers tolerate unknown keys. Good.
- But CHANGELOG v0.8.0 (CHANGELOG.md lines 23–27) established `schema_version: "1.0"` publicly; CHANGELOG v0.9.0 will say `schema_version: "1.1"` for search. An operator diffing JSON outputs across commands WILL ask "why is synthesize still 1.0?" and have no in-changelog answer.
- §2.1 lists `CHANGELOG.md` as an allowed path but the spec doesn't draft even a rough outline of the entry (the way v0.8.0 spec did). Without a draft, the Phase-C implementer may ship a CHANGELOG that doesn't foreshadow the per-producer-versioning-policy story.

**Why it matters (operator-impact reasoning):** Consultants building tooling against `folio *--json` (plausible; they're the target persona for proposal review) will hit the version asymmetry within hours. If the CHANGELOG explains it ("each command versions its own envelope independently; consumers tolerate unknown keys at major 1.x"), they spend 30 seconds understanding; if the CHANGELOG is silent, they spend an hour writing defensive code or filing an issue.

**Proposed closure:** Add a §2.1-adjacent subsection (or embed in §3.3) with a ~15-line rough draft CHANGELOG v0.9.0 entry that matches the v0.8.0 CHANGELOG tone AND calls out:

1. `search --json` envelope is at `schema_version: "1.1"` because search added a `query` top-level key.
2. `synthesize --json` remains at `"1.0"` (unchanged — no new top-level keys).
3. `graph doctor --json` remains pre-envelope (not yet migrated).
4. Explicit one-sentence "why": "Each command bumps its own envelope minor independently; downstream consumers should tolerate unknown top-level keys at major version 1.x and check `command` / `schema_version` before parsing."

This is ~15 lines in the spec and closes a predictable downstream-consumer confusion.

### PROD-SF-005 Acceptance criteria §11 missing two CLI observables: `--scope -` dash-alias parity with synthesize MIN-1, and `--limit 0` behavior

**Severity:** should-fix (acceptance-coverage gap on user-facing observables)

**Evidence:**
- Synthesize spec §11 line 721 has `test_synthesize_envelope_scope_null_for_dash_alias` (MIN-1 closure for `folio synthesize -` → `scope: null`).
- Search spec §11 line 839 covers `envelope["scope"] is None` when no `--scope` arg, but does NOT cover `folio search foo --scope -` → `scope: null`. §3.4 line 462 help copy says "Omit or pass '-' for library-wide." — help copy promises the behavior, AC does not verify it.
- `--limit 0` is undefined. Synthesize v1.1 used `click.IntRange` (CHANGELOG.md line 33: "must be ≥ 0 per click.IntRange"). Search §3.4 line 484–488 uses plain `type=int`, no range constraint. Operator could type `--limit -5` or `--limit 0` and get undefined behavior.

**Why it matters (operator-impact reasoning):** The dash-alias gap is a pure parity miss: synthesize promises dash-alias, search help promises dash-alias, but only synthesize tests it. The limit-zero-and-negative gap is a silent footgun: `--limit 0` could plausibly return zero findings (useful for "just tell me the count") or raise (Click int accepts any integer). Either is fine, but the spec should PICK one and test it.

**Proposed closure:** Add two acceptance criteria + two tests:

1. `[ ] folio search foo --scope - → envelope["scope"] is None (dash-alias, MIN-1 parity)` — test SRC-CLI-24 (or renumber to avoid collision with SF-002 suggestion).
2. `[ ] --limit has click.IntRange(min=0) constraint; --limit -1 exits 2 with Click usage error; --limit 0 returns empty findings list with excluded_flagged_count unchanged` — test SRC-CLI-25.

Also update §3.4 line 486 to use `type=click.IntRange(min=0)` rather than `type=int`, matching synthesize CHANGELOG v0.8.0.

## Product minor (PROD-MIN-NNN)

### PROD-MIN-001 Help copy could differentiate search from synthesize in one clause

**Severity:** minor (discoverability / competing-tools clarity)

**Evidence:** §3.4 help copy (lines 499–511) describes what search does but does not position it against the adjacent `folio synthesize` and `folio links review` commands. An operator who has used synthesize may wonder "when do I use search vs synthesize?" Both operate on pending relationship proposals; synthesize groups by scope; search filters by QUERY substring.

**Proposed closure:** Add a one-clause differentiator to the help copy. Example: `Like `folio synthesize`, but narrows results by a QUERY substring across six text fields.` — this is ~1 line and positions search as "synthesize + filter" mentally.

Spec may legitimately reject this as overreach (each command's help should stand alone); recording as minor.

### PROD-MIN-003 §1.1 rule 9 breadcrumb quotes QUERY but doesn't escape — what if operator types QUERY with a backtick?

**Severity:** minor (edge-case polish)

**Evidence:** §1.1 line 88 and §3.5 lines 569–574 interpolate `report.query` inside an f-string that is echoed via `click.echo()`. If an operator runs `folio search '`rm -rf /`'`, the breadcrumb prints the literal text — no shell injection (click.echo doesn't exec) — but the rendered output would show the backticks and could look alarming. Low severity because click.echo is safe; noting for completeness.

**Proposed closure:** None required; recording for B.1 board completeness. If spec author wants to close, they can add an explicit `repr(report.query)` or a sanitization note in §3.5; I'd recommend leaving as-is (repr-wrapping clutters the breadcrumb for 99% of queries).

## Accept-as-is

- **§10 Deferred Items (12 entries):** No item obviously pulls forward for v0.9.0 on operator-pain grounds. Regex, ranking, LLM search, `--relation` / `--lifecycle`, match-annotation, locator-rendering, envelope-hoist, graph-migration — all correctly deferred. Field-scoped syntax (`producer:X reason:Y`) is correctly deferred-until-demand.
- **Exit code discipline (§4 + §8):** `0` = success (incl. zero matches), `1` = scope/registry, `2` = Click usage — matches synthesize. Operator muscle-memory transfers cleanly.
- **Trust gating (§5) parity with synthesize:** Rules 1–5 honored uniformly; Rule 6 correctly flagged as non-concern.
- **Onboarding path:** An operator who types `folio search` with no arg gets Click's standard usage error (exit 2) which contains the command signature `Usage: folio search [OPTIONS] QUERY`. That IS the right next breadcrumb (reads as "ah, I need a QUERY, let me `--help`"). Spec correctly does not re-invent Click's default usage error.
- **Shared-envelope minor-bump policy first exercise:** §3.3 "Graph + synthesize migration disposition" (lines 440–445) is the right spec-level answer; PROD-SF-004 asks only that the CHANGELOG-facing version of that answer also ship.
- **`--include-flagged` + `trust_override_active` mirror (E4):** Operator-legible and parity with synthesize.
- **QUERY verbatim round-trip (E8, SRC-29, SRC-CLI-8/9):** Correctly requires envelope to preserve operator input bytes-verbatim (not casefolded). This is the right design for downstream tooling and operator trust.
