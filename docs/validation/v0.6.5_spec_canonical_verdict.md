## Preserved blockers (must close in v1.1)

| ID | Source | Description | Required resolution |
|----|--------|-------------|---------------------|
| B-1 | ADV-001 | Schema-version bump creates downgrade break. `_ENTITY_SCHEMA_VERSION = 1` is enforced strictly by loader (`entities.py:555-561`); bumping to 2 makes v0.6.4 readers fail on v0.6.5 files with `EntitySchemaVersionError`. Spec §3.1 and §6 claim both schema bump AND downgrade tolerance — mutually exclusive against current loader. | **Do NOT bump `_ENTITY_SCHEMA_VERSION`.** Additive top-level key `rejected_merges` doesn't require a version bump; `save()` preserves unknown top-level keys. Update §3.1 (remove "bumped by +1"), §6 (reframe risk as "old writers produce files without rejected_merges — handled by load-time default"), §4.1 (remove schema-bump step that was in peer S-5). |
| B-2 | ADV-002 | `reject-merge` load-mutate-save without lock permits lost updates. Two concurrent rejects erase each other; atomic write prevents torn JSON not lost RMW. Violates parent §10.1 "rejected proposals must stay suppressed." | Mandate `library_lock(library_root, "entities")` around load-reject-save critical section. Apply to the new `reject-merge` command at minimum; document whether broader entity-family locking is in/out of scope. Add regression test that existing lock blocks `reject-merge`. |
