## Summary

BLOCK. The draft is not safe to implement as written. The hardest failure is an actual API-contract mismatch: the spec and SYN-7 treat `collect_pending_relationship_proposals` as returning `(views, int)`, but the code returns `(views, SuppressionCounts)`. Separately, the spec claims to consume and surface the parent Section 5 shared proposal contract while explicitly omitting two parent-minimum fields that the sibling graph consumer already emits. The shared JSON envelope also freezes exact top-level keys while claiming additive evolution can stay on the same schema line, which makes the v0.8.1 `findings_truncated` case break by design.
