# v1.0.0 Spec Product Review (claude-sub)

**Phase:** B.1
**Role:** product
**Family:** claude-sub
**Date:** 2026-04-17
**Target:** docs/specs/v1.0.0_folio_enrich_diagnose_spec.md v1.0
**Verdict:** APPROVE-WITH-SHOULD-FIX

Summary up top: the spec is materially operator-ready. Exit-code discipline, `[severity] code subject_id: detail` line shape, `Action:` indented trail, and empty-result message all mirror `folio graph doctor` — operator muscle memory transfers cleanly. Trust-posture annotation-only rationale (§6) is the best-defended product-shape decision in the slice chain so far; it names the sibling-command conventions and explains why diagnose's contract inverts them. Zero blockers. Six should-fix items concern operator-facing ergonomics that are cheap to close at B.1 and would bite at first-touch if shipped as-is: (1) a spec self-contradiction on output grouping, (2) two `recommended_action` strings that reference a nonexistent operator affordance, (3) `<scope>` placeholders inside action text that should be substituted, (4) no scope-resolution hint on silent zero-results (mirrors v0.9.0 PROD-SF-003), (5) help text does not explain the `[flagged]` suffix nor the intentional absence of `--include-flagged`, and (6) `folio enrich` itself does not breadcrumb to diagnose when protected / conflicted counts > 0 (the primary discovery path for new operators). Three minor items round out.

## 1. User-value assessment

The user is the folio.love operator (consulting-shop persona) who has already onboarded to `folio enrich` and has silently-failing managed-section updates they can't explain. Their current recourse when `folio enrich` reports "18 updated, 20 unchanged, **2 protected, 1 conflicted**" (pattern at parent spec §7.5) is to open each affected note, read frontmatter, hand-diff the managed body against history, and guess which rule caught the note. Diagnose compresses that into a read-only command that names the rule and the remediation step per note.

Does it deliver that value at spec time? Yes — the finding-code table (§5) enumerates every non-eligible disposition from `_determine_disposition` with a stable machine-readable code AND a human-readable action. The annotation-only trust posture (§6) is correct for a diagnostic surface; the rationale for inverting the sibling-command `--include-flagged` convention is defensible and well-argued (a flagged note is exactly what an operator wants diagnosed).

Where user-value leaks at spec time: three of six `recommended_action` strings don't give the operator a directly-runnable next step (see PROD-SF-002 + PROD-SF-003). An operator who reads "Lower curation_level" cannot act on it with any existing folio command — `folio promote` rejects downward transitions. The recommended-action text is the most operator-facing payload in the finding and the place where the diagnostic becomes actionable or not.

The primary operator discovery path (from `folio enrich` output to `folio enrich diagnose`) is not closed: enrich's "2 protected, 1 conflicted" footer still does not breadcrumb to diagnose (PROD-SF-006). That is the single most impactful spec-time improvement available and explicitly scope-compatible (spec §2.1 allows `folio/cli.py` modifications and the `enrich` docstring already changes as part of the group reshape).

## 8. Positive observations

- **§6 Trust-posture firewall is the best-defended product-shape decision in the slice chain.** §6.2 explicitly enumerates the five slice-4 discovery-surface rules and rejects rules 1-4 with reasoning; §6.3 adds a manifest-enforced grep firewall (G-scope-5) so the decision cannot regress silently. An operator reading §6 understands why diagnose looks different from its siblings — and the author anticipated the reviewer objection ("why not use `derive_trust_status`?") explicitly.
- **§5.2 defensive default for unknown `protect` reasons** is exactly right for a diagnostic surface: future `_determine_disposition` reasons surface as `managed_sections_unidentified` rather than silently vanishing. ED-15 hard-codes the test.
- **Exit-code discipline mirrors `folio graph doctor`** (parity, muscle-memory carry).
- **Empty-result message "No enrichment hygiene findings."** parallels `No graph-health findings.` in cli.py:2188.
- **Read-only invariant explicitly tested (ED-13 mtime snapshot + byte-identical content assertion)** is a stronger guarantee than most diagnostic surfaces ship with.
- **Group reshape preserves legacy `folio enrich [scope]` invocation** with explicit regression tests ED-CLI-2 + ED-CLI-3. Operators on existing workflows see zero breaking change.
- **Trailing summary line "5 findings (1 flagged): 2 error, 2 warning, 1 info."** is more informative than graph doctor's equivalent — surfaces the flagged count as first-class (not buried in per-line suffixes) and counts by severity.
- **JSON envelope includes `summary.flagged_total`** — downstream tooling can alert on flagged-count thresholds without walking the per-finding array.
- **`subject_id` stability** (§7.3 line 683: "Registry deck_id, NOT the markdown path") is the right contract — markdown paths can change, deck_ids cannot.
- **§11 Carry-Forwards is honest** — the "Parent enrich spec §7.7 wording update ('fixed schema' → 'minimum schema')" entry explicitly calls out the doc-maintenance debt the slice introduces without hiding it.

## 9. Verdict

APPROVE-WITH-SHOULD-FIX.
