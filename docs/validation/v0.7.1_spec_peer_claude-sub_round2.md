# Peer Review — folio-graph-generalized-proposals-v0-7-1 / B.2 / claude-sub (Round 2)

This is a CLOSURE-VERIFICATION round on spec v1.1. Scope is narrow: did
the six B.1 Round 1 issues I flagged (P-1, P-2, P-3, P-4, P-5, P-6)
close cleanly, and did the v1.1 expansion (§3.3 gate, §3.4 CLI, §3.5
dispatch hook) introduce any new peer-lens defects?

## 1. Completeness check

The spec v1.1 adds three new subsections (§3.3, §3.4, §3.5), a new
top-level finding key (`proposal_id`), a new row in the §2.1 table
(the `proposal_id` legacy-identifier row), and three new test entries
(§4 tests 19–23 plus the fixture-based invariant). The revision note
enumerates every B.1 closure with anchor pointers. The §2.2
carry-forward table now includes the rows A-1 flagged (§10.3, §11,
§12, §15.6 folio-graph). Structurally, the spec is complete for a
Round 2 closure.

Closure check per B.1 issue:

- **P-1 (subject_id stdout regression):** closed. §3.1 emits a new
  top-level `proposal_id` key sourced from `view.proposal.proposal_id`
  (§2.1 table row 11, §3.1 pseudocode line "proposal_id": …). §3.4
  updates `folio/cli.py:2019` to read `finding.get("proposal_id")`
  with a fallback chain to `finding.get("subject_id")` for
  non-proposal findings. The fallback is the right call — direct-run
  against `folio/cli.py:2016–2021` confirms the template today is
  shared across all findings, not branched by code, and non-proposal
  findings (`orphaned_canonical_relation`, `stale_analysis_artifact`,
  etc.) still carry a meaningful `subject_id`. §4 test 21
  (`test_graph_doctor_cli_stdout_renders_proposal_id`) asserts stdout
  contains `proposal_id` and not literal `"None"`, and
  `G-test-3` pins this as a gate. Evidence is adequate.

- **P-2 (test_graph_cli.py:389–391 migration):** closed. §4 test 19
  explicitly calls out the migration and branches by `finding["code"]`
  — pending-proposal findings carry the new 16-key set (11 §5 + 1
  `proposal_id` + 4 legacy), non-proposal findings keep the exact
  legacy 5-key set. The cited `{"code", "severity", "subject_id",
  "detail", "recommended_action"}` equality is addressed.

- **P-3 (9 vs 11 field count):** closed. §1 ("nine parent §5 rows
  represented by 11 emitted keys after ID-triad split"), §2.1 B1
  ("parent §5 contract has 9 rows … slice emits 11 keys because the
  §5 ID triad … is split into three keys"), and §2.2 first row
  ("Proposal Object Contract (9 rows, 11 emitted keys)") all use the
  same reconciled phrasing.

- **P-4 (proposal-ID recovery tuple):** closed. §3.1 "Proposal-ID
  recovery tuple" paragraph and OQ-1 v1.1 both name
  `(source_id, relation, target_id, basis_fingerprint)` and cite
  `folio/pipeline/enrich_data.py:120-126`. Static inspection of
  `enrich_data.py:120-126` confirms the cited tuple matches the
  `compute_relationship_proposal_id` call site. `producer` is
  explicitly called out as NOT part of identity.

- **P-5 (CLI --json shape test):** closed. §4 test 20
  (`test_graph_doctor_cli_json_surfaces_contract`) exercises
  `CliRunner` against `folio graph doctor --json` and asserts
  serialization of all 11 contract keys plus `proposal_id`.

- **P-6 (private-constant import ambiguity):** closed **by design
  change** rather than by documentation. The constant is renamed to
  `_SHARED_PROPOSAL_CONTRACT_EMITTED_KEYS` (same leading-underscore
  convention), but the cardinality gates (`G-cardinality-1`,
  `G-cardinality-2`) no longer depend on importing the private
  symbol — manifest v1.1 line 108–113 replaces `inspect.getsource`
  substring matches with runtime pytest invocations
  (`test_graph_doctor_fixture_mandatory_contract_invariant` and
  `test_graph_doctor_proposal_id_preserves_view_proposal_id`). §4
  test 1 still references the constant name in its
  assertion ("each pending-proposal finding contains every key in
  `_SHARED_PROPOSAL_CONTRACT_EMITTED_KEYS` + `proposal_id`"), which
  means the unit test still imports a leading-underscore symbol. The
  convention is well-established in this codebase and this is fine,
  but the spec still does not state the import path explicitly.
  This is a minor carryover, not a blocker.

Completeness gaps that remain or are newly introduced:

- **§3.3 gate definition of "deck IDs" vs "note/entity IDs".** §3.3
  rule 1 defines `target_exists` as "`view.proposal.target_id` must
  be a key in `registry.json`'s deck IDs." Direct-run against
  `folio/graph.py:162` confirms `all_ids = set(registry_data.get("decks", {}).keys())`
  — deck IDs only. Relationship proposals produced by `enrich` can
  target any document recorded in the registry, and today's
  canonical-relation orphan check (`folio/graph.py:113` and
  `folio/graph.py:185`) uses the same `all_ids` set. The gate is
  internally consistent with existing code, **but** the spec does not
  justify why relationship-proposal `target_id` is guaranteed to be
  a deck ID (entities and notes are elsewhere in the registry). If
  relationship proposals legitimately target non-deck IDs today, the
  `target_exists` rule will mis-fire as a `fail` on valid targets.
  See N-1 below.

- **§3.4 fallback to `subject_id` for non-proposal findings.** The
  pseudocode uses `finding.get("proposal_id") or finding.get("subject_id") or "—"`.
  The `or` chain treats empty-string and zero-length as falsy, which
  is the right behavior for "—", but there is an edge case: if a
  pending-proposal finding ever surfaces with a blank `proposal_id`
  (e.g., historical data where `compute_relationship_proposal_id`
  was not backfilled), the chain falls back to
  `finding.get("subject_id")` — which v1.1 pins to `None` — and then
  renders `"—"`. Operator UX remains reasonable. But see N-2: the
  spec does not define what happens for the CLI-level stdout render
  when `proposal_id` is the empty string vs. `None`.

## 3. Quality assessment

The closures are well-targeted. Each of my B.1 findings has a
narrow, traceable resolution in v1.1, and the revision note names
them by ID. The design choice to add `proposal_id` as a new top-level
key (rather than keep `subject_id` carrying the proposal ID and
shipping a comment-level note about semantic mismatch) is the right
call — it preserves operator-facing identity without violating §5
semantics, and §3.4's shared-fallback pattern means non-proposal
findings are not affected.

The §3.3 minimal gate is a principled, conservative addition.
Reusing `all_ids` and `SUPPORTED_RELATIONS` from the existing
`folio/graph.py` / `folio/links.py` surfaces means no new imports
and no new failure modes in the renderer. The short-circuit on
`target_exists` before `supported_relation` (implicit in OQ-4's
discussion of return shape) keeps the failure-shape stable at a
2-key dict.

The §3.5 dispatch-hook subsection is advisory-only, as stated, and
correctly defers factorization to sub-slice 2. I flagged helper
promotion as ADV-SF6 in Round 1 via the peer lens adjacent to
alignment; v1.1's §3.2 commitment to promote `_derive_trust_status`
as a sub-slice-2 acceptance criterion is adequate.

Two quality concerns remain:

First, §3.3's scope assumption. The rule `target_exists` reads the
same `all_ids` set that today's canonical-relation orphan check
uses, which is deck IDs only. The spec should state explicitly
whether relationship-proposal `target_id` is guaranteed to be a
deck ID in today's producer set, or acknowledge that non-deck
targets (entity IDs, note IDs if any) will fail the gate. A
mid-level engineer reading §3.3 verbatim may implement the rule,
watch it fire on a legitimate entity-target proposal in test
fixtures, and not know whether the spec is wrong or the producer
is wrong. See B-1 below.

Second, §3.4's implicit assumption about which findings have
`subject_id`. The fallback chain `finding.get("proposal_id") or finding.get("subject_id")`
assumes non-proposal findings still emit `subject_id`. Direct-run
against `folio/graph.py:180-260` confirms that today's non-proposal
findings (`orphaned_canonical_relation`, `protected_enrich_body`,
`stale_analysis_artifact`, `unconfirmed_entity`, `missing_entity_stub`,
`duplicate_person_candidate`) all carry `subject_id`, so the
fallback works. The spec does not state this as a contract — it is
load-bearing that B4 ("findings that are NOT proposals … are
unchanged by this slice") holds, so the CLI fallback works. Soft
dependency; see N-3.

Implementability is high. The pseudocode in §3.1, §3.3, and §3.4
is sufficient for a mid-level engineer to write code matching the
spec's intent. The §4 test plan covers the new invariants (21
numbered + 2 additional regression guards), and manifest `G-test-3`
+ `G-cardinality-1/2` wire the tests to gates.

## 4. UX review

The v1.0 stdout regression (`[medium] pending_relationship_proposal None: …`)
is resolved. The new stdout contract:

```
[medium] pending_relationship_proposal <proposal_id>: <detail>
  Action: Review with `folio links review` and confirm or reject it.
```

with optional `[flagged]` and `[schema-gate: <rule>]` severity
annotations. The suffixed annotations are a clean progressive
disclosure — operators see only what's non-default. Concatenation
order is documented in §3.4 pseudocode (flagged first, schema-gate
second). Minor: §3.4 and OQ-3 CHANGELOG both mention "schema-gate:
<rule>" but do not say what renders when `rule` is missing; §3.4
pseudocode says `gate.get('rule', 'unknown')`. Fine, but the
CHANGELOG language should probably say "schema-gate: <rule-id>" so
operators know what the placeholder represents. Minor nit, not a
blocker.

The CHANGELOG block in OQ-3 is verbatim-ready and cites both the
breaking `subject_id: null` change and the new `proposal_id` key.
It also calls out the `input_fingerprint` legacy-alias limitation,
which is appropriate given the A-2 narrowing.

## 7. Verdict

**Needs Fixes.**

The six B.1 issues I flagged (P-1, P-2, P-3, P-4, P-5, P-6) all
close acceptably in v1.1. The design choices — new `proposal_id`
key, runtime fixture gates, CLI fallback chain, tuple recovery
citation — are well-targeted and traceable.

However, the v1.1 expansion of §3 introduces one new blocking
peer-lens defect (P-R2-1): the §3.3 `target_exists` rule asserts
an un-cited producer invariant that `relationship` proposal
targets are always deck IDs. If that invariant does not hold
under today's producer set, the new schema gate will emit
false-positive `fail` results on legitimate proposals — the
exact review-noise the v1.1 risk framing in §1 calls out.
Resolving this requires either citing the producer contract or
broadening/narrowing the gate's target-lookup set.

Two should-fix items (P-R2-2 blank-proposal_id edge case,
P-R2-3 §3.4 load-bearing dependency on B4) are mechanical spec
additions, not design changes.

Three minor items are nits carried over or newly surfaced in
§3.5 and OQ-3.

The spec is close. The Round 1 blockers are resolved; only the
new §3.3 target-set assumption needs citation before canonical
verdict can approve.
