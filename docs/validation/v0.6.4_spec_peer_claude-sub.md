## Blockers

### B-001: `suppression_counts["flagged_input"]` pollutes rejection-memory total

**Location:** Spec section 3.2, 4.3, 5.3 vs. current CLI lines 1622-1628.

The spec adds `"flagged_input"` as a new key in the shared
`suppression_counts` dict (section 4.3). The existing CLI disclosure
at line 1622 computes:

```python
total_suppressed = sum(suppression_counts.values())
```

This sums ALL values in the dict, which after this change would include
the `"flagged_input"` count alongside producer-keyed rejection-memory
counts (`"enrich"`, `"digest"`, etc.). The rejection-memory disclosure
line would then report an inflated number that conflates two distinct
suppression reasons.

The spec's section 5.3 adds a separate flagged-input disclosure line
but does NOT specify updating the existing `total_suppressed` computation
to exclude the `"flagged_input"` key. This is a behavioral bug.

**Fix:** The spec must specify that the existing rejection-memory
summation excludes the `"flagged_input"` key. For example:

```python
flagged_count = suppression_counts.pop("flagged_input", 0)
total_suppressed = sum(suppression_counts.values())
```

Or explicitly filter:

```python
rejection_counts = {k: v for k, v in suppression_counts.items() if k != "flagged_input"}
total_suppressed = sum(rejection_counts.values())
```

### B-002: Missing `field` import in `links.py`

**Location:** Spec section 3.1 vs. `folio/links.py` line 5.

The spec proposes:

```python
flagged_inputs: list[str] = field(default_factory=list)
```

But `folio/links.py` currently imports only `dataclass` from
`dataclasses`, not `field`:

```python
from dataclasses import dataclass
```

The spec's allowed-paths section (2.1) lists `folio/links.py` as
modifiable, but does not mention adding the `field` import. An
implementer following the spec literally would hit an `ImportError`
at runtime.

**Fix:** Add to section 3.1 (or a preamble note) that the import line
must change to `from dataclasses import dataclass, field`.

### B-003: Source vs. target flagged-status lookup uses inconsistent data sources

**Location:** Spec section 4.2.

The spec defines two lookup paths:

1. **Source flagged:** `fm.get("review_status") == "flagged"` -- reads
   from the markdown frontmatter of the source document (already loaded).
2. **Target flagged:** `registry_data["decks"].get(proposal.target_id, {}).get("review_status") == "flagged"` -- reads from the registry JSON dict.

These two sources may disagree. The `review_status` field is
frontmatter-authoritative (see `reconcile_from_frontmatter` in
`registry.py` lines 226-268), meaning the registry is only updated
when reconciliation runs. Between an operator manually setting
`review_status: flagged` in a markdown file and the next
`folio provenance sync` (which triggers reconciliation), the registry
dict will hold the stale value.

For source documents this is not a problem because the function already
reads frontmatter directly. For target documents the function only has
the registry dict available (the target's markdown is not read in the
current loop). The spec should acknowledge this staleness window and
either:

(a) Accept the staleness as a known limitation (add a note to section 6),
or (b) Specify reading the target's frontmatter directly when needed
(adds I/O cost).

**Recommendation:** Option (a) is acceptable given the reconciliation
happens on every `folio provenance sync`. But the spec must explicitly
state this design trade-off so implementers do not "fix" it ad-hoc.

## Should-Fix

### SF-001: No test for `--include-flagged` CLI flag end-to-end

**Location:** Spec section 7 test plan.

Test S4-3 (`test_include_flagged_surfaces_proposals`) tests the Python
function with `include_flagged=True`, and S4-6 (`test_silent_empty_protection`)
tests CLI output. But there is no test that invokes the CLI with
`--include-flagged` flag and verifies the flag is correctly threaded
through to `collect_pending_relationship_proposals`. This is a
test-coverage gap for the actual Click option wiring.

**Fix:** Add a test (e.g., S4-9) that uses `CliRunner` to invoke
`folio links review --include-flagged` and verifies flagged proposals
appear in the CLI output with the trust-posture tag.

### SF-002: No test for trust-posture tag rendering in CLI output

**Location:** Spec section 5.2, section 7.

The spec defines a specific tag format:
`(flagged: source)`, `(flagged: target)`, `(flagged: source, target)`.
No test in the plan verifies that the CLI output actually renders this
tag. S4-4 only checks the `flagged_inputs` field on the view object,
not the rendered CLI string.

**Fix:** Either expand S4-6 or add a dedicated test that asserts the
`(flagged: ...)` tag appears in `result.output` from a `CliRunner`
invocation with `--include-flagged`.

### SF-003: Spec does not specify tag ordering relative to revived tag

**Location:** Spec section 5.2.

The spec says "The tag renders after the revived tag (if present). Both
may co-occur." The current code builds the revived tag as:

```python
revived_tag = " (revived -- basis changed)" if view.revived else ""
```

and appends it to the line. The spec should specify the exact concatenation
order in pseudocode, since an implementer could reasonably place the
flagged tag before or after the revived tag, or combine them differently.

**Fix:** Show the combined format explicitly, e.g.:

```
[<id>] source -> target (relation, confidence, producer) (revived -- basis changed) (flagged: source)
```

### SF-004: `suppression_counts` key name `"flagged_input"` could collide with a producer name

**Location:** Spec sections 3.2, 4.3.

Existing `suppression_counts` keys are producer names (e.g., `"enrich"`,
`"digest"`). The new `"flagged_input"` key is a fixed string in the same
dict namespace. If a producer were ever named `"flagged_input"`, the counts
would collide and corrupt both values.

This is unlikely in practice (the `_iter_producer_proposals` function
skips the `"links"` namespace already), but the spec should either:
(a) Document that `"flagged_input"` is a reserved key in
    `suppression_counts`, or
(b) Use a namespaced key like `"_flagged_input"` to avoid potential
    collision with producer names.

### SF-005: Missing test for `review_status: None` (legacy/unset) documents

**Location:** Spec section 6, section 7.

The spec states: "If `review_status` is missing or not a string, the
document is treated as NOT flagged (fail-open)." But no test in the plan
covers a document with `review_status: null` or a document where
`review_status` is absent entirely from frontmatter. Test S4-7 only
covers `review_status: clean`.

**Fix:** Add a test (e.g., S4-10) where the source or target document
has no `review_status` field at all, verifying the proposal passes
through unaffected (fail-open behavior).

## Informational

### I-001: Section 5.4 header format diverges from current header

The spec proposes replacing the header `Pending (0 item(s))` with:

```
Pending (0 items -- N excluded due to flagged inputs; use --include-flagged to review)
```

The current code at line 1607 uses `Pending ({len(proposals)} item(s))`
with the `(s)` pluralization pattern. The proposed replacement uses
`items` (always plural). This is fine for the zero-count case but
the spec should confirm this is intentional divergence from the
pluralization pattern.

### I-002: Carry-forward section 9.3 about `confirm-doc` / `reject-doc`

The spec correctly notes that `confirm-doc` and `reject-doc` use the
default (unflagged) view. This means an operator cannot confirm or
reject flagged-input proposals without first reviewing them via
`folio links review --include-flagged` and then using `folio links confirm`
on individual proposals. But individual `confirm` and `reject` commands
also use `_find_pending_view`, which calls `collect_pending_relationship_proposals`
without `include_flagged`. So even individual confirm/reject of a
flagged-input proposal is blocked.

This is logically consistent (flagged proposals are invisible to all
operations unless `--include-flagged` is used), but the carry-forward
note does not explicitly mention that individual `confirm`/`reject`
commands are also affected, not just the `-doc` variants.

### I-003: Test helper `_base_note` sets `review_status: "clean"` by default

The spec's section 7.1 correctly identifies this, ensuring existing
tests are unaffected. This is a good design note -- new tests for
flagged behavior will need to override this default explicitly.

## Summary

The spec is well-structured and the core trust-gating design is sound.
The main concern is B-001 (rejection-memory count inflation), which is
a behavioral bug that would produce incorrect CLI output. B-002 is a
straightforward import omission. B-003 is a design-trade-off that needs
explicit acknowledgment rather than being silently ambiguous.

The should-fix items are primarily test coverage gaps -- the spec's
8-test plan covers the Python layer well but lacks CLI integration tests
for the new flag and tag rendering.

**Verdict: Needs Fixes** -- B-001 must be resolved before implementation.
