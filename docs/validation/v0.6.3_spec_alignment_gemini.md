Here is the alignment review for `docs/specs/v0.6.3_emission_time_rejection_spec.md` against the parent Tier 4 Shipping Plan.

### Focus Area Summary
1. **Missing features?** None. The spec correctly moves the rejection-memory filter to emission time and ships the queue-cap enforcement. It correctly defers the CLI surfacing of suppression diagnostics because `folio/cli.py` is excluded from the §15.3 scope lock.
2. **Scope expansion?** Yes. The spec silently reinterprets the semantics of `proposals_count` (§7.3) and sneaks in a resolution for a prior slice's carry-forward (PR-A-13) without authorization.
3. **Queue-cap behavior (§9.1):** The limits and units ("engagement" = library) are correct. However, the implementation does a full cross-document file scan on *every* enrichment run, which introduces massive performance overhead. 
4. **Rejection-memory behavior (§10.1):** Faithfully reproduced.
5. **Preservation fix behavior:** It is necessary to honor the zero-defect invariant in §10.1 and does not introduce unauthorized behavior. Dropping old `"suppressed"` entries is logically sound, though the hardcoded `"rejected"` check will drop future terminal states.
6. **Scope-expansion justifications (§15.7):** The justification for test files and CHANGELOG is valid. The implicit justification for the preservation fix is valid. The modification to `proposals_count` has no valid justification under §15.7.

### Findings

**ID: AL-1**
- **Severity:** blocker
- **Alignment Issue:** Silent semantic scope expansion on `EnrichResult.proposals_count`. Spec §7.3 proposes modifying this field to count *only* `queued` proposals instead of all proposals. Repurposing an existing core data field to fix a CLI UX issue (without touching `cli.py`) is a semantic reinterpretation not authorized by the parent proposal. Crucially, because provenance proposals have not yet migrated to `lifecycle_state` and still use `status="pending_human_confirmation"` (as defined in the Slice 2 spec §3.6), filtering strictly for `"queued"` will incorrectly drop provenance counts to 0, breaking cross-system functionality.
- **Parent section violated:** §15.7 ("Contract: slice manifests narrow, they don't reinterpret"), §15.3.

**ID: AL-2**
- **Severity:** should-fix
- **Alignment Issue:** Unauthorized queue-cap tie-breaking behavior. Spec §5.3 step 4 introduces tie-breaking by confidence to resolve "PR-A-13", a carry-forward from Slice 1. Parent §9.1 only specifies that the pipeline "must refuse to enqueue the additional proposal" when at the cap; it does not mandate evaluating a batch and sorting by confidence. While operationally useful, this resolves a carry-forward item not explicitly allocated to this slice.
- **Parent section exceeded:** §9.1, §15.3, §15.7.

**ID: AL-3**
- **Severity:** should-fix
- **Alignment Issue:** Incorrect parent section cross-reference. Spec §4.2 justifies ignoring the `force` flag by citing "spec §7.3 -- rejection memory is absolute". The parent proposal does not contain a §7.3. The correct reference for the stable-state zero-defect invariant is §10.1.
- **Parent section violated:** §10.1.

**ID: AL-4**
- **Severity:** informational
- **Alignment Issue:** $O(N \times M)$ performance scaling for queue-cap enforcement. To enforce the library-wide cap without new persistence, Spec §5.3 dictates reading `registry.json` and scanning frontmatter across *all* documents in the library during *every* document's enrich run. For a batch enrichment of N documents in a library of size M, this causes $O(N \times M)$ file reads. While technically compliant with §9.1 ("without new persistence"), it scales poorly compared to `folio/links.py`'s query-time aggregation. 
- **Parent section:** §9.1.

**ID: AL-5**
- **Severity:** informational
- **Alignment Issue:** Future-proofing of rejected-proposal preservation. The fix in §3 is absolutely required to honor the §10.1 zero-defect invariant. However, the logic explicitly preserves *only* `lifecycle_state == "rejected"` (§3.2). While intentionally dropping `"suppressed"` is correct (they get re-evaluated), strictly ignoring future terminal states like `"stale"` or `"superseded"` (defined in §6) will cause them to be permanently lost upon re-enrichment when those states are implemented in future slices (Slice 4, Slice 6+).
- **Parent section:** §6 (Lifecycle states), §10.1.

**Verdict:** Needs Fixes
