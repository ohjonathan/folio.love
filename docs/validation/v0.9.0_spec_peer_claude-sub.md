### PEER-B1-001 SRC-25 transitive-LLM test as specified will false-positive against `folio.llm`

**Severity:** blocker

**Evidence:**

v0.9.0 spec §9.1 line 757 (SRC-25):

> | SRC-25 | `test_no_transitive_llm_imports_after_import` | Import + invoke `search()` in subprocess; assert `folio.llm`, `openai`, `anthropic` NOT in `sys.modules`. |

```python
# Narrowed to external provider SDKs (openai, anthropic). folio.llm is

```

And `folio/config.py:11`: `from .llm.types import ProviderRuntimeSettings` — confirming `folio.llm` enters `sys.modules` as soon as `FolioConfig` is imported, which happens in the CLI path and in any reasonable `search()` subprocess fixture.

**Reasoning:**

This is a carry-forward regression of v0.8.0's SF-C7 / ADV-SF-005 closure. The v0.8.0 spec initially made the same mistake; it was corrected during implementation by narrowing the forbidden set to `{openai, anthropic}` only. The v0.9.0 spec as written will either (a) force Phase C to deviate from the test table (which erodes spec fidelity), or (b) ship a test that always fails. Either way, the spec needs to say at Phase B what it will actually ship — peer review exists to catch this without a D.2 round-trip.

Related: SRC-24 `test_no_direct_llm_imports_in_module` at line 756 is fine (scans source, not sys.modules). It's only the transitive sys.modules check that breaks.

**Proposed closure:**

Rewrite SRC-25 to read:

> SRC-25 | `test_no_transitive_llm_imports_after_import` | Import + invoke `search()` in subprocess; assert `openai`, `anthropic` NOT in `sys.modules` after `search()` returns. `folio.llm` is intentionally excluded: it is transitively imported by `folio.config` (LLMConfig → provider types) and entering `sys.modules` does NOT imply an LLM call; the SRC-24 + SRC-25 guards are against provider-SDK loading, which is the actual indicator of a direct or transitive LLM invocation path. Carry-forward of v0.8.0 SF-C7 / ADV-SF-005 closure.

Also update Acceptance Criteria line 852 (currently: "No direct or transitive LLM imports after `search()` runs") to say "No direct import of `folio.llm` in module source; no provider-SDK (`openai` / `anthropic`) imports, direct or transitive, after `search()` runs." Consistency with the narrowed test.

### PEER-SF-001 §3.1 public surface does not define `_resolve_scope`, `_normalize_scope_arg`, `_view_matches`, or `_view_to_finding`

**Severity:** should-fix

**Evidence:**

§3.1 pseudocode (spec lines 322-341) references four private helpers that are never defined in the code block:

```python
# Pseudocode:
#   doc_id, subtree_scope = _resolve_scope(config, scope)
#   ...
#   matched_views = [v for v in views if _view_matches(v, needle)]
#   ...
#   findings = [_view_to_finding(v) for v in matched_views]
#   return SearchReport(scope=_normalize_scope_arg(scope), ...)
```

Two (`_resolve_scope`, `_normalize_scope_arg`) are defined in `folio/synthesize.py:88-126` — but `folio/synthesize.py` is a forbidden path per §2.2, so search must **re-implement** these helpers, not import them. §2.4 read-only dependencies does not list synthesize either. The spec is silent on whether these are required to be byte-identical copies, semantically equivalent re-implementations, or subject to operator-specific customization.

`_view_matches` and `_view_to_finding` have no precedent in synthesize.py (synthesize inlines the finding construction inside `synthesize()`). They are search-specific helpers implied by pseudocode but not explicitly required.

**Reasoning:**

This is Phase C ambiguity surface. A Phase C implementer would have to:

1. Infer that `_resolve_scope` and `_normalize_scope_arg` must be re-implemented (forbidden to import from synthesize).
2. Decide on implementation — ideally byte-identical to synthesize's version for cross-consumer consistency.
3. Decide whether `_view_matches` should be a top-level module function vs. inlined in the `search()` body, and whether it is testable in isolation (the spec's test plan does not have a direct `_view_matches` unit test — SRC-5..SRC-13 all go through `search()`).

Ambiguity is expensive: any of these decisions could spark a D.2 round-trip. Spec should either show the full function signatures of these helpers in §3.1, OR explicitly state "re-implement from synthesize §3 verbatim" with a pointer. The latter is acceptable given the "acceptable duplication, lift-later" disposition in §3.3 for `render_envelope` — the same disposition naturally extends to `_resolve_scope` / `_normalize_scope_arg`.

**Proposed closure:**

Add a subsection to §3.1 (e.g., after the `render_envelope` function) titled **"Internal helpers (re-implemented from synthesize)"** that lists the expected module-level private functions and their source in synthesize.py:

```markdown
**Internal helpers.** `folio/search.py` also defines the following private
helpers, re-implemented byte-identically from `folio/synthesize.py` because
synthesize is a forbidden-path scope-lock (§2.2). Follow-up slice that hoists
`render_envelope` (§3.3 deferral) can also hoist these:

- `_normalize_scope_arg(scope) -> Optional[str]` (verbatim from
  `folio/synthesize.py:88-99`; maps `None`, `""`, `"-"`, whitespace-only
  to `None`).
- `_resolve_scope(config, scope_arg) -> tuple[Optional[str], Optional[str]]`
  (verbatim from `folio/synthesize.py:102-126`; raises `ScopeResolutionError`).
- `_view_matches(view, needle) -> bool` (search-specific; implements the
  §3.6 6-field any-match rule with evidence_bundle any-element rule).
- `_view_to_finding(view) -> SearchFinding` (search-specific; mirrors the
  `SynthesisFinding(...)` construction at `folio/synthesize.py:158-174`).
```

This also enables a natural SRC-34 (unit test for `_view_matches` in isolation) and SRC-35 (unit test for `_normalize_scope_arg("-") is None`, closing the MIN-1 dash-alias disposition as a regression guard).

### PEER-SF-003 `excluded_flagged_count` UX ambiguity — reflects upstream trust exclusion, not QUERY-matched-then-excluded

**Severity:** should-fix

**Evidence:**

Spec §3.6 matches QUERY after the collector has already filtered flagged inputs (when `include_flagged=False`). The pseudocode at lines 322-341:

```
views, counts = collect_pending_relationship_proposals(..., include_flagged=include_flagged)
...
matched_views = [v for v in views if _view_matches(v, needle)]
```

`counts.flagged_input` is the collector's scalar count of proposals excluded by §11 trust gating — i.e., it includes proposals that **would not have matched QUERY anyway**. Yet the stdout renders:

```
Search for 'framework' in scope: (library-wide)
Matches: 3
Excluded (flagged inputs): 1 (use --include-flagged to include)
```

An operator reading this line naturally reads it as "1 proposal matched 'framework' but was excluded due to trust gating." The actual semantics are weaker: "1 proposal in scope was trust-excluded; it may or may not have matched 'framework'."

**Reasoning:**

This is a UX truthfulness concern. §11 rule 5 (surface should not silently report empty when exclusion is the cause) is honored in the **scope** sense (sibling synthesize does the same). But in a QUERY lens, the more useful disclosure would be either:

1. "N (trust-excluded in scope, may or may not match)" — more accurate phrasing, no code change.
2. Compute `excluded_flagged_count_matching = len([v for v in flagged_views if _view_matches(v, needle)])` — requires collector to return flagged views for post-hoc scoring. Not doable without `include_flagged=True` and a second pass.

Option 1 is a cheap doc/copy fix. Option 2 is a larger scope issue worth a §10 deferral note.

This same concern technically applies to synthesize too (which is scope-only, not QUERY-scoped), but is amplified by search because "Excluded (flagged inputs): N" under a QUERY heading invites the false reading.

**Proposed closure:**

Either (a) tweak stdout to read `Excluded (flagged inputs in scope): N` (adds "in scope" to disambiguate from "matched QUERY"), or (b) add a §3.6 note and §5 row 4 note documenting that `excluded_flagged_count` reflects **scope-level** trust exclusion, not QUERY-matched trust exclusion, and track QUERY-scoped exclusion counting as a v0.9.1 deferral in §10. I recommend (a) — smaller change, solves the ambiguity at the point of display.

### PEER-SF-006 SRC-23 forbidden-symbols list includes `re.compile(` / `re.match(` / `re.search(` — needs clarification about `re.search` false-positive risk

**Severity:** should-fix

**Evidence:**

§2.3 line 166:

> `re.compile(` / `re.match(` / `re.search(` | **Search-specific:** regex is a v0.9.1 non-goal...

SRC-23 test: `AST + substring grep over folio/search.py (§2.3 forbidden list, includes re.*).`

A literal substring grep for `re.search(` or `re.match(` could false-match in docstrings, comments, or error-message strings mentioning these functions (e.g., an error "did you mean re.search?"). **Reasoning:**

Safest approach: AST walk for `ast.Call` whose `.func.attr in {"search", "match", "compile"}` and `.func.value.id == "re"`, plus a parallel check that `import re` / `from re import` is absent from the module's AST imports.

**Proposed closure:**

Rewrite SRC-23 to be explicit about the check strategy:

> SRC-23 | `test_no_forbidden_symbols_in_module` | AST walk over `folio/search.py`: (a) no `ast.Import`/`ast.ImportFrom` nodes import `re` or its members; (b) no `ast.Call` nodes where `func` resolves to `re.search`, `re.match`, `re.compile`; (c) substring grep for the other §2.3 forbidden symbols (`pickle.loads`, `eval(`, `exec(`, `compile(`, `os.system`, etc.). And add a note in §2.3 explaining the AST-vs-grep split.

### PEER-SF-010 SRC-20 description "called exactly once per view pre-filter" — pre-filter relative to what?

**Severity:** should-fix

**Evidence:**

§9.1 SRC-20:

> `test_derive_trust_status_called_once_per_view` — Monkeypatch-spy: called exactly once per view pre-filter.

"Pre-filter" is ambiguous. The pseudocode (§3.1 lines 322-341) shows:
1. `views, counts = collect_pending_relationship_proposals(...)` — returns views
2. `matched_views = [v for v in views if _view_matches(v, needle)]` — QUERY filter
3. `matched_views = [v for v in matched_views if v.producer == producer]` — producer filter
4. `findings = [_view_to_finding(v) for v in matched_views]` — where `_view_to_finding` calls `derive_trust_status(view)`

So `derive_trust_status` is called once per **finding** (post-filter), not per view (pre-filter). If SRC-20 asserts "once per view returned by collector," the test fails because post-filter views may be fewer.

Looking at sub-slice 2's SYN-19 (spec line 695):

> Monkeypatch-spy over `folio.synthesize.derive_trust_status`; run `synthesize()` over 5 views; assert spy called exactly 5 times with each view.

Synthesize has NO QUERY filter, so 5 views → 5 findings → 5 calls. Trivially "once per view" == "once per finding."

For search with QUERY filter, these diverge. The spec should clarify which.

**Reasoning:**

If the implementer follows the pseudocode literally (construct findings only for matched views), `derive_trust_status` is called once per matched view (i.e., per finding). This is efficient.

Alternative: compute `trust_status` for ALL collector views upfront, use it in `_view_matches` (if trust_status is a searchable field, which it isn't per §3.6 non-searchable list), then drop findings that didn't match. This would call `derive_trust_status` once per collector view (pre-filter).

Neither is wrong, but the test and implementation must agree. SRC-20's "pre-filter" phrasing implies the second option, but the pseudocode and efficiency concerns favor the first.

**Proposed closure:**

Clarify SRC-20:

> SRC-20 | `test_derive_trust_status_called_once_per_matched_view` | Monkeypatch-spy over `folio.search.derive_trust_status`; run `search()` over 10 views where QUERY matches 5; assert spy called exactly 5 times (once per matched view that becomes a finding). Regression guard against re-computing trust_status for views that don't surface.

And adjust §3.1 pseudocode comment or add a note:

> Note: `derive_trust_status(view)` is called at finding-construction time (inside `_view_to_finding`), i.e., after QUERY + --producer filtering. Pre-filter views that don't match QUERY never have their trust_status computed. Efficient; consistent with the shared-helper being a pure projection from view.
