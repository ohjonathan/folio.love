# B.1 Peer Review -- emission-time-rejection-v0-6-3

## Findings

| ID | Severity | Description | Suggested Fix |
|----|----------|-------------|---------------|
| P-1 | blocker | **`_suppress_rejected_proposals` return type change breaks existing callers without migration path.** The spec changes the return from `list[RelationshipProposal]` to `tuple[list[RelationshipProposal], int]`. Line 723-727 in `enrich.py` currently assigns `proposals = _suppress_rejected_proposals(...)`. The spec's new flow (section 6.2) shows the tuple-unpacking call, but the spec does not account for any other callers or test imports that call the function directly. The three existing tests in `TestRejectedProposalSuppression` (test_enrich.py lines 525, 554, 583) call `_suppress_rejected_proposals` and assert on the return value as a list (e.g., `assert len(result) == 0`). Section 9.1 acknowledges updating assertions for `test_suppresses_same_basis` and `test_force_still_suppresses_same_basis` to check for `lifecycle_state == "suppressed"` instead of empty list, but does not mention that these tests must also unpack the tuple return. If the test modifications in section 9.1 are not explicit about the destructuring, the implementer may miss this, causing all three existing tests to fail on the type change alone. | Add an explicit note to section 9.1 that all three existing test assertions must destructure the tuple return: `result, count = _suppress_rejected_proposals(...)`. State the expected `count` value for each test (1 for same-basis tests, 0 for changed-basis test). |
| P-2 | blocker | **`_count_library_queued_proposals` reads frontmatter from every document in the registry on every enrichment call, but the spec does not address the cost within the existing `enrich_note` exception boundary.** Lines 740-743 show the entire relationship evaluation block is inside a `try/except Exception` that catches failures and sets `relationships_axis = EnrichAxisResult(status="error")`. The new `_count_library_queued_proposals` call (section 6.2) reads `registry.json` and iterates all documents' frontmatter. If any document has corrupted frontmatter (malformed YAML, missing file), this could raise an exception that aborts the entire relationship pipeline for the current document, even though the rejection-memory and singular-supersedes steps already succeeded. The spec should state whether `_count_library_queued_proposals` failures should be caught independently (with a fallback of cap-not-enforced) or are intentionally allowed to propagate to the outer handler. | Add an explicit error handling policy for `_count_library_queued_proposals`. Recommended: wrap in its own try/except, log a warning, and default to `existing_queued = 0` (fail-open, consistent with the L2 failure policy). Document this as a design decision. |
| P-3 | should-fix | **`_preserve_rejected_proposals` is called after `_enforce_queue_cap` but its implementation is not specified.** Section 6.2 introduces `preserved_rejected = _preserve_rejected_proposals(existing_meta)` as a new function call, but the spec provides no function signature, no implementation pseudocode, and no description of what `existing_meta` means in this context (is it `_get_enrich_meta(fm)` or the raw `fm`?). Compare this to `_suppress_rejected_proposals` and `_enforce_queue_cap`, which both receive detailed signatures and step-by-step logic in sections 4.2 and 5.3. The implementer must infer the entire function from context. | Add a section (e.g., 3.4 or a new section 6.4) with the function signature, input type (`existing_meta` = output of `_get_enrich_meta(fm)`), filtering logic (select proposals where `lifecycle_state == "rejected"`), and return type (`list[dict]` -- raw dicts, not `RelationshipProposal` objects, since these are passed directly to `all_proposal_dicts`). |
| P-4 | should-fix | **Queue-cap tie-breaking sort is underspecified for equal confidence.** Section 5.3 step 4 says: "Sort by confidence (`high` before `medium`), then by list position (first-emitted wins)." However, the proposals list at this point has been through `_enforce_singular_supersedes`, `_suppress_rejected_proposals`, and `_remove_promoted_proposals` -- the original emission order may not correspond to list index. If the sort is stable (Python's `sorted` is stable), list position is preserved for equal keys, but the spec should explicitly state that a stable sort is required. If an implementer uses an unstable sort, tie-breaking becomes nondeterministic. | Add "(stable sort required)" to section 5.3 step 4, or specify the sort implementation as `sorted(queued, key=lambda p: 0 if p.lifecycle_state == "queued" and p.confidence == "high" else 1)` or equivalent stable-sort pattern. |
| P-5 | should-fix | **`proposals_count` semantic change (section 7.3) is described but not tested.** The spec says `proposals_count` should count only `queued` proposals instead of all proposals. This changes the value surfaced by the CLI's summary output. But neither section 9.1 (existing test modifications) nor section 9.2 (new tests) includes a test that asserts `proposals_count` reflects only queued proposals. This is a user-visible semantic change (the CLI shows "N proposals" to operators) that could regress silently. | Add a test (e.g., S3-10) that constructs an `EnrichResult` with a mix of queued, suppressed, and rejected proposals, and asserts `proposals_count` equals only the queued count. |
| P-6 | should-fix | **`existing_meta` parameter in the new flow (section 6.2) is used but not defined in the surrounding code.** The current code at line 723-726 uses `_get_enrich_meta(fm)` inline. The spec's new flow at section 6.2 passes `existing_meta=existing_meta` to `_suppress_rejected_proposals`, implying a variable `existing_meta` is assigned before the flow begins. But section 6.2 does not show where `existing_meta = _get_enrich_meta(fm)` is assigned. This could lead to a NameError if the implementer copies the flow verbatim. Similarly, `_preserve_rejected_proposals(existing_meta)` uses the same unassigned variable. | Show `existing_meta = _get_enrich_meta(fm)` at the top of the section 6.2 flow, before the `_suppress_rejected_proposals` call. |
| P-7 | should-fix | **`_enforce_queue_cap` receives proposals that may include non-`RelationshipProposal` types after `_remove_promoted_proposals`.** `_remove_promoted_proposals` (line 730) filters proposals, but section 5.3 step 1 says "Separate proposals into `queued` ... and `non_queued`" by checking `lifecycle_state`. If any proposal object has an unexpected `lifecycle_state` value (e.g., from a future state addition), it would fall into `non_queued` and be returned unchanged -- which is safe. But the spec should confirm that `_enforce_queue_cap` only sets `lifecycle_state = "suppressed"` on proposals that were `"queued"`, never on proposals already in another state. The current description implies this but does not make it explicit. | Add a sentence to section 5.3: "Only proposals with `lifecycle_state == 'queued'` are eligible for cap suppression. Proposals already in any other lifecycle state pass through unchanged." |
| P-8 | informational | **S3-4 boundary test arithmetic assumes specific counts.** Test S3-4 says "18 existing queued + 5 new = 2 queued + 3 suppressed." This is correct (cap 20, remaining capacity = 2), but the test would be clearer if it also validated that the 2 surviving proposals are the highest-confidence ones. If all 5 are the same confidence, the test cannot verify tie-breaking by emission order without checking specific proposals. | Consider making S3-4 use mixed confidences (e.g., 3 high + 2 medium) so the assertion can verify both the count and the priority ordering. |
| P-9 | informational | **The `QUEUE_CAP` constant placement rationale (section 8) is sound, but the value 20 should be cross-referenced to the parent spec section.** Section 8 says the value comes from section 9.1 of the parent spec but does not include a code comment or docstring referencing the source. | Add a comment: `# Per tier4_discovery_proposal_layer_spec.md section 9.1` next to the constant definition. |

## Verdict

Needs Fixes

## Rationale

Two blockers prevent approval.

P-1 is a straightforward oversight: the return type change to `_suppress_rejected_proposals` affects existing test callsites that the spec acknowledges but does not fully specify how to update. The tuple-unpacking requirement must be made explicit or the existing tests will fail with a `TypeError` or incorrect assertion on the tuple object.

P-2 is a design gap: the queue-cap counting function introduces I/O (reading frontmatter from every registered document) inside an existing exception boundary. The spec must declare the failure policy for this new I/O path. The project's three-layer failure policy (L1 raise, L2 catch+log+fallback, L3 degrade) provides a clear framework -- the spec should state which layer applies.

The five should-fix items address implementation gaps that would slow Phase C or risk subtle bugs: the missing `_preserve_rejected_proposals` specification (P-3), the underspecified sort stability (P-4), the untested `proposals_count` semantic change (P-5), the undefined `existing_meta` variable (P-6), and the implicit cap-suppression eligibility rule (P-7). None are architecturally blocking, but each would require the implementer to make an unguided decision.
