# Adversarial Review — folio-graph-generalized-proposals-v0-7-1 / B.1 / codex

## 1. Assumption attack
| Assumption | Why it might be wrong | Impact if wrong |
|------------|------------------------|-----------------|
| A renderer-only retrofit can claim a stable shape for future `proposal_type` values. | The proposed code still lives inside the current relationship-only loop and hardcodes `proposal_type: "relationship"` ([spec](../specs/v0.7.1_folio_graph_generalized_proposals_spec.md):44, 120-142). | Future merge or diagram findings can miss required fields while the relationship fixture remains green. |
| `schema_gate_result: None` is honest because no schema gate exists today. | Parent section 5 defines this as relation-rule checks before review ([parent](../specs/tier4_discovery_proposal_layer_spec.md):137), and target-existence is already computable from the registry. | Invalid proposals can be surfaced for review with an explicit "no gate" marker instead of a warning. |
| `subject_id` can change from `proposal_id` to `None` with only CHANGELOG guidance. | Current `graph_doctor()` uses `subject_id` for `view.proposal.proposal_id` ([graph.py](../../folio/graph.py):168-170), and the spec gives inconsistent replacement tuples. | Downstream JSON consumers lose the only proposal identifier in the finding row. |
| `basis_fingerprint` is equivalent to parent `input_fingerprint`. | Parent section 7 requires relation kind and producer identity ([parent](../specs/tier4_discovery_proposal_layer_spec.md):173-184), while current basis computation omits relation and producer ([enrich.py](../../folio/enrich.py):1204-1208). | Fingerprint-based identity can collide across distinct relationship claims. |

## 3. Diagram completeness attack
No diagrams are present in this narrowing spec, so I found no prose/diagram mismatch. The main completeness gap is prose/test mismatch: the spec claims a future-stable generalized proposal surface, but the pseudocode and tests remain relationship-only.

## 4. Edge case inventory
- Pending relationship proposal with `target_id` not present in `registry.json`.
- Two proposals with the same source, target, producer, and basis but different `relation`.
- `flagged_inputs=None`, `[]`, `["source"]`, `["target"]`, `["both"]`, `["inputs"]`.
- Relationship-only fixture with no non-proposal findings, which would make scope-regression assertions too weak.
- Future entity-merge or diagram finding introduced without passing through a common proposal-contract renderer.
- Comment/docstring containing `"proposal_type"` or `"trust_status"` without adding the output keys.

## 5. Security surface
No authN, authZ, injection, secret, SSRF, deserialization, or race-condition security issue surfaced in this spec slice. The closest operator-safety issue is data integrity: invalid proposal targets can be presented without a schema warning.

## 6. Blind-spot identification
The spec is treating "field appears in relationship proposal JSON" as equivalent to "shared proposal contract is generalized." That is the blind spot. Generalization needs either a common renderer boundary or explicit per-`proposal_type` fixture coverage. The current plan adds fields to one existing dict and relies on follow-up slices to behave the same way.

## 7. Risk-assessment override
I do not agree with the author's `low` risk rating. This should be `medium`: the code touch is narrow, but the deliverable intentionally changes a JSON field's semantics, claims future generalized stability, and locks `schema_gate_result: None` even where a minimal rule failure is already computable.

### Detailed findings

**ADV-B1.** The direct-run reproduction used only current public code: `FolioConfig.load`, `graph_doctor`, and a temporary library with a queued relationship proposal. Because `registry.json` already loads `all_ids` in `graph_doctor()` ([graph.py](../../folio/graph.py):160-164), target-existence is available at the same renderer layer the spec modifies. Locking `schema_gate_result is None` for all relationship proposals turns a computable relation-rule failure into an explicit absence of gate data.

**ADV-SF1.** The spec's own language conflicts: line 56 says legacy keys are retained unchanged, line 87 says the change is purely additive, and line 146 says `subject_id` is deliberately breaking. The right shape is not to overload `subject_id` again, but to preserve machine identity through a new `proposal_id` key before nulling section-5 `subject_id`.

**ADV-SF4.** PR-A-14 is directly relevant: direction preservation lived outside `basis_fingerprint` in the rejection key, not inside the fingerprint. This v0.7.1 spec should not reverse that lesson by telling JSON consumers to treat `input_fingerprint` as a proposal identifier.
