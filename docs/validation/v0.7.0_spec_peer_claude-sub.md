## 4. Self-consistency check

The spec is mostly internally consistent. Notable deltas found:

1. **API-name drift between §3.2 / §4 and the codebase.** Both sections
   reference `LLMConfig.profile_for_task`, but the actual method is
   `LLMConfig.resolve_profile`. Internally consistent (the spec uses the
   same wrong name throughout) but inconsistent with the codebase
   (PR-3 above).

2. **CLI shape inconsistency.** §3.3 declares
   `@click.argument("scope", required=True)`, but the established
   `ScopeOrCommandGroup` pattern (which §3.3 explicitly cites as the
   reference) uses `@click.option("--scope", default=None, hidden=True)`.
   §4 then describes scope as a "Required positional", which is
   compatible only with the `argument` form — not the option form needed
   for `ScopeOrCommandGroup`. The spec needs to commit to one model and
   describe how required-validation is implemented in the body
   (PR-1 + PR-4 above).

3. **CF-1 framing vs. cli.py reality.** §8.4 row 3 says rerun guidance is
   "DEFERRED to a follow-up slice", but cli.py:1055 already emits a
   guidance message for analysis rows under refresh. The deferral is
   really about wording, not existence (PR-8 above).

4. **§3.2 `_call_llm` error contract is undefined.** The helper table
   says `_call_llm(config, prompt, llm_profile) -> str` but §11 commits
   to "LLM call raises … exit non-zero". The retry orchestration for
   §11 row 6 (one retry on missing heading) is not assigned to any
   listed helper either (PR-6 above).

5. **§9.3 condition row 4 is "(impossible)" but is not tested as
   impossible.** The spec declares the row impossible by construction
   but offers no test that would fire if a future change broke the
   invariant (e.g., a regression where `--include-flagged` and
   `excluded_flagged_count` could both be non-zero in a hybrid mode).
   Minor — PR-10 above.

6. **§3.1 weekly `include_flagged` parameter is "accepted but not
   consulted"** and §12 row 4 confirms this. DG-CLI-14 asserts the
   silent-accept path. Internally consistent ✓.

7. **§7 path-stability invariant.** Same `(scope, period, type)` triple
   → same `(digest_id, path)` deterministically. §3.2's
   `_compute_digest_id` and `_compute_digest_path` are pure functions
   over those inputs; consistent ✓. (Note: §7's `client` field source —
   "From `engagement_root` (resolved client token)" — assumes
   `_resolve_engagement_scope` returns `client` as a sanitized-or-raw
   string. The helper-table entry says it returns `(client, engagement,
   engagement_root)` without specifying whether tokens are
   pre-sanitized; PR-11 above touches this.)

8. **§13 test plan vs. §3.2 helper table coverage.** Helpers
   `_render_frontmatter`, `_atomic_write`, and `_register_digest` are
   listed in §3.2 but have no dedicated DG-* test (DG-17/18/19 cover
   them transitively through `generate_daily_digest`). This is fine for
   `_render_frontmatter` (covered transitively), but `_atomic_write`
   and `_register_digest` benefit from dedicated unit tests for the
   failure paths (PR-7 + PR-9).

9. **F-050 disposition is correct.** The spec's commitment to use
   `analysis_docs` read-only is the right call:
   `analysis_docs.create_analysis_document` (a)
   `raise FileExistsError` on rerun (line 139–140) — incompatible with
   digest's version-incrementing rerun, (b) uses `datetime.now()` in
   `build_analysis_id` (line 36) — incompatible with `--date`
   parameterisation, (c) writes to `analysis/<subtype>/<id>.md` flat
   (line 57–64) — incompatible with digest's `analysis/digests/<id>/<id>.md`
   per-folder layout, and (d) hardcodes `status: "active"` (line 148) —
   inconsistent with digest's `status: "complete"`. Extending
   `analysis_docs` would require manifest amendment + four breaking
   changes to existing callers. Verified ✓.

10. **DG-REG-1..4 are not pure tautologies.** `tests/test_analysis_docs.py`
    already covers the generic source-less analysis flow under refresh
    (line 200–203) and rebuild (line 219–245). The DG-REG tests
    re-assert these behaviours specifically with `subtype: digest`
    rows, which adds value as a canary against future digest-specific
    regressions (e.g., if a future scan/refresh added `if subtype ==
    "digest"` special-casing). They are duplicative with existing
    coverage but not tautological. Acceptable as written.

## 5. Verdict

**Needs Fixes.**

Two **blockers** prevent Phase C from implementing the spec verbatim:

- **PR-1.** `@click.argument("scope", required=True)` is incompatible
  with `ScopeOrCommandGroup.parse_args` (cli.py:30–33), which rewrites
  positional args into `--scope` options. The two existing consumers of
  `ScopeOrCommandGroup` use `@click.option("--scope", default=None,
  hidden=True)`. The spec's CLI design will not function as written.

- **PR-2.** §3.3 cites cli.py:1174 as the "enrich group" pattern. Line
  1174 is the `provenance` group, and `enrich` is a flat
  `@cli.command()` not a group. The load-bearing pattern reference is
  doubly wrong, which both undermines the §3.3 design rationale and
  signals that the spec author did not verify the citation.

Six **should-fix** items (PR-3 through PR-8) cover the invented
`profile_for_task` API name, scope-argument-shape consistency,
`_call_llm` error-contract ambiguity, range-truncation in the
`config.py` citations, the tempfile-leak bug in `_atomic_write`, and
CF-1's misrepresentation of the existing refresh guidance message.

Seven **minor** items (PR-9 through PR-15) cover test-coverage gaps and
small documentation precision issues.

The F-050 disposition (use `analysis_docs` read-only + reimplement
helpers) is **correct**: extending `analysis_docs.create_analysis_document`
would require four breaking changes plus a manifest amendment.

The DG-REG-1..4 verification tests are **meaningful, not tautological**:
they re-assert the source-less analysis behaviour specifically for
`subtype: digest` rows, providing canary coverage against future
digest-specific regressions in pre-existing commands.

CF-1 (refresh rerun-guidance message) should be **re-disposed**, not
deferred-as-written: the existing message at cli.py:1055 already
satisfies design §8 rule 3 in spirit; the gap is wording specificity.
Either close CF-1 as "satisfied by existing guidance" or scope-expand
to a one-line cli.py edit (with manifest amendment).

The cardinality assertions (G-cardinality-1, G-cardinality-2) are
addressable, the test-plan structure is sound, and §§5/6/7/8/9/10/11/12
are otherwise implementable once the two blockers and the
`profile_for_task` rename are resolved.

Phase C cannot start until PR-1, PR-2, and PR-3 are addressed.
