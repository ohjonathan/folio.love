## Targeted Checks

### ADV-001: schema bump

Closed. §3.1 uses `_schema_version: 1` in the example and states "No schema-version bump." §4.1 repeats that `_empty_entity_registry()` adds `rejected_merges` with no bump and that `load()` applies `setdefault("rejected_merges", [])`. The risk row now relies on `save()` preserving unknown top-level keys and notes that downgrade works because v0.6.5 files stay at version 1.

### ADV-002: lock

Closed for the original race. §4.2 now requires `library_lock(library_root, "entities")` around `suggest-merges` read work and the full `reject-merge` load-mutate-save path. T-15 exists. The test wording is wrong against current `folio/lock.py`; see R2-ADV-001.

### ADV-003: graph_status inventory

Closed for original inventory. §4.3 enumerates both current `suggest_person_merges()` consumers in `folio/graph.py`: `graph_status()` lines 126-134 and `graph_doctor()` lines 224-260. T-11b exists and asserts the CLI label `Reviewable duplicate person candidates`. The label commitment is clean in §4.3/T-11b, but the changelog does not carry the operator-facing change; see R2-ADV-002.

### ADV-004: fingerprint encoding

Closed. §3.2 uses canonical JSON with `sort_keys=True`, `separators=(",", ":")`, and `ensure_ascii=False`. T-4 now covers the delimiter-injection case that Round 1 called out.

### ADV-005: already-merged keys

Closed. §4.1 and §4.2 commit to a stale-candidate `EntityRegistryError` / non-zero CLI error, and T-9c covers the CLI behavior.

### `count_entity_merge_suppressions()`

No issue found. §4.1 specifies `EntityRegistry.count_entity_merge_suppressions() -> int` as a scalar count of stored rejection records, used only for the `M total rejections recorded` disclosure. T-12 directly tests the scalar helper independent of `suggest_person_merges()`. I do not see lingering coupling to the former dict-shaped helper.

## New Findings

### R2-ADV-001 (should-fix): T-15 specifies blocking wait semantics that `library_lock` does not provide

Spec references: §4.2, §7.1 T-15.

Source evidence:

- v1.1 §4.2 correctly requires `library_lock(library_root, "entities")` around `reject-merge`.
- T-15 says: "Pre-acquire `library_lock(library_root, "entities")` in one process; `folio entities reject-merge` in another blocks until released."
- Current `folio/lock.py` does not block until release. On a live non-stale `.folio.lock`, it raises `LibraryLockError("library lock already held ...")` immediately (`folio/lock.py:69-84`).

Failure mode:

An implementer following T-15 literally will either write a failing test against the current lock behavior or broaden `library_lock` into a waiting lock, which would silently change all existing consumers of the shared lock. The lost-update fix does not require wait semantics; it only requires preventing concurrent read-modify-write.

Required fix:

Choose one explicit lock policy:

1. Preferable for this slice: update T-15 to assert that a held `.folio.lock` prevents `reject-merge` from entering the critical section and exits non-zero with the existing lock-held error.
2. Or, if wait-until-release is intended, update §4.2 to declare a broader `library_lock` semantic change, define timeout / interrupt behavior, and add regression coverage for existing lock users.

### R2-ADV-002 (should-fix): `graph status` label/semantics change is not called out in the CHANGELOG

Spec references: §4.3, §4.4, §6, §7.1 T-11b.

Source evidence:

- §4.3 changes `graph_status()` duplicate counting to filtered "reviewable" semantics and says to change the CLI label from `Duplicate person candidates: {N}` to `Reviewable duplicate person candidates: {N}` when observable.
- T-11b asserts the CLI label is `Reviewable duplicate person candidates`.
- §6 risk row says: `CHANGELOG entry calls out the semantic change`.
- §4.4 CHANGELOG block mentions `folio entities suggest-merges` and `folio graph doctor`, but does not mention `folio graph status`, the filtered count semantics, or the label rename.
- Current CLI renders `Duplicate person candidates: {status.duplicate_person_candidates}` at `folio/cli.py:1865`, so the label change is real operator-facing output.

Failure mode:

Operators or scripts scraping `folio graph status` for `Duplicate person candidates:` get a silent label break. The spec makes the rename defensible for clarity, but the changelog currently fails to warn about it even though §6 claims it does.

Required fix:

Update §4.4 CHANGELOG `Changed` bullets to explicitly state that `folio graph status` now reports filtered reviewable duplicate candidates and that the label changes to `Reviewable duplicate person candidates`. If the project wants to avoid breaking automation, specify a backwards-compatible output strategy instead, such as retaining the old label and adding a second explanatory line.

### R2-ADV-003 (minor): Graph call sites rely on a default for a user-facing semantic

Spec reference: §4.3 line 166.

§4.3 says both graph call sites use `suggest_person_merges()` with no explicit parameter because the default is the contract. That is coherent in v1.1 because `apply_rejection_memory=True` is pinned and T-11/T-11b should catch drift. It is still the same class of latent call-site coupling that caused ADV-003: if a future slice changes the default, `graph status` and `graph doctor` silently change again.

Recommendation: require explicit `apply_rejection_memory=True` in `graph_status()` and `graph_doctor()` even though it matches the default. The call sites are operator surfaces; making the semantic visible in code is worth the small verbosity.
