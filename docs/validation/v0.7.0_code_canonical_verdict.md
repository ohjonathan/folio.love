## Blocker (must close in D.4)

### B-201 — §10.4 orphan self-heal not implemented before LLM call

**Source:** codex ADV-201 (with peer PR-203 + gemini ALIGN-201 corroborating).

**Evidence:** codex provided static-inspection + direct-run reproduction:
1. Pre-create digest file with valid frontmatter
2. Leave digest row absent from registry.json
3. Patch `_call_llm` to raise
4. Run `generate_daily_digest`
5. **Expected:** registry has digest entry (self-healed before LLM call)
6. **Observed:** registry does NOT have digest entry (file remains orphaned)

**Spec §10.4 row 5 ("Orphan: file exists, registry row absent — SF-4 self-heal"):** "Before LLM call, build RegistryEntry from existing-file frontmatter and call _register_digest to upsert. Then proceed with rerun. Eliminates the orphan state."

**Required fix in D.4:** In `generate_daily_digest` and `generate_weekly_digest`, after computing `digest_id`/`path` and BEFORE building the LLM prompt:
1. Load registry.
2. If `path.exists() AND digest_id NOT IN registry["decks"] AND _load_existing_digest(path) returns dict`:
   - Call `_register_digest(...)` immediately with the existing-file frontmatter to recreate the registry row.
   - Continue with the rerun (LLM call + write + final upsert).

Add tests DG-20c (daily orphan self-heal pre-LLM verified by reading registry after a patched-LLM-failure rerun) and DG-WEEKLY-ORPHAN equivalent.

## Should-fix (close in D.4)

### SF-201 — §10.4 stale-registry warning missing (ALIGN-202)

**Source:** gemini ALIGN-202.

§10.4 row 6 specifies "Stale: registry row exists, file absent — Treat as fresh write (version: 1); registry will be overwritten by the new entry. Stderr warning (`↷ Registry referenced missing file at {path}; regenerating.`)." Implementation treats it as fresh write but does NOT emit the warning.

**Fix:** Detect the stale-registry case in `generate_*_digest` and `logger.warning("↷ Registry referenced missing file at %s; regenerating.", path)`. Add test DG-19c.

### SF-202 — `--date` parse error wording (PROD-201)

**Source:** product PROD-201.

`digest.py:166-168` raises `invalid --date '...': expected YYYY-MM-DD format` (lowercase 'i', no period). Spec §11 row 2 prescribes `Invalid --date '{value}': expected YYYY-MM-DD format.` (capital I, trailing period).

**Fix:** Update `_parse_date` ValueError message to match spec capitalization and trailing-period.

### SF-203 — Empty + --include-flagged echo punctuation (PROD-202)

**Source:** product PROD-202.

Implementation produces `... ClientA/DD_Q1_2026. (--include-flagged was honored; ...)` (period before parenthetical, no terminal period). Spec §4 prescribes `... ClientA/DD_Q1_2026 (--include-flagged was honored; ...).` (parenthetical THEN terminal period).

**Fix:** In `generate_daily_digest`'s empty-path message construction, drop the period before the parenthetical and add terminal period after it. Update DG-CLI-10b assertion to be exact-string (not substring) to catch regressions.

### SF-204 — ATX indented headings evade heading detection (ADV-203)

**Source:** codex ADV-203.

`_heading_positions` matches `^##\s+...` — pure line-start. ATX spec allows up to 3 leading spaces of indentation. An LLM that emits `   ## Trust Notes` (3 spaces) bypasses both validation AND scrubbing.

**Fix:** Update `_heading_positions` regex to `^[ ]{0,3}##\s+...` (0-3 leading spaces per CommonMark ATX). Add test for indented headings.

### SF-205 — Corrective re-call exception loses original validation context (ADV-204)

**Source:** codex ADV-204.

If the corrective LLM re-call raises (transient error), the caught exception propagates with only the new error message; the original missing/duplicate-heading validation context is lost. Operator sees "Cause: <transient error>" instead of "first attempt missed [heading]; retry failed with <transient>".

Also: corrective prompt formats heading names without the `## ` prefix (currently `[missing]` is `["What Moved Today"]` not `["## What Moved Today"]`).

**Fix:** In `_build_body_with_validation_retry`:
1. Capture missing/dups before the corrective call.
2. If the corrective call raises, wrap the exception with `f"Validation failed first response: missing={missing}, duplicate={dups}; corrective call raised: {original_exc}"`.
3. Format heading names in corrective prompt as `## <heading>` (Markdown form per spec).
