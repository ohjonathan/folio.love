# Adversarial Review - v0.6.2 Provenance Lifecycle Rename

Artifact under review: `docs/specs/v0.6.2_provenance_lifecycle_rename_spec.md`

Loaded: `ontos_context_map`, `provenance-lifecycle-rename-v0-6-2`, `proposal-lifecycle-rename-v0-6-1`, `folio/provenance.py`, `folio/cli.py`.

## Verdict

Needs Fixes.

The spec correctly identifies the provenance proposal namespace, correctly excludes pair-processing metadata, and correctly scopes `_mark_enrich_stale()` as the provenance counterpart omitted from v0.6.1. However, the raw-dict pending-state migration rule is not backward-compatible as written. This violates the spec's own "no operator action required" and "no behavior change" commitments.

## Blocking Finding

### B-1: Raw-dict pending readers would drop legacy pending proposals

Spec references: §3.3, §4.1 items 5 and 11.

Affected code sites:

- `folio/provenance.py:271` in `_has_pending_repair_proposals`
- `folio/provenance.py:1535` in `provenance_status_summary`

The spec says raw-dict readers should use a `lifecycle_state` fallback and, for old pending comparisons, "update to compare to `queued`." That is insufficient for legacy frontmatter that still has only:

```python
{"status": "pending_human_confirmation"}
```

With the spec's stated pattern:

```python
state = raw.get("lifecycle_state")
if state is None:
    state = raw.get("status")
if state == "queued":
    ...
```

legacy pending entries produce `state == "pending_human_confirmation"` and fail the comparison.

Impact:

- `_has_pending_repair_proposals()` no longer detects legacy pending repair proposals, so a later provenance run can generate duplicate repair proposals instead of reporting "repair proposal pending review."
- `provenance_status_summary()` undercounts pending proposals in legacy documents after upgrade.
- `collect_pending_proposals()` is not affected because it goes through `ProvenanceProposal.from_dict()`, which maps legacy `status` to `lifecycle_state`.

Required fix: for raw-dict pending readers, either apply `_STATUS_TO_LIFECYCLE` after fallback or explicitly dual-check:

```python
state = raw.get("lifecycle_state")
if state is None:
    state = raw.get("status")
if state in ("queued", "pending_human_confirmation"):
    ...
```

This is the same compatibility principle the spec already applies correctly to `cli.py` §3.4.

## Edge-Case Review

`stale_pending` handling is conceptually sound. It is provenance-specific, should pass through unchanged, and should not be normalized to `stale` in this migration-only slice. Existing behavior already treats `stale_pending` as neither pending nor rejected in status counts, so preserving that value keeps behavior stable.

The §3.4 `_mark_enrich_stale()` dual-check is correct:

```python
state = proposal.get("lifecycle_state")
if state is None:
    state = proposal.get("status")
if state in ("queued", "pending_human_confirmation"):
    proposal["lifecycle_state"] = "stale_pending"
```

This handles both new and legacy pending proposals. One non-blocking cleanup question: the snippet does not remove a legacy `status` key after writing `lifecycle_state`. Because readers prefer `lifecycle_state`, this is behaviorally safe. If the goal is strict storage cleanup whenever touched, the spec should add `proposal.pop("status", None)` after the write.

## Recommended Spec Patch

Update §3.3 and §4.1 items 5 and 11 to say pending raw-dict readers must either map legacy values through `_STATUS_TO_LIFECYCLE` or dual-check `("queued", "pending_human_confirmation")`.

Add one test assertion for each affected raw-dict path:

- `_has_pending_repair_proposals()` returns true for a legacy `status: pending_human_confirmation` repair proposal.
- `provenance_status_summary()` counts a legacy `status: pending_human_confirmation` proposal as pending.

After that change, this spec is approvable.
