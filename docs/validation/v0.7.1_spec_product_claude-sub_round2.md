# Product Review — folio-graph-generalized-proposals-v0-7-1 / B.2 / claude-sub

## 1. User-value assessment

Closure-verification round on spec v1.1. The user, Johnny, runs
`folio graph doctor` in a terminal to triage the pending-proposal queue
before flipping to `folio links review`. In R1 I flagged that v1.0
solved the `--json` scripted-triage job well but regressed the stdout
path (literal `"None"` after freeing `subject_id` for §5 semantics), and
left three should-fixes around breaking-change signaling, `--help` copy,
and trust-aware `recommended_action`.

v1.1's substantive product-relevant moves:

1. Adds `folio/cli.py` to scope-lock (manifest `allowed_paths:53`) and
   rewrites folio/cli.py:2019 via §3.4 pseudocode. The new rendering
   reads `finding.get("proposal_id") or finding.get("subject_id") or "—"`,
   appends `" [flagged]"` when `trust_status == "flagged"`, and appends
   `" [schema-gate: <rule>]"` when the gate fails. PRD-1 is materially
   addressed.
2. Specifies the CHANGELOG entry verbatim in §6 OQ-3, including a
   "Breaking" stanza that names the specific read-site migration
   (`subject_id` → `proposal_id`) and a "Known gaps" stanza that calls
   out the `input_fingerprint` legacy alias and the deferred full
   `evidence_bundle` rendering. PRD-2 is meaningfully closed.
3. Adds a minimal today-computable schema gate (§3.3) that surfaces
   `target_exists` and `supported_relation` failures — a scope expansion
   from R1, initiated by ADV-B1, that changes the product surface for
   operator triage in ways worth evaluating.
4. Pins `evidence_bundle` type semantics via test 15
   (`== list(view.proposal.signals)`) — `signals: list[str]` is
   confirmed at `folio/pipeline/enrich_data.py:79`. PRD-5 resolved.
5. Reconciles the 9-vs-11 prose inconsistency throughout (§1, §2.1,
   §2.2 table, §2.3). PRD-6 resolved.

The closure is strong on PRD-1, PRD-2, PRD-5, PRD-6. Two R1 findings
remain open as should-fixes in v1.1: PRD-3 (`--help` copy) is not
addressed at all, and PRD-4 (`recommended_action` identical for flagged
vs ok) is not addressed — v1.1 keeps `recommended_action` as a single
literal string for all pending proposals. Two new product questions
arise from v1.1's own additions: the `[flagged]` / `[schema-gate]`
annotations need a sanity check for triage clarity, and the minimal
schema-gate introduces a new class of finding (failing proposals that
were previously silently-pending) whose UX flow is not spelled out.

Overall direction: the slice is now closer to delivering operator
value on both JSON and stdout paths. v1.1 does not fully close the R1
slate, but the remaining gaps are `should-fix`, not blocking. The two
new R2 concerns are also `should-fix`, not blocking.

## 8. Positive observations

- **R1 blocker (PRD-1) is cleanly resolved.** The §3.4 pseudocode
  update is minimal, surgical, and bounded by the new `G-scope-3` gate
  (folio/cli.py diff ≤25 lines). The fallback chain
  `proposal_id or subject_id or "—"` handles non-proposal findings
  correctly — orphaned-relation and stale-analysis findings, which
  still carry a `subject_id`, render unchanged. Operator experience
  on the primary terminal path is preserved, and the new annotations
  are additive.
- **CHANGELOG copy is drafted verbatim.** §6 OQ-3 includes the full
  stanza, not a "write this later" promise. This directly addresses
  the R1 PRD-2 copy-at-spec-time concern — one of the harder gaps to
  close retroactively if Phase C authors improvise.
- **"Known gaps" stanza in CHANGELOG** is unusual for a release note
  and sets a good precedent. Explicitly telling operators that
  `input_fingerprint` is a legacy alias and that full
  `evidence_bundle` stdout rendering is deferred to sub-slice 2+ is
  honest and sets expectations for the v0.8.0 sub-slice.
- **Stable naming for `proposal_id`.** The v1.0 → v1.1 transition
  considered `proposal_id` vs alternatives. v1.1 commits consistently:
  §2.1 B1 table row; §3.1 new key in pseudocode; §3.4 stdout fallback;
  §6 OQ-3 CHANGELOG stanza; test names (`test_graph_doctor_proposal_id_preserves_view_proposal_id`);
  `G-cardinality-2` gate. No drift. A consumer reading any one of
  these anchors gets the same name.
- **Manifest scope lock tightened.** Adding `folio/cli.py` to
  `allowed_paths` with a bounded diff gate (`G-scope-3`, ≤25 lines)
  is the right way to broaden scope — the broadening is necessary
  for PRD-1 closure, but the blast radius is bounded by a mechanical
  gate, not trust.
- **Regression guard for R1 closure.** `G-test-3` (CLI-level
  stdout-regression test) and test 21 (`test_graph_doctor_cli_stdout_renders_proposal_id`)
  explicitly guard the R1 blocker; PRD-1 cannot silently re-open in
  Phase C.
- **Schema-gate return shape is forward-compatible.** OQ-4's
  recommendation to keep the single-rule 2-key dict shape
  `{"status","rule"}` for v0.7.1 with a documented path to evolve
  additively is correct product discipline — don't speculate on
  multi-rule failure shape until it's needed.
- **Test surface for new annotations.** Tests 22 and 23 explicitly
  assert stdout renders `"[flagged]"` and `"[schema-gate: target_exists]"`
  literals. Copy is tested, not just specified.

## 9. Verdict

Needs Fixes

R1 blocker (PRD-1) resolved cleanly. R1 should-fixes partially
resolved: PRD-2 materially closed on the CHANGELOG copy axis (still
open on payload-level signaling as PRD-2-R2); PRD-5 and PRD-6 fully
resolved. PRD-3 and PRD-4 are **carried forward unchanged** as
PRD-3-R2 and PRD-4-R2 — v1.1 did not address `--help` copy or
trust-aware `recommended_action`, and the new schema-gate annotations
make PRD-4-R2 materially worse by adding a second decorative state.
Two new R2 should-fixes (PRD-7 on schema-gate UX, PRD-8 on annotation
bracket style) emerge from v1.1's own additions.

This is a "Needs Fixes" at the should-fix tier, not the blocking tier.
The operator experience on both JSON and stdout paths is net-improved
over v1.0 and over the pre-retrofit baseline. Remaining gaps are
copy and recommended-action variation — solvable with spec-level copy
decisions, not structural rework. B.3 consolidator should decide
whether PRD-3-R2 and PRD-4-R2 warrant a v1.2 spec revision (my
recommendation: yes — two rounds of "should-fix" without closure is a
pattern worth breaking; PRD-4-R2 in particular turns the new
annotations into decoration if unaddressed) or can be captured as
Phase C / D.4 fix-summary items with literal copy drafted in the
consolidator verdict.
