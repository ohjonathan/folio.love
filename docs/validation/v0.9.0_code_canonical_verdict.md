## 2. Canonical D.2 should-fix (DCB-NNN) — close in D.4

### DCB-1 — Producer zero-match hint emits wrong message when QUERY matches zero views

**Convergence:** 3-LENS (codex ADV-SF-002 + peer PEER-SF-D2-001 + product PROD-SF-D2-001).
**Evidence:** `folio/search.py:266` — `producer_hint_names = [v.producer for v in query_matched]`. When `query_matched` is empty (QUERY matched no views), `producer_hint_names == []`, and the stdout renderer's empty-list branch prints "Hint: no proposals in scope for any producer" — FALSE when the scope actually has proposals, just none matching QUERY.
**Closure:** Distinguish "QUERY miss" from "producer mismatch." When `query_matched` is empty, set `producer_hint_names` to None so the renderer falls through to the normal "No matches for '<QUERY>'" breadcrumb instead of the misleading producer branch. Only populate `producer_hint_names` when QUERY did match AND --producer filter killed all.

### DCB-2 — `--limit 0` produces contradictory zero-match diagnostics

**Source:** codex ADV-SF-001 (standalone, real defect).
**Evidence:** `folio/search.py:347` — `_render_search_stdout` checks `not report.findings` for the zero-match breadcrumb, but with `--limit 0` that predicate is true even when `total_available > 0`. Result: `folio search framework --limit 0` with real matches prints both "No matches for 'framework'…" AND "(showing 0 of N; use --limit to adjust)" — contradictory.
**Closure:** Gate the zero-match breadcrumb on `report.total_available == 0` rather than `not report.findings`. Reserve the truncation footer for the `report.truncated` condition. Same fix for the producer-hint branch.

### DCB-3 — `_view_to_finding()` is less defensive than `_view_matches()` for malformed views

**Source:** codex ADV-SF-003 (standalone, substantive).
**Evidence:** `folio/search.py:204` `evidence_bundle=list(view.proposal.signals)` raises on `signals=None`; `folio/search.py:206` `flagged_inputs=list(view.flagged_inputs)` raises on `None`; bare attribute access assumes all proposal fields exist.
**Closure:** Add defensive coercions: `list(view.proposal.signals or [])`, `list(view.flagged_inputs or [])`. Add SRC-42 unit test with SimpleNamespace fixture having `signals=None`, `flagged_inputs=None`.

### DCB-4 — SRC-23 forbidden-symbol guard weaker than spec §2.3

**Source:** codex ADV-SF-004 (standalone).
**Evidence:** `tests/test_search.py:501` AST walk blocks only `re`/`fnmatch` imports and `re.*` calls. Misses `regex`/`re2` third-party engines, `compile(`, `importlib.import_module`, `globals()[`/`locals()[` subscripting, `subprocess.*shell=True`, `Path.match`/`Path.glob`.
**Closure:** Extend SRC-23 AST walk to cover all spec §2.3 entries. Close by adding checks for: third-party regex imports (`regex`, `re2`), `compile()` builtin calls, `importlib.import_module()` calls, subscript access on `globals()`/`locals()`, `subprocess.*` with `shell=True`, `Path.match`/`Path.glob` on any `pathlib.Path`-typed object.

### DCB-5 — Public `search(limit=-1)` not enforced

**Source:** codex ADV-SF-005.
**Evidence:** `folio/search.py:216` — function-level doesn't validate `limit ≥ 0`; CLI IntRange is the only guard. Direct `search(config, query="x", limit=-1)` silently uses Python slice semantics.
**Closure:** Raise `ValueError` in `search()` when `limit is not None and limit < 0`. Add SRC-43.
