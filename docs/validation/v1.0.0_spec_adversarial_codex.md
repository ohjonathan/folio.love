## Blockers (ADV-B1-NNN)

### ADV-B1-001 Spec frontmatter is invalid YAML and breaks Ontos activation
**Severity:** blocker
**Evidence:** `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:1` starts YAML frontmatter, and `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:10-11` use unquoted scalar values containing a quoted heading with a colon: `§"Week 26-27: Graph Quality Layer"`. The framework requires artifacts to carry frontmatter at `frameworks/llm-dev-v1/framework.md:308-309`, and project activation requires `ontos map` before work.
**Reproduction:** direct-run. `ontos map` exits 1 with `Error parsing v1.0.0_folio_enrich_diagnose_spec.md: Invalid YAML frontmatter: mapping values are not allowed here ... line 11, column 81`. A focused `yaml.safe_load()` of the frontmatter also raises `ScannerError` at the same line/column.
**Why it blocks:** This artifact cannot be indexed by the project documentation system. The spec also claims `Ontos_Context_Map.md` is an allowed regenerated artifact at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:125`, but regeneration fails because of this spec.
**Proposed closure:** Quote `consumes_parent` and `consumes_roadmap` values or convert them to block scalars / structured YAML fields. Re-run `ontos map` and require exit 0 before B.3.

### ADV-B1-002 The group reshape regresses existing `folio enrich SCOPE` parsing
**Severity:** blocker
**Evidence:** Existing `folio enrich` is a flat Click command with a positional `scope` at `folio/cli.py:1128-1137`. The spec replaces it with `@cli.group(cls=ScopeOrCommandGroup, invoke_without_command=True)` plus `@click.argument("scope", ...)` at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:392-401`. But `ScopeOrCommandGroup.parse_args()` rewrites the first bare token to `--scope` at `folio/cli.py:27-33`, and the working `digest` precedent only works because it defines hidden `@click.option("--scope", ...)` at `folio/cli.py:1179-1180`.
**Reproduction:** direct-run minimal Click reproduction using the existing `ScopeOrCommandGroup` and the spec's `@click.argument("scope")` shape:
`runner.invoke(broken, ["ClientA"])` exits 2 with `Error: No such option: --scope`.
Even after copying digest's hidden `--scope` option, a flat-command-compatible invocation with options before scope regresses: a flat command accepts `["--dry-run", "ClientA"]` with exit 0, while the grouped shape exits 2 with `Error: No such command 'ClientA'`.
**Why it blocks:** The dispatch hard halt says group reshape regression of any existing `folio enrich` invocation is a blocker. This breaks shell history and programmatic callers that rely on the existing flat-command parser.
**Proposed closure:** Do not use the spec's `@click.argument("scope")` group shape. Either keep a parser that accepts bare scope in every flat-command-compatible position, or extend `ScopeOrCommandGroup` and tests to cover at least `folio enrich ClientA --dry-run`, `folio enrich --dry-run ClientA`, `folio enrich --llm-profile X ClientA`, and `folio enrich --force ClientA`.

### ADV-B1-003 Invalid scope/unreadable-registry behavior violates parent §7.7
**Severity:** blocker
**Evidence:** Parent §7.7 requires the same scope semantics as `folio enrich` at `docs/specs/folio_enrich_spec.md:610-616` and says invalid scope, unreadable registry, or invalid config is fatal and exits non-zero at `docs/specs/folio_enrich_spec.md:637-642`. This spec contradicts that by saying a scope arg that matches no decks returns empty findings and exit 0 at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:780`, and a corrupted registry rebuilds in memory with the same non-fatal outcome at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:779`.
**Reproduction:** direct-run against current `plan_enrichment`: in a temp library with one registered `ClientA` note, `plan_enrichment(config, scope="ClientA", dry_run=True)` returns length 1, while `plan_enrichment(config, scope="NoSuchScope", dry_run=True)` returns length 0. The current scope filter just skips non-matches at `folio/enrich.py:294-298`; it does not distinguish "valid empty scope" from a typo.
**Why it blocks:** A mistyped scope can produce `No enrichment hygiene findings.` and exit 0, falsely telling the operator the scoped library is healthy. That is exactly the class of silent graph-quality stall the command is supposed to prevent.
**Proposed closure:** Add an explicit scope-resolution step before calling `plan_enrichment`, and align corrupted/unreadable registry behavior with parent §7.7. Add CLI tests for invalid scope and unreadable/corrupt registry exit non-zero.

### ADV-B1-004 Sub-item B firewall gate misses the CLI path where `--include-flagged` would be added
**Severity:** blocker
**Evidence:** The spec forbids the `--include-flagged` CLI option name at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:146-159`, and §6 says the firewall is manifest-enforced at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:601-613`. The actual manifest gate only scans `folio/enrich.py` at `frameworks/manifests/folio-enrich-diagnose-v1-0-0.yaml:344-349`, while the diagnose CLI option would live in `folio/cli.py`, which is an allowed edit path at `frameworks/manifests/folio-enrich-diagnose-v1-0-0.yaml:76-80`.
**Reproduction:** direct-run synthetic gate test. In a temp tree with clean `folio/enrich.py` and `folio/cli.py` containing `@click.option("--include-flagged", is_flag=True)`, the manifest G-scope-5 equivalent (`! grep -nE 'derive_trust_status|include[-_]flagged' folio/enrich.py`) exits 0 while `cli.py` contains the forbidden flag.
**Why it blocks:** The hard halt condition says any leak in §6's firewall is a blocker. As written, an implementation can pre-merge sub-item B's CLI override on the diagnose command and still pass G-scope-5.
**Proposed closure:** Gate the actual diagnose code path across both `folio/enrich.py` and `folio/cli.py`. Prefer AST or marker-bounded checks over whole-file grep so existing unrelated `digest --include-flagged` does not create false positives. Add an ED-CLI test asserting `folio enrich diagnose --help` does not expose `--include-flagged`.

## Should-fix items (ADV-SF-NNN)

### ADV-SF-001 The additive `trust_status` key needs parent-contract authority
**Severity:** should-fix
**Evidence:** Parent §7.7 says `--json` emits findings with a "fixed schema" of five keys at `docs/specs/folio_enrich_spec.md:626-635`. This spec emits those five keys plus `trust_status` at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:85` and makes `trust_status` an exhaustive per-finding key at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:677-686`, while deferring the parent wording fix to a future rev at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:623` and `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:814`.
**Why it blocks:** Not blocker-grade because the parent five-key minimum is preserved, but a strict consumer or reviewer can reasonably read "fixed schema" as exact.
**Proposed closure:** Either revise parent §7.7 before Phase C to authorize additive keys, or keep the finding object exactly five-keyed and place trust annotation elsewhere in the envelope.

### ADV-SF-002 Text mode is not actually grouped by finding class
**Severity:** should-fix
**Evidence:** Parent §7.7 requires human-readable mode to print actionable rows grouped by finding class at `docs/specs/folio_enrich_spec.md:626-629`. This spec says output is grouped by finding code at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:81`, but the normative sort key is severity then `subject_id` at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:266-301` and repeated at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:544-547`.
**Why it blocks:** With multiple warning codes, `subject_id` ordering can interleave finding classes. Operators lose the grouping the parent contract promises.
**Proposed closure:** Render explicit code-group headings or change the sort to group by `code` first, then severity/subject. Add a CLI test with two same-severity codes whose subject IDs would otherwise interleave.

### ADV-SF-003 Future `conflict` or `analyze` reasons can disappear silently
**Severity:** should-fix
**Evidence:** The spec says diagnose surfaces every non-eligible `(disposition, reason)` tuple at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:555-557`, but `_entry_to_finding` returns `None` for any unrecognized non-protect tuple at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:368-370`. The spec explicitly acknowledges that future `conflict` or `analyze` reasons would be silently omitted at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:572-576`.
**Why it blocks:** This is a known blind spot in a diagnostic command. A future enrich hardening can add a new unsafe conflict reason and diagnose will report a clean run unless the implementer remembers to update this map.
**Proposed closure:** Add a defensive fallback for unknown `conflict` reasons and for unknown non-eligible dispositions, or make `_entry_to_finding` raise in tests for unmapped tuples. Add synthetic tests parallel to ED-15.

### ADV-SF-004 `diagnose_notes(limit=...)` lacks function-level validation
**Severity:** should-fix
**Evidence:** `diagnose_notes` exposes `limit: int | None` at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:240-245`, but truncates with raw slicing at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:269-273`. Only the CLI validates `--limit` via `click.IntRange(min=1)` at `docs/specs/v1.0.0_folio_enrich_diagnose_spec.md:496-500`.
**Why it blocks:** Programmatic callers can pass `limit=0` or `limit=-1` and receive surprising output rather than a contract error. The public module API is part of the spec surface.
**Proposed closure:** Validate `limit is None or limit >= 1` inside `diagnose_notes`; add a module-level test for invalid limits.

### ADV-SF-005 G-cardinality-5 does not prove exact dataclass field sets
**Severity:** should-fix
**Evidence:** The manifest says G-cardinality-5 verifies dataclass field order at `frameworks/manifests/folio-enrich-diagnose-v1-0-0.yaml:374-379`, but the command uses `zip(fields(...), expected_names)` without checking lengths at `frameworks/manifests/folio-enrich-diagnose-v1-0-0.yaml:378`. Extra trailing fields would pass.
**Why it blocks:** This weakens the same schema-drift guard the spec relies on for exact JSON/key order confidence.
**Proposed closure:** Assert exact lists, e.g. `[f.name for f in fields(DiagnoseFinding)] == [...]`, and do the same for `DiagnoseResult` and `DiagnoseSummary`.

## Summary

Verdict is BLOCK. The spec currently cannot be parsed by Ontos, the proposed Click group reshape breaks existing flat-command parsing, the failure policy contradicts parent §7.7 for invalid scopes/unreadable registries, and the sub-item B firewall gate does not cover the CLI path where the forbidden flag would be added. The remaining issues are fixable, but these blockers need closure before Phase C.
