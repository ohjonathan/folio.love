# Adversarial Review — proposal-review-hardening-v0-6-0 / B.1 / claude-sub

## Verdict

**Needs Fixes.** Two blocker-class regressions, each reproducible by static inspection of current code against the spec's own §4 / §7 instructions. The core feature design (rejection filter + dual-source aggregator + warm-up) is sound; the *integration surface* of the return-type change is under-specified and will silently break four internal call sites the spec does not enumerate.

Adversarial lens summary: the spec's §4 pseudo-code is defensible. The spec's §7 migration narrative ("only two internal callers") is **factually incorrect** against the current repo — there are seven call sites across three files, four of which live inside `folio/links.py` itself and none of which §7 acknowledges. A Phase C implementer who reads §7 as the authoritative caller inventory will ship a broken PR.

Risk-rating override: author self-rated `medium`. Adversarial concurs on `medium` for the *feature* risk, but the *implementation-scope* risk around the return-type change is **high** and is currently mis-accounted for in §8's risk table (which rates "Return-type change breaks an unexpected caller" as Low/Low). That rating is wrong on this evidence.

## Should-fix findings

| ID | Description | Location | Evidence |
|----|-------------|----------|----------|
| SF-1 | §4.3 aggregator's "both sources" read treats `_llm_metadata.links.confirmed_relationships` as a separate namespace from producer keys, but `_llm_metadata` literally contains `links` as a first-class key (set by `_ensure_confirmation_meta` at folio/links.py:202). A producer named `"links"` (not currently possible but not guarded against) would collide, and more subtly the aggregator's iterator in §4.3 must explicitly skip the `"links"` key when iterating `_llm_metadata.items()` to avoid treating confirmed records as a malformed producer axis. Spec §4.3 does not say this. | folio/links.py:202; docs/specs/v0.6.0_proposal_review_hardening_spec.md §4.3 | static-inspection |
| SF-2 | §7 "Backwards-compatibility for stored frontmatter" claims rejection entries from prior versions "have the expected shape (`status: rejected`, `basis_fingerprint: <string>`)" — but §4.2's filter uses `.get()` on `source_id` and `target_id` which are populated at *read time* by `_proposal_from_raw`, not stored (folio/links.py:57–58). In the pseudo-code at spec lines 93–94, `p["source_id"]` / `p["target_id"]` are direct-dict indexes that will `KeyError` on entries where `source_id` was not explicitly stored. Current `_iter_producer_proposals` avoids this by letting `_proposal_from_raw` populate missing keys via `setdefault`. The §4.2 pseudo-code does not route through `_proposal_from_raw`, so it is susceptible. Filter will either (a) crash on older frontmatter or (b) see mismatched keys (stored target_id but missing source_id) — false-negative for the filter. | docs/specs/v0.6.0_proposal_review_hardening_spec.md §4.2 pseudo-code (lines 93–101); folio/links.py:55–67 | static-inspection |
| SF-3 | §4.3 aggregator's "future-proofing" filter excludes `suppressed` and `stale` states from the denominator, but current code only writes `pending_human_confirmation` and `rejected`. The test `test_doctor_aggregator_excludes_suppressed_and_stale` (§4.6 #5) seeds these states manually. Issue: this test *asserts behavior the spec deliberately declares as future-proofing-only* — if a later slice adds `suppressed`/`stale` semantics with different aggregation rules (e.g., `suppressed` counted as an implicit rejection), this test locks in the wrong behavior and becomes a false-pass gate for the wrong policy. Document the test's stability contract or drop it until the state model is committed. | docs/specs/v0.6.0_proposal_review_hardening_spec.md §4.3 (Constraints bullet 3), §4.6 test 5 | static-inspection |
| SF-4 | §4.3 warm-up threshold = 10 hardcoded across `ProducerAcceptanceRate.warmup`, doctor output rendering, and test assertions. No single named constant. A later slice that wants to tune the threshold (per proposal rev 4 §9.2's explicit invitation to revisit) has to grep and edit three sites and five tests. Define `_WARMUP_REVIEWED_COUNT = 10` (or similar) as a module-level constant in `folio/graph.py` and reference it in both code and test. | docs/specs/v0.6.0_proposal_review_hardening_spec.md §4.3, §4.6 tests 1–3 | static-inspection |
| SF-5 | §9.2 of the proposal doc (rev 4) notes that rejection does not write a `rejected_at` timestamp. The spec under review correctly does not assume timestamps. But §9 Exclusion List bullet "Does NOT introduce `rejected_at` timestamps or any new frontmatter fields" means the rejection-memory filter has no tiebreaker: if a producer re-emits a rejected proposal with an identical fingerprint, the operator's only signal that this is the *same old rejection* is that the count-suppressed-by-rejection-memory non-zero line shows up. The review surface shows `"N proposals suppressed by rejection memory."` but does **not** show *which* proposals were suppressed, so an operator debugging "why isn't my expected proposal in the pending queue?" has no path to find the culprit. Add either (a) a `--show-suppressed` flag that surfaces suppressed-proposal IDs, or (b) note in §5 OQ as a committed follow-up. | docs/specs/v0.6.0_proposal_review_hardening_spec.md §4.2, §4.4, OQ-2 | static-inspection |
| SF-6 | The diagram at §10.1 lists "relationship-extractor-v1" and "entity-merge-suggester" as example producers in the doctor-rendering illustration (spec lines 146–149), but §2 explicitly scopes entity-merge rejection memory *out* of this slice. A reviewer reading the diagram will reasonably infer that `entity-merge-suggester`'s 2/8 row is within-scope for this slice; it is not. Rewrite the example to use two relationship-producer names only, or add a footnote that entity-merge rows are illustrative for the aggregator surface only (entity-merge *filter* lands in a follow-up slice). | docs/specs/v0.6.0_proposal_review_hardening_spec.md §10.1 (lines 146–149), §2 out-of-scope table | static-inspection |
| SF-7 | §4.5 test list does not include a test for the `_find_pending_view` / `confirm_proposal` / `reject_proposal` path under the new return-type. After this slice lands, `_find_pending_view` at folio/links.py:267 destructures the function return into the existing comprehension `[view for view in collect_pending_relationship_proposals(config)]` — Python iterates the 2-tuple, yielding the list *then* the dict, so the first "view" is a list and crashes on `view.proposal.proposal_id`. Add an explicit regression test covering `confirm_proposal` and `reject_proposal` after the return-type change, *before* the scope-lock expansion permits the five-site update. This is adjacent to X-1 but is a distinct test-coverage gap not resolved by `test_suppression_counter_returned_alongside_views`. | docs/specs/v0.6.0_proposal_review_hardening_spec.md §4.5; folio/links.py:266–275 | static-inspection |
| SF-8 | §8 adversarial-attack note 4 ("What if `confirmed_relationships` is missing the `producer` field on older accepted entries?") commits the aggregator to "exclude those entries from per-producer rates with a warning" — but §4.3 helper signature and output dataclass make no provision for emitting warnings. The doctor rendering table has no column for "entries excluded due to missing producer." Either add the warning emission path to §4.3 (with a rendering row like `"N accepted entries excluded: missing producer field"`) or drop §8 note 4's commitment. As written, the spec commits to behavior it does not plumb. | docs/specs/v0.6.0_proposal_review_hardening_spec.md §4.3, §8 Adversarial Attack Notes item 4 | static-inspection |
