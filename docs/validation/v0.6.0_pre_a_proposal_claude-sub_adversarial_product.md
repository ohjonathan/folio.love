# Proposal Review — proposal-review-hardening-v0-6-0 (claude-sub, Adversarial+Product)

## 1. Context header

- Overall verdict: Revise and re-review

## 2. Feasibility

The proposal CAN become a Phase A spec in a reasonable pass, but only after (a) the internal contradiction between §10.1 (zero-tolerance on exact rejected-suggestion resurfacing) and §13.1 gate 4 (5% tolerance on the same event) is resolved, (b) the queue-bound operationalization ("per engagement per run") is pinned to an existing frontmatter-plus-registry accounting surface rather than an implied runtime counter that does not exist today, and (c) the `subject_id` / `source_id` / `target_id` "as applicable" ambiguity is reduced to a per-`proposal_type` field requirement table. Capability gaps: none — existing `folio/links.py` already implements a narrower version of this contract for document relationships, so the proposal's surface area is mostly codifying-and-extending, not greenfield. Resource gap: three of six "shared consumers" in §12 (`folio synthesize`, `folio search`, fully generalized `folio graph` proposal handling) do not yet exist, so committing them to this contract is a forward promise the spec cannot validate without those surfaces.

## 4. Scope sanity

The proposal is sized for a single deliverable **if** §13 "Validation Workstream" is treated as a separate downstream phase rather than baked into this spec's acceptance criteria. As written, §13.1 mixes three unrelated validation tracks (document relationships, entity merges, diagram archetypes) into one workstream with one set of numeric gates, and §14 acceptance criterion 6 demands the validation workstream land inside this spec. That is three deliverables collapsed into one.

The cleanest decomposition would be:
1. Proposal object contract + lifecycle (§§5–7) — this is the governance primitive.
2. Review surface + rejection memory + queue bounds (§§8–10) — this is the operator-facing behavior.
3. Trust-gated surfacing (§11) — already has concrete override design on `folio digest` via FR-811; can ship independently.

The §13 validation workstream should be a separate `validation-run` phase, not a scope item in this spec. Rev 2's narrowing was directionally correct but §13 was not pruned in proportion — see PR-P-6, PR-A-10.

Not strongly recommending Split yet because the three chunks above are tightly coupled via `input_fingerprint` and `lifecycle_state` fields. They share the same object. A Revise-and-re-review that either (a) clearly separates the object contract from the validation-run instrumentation, or (b) chooses Split explicitly, is the better path.

## 6. Unknowns worth resolving before Phase A

- **must-resolve-pre-A**: §10.1 vs. §13.1 gate 4 contradiction on rejected-suggestion resurfacing (PR-A-1). One must be corrected before a spec author can proceed; otherwise the spec has no single source of truth for this behavior.
- **must-resolve-pre-A**: `input_fingerprint`'s "normalized claim identity" definition (PR-A-3). Without it, `basis_fingerprint` (the existing `folio links` field) and the new `input_fingerprint` cannot be reconciled, and rejection memory cannot be implemented consistently.
- **must-resolve-pre-A**: "per engagement per run" operationalization (PR-A-2). The spec needs to pick a persistence layer for the run counter or explicitly re-scope the cap to a surface-time cap ("≤20 rendered rows per page") rather than a production-time cap. These are different requirements.
- **must-resolve-pre-A**: §13 validation workstream's position in scope (PR-P-6, PR-A-10, §4 scope sanity). Decide whether §13 is committed spec scope or a separate validation-run phase.
- **resolve-during-A**: §5 field-requirement table per `proposal_type` (PR-A-5). Can be elaborated in the Phase A spec as a table; does not need to be settled before Phase A begins.
- **resolve-during-A**: §8 review-surface medium (CLI vs. Obsidian) (PR-P-5). Phase A spec can commit to one and defer the other.
- **resolve-during-A**: `trust_status`-change staleness direction (PR-A-4). A spec author can encode "only trust-posture degradation triggers stale; improvement does not re-queue rejected proposals."
- **resolve-during-A**: `extraction_confidence` vs. `review_status: flagged` separation (PR-A-6). Phase A can state the invariant ("`review_status: flagged` SHALL be operator-set or explicitly-rule-set, not auto-derived from `extraction_confidence`") or accept the coupling.
- **defer**: `expired` state handling (PR-A-9). Non-blocking minor that can be handled in a follow-up.
- **defer**: Evidence-locator cap edge cases (PR-A-11). Rendering detail; Phase B territory.

## 7. Verdict

**Revise and re-review.**

Justification: Two blocker-severity adversarial findings (PR-A-1 internal numeric contradiction; PR-A-2 unoperationalized queue bound) and one blocker-severity definitional gap (PR-A-3 undefined "normalized claim identity") mean a Phase A spec author cannot proceed without making silent judgment calls that reviewers will correctly flag as unsupported. The Product lens additionally identifies scope leakage in §13 (PR-P-6) and acceptance-criteria-vs-validation-gate divergence (PR-P-3) that will make §14 pass even if the real user value does not. These are all fixable in one revision pass — the proposal is not fundamentally unsound and the rev 2 narrowing was directionally correct. A Split is tempting given §4's scope-sanity analysis but the three candidate chunks are tightly coupled through the shared object contract, so one more revision of this single spec is the lower-risk path. No facts-level disagreements with the Technical lens are pre-identified; Product-vs-Technical disagreement candidates in §3.3 are judgment differences that the orchestrator's consolidation step can resolve.
