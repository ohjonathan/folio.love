## Rationale

Rev 4's §9.2 rewrite replaced the rolling-window framing (which required global chronological ordering, not supported by `folio/links.py`'s `reject_proposal`) with a cumulative acceptance-rate gate plus a 10-reviewed-proposal warm-up. This framing:

1. **Eliminates the `rejected_at` dependency.** Cumulative sums of terminal-state values (accepted / rejected) do not require ordering. Verified independently by both reviewers against `folio/links.py` code.
2. **Is computable from existing frontmatter.** Both reviewers noted the aggregation sources are already present: rejections in `_llm_metadata.<producer>.axes.relationships.proposals` (`status == "rejected"`); acceptances in `_llm_metadata.links.confirmed_relationships` (with `producer` attribution).
3. **Handles producer cold-start via the 10-proposal warm-up.** Claude-sub verified the warm-up does not introduce a meaningful gaming attack surface under folio's closed-set producer threat model (producers are authored by folio, not user-supplied).
4. **Names the tradeoff explicitly.** The spec commits to the cumulative framing while acknowledging its responsiveness tradeoff vs. a rolling window, and explicitly commits to revisiting if field evidence shows tail-drag harms recovered producers. This makes the tradeoff auditable without bloating the current spec.

## New findings from Round 3 (all Phase-A carry-forward, none blocking)

- **PR-A-15 (claude-sub):** Rev 4's §9.2 aggregation description refers to `_llm_metadata.<producer>.axes.relationships.proposals` but accepted proposals have been *moved* to `_llm_metadata.links.confirmed_relationships` by `confirm_proposal`. The spec should list both sources for the aggregation formula. **Severity: minor spec-docs correction.** Phase A can fix this in the Technical Design section of the spec it authors.
- **PR-A-16 (claude-sub):** The 10-proposal warm-up has a theoretical padding attack (emit 9 low-stakes easily-accepted proposals to pad the warm-up, then emit real noise). **Severity: should-fix, not blocking.** Mitigated by folio's closed-set producer threat model; a Phase A spec can note this explicitly.
- **PR-T-data-path (gemini):** Same as PR-A-15. Identified independently by both reviewers. **Severity: minor.**

None of these three are blocking. All three are single-sentence clarifications that a Phase A spec author will naturally address when writing the Technical Design section against current `folio/links.py` state.
