## 3. Quality assessment

Strengths. The (disposition, reason) → finding-code mapping in §5.1 is
exemplary: each row cites the exact source line in `folio/enrich.py` where
`_determine_disposition` produces that tuple, so an implementer can walk the
spec and the source side-by-side. §5.2 (defensive default for unknown protect
reasons) is a thoughtful forward-compat decision with a matching test
(ED-15). §6.3 turns the "do not consume `derive_trust_status`" constraint
into a runnable grep gate (G-scope-5), which is the doctrinally clean way to
encode a scope firewall. §7.4 preempts the likely alignment/adversarial
question ("why not reuse the v0.9.0 shared envelope?") with a concrete
11-key enumeration of what reuse would cost.

Weakness. The §3.3 CLI code block is the one place where the spec's
execution-readiness breaks down. The author cites the digest precedent at
cli.py:1179-1234 as the proven pattern, but the code block itself deviates
from that precedent in a way that does not run. Details under blockers —
the net effect is that a mid-level engineer who copies §3.3 verbatim will
find both `folio enrich ClientA` and `folio enrich diagnose` broken, and
will need either to re-derive the Click group shape from the digest
precedent or to infer the fix from test failures in ED-CLI-2/ED-CLI-3/
ED-CLI-15.

Implementability, net. Once the §3.3 defect is fixed, a mid-level engineer
can build this without follow-up questions. The dataclass shapes, envelope
keys, sort rules, and exit semantics are unambiguous. Lines 256-289
(`diagnose_notes` body) are close to pseudocode-level executable as-is.

## 4. UX review

Text output (§4.3). Operator-friendly: severity prefix, finding code,
subject_id, detail, `Action:` line, trailing summary. Matches how an
operator would scan diagnostic output. Two concerns:

- `[flagged]` suffix placement (end of detail line) differs from the
  existing `graph doctor` precedent (cli.py:2177-2185) which places the
  flagged marker inside the severity bracket. Not wrong — diagnose is a
  different surface — but operators will hit both commands in the same
  session and the inconsistency will register. See PEER-SF-003.
- The trailing summary `"N findings (M flagged): X error, Y warning, Z info."`
  is new. Reads well. No finding.

JSON output (§7.1-§7.3). Clean envelope; 7 top-level keys; per-finding keys
ordered to match the parent §7.7 fixed schema with the additive
`trust_status` trailing. `by_code` as a post-truncation map is the right
choice (otherwise `total` and `sum(by_code.values())` disagree).

Error messages. §9 table matches existing `folio enrich` / `folio graph
doctor` patterns (`✗ Fatal error:`, exit 1). No UX regression.

## 6. Positive observations

- §5.1 mapping table is the best part of this spec: each (disposition,
  reason) row cites the exact source line in `folio/enrich.py`. An
  implementer can walk the spec and the source side-by-side without
  context-switching.
- §5.2 defensive default for unknown protect reasons, plus the matching
  ED-15 synthetic-entry test, pre-empts a whole class of silent-omission
  bugs when `_determine_disposition` is later extended.
- §6.3 encodes the sub-item B firewall as a runnable grep gate
  (G-scope-5) rather than prose-only. Matches the framework's "scope is
  enforced by runnable checks" principle (P6).
- §7.4 pre-writes the answer to the likely alignment/adversarial question
  "why not reuse the v0.9.0 shared envelope?" with a concrete 11-key cost
  enumeration. Saves a review round.
- §11 carry-forwards are itemised with explicit defer-target AND rationale
  for each. This is the discipline that keeps v1.x slices from accreting
  regret.
- §8.1 ED-14 pins the dataclass field order against positional-tuple
  regression (SF-108 closure carry-forward) — good defensive-test taste.
- Test plan is exhaustive: 20 module tests + 18 CLI tests exhaustively
  cover the 6 finding codes, the two omit cases, scope filtering, limit
  truncation, deterministic sort, trust annotation, read-only invariant,
  defensive default, and forbidden-symbol scope. An implementer has very
  little room to under-cover.
- Manifest's G-scope-3 line caps (140 for cli.py, 180 for enrich.py) are
  conservative enough to catch scope drift without starving the
  group-reshape + subcommand + renderers of headroom.
