of PROD-SF-1..5, PROD-MN-1..2, and PROD-I-1..2 is closed in spec text with

informational note on disclosure precedence (PROD-I-101).

Verdict: **Needs Fixes** (1 new should-fix introduced by v1.1; 1 new minor;

## Detailed findings

### PROD-SF-101 (should-fix): `N flagged input(s) included` regresses SF-12 in test and example block

**Location:** `docs/specs/v0.7.0_folio_digest_spec.md` DG-CLI-9c (line 994).

**Test text as specified:**

```
DG-CLI-9c | test_digest_success_stdout_include_flagged_echo | `--include-flagged` honored AND inputs included → success line includes `(--include-flagged honored; N flagged input(s) included)` (MN-9 / PROD-I-2 stdout disclosure)
```

**Why this is a regression:**

SF-12 was introduced to eliminate the `input(s)` bracket-suffix shortcut
across the spec, with reference `cli.py:2536-2541` as the canonical
three-branch pattern. Every other pluralization in §4 and §9.3 was
rewritten to match. This test assertion — which is the only surface that
tests the success-plus-override echo — reverts to the exact shortcut
SF-12 forbids. An implementer reading the test table as the source of
truth for what to assert will reproduce the bad copy.

The `§4` example at line 397 shows `2 flagged inputs included` (plural,
correct for N>1) but does NOT show the N==1 variant. A reader looking at
§4 alone sees only the plural case and may assume the singular reads
`1 flagged inputs included` — which is the slice-6a bug this entire
exercise was meant to prevent.

**Body vs. test surface asymmetry:** the `§4` example is itself
inconsistent with DG-CLI-9c. The example literal is
`(--include-flagged honored; 2 flagged inputs included)` but the test
asserts `(--include-flagged honored; N flagged input(s) included)`.
Either the example is wrong or the test is wrong; they don't agree on
the pluralization rule.

**Recommended rewrite (§4 stdout example block):** Show both branches,
matching the pattern used for `Drawn from 1 input.` / `Drawn from 7 inputs.`:

```
# Successful daily with --include-flagged, N flagged >1
✓ Wrote daily digest: ...
  Drawn from 9 inputs (--include-flagged honored; 2 flagged inputs included).

# Successful daily with --include-flagged, N flagged ==1
✓ Wrote daily digest: ...
  Drawn from 4 inputs (--include-flagged honored; 1 flagged input included).
```

**Recommended rewrite (DG-CLI-9c):** Split into two asserts, matching
DG-CLI-9 / DG-CLI-9b which already split singular/plural for `Drawn from`:

```
DG-CLI-9c  | test_digest_success_stdout_include_flagged_echo_plural
           | success line with 2+ flagged inputs includes
             `(--include-flagged honored; {N} flagged inputs included)`
             (plural branch per SF-12 / MN-9)

DG-CLI-9d  | test_digest_success_stdout_include_flagged_echo_singular
           | success line with 1 flagged input includes
             `(--include-flagged honored; 1 flagged input included)`
             (singular branch per SF-12 / MN-9)
```

Alternative minimum fix: keep DG-CLI-9c as one test but rewrite assertion
text to enumerate both branches explicitly (e.g., "asserts `1 flagged
input included` when N==1 and `{N} flagged inputs included` when N>1").
Test-ID proliferation is secondary; the primary fix is eliminating the
`input(s)` shorthand and showing both branches.

**Severity rationale:** should-fix rather than minor because this is the
exact pattern PROD-SF-1 round-1 explicitly flagged as a recurring defect
and SF-12 explicitly resolved. Missing this after a round-1 call-out
elevates it above minor.

### PROD-MN-102 (minor): `⚠` warning prefix not in existing folio CLI vocabulary

**Location:** §10.4 lines 870, 871, 874.

**Spec text:**

```
⚠ Existing digest at {path} has malformed frontmatter; treating as fresh write.
⚠ Registry referenced missing file at {path}; regenerating.
```

**Observation:** `⚠` is introduced here for stderr warnings, but the
existing folio CLI vocabulary uses:

- `✓` for success
- `✗` for error
- `↷` for skip (e.g., `cli.py:1046, 1054, 1062` for various skipped-row reasons)
- Plain (no prefix) for informational advisories (e.g., `cli.py:1051, 1059` count summaries; §4 line 415 `Note:` advisory)

There is no precedent for `⚠`. Other CLI tools do use it, and it's
visually clear, but for a slice whose product review already focused on
vocabulary consistency, introducing a new symbol without precedent is
mild drift.

**Options:**

1. Reuse `↷` (matches the refresh skipped-row idiom — "we saw something, we
   dealt with it by falling back"). `↷ Existing digest has malformed frontmatter; treating as fresh write.`
2. Use plain `Note:` prefix to match §4's advisory line: `Note: existing digest at {path} has malformed frontmatter; treating as fresh write.`
3. Keep `⚠` and explicitly establish it in §11 as the new warning prefix
   (the spec MUST commit if it's introducing a new symbol).

Severity minor because warnings are stderr-only, not primary operator-
visible output. But worth a round-2 call-out because the round-1 review
explicitly focused on visual-marker consistency.

### PROD-I-101 (informational): Disclosure precedence between SF-14 advisory and MN-9 echo

Two disclosure surfaces are enumerated in v1.1:

- SF-14 advisory (`Note: --include-flagged has no effect in --week mode; ...`) — `--week` mode
- MN-9 echo suffix (`(--include-flagged honored; N flagged inputs included)`) — daily mode, on the success line

By the §4 / §12 algorithm, `--week` mode always emits SF-14 when
`--include-flagged` is passed, and never emits MN-9 (weekly doesn't
consult `include_flagged`). Daily mode emits MN-9 when inputs were
included, and never emits SF-14.

The two are mutually exclusive by construction. The spec could say so in
one sentence (e.g., in §4 or §12) to spare an implementer the "can both
fire?" question.

Not a blocker and not a should-fix — genuinely informational.

PROD-I-101 (disclosure precedence) are low-cost polishes that can be
