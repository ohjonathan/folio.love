## Should-Fix

### D-P-SF-001: `links status` missing `--include-flagged` flag declared in spec §5.1

**Location:** `folio/cli.py` lines 1661-1699 (`links_status_cmd`);
spec §5.1 table entry for `folio links status`.

Spec §5.1 lists `folio links status` among commands that gain
`--include-flagged`. The implementation's status command does not declare
this option. Functionally, the table always discloses the `Flagged Excluded`
column and the total footer, so a `--include-flagged` flag would be a no-op
for the current renderer, which is likely why it was omitted. Still, the
spec asserts the flag exists on all six commands.

**Recommended fix (trivial):** add the flag as a no-op (document it as
reserved / present-for-API-consistency), OR amend the spec to carve status
out explicitly. Given that no test asserts the flag exists on `status`, the
cheaper path is a spec clarification rather than a CLI change; either
resolves the literal discrepancy.

### D-P-SF-002: `links status` empty-table disclosure points at `folio links review`, spec §5.5 says `folio links status`

**Location:** `folio/cli.py` line 1675.
Spec §5.5 prescribes:
```
... use folio links status --include-flagged to inspect.
```
The implementation emits:
```
... use folio links review --include-flagged to inspect.
```

In practice `review` is the command that actually renders per-proposal
flagged tags, so the implementation's text is arguably better UX. But it
is a literal spec deviation. Either update the message to match the spec
exactly, or update the spec phrasing to reflect the `review`-pointed
messaging. Related to D-P-SF-001 — if `--include-flagged` is added to
`status`, the spec's original wording becomes sensible; if not, the
implementation's wording is.

### D-P-SF-003: `confirm-doc` / `reject-doc` swallow mixed-case flagged disclosure

**Location:** `folio/cli.py` lines 1781-1786, 1812-1817.

Current guard:
```python
if acted == 0 and flagged_excluded > 0:
    click.echo(f"{flagged_excluded} proposal(s) excluded from {doc_id} ...")
```

When a document has both clean and flagged proposals, the operator runs
`links confirm-doc <id>`, `acted` is non-zero (clean proposals confirmed),
and the flagged count is silently dropped. Spec §5.6 literally says
"When `acted == 0` and `flagged_excluded > 0`", so the implementation
matches the spec text — but the underlying rule-5 intent ("no silent empty
when flagged inputs are the real cause") arguably covers the mixed case too,
since the operator has no surface that tells them "N additional proposals
were skipped because they were flagged."

**Recommended fix:** drop the `acted == 0` guard and always emit the
disclosure when `flagged_excluded > 0`. Non-blocking because the spec
wording is permissive, but it closes a silent-miss edge case for a typical
"clean up a mixed doc" workflow.

### D-P-I-009: Performance / complexity

- `_resolve_target_flagged` memoizes per-call. Target fm reads bounded at
  `O(unique_target_ids)` per `collect_pending` invocation.
- `confirm_doc` / `reject_doc` call `confirm_proposal` / `reject_proposal`
  per proposal, each of which triggers another
  `collect_pending_relationship_proposals` pass via `_find_pending_view`.
  This is `O(proposals × registry_size)` — pre-existing behavior, not
  introduced by this slice. For realistic doc sizes (dozens of proposals)

- `relationship_status_summary` walks the registry twice (once via
  `collect_pending`, once over `decks` to find zero-pending sources). This
  is `O(registry_size)` overall. Fine.
