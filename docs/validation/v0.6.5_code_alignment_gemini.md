## Findings

### 2. Forbidden Paths

**Status: PASS**

No changes were made to any paths listed in `scope.forbidden_paths` (`folio/pipeline/`, `folio/enrich.py`, `folio/links.py`, `folio/provenance.py`, `docs/product/`, `docs/architecture/`, `frameworks/llm-dev-v1/`, `AGENTS.md`, `Ontos_Context_Map.md`).

### 3. Spec §4 Technical Design Items

**Status: PASS**

All technical design items specified in §4 are correctly implemented:
- `_empty_entity_registry()` defaults `rejected_merges` to `[]`.
- `EntityRegistry.load()` handles legacy files gracefully by applying `setdefault("rejected_merges", [])`.
- `_compute_entity_merge_basis_fingerprint` uses canonical JSON serialization.
- `suggest_person_merges(apply_rejection_memory=True)` correctly filters exact fingerprint matches and annotates `revived` status.
- `PersonMergeSuggestion` dataclass updated with new fields.
- `reject_person_merge` properly enforces key existence, idiosyncrasy, computes basis reasons, checks idempotency, and updates timestamp on append.
- `count_entity_merge_suppressions` returns the integer count of `rejected_merges`.

### 4. Graph Status Label Rename

**Status: PASS**

The output label rename from `Duplicate person candidates` to `Reviewable duplicate person candidates` is correctly wired in `graph_status_cmd` within `folio/cli.py` and is fully functional.

### 5. Library Lock

**Status: PASS**

`library_lock` is successfully applied to the critical sections of both commands in `folio/cli.py`:
- `suggest_merges_cmd`: `with library_lock(library_root, "entities suggest-merges")` wraps the registry loading and suggestion computation.
- `reject_merge_cmd`: `with library_lock(library_root, "entities reject-merge")` wraps the entire load-mutate-save flow.
