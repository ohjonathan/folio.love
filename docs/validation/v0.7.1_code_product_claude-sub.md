# Product Review — folio-graph-generalized-proposals-v0-7-1 / D.2 / claude-sub

## 1. User-value assessment

The user here is the Folio **operator** — the human who runs `folio graph
doctor` in the terminal, reads the findings, and has to decide what to
do next. The job-to-be-done is "triage the graph-health queue": for
each pending relationship proposal, decide whether to confirm, reject,
or route upstream for a data fix. Pre-v0.7.1, every pending finding
printed the same `recommended_action` string ("Review with `folio links
review` ...") regardless of whether the proposal's target was missing
from the registry, whether the source/target was flagged, or whether
the proposal was structurally clean. That flatness was the operator
problem: `graph doctor` was a list of items that all looked identical
in terms of next-step guidance, so the operator had to mentally classify
each one before acting.

v0.7.1 delivers three operator-visible improvements that directly
address the classification problem. (a) The severity line now carries
inline annotations — `[flagged]` when the underlying documents are
trust-compromised, `[schema-gate: <rule>]` when a today-computable
structural rule failed — so the operator can visually sort the queue
without reading every `detail` line. (b) `recommended_action` now
branches by failure mode: a `target_registered` failure routes the
operator to `folio refresh`/`folio ingest` (data pipeline) instead of
`folio links review` (governance decision). That is the right
pipeline-vs-governance split — an operator who sees "target not in
registry" and is told to run `folio links review` is being led to
reject a proposal that was never the problem. (c) `--include-flagged`
at the `graph doctor` surface gives the operator parity with
`folio links`, so the trust-gating story is consistent across the two
surfaces an operator uses to work the queue.

The spec's §1 operator story ("operators scanning `folio graph doctor`
output should see at-a-glance whether a finding needs upstream data
work or is a normal review-queue item" — spec §3.6) is delivered. The
four fixtures I walked with CliRunner (happy-path, flagged, missing-
target, unsupported-relation) produced unambiguous stdout in every case
except the fourth, which surfaces a real product concern documented in
§7 below.

One caveat on user-value: the `subject_id` → `proposal_id` rename is
a net positive for correctness (§5 semantics now honest) but imposes a
migration cost on any external script that parsed the pre-v0.7.1
`--json` output. The CHANGELOG is explicit and the CLI help adds a
one-line pointer; the friction is acknowledged rather than hidden.

## 8. Positive observations

- **Target-missing action copy (C-3) is the strongest piece of copy in the set.** Naming both `folio refresh` and `folio ingest`, plus the intentional-miss escape hatch via `folio links review`, covers all three operator mental models (data-pipeline-lag, missing-upstream-ingest, intentionally-rejected-target) in a single line. This is the fix for PRD-7 done right.
- **Annotation placement on the severity line** is the right choice. Putting `[flagged]` / `[schema-gate: ...]` next to `[medium]` lets the operator scan the queue by severity-column alignment, which is how terminal operators naturally read tabular output. Putting it in the detail field (the v1.1 mistake ADV-R2-M1 caught) would have broken that scan.
- **The `—` fallback for blank proposal_id** closes the P-R2-2 edge case with visible signal rather than silent blankness. "No string between `code` and `:`" would have been a subtler bug than "None", and the em-dash choice says "something was expected here" to the reader.
- **CHANGELOG Breaking stanza** uses MUST (RFC 2119-style) and names the exact before/after key pair. A consumer running `jq '.findings[].subject_id'` gets a precise instruction without having to diff the spec.
- **`--include-flagged` help copy** references the underlying `review_status: flagged` field, so an operator who sees the flag and wonders "what does 'flagged' mean?" can trace it to the frontmatter field without reading the trust-gating spec.
- **The migration cost is acknowledged, not hidden.** The Breaking stanza does not try to soften the change; the `--help` addendum is a reminder, not a pitch. This is honest product communication.
- **`recommended_action` branching by failure mode (S-6 through S-10)** directly delivers the §1 operator story. Pre-v0.7.1, every pending finding recommended the same action; v0.7.1 differentiates pipeline issues from governance decisions, which is the core product improvement.

## 9. Verdict

**Needs Fixes**

Two should-fix items (PRD-D2-1 operator-honesty gap on `supported_relation`
unreachability; PRD-D2-2 spec-jargon leak in `--help` copy) degrade the
shipped operator story without blocking value delivery. The
user-experience improvement vs pre-v0.7.1 is real and material; the
fixes are copy-level, not structural. Both are addressable by
CHANGELOG edit + one-line CLI help-text revision in D.4.

If the meta-consolidator determines the should-fix items fall below
the D.2 Needs-Fixes threshold under the family's calibration, Approve
is defensible; from the Product lens I record Needs Fixes to surface
both items into the D.3 canonical verdict.

## 10. Notes

- Cross-cutting concern (technical consequence of a product decision): the
  upstream-filter question (PRD-D2-1) has a structural fix option (move
  the filter semantics) that belongs to Peer / Alignment, not Product.
  From the Product lens I recommend the copy-level CHANGELOG fix
  because the current behavior (silent filter, no operator-visible
  surface for unsupported relations) is acceptable in v0.7.1 — there is
  no operator harm, only a documentation-honesty gap.
- `--include-flagged` defaulting to off is the right choice. The flag
  is a deliberate opt-in to act on trust-compromised inputs, matching
  the `folio links` convention. Operators who need it will find it.
- No boundary violations with the other lenses: scope compliance
  (forbidden paths / cardinality) is the mechanical gate; code
  structure is Peer; adversarial failure modes is Adversarial;
  alignment with parent spec is Alignment. PRD-D2-1 is framed as
  "the operator can't see the surface we advertised," which is the
  Product framing, not "the gate is wrong" (which would be Peer).
