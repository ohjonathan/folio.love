## Canonical Blockers (DCB-N)

### DCB-001 - Truncation footer cannot render the promised `N of M` count

**Evidence:** codex DSF-001, PEER-DCB-001, and PROD-DCB-001 converge on the same root cause. `diagnose_notes()` computes `unfiltered_count = len(raw_findings)` at `folio/enrich.py:2141` but discards it. `DiagnoseResult` has no pre-truncation count (`folio/enrich.py:1941-1950`), so `_render_diagnose_text()` uses `result.summary.total` and hardcodes `more`, producing `... (showing 1 of more; raise --limit to see more)` at `folio/cli.py:1326-1329`.

**Required D.4 closure:** Carry the pre-truncation count through the diagnose result path and render the exact spec intent: `... (showing {len(result.findings)} of {M}; raise --limit to see more)`. Add or update CLI regression coverage so `--limit 1` on a 3-finding fixture asserts `showing 1 of 3`, not merely `showing`. If D.4 exposes `M` in JSON, bump the diagnose schema/version contract and update the spec, tests, and manifest gates coherently; do not silently add a key under `schema_version: "1.0"`. If D.4 keeps the count internal for text only, update field-order/cardinality checks for any internal dataclass field change.

### DCB-002 - Library-wide diagnose false-cleans corrupt or unreadable registry state

**Evidence:** codex DCB-001 direct-run reproduction exits 0 with `No enrichment hygiene findings.` on a corrupt registry when `scope=None`. Peer DSF-002 independently identifies the same bypass. At HEAD, `diagnose_notes()` only calls `_resolve_scope_or_raise()` when `scope is not None` (`folio/enrich.py:2123-2124`), while the `_corrupt` registry check lives inside `_resolve_scope_or_raise()` (`folio/enrich.py:1973-1978`). Library-wide diagnose therefore falls through to `plan_enrichment()`, whose dry-run path rebuilds corrupt registry state in memory (`folio/enrich.py:253-258`).

**Required D.4 closure:** Add an unconditional diagnose preflight before scope resolution and before `plan_enrichment()`: verify `config.library_root` exists and is a directory; if `registry.json` exists, load it and fail with `ScopeResolutionError` when `load_registry()` returns `_corrupt`. Preserve the existing valid-empty-library behavior when `registry.json` is simply absent. Add tests for library-wide corrupt registry, unreadable registry, and missing/non-directory `library_root`; the existing scoped corrupt-registry test is insufficient.

### DCB-003 - Suggested shell commands interpolate unquoted user-controlled tokens

**Evidence:** codex DSF-002 reports copy-paste command injection in rendered command suggestions. At HEAD, the enrich breadcrumb interpolates `scope` into a backtick command (`folio/cli.py:1250-1252`), and recommended actions interpolate `subject_id` into `folio enrich --force ...` and `folio enrich ...` snippets (`folio/enrich.py:2050-2053`, `folio/enrich.py:2062-2064`). Folio does not execute these strings, but it explicitly presents them as shell commands for operators to copy.

**Required D.4 closure:** Either quote every embedded scope/subject token with `shlex.quote()` or stop rendering exact shell snippets that contain untrusted IDs. Cover both breadcrumb scope and `recommended_action` subject IDs. Add regression tests using metacharacters/backticks in scope and subject IDs and assert the rendered output is not an executable unquoted shell fragment.

### Closed D.2 blocker - Package metadata version drift

codex DCB-002 is closed at the D.3 target commit. Verification against `HEAD:pyproject.toml` shows `version = "1.0.0"`, matching the v1.0.0 contract and changelog. D.4 should keep this fixed and may add a release-identity guard if cheap, but no blocker action remains.
