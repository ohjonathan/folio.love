## Notes

- The JSON/text split decided in §5 OQ-6 is the cleanest part of v1.1:
  machine-stable enum for tooling, threshold-disclosing text for
  humans. Future iterations that tune the thresholds won't need a
  schema bump — only the text suffix changes. Good design hygiene.
- The always-render-suppression-count decision (UX-1 + FV-4 co-closure)
  is a small but important product call: discoverability beats
  terminal quietness when the feature being disclosed is a silent
  filter. The precedent set here applies to future review-layer
  filters.
- The `(revived — basis changed)` copy is short, specific, and
  operator-legible. "basis changed" is honest — it doesn't promise
  "new evidence" or "updated context" (which would overclaim);
  it describes the exact technical condition (different
  `basis_fingerprint`) in operator-legible terms.
- The row-ordering decision (`low-acceptance` → `ok` → `warmup`) is
  correctly justified as "problems first triage". Johnny's SteerCo-prep
  scan pattern is captured.
- Scope-disclosure footer is doing triple duty (FV-3 + SC-2 + CPY-2
  co-closure). Landing it as a tested copy string in §4.6 test 13
  prevents silent drift if a future PR trims the note.
- Data-integrity row elision-when-zero vs suppression-count
  always-render is an intentional asymmetry: data-integrity is a
  producer defect worth surfacing only when it fires; suppression
  count is a feature-discoverability signal worth surfacing always.
  Both choices are correct for their respective roles.
- No accessibility regression. Pipe-table format preserved; no
  color/unicode-box traps introduced.
- Recommend Approve; no further product-lens revision needed before
  Phase C dispatch.
