### PR-SF-2 (closed): Thin reject echo

**R1 recommendation:** echo reasons + both names + both keys + ✓ prefix.

**v1.1 §4.2 item 2 commit (line 153):**

```
"✓ Rejected merge candidate: {left_name} [{left_key}] <> {right_name} [{right_key}] (reasons: {', '.join(reasons)})"
```

Verbatim match to R1 recommendation. All four elements present (prefix, both names, both keys, reasons). The `{', '.join(reasons)}` expansion is correct and matches the comma-separated rendering used by `suggest-merges` output rows, so the tie-back between reject-time and later revival annotation is preserved.

**Error-path copy (R1 addendum):** v1.1 §4.2 item 2 line 155 commits to:

```
"✗ Merge candidate is stale: '{key}' no longer exists (possibly already merged)"
```

This is stronger than the R1 recommendation (which asked for an error but didn't specify stale-vs-unknown semantics). The "possibly already merged" clue is genuinely helpful for the solo operator who just ran `folio entities merge` in a prior terminal. Closed with improvement.

**Test coverage:** T-9 (line 230) covers happy path; T-9c (line 232) covers stale key. Closed.

### PR-SF-3 (closed): Idempotent silence

**R1 recommendation:** distinct message on second reject so operator can distinguish "stored" from "already rejected."

**v1.1 §4.2 item 2 commit (line 154):**

```
"= Already rejected: {left_name} <> {right_name} (no change)"
```

Commits to a distinct message. Two intentional divergences from the R1 recommendation that are operator-appropriate and worth noting (not flagging):

1. **Prefix: `=` instead of `✓`.** R1 didn't specify a prefix. `=` ("unchanged") is a reasonable semantic marker and visually distinct from the `✓` of a real-write. This is better than the R1 wording.
2. **Drops `[key]` brackets from names.** On the first-reject line the `[key]` brackets help the operator match against `suggest-merges` output. On a no-op re-reject the keys are less useful (the operator *just typed them*). Slightly less symmetric, but not a friction point.

Test coverage: T-9b (line 231) *"Second invocation prints 'Already rejected … (no change)' and exits 0."* Exit-code idempotency covered. Closed.

### PR-SF-4 (closed): Verification surface

**R1 recommendation:** minimum-scope solution — add `"({M} total rejections recorded.)"` line to `suggest-merges` output so "did my reject stick?" has an answer.

**v1.1 §4.2 item 1 commit (line 145):**

> Additionally render total-rejections disclosure (B.1 R1 product PR-SF-4): `({M} total rejections recorded.)` where `M = registry.count_entity_merge_suppressions()`. Rendered on a line immediately after the suppression-count line. **When `M == 0`, render `(0 total rejections recorded.)` rather than suppressing — operator sees the surface even when pristine.**

The M=0 behavior is the key operator-experience detail: a pristine operator running `suggest-merges` before their first reject still sees the line, so the surface is *discoverable* rather than conditional. This is the right call and closes the finding fully.

**Test coverage:**
- T-10b (line 234): *"Output contains `({M} total rejections recorded.)` (including `(0 total rejections recorded.)` when pristine)."* — explicitly tests the pristine case.
- T-12 (line 239) covers the registry helper directly.

Closed.

Note: `count_entity_merge_suppressions()` rename (from `collect_entity_merge_suppression_counts`) resolves S-4 but is also a small product win — the verification-surface copy `"{M} total rejections recorded"` reads more naturally when fed by a scalar-named helper.

## New observations in v1.1

### PR-R2-I-1 (informational): `Reviewable duplicate person candidates` label change — scope check

**Observation:** §4.3 line 168 introduces a label change on `folio graph status` output:

> if the CLI currently renders `Duplicate person candidates: {N}`, change label to `Reviewable duplicate person candidates: {N}`

This was **not** proposed in Round 1. It appeared in the canonical verdict's SF-1 resolution (codex ADV-003 + peer S-1 enumeration of the two `graph.py` call sites), where the author chose "filter both sites + rename label to disclose semantics" rather than "leave status unfiltered."

**Operator impact (solo operator):**
- Muscle memory: minor. The operator's eyes find the number, not the label prefix. The number's meaning is slightly different ("reviewable" vs raw) but this is the *point* of the filter.
- Screenshot / doc drift: negligible; internal-only surface.
- Semantic win: the `Reviewable` prefix makes it explicit that `graph status` and `graph doctor` now honor rejection memory, so an operator seeing `0 reviewable duplicates` can't mistakenly think "I have no duplicates" when they in fact have a growing `rejected_merges` list.

**In-scope for this slice?** Defensible yes. Rationale chain:
1. SF-1 required filtering both call sites (codex blocker + peer should-fix convergent).
2. Silent filter on `graph status` count is a Round 1 finding (it would change the number operators see without disclosure).
3. Either (a) don't filter `graph status` — rejected by spec for symmetry with `graph doctor`, or (b) filter it and disclose via label — accepted.
4. The label rename is the minimum disclosure surface; the alternative (a CHANGELOG-only disclosure with no label change) would reintroduce the original "silent filter" problem.

So the label change is a derived consequence of closing SF-1, not a separate proposal. In scope. The spec's §6 risk table line 209 explicitly tracks this as "Mitigation: Label changed to 'Reviewable duplicate person candidates' per §4.3 to disclose filtering; CHANGELOG entry calls out the semantic change." Good traceability.

**One suggestion (not required for approval):** The CHANGELOG §4.4 entry at lines 175-189 does not currently call out the `graph status` label change explicitly. The risk-table line promises it. Recommend adding a `### Changed` bullet:

```
- `folio graph status` now labels duplicate-person rows as
  `Reviewable duplicate person candidates` and reports the filtered
  count (excluding rejections). Unchanged count when no rejections exist.
```

Pre-approval nit only. Not blocking.
