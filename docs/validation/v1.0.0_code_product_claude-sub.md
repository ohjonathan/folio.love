# Product Review — folio-enrich-diagnose-v1-0-0 / D.2 / claude-sub

## 1. User-value assessment

The operator problem is: "my `folio enrich` run reported N protected / M
conflicted notes, and I don't know why." Before this slice, the operator
had to inspect registry entries and frontmatter by hand to figure out
which blocker (curation tier, review status, fingerprint mismatch, stale
enrich state, unparseable frontmatter, or unrecognizable managed
sections) was stopping enrich. Running `folio enrich diagnose [scope]`
now answers that question in one command, read-only, without any LLM
calls and without mutating the library. The breadcrumb inside
`folio enrich`'s dry-run / normal exit path ("Tip: run
`folio enrich diagnose <scope>` to see why these notes stalled") closes
the discovery gap — operators don't need to know the diagnose subcommand
exists before they need it.

The surface does solve a real user job. The sort order (error-first,
then warning, then info, grouped by code, then subject asc), the
`[flagged]` annotation inside the severity bracket, and the
`recommended_action` line per finding are the right operator-facing
choices — a scrolling operator can triage the worst problems first,
spot trust-annotated notes at a glance, and has a concrete next step
on every finding line.

The `--json` mode produces a stable envelope that downstream tooling
(CI, dashboards, ad-hoc scripts) can consume directly without text
parsing — this is an underrated UX win because it means the text
surface doesn't have to compromise on legibility to remain
machine-parseable.

Where the implementation falls short of the spec's user-value promise:
the **truncation footer is broken** (§7 PROD-DCB-001 below), which
silently degrades the `--limit N` flow from "you are looking at 5 of 37
findings, run with a higher limit to see more" to "you are looking at
some findings of some total" — an operator running diagnose on a large
engagement with a `--limit 10` cap has no way to know whether the 10
findings are the whole story or a sample. That is a concrete
user-visible regression from the spec's intent.

## 7. Issues found

### Blocking (Critical — user-facing regression from the spec's promise)

| ID | Description | Location | Evidence | Reproduction | Suggested action |
|----|------------|---------|---------|-------------|-----------------|
| PROD-DCB-001 | **Truncation footer is broken.** Spec §4.3 mandates `... (showing N of M; raise --limit to see more)` where N is the rendered (post-limit) count and M is the unfiltered total. The implementation emits `... (showing {unfiltered} of more; raise --limit to see more)` where `unfiltered = result.summary.total` — but `summary.total` equals the post-truncation length (`len(rendered)` per `folio/enrich.py:2148-2150`), and the word `more` is a hardcoded literal, not the pre-truncation count. Root cause: `DiagnoseResult` does not carry the unfiltered count as a field; the text renderer has no source of truth for M. User impact: running `folio enrich diagnose --limit 10` on a library with 37 blockers prints `... (showing 10 of more; raise --limit to see more)` — the operator cannot tell whether the 10 findings are the whole story, cannot size the gap, and cannot pick a sensible next `--limit`. | `folio/cli.py:1326-1329`; underlying data-model gap at `folio/enrich.py:1942-1957` (DiagnoseResult has no `unfiltered_total`) and `folio/enrich.py:2141-2152` (unfiltered count computed but discarded). | direct-run — see §3 U-1 above (`... (showing 1 of more; raise --limit to see more)` when run with `--limit 1` on a library with 3 findings). | (1) Add an `unfiltered_total: int` field to `DiagnoseResult` (or plumb via a new `summary` sub-field — note this is a schema-minor for the JSON envelope and should be declared as such). (2) Compute and pass `unfiltered_count` from `diagnose_notes`. (3) In `_render_diagnose_text`, emit `f"... (showing {len(result.findings)} of {result.unfiltered_total}; raise --limit to see more)"`. (4) Add a CLI regression test that greps the footer for the actual `M` digit, not just the literal `showing`. |

## 8. Positive observations

1. **`[flagged]` annotation inside the severity bracket is correct and consistently rendered.** Verified by direct-run — a flagged `protect`-class note renders as `[warning [flagged]] protected_by_curation_level <id>: …`. CSF-2 closure landed cleanly; parity with `folio graph doctor` preserved.
2. **Scope-typo UX is a clear win.** `folio enrich diagnose ClinetA_typo` exits 1 with `✗ Scope 'ClinetA_typo' did not match any registered evidence or interaction deck. Run 'folio status' to list registered scopes.` — the recovery command is named inline, not left for the operator to discover. CB-3 closure landed.
3. **`--limit 0` / `--limit -1` are rejected at the CLI layer with exit 2 + Click usage message** before any registry work happens, so the operator gets instant feedback on the invalid input. Spec §4.1 / §4.2 honored.
4. **Action text for the two "no-remediation-command-exists" findings honestly names the manual path** ("no command demotion path exists today", "no `folio unflag` command exists today"). PROD-SF-002 closure reads well — no false promises of a `folio promote` downward flow.
5. **Subject_id substitution in `managed_body_conflict` and `enrich_status_stale` action text works correctly** — literal `<scope>` does not leak into the recommended_action string. PROD-SF-003 closure landed; the operator can copy-paste the suggested command verbatim.
6. **JSON envelope is consumable.** Top-level keys match the spec §7.2 inventory exactly (`schema_version`, `command`, `scope`, `limit`, `findings`, `summary`, `truncated`), per-finding keys match §7.3 exactly (6 keys, `trust_status` always present). Indentation (`json.dumps(..., indent=2)`) + key order matches the spec rendering so diffs between runs are stable. Downstream tooling can key off `schema_version: "1.0"` for the minor-bump contract.
7. **The enrich → diagnose breadcrumb fires in exactly the right conditions** (`result.protected + result.conflicted > 0`) and not otherwise. Verified direct-run with a healthy library (no breadcrumb) and a protected-note library (breadcrumb fires). PROD-SF-006 intent is honored modulo the whitespace nit in PROD-DSF-001.
8. **Empty-findings output is clean** — text mode prints `No enrichment hygiene findings.`; JSON mode emits `{"findings": [], "summary": {"total": 0, "by_code": {}, "flagged_total": 0}, "truncated": false}` with all envelope keys present. Operator tooling has no missing-key branch to handle.
9. **Help text for `folio enrich diagnose` is operator-oriented** — the "Notes:" section explains the `[flagged]` annotation AND the deliberate absence of `--include-flagged` with a one-line rationale ("diagnose is a diagnostic surface, not a discovery surface"). An operator reading `--help` learns enough to know they're not missing a flag.
10. **Read-only invariant is visible at the help-text level** (`Read-only diagnostic — never writes notes, frontmatter, or registry.`) — operators can confidently run diagnose against shared / production libraries without "is this safe?" hesitation.

## 9. Verdict

**Needs Fixes**

Rationale: one blocking issue (PROD-DCB-001, broken truncation footer
— a user-facing regression from the spec's §4.3 contract, creates a
failure-visibility dead-end on the `--limit N` flow) plus two
should-fix copy defects in high-visibility surfaces (the breadcrumb,
the summary line). The blocking issue is not hypothetical — it fires
every time `--limit` truncates, which is precisely the flow operators
use on libraries large enough to need the flag. The should-fixes
degrade the polish of two surfaces that are otherwise quality wins.

All three findings are localized copy + one-small-data-model-field
additions; none require architectural change. Phase D.3 fix round
should be quick.
