## Product should-fix (PROD-SF-D2-NNN)

### PROD-SF-D2-001 — Producer-hint misdirection when QUERY misses but producer is valid

**Severity:** should-fix (operator misleading hint; first-touch frustration)

**Observed behavior** (reproduced against a library with 3 queued proposals
from `producer == "enrich"`):

```
$ folio search nonmatching --producer enrich
Search for 'nonmatching' in scope: (library-wide)
Matches: 0
Excluded (flagged inputs in scope): 0
Hint: no proposals in scope for any producer. Run `folio ingest` /
`folio enrich` to populate the queue.
```

The operator sees "no proposals in scope for any producer" and is directed
to run `folio ingest` / `folio enrich` — but the scope in fact contains
3 proposals from producer "enrich". The misleading part is that the
**QUERY is what filtered the view set to empty**, not any absence of
proposals.

**Root cause** (`folio/search.py:265-269`):

```python
if producer is not None and not findings:
    # CSF-3: collect observed producer names from the QUERY-matched
    # views (pre-producer-filter) so stdout can hint at case mismatch.
    producer_hint_names = [v.producer for v in query_matched]
```

`producer_hint_names` is populated from `query_matched` — the
post-QUERY-filter set. When QUERY eliminates all views, this list is
empty, and `_render_search_stdout` at `folio/search.py:354-358` takes
the empty-list branch and emits the misleading "no proposals ... for any
producer" hint.

The CSF-3 intent (per spec §3.5) was a **case-mismatch hint** — "you
passed `--producer Enrich`, did you mean `enrich`?" — which is useful
ONLY when the QUERY succeeded but the producer filter failed. When the
QUERY is what failed, showing any producer hint sends the operator in the
wrong direction.

**Suggested fix:**

- Gate the producer hint on `query_matched` being non-empty. When QUERY
  eliminates everything AND `--producer` was supplied, fall through to
  the generic "No matches for '<QUERY>'..." breadcrumb instead. The hint
  text can still mention `--producer` in context ("QUERY matched 0; your
  --producer filter was not applied") — but promising "no proposals ...
  for any producer" when the queue has proposals is a correctness issue,
  not just a polish issue.

A minimal patch:

```python
# search.py line ~267
if producer is not None and not findings and query_matched:
    producer_hint_names = [v.producer for v in query_matched]
# else: leave producer_hint_names=None so the generic no-matches
# breadcrumb fires
```

**Impact:** an operator debugging a typo in QUERY would, under the current
behavior, be told to run `folio ingest` / `folio enrich` — work they
don't need to do, may chew up significant time to kick off, and may
trigger downstream changes. Fixing the hint prevents a concrete
false-action trail.

### PROD-SF-D2-002 — `--limit` truncation is invisible in `--json` output

**Severity:** should-fix (stdout/JSON UX parity)

**Observed behavior:**

Stdout with truncation:
```
$ folio search framework --limit 1
Search for 'framework' in scope: (library-wide)
Matches: 1
Excluded (flagged inputs in scope): 0

[queued] clienta_evidence_s0 --impacts--> clienta_evidence_t0  (enrich)
  shared framework concept
(showing 1 of 3; use --limit to adjust)
```

JSON with the same truncation:
```json
{
  "schema_version": "1.1",
  "command": "search",
  "scope": null,
  "query": "framework",
  "trust_override_active": false,
  "excluded_flagged_count": 0,
  "findings": [ ... 1 entry only ... ]
}
```

The JSON consumer has no way to tell that there are 2 more findings
available. An operator (or downstream tool) writing `jq
'.findings | length'` against this envelope would read 1 and conclude
that's the total match count — it isn't.

This is a stdout-only observable that should also be in JSON. The report
dataclass already carries `total_available` and `truncated`
(`folio/search.py:91-92`). Surfacing them in the envelope is a pure
additive and a natural v0.9.0 ship item.

**Why this is a parity issue in particular:** operator-facing docs say
"JSON is the machine-readable envelope." If the stdout version has a
load-bearing "(showing N of M)" footer that the JSON omits, the envelope
is actively less complete than the human renderer for the same query. In
synthesize v0.8.0 the same shape was explicitly marked as deferred (per
v0.8.0 CHANGELOG, "`findings_truncated: bool` envelope key (when `--limit`
caps) is reserved for v0.8.1 with a `schema_version` bump to `"1.1"`") —
so v0.9.0 **already has the schema_version bump** (1.1). This is
effectively the moment that deferred key should land, since paying the
minor-bump cost without using it for truncation observability is a
missed opportunity AND leaves JSON consumers in a worse state than
stdout.

**Suggested fix:**

Add `"findings_truncated": bool` (or `"truncated": bool`) as an 8th
top-level key in `render_envelope`. No further minor-bump needed — we're
already at "1.1". Update SRC-CLI-21/24 pair to assert the new key
alongside the stdout footer.

Alternatively, accept the miss for this slice and defer to v0.9.1 with a
note that v0.9.0 JSON consumers must re-query without `--limit` to
determine truncation. If deferred, the CHANGELOG "Known gaps" section
should say so explicitly — currently there is no such disclosure.

### PROD-SF-D2-003 — `Try \`folio status\`` suggestion is polite but practically incomplete

**Severity:** should-fix (error-message utility)

**Observed behavior:**

```
$ folio search framework --scope nonexistent-scope
Error: scope 'nonexistent-scope' does not resolve to an engagement or
document. Try `folio status` to list valid scopes.
```

Then the operator runs `folio status`:
```
$ folio status
Library: 6 documents
  ✓ Current: 6
```

That's the entire output. `folio status` does not actually list scopes
(valid --scope argument values: engagement subtree strings, document IDs,
etc.). The operator who just hit a scope typo is now told to run a
command whose output does not help them recover.

This is a shipped-as-suggested behavior (PROD-SF-003 closure from B.1) but
the closure was spec-level — it did not verify that `folio status` is
actually a useful destination. The suggestion pattern is correct (point
operators to a discovery command), but the suggested command doesn't
carry the information.

**Options:**

1. Change the error-message suggestion to something that actually lists
   scopes — e.g., `Try \`folio status\` for library counts, or list your
   engagement subtrees under your \`library_root\`.` (verbose, but
   accurate).

2. Leave the suggestion as-is and raise a follow-up slice to make `folio
   status` list valid --scope values (e.g., enumerate registry `decks`
   entries' `engagement` / `markdown_path` / `deck_dir` uniques). This is
   the more coherent long-term fix.

3. Take a middle path: reword to `Valid --scope values are: registered
   document IDs (see registry.json), engagement subtrees under
   library_root, or '-' for library-wide.` — doesn't point to a command
   at all, just describes the contract.

**Suggested fix:** take option (2) as a tracking note — defer the
suggestion fix to the `folio status` enhancement slice — AND in this
slice reword the error text to option (3). The current text sets up
operators for wasted effort.

**Why should-fix not blocker:** most operators will iterate on the scope
string (try a client name, an engagement) before reaching for `folio
status`, so the impact is contained. But it IS the first error an
operator hits when exploring search, and "the suggested command doesn't
help" is a credibility hit on a new surface.

### PROD-SF-D2-004 — Zero-flagged-excluded line adds noise on every run

**Severity:** should-fix (stdout readability, magnified by frequency)

**Observed behavior on the clean-library 3-proposal case:**

```
$ folio search framework
Search for 'framework' in scope: (library-wide)
Matches: 3
Excluded (flagged inputs in scope): 0

[queued] clienta_evidence_s0 --impacts--> clienta_evidence_t0  (enrich)
  shared framework concept
...
```

The `Excluded (flagged inputs in scope): 0` line is printed on EVERY
search (parent §11 rule 5 "no surface silently reports empty when
exclusion is the cause"). When the count is 0, this line is informational
noise — nothing was excluded, there's nothing for the operator to act on.

Parent §11 rule 5 is about **non-silent exclusion**, i.e., when an
exclusion OCCURS, surface it. When no exclusion occurred, the rule is
satisfied by the zero. Making the line conditional on `N > 0` would
be cleaner; the "always print zero" decision predates the §5 disclosure
discipline and was preserved conservatively from synthesize.

**Counter-argument:** the "always show" behavior signals to operators
that the search is trust-gating-aware by default — a security property
visible at every run. That's a legitimate design choice.

**Suggested fix:** either (a) gate the line on `N > 0` and add a single
`--verbose` or `--show-trust` affordance for always-show, OR (b) leave
as-is but update help/CHANGELOG to explicitly document "the zero line is
a by-design trust-surfacing signal."

**Why should-fix not blocker:** synthesize already ships the same
always-print line, so search is parity-consistent. But the noise
compounds across consumers; worth revisiting before the third consumer
bakes in the pattern permanently.

### PROD-SF-D2-005 — CHANGELOG operator-notes buries the `--producer` case-sensitivity trap

**Severity:** should-fix (CHANGELOG readability)

**Observed CHANGELOG text:**

> `--producer` is case-sensitive exact match. `folio search foo
> --producer Enrich` (mixed-case) prints a hint listing observed
> producer names in scope.

This reads as a feature description, not a caution. Operators reading the
CHANGELOG will likely skim past without registering that the flag is an
**unexpected** case-sensitive exception in a CLI where QUERY matching is
case-insensitive. Mixing one case-sensitive flag into an otherwise
case-insensitive surface is a subtle footgun.

**Why it matters:** `--producer enrich` works, `--producer Enrich` does
not (though the CSF-3 hint does partially recover, modulo the
misdirection described in PROD-SF-D2-001). Operators who've built muscle
memory around case-insensitive QUERY may paste a producer name with a
capital letter and get zero findings.

**Suggested fix:** reword the operator note to lead with the contrast,
e.g.:

> **Note:** `--producer` is case-sensitive (contrast: QUERY matching is
> case-insensitive). `folio search foo --producer Enrich` returns zero
> results and prints a hint. Producer names in the registry are
> lowercase by convention.

Equivalent polish in `--help` text for `--producer` would be valuable
too — current help reads `"Restrict to findings emitted by this exact
producer name (case-sensitive)."`. Adding "(contrast with
case-insensitive QUERY)" in help would prevent first-touch confusion.

## Product minor (PROD-MIN-D2-NNN)

### PROD-MIN-D2-001 — "(showing 1 of 3; use --limit to adjust)" reads awkward

The truncation footer says "use --limit to adjust" — but the operator
**just used** `--limit`. The phrasing presumes the operator wanted to
see all 3 and is being invited to lift the limit. A clearer wording:

```
(showing 1 of 3; increase --limit to see more)
```

or

```
(showing 1 of 3 findings; 2 additional matches elided — pass
--limit N to see more)
```

Minor cosmetic; the current version is understandable on second read.

### PROD-MIN-D2-002 — "in scope: (library-wide)" parenthesis is visually distracting

Current first line:
```
Search for 'framework' in scope: (library-wide)
```

The parenthesis around a value that's not actually parenthetical reads
like a footnote. Two cleaner options:

```
Search for 'framework' across library
Search for 'framework' (library-wide)  # scope label is the parenthetical
```

Trivial polish; ship-as-is is fine.

### PROD-MIN-D2-003 — `folio search --help` "Tip:" does not mention other likely over-matches

The tip calls out `draws_from`:

> Tip: QUERY matches `relation` too, so `folio search draws_from` will
> match every draws_from-typed proposal in scope (~1/6 of the queue).

But the same applies to QUERY against `producer` (`folio search enrich`
matches all enrich-produced proposals — potentially 100% of the queue).
A slightly broader tip:

> Tip: QUERY matches across all six text fields including `relation` and
> `producer`. E.g., `folio search draws_from` matches every draws_from
> proposal; `folio search enrich` matches everything the enrich producer
> emitted. Use `--scope ENGAGEMENT` or `--producer P` to narrow.

Minor; the `draws_from` example is sufficient to communicate the concept.

---

### PROD-MIN-D2-004 — JSON pretty-print format lacks a machine-readable compact variant

`--json` emits indented JSON (2-space). For machine consumers piping to
`jq`, indentation is fine. For logging / streaming / append-only archives,
a compact one-line variant (`--json-compact` or `--json --compact`) would
be cleaner.

Not a v0.9.0 concern; noted for completeness.

## Accept-as-is

The following behaviors tested cleanly and need no product-side change:

- **Click missing-argument message** (`folio search` with no QUERY): crisp
  `Usage:` line + `Error: Missing argument 'QUERY'.` + exit 2. Standard
  Click idiom, operator-familiar.

- **Empty / whitespace-only QUERY rejection**: `folio search ""` and
  `folio search "   "` both emit
  `Error: QUERY must contain at least one non-whitespace character.` at
  exit 2 via the `_validate_query` callback. The error text is
  actionable.

- **Negative --limit rejection**: `folio search foo --limit -1` emits
  `Error: Invalid value for '--limit': -1 is not in the range x>=0.` at
  exit 2 via `click.IntRange(min=0)`. Idiomatic Click, operator-familiar.

- **Missing registry.json**: `folio search foo` on a library with no
  `registry.json` exits 0 with the generic no-matches breadcrumb. No
  error surface (CB-3 closure held). Consistent first-touch shape.

- **JSON envelope on empty library**: `{schema_version: "1.1", command:
  "search", scope: null, query: "foo", trust_override_active: false,
  excluded_flagged_count: 0, findings: []}` — 7 keys, parseable,
  semantically obvious. `schema_version: "1.1"` is clearly new (not
  "1.0"); operators piping through `jq` will notice the bump
  immediately.

- **Help --scope copy**: `Omit or pass '-' for library-wide.` is concise
  and the `-` alias actually works. Good UX.

- **v0.9.1+ foreshadowing**: `--help` says "planned for v0.9.1+" with no
  date, consistent with v0.8.0's "planned for a future version"
  pattern and the CHANGELOG convention.

- **Library-wide scope label**: stdout header renders `(library-wide)`
  when no `--scope` supplied; JSON serializes `"scope": null`. Clear
  mental mapping.

- **Arrow rendering**: `[queued] src --impacts--> tgt  (enrich)` reads
  naturally; lifecycle state and producer annotation are visibly
  contextualized. `[flagged]` / `[flagged: source]` annotations in the
  `--include-flagged` case are clear. `[queued]` + `—` for missing
  `target_id` degrades gracefully.

- **Case-insensitive QUERY matching**: `folio search IMPACTS` matches
  proposals with `relation: "impacts"` — operator mental model matches
  implementation.

- **Help --limit**: `0 emits zero findings but still reports counts` is
  an unusually thoughtful help string, explicitly addressing the edge.

- **Trust override stdout line**: `Trust override active:
  --include-flagged` correctly annotates the `--include-flagged` path;
  `flagged_inputs` in JSON carries `["source"]` / `["target"]` for
  audit purposes.

- **Schema-version-bump storytelling**: the CHANGELOG "Changed" section
  explains the bump reason, cites synthesize §3.3, and clarifies that
  sibling consumers (graph, synthesize) do NOT bump. This reads well for
  an operator who'll eventually wonder why the three envelopes have
  different schema_version values.

- **CHANGELOG "Contract scaffolding" footnote**: the envelope-shape
  explicit listing is a nice touch for downstream tool authors.

## Summary

**Verdict: APPROVE-WITH-SHOULD-FIX.**

The implementation is operator-safe: all documented acceptance-criteria
behaviors fire correctly, error messages are crisp and actionable, and
the `--json` envelope parses cleanly on an empty library with clearly
new `schema_version: "1.1"`. The `--help` text leads with the "v0.9.0
structural MVP" signature, foreshadows v0.9.1 without a date, and
correctly calls out the over-match gotcha for `relation`-typed
queries.

**Five should-fix items land, of which the first two are the most
actionable:**

1. **PROD-SF-D2-001** — producer-hint misdirection when QUERY misses
   AND `--producer` is valid. Current behavior tells operators to run
   ingest/enrich when the queue actually has proposals; root cause is a
   one-line fix in `folio/search.py:267`. Highest first-touch hazard
   identified.

2. **PROD-SF-D2-002** — `--limit` truncation is invisible in `--json`.
   The JSON envelope has fewer observables than stdout for the same
   query; this is a stdout/JSON parity bug. The `schema_version` is
   already at "1.1" (paid the minor-bump cost); adding a
   `findings_truncated` top-level key would close this without further
   bump.

3. **PROD-SF-D2-003** — `Try \`folio status\`` suggestion points to a
   command that doesn't actually list scopes. Either enhance `folio
   status` (out-of-slice) or reword the suggestion inline
   (in-slice polish).

4. **PROD-SF-D2-004** — zero-flagged-excluded line renders on every run,
   adding noise. Parity with synthesize; worth revisiting before the
   pattern permanently bakes in.

5. **PROD-SF-D2-005** — CHANGELOG operator-note does not flag
   `--producer` case-sensitivity as an unexpected behavior; rewording
   to foreground the contrast with case-insensitive QUERY matching
   would close the trap.

**Four product-minor items** note cosmetic or deferred-observability
gaps (truncation-footer wording, stdout label parentheses, help-tip
coverage of `producer`-field over-match, compact-JSON variant).

**Nothing blocking.** The slice is shippable as-is if PROD-SF-D2-001
and PROD-SF-D2-002 are either addressed in a fix pass or explicitly
documented as known-gaps in the CHANGELOG before merge. PROD-SF-D2-003/4/5
and all PROD-MIN items are safely deferrable to a polish slice.
