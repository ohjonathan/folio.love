## Rationale

The explicitly cited relationship-proposal references are accurate against the current code on disk: `RelationshipProposal` spans `folio/pipeline/enrich_data.py:53`-`:109`; `folio/enrich.py:715` and `:985` are the emission and rejected-key filter; `folio/links.py:169`, `:184`, and `:371` are the consumer reads/write; `folio/graph.py:320` is the acceptance-rate rejected-count read; and `folio/cli.py:133` is a raw proposal status read.

The blocking issue is scope completeness. The spec says this is a proposal lifecycle rename and even reaches into provenance metadata via `cli.py:133`, but it omits `folio/provenance.py`, which owns the provenance proposal schema and many old-status code paths. That leaves Phase C with an ambiguous implementation boundary and a high chance of a partially migrated persisted schema.

For §3.4 edge cases: when both `status` and `lifecycle_state` are present, `from_dict` prefers `lifecycle_state`, and T-3 covers that behavior. When neither key is present, the proposed `from_dict` code defaults through `pending_human_confirmation` and maps to `queued`, but the test plan should add an explicit regression test for that default. Raw dict readers need the same old-to-new value mapping or explicit dual comparisons; the current §3.5 pattern is not enough for legacy pending values.
