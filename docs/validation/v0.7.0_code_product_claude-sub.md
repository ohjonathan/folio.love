# D.2 Product Review — folio-digest-v0-7-0

## Summary

The `folio digest` operator surface is largely faithful to spec v1.2 §4 + §11
+ §9.3. Across 12 directly-executed CLI scenarios (CliRunner; live LLM
mocked via `_call_llm`), every prefix (`✓` / `✗` / `↷`), pluralization
ladder (singular/plural/zero), Trust Notes 6-branch matrix, refresh
subtype-aware patch (DG-REG-5 path), Trust Notes deterministic rendering,
LLM-failure message format, weekly `--include-flagged` advisory, rerun
"Updated… (version N → N+1)" line, and `--help` example block reproduce
correctly. All 73 module + CLI tests pass; the 6 Trust Notes branches
match §9.3 character-for-character; the `Documents Drawn From` wikilink
section renders deterministically; the refresh patch correctly emits
`↷ <id>: skipping digest (source-less); rerun \`folio digest\` instead`
for digest rows and the generic non-digest message for other analysis
subtypes.

Two wording divergences from §4 / §11 prevent **Approve**:

1. **PROD-201 (should-fix):** `--date` parse error capitalization + missing
   trailing period diverges from spec §11 row 2 prescribed text.
2. **PROD-202 (should-fix):** Empty + override echo (SF-15) places the
   period inside the parenthetical rather than after, contradicting the
   §4 stdout example at line 448 character-for-character.

Both are surgical one-line fixes in `folio/digest.py`; neither blocks
`folio digest` semantics, but both are wording divergences from the
exact §4/§11 text the round-2/round-3 verdict committed v1.2 to ship.
One additional minor and two informational items are filed below.

## Detailed findings

### PROD-201 — `--date` parse error wording mismatch (should-fix)

**Source:** direct-run scenario B; spec §11 line 1042.

**Spec §11 row 2 prescribes** (line 1042):

> `✗ Invalid --date '{value}': expected YYYY-MM-DD format.`

**Implementation produces** (`folio/digest.py:166-168`):

```
✗ invalid --date 'bogus': expected YYYY-MM-DD format
```

Differences:
- Capitalization: spec has `Invalid` (capital I); impl has `invalid` (lowercase i).
- Trailing period: spec ends with a period; impl ends without one.

The CLI prepends `✗ ` correctly via `_emit_digest_result` for `status == "error"`, so the prefix is fine; only the message body diverges.

**Why this is should-fix and not minor:** the §11 failure-message table is the spec's canonical wording contract for operator-facing diagnostics; v1.2's verdict sequence (R2 → R3 closure) explicitly committed to "operator and reviewer can verify" the §11 messages match. Two character-level divergences in one error message break that commit. Both are one-line fixes:

```python
# folio/digest.py:166-168
raise ValueError(
    f"Invalid --date '{date_str}': expected YYYY-MM-DD format."
)
```

Spec §11 row 1 ("Invalid `<scope>`") follows the same `Invalid {thing} '{value}': {reason}` capitalization pattern, and the actual CLI output does emit `✗ Invalid scope '../../etc': scope ...` correctly (scenario C confirms capital "I"). The `--date` path is the lone outlier.

### PROD-202 — Empty + override echo punctuation mismatch (should-fix)

**Source:** direct-run scenario E; spec §4 line 448.

**Spec §4 line 448 prescribes** (the SF-15 override-echo branch):

```
No eligible inputs for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026 (--include-flagged was honored; no flagged inputs found either).
```

**Implementation produces** (`folio/digest.py:937-952`):

```
No eligible inputs for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026. (--include-flagged was honored; no flagged inputs found either)
```

Differences:
- The base message ends with a period (`...scope {scope}.`), and the override suffix is appended via `message += " (--include-flagged was honored; no flagged inputs found either)"`. This produces `... ClientA/DD_Q1_2026. (--include-flagged was honored; no flagged inputs found either)` — period **before** the parenthetical.
- The spec example places the period **after** the closing `)`.

Visually: spec has one sentence with parenthetical-then-period; impl has two-clause stutter (`. (`) and no terminal period.

The DG-CLI-10b test (`tests/test_cli_digest.py:303-313`) asserts only the substring `--include-flagged was honored`, which passes either way — so the test does NOT catch the divergence. The fix is in `digest.py`:

```python
# Replace lines 937-952 with:
if not selection.eligible:
    if selection.counts.excluded > 0 and not include_flagged:
        # ... existing flagged-only branch is fine (uses two-sentence form per spec §4 line 451) ...
    elif include_flagged:
        message = (
            f"No eligible inputs for daily digest on {day} in scope {scope} "
            f"(--include-flagged was honored; no flagged inputs found either)."
        )
    else:
        message = (
            f"No eligible inputs for daily digest on {day} in scope {scope}."
        )
```

**Severity rationale:** classified should-fix because the §4 stdout block is the explicit MN-103 / SF-107 closure surface — the spec literally enumerates the punctuation. Operators reading both the spec and the live CLI will see disagreement on a single sentence.

### PROD-203 — `_call_llm` profile-resolver kwarg shape (minor; verify, not assert)

**Source:** static-inspection `digest.py:704` vs spec §3.2 row `_call_llm`.

Spec §3.2 calls out `config.llm.resolve_profile(override=llm_profile, task=task)`. The implementation at `digest.py:704`:

```python
profile = config.llm.resolve_profile(llm_profile, task=task)
```

Passes `llm_profile` positionally (as the first arg `override`). This works if `LLMConfig.resolve_profile` signature is `def resolve_profile(self, override=None, *, task=...)`. The spec calls it `resolve_profile(override=llm_profile, task=task)` (named keyword) — both forms are equivalent if the signature is positional-allowed. Operator impact: zero. Test impact: none — the existing tests mock `_call_llm` directly, never reaching this line.

**Recommendation:** either add a one-line comment confirming the positional-arg form is intentional and matches the resolver signature, or align to spec's keyword form for symmetry with the spec text. Filed as minor only because operator surface is unaffected.

### PROD-I-201 — Rerun emits `Path:`; spec §4 example omits it (informational)

**Source:** direct-run scenario J vs spec §4 line 431–432.

Spec §4 rerun example (lines 431–432) shows two lines only:

```
✓ Updated daily digest: usbank_techresilience2026_analysis_20260404_daily-digest (version 3 → 4)
  Drawn from 8 inputs.
```

Implementation (scenario J) produces three lines:

```
✓ Updated daily digest: clienta_ddq126_analysis_20260404_daily-digest (version 1 → 2)
  Path: /tmp/.../clienta_ddq126_analysis_20260404_daily-digest.md
  Drawn from 1 input.
```

The `Path:` line is unconditionally echoed by `_emit_digest_result` (cli.py:1260-1261) for every successful write/rerun. **Operationally this is a strict improvement** — the operator gets the path on every rerun without needing to compute it. The spec example omits Path likely for brevity rather than as a prohibition (no normative language excludes it).

**Recommendation:** treat as deliberate enhancement; either annotate the spec rerun example to show Path is present (one-line edit to §4) or leave as-is. No code change required. Filed informational only.

### PROD-I-202 — LLM failure message single-line vs continuation indent (informational)

**Source:** direct-run scenario K.

The LLM-failure example in spec §11 row 5 reads as one continuous prose line:

> `✗ LLM synthesis failed for {type} digest on {period} in scope {scope} (existing digest, if any, preserved). Cause: {upstream}`

Implementation matches this exactly — single line, no continuation indent. The spec's other multi-line success cases use two-space continuation indent (`Path:`, `Drawn from`); the failure case does not. Operator-experience perspective: a long failure message on one line may wrap badly in narrow terminals. Could be improved by emitting `Cause: {upstream}` as a continuation line:

```
✗ LLM synthesis failed for daily digest on 2026-04-04 in scope ClientA/DD_Q1_2026 (existing digest, if any, preserved).
  Cause: api timeout
```

But spec §11 row 5 unambiguously prescribes the single-line form, so the implementation is correct-as-spec'd. Filed informational only as a possible v0.7.x ergonomics improvement, not a v0.7.0 fix.

## Verdict

**Needs Fixes.**

Two should-fix wording divergences from spec §4 line 448 and §11 row 2 prevent Approve. Both are mechanical one-line fixes in `folio/digest.py`:

1. **PROD-201:** `digest.py:166-168` — capitalize `invalid` → `Invalid`, append trailing period.
2. **PROD-202:** `digest.py:937-952` — restructure empty + override echo so the period closes the full sentence (after the parenthetical) rather than splitting it into two clauses.

PROD-203 is a recommendation (verify resolver-call kwarg shape matches spec §3.2 text); operator impact is zero. PROD-I-201 / PROD-I-202 are informational and do not require code changes.

After PROD-201 + PROD-202 fixes:
- Add a regression test asserting the exact `Invalid --date 'bogus': expected YYYY-MM-DD format.` string (currently DG-CLI-4 only checks for substring `YYYY-MM-DD`).
- Tighten DG-CLI-10b to assert the exact §4 line 448 text rather than a substring (`--include-flagged was honored`), to prevent recurrence of the punctuation drift.

Operator-experience baseline is otherwise solid: every other §4 example, all 6 Trust Notes branches, the refresh-patch DG-REG-5 path, and the LLM-failure message format reproduce exactly. The CLI `--help` block is discoverable and matches the §3.3 docstring. v0.6.4 / v0.6.5 vocabulary (`✓` / `✗` / `↷`, two-space indent, three-branch pluralization) is honored throughout.
