## Accept-as-is / clean narrowing

The v0.9.0 `folio search` spec represents an exemplary clean narrowing of the parent `tier4_discovery_proposal_layer_spec.md` (and its sub-slice 1 and 2 predecessors). 

**Key Alignment Confirmations:**
- **Parent §5 (Proposal Object Contract):** The spec faithfully maps all 11 required minimum-contract keys to `SearchFinding` and the JSON envelope, ensuring strict parity with graph (v0.7.1) and synthesize (v0.8.0). `schema_gate_result` and `input_fingerprint` are correctly handled as `None` and legacy-alias respectively, mirroring the sibling consumers.
- **Parent §11 (Trust-Gated Surfacing):** Rules 1-5 are completely honored. The default exclusion of flagged inputs, the explicit `--include-flagged` override, and the routing of `excluded_flagged_count` from `SuppressionCounts.flagged_input` directly carry forward the Phase 0 baseline behavior.
- **Parent §12 (Shared Consumers):** The 3-way (N=3) shared-consumer invariant is explicitly defined and enforced in tests (`test_search_module_uses_shared_helper`). Search correctly uses the canonical `collect_pending_relationship_proposals` and `derive_trust_status` contract boundaries.
- **Parent §15.6 & §15.7 (Manifests narrow, don't reinterpret):** The spec meticulously catalogs what is implemented and what is deferred (e.g., regex, multi-lifecycle search, ranking). Parent sections are cited by number and name without paraphrase.
- **Envelope Versioning Policy:** The minor-version bump (`schema_version: "1.0"` → `"1.1"`) is a flawless application of the synthesize v0.8.0 §3.3 policy ("Producers MUST bump minor when adding a top-level key"). The addition of the `query` top-level key justifies the exact bump executed here.
