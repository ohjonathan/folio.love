## 3. New findings

### PR-101 — `_atomic_write` tempfile cleanup has subtle ordering defect (should-fix)

**Severity:** should-fix (functional defect; the fix for PR-7 doesn't fully close PR-7).

**Evidence:** §10.2 lines 826-839 implements:

```python
tmp_path: Path | None = None
try:
    with tempfile.NamedTemporaryFile(
        mode="w", encoding="utf-8", dir=str(path.parent), delete=False, suffix=".tmp"
    ) as tmp:
        tmp.write(content)               # ← if THIS raises, tmp_path is still None
        tmp_path = Path(tmp.name)         # ← runs only on successful write
    os.replace(tmp_path, path)
    tmp_path = None
except Exception:
    if tmp_path is not None and tmp_path.exists():
        tmp_path.unlink(missing_ok=True)
    raise
```

`NamedTemporaryFile(..., delete=False)` creates the temp file on disk *immediately*. The current ordering captures `tmp_path = Path(tmp.name)` AFTER `tmp.write(content)`. If `tmp.write` raises (e.g., disk full mid-write, encoding error), `tmp_path` is still `None`, and the except branch's `if tmp_path is not None` guard skips the cleanup. The orphan tempfile remains on disk — exactly the bug PR-7 was trying to close.

**Required fix:** Capture `tmp_path` BEFORE the write call:

```python
with tempfile.NamedTemporaryFile(
    mode="w", encoding="utf-8", dir=str(path.parent), delete=False, suffix=".tmp"
) as tmp:
    tmp_path = Path(tmp.name)            # capture name FIRST
    tmp.write(content)
os.replace(tmp_path, path)
tmp_path = None
```

Test DG-20b at spec line 959 asserts "no orphan `.tmp` file remains in digest folder" but does not specify which write failure mode is patched. To prove the fix, DG-20b's mock should fail INSIDE `tmp.write(content)` (e.g., monkeypatch `tempfile.NamedTemporaryFile` to return a tmp whose `write` raises), not just after the with-block exit. Otherwise DG-20b passes vacuously and the bug ships.

### PR-105 — `_strip_duplicate_section` naming/contract ambiguity (minor)

**Severity:** minor (contract clarity).

**Evidence:** §3.2 line 238 names the helper `_strip_duplicate_section(body, heading) -> str` and describes: "Remove a duplicate `## <heading>` section". The wording "a duplicate" suggests the helper removes one duplicate.

But §9.5 line 795 says: "calls `_strip_duplicate_section(llm_body, "Trust Notes")` to remove **any** LLM-emitted `## Trust Notes` block". Here "any" implies zero, one, or more.

The actual semantics matter for the §9.5 invariant ("the final digest body has exactly one `## Trust Notes` section"). If the LLM emits exactly one `## Trust Notes` section, `_strip_duplicate_section` should remove it (so the programmatic block is the only one). If the LLM emits zero, the helper should be a no-op. If the LLM emits two or more, the helper should remove all.

The name `_strip_duplicate_section` reads as "remove second-and-subsequent occurrences" (the "duplicate" connotation). But the §9.5 use case requires "remove all occurrences regardless of count". Pick one:

- **(a)** Rename to `_strip_section(body, heading)` — semantics: remove ALL occurrences of `## <heading>` sections from `body`. Caller responsibility: only call when the section will be re-added programmatically.
- **(b)** Keep `_strip_duplicate_section` with the strict interpretation "remove second-and-subsequent" — and add a separate guard in §9.5 to handle the "exactly one but rogue" case.

Option (a) matches §9.5's intent more cleanly.
