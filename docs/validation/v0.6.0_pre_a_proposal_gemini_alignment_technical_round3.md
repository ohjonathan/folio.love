## 2. B-4 closure assessment (THE scope)

- Rev 4 change summary (quote the relevant sentences of §9.2): "Cumulative acceptance-rate gate. Any producer must sustain a cumulative acceptance rate of at least 50% across all its *reviewed* proposals... The rate is computed at query time by aggregating `lifecycle_state` values from all entries of `_llm_metadata.<producer>.axes.relationships.proposals` across all managed documents — no `rejected_at` timestamp, no global chronological ordering, and no new per-producer persistence is required."
- Closure verdict: closed
- Evidence (cross-check against folio/links.py):
  - Does the cumulative framing avoid the `rejected_at` dependency? Yes. A cumulative calculation sums the historical terminal states directly, eliminating the need for chronological ordering and `rejected_at` timestamps.
  - Is the aggregation computable from existing frontmatter? Yes. All necessary data is present without new persistence. Rejections are found in the producer's proposals list (`status == "rejected"`), and acceptances are found in `_llm_metadata.links.confirmed_relationships`, which explicitly tracks the `producer` of the accepted proposal.
- New alignment / technical issues introduced by Rev 4: There is a minor technical inaccuracy in the spec's data path. §9.2 states the rate is computed by aggregating "from all entries of `_llm_metadata.<producer>.axes.relationships.proposals`". However, `folio/links.py`'s `confirm_proposal` function removes accepted proposals from that list and moves them to `_llm_metadata.links.confirmed_relationships`. Computing the rate *solely* from the producer proposals list would incorrectly yield 0 acceptances. This is a spec documentation bug, not an architectural blocker, as the data is readily available in the current frontmatter.

## 3. Scope sanity
Did Rev 4's fix grow scope beyond what the spec can carry?
No. Using a cumulative rate rather than a rolling window reduces complexity and keeps scope bounded to the existing frontmatter state model.

## 5. Verdict
Proceed to Phase A

Justification: The architectural blocker B-4 is successfully closed. The core issue in Rev 3 was the dependency on non-existent `rejected_at` timestamps to compute a rolling window. Rev 4's shift to a cumulative acceptance-rate gate completely eliminates this dependency. While Rev 4 introduces a minor technical inaccuracy regarding the exact JSON path for accepted proposals (they reside in `links.confirmed_relationships`, not the producer's proposal list), this does not require any new persistence or state changes. The required aggregation is fully operationalizable against the current `folio/links.py` state model, and the data path correction can be handled cleanly during Phase A implementation.
