# Peer Review — folio-graph-generalized-proposals-v0-7-1 / B.1 / claude-sub

## 1. Completeness check

The spec is structurally complete: frontmatter, Overview (§1), Scope
(§2.1 in-scope / §2.2 carry-forward / §2.3 out-of-scope), Changes (§3),
Tests (§4), Validation (§5), Open Questions (§6), References (§7). The
revision note is present. The `consumes_proposal` frontmatter key
correctly cites the parent spec at rev 5.

However, several completeness gaps surface on close read:

- **§3 renderer pseudocode does not cover the CLI stringification path.**
  The spec changes `subject_id` from `view.proposal.proposal_id` to
  literal `None` (§3.1 "Note on subject_id conflict") but does not
  account for the existing non-JSON CLI printer at
  `folio/cli.py:2019` which interpolates `finding['subject_id']` into
  the human-readable line. After this change the operator sees
  `[medium] pending_relationship_proposal None: <source> has pending …`.
  Neither §3 nor §4 mention updating `graph_doctor_cmd` or adjusting
  its stringification for the new shape. Direct-run evidence:
  `folio/cli.py:2019` reads `f"[{finding['severity']}] {finding['code']} {finding['subject_id']}: {finding['detail']}"`.
- **§4 does not mention the pre-existing test that will fail.**
  `tests/test_graph_cli.py:389-392` asserts finding keys are EXACTLY
  `{"code", "severity", "subject_id", "detail", "recommended_action"}`.
  Adding nine/eleven §5 keys will fail this equality assertion. The
  test plan in §4 only lists *new* tests; it does not call out the
  existing assertion that must be relaxed or replaced. An implementer
  following §4 verbatim will ship a broken regression.
- **§3 does not specify where `_SHARED_PROPOSAL_CONTRACT_FIELDS` is
  re-exported for tests.** §4 test 1 reads "assert every name in
  `_SHARED_PROPOSAL_CONTRACT_FIELDS` is a key on the finding dict" but
  the constant is module-private (leading underscore). Either tests
  import the private constant (convention-permitted but
  unstated) or the spec should promote it; the implementer will have
  to guess.
- **No `--json` golden-shape sample.** B2 says "The `--json` output …
  MUST serialize the above nine §5 fields by the §5 names" but §4 does
  not list a JSON-shape test using the CLI runner (the existing
  precedent at `tests/test_graph_cli.py:378` tests JSON shape). The
  §4 tests all appear to exercise `graph_doctor(...)` directly, not
  the CLI json output.
- **Field count inconsistency.** §1 Overview, B2, B3, §5.1, §5.3 all
  say "nine §5 fields". The §2.1 table enumerates ten `§5 field`
  rows (proposal_type, source_id, target_id, subject_id,
  evidence_bundle, reason_summary, trust_status, schema_gate_result,
  producer, input_fingerprint, lifecycle_state = 11 distinct names),
  and `_SHARED_PROPOSAL_CONTRACT_FIELDS` in §3 has 11 entries. The
  parent spec §5 table at `docs/specs/tier4_discovery_proposal_layer_spec.md:130-140`
  has 9 rows because it collapses source_id/target_id/subject_id into
  one row. The narrowed spec should reconcile: either call them
  "nine §5 rows surfaced as eleven keys" or consistently say "eleven
  field keys". As written, an implementer reading "nine" and counting
  eleven cannot tell whether something is missing.

## 3. Quality assessment

The design is sound in its intent: surface every §5 field by its
contract name so that future proposal types flow through a shared
renderer. The narrowing discipline (§2.2 carry-forward table) is
correctly structured against parent §15.7, and the helper-divergence
disclosure at §3.2 meets F-050's shape. The risk framing in §1 is
calibrated to the actual change surface.

The implementability bar is, however, lower than the spec's
confidence suggests. Three issues combine to make this harder than
"a mid-level engineer implements without questions":

First, the rename of `subject_id` from a meaningful value (the
proposal ID) to literal `None` is documented as a deliberate breaking
change in §3.1, but the human-readable CLI path that consumes
`finding['subject_id']` is not addressed. An engineer will either
ship the literal `"None"` in operator output (bad UX) or will edit
`folio/cli.py` to branch on `code` (a scope expansion the spec does
not authorize). Neither path is documented. OQ-1 reaches the right
conclusion but omits this follow-on consequence.

Second, the §4 test plan is dense on new conformance but silent on
regression. The existing `assert set(finding.keys()) == {...}` at
`tests/test_graph_cli.py:390` will break. §4 test 11
("test_graph_doctor_legacy_keys_preserved") asserts presence but not
equality, so it does not replace the existing equality test. The
spec should either (a) state that line 390 is updated to use
`<=` (subset) or to a known-exact superset, or (b) add a test
migration note.

Third, the proposal-ID displacement is under-justified. §3.1 claims
"Consumers that needed the proposal ID should read
`input_fingerprint` (stable across reruns) or derive from
`(source_id, target_id, producer)`." But `compute_relationship_proposal_id`
at `folio/pipeline/enrich_data.py:120-126` uses the tuple
`(source_id, relation, target_id, basis_fingerprint)` — the provided
guidance omits `relation`, so a consumer following the spec's recipe
will collide proposals that differ only by `relation`. Either
document `relation` in the recovery tuple or, more cleanly, preserve
`proposal_id` under a non-§5 key (e.g., `legacy_proposal_id` or
`finding_id`) so that CLI stringification, logs, and `folio links
review` cross-references continue to work without string surgery.

## 4. UX review

The `folio graph doctor` non-JSON path prints
`[severity] code subject_id: detail` today. After the spec:
`[medium] pending_relationship_proposal None: <source> has pending
impacts -> <target> from enrich`. The literal word `None` in
operator output is the UX regression this spec leaves on the table.
Even if `subject_id` semantics were wrong before, shipping literal
`None` is worse than shipping a proposal ID. This must be fixed
before merge — either by branching the CLI printer on `code`, by
preserving the proposal ID under a different key, or by passing an
alternate label (e.g., `view.source_id`) through `subject_id` for
relationship proposals.

The `--json` path is additive and does not regress existing
consumers that only read legacy keys. New consumers get the §5 shape.
OQ-3 correctly flags the CHANGELOG Breaking entry.

Error messages, help text, and command surface are unchanged — the
cardinality assertion at manifest line 105-107 pins `["doctor",
"status"]` and is verified in this review (direct-run output:
`['doctor', 'status']`).

## 6. Positive observations

- **§2.2 carry-forward table is exemplary.** It cleanly partitions
  parent §§ into implements / inherits / defers with notes, following
  the v0.6.0 §11 pattern. This is the single most important part of
  a §15.7-compliant narrowing spec and it is done well.
- **§3.2 helper-divergence disclosure is correctly F-050-shaped.** It
  documents derived-vs-contract explicitly, names the producer that
  is *not* being changed, and justifies locality.
- **The `subject_id` rename is flagged honestly as a breaking
  change** (§3.1 final paragraph, OQ-3) rather than smuggled. Even
  though P-1 shows the downstream consequence is incomplete, the
  intent to surface the change is the right discipline.
- **Risk framing in §1 is calibrated.** "Narrow retrofit; no new
  machinery; no migration" is accurate given the forbidden-path
  scope lock at manifest lines 76-94. Primary-risk callout
  (silent-field-drop) maps directly to test 1.
- **Cardinality assertions in §5.2 are sharp and mechanical.** Each
  of the three can fail cleanly and be read from source, matching
  the v1.2.0 tightening of verify-cardinality semantics.
- **OQ-1 and OQ-2 both reach defensible conclusions with recorded
  rationale**, and the dispositions ("resolved-during-A") are
  appropriate for B.1-entry state.

## 7. Verdict

**Needs Fixes.**

Two blocking findings (P-1 CLI stringification regression, P-2
missing test-migration note) must be resolved in B.2 before the
canonical spec verdict can approve. Four should-fix (P-3 field-count
inconsistency, P-4 proposal-ID recovery recipe, P-5 missing CLI
JSON test, P-6 private-constant import ambiguity) are all correctness
or implementability issues a mid-level engineer would legitimately
ask about.

The spec is close — the design is sound, the narrowing discipline
is correct, and the test surface is nearly right. But §3.1's
`subject_id` rename has a cascading downstream UX regression the
spec does not confront, and the test plan does not account for the
one existing equality assertion the additive shape will break. Both
are mechanical fixes but both are real.
