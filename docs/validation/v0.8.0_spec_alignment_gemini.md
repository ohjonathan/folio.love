# v0.8.0 Folio Synthesize — Alignment Review

## Summary
The v0.8.0 specification successfully establishes `folio synthesize` as a structural read-only surface, forming a legitimate §15.7 narrowing of the parent product shape. It honors the v0.7.1 carry-forwards (trust helper promotion and JSON schema versioning) and correctly implements the parent §11 trust-gating contract. However, the spec contains a critical contradiction between its §1.2 claims and actual mechanisms regarding the §8 review rendering contract, and it improperly drops two mandatory §5 contract fields (`schema_gate_result` and `input_fingerprint`) from its envelope, breaking shared-consumer uniformity.

## Dispositions that look like reinterpretation

1. **Missing §5 fields:** Target spec §6 states that `schema_gate_result` and `input_fingerprint` are "(NOT surfaced in v0.8.0 envelope)". Sub-slice 1 (`folio graph doctor`, v0.7.1) successfully emitted these 11 keys (with the minimal gate for `schema_gate_result` and legacy alias for `input_fingerprint`). Omitting them from the v0.8.0 envelope breaks the "minimum shared contract" required by parent §5 and parent §12 for shared consumers. The JSON envelope must include these keys for uniformity.
2. **§1.2 vs §3.5 Contradiction:** Target spec §1.2 claims `schema-gate summary on stdout` and `full evidence-bundle rendering (1-3 locator lines per finding)` are implemented. This directly contradicts the mechanisms defined in §3.5 (which lacks both) and §2.5 (which explicitly defers full evidence-bundle rendering). §1.2 must be corrected to reflect the actual partial implementation (or §3.5 must be updated to include the schema-gate summary).

## Missing parent-§ coverage
None. The spec references all relevant parent §§, explicitly deferring out-of-scope sections like §9 (Queue Bounds), §10 (Rejection memory at surface time, already handled upstream), and §13 (Validation workstream).

## Product-Shape Narrowing Check
Parent §15.6 leaves the synthesize product-shape open. The v0.8.0 spec's decision to ship a structural-only MVP (no LLM narrative synthesis, no new doc artifacts) is a **legitimate §15.7 narrowing**. By isolating the risk of the greenfield CLI and shared-consumer contract integration from the risk of LLM prompt design, it avoids taking on orthogonal risks simultaneously. It does not underclaim, as it fulfills the §15.6 shared-consumer requirement by grouping and surfacing proposal findings accurately.

## Attack Check Responses
1. **Does the spec's §1.2 "sections implemented" table match what the spec actually implements?** No. §1.2 claims §8 Review Rendering is implemented with "schema-gate summary on stdout" and "full evidence-bundle rendering in §6 (1-3 locator lines per finding)". Neither is implemented in the §3.5 stdout renderer, and §2.5 explicitly defers the evidence locators.
2. **Does §1.3 "sections NOT implemented" table match what the parent requires for an in-scope shared consumer?** Yes, no silent deferral of an in-scope requirement, aside from the §5 fields improperly dropped from the mapping in §6.
3. **Does §2.2 forbidden_paths match the spec's §2.1 allowed_paths?** Yes, there are no contradictions and no missing exclusions.
4. **Does §3.3 envelope design honor parent §8?** Partially. It implements evidence bundle, reason summary, and trust status, but explicitly omits `schema_gate_result` (schema warning), violating the full parent §8 rendering contract. 
5. **Does §5 trust-gating map every parent §11 rule 1-5 to a mechanism?** Yes. All rules (default exclusion, override flag, explicit trust label, excluded count disclosure) are correctly wired to the CLI and JSON outputs.
6. **Does §6 §5-field-mapping table accurately cite parent §5 rows?** It maps the rows, but it explicitly drops Row 6 (`schema_gate_result`) and Row 8 (`input_fingerprint`), failing to satisfy the "minimum shared contract" required by parent §5.
