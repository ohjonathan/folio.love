## Blockers (must close before Phase C)

### B-001 — `@click.argument("scope", required=True)` incompatible with `ScopeOrCommandGroup`

**Source:** peer PR-1.

**Evidence:** `folio/cli.py:30-33` defines `ScopeOrCommandGroup.parse_args` which rewrites a non-flag first positional into `--scope <arg>` *before* calling `super().parse_args`. The two existing consumers (`provenance` at cli.py:1174-1187, `links` at cli.py:1580-1583) both use `@click.option("--scope", default=None, hidden=True)`. With the spec as written, `folio digest ClientA/DD_Q1_2026` would be silently rewritten by the parser and the function would receive `None` for the `scope` positional, then fail required-validation.

**Required fix in v1.1:** Change spec §3.3 from `@click.argument("scope", required=True)` to `@click.option("--scope", default=None, hidden=True)`, plus an explicit `if scope is None: raise click.UsageError("scope is required")` in the body (or `ctx.fail` for click-native error path). Also update §4 to clarify that `<scope>` is presented as a positional in user docs but parsed as a hidden option per the established `ScopeOrCommandGroup` convention. Add a test (DG-CLI-3 already exists; expand its assertion).

### B-002 — §3.3 cites the wrong pattern (cli.py:1174 is `provenance`, not `enrich`)

**Source:** peer PR-2.

**Evidence:** `folio/cli.py:1174` is `@cli.group(cls=ScopeOrCommandGroup, invoke_without_command=True)` for `provenance`. The `enrich` symbol at cli.py:1132 is `@cli.command()` (a flat command, not a group). The spec's §3.3 text "Pattern matches the existing `enrich` group (cli.py:1174 — same `ScopeOrCommandGroup` + `invoke_without_command=True`)" is wrong on both axes (wrong symbol named, and `enrich` doesn't even use this pattern).

**Required fix in v1.1:** Replace the pattern citation with `provenance` (cli.py:1174) and also reference `links` (cli.py:1580) as the second consumer of `ScopeOrCommandGroup`. This is purely a documentation correction; the underlying design intent (use `ScopeOrCommandGroup` with `invoke_without_command=True`) is correct and aligns with B-001's fix.

## Should-fix (close in v1.1)

Numbered in priority order (highest-impact first). Each maps to source finding(s).

### SF-1 — Rename invented API `LLMConfig.profile_for_task` → `resolve_profile`

**Source:** peer PR-3. Spec §3.2 and §4 reference `LLMConfig.profile_for_task` — no such method exists. Actual API is `LLMConfig.resolve_profile(override, task=...)` at config.py:107, used by 14 callers (enrich.py:265/534 and others).

**Fix:** Replace all references in §3.2 and §4 with `LLMConfig.resolve_profile(override, task="digest")`. Update `_call_llm` signature comment to match.

### SF-2 — Re-dispose CF-1 (cli.refresh rerun-guidance message)

**Source:** peer PR-8 + product PROD-SF-5 + adversarial ADV-002 partial.

The existing message at cli.py:1053-1056 already emits guidance — `"↷ {entry.id}: skipping analysis document (source-less); rerun the originating analysis workflow instead"`. CF-1's framing in v1.0 ("would require modifying cli.py outside the digest group") is wrong: cli.py IS in the manifest's allowed_paths.

**Fix:** Choose option 1 (recommended by product PROD-SF-5): expand cli.py:1053-1056 to detect `subtype == "digest"` and emit `"... rerun 'folio digest' instead"` for digest rows. ~5 lines. Update §14 CF-1 to mark as closed-in-this-slice. Adds a test (DG-REG-5) to assert the digest-specific wording.

### SF-3 — Define source-backed frontmatter read failure policy (fail-open per v0.6.4 contract)

**Source:** adversarial ADV-001.

Spec §5 step 3.iv calls `_is_flagged(fm.get("review_status"))` but doesn't define behavior when `registry._read_frontmatter` returns `None` (file missing or YAML malformed; registry.py:427 returns `None` in those cases). Mirror v0.6.4 CB-1 contract: treat unreadable frontmatter as "not flagged" (fail-open) — consistent with `_resolve_target_flagged` behavior at links.py:113.

**Fix:** Add explicit policy in §5 step 3.iv: "If `_read_frontmatter` returns `None` (missing file, malformed YAML), treat the input as not-flagged for trust-gating decisions; the input is still eligible if other predicates pass. This matches the v0.6.4 fail-open contract for unreadable trust state." Add tests for missing-file and malformed-YAML cases (extend DG-11 + add DG-11b/c).

### SF-4 — Define write-then-register orphan recovery (and verify §11 recovery guidance)

**Source:** adversarial ADV-002.

Spec §3.2 ordering writes the file before `_register_digest`. If `registry.upsert_entry` raises (corrupt registry, OSError), the file exists but registry has no entry. Spec §11 says "instructs operator to run `folio refresh`" — but `cli.refresh` does NOT scan the filesystem for orphan source-less analysis files; only `rebuild_registry` does (registry.py:145-175). The recovery guidance is wrong.

**Fix:** Two parts:
1. Update §11 row "Registry upsert failure" recovery guidance to either (a) recommend `rebuild_registry` (the correct command) or (b) say "rerun the same `folio digest` command" (which will recompute the same path and re-register).
2. Add §3.2 helper `_load_existing_digest`: extend so that on digest start, if computed path exists but registry row is missing, load the existing digest frontmatter and upsert before proceeding (self-healing). Note this in §10.1 rerun semantics.

Add test DG-20b: kill registry mid-write, verify rerun re-registers cleanly.

### SF-5 — Define daily-input date-comparison timezone explicitly

**Source:** adversarial ADV-003.

Spec §5 uses `.split("T")[0]` for ISO timestamps but doesn't specify timezone interpretation for the `--date` argument or the local current date default. A `modified: 2026-04-04T23:00:00-05:00` value would compare as `2026-04-04` via split — but in UTC it's `2026-04-05`. ISO week boundary in §6 has the same ambiguity.

**Fix:** Add §5 paragraph: "Date comparisons use the literal `YYYY-MM-DD` substring of the frontmatter `modified`/`converted` field (via `.split('T')[0]`), regardless of any timezone offset suffix. The `--date` argument and the default-current-date are interpreted as the operator's local-system date. Cross-timezone notes whose local date differs from their stored UTC date may match the digest of a different day; this is intentional for v0.7.0 (single-engagement, single-operator workflow) and revisited if multi-operator/multi-timezone use surfaces." Add test DG-13b for timezone-bearing timestamps.

### SF-6 — Define rerun corruption cases (corrupt YAML, file/registry divergence, non-integer version)

**Source:** adversarial ADV-004.

Spec §10 says rerun increments version but doesn't define behavior for: corrupt existing-digest YAML (`_load_existing_digest` cannot parse), non-integer `version` field, missing `created` field, file exists but registry row absent (orphan, see SF-4), registry row exists but file absent.

**Fix:** Add §10.4 "Rerun corruption fallback":
- Corrupt YAML: log warning, treat as `version: 1` and overwrite (rationale: digests are regenerable; data loss is bounded to the prior synthesis text).
- Non-integer version: coerce to `int(value) + 1`; on failure, fall back to `version: 1`.
- Missing created: use `today`.
- File-without-registry: see SF-4 self-heal.
- Registry-without-file: treat as fresh write (`version: 1`), warn.
Add tests DG-19b (corrupt YAML rerun), DG-19c (missing-file with stale registry).

### SF-7 — Trust Notes collision detection

**Source:** adversarial ADV-005.

Spec §9.3 renders Trust Notes programmatically but `_validate_body_sections` (§3.2) only checks heading PRESENCE. If the LLM body output happens to contain its own `## Trust Notes` heading, the final body has TWO trust-notes sections — semantically conflicting.

**Fix:** Update `_validate_body_sections` contract in §3.2 to detect and remove duplicate `## Trust Notes` from the LLM-returned body BEFORE the programmatic block is appended. Or alternatively: scan LLM body, raise on duplicate trust-notes heading (one-retry path per §11). Add test DG-23b for LLM-returned body containing rogue trust-notes.

### SF-8 — Define LLM retry semantics

**Source:** adversarial ADV-006.

Spec §9.1 last paragraph and §11 row 6 say "missing required heading after one retry" — but doesn't say (a) is retry the same prompt or a corrective prompt naming the missing heading, (b) is `_call_llm` itself responsible or does the caller orchestrate, (c) is retry on transient errors (timeout, rate limit) separate from retry on validation failure?

**Fix:** Add §9.4 "Retry policy":
- Validation retry (missing heading): one corrective re-prompt naming the missing heading; if second response still invalid, fail per §11. Helper: `_call_llm` returns raw string; caller (`generate_*_digest`) orchestrates retry.
- Transient error retry (timeout, rate-limit, 5xx): NOT in scope for v0.7.0; transient errors fail-fast per §11. Carry-forward CF-8 added.
Add test DG-23c: missing-heading first response, valid second response → succeeds. DG-23d: missing-heading both responses → exit non-zero.

### SF-17 — `_atomic_write` tempfile cleanup

**Source:** peer PR-7.

`NamedTemporaryFile(delete=False)` + `tmp.write(content)` — if write raises mid-stream, tempfile is orphaned.

**Fix:** Wrap §10.2 atomic-write code in try/except that removes the tempfile on any exception before re-raising:

```python
tmp_path = None
try:
    with tempfile.NamedTemporaryFile(...) as tmp:
        tmp.write(content)
        tmp_path = Path(tmp.name)
    os.replace(tmp_path, path)
except Exception:
    if tmp_path and tmp_path.exists():
        tmp_path.unlink(missing_ok=True)
    raise
```

Add test DG-20b (write-failure leaves no tempfile).
