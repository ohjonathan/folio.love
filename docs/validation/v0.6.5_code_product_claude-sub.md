## Operator migration story (solo operator walkthrough)

Walk-through: operator on v0.6.4 → v0.6.5 without reading the spec.

1. **Runs `folio entities suggest-merges`** — sees the familiar candidate list, and now two extra lines at the bottom: `No merge candidates suppressed by rejection memory.` + `(0 total rejections recorded.)`. The phrasing is self-explanatory; no pointer to `reject-merge` is required because the operator can infer from context. **No friction.**
2. **Runs `folio entities --help`** — `reject-merge` appears in the command list with a one-line summary (`"Reject a merge candidate so it stops surfacing in..."`). **Discoverable.**
3. **Runs `folio entities reject-merge alice_chen chen_alice`** — sees `✓ Rejected merge candidate: Alice Chen [alice_chen] <> Chen, Alice [chen_alice] (reasons: ...)`. The echoed reasons give the operator an audit trail of why the candidate was flagged in the first place. **Confidence-building.**
4. **Re-runs `folio entities suggest-merges`** — the pair is gone from the list, `1 merge candidate suppressed by rejection memory.` and `(1 total rejections recorded.)` both increment. **The feedback loop closes; the operator can see their reject "stuck".**
5. **Re-runs `folio graph status`** — `Reviewable duplicate person candidates: 0` (was 1 before). The label change signals the semantic shift. **Matches CHANGELOG.**

### `folio entities merge` workflow friction check

The `merge` command (`entities.command("merge")`) is unchanged by this slice. The spec §4.2 locking-scope decision explicitly defers broader entity-family locking to a follow-up. The documented failure mode — "Operator running `merge` and `reject-merge` concurrently may see an `EntityRegistryError` for stale keys (documented, not silent corruption)" — is an acceptable tradeoff for a solo operator (who is running these commands serially in one terminal). The `merge` command's help text does not cross-reference `reject-merge`, but that's OK: `reject-merge` is surfaced adjacent to `merge` in `entities --help` and from the `suggest-merges` disclosure flow, so operators will find it by the natural "I don't want to merge this one" impulse. **No friction flagged.**

Informational only.
- CHANGELOG does NOT separately call out `reject-merge`'s exit-code contract (0 for happy/idempotent, non-zero for stale). The spec §4.2 defines it; operators scripting this command would want to know. Minor.

### Informational (I)

**I-PR-001: CHANGELOG does not enumerate `reject-merge` exit-code contract.** The v0.6.5 CHANGELOG entry for `folio entities reject-merge` describes persistence and locking but does not mention the three-state exit-code semantics (`0` on successful reject, `0` on idempotent no-op, non-zero on stale key). Operators scripting `reject-merge` in a shell loop would need to read the spec or the source to know that `= Already rejected:` is exit-0 and not a failure. Not a blocker — symmetric to slice 1 `folio links reject` CHANGELOG which also omits exit-code tables — but worth noting for a future docs pass. Spec §4.2 already gets this right.

**I-PR-002: `reject-merge` success-line reasons echo depends on the last `rejected_merges` record being the one just written.** `cli.py:2599-2607` reads `reg._data.get("rejected_merges", [])[-1]` after `reg.save()`. If a concurrent writer appended a different record between save and readback, the operator would see wrong reasons in the echo. The `library_lock` wrapper makes this vanishingly unlikely (both writers hold the same lock), and the impact is cosmetic (reasons display only), but the pattern of "read the record we just wrote via index `[-1]`" is a latent footgun if locking ever weakens. A cleaner pattern would be to have `reject_person_merge` return the stored record (or reasons list) directly. Informational only — the spec §4.1 item 6 specifies the tuple return signature, so changing it is out of scope for this slice.

**I-PR-003: Non-person `--type` branch is unreachable under current Click choice constraint.** `cli.py:2497` (after diff) renders `"No merge heuristics available for this entity type."` + disclosure for non-person types, but `--type` is declared as `Choice(["person"])` so Click rejects other values at argument-parsing time (Scenario G confirms exit-2 with `Invalid value for '--type'`). The defensive dead-code path is OK — if `suggest_person_merges` ever generalizes to other entity types the scaffolding is in place. No fix required.
