## Should-fix

**SF-peer-1 — Dataclass field order diverges from spec `asdict` ordering (maintainability).**
Spec §3.3 example envelope lists the 11 shared-contract keys in a specific order (matching graph's `_SHARED_PROPOSAL_CONTRACT_EMITTED_KEYS` at `folio/graph.py:30-42`). Because `render_envelope` uses `asdict(f)` (folio/synthesize.py:101), the JSON key order reflects the dataclass field order at `folio/synthesize.py:32-44`, not the graph tuple order. Specifically: synthesize emits `proposal_id, proposal_type, source_id, target_id, subject_id, relation, evidence_bundle, reason_summary, trust_status, producer, lifecycle_state, schema_gate_result, input_fingerprint` — but graph's tuple puts `schema_gate_result` between `trust_status` and `producer`, and `input_fingerprint` between `producer` and `lifecycle_state`. The spec §3.3 JSON sample (spec:404-417) matches graph's order, not synthesize's. Tests only assert key *presence* (SYN-12, SYN-CLI-18), so this passes, but cross-consumer JSON diffs will look noisier than necessary. Recommend reordering `SynthesisFinding` fields to match graph's tuple (schema_gate_result → after trust_status; input_fingerprint → after producer), or document that ordering isn't load-bearing (consumers tolerate-unknown per CB-3). Not a spec-contract violation because E6 is presence-based.

**SF-peer-2 — `_render_synthesis_stdout` docstring stripped vs spec (maintainability).**
Spec §3.5 shows a meaningful docstring ("Compact human-readable summary: one line per finding + header + footer. Always prints the excluded-flagged-count line..."). The implementation at folio/synthesize.py:105 has no docstring. Similarly `synthesize()` at folio/synthesize.py:55 and `render_envelope()` at folio/synthesize.py:94 have no docstrings. The spec §3.1 `synthesize` docstring (spec:323-366) and §3.3 `render_envelope` docstring (spec:370-379) explicitly define derivation semantics — losing them in code removes onboarding signal. Review item #12 in the instructions calls docstrings out explicitly. Module docstring is present and excellent (folio/synthesize.py:1-13); function/dataclass docstrings absent.

**SF-peer-3 — SYN-20 narrowing comment is correct but masks a latent footgun (maintainability).**
folio/synthesize.py (no comment in module) / tests/test_synthesize.py:373-398 narrows SYN-20 from "no `folio.llm`, openai, anthropic in sys.modules" to "only openai / anthropic". The inline comment at test_synthesize.py:374-378 justifies this well: `folio.config → LLMConfig → folio.llm` is a transitive import, not a call. That narrowing is defensible. However: the spec §9.1 SYN-20 row (spec:696) explicitly lists `folio.llm` in the guard set. The narrowing is a test-vs-spec divergence — acceptable because the concrete concern (provider-SDK loading = invocation path) is preserved, but the SYN-20 row should be surgically amended in the spec or a one-liner added to the D.4 fix summary. Not blocking (reviewer consent is captured in the narrowing comment); flagging for audit-trail cleanliness.

**SF-peer-4 — CLI imports inside function are correct per graph pattern but violate PEP 8 module-level convention (idiom).**
folio/cli.py:1978-1982 imports `_render_synthesis_stdout, render_envelope, synthesize` from `.synthesize` *inside* the CLI function body. Spec §3.4 (spec:504) shows this exact pattern ("from .synthesize import synthesize as run_synthesize, render_envelope"), and it matches `folio/cli.py:2006` (graph import) and other per-command imports — it's clearly the project convention for lazy CLI loading. However, the new greenfield module would benefit from a brief comment like `# Lazy import to keep CLI startup fast (matches graph pattern at cli.py:2006)` for future-author context. Not a bug; a minor idiom polish.
