## Verdict

Needs Fixes

## Rationale

The spec has one critical omission: `folio/provenance.py` is entirely absent from scope despite containing a parallel `ProvenanceProposal` dataclass with its own `status` field, plus 11 raw-dict reads and 1 raw-dict write of `"status"` against proposal dictionaries. This is a blocker because:

1. **Silent data divergence.** Once `enrich_data.py` emits `lifecycle_state`, provenance code that reads `raw.get("status")` on those same proposals will get `None` instead of the actual state. Rejection-memory filters (lines 521, 1009, 1239), pending-proposal collectors (line 271, 1413, 1535), and rejection writers (line 1692) will all silently malfunction.

2. **Schema inconsistency.** The provenance subsystem has its own `ProvenanceProposal` dataclass with the same `status` / `pending_human_confirmation` vocabulary. Leaving it unrenamed while renaming `RelationshipProposal` creates two incompatible proposal schemas within the same codebase.

The spec must either (a) expand scope to include `provenance.py`, or (b) explicitly defer `provenance.py` migration to a named follow-up slice with a justification for why the two proposal types can safely diverge in the interim. Option (b) requires verifying that `RelationshipProposal` and `ProvenanceProposal` raw dicts never flow through the same code paths (they appear to use separate frontmatter namespaces, so this may be viable, but the spec must state this explicitly).

The should-fix items (A-3 through A-6) are implementation-quality issues that could introduce subtle bugs but are individually addressable during code review. The minor items (A-7 through A-9) are documentation accuracy concerns.
