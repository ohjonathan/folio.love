## Rendering Contract

### 1. Mermaid: `graph_to_mermaid()`

Required signature:

```python
def graph_to_mermaid(graph: DiagramGraph) -> str:
    ...
```

#### General rules

- Deterministic: same `DiagramGraph` -> byte-identical Mermaid
- Default direction: `graph TD`
- Empty graph: return `""` or a minimal valid Mermaid string, but do not crash
- Use stable ordering everywhere:
  - groups: deterministic order, recommended alphabetical by `name` then `id`
  - nodes within groups: deterministic order, recommended alphabetical by label then id
  - ungrouped nodes: deterministic order after grouped nodes
  - edges: deterministic order, recommended by source label then target label then edge id

#### Mermaid node shape mapping

Map the **live runtime kinds** first:

- `service` -> rectangle `[label]`
- `database` -> cylinder `[(label)]`
- `queue` -> parallelogram `[/label/]`
- `user` -> circle `((label))`
- `external` -> stadium or rectangle fallback; choose one and keep it deterministic
- `process` -> rectangle or rounded box; choose one and keep it deterministic
- `decision` -> diamond `{label}`
- `start` -> circle `((label))`
- `end` -> double circle fallback if Mermaid-safe, otherwise circle
- `note` -> bracketed note-style fallback
- `container` -> rectangle unless represented as group context
- `other` -> rectangle

Also accept proposal-era aliases defensively:

- `datastore` -> same as `database`
- `actor` -> same as `user`
- `boundary` -> prefer rendering as group context when group exists, otherwise safe rectangle fallback
- `unknown` -> rectangle

Do not assume `boundary` nodes are common in the live graph today.

#### Mermaid edge mapping

Map current runtime directions:

- `forward` -> `-->`
- `reverse` -> reverse the endpoints and render `-->`, or use the Mermaid reverse arrow form if you choose it consistently
- `bidirectional` -> `<-->`
- `none` -> `---`

Also accept legacy values:

- `->` -> `-->`
- `<->` -> `<-->`
- `undirected` -> `---`
- `unknown` -> `---`

#### Edge labels

- labeled edge -> `-->|label|` or equivalent arrow-specific labeled form
- unlabeled edge -> plain arrow

#### Technology labels

If a node has `technology`, render it on a second line:

- `Web Application<br/>React`

Keep this deterministic and sanitized.

#### Groups and nesting

Implement recursive `subgraph` rendering from `DiagramGroup.contains_groups`.

Required safety rules:

- detect cycles via visited-group tracking
- enforce a maximum nesting depth, recommended `5`
- if the depth limit is exceeded or a cycle is detected, flatten remaining children into the current level and record an uncertainty
- do not recurse forever

#### Sanitization (critical)

Sanitize all labels before emitting Mermaid.

You must handle at least:

- parentheses: `(` `)`
- brackets: `[` `]`
- braces when used in labels
- double quotes: `"`
- single quotes only if your chosen Mermaid form requires it
- pipes: `|`
- semicolons: `;`
- backticks if you use quoted node labels
- reserved words that can break Mermaid statements:
  - `end`
  - `subgraph`
  - `graph`
  - `direction`

Also:

- preserve Unicode when Mermaid accepts it
- if label becomes empty after sanitization, fall back to node id

If a specific node or edge cannot be rendered safely:

- omit only that element from Mermaid
- append a concise string to `DiagramAnalysis.uncertainties`
- set `review_required=True`
- add a short render-time note/question if helpful
- do **not** crash the whole Mermaid block

Keep the component/connection tables complete even when Mermaid omits an unsafe element.

### 2. Prose: `graph_to_prose()`

Required signature:

```python
def graph_to_prose(graph: DiagramGraph) -> str:
    ...
```

Rules:

- strictly graph-bound
- no semantic inference
- no architecture interpretation
- if it is not in nodes, edges, or groups, it does not belong in the prose

Recommended structure:

- one sentence per edge:
  - `"Web Application connects to API Gateway via HTTPS/REST."`
- optional group summary sentence:
  - `"Components are organized into VPC and Data Platform groups."`
- if no edges:
  - `"<N> components identified: ..."`

Technology may be included only as explicit graph data:

- `"Order Database (PostgreSQL) connects to API Gateway."`

Do not generate claims like:

- purpose
- ownership
- security role
- performance role
- inferred semantics

    ...

Target Markdown columns:

Rules:

Rules:

- deterministic ordering by source label, then target label, then edge id
- human-readable direction symbols:
  - forward -> `->` or `->`-style human string
  - bidirectional -> `<->`
  - none / unknown -> `--`
- confidence rendered to 2 decimal places
- empty graph -> empty table or deterministic "No connections identified." message

### 5. Entity resolution: `resolve_entity()`

Required signature:

```python
def resolve_entity(technology: str | None) -> str:
    ...
```

Rules:

- technology labels only

Use a module-level constant that is easy to extend.

Decide and test case handling explicitly. Recommended:

- case-insensitive lookup
- canonical output casing in the wiki-link, e.g. `postgresql` -> `[[PostgreSQL]]`
