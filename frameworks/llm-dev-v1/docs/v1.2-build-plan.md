# v1.2.0 Build Plan

Author: Jonathan Oh. Drafted 2026-04-16 following the v1.1.1 patch release
(PR #51, merge `ab4bc27`) and in parallel with `folio.love`'s Slice 5/6
Tier-4 work.

## Context

Five slices of production adoption in `folio.love` produced 46 friction
entries (`F-001`..`F-046`) and 25 concrete recommendations consolidated at
`frameworks/manifests/framework-learnings-v1.1-adoption.md`. v1.1.1 shipped
the bug-fix subset (C-1, C-2, C-4, C-5, B-5). v1.2 ships the
framework-coherence improvements that require template, playbook, schema,
or tooling additions.

**Freeze timing.** v1.2 scope is *drafted* at this build plan's status
`not-started`, but the scope itself is not frozen until `folio.love`
completes its next full D-phase retrospective. Most framework friction
historically surfaces in B/D review-board rounds, so locking v1.2 before a
full D-phase runs against v1.2's should-ship changes risks re-opening the
scope mid-release. If `folio.love` Slices 5/6 surface new categories after
freeze, those slip to v1.3 — v1.2 does not stay perpetually open.

**Target adopter.** `folio.love` remains the first-production adopter.
This document is also scoped for `johnny-os` D4 (the next framework
deliverable originating outside folio.love), which benefits from v1.1.1's
bug fixes already shipped and will benefit from v1.2 once it lands.

## Decisions taken before this plan was written

1. **v1.1.1 shipped separately**, not folded into v1.2. Bug-vs-feature
   distinction preserved in the CHANGELOG; reduces v1.2 scope by 5 items.
2. **A-3 = tighten Template 16**, not a new Template 16b. Evidence
   (F-013) is about *same-variant multi-reviewer disagreement* on a
   single Pre-A proposal. Cross-variant consolidation (Triage +
   Proposal running on one deliverable) has no friction evidence and
   is deferred to v1.3 or later.
3. **B-3 = must-ship, not should-ship.** D.6 is mechanically
   checkable (file existence, test pass, grep empty) — every other
   gate is already a script. D.6 being prose-only was a structural gap,
   not a nice-to-have. Removing the reviewer dispatch from D.6 simplifies
   Phase D architecture.
4. **C-13 = should-ship, not must-ship.** One data point (Slice 4's
   alignment/product overlap on §11 rule 5) isn't enough to call it
   structural. Promote to must if Slices 5/6 repeat it; downgrade or
   close as a Slice-4 artifact if not.

## Versioning rules inherited from ROADMAP.md

- Minor bump stays in `frameworks/llm-dev-v1/` (no folder move).
- v1.2 must remain backward-compatible with stamped v1.0.0 and v1.1.0
  manifests. `manifest_version` comparison: `"1.0.0"` and `"1.1.0"`

  extracted project).
- `verify-all.sh` gains new checks but the existing 8 remain passing.

## Phase 1 — Must-ship doctrine + policy

**Objective.** Land the two policy/spec changes first so downstream
tooling encodes the new rules.

Files: `framework.md`, `templates/04-review-board-adversarial.md`,
`templates/05-review-board-adversarial.md` (if Template 04 is the peer
and 05 is adversarial; verify exact numbering when editing),
`manifest/deliverable-manifest.schema.yaml`,
`templates/16-proposal-review.md`.

### A-1 — Codex adversarial as default (F-006, F-006-confirmed)

- `framework.md § P3`: add **adversarial-family invariant**. The
  family assigned to the adversarial role at B.1, D.2, and any
  cross-provider review round MUST NOT be the same provider as the
  artifact author. Same-provider adversarial is documented as
  "advisory only" and requires a cross-provider second pass before D.6
  acceptance. Rationale: training-data diversity is empirically a
  stronger defect-catching signal than within-family lensing
  (F-006-confirmed).
- `templates/<adversarial-template>.md`: role-header line adds
  "adversarial family MUST be a different provider from the author
  family". Halt condition: adversarial role dispatched to same
  provider as author → halt and re-dispatch.
- `manifest/deliverable-manifest.schema.yaml`: extend
  `model_assignments` entry validation. For assignments with
  `role: adversarial` on phase `B.1` or `D.2`, the family must differ
  from the phase's author family. Enforce via custom assertion in
  `verify-p3.sh` (the schema alone cannot cross-reference author).
- `scripts/verify-p3.sh`: add adversarial-family check.

**Acceptance.** `verify-p3.sh` fails on a manifest that assigns
adversarial to the same provider family as author on B.1/D.2.
Same-provider adversarial remains legal when paired with a
cross-provider second pass documented in `gate_prerequisites`.

### A-3 — Pre-A tightening (F-013)

- `templates/16-proposal-review.md`: add **"P5-style consolidation
  rules"** section applicable when two reviewers produce divergent
  verdicts (e.g., Proceed vs Revise). Steps:
  1. Extract blocker deltas — enumerate each reviewer's blockers side
     by side.
  2. Apply evidence-class priority: `BUG > STRUCTURAL > COSMETIC`.
     Higher-priority class wins when a blocker conflicts with a
     lower-priority accept.
  3. Produce a consolidated verdict with visible reasoning (which
     reviewer contributed what, how the priority was applied).
- Template 16 remains single-variant (Proposal review only). No new
  Template 16b; cross-variant consolidation is not in scope.

**Acceptance.** When Template 16 runs with two reviewers producing
divergent verdicts, the template's consolidation section produces a
single Proceed / Revise / Drop verdict traceable to the priority
rules. Existing single-reviewer runs are unchanged.

Exit: A-1 and A-3 land; `verify-all.sh` 8/8 green; the two new
assertions in `verify-p3.sh` are added; v1.0.0 and v1.1.0 manifests
still validate.

## Phase 2 — Must-ship tooling

**Objective.** Ship the two net-new scripts that mechanize existing
prose-only gates.

New files: `scripts/verify-adopter.sh`, `scripts/verify-d6-gate.sh`.

### A-2 — `verify-adopter.sh <manifest-path>` (F-007, F-010)

Unified entrypoint for adopter-supplied manifests.

- Runs (in order): `verify-schema.sh --manifest <path>`,
  `verify-p3.sh --manifest <path>`, `verify-gate-categories.sh --manifest <path>`,
  `verify-artifact-paths.sh --manifest <path>`.
- Each `verify-*.sh` gains a `--manifest <path>` flag as a secondary
  interface. Default behavior unchanged (validates the two example
  manifests).
- `verify-adopter.sh` reports per-check PASS/FAIL and aggregate.
- Document in `README.md` adopter-onboarding section.

**Acceptance.** Running
`bash scripts/verify-adopter.sh frameworks/manifests/my-deliverable.yaml`
(on a folio.love-style adopter manifest) produces a PASS result. The
existing `bash scripts/verify-all.sh` still passes 8/8 — the adopter
entry point is additive.

### B-3 — `verify-d6-gate.sh <final-approval-path>` (F-030)

Mechanical D.6 gate. Removes the reviewer dispatch that v1.1 requires
for final approval.

- Input: path to the final-approval artifact emitted by Template
  `07-final-approval-gate.md`.
- Parses the artifact's gate-prerequisite rows (14 standard rows per
  Template 07's schema).
- Asserts every row is `PASSED` with an evidence-class tag
  (`direct-run`, `file-exists`, `grep-empty`, `test-pass`, etc.).
- Fails with row-level detail on any row missing PASSED / evidence.
- `playbook.md § D.6`: update to describe the script as the primary
  gate; reviewer dispatch is deprecated (still allowed for edge
  cases, but the default D.6 flow is script-driven).
- `templates/07-final-approval-gate.md`: update output schema to
  emit machine-readable evidence-class tags inline with each row so
  the script can parse without heuristics.

**Acceptance.** On folio.love's v0.6.4 final-approval artifact (or
equivalent synthetic fixture), `verify-d6-gate.sh` produces the same
verdict as the D.6 reviewer-based gate would. `verify-all.sh` does
NOT invoke verify-d6-gate by default (the script is per-deliverable,
not per-bundle) but adds a regression test that it runs correctly
on a known-good fixture stored in `examples/`.

Exit: Both scripts land; `verify-all.sh` still 8/8; adopter onboarding
has a single entry point for manifest validation; D.6 gate is
mechanically executable.

## Phase 3 — Should-ship

**Objective.** Ship five recommendations that are high-leverage but
not structurally required for v1.2's coherence claim.

### A-4 — Adoption doc v2 (F-001, F-002, F-003, F-008, F-018, F-019)

- `README.md` adoption section: single `<ADOPTER_REPO_ROOT>` variable
  replaces the inline mixture of paths. Per-language appendix
  (Python / TS / Go) for the `scope.allowed_paths` and
  `scope.forbidden_paths` conventions. Concrete `day-one.sh`
  bootstrap script that adopters run to create their first
  manifest + scope-lock skeleton.
- `tokens.md` gains `<MANIFEST_DIR>` (new required token) with the
  convention `frameworks/manifests/` for monorepo adopters,
  `manifests/` for standalone.

**Acceptance.** A new adopter can read the README → run
`day-one.sh` → get a working manifest + scope-lock skeleton without
any non-documented guesses.

### B-2 — Orchestrator consolidation on unanimous verdicts (F-023, F-028, F-029)

- `playbook.md § B.3 / § D.3`: add **"Orchestrator consolidation
  fast-path"**. When all lenses return the same verdict (all-Approve
  or all-Needs-Fixes), the orchestrator may author the canonical
  verdict directly without a separate meta-consolidator family
  dispatch. Reserved for split verdicts only.
- `templates/06-meta-consolidator.md`: note the fast-path at the top;
  clarify that the meta-consolidator template runs only on split
  verdicts.

**Acceptance.** On a deliverable where all 3 lenses return Approve,
the orchestrator can finalize B.3 / D.3 without dispatching a 4th
family. When verdicts split, the existing meta-consolidator path
remains mandatory.

### C-9 — Codex adversarial prompt template (F-042 mitigated, F-044)

- `templates/<adversarial-template>.md`: codify the Slice-4-mitigated
  **targeted-prompt pattern**. Suggested structure:

  ```
  You are the adversarial reviewer for <DELIVERABLE_ID>. Find what breaks.
  Read (sparingly): <FILE_LIST>.
  Target failure modes:
  1. <MODE_1>
  2. <MODE_2>
  ...
  Classify: Blockers, Should-fix, Informational.
  ```

- Mark F-042 as framework-mitigated (the adopter's workaround is now
  documented as the framework's prescribed pattern).

**Acceptance.** An adversarial dispatch using the templated prompt
produces a structured review (not an exploration transcript) on a
representative artifact.

### C-12 — Contract-enumeration checklist + model-assignments preflight (Slice 4 DC-1, F-045)

Two template additions.

- `templates/12-spec-author.md` gains a "Contract enumeration
  checklist" section: author extracts every §-level enum table from
  the spec and asserts an implementation anchor exists for each item
  (function, CLI flag, class field, test). Produces a row-level
  cross-reference the D.2 adversarial reviewer can audit against.
- `templates/<phase-launch-templates>.md` (B.1, D.2): add preflight
  command `yq '.model_assignments[] | select(.phase == "<PHASE>") ' manifest.yaml`
  and an explicit "Confirm all assignments dispatched. Proceed?"
  checkpoint. Prevents the Slice 4 dispatch-hygiene miss (F-045).

**Acceptance.** Spec authors land a contract enumeration section;
phase launches include the preflight check; missed-reviewer-family
dispatches become a noisy failure mode instead of a silent miss.

### C-13 — B.1 role-overlap reduction (Slice 4 B.1 overlap, F-046)

- `templates/03-review-board-peer.md`,
  `templates/04-review-board-alignment.md`,
  `templates/19-review-board-product.md`: tighten role boundaries.
  - Alignment = parent-spec fidelity only. No surface enumeration.
  - Product = operator experience only. No scope compliance.
  - Peer = code/spec correctness only. Boundaries documented.
- Promote to must-ship if Slices 5/6 surface repeat overlap.
  Downgrade to v1.3 or close as Slice-4 artifact if not.

**Acceptance.** A B.1 board on a user-facing deliverable produces
findings where alignment and product lenses do not re-report the
same issue from different framings.

Exit: Phase 3 items land as additive, backward-compat changes. No
existing tests or templates break.

## Phase 4 — Conformance + release

**Objective.** Finalize the conformance suite, extend backward-compat
guards, ship v1.2.

- `scripts/verify-all.sh`: invoke the new assertions
  (`verify-p3.sh` adversarial-family check). Does NOT invoke
  `verify-adopter.sh` (per-deliverable) or `verify-d6-gate.sh`
  (per-deliverable) by default, but the bundle conformance suite
  confirms both scripts run correctly against the shipped example
  manifests and fixtures.
- `CHANGELOG.md`: add `## v1.2.0 — Released YYYY-MM-DD (PR #N, merge <SHA>)`
  with must-ship + should-ship rows. Format matches v1.1.0 + v1.1.1 entries.
- `ROADMAP.md`: move v1.2 from `## v1.2.0 — Planned` to
  `## v1.2.0 — Released`.
- Full 3-family review-board round (not the 2-family light review
  used for v1.1.1). Product lens runs where `<USER_FACING>` is true
  for any example manifest touched by v1.2 changes.

**Acceptance.** Canonical 3-family review-board verdict emits
**Approve**; conformance suite 8/8 green with the new assertions in
place; backward-compat regression guards intact (v1.0.0 +
v1.1.0 + v1.1.1 manifests all validate).
