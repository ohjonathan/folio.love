# v1.2.0 Build Plan

Author: Jonathan Oh. Drafted 2026-04-16.

## Context

Two independent v1.2 scope proposals produced by parallel adopters:

- **johnny-os** (this repo) — 14 findings from the D3 Manifest Spec run
  (`docs/v1.2-scope-proposal.md`). Single deliverable, one halt event.
- **folio.love** — 25 recommendations from 46 friction entries across 5
  adoption slices (`folio.love` @`11312e75`:
  `frameworks/llm-dev-v1/docs/v1.2-build-plan.md`).

**Divergence to resolve.** folio.love shipped a local `v1.1.1` release
(PR #51, merge `ab4bc27`) containing 5 bug fixes (folio recommendations
C-1, C-2, C-4, C-5, B-5). johnny-os's canonical bundle does NOT have
those commits. Phase 0 of this plan cherry-picks folio's v1.1.1 back
into johnny-os as the canonical v1.1.1 release, restoring a single
authoritative source.

**User decisions taken before this plan was written:**

1. **Scope:** ship v1.1.1 first (Phase 0), then v1.2 (Phases 1–5).
2. **v1.1.1 reconciliation:** cherry-pick folio's v1.1.1 into johnny-os
   as-is. folio is the faster-moving adopter; patches flow upstream.
3. **Priority authority for v1.2:** evidence-weight merge. Cross-run
   findings (present in both retros) get must-ship. Single-retro
   findings go should-ship unless coherence-critical.
4. **Review boards:**
   - v1.1.1 → focused 2-family review (Claude + Codex) per patch-release
     pattern.
   - v1.2 → full 3-family review board (Claude + Codex + Gemini) with
     Product lens where `<USER_FACING>` applies.
5. **Licensing:** Apache 2.0 (unchanged from v1.1).

## Versioning rules inherited from ROADMAP.md

- Minor bump stays in `frameworks/llm-dev-v1/` (no folder move).
- v1.2 must remain backward-compatible with stamped v1.0.0, v1.1.0, and
  v1.1.1 manifests. Schema additions permitted; removals/renames not.
- No git tag in this repo (tags reserved for the eventual extracted
  project).
- `verify-all.sh` gains new checks; the existing 8 must remain passing.

## Phase 0 — v1.1.1 reconciliation (ships independently before v1.2)

**Objective.** Cherry-pick folio.love's v1.1.1 commits into johnny-os
so the canonical bundle and folio's local bundle are byte-equivalent
for the v1.1.1 scope. Ships as its own PR + release.

### Process

1. Open branch `frameworks/llm-dev-v1.1.1` off `main`.
2. Fetch folio.love's v1.1.1 diff at merge commit `ab4bc27` (PR #51):
   ```bash
   gh api "repos/ohjonathan/folio.love/commits/ab4bc27" \
     -H "Accept: application/vnd.github.diff" > /tmp/folio-v1.1.1.patch
   ```
3. Inspect the patch. Apply only hunks that touch
   `frameworks/llm-dev-v1/` in folio (mirror them into this repo's
   `frameworks/llm-dev-v1/`). If folio's v1.1.1 touched adopter-local
   files (e.g., `frameworks/manifests/`), those do NOT come back —
   folio's fork keeps them.
4. Commit per-scope-item atomically (one commit per folio
   recommendation ID: C-1, C-2, C-4, C-5, B-5). Each commit message
   cites `folio.love#51 ab4bc27` as the source.
5. Run `bash scripts/verify-all.sh` after each commit — must remain
   8/8 green.
6. Author `CHANGELOG.md` entry:
   ```
   ## v1.1.1 — Released YYYY-MM-DD (PR #N, merge <SHA>)
   Cherry-picked from folio.love PR #51 (merge ab4bc27).
   - C-1: <title>
   - C-2: <title>
   - C-4: <title>
   - C-5: <title>
   - B-5: <title>
   ```
7. `ROADMAP.md`: add a `## v1.1.1 — Released` section under v1.1.0
   with scope summary and the cherry-pick provenance.
8. Update `PROVENANCE.md` non-normative table with `v1.1.1` entries.
9. Dispatch **2-family focused review board** (Claude + Codex). Gemini
   runs as reviewer-of-reviewer (audits the meta-consolidator output).
   Output at
   `frameworks/llm-dev-v1/review-board/v1.1.1-spec-verdict.md`.
10. On Approve, merge the PR (`--no-ff`), capture merge SHA, update
    CHANGELOG + ROADMAP with the real SHA, archive session via
    `ontos log -e "v1-1-1-release"`.

### Exit

- johnny-os bundle byte-equivalent to folio's v1.1.1 for the 5 scope
  items.
- v1.1.1 merged and stamped in CHANGELOG + ROADMAP.
- `docs/handoffs/2026-04-15_folio-love-llm-dev-adoption.md` updated
  to point folio at the johnny-os v1.1.1 merge SHA so folio can resync
  on the next bundle pull.
- **v1.2 work begins only after v1.1.1 merges.** Phases 1–5 branch off
  `main` post-v1.1.1.

### 1.1 — Adversarial family MUST differ from author family (folio A-1)

- **folio evidence.** F-006, F-006-confirmed. `must-ship`.
- **johnny-os evidence.** D3's B.3-r3 Gemini-adversarial artifacts all
  required post-hoc P5 downgrade (5× total) when sandboxed; same
  provider as author plus sandbox together produced advisory-only
  quality. Cross-run signal: adversarial quality degrades when
  same-provider.
- **Fix.**
  - `framework.md § P3`: add **adversarial-family invariant.** The
    family assigned `adversarial` at B.1, D.2, or any cross-provider
    review round MUST NOT be the same provider as the author family.
    Same-provider adversarial is `advisory-only`; a cross-provider
    second pass is required before D.6.
  - `templates/05-review-board-adversarial.md`: role-header gains the
    "adversarial family MUST differ from author" line + halt condition
    on same-provider dispatch.
  - `manifest/deliverable-manifest.schema.yaml`: extend
    `model_assignments` validation. Cross-reference author family per
    phase (schema alone can't express this; assertion goes in
    `verify-p3.sh`).
  - `scripts/verify-p3.sh`: add adversarial-family check.

**Acceptance.** `verify-p3.sh` fails on a manifest assigning
adversarial to the author's provider on B.1/D.2. Same-provider
adversarial remains legal when `gate_prerequisites` documents the
cross-provider second pass.

### 1.2 — `scripts/verify-d6-gate.sh` (folio B-3 + johnny-os #8)

- **folio evidence.** F-030. `must-ship`.
- **johnny-os evidence.** #8 in scope-proposal, ranked should-ship rank
  5. Both retros identify D.6 as prose-only; coherence argument wins:
  every other gate (`verify-p3`, `verify-schema`, `verify-artifact-paths`)
  is mechanical, so D.6 being prose-only is a structural gap, not a
  nice-to-have.
- **Fix.**
  - New `scripts/verify-d6-gate.sh <final-approval-path>`. Parses the
    14-row gate table from Template 07's output. Asserts every row is
    `PASSED` with an evidence-class tag (`direct-run`, `file-exists`,
    `grep-empty`, `test-pass`, etc.). Fails with row-level detail.
  - `templates/07-final-approval-gate.md`: update output schema to emit
    machine-readable evidence-class tags inline per row.
  - `playbook.md § D.6`: script is the primary gate; reviewer dispatch
    deprecated (still allowed for edge cases, default flow is
    script-driven).
  - `scripts/verify-all.sh` does NOT invoke `verify-d6-gate.sh` per
    bundle run (per-deliverable), but adds a regression test that it
    runs correctly against a known-good fixture in `examples/`.

**Acceptance.** On a known-good final-approval fixture,
`verify-d6-gate.sh` produces the same verdict as the v1.1 reviewer-based
gate. Row-level failure diagnostics when a row lacks PASSED + evidence.

### 1.3 — Circuit-breaker preserved-blocker-ID carry-forward (johnny-os #4 v1.2 slice)

- **johnny-os evidence.** Scope-proposal §3 classification:
  framework-too-strict. The one halt this run (`048f4bd`). Without the
  override, a converging review that caught 3 production-grade crash
  bugs would have been killed.
- **folio evidence.** folio parks severity-aware CB as `C-7` deferred.
  This is different scope (severity vs. ID carry-forward); johnny-os's
  signal is not contradicted.
- **Coherence-critical override.** The halt's cause — new blockers in
  an improved artifact vs. same blockers cycling — is a v1.1 template
  bug that repeats across future runs without the fix. Must-ship.
- **Fix.**
  - Prose slice (already in v1.1.1 per johnny-os scope §4.1) — Template
    06 gains escalation-record format and new-vs-recurring orchestrator
    judgment rule. **NOTE:** If this wasn't included in folio's v1.1.1
    cherry-pick, add it as a v1.2 Phase 1 prose change to Template 06.
  - Mechanical slice (v1.2): schema adds
    `review_rounds[].preserved_blocker_ids`. Template 06 emits it in
    each consolidator run. Breaker fires only on ID overlap across
    rounds (same IDs cycling = halt; new IDs found = not a halt). New
    `scripts/verify-circuit-breaker.sh` asserts the field is present
    on every round record and that breaker logic matches the schema
    rule.
  - Example manifest updated to demonstrate the field.

**Acceptance.** Synthetic fixture with 2 rounds, same blocker IDs:
breaker fires. Synthetic fixture with 2 rounds, different blocker IDs:
breaker does not fire. Verify-circuit-breaker catches a missing
`preserved_blocker_ids` field.

### 2.1 — `scripts/verify-adopter.sh <manifest-path>` (folio A-2)

- **folio evidence.** F-007, F-010. `must-ship`.
- **johnny-os evidence.** Silent — not surfaced because johnny-os is
  first-party (runs `verify-all.sh` directly). Adopters need a unified
  entrypoint to validate their own manifest without dragging in the
  bundle's own example validations.
- **Fix.**
  - New `scripts/verify-adopter.sh <manifest-path>`: unified entrypoint.
    Runs `verify-schema.sh --manifest <path>`, `verify-p3.sh
    --manifest <path>`, `verify-gate-categories.sh --manifest <path>`,
    `verify-artifact-paths.sh --manifest <path>`. Reports per-check
    PASS/FAIL and aggregate.
  - Each `verify-*.sh` gains a `--manifest <path>` flag (secondary
    interface). Default behavior unchanged (validates the two example
    manifests).
  - `README.md` adopter-onboarding section documents the new entry
    point.

**Acceptance.**
`bash scripts/verify-adopter.sh frameworks/manifests/my-deliverable.yaml`
on a folio.love-style adopter manifest produces PASS. `verify-all.sh`
still 8/8 green (adopter entry is additive).

### 2.2 — Template 16 P5-style consolidation rules (folio A-3)

- **folio evidence.** F-013 (same-variant multi-reviewer disagreement
  on single Pre-A proposal). Per user's earlier decision: tighten
  Template 16, not new Template 16b.
- **johnny-os evidence.** D3 did not exercise this path (Pre-A.proposal
  had a single reviewer). No contradiction.
- **Fix.** `templates/16-proposal-review.md` gains a **P5-style
  consolidation rules** section for divergent-reviewer cases:
  1. Extract blocker deltas — enumerate each reviewer's blockers side
     by side.
  2. Apply evidence-class priority: `BUG > STRUCTURAL > COSMETIC`.
     Higher-priority wins when a blocker conflicts with a lower-priority
     accept.
  3. Produce consolidated verdict (Proceed / Revise / Split / Drop)
     with visible reasoning.
  Template stays single-variant; cross-variant consolidation deferred
  to v1.3.

**Acceptance.** Template 16 run with two divergent reviewers produces a
single verdict traceable to the priority rules. Single-reviewer runs
unchanged.

Exit for Phase 2: 2.1 + 2.2 land. `verify-all.sh` green. Adopter
entrypoint works end-to-end on folio-shaped manifests.
