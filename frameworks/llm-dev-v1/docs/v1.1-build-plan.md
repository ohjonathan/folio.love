# v1.1.0 Build Plan

Author: Jonathan Oh. Drafted 2026-04-15 at end of v1.0.0 release session.

## Context

v1.0.0 shipped on 2026-04-15 (PR #3 merged at `1a1d9e5`). The bundle stays
in johnny-os at `frameworks/llm-dev-v1/` until mature; extraction to its
own repo is deferred. v1.1 will be the first production use — the target
adopter is the `folio.love` project
(`https://github.com/ohjonathan/folio.love`), a user-facing product.

Decisions taken before this plan was written:

1. **Scope:** ship all four v1.1 items listed in `ROADMAP.md` —
   Proposal Review pre-phase, Product lens, Triage and Validation
   pre-phase variants, plus any patch-sized re-review blockers.
2. **Extraction:** NOT now. Bundle stays in johnny-os until feedback
   from folio.love matures it. Handoff to folio.love is a prompt-level
   artifact, not a repo split.
3. **Product lens depth:** heavy. New role, state-machine entry, meta-
   consolidator intake, manifest schema changes, P3 extension for
   user-facing phases.
4. **License:** Apache 2.0.
5. **Review board:** same 3-family (Claude + Codex + Gemini) with
   internal Peer/Alignment/Adversarial, meta-consolidated, same
   playbook as v1.0.0.

## Versioning rules inherited from ROADMAP.md

- Minor bump stays in the same folder (`frameworks/llm-dev-v1/`).
- v1.1 must remain backward-compatible with stamped v1.0.0 manifests
  (manifests that carry `manifest_version: "1.0.0"`; the `required`
  list is unchanged from v1.0.0 so pre-stamp drafts still need the
  field). Schema additions are permitted; schema removals/renames
  are not.
- No git tag is created in this repo (tags reserved for the eventual
  extracted project).

## Phase 1 — Doctrine + state machine (heavy Product lens)

**Objective.** Add Product as a first-class fourth lens; add pre-A
variants to the state machine.

Files: `framework.md`, `playbook.md`, `README.md`.

- `framework.md`: extend the Roles table with `product` role. Extend
  the state machine: pre-A variants `-A.proposal`, `-A.triage`,
  `-A.validation` feed into A or into a fast-patch exit.
- `framework.md § P3`: extend strict rule. For user-facing phases
  (B/D on deliverables flagged `user_facing: true`), the rule becomes
  **≥3 non-author engineering families + ≥1 Product reviewer** (which
  may be the same family as one of the engineering reviewers, but in a
  separate session per P10 single-role). Non-user-facing phases keep
  the v1.0.0 rule unchanged.
- `framework.md § Compatibility`: note that v1.1 is additive; v1.0.0
  manifests parse unchanged (new fields are optional with safe defaults).
- `playbook.md`: add a short section on when Product applies (trigger:
  artifact touches a user-facing surface — UI copy, UX flow, public
  API signature, pricing surface, marketing claim).
- `playbook.md`: add pre-A variants — when each is correct entry
  (Proposal = new direction, Triage = bug, Validation = assumption-test).
- `README.md`: update bundle table and known-limits section.

Exit: Product is documented as a first-class lens; pre-A variants
are documented; v1.0.0 manifests still parse.

## Phase 2 — Pre-A variants (three new templates)

**Objective.** Ship the three pre-A entry-point templates the playbook
supports.

New files:

- `templates/16-proposal-review.md` — review a direction / proposal doc
  before committing to a spec. Verdict: Proceed to A | Revise proposal
  | Drop. Sections: feasibility, scope sanity, reviewer diversity,
  unknowns worth resolving before spec.
- `templates/17-triage.md` — bug triage entry. Decides whether the
  report deserves a full spec (escalate to A) or a fast-patch
  (C-direct with regression test, skipping A/B). Includes reproduction
  and severity sections.
- `templates/18-validation.md` — validation / assumption-test entry.
  Ships a prototype, measures, emits a decision doc. Decision: escalate
  to A with the validated approach | drop the assumption | re-run with
  different design.

Updates:

- `templates/01-worker-session-contract.md`: add halt-catalog entries
  for `proposal-reviewer`, `triage-author`, `validation-author`.
- `README.md` bundle table: list templates 16–18.
- `tokens.md`: add tokens these templates need (e.g.,
  `<PROPOSAL_DOC_PATH>`, `<TRIAGE_REPORT_PATH>`, `<VALIDATION_BUDGET?>`).

Exit: three templates exist; frontmatter-token coverage clean; each
verdict has a clear exit into the main state machine.

## Phase 3 — Product lens template + meta-consolidator update

**Objective.** Ship the Product review template and update
meta-consolidation for 4-verdict intake on user-facing phases.

New files:

- `templates/19-review-board-product.md` — 4th review lens. Sections
  (first draft; refine with playbook grounding):
  - User-value assessment (does this solve a real user problem?)
  - UX-friction inventory (flows that add steps or cognitive load)
  - Copy review (clarity, tone, honesty — including error messages)
  - Accessibility surface (contrast, keyboard nav, screen-reader hints,
    localization hooks)
  - Failure-visibility (how the user learns something failed;
    recoverable vs. dead-end states)
  - Positive observations section (parallel to other three lenses)

Updates:

- `templates/06-meta-consolidator.md`: intake 4 verdicts when
  `user_facing: true`. Halt condition: missing Product verdict on a
  user-facing phase. Canonical verdict frontmatter gains a
  `product_verdict` field.
- `templates/05-review-board-adversarial.md`: no change (its existing
  posture already permits overlap with Product's failure-visibility
  section; overlap is tolerated by design).
- `framework.md § Artifact contracts`: document the product-verdict
  artifact path convention.

Exit: Product template ships; meta-consolidator correctly gates on
Product presence for user-facing phases.

## Phase 4 — Manifest + schema + generator-spec

**Objective.** Backward-compatible schema additions for Product lens
and pre-A variants.

Files: `manifest/deliverable-manifest.schema.yaml`,
`manifest/example-manifest.yaml`, `manifest/generator-spec.md`,
`examples/walkthrough-mini-deliverable.md`.

- **Schema.**
  - Add `user_facing: bool` at deliverable root (default `false`).
  - Add pre-A phase IDs to the phase enum: `-A.proposal`, `-A.triage`,
    `-A.validation`.
  - Add `product` to the role-label enum.
  - Extend the P3 invariant: when `user_facing: true`, require at
    least one `product` role in `model_assignments` for phases B and D.
  - New top-level `pre_a` optional block: `{ entry: proposal | triage |
    validation, artifact_path: <path> }`.
  - Add `product_verdict` to `artifacts.canonical_verdict` pattern
    options.
- **Example manifest.** Ship a user-facing example (generic shape —
  an info page, a CTA flow, or similar — NO folio-specific content).
  Demonstrates: `user_facing: true`, pre-A block present, Product
  reviewer in B/D `model_assignments`, 4-verdict meta-consolidation.
- **Generator-spec.** Add invariants 15–17:
  - 15. Product-presence: user-facing deliverables must have a
        product reviewer in B and D.
  - 16. Pre-A exit: if `pre_a` is present, its artifact is produced
        before Phase A is dispatched.
  - 17. Pre-A-variant coherence: Triage with spec-unnecessary verdict
        jumps to C-direct; Validation with drop-assumption verdict
        halts the deliverable.

Exit: schema validates the v1.0.0 example unchanged AND validates the
new user-facing example; all three generator invariants documented and
reflected in verify scripts (Phase 5).

## Phase 5 — Conformance suite + packaging

**Objective.** Extend mechanical verification and add license.

Files: `scripts/verify-p3.sh`, `scripts/verify-pre-a.sh` (new),
`scripts/verify-all.sh`, `LICENSE` (new), `CHANGELOG.md`, `ROADMAP.md`.

- `scripts/verify-p3.sh`: extend with user-facing branch — if
  `user_facing: true`, assert `product` role exists in B and D
  model_assignments.
- `scripts/verify-pre-a.sh` (new): when `pre_a` present, assert the
  artifact at `artifact_path` exists before Phase A entry in the
  example. Static check on the example; runtime check is the
  orchestrator's job.
- `scripts/verify-all.sh`: add the new script to the suite.
- `LICENSE` at bundle root: Apache 2.0 full text.
- `README.md`: add "License" section pointing at `LICENSE`.
- `CHANGELOG.md`: v1.1.0 entry summarizing the four scope items +
  licensing.
- `ROADMAP.md`: mark v1.1 shipped; move Triage/Validation out of
  "deferred"; confirm v1.2 remains open for retrospective-driven
  patch/minor items.

Exit: `bash scripts/verify-all.sh` exits 0; LICENSE present;
CHANGELOG and ROADMAP reflect v1.1 state.

## Phase 6 — folio.love adoption handoff

**Objective.** Produce a handoff prompt that folio.love's orchestrator
can read to bootstrap using v1.1.

New file (in johnny-os, NOT in the bundle — adoption guidance is not
part of the normative framework):

- `docs/handoffs/2026-xx-xx_folio-love-llm-dev-adoption.md`. Contents:
  - How to pull the bundle (git subtree import, submodule, or verbatim
    copy).
  - How to fill `tokens.md` for folio.love specifics
    (workspace, repo URL, CLIs, scope-lock paths typical for a Next.js
    /TypeScript project if that's the stack, doc-index tool choice).
  - Which pre-A entry to pick for the first deliverable (likely
    Validation for a new assumption-test, or Proposal for a direction
    doc).
  - Note that folio.love is the first production run — feedback should
    return as either a GitHub issue on johnny-os or a retro doc.
  - Pointer to `frameworks/llm-dev-v1/README.md` for the 10-minute
    adoption path.

Exit: handoff file exists; someone unfamiliar with johnny-os but
familiar with folio.love can follow it without needing a live
conversation.

## Phase 7 — 3-family review board for v1.1

**Objective.** Reproduce the v1.0.0 review board with v1.1 scope.

- Dispatch same three families (Claude + Codex + Gemini), each running
  internal Peer/Alignment/Adversarial, Codex extracts to a canonical
  verdict at `frameworks/llm-dev-v1/review-board/v1.1.0-spec-verdict.md`.
- Review scope:
  - **Doctrine consistency** (Phase 1): Product lens integration does
    not break v1.0.0 principles; strict P3 extension is unambiguous.
  - **Template fidelity** (Phases 2–3): new templates match the
    existing scaffolding standard (§9.3 port equivalents, halt
    conditions, frontmatter discipline).
  - **Schema correctness** (Phase 4): backward-compatibility verified
    against v1.0.0 example; new invariants are mechanically checkable.
  - **Conformance suite** (Phase 5): verify-all.sh passes.
  - **Handoff clarity** (Phase 6): a reader who has never touched
    johnny-os can follow the adoption prompt.
- Address blockers in a v1.1 `spec-verdict-response.md`, same format
  as v1.0.0.
- Stamp v1.1.0 in CHANGELOG after all blockers close. No git tag.
- Session archive at close.

Exit: v1.1.0 stamped; bundle ready for folio.love adoption.
