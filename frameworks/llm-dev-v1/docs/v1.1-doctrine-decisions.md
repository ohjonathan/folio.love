# v1.1.0 Doctrine Decisions

Author: Jonathan Oh. Captured 2026-04-15 at the start of the v1.1 execution
session.

## Context

`docs/v1.1-build-plan.md` scopes four additive items for v1.1 but leaves four
doctrine questions unresolved — each driven by a tension between the build
plan and the source playbook
(`docs/reference/llm-development-playbook.md`). Phase 1 of the execution
session (exploration) surfaced these tensions; the questions were resolved in
a pre-plan question round before any normative file was touched. This annex
records the resolutions so Phases 1-7 edit from settled ground.

The annex is non-normative. The normative expressions of each decision land in
`framework.md`, `playbook.md`, the new templates, the schema, and the
CHANGELOG during Phases 1-5. If a conflict emerges between this annex and a
normative file once v1.1.0 stamps, the normative file wins and this annex is
amended.

---

## 1. Template 18 = Validation / Observation Run (playbook §15.3 Validation/Observation Runs)

The build plan framed Template 18 as a prototype-measure-decide loop. The
source playbook has no such construct. Playbook §15.3 Validation/Observation Runs defines
"Validation / Observation Run" as a gate protocol over deployed code —
the worker runs the code, measures, and emits a structured report that feeds
Evidence-Based Patch (§15.2) or a new Phase A; the report is not a review
verdict.

**Decision.** Template 18 is redefined as `18-validation-run.md`, a
structural port of playbook §15.3 Validation/Observation Runs. The file name carries `-run` to prevent
confusion with Phase B spec validation. Verdict options:
`Run clean — proceed to A | Run inconclusive — re-run with revised plan |
Run exposed defect — escalate to hotfix or incident`. Frontmatter pattern
mirrors `09-incident-postmortem.md` (structured observation report), not the
meta-consolidator pattern. The divergence from Templates 16 and 17 (which
adjudicate proposals/backlog) is intentional and is documented in
`framework.md § Compatibility with source playbook` so a reader expecting
template symmetry in the pre-A layer is not surprised.

## 2. Template 16 verdict labels (playbook §13 alignment)

The playbook §13 ships the Proposal Review table headers but does not populate
the verdict cells. The build plan's draft ("Proceed to A | Revise proposal |
Drop") is closer to three Phase-B-style labels than to playbook §13.5's
Round-3 escalation paths.

**Decision.** Template 16 verdicts are:
`Proceed to Phase A | Revise and re-review | Split into multiple proposals |
Abandon direction`. Four labels, all mapped to playbook §13.3 + §13.5
constructs. "Timebox one final round" from §13.5 is not a verdict — it is
orchestrator instrumentation and lives in the halt-catalog guidance of
`01-worker-session-contract.md`, not in the Template 16 output schema.

## 3. P3 extension for Product lens on user-facing phases

The build plan extended P3 for `user_facing: true` deliverables to require
"≥3 non-author engineering families + ≥1 Product reviewer." The question was
whether the Product family must be distinct from the three engineering
families (raising the adoption floor to 5 families) or may overlap.

**Decision.** The adoption floor stays at 4 families (1 author + 3
engineering reviewers). When `user_facing: true`:
- Phase B.1 / B.2 / D.2 require ≥3 non-author engineering families (strict P3
  unchanged) AND a Product reviewer.
- The Product family MAY be the same as one of the three engineering families,
  provided the Product review runs in a separate worker session under P10
  (one role per session). The session-uniqueness guarantee preserves role
  fatigue protection even when the family overlaps.
- Proposal Review (Template 16) is excluded from this rule — it is pre-A and
  uses the 2-reviewer playbook §13.4 configuration, not the phase-advance
  review board.

Adopters who prefer a strictly-disjoint 5-family composition (1 author + 3
engineering + 1 Product, all distinct) are free to configure their manifest
that way; v1.1 does not mandate it. `verify-p3.sh` gets a branched check —
three distinct engineering families AND Product presence — rather than a
single-assertion "≥5 distinct families" check.

## 4. User-facing walkthrough placement

The build plan was ambiguous on whether `examples/walkthrough-mini-deliverable.md`
(the v1.0.0 currency-converter) should be retrofitted with `user_facing: false`
or supplemented with a second walkthrough.

**Decision.** Supplement, not replace or retrofit. The v1.0.0
currency-converter walkthrough stays byte-identical in v1.1 as the
backward-compatibility witness. A second file,
`examples/walkthrough-user-facing.md`, ships alongside it. It demonstrates
`user_facing: true`, a `pre_a` block, a Product reviewer in B.1/B.2/D.2
model assignments, and 4-verdict meta-consolidation. Content is generic
(settings-page or CTA-flow shape) with no folio.love-specific tokens — the
bundle stays portable. The `examples/` directory after v1.1 therefore
contains exactly two walkthroughs: v1.0.0 (currency-converter) and v1.1
(user-facing generic).
