# Adversarial Review — frameworks-llm-dev-v1 / v1.2.0 / claude

## BEGIN ADVERSARIAL REVIEW

Lens: failure. Core question: how does v1.2 fail, what bypasses the
new mechanics, what silently passes? v1.2 adds four new verify scripts
(`verify-d6-gate.sh`, `verify-circuit-breaker.sh`, `verify-adopter.sh`,
plus the `--probe-codex-model` flag on `verify-tokens.sh`), a new
day-one bootstrap, one cross-provider adversarial invariant in
`verify-p3.sh`, and 3.9's role-boundary tightening that re-routes
several categories of finding from Peer/Alignment/Product to the
mechanical gate. Every one of those surfaces is a new attack target.

## 3. Diagram / prose completeness attack

State-machine prose and pre-A diagram unchanged from v1.1. The new
role-boundary table in templates 03/04/19 introduces prose claims that
do not all have mechanical backing:

- **04 § "What is NOT your lens"** says scope compliance is handled by
  `verify-artifact-paths, verify-portability, D.6's gate table`. Of
  those three, only D.6 actually checks scope (and only through
  adopter-authored commands). The two `verify-*.sh` scripts named do
  different things. SHOULD-FIX wording, OR harden one of the named
  scripts to evaluate `scope.forbidden_paths` against a diff.
- **04 § "Surface enumeration audit"** promises Alignment audits § 11
  with "missing rows or missing anchors are blocker-grade findings".
  Enforcement is reviewer-discipline; no script cross-references.
- **05 § "Codex targeted-prompt pattern"** is purely dispatch-side
  prose. No attack surface.

## 4. Edge case inventory

Against all four new scripts, I ran the full documented + undocumented
edge matrix. Direct-run reproductions; clean exit codes verified (not
just stdout).

- **verify-p3 (adv-provider rule).** v1.0/1.1 grandfathered; v1.2 fails
  same-provider adversarial on B.1, B.2, D.2; escape hatch with matching
  `G-xprov-adv-<phase>` prefix passes; hollow escape-hatch (no artifact
  reference) passes (see ADV-SF1); per-phase partial escape scopes
  correctly; hyphen-less author family works. Multi-author manifests
  trigger the check against both authors (union of providers).
- **verify-d6-gate.** Permuted header column order → OK (columns are
  positional by name). Missing Evidence-class column → FAIL. `not-run` /
  `static-inspection` values → FAIL. Empty-data table → FAIL. Missing
  separator → FAIL. Lowercase `## gate table` → OK (by case-fold). No
  `## Gate table` heading → FAIL with clear diagnostic.
- **verify-circuit-breaker.** `review_rounds: []` or `null` → OK (no-op).
  Duplicate rounds in a phase → FAIL. `round: 0` / `-1` → FAIL.
  `preserved_blocker_ids: null` → FAIL. `preserved_blocker_ids: []` on
  single round → OK (zero preserved). Canonical fixtures: missing-field
  → FAIL per documented diagnostic; new-ids → CB quiescent;
  same-ids → CB fires.
- **verify-adopter.** Path with spaces → OK. Invalid manifest → sub-
  check identifies `verify-schema.sh`. Missing path → exit 2. Symlink
  → resolves and validates. `--manifest <path>` syntax → treated as
  positional (exit 2, "cannot read --manifest"). By-design; README
  prescribes positional.
- **verify-tokens --probe-codex-model.** Missing model → exit 1 usage.
  Codex not on PATH → exit 2. Did not run actual probe.
- **day-one.sh.** Invalid IDs: `INVALID-UPPERCASE`, `_`, space, slash,
  leading dash, leading dot, leading digit, trailing dash, empty → all
  FAIL with the schema-regex diagnostic. Valid IDs → bootstrap. Re-run
  on existing manifest → "not overwriting" + exit 1.

## 5. Security surface

No new security surface beyond v1.1. All YAML loads use `yaml.safe_load`.
`verify-tokens.sh --probe-codex-model` pipes literal `"ping"` to codex
stdin, no user input injection. `day-one.sh` uses `sed` with quoted
substitutions; no shell-injection vector on the deliverable-id (guarded
by the regex check first).

Token substitution remains the generator-v2 surface it was in v1.1. No
new exposure.

## 6. Blind-spot identification

**Claude's own blind spot.** Claude authored much of v1.2's 3.9
role-boundary tightening. The blind spot is coverage-asymmetry: the
prose promises a clean routing ("X belongs to the mechanical gate") but
the named mechanical gates do not all perform X. Specifically
`verify-artifact-paths` is a PLACEHOLDER-SHAPE check, not a scope-compliance
check; `verify-portability` scans the bundle, not the adopter
implementation. Claude wrote the templates from the orchestrator's
vantage (where D.6's gate-prerequisites commands are the catch net) and
didn't revisit the template-prose footnotes. A non-Claude reader
reading 03/04/19 in isolation would expect named scripts to do what
their names suggest.

**Author blind spot (general).** The same pattern recurs at 1.1: the
adversarial-provider invariant has a real mechanical check, but the
escape-hatch check is narrower than the prose implies. "Names a
cross-provider second-pass artifact" is aspirational.

## 7. Risk-assessment override

Author did not assert a risk level in the build-plan review-board
scope. I assign **medium** — same as v1.1.

Justification: every new script is structurally sound and direct-run
tests pass their documented negative-case matrices. The gap is prose-
vs-script asymmetry in two places (1.1 escape hatch, 3.9 mechanical-
gate redirect). Neither breaks the merge path; both narrow the claimed
invariants. folio.love already consumes v1.1; v1.2 will inherit one
previous-adopter and one new-adopter (the framework maintainer's
internal D3 slice). Risk transfer to a live project is unchanged from
v1.1.

Not high because verify-all.sh remains 10/10 green and all prose-only
invariants halt at orchestrator-review time rather than corrupting
merged artifacts (P8 safety intact).

## 9. Verdict

**Needs Fixes.** No blockers; three should-fix findings follow the
v1.1-review pattern (prose-vs-script asymmetry). ADV-SF1 and ADV-SF2
both stem from 3.9's role-boundary tightening: templates route
findings to scripts whose coverage is narrower than the prose claims.
ADV-SF3 removes Peer's redundant § 11 check without a mechanical
replacement.

v1.2's bundle is structurally clean: 10/10 conformance green; all four
new scripts handle every negative edge I constructed with correct
diagnostics; v1.0/v1.1 manifests validate unchanged; the walkthrough
byte-unchanged guard holds; the adversarial-provider invariant works
correctly on version-stamped twins. The three should-fix issues are
textual / narrow-scope strengthenings, not scope problems. Recommend
ship with SF1–SF3 as fix-forward.
