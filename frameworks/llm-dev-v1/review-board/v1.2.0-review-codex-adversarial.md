## 4. Role-boundary coverage attack

3.9 tightened Templates 03, 04, and 19 by moving raw scope compliance out
of reviewer lenses. The direction is clean if the mechanical gate is
coverage-equivalent. It is not yet coverage-equivalent.

Relevant static evidence:

- `templates/03-review-board-peer.md:50-53`: Peer treats scope-lock /
  forbidden-path violations as mechanical gates unless there is a
  design-quality issue.
- `templates/04-review-board-alignment.md:54-62`: Alignment does not
  perform `scope.allowed_paths` / `scope.forbidden_paths` /
  `forbidden_symbols` compliance except as parent-spec fidelity.
- `templates/19-review-board-product.md:74-82`: Product does not perform
  scope compliance.
- `scripts/verify-gate-categories.sh:30-40`: the script derives only the
  set of category names present.
- `scripts/verify-d6-gate.sh:132-167`: the D.6 parser checks `PASSED`
  plus allowed Evidence class, not whether the reproduction command
  matches the manifest's declared scope.

Direct-run reproduction:

```text
CASE verify-adopter hollow-scope
verify-adopter: PASSED (4/4 checks against /tmp/codex-v12-adv/hollow-scope.yaml)
RC=0

CASE verify-d6 hollow-scope
verify-d6-gate: OK (1 rows validated; all PASSED with allowed evidence class)
RC=0
```

The hollow manifest kept the required `scope` gate category but replaced
both scope gate commands with `true`. The D.6 artifact then claimed that
scope gate as `PASSED` with `command-exit-0`. Both mechanics accepted it.

## 5. Security surface

No new shell-injection issue surfaced in the tested scripts. Paths with
spaces are quoted through `verify-adopter.sh` and its sub-checks. The
probe mode does not run without an explicit model argument and reports a
missing `codex` binary before dispatching anything.

The main trust-boundary issue is semantic, not injection: D.6 gate
evidence is self-reported by the artifact table. `verify-d6-gate.sh`
validates the table grammar and evidence tag, not the command's
relationship to manifest scope.

## 6. Blind-spot identification

Codex-family blind spot: this release correctly hardened several exact
bugs from v1.1, so the obvious attacks now pass. The remaining weak point
is the same class at a higher layer: a prose routing decision moved work
from humans to "the mechanical gate", but the named gate only proves that
a row exists and is labeled as passed. The adversarial question is not
"does the table parse"; it is "does the table prove the same thing the
human reviewer was told not to review?"

## 7. Risk-assessment override

Risk: medium.

The release is not a re-scope. The requested negative tests mostly pass,
the schema enum regression is closed, and no v1.0/v1.1/v1.1.1 manifest
regression appeared. The remaining issue can let a pure scope violation
escape when adopters provide weak scope gates and reviewers obey the new
3.9 boundaries. That is a release-quality problem but not a doctrine
rewrite.

## 9. Verdict

**Needs Fixes.** No blocker and no re-scope finding. The release's direct
checks behave correctly except for the 3.9 coverage-equivalence gap. That
gap should be fixed before treating the tightened reviewer boundaries as
safe: either make the mechanical scope gate prove actual scope coverage,
or keep raw scope violations inside a human lens until the gate is strong
enough.
