## 1. Architecture compliance

No architecture-layer violation in the executable or template changes.
`git diff main --name-status -- frameworks/llm-dev-v1/templates` reports only:

- `M frameworks/llm-dev-v1/templates/06-meta-consolidator.md`
- `M frameworks/llm-dev-v1/templates/08-retrospective.md`

No new template is introduced, no template is removed, and no state-machine
phase is renamed. C-4 attaches archive ordering to existing Phase E
retrospectives (`templates/08-retrospective.md:116-123`). B-5 attaches
cardinality re-baseline to existing B.3 meta-consolidation
(`templates/06-meta-consolidator.md:171-186`).

C-1 widens only the `id` pattern to permit dot-separated components while
leaving `slug` dot-free (`manifest/deliverable-manifest.schema.yaml:145-158`).
This is schema widening, not a schema rename. C-2 adds an orchestrator-only
suppression section and script extraction path; the direct `verify-all.sh` run
confirms the token suite now passes with no unresolved token warning.

C-5 is user-visible because `verify-schema.sh` now exits 1 rather than 2 when
`check-jsonschema` is missing (`scripts/verify-schema.sh:23-29`). I do not
classify it as a v1.2-only contract break: the README now marks
`check-jsonschema` required (`README.md:66-70`, `README.md:116-121`), and P12
requires prompt suites to be validated before dispatch. The old soft-skip was a
degraded validation mode, so hard-failing is patch-level hardening rather than
feature scope.

## 3. Roadmap alignment

The patch scope matches the five named folio.love cherry-picks, and the
code-level changes are local to the declared files. However, the release/stamp
prose overclaims the current state of the branch and understates the exact kind
of changes shipped. See ALIGN-SF1 and ALIGN-SF2.

The main alignment risk is not the fixes themselves; it is adopter-facing
documentation that currently says:

- v1.1.1 is "Released" with `PR #TBD` / merge `TBD` (`CHANGELOG.md:6`).
- exit criteria are already met and a canonical verdict exists at
  `review-board/v1.1.1-spec-verdict.md` (`ROADMAP.md:112-116`).
- the scope has "no schema or template-contract changes" while the same
  section lists a schema pattern change and normative template additions
  (`ROADMAP.md:89-105`; `docs/handoffs/2026-04-15_folio-love-llm-dev-adoption.md:217-220`).

Those are fixable documentation/stamp issues, but they should be corrected
before merge because downstream adopters read these files as release authority.

## 5. Backward compatibility

No backward-compat regression found.

The two promised manifest witnesses validate unchanged under the v1.1.1 schema.
The v1.0.0 walkthrough diff is empty. C-1 is additive for `id` values: existing
dash-only IDs remain valid, and adopters gain semver-friendly dot components.
`slug` remains governed by the original dot-free filesystem-safe pattern.

C-5 is the only user-visible script-exit change. It affects an invalid
verification environment (missing required validator), not valid manifests. In
doctrine terms it closes a degraded validation mode rather than introducing one.
The README and `verify-all.sh` comments document the exception, and the required
direct run confirms the normal environment remains green.

B-5 does not replace P6's runnable cardinality checks with prose. It adds a
meta-consolidator duty to keep the manifest's mechanical assertions aligned
with the finalized spec before B.3 Approve. The eventual D.6 failure/pass still
depends on runnable assertions; B-5 is a process guard against stale expected
values, not a substitute for mechanical scope enforcement.

## 9. Verdict

**Needs-Fixes**

The code-level v1.1.1 patch is compatible: required direct-run checks are green,
both v1.0.0/v1.1.0 manifest witnesses validate unchanged, the v1.0.0 walkthrough
is byte-unchanged, no new template appears, no token definition is removed, and
C-5 removes rather than adds a degraded validation mode.

The merge should wait on two documentation/stamp corrections. Both are
adopter-facing authority issues: the release files currently claim completion
before the canonical verdict exists, and they describe the patch as having no
schema or template-contract changes even though the branch intentionally widens
one schema pattern and adds normative text to two existing templates. Fixing
those statements should be sufficient for approval from this lens.
