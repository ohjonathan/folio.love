## 5. Backward compatibility

Every witness the v1.1 Alignment-lens review examined has been re-run.
Core findings:

- **v1.0.0 manifest validates unchanged.** `check-jsonschema` direct-run
  passes against the HEAD schema. Same result on both the branch-HEAD
  copy (which has new optional fields populated for demonstration —
  `evidence_cap`, `review_rounds`) and the main-tree byte-copy of the
  v1.0.0 manifest. See ALIGN-MIN1 about the in-branch v1.0.0 example
  drift; this is cosmetic drift, not a contract break.
- **v1.1.0 user-facing manifest validates unchanged.** Direct-run OK.
- **Walkthrough byte-unchanged.** `git diff main..HEAD` returned 0 lines
  for both `walkthrough-mini-deliverable.md` and
  `walkthrough-user-facing.md`.
- **Manifest version enum.** This is the contract-break vector and it
  does **not** break: the enum stays `["1.0.0", "1.1.0"]`. Every
  previously-stamped manifest parses unchanged. BUT: see ALIGN-B1. The
  new v1.2 adversarial-family invariant is gated on `manifest_version ≥
  1.2.0`; because `1.2.0` is not a legal value under the current enum,
  no adopter can opt into the new invariant. The gate works in one
  direction (existing v1.0/v1.1/v1.1.1 manifests are correctly
  grandfathered — no regression), but in the other direction the gate
  is closed (no manifest can legally claim v1.2.0 to opt in).

No breaking changes to Templates, scripts, or tokens. The Template 06
default path (`<USER_FACING>=false`) still produces a 3-row family
verdict table, semantic preserved from v1.1; the v1.2+ metrics block is
appended after the existing sections (lines 169-193). Template 07 Gate
table gains one column (`Evidence class`) and rows use allowed tag
values — any v1.1-era final-approval artifact without that column
would fail `verify-d6-gate.sh` if run against it, but v1.1 did not ship
a mechanized D.6 gate. Reviewer-dispatched alternative is explicitly
preserved for edge cases (playbook.md:232-239).

## 9. Verdict

**Needs Fixes** (one blocker, two minors).

The v1.2 bundle satisfies the alignment-lens criteria on every axis
except the schema `manifest_version` enum:

- P3 adversarial-family invariant is explicit and version-gated (one
  paragraph per v1.2+ trigger, cross-reference to scope-proposal and
  folio F-006). Doctrinal language tight.
- Adopter-authority preserved: folio v1.1.1 was cherry-picked into
  canonical (not merged from adopter back), and v1.2 items 2.1 / 3.5
  treat the adopter as a read-only consumer of canonical.
- Backward-compat witnesses intact: v1.0.0 walkthrough byte-unchanged
  (0-line diff), v1.1 walkthrough byte-unchanged, both example
  manifests validate under the v1.2 schema.
- Conformance suite 10/10 green (direct-run confirmed on the HEAD
  commit).
- Build-plan ↔ commit mapping: 1:1 for all 18 build-plan line items;
  Phase 4.3 re-defer correctly landed as prose-only.
- P5 evidence-rule self-check: prose-vs-mechanical classifications are
  honest — 1.3 / 3.2 / 1.2 are mechanical with fixtures; 1.4 / 3.6 / 3.8
  are honestly prose.
- Cross-run evidence weighting preserved from build plan to CHANGELOG.

The one blocker (ALIGN-B1) is mechanically fixable in a one-line schema
edit plus a one-line generator-spec update. Its classification as
blocker rests on the coherence argument in the scope-proposal (the
exact argument that elevated 1.3 from should-ship to must-ship): a
mechanical gate that exists but is unreachable is a structural defect,
not a rough edge. The invariant was explicitly called out in the
alignment-lens dispatch ("confirm verify-p3.sh actually version-gates —
direct-run the v1.0 example; it must still pass") — the version-gate
works in the grandfather direction; the enum-widening direction is what
fails.

Two minors (ALIGN-MIN1, ALIGN-MIN2) are cosmetic drift that reduce
witness clarity; neither is block-merge.
