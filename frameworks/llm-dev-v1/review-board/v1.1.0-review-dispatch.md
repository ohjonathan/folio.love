per `docs/v1.1-build-plan.md` Phases 0-6. Phase 7 dispatches a

## Review scope (scope items per `docs/v1.1-build-plan.md` Phase 7)

All three families evaluate the same bundle but with slice-specific
emphasis per family strength:

### Codex — mechanics + schema slice

**Primary scope.** Verify every executable-contract claim in v1.1.

1. Both example manifests validate (backward-compat guard):
   `bash scripts/verify-schema.sh` — confirm exit 0, both
   manifests listed as validated.
2. `verify-p3.sh` aggregation logic: confirm the script correctly
   aggregates across multiple model-assignment entries per phase (P10
   same-family-separate-session pattern used by the user-facing example).
3. `verify-pre-a.sh` coherence check: confirm it accepts the
   user-facing manifest, rejects a malformed `pre_a` block (construct
   a negative test and run the script).
4. Generator-spec invariants 15-17: read each, confirm the cited
   script would enforce it, or flag unenforceable invariants.
5. Expectation grammar: every `expect:` field in both manifests
   conforms to the schema's `definitions.expectation`.
6. New schema fields: `user_facing`, `pre_a`, `product_verdict` and
   related — confirm backward compat (v1.0.0 shape validates
   unchanged against v1.1 schema).
7. New tokens in `tokens.md` Categories 6 and 13: every new token has
   a meaning, example fill, and is referenced by at least one
   template.

**Verdict format:** Approve | Needs Fixes | Re-scope.
**Output path:** `review-board/v1.1.0-review-codex.md`.
**Template reference:** `templates/05-review-board-adversarial.md`
(mechanics-focused adversarial posture).

### Claude — doctrine slice

**Primary scope.** Verify doctrine consistency for the v1.1 additions.

1. P3 carve-out in `framework.md`: read the pre-A carve-out and
   user-facing extension paragraphs. Confirm no degraded-mode escape
   hatches, no ambiguity about when ≥3 families applies, Product
   overlap rule explicitly stated.
2. Templates 16, 17, 18, 19 each define a distinct role that does
   not overlap with existing template roles. Confirm halt conditions
   are non-empty and mechanically checkable.
3. Meta-consolidator conditional intake: read the 4-row stanza in
   `06-meta-consolidator.md`. Confirm the default (3-row) path is
   unchanged from v1.0.0 and the 4-row path activates only when
   `<USER_FACING>` is true.
4. Compatibility table in `framework.md`: confirm the v1.1 extension
   note about Product lens scope extension beyond playbook §13.4 is
   present and accurate.
5. Phase state machine + entry/exit criteria table: confirm the three
   pre-A variants are reachable, have well-defined exits, and do not
   create ambiguous paths (e.g., `-A.validation` defect → which exit?).

**Verdict format:** Approve | Needs Fixes | Re-scope.
**Output path:** `review-board/v1.1.0-review-claude.md`.
**Template reference:** `templates/04-review-board-alignment.md`
(doctrine-compliance alignment posture).

### Gemini — template + walkthrough slice

codex (or another family chosen by the orchestrator) with
`templates/06-meta-consolidator.md` to produce:

`review-board/v1.1.0-spec-verdict.md`

The canonical verdict follows the v1.0.0-era shape: context header,

3 rows — no Product verdict needed since the bundle itself is

blockers, should-fix, minor, contradictions (if any), agreement
analysis, required actions, decision summary.

  schema, and template issues).

  format as v1.0.0's response doc — row-per-blocker table with fix
  description, commit SHA, evidence).

## Stamp criteria

v1.1.0 stamps when:

1. Canonical verdict is `Approve` (possibly after 1-2 rounds of fix +
   re-review).
2. `bash scripts/verify-all.sh` still passes on final HEAD.
3. `CHANGELOG.md` top entry status flips from "Unreleased (pending
   review board)" to "Released YYYY-MM-DD (PR #N, merge `SHA`)".
4. `ROADMAP.md` v1.1.0 status flips from "Unreleased" to "Released".
5. Session archived via `ontos log -e "v1-1-0-release"`.

No git tag is created (per `ROADMAP.md`; tags reserved for a future
extracted project).
