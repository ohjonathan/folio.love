fidelity). Scope: v1.2.0 bundle release, full 3-family minor-release

against example-manifest.yaml). Structural/editorial findings are
`static-inspection` as in v1.1.0 — prose bundles offer limited
executable surface beyond the verify suite.

## 6. Positive observations

- **Cross-run-consensus gating for must-ship is operationally sound.**
  Four Phase 1 items all have cross-run evidence — 1.1 (folio F-006 +
  D3 retro §2), 1.2 (folio F-030 + D3 scope-proposal #8), 1.3 (D3
  scope §3 + folio C-7 deferral), 1.4 (D3 scope §4.1 prose slice).
  Two independent adopter retros concurring on a friction is a higher
  signal than any single retro; the build plan's phase taxonomy made
  this visible by construction. This is the v1.2 build's biggest
  doctrine validation — the cross-family review board + two-adopter
  retro loop produced consensus, then consensus drove the must-ship
  selection.
- **Fixture-based script regression is the right pattern.** The
  d6-gate fixture in `examples/d6-gate-fixture.md` + the three
  cb-fixtures in `examples/cb-fixture-*.yaml` give `verify-all.sh`
  exactly the regression signal the script needs — parser drift or
  schema-shape drift fails the regression before an adopter's real
  deliverable run does. The "intentionally non-normative" note in the
  d6 fixture prose is the right kind of signaling.
- **Decision-only v1.3 deferrals are named in the CHANGELOG.** The
  Deferred-to-v1.3 block lists 4.3 ADV-SF5 (token↔manifest mechanical
  linkage) + cross-variant Pre-A consolidation (Template 16b) + the
  folio.love-parked items (12 items tracked by name). This is the
  right audit-trail discipline — a reader knows what was consciously
  deferred vs what was missed. Matches the v1.1.1 pattern of naming
  the slice set even when the fix was cherry-picked.
- **Scope-proposal + build-plan provenance is tracked in
  PROVENANCE.md.** Two new rows (`docs/v1.2-scope-proposal.md` and
  `docs/v1.2-build-plan.md`) in the non-normative-artifacts table.
  Downstream adopters who want to audit decision trails can walk
  scope-proposal → build-plan → per-item commit → review-board
  verdict → CHANGELOG stamp in sequence.
- **Orchestrator consolidation fast-path (3.6) is conservative in
  the right direction.** Fast-path requires unanimous lens verdicts
  AND no cross-lens conflicts. Any disagreement routes back to full
  meta-consolidator dispatch. This is a strict-subset optimization —
  no split-verdict round can accidentally take the fast-path. The
  `consolidation_mode: fast-path` frontmatter tag makes the choice
  auditable.
