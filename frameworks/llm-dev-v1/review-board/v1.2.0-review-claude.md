### PEER-SF1 — `<MERGE_WORKSPACE?>` token defined but not referenced
- **Lens:** Peer | **Evidence:** direct-run
- **Location:** `frameworks/llm-dev-v1/tokens.md:85`; `scripts/verify-tokens.sh:99-110`
- **Description:** `<MERGE_WORKSPACE?>` is defined in `tokens.md` Category 1 (v1.2 Phase 4.1) but never referenced in any scanned file (templates, framework.md, playbook.md, generator-spec.md). `verify-tokens.sh` emits `[warn] tokens defined but not referenced: <MERGE_WORKSPACE>`. The warning is non-fatal (`verify-tokens: OK` still returns at the end) but the token is defined-but-unused until v1.3.
- **Reproduction:** `bash scripts/verify-tokens.sh 2>&1 | head -3` returns `verify-tokens: [warn] tokens defined but not referenced: <MERGE_WORKSPACE>`.
- **Suggested action:** Minimum-fix: add `<MERGE_WORKSPACE?>` to the "Orchestrator-only tokens" section of `tokens.md` (same mechanism v1.1.1 used for `<MANIFEST_DIR>` — verify-tokens suppresses the warning). Stronger fix: wire the token into `playbook.md § Merge safety` or `00-orchestrator-runbook.md`'s final-merge step, which is the natural consumer.

### ADV-SF1 — Adversarial-provider escape-hatch is pure id-prefix match
- **Lens:** Adversarial | **Evidence:** direct-run
- **Location:** `scripts/verify-p3.sh:172-177`
- **Description:** The v1.2+ P3 adversarial-family MUST-differ-provider invariant is enforced in `verify-p3.sh`. Adopters can authorize same-provider adversarial by adding a `gate_prerequisites` entry with id prefix `G-xprov-adv-<phase>`. Prose (script comment lines 20-24 + framework.md §P3) says that entry "names a cross-provider second-pass artifact". The script inspects only the `id` field — not command shape, not artifact-path pointer. A hollow entry with `command: true` passes.
- **Reproduction:** Synthesize a v1.2.0 manifest with `claude-opus` author + `claude-sonnet` adversarial on B.1 and a `G-xprov-adv-B.1` entry with `{command: "true", expect: "exit-0"}` → `verify-p3.sh --manifest /tmp/mf.yaml` returns OK, exit 0.
- **Suggested action:** Add a narrow schema constraint on `gate_prerequisites` entries with id prefix `G-xprov-adv-`: require `description` mentioning a family name from `model_assignments` with a distinct provider, OR a new optional `cross_provider_family` field. Even validating `description` is >20 chars closes the worst case.

### ADV-SF2 — 3.9 role-boundary prose routes scope-compliance to scripts that don't check it
- **Lens:** Adversarial | **Evidence:** direct-run
- **Location:** `templates/03-review-board-peer.md:45-51`; `templates/04-review-board-alignment.md:47-57`; `templates/19-review-board-product.md:74-81`; `scripts/verify-artifact-paths.sh:5-18`; `scripts/verify-portability.sh`
- **Description:** v1.2 Phase 3.9 templates tell Peer/Alignment/Product reviewers that scope-compliance findings belong to "the mechanical gate (verify-artifact-paths, verify-portability, D.6)". Two of the three named scripts don't actually check scope compliance: `verify-artifact-paths.sh` only validates placeholder shapes of `artifacts.*` fields; `verify-portability.sh` scans the bundle's own normative files for host-project strings. Only D.6's adopter-authored `G-scope-*` commands catch scope violations — and only as deep as those commands go.
- **Reproduction:** Copy `example-manifest.yaml`; replace `G-scope-1` + `G-scope-2` verification commands with `{command: "true", expect: "exit-0"}`; `bash scripts/verify-adopter.sh /tmp/mf-hollow-scope.yaml` → PASSED. No bundle script checks that e.g. `src/auth/` wasn't touched.
- **Suggested action:** Fastest fix: strike `verify-artifact-paths, verify-portability` from the parenthetical in T03/T04/T19 and replace with "the D.6 gate's G-scope-* prerequisites". Stronger fix: add `scripts/verify-scope-diff.sh <manifest> <base> <head>` that evaluates `scope.forbidden_paths` against a diff range.

### ADV-SF3 — §11 Contract-enumeration audit is reviewer-discipline with no script backing
