# Adversarial Review — frameworks-llm-dev-v1 / v1.1.0 / gemini

## 1. Assumption attack

| Assumption | Why it might be wrong | Impact if wrong |
|------------|------------------------|-----------------|
| "The `user_facing` boolean flag correctly determines if the Product lens is enforced." | The flag's value is entirely self-reported by the deliverable author. There is no automated heuristic (e.g., checking `allowed_paths` for UI/frontend directories) to force `user_facing: true`. An author can bypass the heavy Product scrutiny by simply leaving it `false`. | A critical user-facing feature merges without Product-lens review. The entire v1.1 user-facing security model is opt-in and trivially bypassed. |
| "v1.0.0 manifests parse unchanged under v1.1." | The schema lists `manifest_version` in the `required` array (unchanged from v1.0.0). If an adopter holds a v1.0.0 draft manifest without this field (as it was an unenforced convention prior to v1.0.0 stamping), it fails to validate against the v1.1 schema. | Breaking change for early adopters holding drafts; forces manual intervention to inject the `manifest_version` field to pass validation. |
| "verify-pre-a.sh enforces pre-A routing coherence." | The script only checks if the manifest fields are populated. It never reads the actual pre-A artifact verdict to determine correct downstream routing (e.g., Triage to C-direct vs. Phase A). The enforcement is exclusively prose-based in `generator-spec.md`. | Routing invariant 17 is essentially a human-operated suggestion. An orchestrator could trivially route a rejected triage finding into Phase A without any mechanical consequence. |

## 2. Failure mode analysis

| Failure | How it happens | Would we notice? |
|---------|----------------|-------------------|
| Meta-consolidator silent token failure | The orchestrator fails to provide a value for the optional `<USER_FACING>` token. `06-meta-consolidator.md` falls back to the default 3-row review board, completely ignoring any Product verdicts written to disk. | No. The review passes with only 3 families, invalidating the v1.1 Product lens extension silently. |
| `verify-artifact-paths.sh` bypass | The script hardcodes `example-manifest.yaml` and never iterates over the user-facing example. Consequently, the grammar rules for `product_verdict` (requiring `<phase>` and `<family>`, forbidding `<role>`) are completely unenforced. | No. A malformed `product_verdict` path passes all CI checks but causes the meta-consolidator to halt or write to an invalid path at runtime. |
| Multiple product roles per phase | The schema does not restrict the number of `product` roles in `model_assignments`. `verify-p3.sh` checks for "at least one", but `06-meta-consolidator.md` uses a singular `<PRODUCT_VERDICT_PATH>`. If an author assigns two Product reviewers, one verdict is silently overwritten or the pipeline crashes. | No static check prevents this; pipeline fails at runtime. |

## 3. Diagram completeness attack

The state machine diagram logic for Pre-A exits in `framework.md` lacks the `C-direct` fast-patch route defined in generator invariant 17(a). `framework.md:70` shows `C direct (trivial fix)` but the relationship from `-A.triage` specifically is only defined in prose. This is a minor misalignment but does not block the release. Similarly, invariant 17(b)'s `exit_path` is referenced in the generator spec but nowhere in the Validation template enum.

## 4. Edge case inventory

- **`pre_a.artifact_path` Empty String:** The schema defines this as a string but lacks `minLength: 1`. An empty string `""` validates against the JSON schema, although `verify-pre-a.sh` catches it (Schema vs. Script divergence).
- **`<USER_FACING>` token empty:** If an empty string is passed to `06-meta-consolidator.md`, the markdown renders malformed lines (e.g., `- **User-facing:**   (true...`).
- **`user_facing: true` but missing `product_verdict`:** Schema lists it as optional, so JSON schema validation passes. `verify-p3.sh` checks for it, but relying exclusively on bash scripts for presence validation when the schema claims authority is brittle.
- **Regex ReDoS in `expect`:** The `expectation` schema allows arbitrary regex (`regex:.+`). This is safe for the framework itself but poses a risk for downstream evaluators processing untrusted manifests.

## 5. Security surface

- **Token Substitution Injection:** As noted in the architecture, tokens like `<DELIVERABLE_ID>` are substituted dynamically. If an author uses a payload containing nested brackets, it could manipulate the prompt structure and bypass token safety guards.
- **Opt-in Product Lens:** The most significant security/process surface is the `user_facing` flag. Because it is opt-in, the framework relies entirely on human honesty to trigger its most rigorous review path, bypassing the zero-trust adversarial model.

## 6. Blind-spot identification

The framework architects assume that JSON Schema validation and bash script validation are perfectly equivalent or complementary. There are multiple instances where the schema permits invalid states (e.g., missing `product_verdict` when `user_facing` is true, empty string `artifact_path`) that are only caught by bash scripts, or conversely, bash scripts that fail to enforce stated schema invariants (like `verify-artifact-paths.sh` entirely skipping the user-facing manifest). This creates a "Swiss cheese" validation model where downstream consumers running only JSON schema validation will incorrectly assume the manifest is valid.

## 7. Risk-assessment override

The author implies a low risk. I override this to **Medium**. The v1.1 additions rely heavily on cross-file coherence that is not mechanically enforced across the full suite (specifically `verify-artifact-paths.sh` omitting the user-facing example, and the schema marking mandatory fields as optional). Since this is the framework's first production adoption (`folio.love`), these mechanical gaps will immediately surface in the orchestrator as runtime crashes or silent security downgrades.

## 10. Notes
The v1.1 extension introduces a major architectural trade-off by relying heavily on bash scripts to enforce invariants that JSON Schema Draft 07 cannot cleanly express. While this is acceptable for a CLI-driven workflow, it creates severe blind spots where the scripts themselves do not cover the full matrix of valid manifests. Fixing the `verify-artifact-paths.sh` array and tightening the JSON Schema constraints will close the most dangerous mechanical gaps.
