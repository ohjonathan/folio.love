# Alignment Review - frameworks-llm-dev-v1 / v1.2.0 / codex

Reviewer posture: doctrine compliance, backward compatibility, build-plan
fidelity, adopter-authority preservation. Evidence drawn from branch HEAD on
`frameworks/llm-dev-v1.2`, static inspection of the normative bundle and
handoffs, `git log main..HEAD --oneline`, direct conformance runs, and
synthetic manifest checks under `/tmp`.

## 1. Architecture compliance

The v1.2 branch is mostly additive: no template contract renames surfaced in
the diff, token-table changes are additions only (`<CLI_CODEX_MODEL?>`,
`<MANIFEST_DIR>`, `<MERGE_WORKSPACE?>`, `<EVIDENCE_CAP?>`), and the schema
adds optional `review_rounds[]` plus optional
`cli_capability_matrix[].evidence_cap`. The v1.0.0 walkthrough is byte-
unchanged.

One doctrine blocker remains: the same-provider adversarial escape hatch is
too weak mechanically. `framework.md:164-174` says same-provider adversarial
is permitted only when a gate prerequisite documents a cross-provider second
pass artifact before D.6, and says `verify-p3.sh` enforces this
mechanically. In practice, `verify-p3.sh:170-188` only records a phase as
escaped when a `gate_prerequisites[].id` equals `G-xprov-adv-<phase>`.
It does not verify that the prerequisite names a real artifact, that the
artifact is cross-provider, or that it runs before D.6. See ALIGN-BLOCKER1.

There is also a doctrine consistency gap in the v1.2 fast-path. P3 still
says every round produces exactly three family verdicts plus one canonical
verdict "from the meta-consolidator" (`framework.md:134-137`), and the
artifact contract still lists Canonical verdict owner as Meta-Consolidator
(`framework.md:271-277`). `playbook.md:123-136` newly allows the
orchestrator to author the canonical verdict directly on unanimous rounds.
That may be a valid v1.2 extension, but it needs an explicit doctrine
exception and version/opt-in boundary before release. See ALIGN-SF1.

## 2. Diagram and state-machine cross-reference

No state-machine break found in the Phase 0/A/B/C/D/E or pre-A diagrams.
The v1.2 changes do not add new phase ids. The adversarial-family rule is
scoped to B.1, B.2, and D.2, matching the review-board phases where the
Adversarial role appears. The fast-path issue is not diagram-shaped; it is
an owner/role-contract mismatch for the canonical verdict.

`verify-d6-gate.sh` rejects non-PASSED rows and disallowed evidence classes.

The circuit-breaker check is also mechanical for the manifest state it

The same-provider adversarial escape hatch is the one P5/P3-style rule that

bare id, not the evidence artifact the doctrine requires. See
ALIGN-BLOCKER1.

## 8. Schema enum fix (ALIGN-B1 close)

The latest commit does widen the schema enum:
`deliverable-manifest.schema.yaml:120-130` accepts `"1.0.0"`, `"1.1.0"`,
`"1.1.1"`, and `"1.2.0"`. Direct validation of synthesized 1.1.1 and 1.2.0
manifests passed.

That closes the schema-validator gate. It does not fully close the
normative generator gate: `manifest/generator-spec.md:212-219` still says
the generator targets only `manifest_version: 1.0.0` or `1.1.0` and rejects
higher versions. Since the generator spec is normative even though the
implementation is deferred, this leaves stale adopter-authority text in the
manifest toolchain. See ALIGN-SF2.

## 10. Verdict

**Needs-Fixes**

The direct backward-compat gates are green: `verify-all.sh` passed, both
bundled examples validate, the original v1.0 manifest from `main` validates
against the v1.2 schema, the v1.0 walkthrough is byte-unchanged, and the
adversarial-provider rule is correctly version-gated for the no-escape case.

Approval is blocked by ALIGN-BLOCKER1. The must-ship adversarial-family
invariant can be bypassed by a dummy gate id while still passing
`verify-adopter.sh`, so v1.2 would advertise a mechanical cross-provider
guard that the adopter gate does not actually enforce. The other should-fix
items are doctrine/spec consistency and build-plan completion gaps that
should be closed before the final v1.2 canonical verdict.
