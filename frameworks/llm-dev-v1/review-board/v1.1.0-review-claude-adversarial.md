# Adversarial Review — frameworks-llm-dev-v1 / v1.1.0 / claude

## BEGIN ADVERSARIAL REVIEW

Lens: failure. Core question: how does this fail, what bypasses the
mechanics, what silently passes? The v1.1 build plan declared no risk
level; I assume author would rate low, and attempt to prove that wrong.

## 3. Diagram completeness attack

State-machine prose at framework.md:69-71 names three pre-A exits:

- `-A.validation → A (revised plan) | hotfix/incident | re-run`

The explicit `-A.validation` verdict "Run exposed defect — escalate to
hotfix or incident" (docs/v1.1-doctrine-decisions.md §1) maps to
`hotfix/incident` in the diagram. No blocker. However the diagram does
NOT show the C-direct fast-patch exit that invariant 17(a) promises
from `-A.triage`. framework.md:70 only shows `A (In-scope work) | C
direct (trivial fix) | ∅ (Deferred only)` — OK, C direct is present
in triage row. Proposal row line 69 shows `∅ (Abandon)` — consistent.

No blocking diagram/prose mismatch. One should-fix: invariant 17(b)
verdict-to-routing mapping (`exit_path` field in the generator-spec)
is not explicitly shown in the state diagram nor in template 18's
verdict enum (templates/18-validation-run.md). The generator-spec
introduces an `exit_path` field that no template defines.

## 4. Edge case inventory

- **Empty `pre_a.artifact_path`** (`""`): schema accepts (no minLength),
  verify-pre-a.sh catches. Divergence is a should-fix.
- **Unknown `pre_a.entry`**: schema catches via enum. OK.
- **`user_facing: true` but `model_assignments` has zero `product` roles**:
  verify-p3.sh catches for B.1/B.2/D.2. OK.
- **`user_facing: true` but `artifacts.product_verdict` missing**: no
  check; template 06 halts at runtime. BLOCKER.
- **`user_facing: true` but `artifacts.product_verdict` path contains
  `<role>` (grammar violation)**: verify-artifact-paths.sh does not
  inspect user-facing manifest; template-06 path resolution generates a
  literal `<role>` string in the dispatched path. BLOCKER.
- **Two `product` entries same phase, different families**: accepted by
  all static checks; runtime contradiction with singular
  `<PRODUCT_VERDICT_PATH>`. SHOULD-FIX.
- **v1.0.0 manifest without `manifest_version`**: schema rejects. Claim
  of "v1.0.0 manifests parse unchanged" breaks for pre-stamp drafts.
  SHOULD-FIX (wording or schema).
- **`-A.validation` verdict = "Run exposed defect"**: invariant 17(b)
  prose-only. No static check. MINOR (doctrine accepts orchestrator
  responsibility, but flagging for honesty).
- **Dispatch of Product role without `<USER_FACING>` token being fed**:
  token is `<USER_FACING?>` (optional); template 06 falls through to
  3-row default. Behavior safe but silent — adopter who mis-fills
  `USER_FACING` as empty but sets `user_facing: true` in manifest gets
  the 3-row path with Product verdicts on disk being ignored. SHOULD-FIX
  (cross-link `<USER_FACING>` to manifest.user_facing mechanically).

## 5. Security surface

Normative bundle files are static text + shell scripts. Threat surface
is narrow:

- **Token substitution injection.** A malicious token fill containing
  `<ANGLE_UPPER>` patterns could introduce new placeholders. `tokens.md`
  § Forbidden patterns bans nested tokens; verify-tokens.sh enforces at
  template-author time but not at fill-time (fill-time substitution is
  generator-v2 scope). Accepted risk, but worth documenting that
  `tokens.local.md` is in the adopter's trust boundary.
- **YAML load** in verify scripts uses `yaml.safe_load` (confirmed in
  verify-p3.sh:33, verify-pre-a.sh:33). Good.
- **`check-jsonschema` / `pyyaml` dependency.** No pinning in the
  bundle; adopters may run with outdated tools. MINOR portability note,
  not a security blocker.
- **Shell injection in verify-portability.sh.** grep pattern built from
  static string; no user input path reaches shell. Safe.
- **`regex:` / `literal:` expectation grammar.** An adopter manifest can
  specify arbitrary `expect: regex:<pattern>`. The runtime evaluator is
  out of scope (generator v2). Accepted.

No exploitable security issue at the bundle layer.

## 6. Blind-spot identification

**Claude's own blind spot.** Claude authored much of the v1.1 scope.
Likely blind spots:

- **Enforcement-vs-prose asymmetry.** v1.0.0 review flagged B3 (token
  mechanics not enforced). v1.1 repeats the pattern at a different
  layer: generator-spec invariants 15-17 describe routing contracts
  (e.g., Triage→C-direct) but only #16 has a mechanical check and
  #15/#17 have partial/no scripts. Claude likely wrote the generator-
  spec row "Enforced by: scripts/verify-pre-a.sh" assuming symmetry
  with #6-14 and did not revisit after writing the script. A
  non-Claude reader would spot the prose-vs-script gap faster.
- **Singular vs plural product verdict.** template 06 assumes one
  Product reviewer per phase. schema/verify-p3 accept multiple. Claude
  privileged the adoption-floor narrative ("4-family floor") and didn't
  pressure-test the "what if adopter configures 5 families with two
  Product reviewers" shape. Codex (mechanics slice) may already have
  caught this; worth cross-checking.
- **Scope of verify-artifact-paths.sh.** The script runs only against
  the v1.0.0 example. Adding the v1.1 manifest to its loop would be a
  two-line change. Claude likely duplicated the loop pattern from
  verify-p3.sh mentally but implemented only the v1.0.0 path here.

**Author blind spot (general).** Design relies on "adopters will not
configure weird P10 overlaps." See Risk Override.

## 7. Risk-assessment override

Author declared no risk. I override to **medium**.

Justification: v1.1 is additive and the conformance suite is 8/8 green,
but the user-facing extension introduces cross-field invariants (schema
field presence ↔ manifest flag ↔ model-assignment role ↔ artifact-path
shape ↔ meta-consolidator intake) where any single link missing a check
silently corrupts the dispatch. Two such links (B1, B2 below) have
direct-run reproductions. folio.love will be the first production adopter
(per build plan §Context), so mis-configuration risk transfers to a live
project.

Not high because failure modes halt at dispatch rather than corrupting
merged artifacts; merge safety (P8) is unchanged.

## 9. Verdict

**Needs Fixes.** Two direct-run-reproduced blockers (ADV-B1, ADV-B2)
demonstrate a prose-only invariant and a scope-limited enforcement
script. Neither is a doctrine or craft issue; both are mechanical gaps
matching the v1.0.0 review's B3/C3 failure-mode family ("prose-level
contract, no executable check"). The fixes are small (a handful of
lines per script + a schema tweak). Backward-compat and routing-
coherence should-fixes follow the same pattern.

v1.1's bundle is structurally sound: verify-all.sh genuinely runs a
useful suite, the doctrine carve-out is clean, the Product template
matches the scaffolding standard, and v1.0.0's known failure modes
(S1 optional risk level, B4 example-manifest invalidity) are closed.
The remaining gaps are enforcement-symmetry issues, not scope
problems.
