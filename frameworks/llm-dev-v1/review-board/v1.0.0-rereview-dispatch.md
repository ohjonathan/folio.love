# Focused Re-Review Dispatch — v1.0.0

Per verdict §10 action 8 ("Re-run focused verification, not a full new
design review"), this dispatch scopes each family to the slice where
they had the highest signal in the original review, then runs a tight
meta-consolidation.

**Inputs to every re-review worker.**

- `review-board/v1.0.0-spec-verdict.md` — the original canonical verdict.
- `review-board/v1.0.0-spec-verdict-response.md` — the fix summary
  mapping each blocker / should-fix to its resolution.
- The bundle at commit HEAD on branch `frameworks/llm-dev-v1`.
- The mechanical verification suite output: `bash scripts/verify-all.sh`
  (must exit 0 before any family is dispatched).

**Pre-dispatch gate (orchestrator runs):**

```bash
bash scripts/verify-all.sh    # must PASS
```

If `verify-all.sh` does not pass, do not dispatch — re-enter the fix
loop.

## Meta-consolidation (Codex)

After the three re-review artifacts land, Codex runs
`06-meta-consolidator.md` with inputs from all three. Because this is a
focused re-review rather than a fresh three-lens pass, the consolidator's
decision rule is simplified:

- If all three families return Approve → re-review PASSES. Orchestrator
  tags `llm-dev-v1.0.0` and updates `CHANGELOG.md`.
- If any family returns Request Further Fixes, preserved with evidence
  (file:line + reproduction or concrete check command + expected
  output), re-enter the fix loop for that slice only — do not restart the
  whole revision.

**Output path:** `review-board/v1.0.0-rereview-verdict.md`.

---

## Halt circuit breaker

If a re-review family raises the same class of blocker (same B-id or
same category) three times across rounds, escalate — do not re-dispatch
a fourth time. The three-halts rule applies to the revision cycle as a
whole.
