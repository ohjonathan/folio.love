## Preserved blockers (merge-blocking)

Each blocker below is preserved exactly as raised. No silent demotion.
Three distinct blockers across lenses; none duplicated across lenses.

### PEER-B1 — Triage disposition vs. generator-spec invariant 17(a) contradiction
- **Raised by:** claude / Peer
- **Description:** Template 17 routes `In-Scope | fast-patch` findings to Phase C-direct (with per-finding regression tests). Generator-spec invariant 17(a) says `pre_a.entry == "triage"` with "all findings classified as `rejected`" routes the generator to emit a Phase C-direct dispatch sequence. Semantically `rejected` means "not a real issue, already handled, or out of scope" and should NOT generate a fix dispatch. Adopters following either specification produce an invalid pipeline.
- **Location:** `manifest/generator-spec.md:121` vs `templates/17-triage.md:62-67`
- **Evidence:** `static-inspection` (side-by-side read of both files)
- **Reproduction:** Read both files side-by-side; templates say fast-patch → C-direct, generator-spec says rejected → C-direct.
- **Required action:** Rewrite invariant 17(a) to match Template 17: `pre_a.entry == "triage"` with all In-Scope findings dispositioned `fast-patch` routes the generator to emit a Phase C-direct dispatch sequence (skipping A and B), with per-finding regression tests as scope-lock. `rejected` findings emit no dispatch.

### ADV-B1 — `artifacts.product_verdict` presence not enforced when `user_facing: true`
- **Raised by:** claude / Adversarial
- **Description:** `artifacts.product_verdict` is schema-optional and nothing statically enforces its presence when `user_facing: true`, despite schema description at line 205 claiming "verify-artifact-paths.sh enforces" and template 06 (line 162) halting at runtime if it's missing. Enforcement is prose-only; `verify-p3.sh` checks role presence but not artifact-path presence, and `verify-artifact-paths.sh` ignores the user-facing manifest entirely.
- **Location:** `manifest/deliverable-manifest.schema.yaml:204-206`; `scripts/verify-p3.sh:91-103`; `scripts/verify-artifact-paths.sh:22`; `templates/06-meta-consolidator.md:16,162`
- **Evidence:** `direct-run`
- **Reproduction:** `python3 -c "import yaml; m=yaml.safe_load(open('manifest/example-user-facing-manifest.yaml')); del m['artifacts']['product_verdict']; yaml.safe_dump(m, open('/tmp/mf-no-pv.yaml','w'))"` then `check-jsonschema --schemafile manifest/deliverable-manifest.schema.yaml /tmp/mf-no-pv.yaml` returns `ok -- validation done, exit=0`. `verify-p3.sh` and `verify-artifact-paths.sh` both also pass.
- **Required action:** Extend `verify-p3.sh` user-facing branch (or `verify-artifact-paths.sh`) with: `if user_facing and not m['artifacts'].get('product_verdict'): fail`. Also widen `verify-artifact-paths.sh` to iterate both manifests (see ADV-B2).

### ADV-B2 — `verify-artifact-paths.sh` hardcoded to single (v1.0.0) manifest
- **Raised by:** claude / Adversarial
- **Description:** `verify-artifact-paths.sh` hardcodes a single manifest path and never validates `product_verdict` placeholder shape for the user-facing manifest. A `product_verdict` missing `<phase>` or `<family>`, or containing `<role>`, passes all v1.1 checks. Schema description at line 205 advertises grammar constraints ("Must contain `<phase>` and `<family>`; must NOT contain `<role>`") with zero mechanical enforcement.
- **Location:** `scripts/verify-artifact-paths.sh:22`
- **Evidence:** `direct-run`
- **Reproduction:** `sed -n '22p' scripts/verify-artifact-paths.sh` shows `manifest="$bundle/manifest/example-manifest.yaml"` (single string). Script never reads `example-user-facing-manifest.yaml`.
- **Required action:** Make `manifests=(…)` an array (matching `verify-p3.sh` pattern) and add require/forbid rules for `product_verdict` when the key exists.

No factual contradictions. The verdict spread (Alignment: Approve / 0
blockers; Peer: Needs Fixes / 1 blocker; Adversarial: Needs Fixes / 2
blockers) is complementary rather than conflicting — each lens looked
at a distinct surface:

- **Alignment** surveyed doctrine compliance, backward compatibility, and P3 constraint verification. It found these clean.
- **Peer** surveyed template quality, completeness, and scaffolding fidelity. It caught the cross-artifact contradiction between Template 17 and generator-spec invariant 17(a) — a spec-vs-spec issue Alignment's "doctrine vs. source playbook" sweep did not probe and Adversarial's "fail-mode / bypass" sweep did not surface.
- **Adversarial** pressure-tested the mechanical enforcement of cross-field invariants (schema ↔ manifest flag ↔ role ↔ artifact-path ↔ meta-consolidator intake). It caught two prose-only invariants Alignment trusted (the schema description at line 205) and Peer did not execute against.

The Alignment "Approve" verdict does NOT override the Peer and Adversarial
blockers — it reflects that doctrine narrative is clean while mechanics
and cross-artifact consistency have gaps. The family verdict therefore
follows the strictest preserved finding, per the consolidator ruleset.

One adjacent observation worth flagging: Adversarial §6 Blind-spot
identification notes that invariants 15-17's "Enforced by" column likely
assumed symmetry with #6-14 without being revisited after script
authoring. Peer's PEER-B1 (same invariants region) independently found
a different defect (disposition wording) in the same spec. This is
an agreement-gap signal — two lenses touched invariant 17 and each
found a distinct issue; neither refutes the other.
