## 5. Security surface

v1.1.1 touches three shell scripts (`verify-schema.sh`,
`verify-tokens.sh`, `verify-all.sh`), one schema regex, one template
(Template 06 paragraph), one template frontmatter addition
(Template 08), and a small `playbook.md` cross-reference. Threat
surface:

- **Id regex `^[a-z][a-z0-9]*([.-][a-z0-9]+)*$`.** No backtracking
  catastrophe risk: the inner alternation `[.-]` is a character class
  (no `|`), and the repeated group `([.-][a-z0-9]+)*` has a mandatory
  `[a-z0-9]+` after each separator, so there is no `(a+)*`-style
  ambiguity. Safe against ReDoS.
- **Awk flag-based range in `verify-tokens.sh`.** awk receives
  `tokens.md`'s content unmodified; no user input enters the awk
  program. No injection risk.
- **Hard-fail message in `verify-schema.sh`.** Emits to stderr only;
  no shell interpolation from manifest content; no command-substitution
  opportunity. Safe.
- **Template 06 paragraph.** Prose-only; no new token references; no
  new file operations. Safe.
- **Placeholder-SHA CHANGELOG / ROADMAP.** `TBD` strings are not
  executable. No risk.

No exploitable security issue at the bundle layer.

## 6. Blind-spot identification

**Claude's own blind spots on this patch release.**

- **Suppression-mechanism brittleness.** The v1.1.1 C-2 comment
  (verify-tokens.sh:49-50) addresses a prior range-based awk bug
  (ADV-SF-001, folio-internal) but the new flag-based approach
  introduces a NEW brittleness: section-name drift. I (Claude) likely
  wrote the comment focused on the specific bug that existed and did
  not consider adjacent drift modes. A non-Claude reader would
  probably write the awk pattern as `/^## Orchestrator[ -]only
  tokens/i` for resilience, or add an explicit assertion that the
  section exists.
- **Cross-reference hygiene for prose changes.** C-4 added "Archive
  ordering" as a named concept to Template 08 but only updated tokens.md
  with semantic (not lexical) cross-reference. A future adopter
  searching the bundle for the canonical phrase across all files
  (natural reader behavior) misses the tokens.md touchpoint. Claude
  typically writes semantic cross-refs because they're more durable,
  but in this case a bidirectional phrase match is cheap and would be
  stronger.
- **B-5 prose-only enforcement.** Claude authored the prose norm and
  immediately deferred the mechanical slice. The gap is acknowledged
  in `docs/v1.2-build-plan.md` but not in the B-5 prose itself — a
  reader of `templates/06-meta-consolidator.md:171-186` learns the
  norm is "normative for v1.1.1 B.3 consolidation" but gets no signal
  that mechanical enforcement is not planned. This is consistent with
  v1.1.0's pattern of "normative prose with mechanical gap"; the gap
  was flagged in v1.1.0 adversarial (ADV-SF3) and now repeats at B-5.
- **Silent-abort risk from `set -euo pipefail`.** Across the v1.1
  suite, `pipefail` is common. The C-2 awk pipeline is vulnerable to
  empty-grep silent abort. A blind-spot: the "silent-skip was removed"
  motivation applied to verify-schema.sh (C-5) was not symmetrically
  applied to verify-tokens.sh — which can still fail silently under
  the renamed-section scenario.

**Author blind spot (general).** v1.1.1 is ~219 lines across 13 files;
the author likely prioritized "each fix passes verify-all.sh" over
"each fix is robust to adjacent edits". C-1 and C-5 are robust; C-2
and B-5 are less so (C-2 for mechanical reasons; B-5 for the
prose-only gap).

## 7. Risk-assessment override

Author declared no risk level. I override to **low-medium**.

Justification: v1.1.1 is a narrow patch release (5 cherry-picks,

the changes, with strong direct-run evidence across all four

consistent with prior v1.1 adversarial findings.

Not `low` because the silent-abort path is genuinely invisible when
`verify-tokens.sh` is run standalone (the typical "I'll run just the
token check" workflow). Not `medium` because `verify-all.sh` correctly
surfaces the failure and names the script, so CI is protected.
