## 2. C-5 contract-change analysis (exit-2 → exit-1)

The orchestrator dispatch prompt asked Alignment to judge whether the
script-exit contract counts as a user-visible contract, and whether
C-5's hardening should have waited for v1.2.

**Is exit-2 vs exit-1 a user-visible contract?** Partially. The existing
`verify-all.sh` contract was documented in `README.md:113-116` at
v1.1.0: "If a dependency is missing the individual script exits 2;
`verify-all.sh` treats that as skipped (not failed) and prints which
checks were skipped so CI can catch the gap." So the contract was:
exit-2 = dependency-missing = skipped (CI-visible but not a gate
failure). CI pipelines written against that contract will behave
differently under v1.1.1:

- **Without check-jsonschema installed, v1.1.0 behavior:** verify-all
  exits 0, prints "skipped (2 not installed)" for verify-schema, CI
  passes with a warning.
- **Without check-jsonschema installed, v1.1.1 behavior:** verify-schema
  exits 1, verify-all exits non-zero, CI fails.

This is a behavior change for CI pipelines that do not install
`check-jsonschema`. However:

- The README/CHANGELOG/ROADMAP all document the change explicitly; no
  silent drift.
- `check-jsonschema` was already *named* as the expected validator in
  v1.1.0 README:63-64 — adopters running verify-suite without it were
  already in a "missing dep" state, which v1.1.0 treated as a soft-warn
  and v1.1.1 treats as a hard-fail.
- The change strengthens evidence discipline (P2): silent-skip of a
  schema check is genuinely indistinguishable from a passing check, and
  P2 exists precisely to forbid that ambiguity.
- The README § Running the checks (lines 116-121 after edit) now
  contains the word "Exception (v1.1.1)" calling this out in the
  preamble to the verify-suite section — adopters reading the suite
  docs will notice.
- The CHANGELOG row for C-5 carries the rationale verbatim.

**Should this have waited for v1.2?** Alignment-lens answer: no.

1. The directionality is strictly safer (more gate, not less). v1.1.1 is
   a patch release shipping five backward-compat-safe fixes pulled from
   an adopter; a CI pipeline that fails hard because a required
   validator is missing is safer than a CI pipeline that passes green
   while actually skipping schema validation.
2. The contract being hardened was a *soft* contract ("will skip
   gracefully") not a *load-bearing* contract (like manifest field
   semantics). No adopter manifest changes; no new template; no schema
   narrowing on a field adopters drafted content against.
3. v1.2 would be the right home if this were a semantic widening (new
   exit code 3 for "validator version mismatch", say) or a schema
   narrowing (reject a field some adopter set). It is neither. It is a
   cleanup of an observability gap in the verify-suite that happens to
   trip CI that was under-provisioned.
4. The build plan scoped C-5 as a patch; the folio.love adoption
   retro surfaced it as a F-005 "masked a real schema failure"
   bug-class finding, not a "feature request deferred to v1.2".

Verdict on C-5: accepted as in-scope for v1.1.1 patch release. No
finding filed. Cross-check with Adversarial — if the Codex lens judges
the CI-break risk differently, the meta-consolidator adjudicates.

**The C-1 regex narrowing note.** The `id` regex widens to accept dots
but also newly rejects three pathological forms that the v1.0.0 regex
accepted:

| Form | Old `^[a-z][a-z0-9-]*$` | New `^[a-z][a-z0-9]*([.-][a-z0-9]+)*$` |
|------|--------------------------|------------------------------------------|
| `abc-` (trailing dash) | accepted | rejected |
| `a--b` (adjacent dashes) | accepted | rejected |
| `abc1-` (trailing dash after digit) | accepted | rejected |

Mechanically this is a narrowing. Both example manifests
(`currency-converter`, `notification-preferences`) are fine under both
patterns. The new schema `description` field explicitly names the
rejected forms — the narrowing is documented, not silent. An adopter's
v1.0.0-authored manifest whose `id` had a trailing dash would break
under v1.1.1; no such adopter exists in-tree. The folio.love adopter
surfaced this fix so their `v0.6.4`-suffixed ids could validate — they
have motivation to keep the change, and the rejected forms are
stylistically inadvisable anyway. Alignment lens judges this as
acceptable refinement with explicit documentation, not a
backward-compat violation.

No blocker. Filed as ALIGN-MIN1 for downstream awareness.

## 4. Adopter-authority model

The orchestrator asked specifically: does the v1.1.1 sync note in the
folio.love adoption handoff preserve the directionality that patches
flow canonical → adopter, not adopter-local → canonical-merge?

**Reading the sync note.** `docs/handoffs/2026-04-15_folio-love-llm-dev-adoption.md:205-234`.

Relevant mechanics (lines 207-215):
- "folio.love shipped PR #51 (merge `ab4bc2735660`) with five
  backward-compat bug fixes ahead of the canonical johnny-os bundle.
  The maintainer cherry-picked those same fixes into the canonical
  bundle at branch `frameworks/llm-dev-v1.1.1`..."
- "The two copies are byte-equivalent for the v1.1.1 scope."
- "folio.love can resync its local bundle from the johnny-os v1.1.1
  merge SHA once it lands — no bundle-level changes required beyond
  re-copying `frameworks/llm-dev-v1/` (verbatim-copy adoption mode)."

And crucially, line 171-174 of the same handoff (which predates the
v1.1.1 note but remains authoritative for the directional contract):
"Do not patch the bundle inside folio.love without upstreaming the fix
to johnny-os first. A locally-patched bundle diverges from the
canonical version and loses the backward-compat guarantee the
framework ships."

**Assessment.** The sync note describes what happened — folio.love did
ship fixes to their local copy before the canonical bundle absorbed
them — but the canonical bundle is what establishes the authoritative
version. The note positions folio.love as the *source of friction
evidence*, not the *source of normative framework text*. The words
"cherry-picked ... into the canonical bundle" are the right framing:
the maintainer pulled fixes upward, then re-broadcast them downward.
The note closes with the correct directional instruction ("folio.love
can resync its local bundle from the johnny-os v1.1.1 merge SHA once
it lands"), which re-asserts the canonical → adopter direction.

The sync note does introduce a tension that deserves a named mention:
for the ~10-day window between folio.love merging PR #51 and the
canonical bundle merging v1.1.1, folio.love ran with a bundle that
diverged from canonical. This is exactly the state the line 171-174
contract warns against. The v1.1.1 backfill resolves the divergence
going forward, but the window existed. Alignment lens treats this as a
process learning for v1.2 (document the "adopter ahead of canonical"
flow explicitly — is it an allowed mode or a strictly-forbidden one?),
not a v1.1.1 blocker.

No adopter-authority drift in the normative framework files
themselves. The handoff document is non-normative (lives at
`docs/handoffs/`, not inside the bundle), so its wording does not
establish bundle doctrine.

Filed as ALIGN-MIN2 (process clarification candidate for v1.2).
