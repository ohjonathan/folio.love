## Preamble

This review executes the three-lens review board on the v1.0.0-rc bundle, as a single Claude session running Peer → Alignment → Adversarial → Consolidation in sequence. The three passes are run serially and not blended; each has its own lens, its own evidence, and its own verdict. The consolidation at the end reconciles my three findings. Cross-model synthesis across Claude / Codex / Gemini is performed downstream by an extraction step keyed on filenames; this file is the Claude input.

The bundle under review is a candidate portable meta-prompt framework extracted from the johnny-os Phase-1 playbook and the D2 orchestration retrospective. v1.0.0 is intended to stamp the bundle as usable doctrine for future deliverables in any project.

# Pass 1 — Peer Review

**Lens:** Is this good?
**Mindset:** Reviewing quality, completeness, clarity, operator usability. Finding problems, not compliance checks.

## 1.1 Completeness check

Present per build plan's directory layout:
- Doctrine surface: `framework.md` (234 lines), `playbook.md` (178), `tokens.md` (138), `README.md`, `CHANGELOG.md`.

- `manifest/` has `deliverable-manifest.schema.yaml`, `example-manifest.yaml`, `generator-spec.md`.

For a framework whose very first step is "Phase A — author a spec," absence of a dispatchable Phase A author template is a usability hole.

## Custom attack surfaces

The architect flagged five high-risk surfaces. I attack each.

### Attack-surface-1 — Portability leakage beyond the grep

The first-pass grep missed:
- **Directory assumption (`docs/`):** Every example path in tokens.md assumes a `docs/` subdirectory (`docs/specs/`, `docs/reviews/`, `docs/retros/`, `docs/incidents/`, `docs/logs/`). Projects with a different convention (`documentation/`, `Docs/`, `docs/src/`) inherit johnny-os's specific layout.
- **CLI tool assumption (`rg`):** The `forbidden_symbols` / `cardinality_assertions` machinery in the schema + example uses `rg` (ripgrep) commands exclusively. A project without ripgrep installed cannot run the scope-lock guards. Not mentioned as a dependency anywhere.
- **CLI tool assumption (`gh`):** tokens.md `<PR_TOOL>` example is `gh` (GitHub CLI). Projects on GitLab, Bitbucket, internal forges inherit GitHub-ness.
- **Git assumption (`git worktree`):** 00-orchestrator-runbook's dispatch lifecycle step 2 assumes `git worktree add`. Projects on git < 2.5 or non-git VCS fail silently.
- **Test runner assumption (`pytest`):** example-manifest smoke checks use `pytest -xvs`. tokens.md `<TEST_COMMAND>` example is `pytest -xvs`. Python-shaped, not tool-agnostic.
- **Date format assumption (ISO 8601):** `<DATE>` example is `2026-04-15`. No policy for locale-specific dates or filename-safe dates.
- **Reviewer model assignments baked in:** 00's required_tokens include `CLI_CLAUDE`, `CLI_CODEX`, `CLI_GEMINI`. These are three specific model families. A project with access to only OpenAI or only Mistral inherits a three-family assumption.
- **File-naming convention:** example-manifest.yaml `artifacts.verdict` uses `<phase>-<family>-<role>.md` — this pattern is the D2 pattern and bakes specific sub-phase structure into a portable bundle.
- **Session-archive convention:** tokens.md `<DOC_INDEX_ARCHIVE>` example fills `ontos log -e "<SLUG>"` — presumes a session-indexing tool.

**Adversarial finding:** The bundle passes the johnny-os string grep but inherits johnny-os's entire toolchain: macOS-first, Python-first, git worktree, ripgrep, GitHub CLI, three-model-family. A strict test would require `X-POR-1` style portability tags on every example.

### Attack-surface-2 — Token fragility

- **Partial substitution:** tokens.md says "A substitution pass must leave zero `<ANGLE_UPPER>` strings remaining in the dispatched prompt, excepting `<FINAL_REPORT_SCHEMA>` and other template-local placeholders explicitly marked in the template." But templates use `<ANGLE_UPPER>` as a meta-placeholder reference (e.g., `02-phase-dispatch-handoff.md:129` "No `<ANGLE_UPPER>` strings remaining"). A naive generator that greps for `<[A-Z_]+>` would falsely flag `<ANGLE_UPPER>` as unsubstituted.
- **Substitution with empty string:** `<FORBIDDEN_SYMBOLS:LIST?>` and `<CARDINALITY_ASSERTIONS:LIST?>` are optional. If not supplied, template 01 line 66–67 becomes `Forbidden symbols:  ` and `Cardinality checks: ` — empty section headers. The worker might interpret "no forbidden symbols" as permissive. A schema that pre-fills `none` as the literal string would prevent this.
- **YAML quoting:** `<DELIVERABLE_ID>` appears in both markdown prose AND frontmatter YAML in the same template (e.g., 01 frontmatter `id: <DELIVERABLE_ID>-<PHASE_ID>-<FAMILY>-peer`). If the operator fills in `<DELIVERABLE_ID>` with something containing a colon or hyphen, YAML parsing breaks. No escaping guidance.
- **Multi-occurrence coupling:** `<DELIVERABLE_ID>` appears 12+ times in 01. A manual edit that misses one occurrence yields an inconsistent dispatch prompt. The v2 generator is supposed to fix this but v1 operators are manual.

**Adversarial reproduction:** fill `<DELIVERABLE_ID>` with `auth: oauth2 refactor` (realistic project name with punctuation). Frontmatter `id: auth: oauth2 refactor-D.2-claude-peer` breaks YAML. Worker's session archive path breaks. Worker halts at first attempted write.
