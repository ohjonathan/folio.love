### 1.4 Cross-Artifact Coherence

The bundle is not fully coherent across doctrine, playbook, example, and schema:

- P3 says "the same model family may not hold more than one role on the same artifact in the same round" (`framework.md:96-100`). `playbook.md` then recommends double-role same-family sessions as one valid option (`playbook.md:97-100`). The walkthrough uses Codex as both peer and adversarial in Phase B (`examples/walkthrough-mini-deliverable.md:74-77`).
- The source playbook defines Phase B as B.1 through B.4 (`docs/reference/llm-development-playbook.md:796-818`) and Phase D as D.1 through D.6 (`docs/reference/llm-development-playbook.md:1097-1119`). The framework state machine compresses B and D into high-level phases (`framework.md:56-63`), while templates reintroduce D.2-D.6. Operators must bridge the naming.
- The manifest schema enum allows `B` and `D.3`, but the example uses `B.consolidate` (`manifest/example-manifest.yaml:47-50`), which schema does not allow (`manifest/deliverable-manifest.schema.yaml:104-107`).
- `framework.md` artifact contracts say family verdicts are `<DELIVERABLE_ID>-<phase>-<family>-<role>.md` (`framework.md:197`), while the source operational convention uses per-family verdicts plus role drafts (`ops/2026-04-13-claude-cli-cross-model-agentic-workflow.md:80-96`). The v1 bundle needs to choose whether a "family verdict" is one artifact or three role artifacts.
- `tokens.md` defines typed tokens (`<TOKEN:LIST>`, `<TOKEN?>`) but templates use untyped tokens. That makes exact validation ambiguous.

### 1.5 Operator UX Walkthrough

Walking `examples/walkthrough-mini-deliverable.md` as a new operator:

- Phase 0 is understandable, but validation is "by eye" (`examples/walkthrough-mini-deliverable.md:38-40`). A new operator has no mechanical validator, and the example manifest actually fails the schema.
- Phase A immediately requires a missing component: "a phase-A-author template" that is project-local (`examples/walkthrough-mini-deliverable.md:46-51`). This is the first hard stop.
- Phase B exposes a manifest assignment that violates P3 (`examples/walkthrough-mini-deliverable.md:71-77`). The walkthrough says the operator catches it manually, then offers a workaround that still contradicts the doctrine before settling on skipping a lens.
- Tokens are not always explicit in the walkthrough. Model assignments are present, but corrected assignments are prose, not an updated manifest.
- Phase C relies on the author role using the worker contract, but there is no implementation-author wrapping template equivalent to source playbook §10.4.
- D.5 says Gemini can only static-inspect but "all three confirm" (`examples/walkthrough-mini-deliverable.md:133-136`). The final gate later requires direct-run evidence for every yes (`templates/07-final-approval-gate.md:59-60`), so the evidentiary effect of Gemini's verifier is unclear.
- D.6 references scripts such as `scripts/spec-coverage.sh` and `scripts/no-open-blockers.sh` (`manifest/example-manifest.yaml:111-132`) that are neither supplied nor declared as required project prerequisites.
- The merge step says use fresh clone and `--no-ff`, but non-standard git setups (no GitHub, trunk-based workflow, protected branches, no local merge permission) are not covered.

### 2.4 Token System Integrity

Exact token comparison found these template tokens not exactly present in `tokens.md`: `<ARTIFACT_OUTPUT_PATHS>`, `<AUTHOR_RISK_LEVEL>`, `<CARDINALITY_ASSERTIONS>`, `<CLI_CLAUDE>`, `<CLI_CODEX>`, `<CLI_GEMINI>`, `<CONTROL_PLANE_HOST>`, `<CONTROL_PLANE_OS>`, `<DOC_INDEX_ACTIVATION>`, `<DOC_INDEX_ARCHIVE>`, `<FORBIDDEN_SYMBOLS>`, `<GATE_PREREQUISITES>`, `<INCIDENT_DATE>`, `<INCIDENT_SLUG>`, `<NO_TOUCH_PATHS>`, `<PACKAGE_MANIFEST>`, `<PHASE_ID+1>`, `<POWER_RESILIENCE_TOOL>`, `<REMOTE_ACCESS_TOOL>`, `<REVIEW_BOARD_FAMILIES>`, `<SCOPE_LOCK_PATHS>`, `<SESSION_TOOL>`, `<SMOKE_CHECKS>`, `<TRACKER_ROW_SCHEMA>`.

Many are false positives caused by glossary notation (`<TOKEN:LIST>` and `<TOKEN?>`). That still matters: a mechanical validator cannot know whether `<SMOKE_CHECKS>` satisfies `<SMOKE_CHECKS:LIST>` unless the grammar is specified.

Real orphan:

- `<PHASE_ID+1>` in `templates/00-orchestrator-runbook.md:140`. This is computed notation, not a declared substitution token.

Unused or underused glossary tokens:

- `<AUTHOR_FAMILY>`, `<META_CONSOLIDATOR_FAMILY>`, `<MODEL_ASSIGNMENTS:LIST>`, `<PR_TITLE_PATTERN>`, `<REPO_URL>`, `<STATIC_CHECKS:LIST?>`, `<DELIVERABLE_SLUG>`, `<SPEC_DIR>`, `<REVIEWS_DIR>`, `<RETRO_DIR>` are defined but not used in templates. Some appear in `framework.md` rather than templates; the glossary says every token should appear in a template or be reserved for v2.
- `<DOC_INDEX_TOOL?>` is defined and referenced in `framework.md`/`playbook.md`, but not used in templates. The orchestrator template lists it as optional frontmatter but does not use it.

Ambiguous tokens:

- `<BRANCH_CONVENTION>`: Is it a literal branch name, a pattern, or a rendered branch per phase? Templates use it as branch name in dispatch (`02:47`) while tokens define it as a pattern.
- `<ARTIFACT_OUTPUT_PATHS>`: Sometimes a single file, sometimes a list, sometimes a directory-like write scope.
- `<SCOPE_LOCK_PATHS>` and `<NO_TOUCH_PATHS>`: Rendering could be newline, comma, glob syntax, or prose list.
- `<FINAL_REPORT_SCHEMA>`: It is both an allowed straggler and a template-local section marker; it is not pure replacement.
- Optional command tokens: absence could mean empty string, omit section, or "not available"; templates do not define rendering.

Substitution semantics: not pure text replacement. The generator spec requires derived tokens from manifest paths (`generator-spec.md:70-80`), optional-token handling, per-phase path rendering, and computed artifact paths. This is a templating language, not simple angle-token substitution.

### 2.7 Manifest Schema Compliance

Field-by-field result:

- Required top-level fields are present: `manifest_version`, `id`, `slug`, `summary`, `scope`, `artifacts`, `model_assignments`, `smoke_checks`, `gate_prerequisites`.
- `scope` contains allowed/forbidden paths, forbidden symbols, and cardinality assertions. Good.
- `artifacts` includes required paths. However, a single `verdict` pattern is asked to serve both per-role family reviews and canonical verdicts, with no separate canonical field.
- `model_assignments` contains invalid phase `B.consolidate` (`example-manifest.yaml:49`) relative to the schema enum (`schema.yaml:104-107`).
- `regression_guards:` has no value (`example-manifest.yaml:103-105`), which YAML parses as null. If present, schema says it must be an array (`schema.yaml:145-154`).
- `cli_capability_matrix` is optional in the schema, but generator invariant #7 requires an entry for every family used (`generator-spec.md:96-103`). The schema cannot enforce a generator-required field.
- `smoke_checks.expect` accepts strings, but the example uses `>=1` (`example-manifest.yaml:99-101`). The generator spec does not define expression operators; schema text only gives examples like `exit-0` and exact stdout in cardinality.
- `gate_prerequisites` has 10 entries and matches minItems. Good, but several commands reference project-local scripts not declared elsewhere.

Generator-spec mismatch:

- Generator outputs `phase-B-consolidate-<family>.md` (`generator-spec.md:36-40`) but schema does not allow a `B.consolidate` phase.
- Generator says each prompt has zero `<ANGLE_UPPER>` strings except `<FINAL_REPORT_SCHEMA>` (`generator-spec.md:57-62`), but templates intentionally include markdown placeholder examples such as `<path>`, `<phase>`, `<family>`, and computed `<PHASE_ID+1>`.
- Generator says it enforces P3 by failing "if the same family holds two roles in one phase" (`generator-spec.md:82-83`), but the schema's assignment map cannot represent the same key twice; it also cannot know author family for a reviewed artifact unless phase relationships are modeled.

### 3.3 Portability Attack (Beyond the Grep)

String grep is not enough. The johnny-os DNA survives structurally:

- Model families are hard-coded culturally, even if not path-specific. `tokens.md` defines `<CLI_CLAUDE?>`, `<CLI_CODEX?>`, and `<CLI_GEMINI?>`; README and walkthrough name all three. A project using OpenAI-only, Anthropic-only, local models, or hosted APIs needs more than token substitution.
- The artifact layout assumes docs/reviews/specs/retros conventions. That is portable-ish, but it is still an opinionated documentation architecture.
- The process assumes git branch/worktree control and `--no-ff` merges. That fails for teams using squash-only protected branches, Gerrit, Perforce, or server-side merge queues.
- The example assumes Python, `pytest`, `rg`, and shell. That is fine as an example, but the final gate and manifest semantics do not abstract command expectation enough for non-shell environments.
- `10-infra-bootstrap.md` references Brewfile-like package manifests and service managers (`launchd`, `systemd`) rather than a truly generic control-plane capability model.
- The source operational pattern of family verdicts plus canonical verdict leaks into artifact naming, but the bundle changes the shape to per-role verdicts without fully adapting the consolidator.

### 3.4 Edge Case Inventory

- Deliverable with zero scope-locked artifacts: schema requires scope allowed/forbidden paths but not `minItems`, so empty arrays are allowed. Worker contract then receives empty allowed paths and may either halt or write nowhere. No "analysis-only deliverable" path exists.
- Deliverable that needs to skip Phase B under evidence-based patch mode: source §15.2 allows Phase A/B collapsed (`playbook.md:1512-1522`), but bundle has no variant template. Operator either violates framework or does unnecessary review.
- Hotfix workflow: source says abbreviated CA + Adversarial only (`playbook.md:1524-1533`). Bundle P4/P3 assumes three-lens boards. No hotfix gate.
- Track-based parallel implementation: source has strict zero-overlap conditions (`playbook.md:1544-1578`). Bundle has no track fields in manifest and no branch/worktree map for multiple C/D cycles.
- Tier 1 agent team workflow: source requires spawning exactly three reviewer agents for review team (`playbook.md:1719-1773`). Bundle templates are single-worker prompt files and do not encode agent-team activation.
- Operator running on a single model family: no degraded protocol. P3 makes review impossible.
- Continuation after halted run: template 11 requires a halt report and original dispatch. If a session dies before writing a halt report, it tells the worker to halt and request original redispatch. That is safe but not a recovery workflow.

### 3.5 Recursive Trap Analysis

The bundle is a review-board framework being reviewed by a review board. It fails some of the principles it wants reviewers to enforce:

- It declares "prompts are software" but ships no validator, no passing manifest example, and no token conformance test.
- It declares P3 model diversity but the built-in walkthrough cannot run a three-lens review without violating or weakening P3.
- It declares the adversarial role must stress failures, but the adversarial template is less rigorous than the review prompt used for this very report.
- It says final approval is not a vibe, but v1 relies on by-eye manifest validation and hand-composed prompts.
- Template 06 is not sufficient to consolidate this review. It expects families x roles. This task has one model running three sequential lenses, then internal consolidation. The bundle cannot represent that legitimate review mode without adaptation.
- Doctrine principles are sometimes unfalsifiable as written. P12 says prompt suites are validated before dispatch, but v1 does not define the validator or minimal checks beyond "zero `<ANGLE_UPPER>`."

### 3.6 Manifest / Generator Attack

Underspecified divergence points:

- Token rendering: comma list vs newline list vs fenced code block.
- Optional tokens: remove line, empty string, literal "none", or explicit "not configured."
- Command expectations: exact stdout, exit status, comparison operators (`>=1`), regex match, JSON assertions.
- Review-board composition: how to represent skipped lenses, same-family multiple sessions, author exclusion, or alternate model variants.
- Artifact path roles: one verdict pattern tries to cover role drafts and canonical verdicts.
- Branch map: one default pattern cannot express product branch vs worker branch vs merge clone.
- Capability matrix: schema does not require it, generator does.
- Missing token references: schema has no field declaring arbitrary extra tokens used by project-local templates.

Schema permits configurations doctrine may forbid:

- `model_assignments` can include only one family for B and still pass schema; P3 requires at least two.
- `scope.allowed_paths` can be empty; P6 expects meaningful scope locks.
- `cli_capability_matrix` can be omitted; generator invariant requires it.
- Author and reviewer relationships cannot be checked across phases, so schema can permit author-family self-review if phases are modeled separately.

Failure mode when manifest references a token not in `tokens.md`: there is no schema field for token references and no v1 validator. A generator might silently pass it through, fail late, or invent a value.

### 3.7 Adversarial Walkthrough

The first ambiguous instruction is Phase 0 validation: "in v1, by eye" (`examples/walkthrough-mini-deliverable.md:38-40`). A new operator does not know what schema compliance means, and the example is not compliant.

The first hard break is Phase A: the walkthrough requires "a phase-A-author template" and says it is project-local (`examples/walkthrough-mini-deliverable.md:46-51`). The bundle cannot start its default workflow from its own files.

The second hard break is Phase B: the manifest assigns Claude as adversarial while Claude authored the spec (`example-manifest.yaml:45-49`; walkthrough calls this a P3 violation at `examples:71-74`). The walkthrough's correction then uses two Codex sessions and labels that a P10 violation, which is doctrinally confused.

The third break is final approval: gate commands reference scripts not included in the bundle (`example-manifest.yaml:111-132`). A third-party operator following the example has to invent scripts or weaken the gate.

### 3.8 Blind Spot Identification

The architect appears to be optimizing for capturing the D2 lessons, not for a cold third-party install. The bundle's strongest parts are the lessons learned from real orchestration. The blind spot is that those lessons still assume an expert orchestrator who knows when to patch the process. The framework claims portability but has not replaced expert judgment with enforceable contracts.

What seems too simple: "fill tokens and dispatch." The tokens are a typed, derived, optional, phase-aware configuration language. Treating them as simple text placeholders is the fragile core.

Conspicuously absent: a minimal conformance suite. Even v1 without a generator should ship checks for token closure, schema example validity, required frontmatter coverage, and portability grep exceptions.

### 3.9 Risk Assessment Override

I override the implied low risk for a v1 documentation/spec bundle. This is medium-high risk for stamp because the artifact is intended to be executed by LLMs, and small ambiguities become workflow divergence.

Top risk vectors:

1. Invalid manifest/spec pairing: a future generator starts from a schema/example pair that already disagrees, producing incompatible prompt suites.
2. Review-board dispatch deadlock: P3 plus author-family exclusion plus three roles leaves no valid default path for common three-family workflows.
3. Token fragility: exact token mismatch and optional/list semantics cause partially substituted prompts that models may "helpfully" interpret rather than halt on.
