# 6. Issues Found

**Critical**

1. **Governance is still unresolved, so the spec is not approval-ready.** The spec now correctly says roadmap/PRD, ontology, and refresh amendments are required preconditions ([folio_provenance_spec.md#L1358](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1358)). But the live roadmap and Tier 3 tracker still define PR D as deliverable-to-evidence provenance with `depends_on`/`draws_from`/`impacts`, not evidence-version-lineage via `supersedes` ([04_Implementation_Roadmap.md#L437](/Users/jonathanoh/Dev/folio.love/docs/product/04_Implementation_Roadmap.md#L437), [tier3_kickoff_checklist.md#L92](/Users/jonathanoh/Dev/folio.love/docs/validation/tier3_kickoff_checklist.md#L92)). This is still a block, not a documentation footnote.

2. **The new same-coordinate stale repair model can preserve wrong canonical links.** Reruns suppress any proposal that lands on the same confirmed coordinates, even under `--force` or content changes ([folio_provenance_spec.md#L1041](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1041)). The only first-class repair is `reconfirm`, which just updates hashes from current content ([folio_provenance_spec.md#L955](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L955)). That means a materially changed claim/passage can be converted back to “fresh” without semantic rematching. For provenance, that is a correctness failure, not just a UX issue.

**Major**

1. **`acknowledged_stale` is internally contradictory.** `acknowledge` says staleness is noted ([folio_provenance_spec.md#L951](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L951)), but stale detection only classifies stale links when `acknowledged_stale` is false ([folio_provenance_spec.md#L989](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L989)). The test plan simultaneously says acknowledged stale links are still excluded from coverage ([folio_provenance_spec.md#L1194](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1194)). The lifecycle needs an explicit acknowledged-stale state in status/review, not a boolean that removes the link from one model but not the other.

2. **Dense-pair handling is deterministic but not practically sufficient.** Passage chunking is now precise, but claims are never chunked, so claims-only overflow hard-fails the pair ([folio_provenance_spec.md#L873](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L873)). Oversized slides are truncated from the front only ([folio_provenance_spec.md#L861](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L861)). On a corpus that already includes 137-slide evidence notes, that is a real operational limit, not a theoretical edge ([tier2_real_vault_validation_report.md#L163](/Users/jonathanoh/Dev/folio.love/docs/validation/tier2_real_vault_validation_report.md#L163)).

3. **The lock story is improved but still overstated.** The spec says commands “must not run concurrently,” yet enforcement remains advisory and explicitly relies on operator discipline ([folio_provenance_spec.md#L1085](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1085), [folio_provenance_spec.md#L1104](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1104)). Without explicit exclusive-create semantics and without protection from non-folio writers, there are still clobber paths.

4. **The slice is still badly sequenced against user value.** Rev 3 is honest that v1 produces zero output until humans confirm `supersedes` first ([folio_provenance_spec.md#L206](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L206)). The roadmap still presents PR D as the moment “deliverable claims” get connected to evidence ([04_Implementation_Roadmap.md#L437](/Users/jonathanoh/Dev/folio.love/docs/product/04_Implementation_Roadmap.md#L437)). This is not fatal if governance changes first, but until then it remains a sequencing red flag.

**Minor**

1. **The spec is not fully frozen while `supersedes` cardinality remains an open reviewer question.** The implementation contract assumes singular everywhere, but section 20 still asks whether that is the right v1 constraint ([folio_provenance_spec.md#L1324](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L1324)).

2. **`claim_hash` is probably too weak for provenance lifecycle decisions.** It is based on `claim_text` alone ([folio_provenance_spec.md#L755](/Users/jonathanoh/Dev/folio.love/docs/specs/folio_provenance_spec.md#L755)). If the grounding quote or extraction context changes but the claim text does not, stale detection and rejection suppression may miss a meaningful basis change.

# 7. Verdict

**Verdict: Block**

Unblock requirements:

1. Approve or land the upstream roadmap/PRD, ontology, and refresh amendments first, so PR D v1 is authorized by the project’s governing docs rather than just by this spec.
2. Redesign same-coordinate stale handling so changed confirmed links can be semantically revalidated. Either allow reproposal against stale confirmed links, or make `reconfirm` a true semantic confirmation flow rather than a hash refresh.
3. Replace the current `acknowledged_stale` boolean with an explicit surfaced lifecycle state that remains visible in review/status and has unambiguous coverage semantics.
4. Tighten the concurrency claim: either specify an actually race-safe lock acquisition contract or lower the safety language to match the remaining advisory behavior."}}
